## Данные

В реальном гемблинг-продукте события лежат в таблице вида «один игрок — одно действие в один момент времени»: регистрация, открытие сессии, депозит, ставка. Для когортного анализа удержания нам достаточно двух фактов: когда игрок зарегистрировался и в какие дни он возвращался (был активен). Сгенерируем это синтетически, но реалистично.

!!! note "Что делает реалистичный генератор"

    Мы закладываем три эффекта, которые есть в любом живом продукте. Первое: удержание падает по геометрическому закону — после первого дня уходит большая часть, дальше кривая выполаживается. Второе: когорты различаются по качеству — мы имитируем маркетинговые волны, когда в какие-то недели приходит более «липкий» трафик, а в какие-то льют дешёвый и плохой. Третье: есть weekly-сезонность — на выходных активность выше. Без этих эффектов heatmap будет скучным и ты не научишься его читать.

Это setup-код, а не задача — скопируй и запусти как есть. Модель проста: для каждого игрока берём дату регистрации, его личный «коэффициент липкости» (зависит от когорты + индивидуальный шум), и для каждого последующего дня бросаем монетку «вернулся / не вернулся» с вероятностью, которая падает со временем. Так мы получаем поток событий активности, неотличимый по структуре от настоящего.

!!! warning "Зачем фиксировать seed"

    Без `seed` каждый запуск даст другие числа, и ты не сможешь сверить результат с воркшопом или воспроизвести баг. Фиксированный seed — основа воспроизводимого анализа. Все критерии-`assert` ниже рассчитаны на `seed=42`.

```python
# generate_data.py
import numpy as np
import pandas as pd

RNG = np.random.default_rng(42)

N_PLAYERS = 8000
START = pd.Timestamp("2025-01-01")
HORIZON_DAYS = 90          # сколько дней наблюдаем после регистрации максимум
STUDY_END = START + pd.Timedelta(days=120)  # дата выгрузки данных

# 1. Регистрации: игроки приходят в течение ~10 недель неравномерно
reg_offsets = RNG.integers(0, 70, size=N_PLAYERS)
reg_dates = START + pd.to_timedelta(reg_offsets, unit="D")

# 2. Качество когорты по неделе регистрации: маркетинговые волны
reg_week = (reg_offsets // 7)
week_quality = {w: q for w, q in zip(range(10), [1.0, 1.1, 0.7, 0.75, 1.25, 1.3, 0.6, 0.9, 1.15, 1.05])}
base_stick = np.array([week_quality.get(w, 1.0) for w in reg_week])

# индивидуальная липкость вокруг качества когорты
player_stick = np.clip(base_stick * RNG.lognormal(mean=0.0, sigma=0.35, size=N_PLAYERS), 0.05, 3.0)

players = pd.DataFrame({
    "player_id": np.arange(N_PLAYERS),
    "reg_date": reg_dates,
    "reg_week": reg_week,
    "stick": player_stick,
})

# 3. Генерируем активность по дням
events = []
for row in players.itertuples(index=False):
    pid, reg, _, stick = row.player_id, row.reg_date, row.reg_week, row.stick
    # день 0 = день регистрации, всегда активен
    events.append((pid, reg))
    for d in range(1, HORIZON_DAYS + 1):
        day = reg + pd.Timedelta(days=d)
        if day > STUDY_END:
            break
        # базовая вероятность возврата падает: ~ stick / (d^0.55)
        p = stick / (d ** 0.55) * 0.45
        # weekly-сезонность: пт-сб-вс активнее
        if day.dayofweek in (4, 5, 6):
            p *= 1.25
        p = min(p, 0.95)
        if RNG.random() < p:
            events.append((pid, day))

events = pd.DataFrame(events, columns=["player_id", "event_date"])
events["event_date"] = pd.to_datetime(events["event_date"])

players[["player_id", "reg_date", "reg_week"]].to_parquet("players.parquet", index=False)
events.to_parquet("events.parquet", index=False)

print("players:", len(players), "| events:", len(events))
print(events.head())
```

**Критерий шага** — запусти сразу после генерации:

```python
assert len(players) == 8000, "ожидаем 8000 игроков"
assert len(events) == 66637, "при seed=42 ровно столько событий активности"
assert (events.groupby("player_id").size() >= 1).all(), "у каждого игрока минимум день регистрации"
print("OK: данные сгенерированы воспроизводимо")
```

Два файла: `players.parquet` (кто когда зарегистрировался) и `events.parquet` (когда был активен, по одной строке на день активности). 8000 игроков и 66 637 событий активности. Зерно зафиксировано (`seed=42`), поэтому у тебя получатся ровно те же числа, что в этом тексте.

!!! tip "Чем заменить на реальные данные"

    Структура «player_id + event_date» — это уже то, что лежит в любой продуктовой БД. Чтобы потренироваться на публичных данных, подставь датасет онлайн-ритейла: на Kaggle есть «Online Retail II» (UCI) с колонками CustomerID и InvoiceDate — переименуй их в player_id и event_date, и весь код ниже отработает без изменений. Для мобильной аналитики подойдёт «Google Analytics Sample» в BigQuery. Логика когорт от домена не зависит: игрок, покупатель, подписчик — всё это «сущность, которая возвращается или нет».

## Ход работы

### Шаг 1: Свести события к «игрок — дата активности»

**Зачем.** Когортный анализ оперирует не отдельными ставками, а фактом «игрок был активен в день X». Один игрок может сделать 50 ставок за день — для удержания это всё равно один активный день. Поэтому первым делом схлопываем события до уникальных пар `(player_id, event_date)`. Этот шаг убирает дубли и резко уменьшает объём.

**Задача.** Прочитай `players.parquet` и `events.parquet`. Построй DataFrame `activity` с уникальными парами `(player_id, event_date)`, присоедини `reg_date` из `players` и посчитай колонку `day_number` — число дней от регистрации до активного дня. Имена `activity` и `day_number` проверяет критерий.

??? tip "Подсказка"

    Уникальность — `drop_duplicates(["player_id", "event_date"])`. Разница дат в днях — `(activity["event_date"] - activity["reg_date"]).dt.days`. Не забудь привести обе колонки к `datetime`, иначе вычитание сломается.

**Критерий шага:**

```python
assert "day_number" in activity.columns, "нужна колонка day_number"
assert (activity["day_number"] == 0).sum() == 8000, "день регистрации (day 0) должен быть у всех 8000"
assert len(activity) == 66637, "после дедупликации столько уникальных активных дней"
assert activity["day_number"].min() == 0, "раньше регистрации активности быть не может"
print("OK: events схлопнуты в activity с day_number")
```

??? success "Решение"

    ```python
    import pandas as pd

    players = pd.read_parquet("players.parquet")
    events = pd.read_parquet("events.parquet")

    # уникальные активные дни на игрока
    activity = events.drop_duplicates(["player_id", "event_date"]).copy()

    # присоединяем дату регистрации, чтобы считать «возраст» активности
    activity = activity.merge(players[["player_id", "reg_date"]], on="player_id", how="left")
    activity["reg_date"] = pd.to_datetime(activity["reg_date"])
    activity["event_date"] = pd.to_datetime(activity["event_date"])

    # day_number = сколько дней прошло с регистрации до этого активного дня
    activity["day_number"] = (activity["event_date"] - activity["reg_date"]).dt.days

    print(activity[["player_id", "reg_date", "event_date", "day_number"]].head())
    ```

    **Почему так.** `day_number = 0` — день регистрации, `day_number = 1` — следующий день и так далее. Это ключевое поле: вокруг него строится вся когортная механика. `day_number` иногда называют «age» или «period» — возраст игрока в днях на момент активности. Дедупликация по `(player_id, event_date)` гарантирует, что один игрок за один день считается один раз, иначе удержание раздуется выше 100%.

### Шаг 2: Назначить когорты по неделе регистрации

**Зачем.** Когорта — это группа игроков, объединённых моментом входа. Брать дневные когорты при 8000 игроков — слишком дробно, числа будут шумными. Недельные когорты — стандарт для продуктовой аналитики: достаточно игроков в группе для устойчивой статистики и при этом видна динамика по времени. Когорту обозначаем датой понедельника той недели, когда игрок зарегистрировался.

**Задача.** Добавь колонку `cohort` (дата начала недели регистрации) и в `activity`, и в `players`. Посчитай Series `cohort_sizes` — число **уникальных игроков** в каждой когорте. Имена `cohort_sizes` и колонки `cohort` проверяет критерий.

??? tip "Подсказка"

    Метка недели — `reg_date.dt.to_period("W-SUN").dt.start_time` (понедельник той недели). Размер когорты — это `nunique()` по `player_id`, сгруппированному по `cohort`, считается по таблице `players`, а не по `activity`.

**Критерий шага:**

```python
assert len(cohort_sizes) == 11, "при разбросе регистраций в 10 недель получается 11 когорт"
assert cohort_sizes.sum() == 8000, "сумма размеров когорт = всем игрокам, каждый в одной когорте"
assert cohort_sizes.min() >= 200, "самая маленькая когорта всё ещё крупная (>= 200)"
print("OK: когорты назначены, размеры по уникальным игрокам")
```

??? success "Решение"

    ```python
    # понедельник недели регистрации = метка когорты
    activity["cohort"] = activity["reg_date"].dt.to_period("W-SUN").dt.start_time

    players["reg_date"] = pd.to_datetime(players["reg_date"])
    players["cohort"] = players["reg_date"].dt.to_period("W-SUN").dt.start_time

    cohort_sizes = players.groupby("cohort")["player_id"].nunique().rename("cohort_size")
    print(cohort_sizes)
    ```

    **Почему так.** `cohort_sizes` — знаменатель для всех расчётов удержания: сколько игроков всего пришло в каждую когорту (размеры от 222 до 843). Запомни правило: размер когорты считается по **игрокам**, а не по событиям, и фиксируется один раз. Если считать по строкам `activity`, знаменатель станет суммарной активностью, а не числом людей.

!!! warning "Период недели должен совпадать у когорты и у периодов"

    Мы используем `W-SUN` (неделя заканчивается воскресеньем, начинается понедельником). Если для когорт взять одну границу недели, а для периодов жизни — другую, числа поедут. Держи единую конвенцию по всему пайплайну — это аукнется на шаге 6.

Проверь понимание:

```text
Q: Почему размер когорты считают по уникальным игрокам (COUNT DISTINCT), а не по числу строк в activity?
[ ] По строкам быстрее и результат тот же
[x] У игрока много строк (по одной на активный день); счёт строк сделает знаменателем суммарную активность, а не число людей
[ ] pandas не умеет считать строки
> В activity на одного игрока приходится много дней активности. Размер когорты - это сколько ЛЮДЕЙ пришло, поэтому только уникальные player_id.
---
Q: Что произойдёт с heatmap, если взять дневные когорты вместо недельных при 8000 игроков?
[ ] Станет точнее и информативнее
[x] Когорты станут мелкими (~115 чел/день), проценты запрыгают из-за шума, heatmap станет рябым
[ ] Ничего не изменится
> Гранулярность когорты выбирается под объём данных. Мелкие дневные когорты дают шумные проценты, тренды тонут.
```

### Шаг 3: Перейти от дней к периодам жизни

### Шаг 4: Три способа посчитать удержание

**Зачем.** Слово «retention» в продукте означает минимум три разные метрики. Если не договориться какую считаешь — два аналитика получат разные числа и потеряют день на сверку. Разберёмся на примере игрока с активными периодами `[0, 1, 3]`:

- **Classic (N-day / N-week retention)**: игрок удержан в периоде $k$, только если он был активен именно в этом периоде. Для нашего игрока: периоды 0,1,3 — удержан; 2 — нет. Отвечает на вопрос «какая доля вернулась ровно на неделе $k$». Самая строгая и самая популярная метрика.
- **Rolling (unbounded / "returned on or after")**: игрок удержан в периоде $k$, если он был активен в периоде $k$ или в любом более позднем. Для нашего игрока период 2 считается удержанным, потому что позже (в периоде 3) он вернулся. Сглаживает «пропуски» и показывает, кто ещё не ушёл насовсем. Отвечает на вопрос «какая доля жива на момент $k$ или позже».
- **Range (bracket retention)**: удержание считается по диапазону — «вернулся хотя бы раз в течение периодов 1..$k$». Удобно для редко-возвращающихся продуктов (банки, страховки), где дневное удержание почти ноль, а помесячное осмысленно.

Формально, для когорты размера $N$ и числа удержанных в периоде $k$ равного $r_k$: $\text{retention}_k = r_k / N$. Разница между методами — только в определении того, кто попадает в $r_k$.

**Задача.** Напиши функцию `retention_rows(player_periods, cohort_sizes, max_period, method)`, возвращающую длинный DataFrame `cohort / period / retention` для метода `"classic"`, `"rolling"` или `"range"`. Посчитай три таблицы `classic`, `rolling`, `rng_ret` (для `max_period=6`) и собери сравнение средних по периодам в `cmp` (колонки `classic`, `rolling`, `range`). Эти имена проверяет критерий.

**Критерий шага:**

```python

assert (cmp["rolling"] >= cmp["classic"] - 1e-9).all(), "rolling никогда не ниже classic"

print("OK: три метрики посчитаны и согласованы")
```

??? success "Решение"

    ```python
    import numpy as np

    MAX_PERIOD = 6   # анализируем первые 7 недель жизни (0..6)

    def retention_rows(player_periods, cohort_sizes, max_period, method):
        out = []
        for cohort, grp in player_periods.groupby("cohort"):
            n = cohort_sizes.loc[cohort]
            for k in range(max_period + 1):
                retained = 0
                for periods in grp["active_periods"]:
                    pset = set(periods)
                    if method == "classic":
                        retained += k in pset
                    elif method == "rolling":
                        retained += any(p >= k for p in pset)
                    elif method == "range":
                        # активен хотя бы раз в диапазоне [1, k]; период 0 = база
                        retained += (k == 0) or any(1 <= p <= k for p in pset)
                out.append({"cohort": cohort, "period": k, "retention": retained / n})
        return pd.DataFrame(out)

    classic = retention_rows(player_periods, cohort_sizes, MAX_PERIOD, "classic")
    rolling = retention_rows(player_periods, cohort_sizes, MAX_PERIOD, "rolling")
    rng_ret = retention_rows(player_periods, cohort_sizes, MAX_PERIOD, "range")

    # сравним среднее по всем когортам в каждом периоде
    cmp = (
        classic.groupby("period")["retention"].mean().rename("classic").to_frame()
        .join(rolling.groupby("period")["retention"].mean().rename("rolling"))
        .join(rng_ret.groupby("period")["retention"].mean().rename("range"))
    )
    print((cmp * 100).round(1))
    ```

    **Почему так.** Ты увидишь характерную картину (в процентах):

    | период | classic | rolling | range |
    |---|---|---|---|
    | 0 | 100.0 | 100.0 | 100.0 |
    | 1 | 64.1 | 98.2 | 64.1 |
    | 2 | 54.3 | 96.6 | 81.4 |
    | 3 | 46.8 | 94.4 | 88.0 |
    | 6 | 36.2 | 78.5 | 95.8 |

    Classic самое низкое (строгое требование «именно в этот период»), rolling выше (засчитывает любую будущую активность), range растёт с шириной окна. Все три равны 100% на периоде 0. Главный вывод: всегда подписывай, какой retention показываешь, иначе число вводит в заблуждение.

!!! example "Когда какой метод"

    Classic — для продуктов с ожидаемой ежедневной/еженедельной активностью (игры, гемблинг, соцсети): важно, вернулся ли игрок именно сейчас. Rolling — когда нужно понять, кто из когорты ещё не ушёл насовсем (churn-анализ, оценка «живой базы»). Range — для редкого использования (финансы, e-commerce с длинным циклом покупки), где «активен хотя бы раз в месяц» осмысленнее, чем «активен в конкретный день».

Проверь понимание:

```text
Q: Игрок активен в периодах [0, 2]. Как его засчитают classic и rolling на периоде 1?
[ ] Оба засчитают удержанным
[x] Classic - НЕ удержан (нет периода 1); rolling - удержан, потому что позже (в периоде 2) он вернулся
[ ] Оба не засчитают
> Classic смотрит ровно на период 1 и не находит игрока. Rolling смотрит "период 1 или позже" - находит активность в периоде 2, значит на момент периода 1 игрок ещё жив.
---
Q: Почему rolling-retention всегда не ниже classic в том же периоде?
[ ] Это совпадение на наших данных
[x] Условие rolling ("активен в k ИЛИ позже") включает условие classic ("активен в k") как частный случай
[ ] rolling считается по большей когорте
> Любой, кто удержан по classic в периоде k, активен в k, а значит автоматически удержан и по rolling. Обратное неверно. Поэтому rolling >= classic всегда.
```

### Шаг 6: То же самое через DuckDB (SQL-ветка)

**Зачем.** В проде когорты чаще считают прямо в SQL — не тащить же миллионы событий в pandas. DuckDB позволяет писать аналитический SQL по parquet-файлам без сервера, прямо из Python. Покажем, что та же classic-матрица получается одним запросом. Это важный навык: уметь воспроизвести расчёт на той стороне, где лежат данные.

**Задача.** Напиши SQL-запрос к `events.parquet` и `players.parquet`, который повторяет classic-матрицу: CTE для активности, для размеров когорт и для удержанных, финальный select с делением. Выполни его в `sql_long`, сверь с pandas-веткой через merge и сохрани максимальное расхождение в `diff`. Имена `sql_long` и `diff` проверяет критерий.

!!! warning "Ловушка деления периода в DuckDB"

    В DuckDB оператор `/` — это деление с плавающей точкой: `(event_date - reg_date) / 7` даст `1.142857`, а не `1`. Тогда `GROUP BY period` разнесёт данные по дробным периодам, и матрица не сойдётся с pandas. Для номера недели нужно **целочисленное деление** `// 7`. Это типичная ошибка переноса pandas-логики (`//`) в SQL.

**Критерий шага:**

```python

assert diff < 0.01, "при // 7 и единой границе недели SQL и pandas совпадают почти точно"
print(f"OK: SQL-матрица сошлась с pandas, max расхождение = {diff:.2e}")
```

??? success "Решение"

    ```python
    import duckdb

    con = duckdb.connect()

    query = """
    WITH activity AS (
        SELECT DISTINCT
            e.player_id,
            CAST(e.event_date AS DATE) AS event_date,
            CAST(p.reg_date AS DATE)   AS reg_date,
            date_trunc('week', CAST(p.reg_date AS DATE)) AS cohort,
            (CAST(e.event_date AS DATE) - CAST(p.reg_date AS DATE)) // 7 AS period
        FROM read_parquet('events.parquet') e
        JOIN read_parquet('players.parquet') p USING (player_id)
    ),
    sizes AS (
        SELECT date_trunc('week', CAST(reg_date AS DATE)) AS cohort,
               COUNT(DISTINCT player_id) AS n
        FROM read_parquet('players.parquet')
        GROUP BY 1
    ),
    retained AS (
        SELECT cohort, period, COUNT(DISTINCT player_id) AS r
        FROM activity
        WHERE period BETWEEN 0 AND 6
        GROUP BY 1, 2
    )
    SELECT r.cohort,
           r.period,
           r.r * 1.0 / s.n AS retention
    FROM retained r
    JOIN sizes s USING (cohort)
    ORDER BY r.cohort, r.period
    """

    sql_long = con.execute(query).df()
    sql_long["period"] = sql_long["period"].astype(int)

    # сверка с pandas-веткой
    merged = sql_long.merge(classic, on=["cohort", "period"], suffixes=("_sql", "_pd"))
    diff = (merged["retention_sql"] - merged["retention_pd"]).abs().max()
    print("Макс. расхождение SQL vs pandas:", diff)
    ```

    **Почему так.** При целочисленном `// 7` и одинаковой границе недели (DuckDB `date_trunc('week')` начинает неделю с понедельника, и `to_period("W-SUN").start_time` в pandas — тоже понедельник) обе ветки дают **идентичные** числа: `diff = 0.0`. Если бы оставил `/ 7`, расхождение подскочило бы до ~0.5 — деление перестало бы давать целые периоды. Большое расхождение в проде — это всегда сигнал, что когорты или периоды нарезаны по разным правилам в двух ветках.

!!! tip "Polars вместо pandas"

    Если данных много, ту же матрицу в polars: `df.group_by(["cohort","period"]).agg(pl.col("player_id").n_unique())` и затем `pivot`. API почти зеркальный pandas, но ленивый режим (`scan_parquet().group_by(...).collect()`) обрабатывает десятки миллионов строк в разы быстрее. На учебных 8000 игроков разница незаметна, на проде — критична.

Проверь понимание:

```text
Q: Почему в SQL-запросе важен COUNT(DISTINCT player_id), а не COUNT(*)?
[ ] DISTINCT работает быстрее
[x] У игрока может быть несколько активных дней в одном периоде; COUNT(*) посчитал бы их все и завысил числитель выше 100%
[ ] COUNT(*) не работает с GROUP BY
> Удержание - это доля ЛЮДЕЙ, вернувшихся в период. COUNT(*) считает строки-события, поэтому удержание могло бы превысить 100%. DISTINCT player_id считает уникальных людей.
---
Q: SQL- и pandas-матрицы вдруг разошлись на 5-10%. Самая вероятная причина?
[ ] Машинная погрешность float
[x] Разъехались границы недели или способ деления периода (/ вместо //) между ветками
[ ] DuckDB округляет иначе
> Машинная погрешность - это 1e-15, а не проценты. Расхождение в проценты означает разную нарезку когорт/периодов: разная граница недели или float-деление вместо целочисленного.
```

**Зачем.** Числа и графики сами по себе не выводы. Аналитик ценен тем, что переводит матрицу в утверждения, на которые можно действовать. Сформулируем выводы по трём осям: где обрыв во времени, какие когорты лучше/хуже, и что это значит для продукта.

Имена `avg_curve`, `worst_period`, `gap` проверяет критерий.

**Критерий шага:**

## Типичные ошибки

- **Считать удержание по событиям, а не по игрокам.** `COUNT(*)` вместо `COUNT(DISTINCT player_id)` завышает числитель и даёт удержание больше 100%. Знаменатель (размер когорты) тоже всегда по уникальным игрокам и фиксируется один раз.
- **Молча смешивать три определения retention.** Classic, rolling и range дают разные числа (на неделе 2: 64% / 98% / 64%). Самый частый конфликт отчётов: один считал classic, другой rolling, оба назвали это «retention». Всегда подписывай метрику.
- **Float-деление периода в SQL.** `(event_date - reg_date) / 7` в DuckDB даёт дробный период, и матрица разъезжается с pandas (~0.5 расхождение). Для номера недели нужно целочисленное `// 7`.
- **Survivorship bias на правом крае матрицы.** Молодые когорты физически не успели прожить 6 недель к дате выгрузки. Если не обрезать незавершённые периоды, последние строки матрицы будут с пропусками или искусственно завышены. Для каждой когорты считай только периоды, полностью уместившиеся в окно (`event_date <= STUDY_END`).
- **Разные границы недели в разных ветках.** `date_trunc('week')` в DuckDB и `to_period("W-SUN")` в pandas должны давать одну границу (понедельник). Если когорты нарезаны по-разному, SQL и pandas не сойдутся. Зафиксируй одну конвенцию.
- **Дневные когорты на малых данных (senior-ловушка).** Соблазн «больше гранулярности» при 8000 игроков делает когорты по ~115 человек — числа становятся шумными, тренды тонут. Гранулярность когорты выбирается под объём, а не под желание детализации.
- **Делать выводы по мелким когортам.** Без фильтра по размеру «лучшая когорта» легко окажется случайным выбросом из 25 человек. Здесь все когорты крупные (≥222), но привычку фильтровать держи всегда.

## Развитие

- **Revenue-когорты.** Замени бинарное «активен/нет» на сумму депозитов в периоде и построй когортную матрицу по выручке (LTV-кривые). Тот же каркас, но в ячейках деньги, а не доли — путь к когортному LTV.
- **Retention по сегментам.** Раздели когорты ещё и по каналу привлечения или гео (добавь поле в генератор) и сравни heatmap по сегментам — найди, какой канал даёт удерживаемый трафик.
- **Аппроксимация кривой удержания.** Подгони к усреднённой кривой степенную модель $r(t) = a \cdot t^{-b}$ или экспоненту и прогнозируй удержание за горизонтом наблюдения — основа для оценки LTV до того, как когорта дожила.
- **Полный SQL-пайплайн на rolling и range.** Перепиши шаг 4 целиком на DuckDB с оконными функциями (`MAX(period) OVER ...` для rolling), чтобы все три метрики считались на стороне БД.
- **Интерактивный дашборд.** Заверни матрицу и графики в Streamlit или собери в DataLens, добавь фильтр по метрике (classic/rolling/range) и по минимальному размеру когорты.
