## Что ты построишь

К концу воркшопа у тебя будет рабочий антифрод-пайплайн без размеченных меток фрода. На входе — таблица транзакций гемблинг-платформы (депозиты, ставки, выводы, начисления бонусов). На выходе — **ранжированный список топ-N подозрительных кейсов** с anomaly score, человеко-читаемым обоснованием по каждому ("почему этот игрок подозрителен"), оценкой **precision@k** на injected-аномалиях и описанием хэндоффа в операционную команду.

Это не "обучили классификатор на размеченных фродах" — таких меток в реальности нет или их единицы. Это **unsupervised anomaly detection**: ты ищешь то, что отклоняется от нормы, ранжируешь по степени отклонения и отдаёшь людям на проверку только верхушку, потому что ручной разбор стоит дорого. Артефакт — это `suspicious_cases.csv` плюс короткая записка для риск-команды, которую не стыдно показать.

Почему это важно именно в гемблинге: фрод тут не "украли карту", а тонкие схемы — bonus abuse (выжать бонус и вывести без отыгрыша), мультиаккаунтинг (один человек = десять "игроков" под один реферальный или welcome-бонус), сговорные паттерны ставок, отмывание через минимальный риск. Все они проявляются как **статистические аномалии в поведении**, а не как явный сигнал.

!!! info "Контекст модулей"

    Воркшоп опирается на M15 (unsupervised learning): кластеризация, метрики расстояния, методы детекции аномалий. Если плывёшь в том, почему StandardScaler обязателен перед distance-based методами или что такое contamination — вернись к M15. Признаковую инженерию мы делаем руками, без отдельного feature store.

## Данные

В реальном проекте ты берёшь транзакции из прода. Здесь мы генерируем **синтетический, но реалистичный** журнал транзакций с **впрыснутыми аномалиями известной доли** — это ключевой приём. Раз мы сами знаем, кто аномален, мы сможем честно посчитать precision@k и не обманывать себя. В проде этой роскоши нет, поэтому синтетика — твой полигон для калибровки.

Генератор моделирует три популяции: массу нормальных игроков и три типа злоумышленников. Seed зафиксирован.

```python
import numpy as np
import pandas as pd

rng = np.random.default_rng(42)

N_NORMAL = 4000
N_BONUS_ABUSE = 90
N_MULTI = 80
N_BET_ANOMALY = 50
CONTAMINATION_TRUE = (N_BONUS_ABUSE + N_MULTI + N_BET_ANOMALY) / (
    N_NORMAL + N_BONUS_ABUSE + N_MULTI + N_BET_ANOMALY
)

def make_normal(n, start_id):
    pid = np.arange(start_id, start_id + n)
    deposits_cnt = rng.poisson(8, n) + 1
    deposit_sum = np.round(rng.gamma(3, 40, n) * deposits_cnt, 2)
    bets_cnt = rng.poisson(120, n) + 5
    bet_sum = np.round(deposit_sum * rng.uniform(3, 12, n), 2)
    withdraw_cnt = rng.poisson(2, n)
    withdraw_sum = np.round(deposit_sum * rng.uniform(0.1, 0.9, n) * (withdraw_cnt > 0), 2)
    bonus_sum = np.round(deposit_sum * rng.uniform(0, 0.3, n), 2)
    account_age_days = rng.integers(30, 900, n)
    distinct_ip = rng.poisson(1.5, n) + 1
    distinct_device = np.minimum(distinct_ip, rng.poisson(1.2, n) + 1)
    avg_bet = np.round(bet_sum / bets_cnt, 2)
    max_bet_share = rng.uniform(0.01, 0.15, n)
    return pd.DataFrame(dict(
        player_id=pid, deposits_cnt=deposits_cnt, deposit_sum=deposit_sum,
        bets_cnt=bets_cnt, bet_sum=bet_sum, withdraw_cnt=withdraw_cnt,
        withdraw_sum=withdraw_sum, bonus_sum=bonus_sum,
        account_age_days=account_age_days, distinct_ip=distinct_ip,
        distinct_device=distinct_device, avg_bet=avg_bet,
        max_bet_share=max_bet_share, label="normal"))

def make_bonus_abuse(n, start_id):
    df = make_normal(n, start_id)
    df["label"] = "bonus_abuse"
    df["bonus_sum"] = np.round(df["deposit_sum"] * rng.uniform(0.8, 2.0, n), 2)
    df["bet_sum"] = np.round(df["deposit_sum"] * rng.uniform(0.5, 1.2, n), 2)  # почти не отыгрывают
    df["withdraw_cnt"] = rng.integers(1, 3, n)
    df["withdraw_sum"] = np.round((df["deposit_sum"] + df["bonus_sum"]) * rng.uniform(0.7, 1.0, n), 2)
    df["account_age_days"] = rng.integers(1, 14, n)  # свежие
    df["bets_cnt"] = rng.poisson(15, n) + 1
    df["avg_bet"] = np.round(df["bet_sum"] / df["bets_cnt"], 2)
    return df

def make_multiaccount(n, start_id):
    df = make_normal(n, start_id)
    df["label"] = "multiaccount"
    df["distinct_ip"] = rng.integers(4, 12, n)       # много игроков с одного IP
    df["distinct_device"] = rng.integers(3, 9, n)
    df["account_age_days"] = rng.integers(1, 20, n)
    df["deposit_sum"] = np.round(rng.gamma(2, 25, n), 2)  # мелкие однотипные депозиты
    df["bonus_sum"] = np.round(df["deposit_sum"] * rng.uniform(0.5, 1.0, n), 2)
    return df

def make_bet_anomaly(n, start_id):
    df = make_normal(n, start_id)
    df["label"] = "bet_anomaly"
    df["max_bet_share"] = rng.uniform(0.6, 0.98, n)   # ва-банк одной ставкой
    df["bets_cnt"] = rng.poisson(6, n) + 1
    df["bet_sum"] = np.round(df["deposit_sum"] * rng.uniform(0.9, 1.1, n), 2)
    df["avg_bet"] = np.round(df["bet_sum"] / df["bets_cnt"], 2)
    df["withdraw_sum"] = np.round(df["deposit_sum"] * rng.uniform(0.0, 1.8, n), 2)
    return df

parts = [
    make_normal(N_NORMAL, 0),
    make_bonus_abuse(N_BONUS_ABUSE, 100000),
    make_multiaccount(N_MULTI, 200000),
    make_bet_anomaly(N_BET_ANOMALY, 300000),
]
df = pd.concat(parts, ignore_index=True).sample(frac=1, random_state=42).reset_index(drop=True)
print(df.shape, "true contamination =", round(CONTAMINATION_TRUE, 4))
df.head()
```

Что получилось: ~4220 строк, по одной на игрока, с агрегированными признаками поведения. Колонка `label` — это **ground truth**, который мы используем только для оценки и прячем от моделей. Истинная доля аномалий ~5.3%.

!!! warning "label не должен попасть в модель"

    Самая частая ошибка новичка — оставить `label` в матрице признаков. Тогда модель "детектит" по утечке, метрики космос, в проде ноль. Мы будем явно собирать матрицу без него.

Чем заменить на реальные данные: подойдёт публичный **Credit Card Fraud Detection** (Kaggle, ULB) — там 284k транзакций с сильным дисбалансом и колонкой Class, которую так же прячут при обучении и используют только для оценки. Для поведенческой гемблинг-специфики ближе **PaySim** (Kaggle) — синтетический мобильный money-transfer лог с типами CASH_OUT/TRANSFER и флагом isFraud. Логика воркшопа переносится один в один: спрятать метку, собрать поведенческие агрегаты, ранжировать по score.

## Ход работы

### Шаг 1: Разобрать паттерны фрода до кода

**Зачем.** Если не понимаешь, как выглядит злоупотребление в данных, не соберёшь признаки, которые его ловят. Детектор аномалий найдёт "что-то странное", но странное бывает и легальным (VIP с огромными ставками — не фрод). Признаки должны кодировать гипотезы о схемах.

Три схемы, которые мы заложили в генератор, и их сигнатуры:

- **Bonus abuse.** Игрок берёт бонус, минимально его отыгрывает (низкий `bet_sum / (deposit_sum + bonus_sum)`), быстро выводит. Сигнал: высокое отношение бонуса к депозиту, низкий wagering ratio, свежий аккаунт, withdraw близок к deposit+bonus.
- **Мультиаккаунтинг.** Один человек регистрирует пачку аккаунтов под welcome-бонусы. Сигнал: много distinct IP/device на игрока, мелкие однотипные депозиты, свежие аккаунты, кучность.
- **Аномальные ставки.** Ва-банк одной ставкой (`max_bet_share` близко к 1), мало ставок, цель — обналичить депозит с минимальным оборотом (низкий риск для отмывания). Сигнал: высокая доля максимальной ставки, низкий bets_cnt.

**Что получилось.** У нас есть карта "схема → наблюдаемые признаки". Дальше превращаем её в числа.

!!! question "Проверь себя"

    1. Почему нельзя просто обучить классификатор на колонке label?
    2. VIP-игрок делает ставки в 50 раз выше средних. Это аномалия? Должен ли антифрод его флагать?
    3. Какой признак отличает bonus abuse от честного игрока, который просто взял welcome-бонус?

??? success "Ответы"

    1. В реальности размеченного label почти нет (фрод выявляют пост-фактум, единичные кейсы), а в нашей синтетике он — оценочный ground truth; его утечка в признаки даёт фиктивное качество. Антифрод по определению работает там, где меток нет — отсюда unsupervised.
    2. Статистически да, это outlier. Но это **легальная** аномалия. Поэтому anomaly score — не приговор, а приоритет для ручного разбора. VIP отсеется на шаге ручного ревью и через бизнес-правила (whitelist). Anomaly detection ищет необычное, бизнес решает, плохое ли оно.
    3. Wagering ratio: честный отыгрывает бонус (высокий bet_sum относительно бонуса), abuser выводит почти без оборота. Плюс возраст аккаунта и паттерн вывода.

### Шаг 2: Собрать признаки

**Зачем.** Сырые суммы зависят от масштаба активности (богатый игрок депозитит больше — это не фрод). Нужны **относительные, поведенческие** признаки: отношения и доли, инвариантные к объёму. Они отделяют "много денег" от "странного поведения".

```python
eps = 1e-6
feat = pd.DataFrame()
feat["player_id"] = df["player_id"]

# поведенческие отношения, а не абсолютные суммы
feat["wagering_ratio"] = df["bet_sum"] / (df["deposit_sum"] + df["bonus_sum"] + eps)
feat["bonus_to_deposit"] = df["bonus_sum"] / (df["deposit_sum"] + eps)
feat["withdraw_to_inflow"] = df["withdraw_sum"] / (df["deposit_sum"] + df["bonus_sum"] + eps)
feat["ip_per_account"] = df["distinct_ip"]
feat["device_per_ip"] = df["distinct_device"] / (df["distinct_ip"] + eps)
feat["max_bet_share"] = df["max_bet_share"]
feat["bets_per_deposit"] = df["bets_cnt"] / (df["deposits_cnt"] + eps)
feat["avg_bet_to_deposit"] = df["avg_bet"] / (df["deposit_sum"] / (df["deposits_cnt"] + eps) + eps)
feat["account_age_days"] = df["account_age_days"]
feat["deposit_sum_log"] = np.log1p(df["deposit_sum"])

FEATURES = [c for c in feat.columns if c != "player_id"]
X = feat[FEATURES].replace([np.inf, -np.inf], np.nan).fillna(0.0)
print(X.describe().T[["mean", "std", "min", "max"]])
```

**Что получилось.** Матрица из 10 поведенческих признаков. Обрати внимание: `label` тут нет вообще — мы его не трогали. Каждый признак — закодированная гипотеза с шага 1. `wagering_ratio` ловит bonus abuse, `ip_per_account` — мультиаккаунтинг, `max_bet_share` — аномальные ставки.

!!! tip "Хэндофф признаков в SQL"

    На проде эти агрегаты считаются не в pandas, а в duckdb/витрине. Пример формы запроса (псевдо): `SELECT player_id, SUM(bet)/NULLIF(SUM(deposit)+SUM(bonus),0) AS wagering_ratio, COUNT(DISTINCT ip) AS ip_per_account FROM tx GROUP BY player_id`. Логика идентична — важно, чтобы офлайн-обучение и онлайн-скоринг считали признаки **одинаково**, иначе train/serve skew.

### Шаг 3: Масштабирование

**Зачем.** LOF и (в меньшей степени) Isolation Forest чувствительны к масштабу. `account_age_days` живёт в сотнях, `device_per_ip` — около единицы. Без стандартизации distance-based LOF будет смотреть почти только на возраст аккаунта. StandardScaler приводит всё к нулевому среднему и единичной дисперсии.

```python
from sklearn.preprocessing import StandardScaler

scaler = StandardScaler()
Xs = scaler.fit_transform(X)
```

**Что получилось.** Матрица `Xs` готова к подаче в модели. Iso Forest формально к масштабу устойчив (он режет по осям), но единый scaling упрощает сравнение и нужен для LOF — держим один пайплайн.

### Шаг 4: Isolation Forest

**Зачем.** Isolation Forest — рабочая лошадь промышленного антифрода. Идея: аномалии "изолируются" случайными разрезами за меньшее число шагов, чем плотная норма. Он линейно масштабируется, не требует расстояний, хорошо тянет много признаков. `contamination` — ожидаемая доля аномалий; задаём близко к нашей оценке, но помним, что в проде это гиперпараметр калибровки, а не известная константа.

```python
from sklearn.ensemble import IsolationForest

iso = IsolationForest(
    n_estimators=300,
    contamination=0.05,
    max_samples="auto",
    random_state=42,
    n_jobs=-1,
)
iso.fit(Xs)
# score_samples: выше = нормальнее. Инвертируем, чтобы "выше = аномальнее"
iso_score = -iso.score_samples(Xs)
```

**Что получилось.** Вектор `iso_score`, где большие значения — кандидаты в аномалии. Знак инвертируем намеренно: sklearn отдаёт "чем выше, тем нормальнее", а нам для ранжирования удобнее обратное.

!!! note "Про contamination"

    contamination влияет на порог метки `predict` (-1/1), но **не на ранжирование** `score_samples`. Поскольку мы работаем precision@k по score, точное значение contamination для топа не критично — оно важно, когда тебе нужна бинарная метка по умолчанию. Это частый источник путаницы.

### Шаг 5: Local Outlier Factor

**Зачем.** Iso Forest ловит глобальные аномалии. LOF ловит **локальные** — точку, которая нормальна глобально, но выбивается из своего соседства (мультиаккаунты образуют плотный странный кластер; кто-то на краю плотной группы). Две модели с разной индуктивной предпосылкой дают взаимодополняющий взгляд. Для скоринга всего набора используем `novelty=False` и `fit_predict`/`negative_outlier_factor_`.

```python
from sklearn.neighbors import LocalOutlierFactor

lof = LocalOutlierFactor(n_neighbors=35, contamination=0.05, n_jobs=-1)
lof.fit_predict(Xs)
lof_score = -lof.negative_outlier_factor_  # выше = аномальнее
```

**Что получилось.** Вектор `lof_score`. `n_neighbors=35` — баланс: слишком мало шумит, слишком много размывает локальность. Это гиперпараметр, который стоит варьировать (20-50) и смотреть стабильность топа.

!!! warning "LOF для онлайн-скоринга"

    `LocalOutlierFactor` по умолчанию transductive — скорит ровно тот набор, на котором обучен, и не имеет `predict` для новых точек. Для онлайна нужен `LocalOutlierFactor(novelty=True)` и отдельный `fit` на "чистой" норме. В этом воркшопе мы скорим весь батч офлайн, поэтому novelty=False корректен.

### Шаг 6: Ранжирование и ансамбль

**Зачем.** Score двух моделей в разных шкалах — складывать сырыми нельзя. Приводим каждый к рангам (или к [0,1]) и усредняем. Ансамбль рангов устойчивее к выбросам шкалы, чем среднее сырых score. Финальный ранг — приоритет для ручной проверки.

```python
from scipy.stats import rankdata

res = df[["player_id", "label"]].copy()
res["iso_score"] = iso_score
res["lof_score"] = lof_score
res["iso_rank"] = rankdata(iso_score) / len(res)
res["lof_rank"] = rankdata(lof_score) / len(res)
res["ensemble_score"] = (res["iso_rank"] + res["lof_rank"]) / 2
res = res.sort_values("ensemble_score", ascending=False).reset_index(drop=True)
res.head(15)[["player_id", "label", "iso_rank", "lof_rank", "ensemble_score"]]
```

**Что получилось.** Отсортированная таблица: вверху — самые подозрительные. Колонку `label` оставляем для оценки (шаг 7) — в проде её здесь не будет. Уже глазами видно, что в топе доминируют bonus_abuse и multiaccount.

!!! question "Проверь себя"

    1. Почему мы усредняем ранги, а не сырые iso_score и lof_score?
    2. Что концептуально ловит LOF, чего не видит Isolation Forest?
    3. Если поднять contamination с 0.05 до 0.2, изменится ли порядок в топ-50?

??? success "Ответы"

    1. Сырые score в разных шкалах и распределениях (iso_score примерно 0.4-0.7, lof_score может уходить далеко). Сумма даст перекос в сторону модели с большим разбросом. Ранги нормируют обе в [0,1] и делают вклад сопоставимым.
    2. Локальные аномалии — точки, нормальные по глобальной плотности, но выбивающиеся из своего ближайшего окружения. Iso Forest режет по осям глобально и такие кейсы (плотный аномальный микрокластер) может пропустить.
    3. Порядок ранжирования по score почти не изменится — contamination двигает бинарный порог, а не score_samples. Изменится число точек, помеченных как -1 по умолчанию.

### Шаг 7: Оценка precision@k

**Зачем.** Recall в антифроде честно посчитать нельзя — ты не знаешь всех фродов в проде. Зато можешь измерить **precision@k**: из топ-k, которые отдал на проверку, сколько реально аномальны. Это прямо отражает экономику: команда проверяет фиксированный бюджет кейсов в день, и нам важно, чтобы в этих k было максимум настоящих фродов. На синтетике у нас есть label, поэтому считаем честно и калибруем k.

```python
def precision_at_k(frame, k, label_col="label"):
    top = frame.head(k)
    hits = (top[label_col] != "normal").sum()
    return hits / k

for k in [20, 50, 100, 150, 200]:
    p = precision_at_k(res, k)
    print(f"precision@{k:>3} = {p:.3f}  (попаданий {int(p*k)} из {k})")

# базовый уровень: доля аномалий в популяции
base_rate = (res["label"] != "normal").mean()
print(f"\nbase rate (случайный отбор) = {base_rate:.3f}")
print(f"lift@100 = {precision_at_k(res, 100) / base_rate:.1f}x")
```

**Что получилось.** Типично precision@50 будет в районе 0.8-0.95, precision@100 — 0.6-0.8, при base rate ~0.053. Lift@100 порядка 12-15x: то есть детектор примерно на порядок эффективнее случайного отбора кейсов. Именно lift, а не абсолютный precision, продаёт систему бизнесу — "ваша команда находит фрод в N раз быстрее".

```python

print(top100[top100.label != "normal"]["label"].value_counts())
```

### Шаг 8: Ручной разбор топ-аномалий

**Зачем.** Score без объяснения операционная команда не примет — им нужно "почему этот". Для каждого кейса в топе генерируем **человеко-читаемое обоснование**: какие признаки сильнее всего отклонены от медианы популяции (в единицах робастного z-score). Это и есть мост между ML и людьми.

```python
med = X.median()
mad = (X - med).abs().median() + eps  # робастная мера разброса

def explain(player_id, top_features=3):
    i = feat.index[feat["player_id"] == player_id][0]
    row = X.loc[i]
    z = (row - med) / (1.4826 * mad)  # робастный z через MAD
    top = z.abs().sort_values(ascending=False).head(top_features)
    parts = []
    for f in top.index:
        direction = "выше" if z[f] > 0 else "ниже"
        parts.append(f"{f} сильно {direction} нормы (z={z[f]:.1f}, факт={row[f]:.2f})")
    return "; ".join(parts)

for pid in res.head(8)["player_id"]:
    lbl = res.loc[res.player_id == pid, "label"].values[0]
    print(f"[{pid}] (истина: {lbl})")
    print("   ", explain(pid), "\n")
```

**Что получилось.** По каждому игроку — топ-3 признака, которые загнали его в аномалии, с направлением и фактическим значением. Например: "bonus_to_deposit сильно выше нормы (z=8.3); wagering_ratio сильно ниже нормы (z=-3.1); account_age_days сильно ниже нормы (z=-2.4)" — это буквально портрет bonus abuser, который аналитик прочитает за 5 секунд. Используем MAD/робастный z, а не обычный, потому что среднее и std сами поедены выбросами.

### Шаг 9: Выбор порога и хэндофф

**Зачем.** Команда не может проверять бесконечно. Порог = **операционный бюджет**, а не магическое число из статистики. Считаем по ёмкости проверки и по тому, где precision@k начинает обваливаться (кривая precision против k). Точка, после которой precision резко падает к base rate, — естественная граница "дальше копать не выгодно".

```python
import matplotlib.pyplot as plt

ks = list(range(10, 400, 10))
precs = [precision_at_k(res, k) for k in ks]
plt.figure(figsize=(8, 4))
plt.plot(ks, precs, marker="o")
plt.axhline(base_rate, ls="--", color="gray", label=f"base rate {base_rate:.3f}")
plt.xlabel("k (число кейсов на проверку)")
plt.ylabel("precision@k")
plt.legend(); plt.title("Где обрезать топ"); plt.tight_layout()
plt.savefig("precision_at_k.png", dpi=120)
```

**Что получилось.** График, по которому виден "локоть". Решение о пороге формулируется так: "Риск-команда проверяет 60 кейсов в день. На k=60 precision@k ~0.85 и lift ~16x. Берём фиксированный дневной топ-60 по ensemble_score; всё ниже — в фоновую очередь."

**Хэндофф в операционную команду** — это процесс, не файл. Опиши его явно:

- **Формат**: ежедневный `suspicious_cases.csv` (priority, player_id, score, reason) + дашборд.
- **SLA**: топ-60 разбираются в течение дня; решение по каждому — fraud / not_fraud / unclear.
- **Действия**: подтверждённый фрод → заморозка вывода, проверка KYC, отзыв бонуса; not_fraud → whitelist; unclear → доп. запрос.
- **Whitelist VIP**: легальные крупные игроки исключаются из топа правилом, чтобы не жечь бюджет проверки на них повторно.

### Шаг 10: Обратная связь и переобучение

**Зачем.** Каждый разобранный кейс даёт метку fraud/not_fraud. Это бесценно: со временем накапливается размеченный сет, и можно перейти от чистого unsupervised к **полу-supervised** (PU-learning, или supervised поверх score как фичи). Без петли обратной связи детектор деградирует — мошенники адаптируются (concept drift), а ты не замечаешь.

Контур, который надо заложить:

- **Сбор вердиктов**: решения команды пишутся обратно в `feedback` таблицу (player_id, verdict, reviewer, ts).
- **Мониторинг**: трекать precision@k по факту вердиктов неделя к неделе. Падение precision = drift, пора переобучать/добавлять признаки.
- **Переобучение по расписанию**: Iso Forest/LOF переобучаются на свежем окне (например, последние 90 дней), чтобы "норма" следовала за сезонностью.
- **Эволюция модели**: когда накоплено хотя бы несколько сотен подтверждённых фродов — обучить supervised-классификатор, где anomaly score становится одним из признаков. Unsupervised остаётся "детектором нового", supervised — "детектором известного".


!!! question "Проверь себя"

    1. Почему recall в проде честно не измерить, а precision@k — можно?
    2. Что произойдёт с детектором через полгода без переобучения?
    3. Как подтверждённые вердикты команды улучшают систему, если мы начинали без меток?

??? success "Ответы"

    1. Recall требует знать **всех** фродов в популяции, включая непойманных — их по определению нет в данных. Precision@k требует проверить только отобранный топ, что команда и так делает; знаменатель k известен, числитель даёт ручной разбор.
    2. Concept drift: мошенники меняют схемы, "норма" дрейфует (новые продукты, сезон). Распределения уезжают, старая модель ловит вчерашний фрод, precision@k падает. Без мониторинга это замечают поздно.
    3. Накапливается размеченный датасет → переход к полу-supervised: anomaly score как признак в supervised-модели, плюс whitelist из not_fraud. Unsupervised продолжает ловить новое, supervised — уже известные паттерны точнее.

## Типичные ошибки

- **Утечка метки в признаки.** Оставили label (или прокси вроде "был ли возврат") в X — получили идеальные метрики и нерабочий прод. Собирай матрицу явно из поведенческих признаков.
- **Абсолютные суммы вместо отношений.** Скоринг по `deposit_sum` ловит богатых, а не мошенников. Всегда относительные/долевые признаки.
- **Забыли StandardScaler перед LOF.** LOF тогда смотрит только на признак с наибольшим масштабом. Iso Forest простит, LOF — нет.
- **Сложение сырых score двух моделей.** Разные шкалы → перекос. Усредняй ранги или нормированные score.
- **LOF с novelty=False для онлайна.** Он transductive, `predict` для новых точек нет. Для скоринга свежих транзакций — novelty=True и fit на чистой норме.
- **Контаминацию приняли за порог ранжирования.** contamination двигает бинарную метку, не score. Топ-k по score устойчив к её значению.
- **(senior) Train/serve skew в признаках.** Офлайн считаешь wagering_ratio в pandas, онлайн — в SQL с другой логикой NULL/деления. Расхождение убивает качество тихо. Один источник определения признака.
- **(senior) Гонка за precision@k вместо lift.** В сильно несбалансированных данных абсолютный precision вводит в заблуждение; lift над base rate честнее показывает ценность.
- **(senior) Нет петли обратной связи.** Без сбора вердиктов и мониторинга precision во времени система незаметно деградирует под адаптацию мошенников.
- **VIP в топе каждый день.** Легальные крупные игроки — стабильные аномалии. Без whitelist команда жжёт бюджет на одних и тех же ложных.

!!! tip "AI-копилот в этом воркшопе"

    Где нейросеть ускорит: генерация скелета feature-инженерии ("предложи 15 поведенческих признаков для bonus abuse"), быстрый разбор API sklearn (параметры LOF novelty), черновик explain-функции и текста хэндоффа для риск-команды, генерация вариантов синтетики. Где подведёт: copilot охотно оставит **утечку label** в матрице, если не указать явно; предложит складывать сырые score без нормировки; забудет про novelty=True для онлайна; сгенерирует "правдоподобные" precision@k числа, которых у тебя нет — всегда считай метрики сам на своих данных, не доверяй сгенерированным цифрам. И главное: AI не знает доменных схем фрода твоей платформы — карту "схема → признак" строишь ты.

## Развитие

- **Третья модель в ансамбль.** Добавь ECOD или COPOD из `pyod` — они параметр-свободны и хорошо ловят хвостовые аномалии; сравни стабильность топа.
- **Графовый мультиаккаунтинг.** Построй граф связей игроков по общим IP/device/платёжным реквизитам (networkx) и ищи плотные компоненты — это поймает сговоры, которые табличные признаки упускают.
- **SHAP для объяснений.** Замени ручной z-score explain на SHAP поверх supervised-модели, обученной на накопленных вердиктах, — обоснования станут точнее.
- **Онлайн-скоринг.** Заверни Iso Forest (novelty) в FastAPI-эндпойнт, считай признаки в duckdb по событию, верни score в реальном времени.
- **Drift-мониторинг.** Подключи отслеживание распределений признаков (PSI/KS-тест неделя к неделе) и алерт, когда норма уезжает.
