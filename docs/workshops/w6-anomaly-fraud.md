## Что ты построишь

К концу воркшопа у тебя будет рабочий антифрод-пайплайн без размеченных меток фрода. На входе — таблица транзакций гемблинг-платформы (депозиты, ставки, выводы, начисления бонусов). На выходе — **ранжированный список топ-N подозрительных кейсов** с anomaly score, человеко-читаемым обоснованием по каждому ("почему этот игрок подозрителен"), оценкой **precision@k** на injected-аномалиях и описанием хэндоффа в операционную команду.

Это не "обучили классификатор на размеченных фродах" — таких меток в реальности нет или их единицы. Это **unsupervised anomaly detection**: ты ищешь то, что отклоняется от нормы, ранжируешь по степени отклонения и отдаёшь людям на проверку только верхушку, потому что ручной разбор стоит дорого. Артефакт — это `suspicious_cases.csv` плюс короткая записка для риск-команды, которую не стыдно показать.

Почему это важно именно в гемблинге: фрод тут не "украли карту", а тонкие схемы — bonus abuse (выжать бонус и вывести без отыгрыша), мультиаккаунтинг (один человек = десять "игроков" под один реферальный или welcome-бонус), сговорные паттерны ставок, отмывание через минимальный риск. Все они проявляются как **статистические аномалии в поведении**, а не как явный сигнал.

!!! info "Как устроен этот воркшоп"

    Это не лекция с готовым кодом, а задачник. Каждый содержательный шаг построен по схеме **Задача → Критерий → Решение**:

    - **Задача** — что именно сделать руками, с явными именами выходных переменных (их проверяет критерий — если назовёшь иначе, поймаешь `NameError`).
    - **Критерий шага** — кусок кода с `assert`, который ты запускаешь после своего решения. Зелёный прогон без ошибки = шаг сдан. Это локальный авто-грейдер. Там, где проверка визуальная или проектная (график, текст хэндоффа), вместо `assert` дан self-check чек-лист.
    - **Решение** спрятано под спойлер `Решение`. Открывай только после своей попытки — чтобы сверить подход, а не списать.
    - **Стат-задачи** проверяются прямо на странице: посчитай число, впиши в поле, нажми «Проверить».

    SETUP-код (генератор данных, окружение) оставлен видимым — это не задача, его копируешь как есть. Все `assert` рассчитаны на сгенерированные данные с `seed=42`. Не меняй seed, иначе числа поплывут.

## Бизнес-кейс

Ты — data scientist в команде iGaming-продукта. К тебе приходит антифрод-лид (руководитель риск-команды): за последние два месяца платёжный отдел заметил рост подозрительных выводов, но ручная проверка не справляется — аналитики разбирают кейсы вслепую и упираются в ёмкость. Размеченного списка "это фрод" почти нет: подтверждённых случаев единицы, остальное — догадки. Нужен инструмент, который из всей массы игроков каждый день выдаёт верхушку самых подозрительных, чтобы команда тратила свои часы на тех, у кого фрод реально вероятен, а не на случайную выборку.

!!! example "Ситуация"

    - Риск-команда физически проверяет около 60 кейсов в день. Сейчас отбор почти случайный: при base rate аномалий ~5% в топ попадает 3-4 реальных фрода из 60 — остальное время аналитиков сгорает впустую.
    - По грубой оценке антифрод-лида, утечка через bonus abuse и мультиаккаунтинг — ориентировочно 1.5-2 млн ₽ в месяц недополученной маржи (невозвратные бонусы, выводы без отыгрыша, повторные welcome-бонусы на одно лицо).
    - От твоего ответа зависит решение: нанимать ли ещё двух проверяющих в риск-команду (ФОТ ~250 тыс ₽/мес) или поднять эффективность текущей через приоритизацию. Если детектор даёт lift хотя бы 10x, найм откладывается.
    - Ограничение: меток фрода нет, делаешь на unsupervised; на пилот — спринт, данные только поведенческие агрегаты по игрокам (депозиты, ставки, выводы, бонусы, IP/девайсы).

## Данные

В реальном проекте ты берёшь транзакции из прода. Здесь мы генерируем **синтетический, но реалистичный** журнал транзакций с **впрыснутыми аномалиями известной доли** — это ключевой приём. Раз мы сами знаем, кто аномален, мы сможем честно посчитать precision@k и не обманывать себя. В проде этой роскоши нет, поэтому синтетика — твой полигон для калибровки.

Генератор моделирует три популяции: массу нормальных игроков и три типа злоумышленников. Это SETUP-код, а не задача — скопируй и запусти как есть. Seed зафиксирован.

```python
import numpy as np
import pandas as pd

rng = np.random.default_rng(42)

N_NORMAL = 4000
N_BONUS_ABUSE = 90
N_MULTI = 80
N_BET_ANOMALY = 50
CONTAMINATION_TRUE = (N_BONUS_ABUSE + N_MULTI + N_BET_ANOMALY) / (
    N_NORMAL + N_BONUS_ABUSE + N_MULTI + N_BET_ANOMALY
)

def make_normal(n, start_id):
    pid = np.arange(start_id, start_id + n)
    deposits_cnt = rng.poisson(8, n) + 1
    deposit_sum = np.round(rng.gamma(3, 40, n) * deposits_cnt, 2)
    bets_cnt = rng.poisson(120, n) + 5
    bet_sum = np.round(deposit_sum * rng.uniform(3, 12, n), 2)
    withdraw_cnt = rng.poisson(2, n)
    withdraw_sum = np.round(deposit_sum * rng.uniform(0.1, 0.9, n) * (withdraw_cnt > 0), 2)
    bonus_sum = np.round(deposit_sum * rng.uniform(0, 0.3, n), 2)
    account_age_days = rng.integers(30, 900, n)
    distinct_ip = rng.poisson(1.5, n) + 1
    distinct_device = np.minimum(distinct_ip, rng.poisson(1.2, n) + 1)
    avg_bet = np.round(bet_sum / bets_cnt, 2)
    max_bet_share = rng.uniform(0.01, 0.15, n)
    return pd.DataFrame(dict(
        player_id=pid, deposits_cnt=deposits_cnt, deposit_sum=deposit_sum,
        bets_cnt=bets_cnt, bet_sum=bet_sum, withdraw_cnt=withdraw_cnt,
        withdraw_sum=withdraw_sum, bonus_sum=bonus_sum,
        account_age_days=account_age_days, distinct_ip=distinct_ip,
        distinct_device=distinct_device, avg_bet=avg_bet,
        max_bet_share=max_bet_share, label="normal"))

def make_bonus_abuse(n, start_id):
    df = make_normal(n, start_id)
    df["label"] = "bonus_abuse"
    df["bonus_sum"] = np.round(df["deposit_sum"] * rng.uniform(0.8, 2.0, n), 2)
    df["bet_sum"] = np.round(df["deposit_sum"] * rng.uniform(0.5, 1.2, n), 2)  # почти не отыгрывают
    df["withdraw_cnt"] = rng.integers(1, 3, n)
    df["withdraw_sum"] = np.round((df["deposit_sum"] + df["bonus_sum"]) * rng.uniform(0.7, 1.0, n), 2)
    df["account_age_days"] = rng.integers(1, 14, n)  # свежие
    df["bets_cnt"] = rng.poisson(15, n) + 1
    df["avg_bet"] = np.round(df["bet_sum"] / df["bets_cnt"], 2)
    return df

def make_multiaccount(n, start_id):
    df = make_normal(n, start_id)
    df["label"] = "multiaccount"
    df["distinct_ip"] = rng.integers(4, 12, n)       # много игроков с одного IP
    df["distinct_device"] = rng.integers(3, 9, n)
    df["account_age_days"] = rng.integers(1, 20, n)
    df["deposit_sum"] = np.round(rng.gamma(2, 25, n), 2)  # мелкие однотипные депозиты
    df["bonus_sum"] = np.round(df["deposit_sum"] * rng.uniform(0.5, 1.0, n), 2)
    return df

def make_bet_anomaly(n, start_id):
    df = make_normal(n, start_id)
    df["label"] = "bet_anomaly"
    df["max_bet_share"] = rng.uniform(0.6, 0.98, n)   # ва-банк одной ставкой
    df["bets_cnt"] = rng.poisson(6, n) + 1
    df["bet_sum"] = np.round(df["deposit_sum"] * rng.uniform(0.9, 1.1, n), 2)
    df["avg_bet"] = np.round(df["bet_sum"] / df["bets_cnt"], 2)
    df["withdraw_sum"] = np.round(df["deposit_sum"] * rng.uniform(0.0, 1.8, n), 2)
    return df

parts = [
    make_normal(N_NORMAL, 0),
    make_bonus_abuse(N_BONUS_ABUSE, 100000),
    make_multiaccount(N_MULTI, 200000),
    make_bet_anomaly(N_BET_ANOMALY, 300000),
]
df = pd.concat(parts, ignore_index=True).sample(frac=1, random_state=42).reset_index(drop=True)
print(df.shape, "true contamination =", round(CONTAMINATION_TRUE, 4))
df.head()
```

Что получилось: **4220 строк** (по одной на игрока) и 14 колонок с агрегированными признаками поведения. Колонка `label` — это **ground truth**, который мы используем только для оценки и прячем от моделей. Истинная доля аномалий — **~5.21%** (220 злоумышленников из 4220).

!!! warning "label не должен попасть в модель"

    Самая частая ошибка новичка — оставить `label` в матрице признаков. Тогда модель "детектит" по утечке, метрики космос, в проде ноль. Мы будем явно собирать матрицу без него.

!!! tip "Чем заменить на реальные данные"

    Подойдёт публичный **Credit Card Fraud Detection** (Kaggle, ULB) — там 284k транзакций с сильным дисбалансом и колонкой Class, которую так же прячут при обучении и используют только для оценки. Для поведенческой гемблинг-специфики ближе **PaySim** (Kaggle) — синтетический мобильный money-transfer лог с типами CASH_OUT/TRANSFER и флагом isFraud. Логика воркшопа переносится один в один: спрятать метку, собрать поведенческие агрегаты, ранжировать по score.

## Ход работы

### Шаг 1: Разобрать паттерны фрода до кода

**Зачем.** Если не понимаешь, как выглядит злоупотребление в данных, не соберёшь признаки, которые его ловят. Детектор аномалий найдёт "что-то странное", но странное бывает и легальным (VIP с огромными ставками — не фрод). Признаки должны кодировать гипотезы о схемах.

**Задача.** До единой строки кода выпиши карту "схема → наблюдаемые признаки" для трёх схем, заложенных в генератор. Для каждой назови, какие из имеющихся колонок (`bonus_sum`, `deposit_sum`, `bet_sum`, `withdraw_sum`, `distinct_ip`, `distinct_device`, `max_bet_share`, `bets_cnt`, `account_age_days`) и в какую сторону должны отклоняться. Кода тут нет — это доменная заготовка, без неё шаг 2 будет угадыванием.

**Критерий шага (self-check).** В твоей карте по каждой схеме должно быть:

- [ ] **Bonus abuse** — высокое отношение бонуса к депозиту, низкий wagering (`bet_sum` мало относительно `deposit_sum + bonus_sum`), свежий аккаунт, `withdraw_sum` близок к `deposit_sum + bonus_sum`.
- [ ] **Мультиаккаунтинг** — много `distinct_ip` / `distinct_device` на игрока, мелкие однотипные депозиты, свежие аккаунты.
- [ ] **Аномальные ставки** — `max_bet_share` близко к 1 (ва-банк), низкий `bets_cnt`, оборот близок к депозиту.
- [ ] У каждого признака явно указано направление отклонения (выше/ниже нормы).

??? success "Решение"

    Карта "схема → сигнатура":

    - **Bonus abuse.** Игрок берёт бонус, минимально отыгрывает, быстро выводит. Сигнал: высокое `bonus_to_deposit`, низкий `wagering_ratio = bet_sum / (deposit_sum + bonus_sum)`, свежий `account_age_days`, `withdraw_sum` ≈ `deposit_sum + bonus_sum`.
    - **Мультиаккаунтинг.** Один человек регистрирует пачку аккаунтов под welcome-бонусы. Сигнал: высокий `distinct_ip` / `distinct_device` на игрока, мелкие однотипные `deposit_sum`, свежие аккаунты, кучность.
    - **Аномальные ставки.** Ва-банк одной ставкой (`max_bet_share` → 1), мало ставок (`bets_cnt`), цель — обналичить депозит с минимальным оборотом (отмывание). Сигнал: высокая `max_bet_share`, низкий `bets_cnt`.

    **Почему так.** Каждая строка карты на шаге 2 станет конкретным относительным признаком. Без карты ты соберёшь случайный набор колонок, и детектор будет ловить шум, а не схемы.

Проверь понимание:

```text
[x] В реальности размеченного label почти нет — фрод выявляют пост-фактум единицами, а в синтетике label это оценочный ground truth, утечка которого даёт фиктивное качество
[ ] Isolation Forest точнее любого классификатора
> Антифрод по определению работает там, где меток нет. label у нас существует только для честной оценки precision@k и обязан быть спрятан от моделей.

[x] Статистически да, это outlier, но легальный — поэтому anomaly score это приоритет на ручной разбор, а не приговор; VIP отсеивается ревью и whitelist
```

    ```python
    feat["player_id"] = df["player_id"]

    # поведенческие отношения, а не абсолютные суммы
    feat["wagering_ratio"] = df["bet_sum"] / (df["deposit_sum"] + df["bonus_sum"] + eps)
    feat["bonus_to_deposit"] = df["bonus_sum"] / (df["deposit_sum"] + eps)
    feat["withdraw_to_inflow"] = df["withdraw_sum"] / (df["deposit_sum"] + df["bonus_sum"] + eps)
    feat["ip_per_account"] = df["distinct_ip"]
    feat["device_per_ip"] = df["distinct_device"] / (df["distinct_ip"] + eps)
    feat["max_bet_share"] = df["max_bet_share"]
    feat["bets_per_deposit"] = df["bets_cnt"] / (df["deposits_cnt"] + eps)
    feat["avg_bet_to_deposit"] = df["avg_bet"] / (df["deposit_sum"] / (df["deposits_cnt"] + eps) + eps)
    feat["account_age_days"] = df["account_age_days"]
    feat["deposit_sum_log"] = np.log1p(df["deposit_sum"])

    FEATURES = [c for c in feat.columns if c != "player_id"]
    X = feat[FEATURES].replace([np.inf, -np.inf], np.nan).fillna(0.0)
    print(X.describe().T[["mean", "std", "min", "max"]])
    ```

    **Почему так.** Матрица из 10 поведенческих признаков. `label` тут нет вообще — мы его не трогали. Каждый признак — закодированная гипотеза с шага 1: `wagering_ratio` ловит bonus abuse, `ip_per_account` — мультиаккаунтинг, `max_bet_share` — аномальные ставки. Относительные признаки делают "много денег" и "странное поведение" разными вещами.

!!! tip "Хэндофф признаков в SQL"

    На проде эти агрегаты считаются не в pandas, а в duckdb/витрине. Пример формы запроса (псевдо): `SELECT player_id, SUM(bet)/NULLIF(SUM(deposit)+SUM(bonus),0) AS wagering_ratio, COUNT(DISTINCT ip) AS ip_per_account FROM tx GROUP BY player_id`. Логика идентична — важно, чтобы офлайн-обучение и онлайн-скоринг считали признаки **одинаково**, иначе train/serve skew.

### Шаг 4: Isolation Forest

**Зачем.** Isolation Forest — рабочая лошадь промышленного антифрода. Идея: аномалии "изолируются" случайными разрезами за меньшее число шагов, чем плотная норма. Он линейно масштабируется, не требует расстояний, хорошо тянет много признаков. `contamination` — ожидаемая доля аномалий; задаём близко к нашей оценке, но помним, что в проде это гиперпараметр калибровки, а не известная константа.

**Задача.** Обучи `IsolationForest` (`n_estimators=300`, `contamination=0.05`, `random_state=42`) на `Xs`. Получи anomaly score в `iso_score` так, чтобы **большее значение = аномальнее** (sklearn `score_samples` отдаёт наоборот — инвертируй знак).

**Критерий шага:**

```python
import numpy as np
assert iso_score.shape == (4220,) and np.isfinite(iso_score).all(), "score на каждого игрока, без NaN/inf"
anom = (df["label"] != "normal").values
assert iso_score[anom].mean() > iso_score[~anom].mean(), "у аномалий средний score должен быть выше нормы"
print("OK: Isolation Forest обучен, аномалии в среднем 'аномальнее'")
```

??? success "Решение"

    ```python
    from sklearn.ensemble import IsolationForest

    iso = IsolationForest(
        n_estimators=300,
        contamination=0.05,
        max_samples="auto",
        random_state=42,
        n_jobs=-1,
    )
    iso.fit(Xs)
    # score_samples: выше = нормальнее. Инвертируем, чтобы "выше = аномальнее"
    iso_score = -iso.score_samples(Xs)
    ```

    **Почему так.** Вектор `iso_score`, где большие значения — кандидаты в аномалии. Знак инвертируем намеренно: sklearn отдаёт "чем выше, тем нормальнее", а нам для ранжирования удобнее обратное. На данных генератора средний score у аномалий заметно выше, чем у нормы (~0.62 против ~0.42) — модель уже улавливает сигнал.

!!! note "Про contamination"

    contamination влияет на порог метки `predict` (-1/1), но **не на ранжирование** `score_samples`. Поскольку мы работаем precision@k по score, точное значение contamination для топа не критично — оно важно, когда тебе нужна бинарная метка по умолчанию. Это частый источник путаницы.

### Шаг 5: Local Outlier Factor

**Зачем.** Iso Forest ловит глобальные аномалии. LOF ловит **локальные** — точку, которая нормальна глобально, но выбивается из своего соседства (мультиаккаунты образуют плотный странный кластер; кто-то на краю плотной группы). Две модели с разной индуктивной предпосылкой дают взаимодополняющий взгляд.

**Задача.** Обучи `LocalOutlierFactor` (`n_neighbors=35`, `contamination=0.05`) на `Xs` через `fit_predict`. Сохрани score в `lof_score` так, чтобы **большее значение = аномальнее** (используй `negative_outlier_factor_` с инверсией знака).

**Критерий шага:**

```python
import numpy as np
assert lof_score.shape == (4220,) and np.isfinite(lof_score).all(), "score на каждого игрока"
anom = (df["label"] != "normal").values
assert lof_score[anom].mean() > lof_score[~anom].mean(), "LOF тоже должен давать аномалиям выше score"
print("OK: LOF обучен")
```

??? success "Решение"

    ```python
    from sklearn.neighbors import LocalOutlierFactor

    lof = LocalOutlierFactor(n_neighbors=35, contamination=0.05, n_jobs=-1)
    lof.fit_predict(Xs)
    lof_score = -lof.negative_outlier_factor_  # выше = аномальнее
    ```

    **Почему так.** Вектор `lof_score`. `n_neighbors=35` — баланс: слишком мало шумит, слишком много размывает локальность. Это гиперпараметр, который стоит варьировать (20-50) и смотреть стабильность топа.

!!! warning "LOF для онлайн-скоринга"

    `LocalOutlierFactor` по умолчанию transductive — скорит ровно тот набор, на котором обучен, и не имеет `predict` для новых точек. Для онлайна нужен `LocalOutlierFactor(novelty=True)` и отдельный `fit` на "чистой" норме. В этом воркшопе мы скорим весь батч офлайн, поэтому `novelty=False` корректен.

### Шаг 6: Ранжирование и ансамбль

**Критерий шага:**

```python

print("OK: ансамбль собран, набор отранжирован")
```

??? success "Решение"

Проверь понимание:

### Шаг 7: Оценка precision@k

**Зачем.** Recall в антифроде честно посчитать нельзя — ты не знаешь всех фродов в проде. Зато можешь измерить **precision@k**: из топ-k, которые отдал на проверку, сколько реально аномальны. Это прямо отражает экономику: команда проверяет фиксированный бюджет кейсов в день, и нам важно, чтобы в этих k было максимум настоящих фродов. На синтетике у нас есть label, поэтому считаем честно и калибруем k.

**Задача.** Напиши функцию `precision_at_k(frame, k, label_col="label")`, возвращающую долю не-`normal` в топ-k. Посчитай её для `k ∈ {20, 50, 100, 150, 200}`, посчитай `base_rate` (доля аномалий в популяции) и `lift@100 = precision@100 / base_rate`.

Сначала прикинь экономику руками:

```text
TASK: precision@100 = 0.68, base rate = 0.052. Чему равен lift@100 = precision@100 / base_rate? Округли до 0.1.
ANSWER: 13.1
TOL: 0.5
PLACEHOLDER: 0.0
EXPLAIN: 0.68 / 0.052 = 13.08 ≈ 13.1x. Lift показывает, во сколько раз отбор лучше случайного: детектор находит реальный фрод примерно на порядок эффективнее. Именно lift, а не абсолютный precision, продаёт систему бизнесу.
---
TASK: На k=60 precision@k = 0.93. Сколько реальных фродов в этих 60 кейсах? Ответ - целое число.
ANSWER: 56
TOL: 1
PLACEHOLDER: целое число
EXPLAIN: 0.93 * 60 = 55.8 ≈ 56 настоящих кейсов из 60 проверок. Сравни с 3 при случайном отборе (стат-разминка) - это и есть скачок выхлопа команды без расширения штата.
```

**Критерий шага:**

```python

p100 = precision_at_k(res, 100)
base_rate = (res["label"] != "normal").mean()
assert p50 >= 0.85, "в топ-50 точность должна быть высокой"
assert 0.55 <= p100 <= 0.80, "в топ-100 точность в районе 0.6-0.8"
assert p100 / base_rate > 8, "lift@100 должен сильно превышать случайный отбор"
print(f"OK: precision@50={p50:.2f}, precision@100={p100:.2f}, lift@100={p100/base_rate:.1f}x")
```

??? success "Решение"

    ```python
    def precision_at_k(frame, k, label_col="label"):
        top = frame.head(k)
        hits = (top[label_col] != "normal").sum()
        return hits / k

    for k in [20, 50, 100, 150, 200]:
        p = precision_at_k(res, k)
        print(f"precision@{k:>3} = {p:.3f}  (попаданий {int(round(p*k))} из {k})")

    # базовый уровень: доля аномалий в популяции
    base_rate = (res["label"] != "normal").mean()
    print(f"\nbase rate (случайный отбор) = {base_rate:.3f}")
    print(f"lift@100 = {precision_at_k(res, 100) / base_rate:.1f}x")

    # вклад моделей по типам аномалий в топ-100
    top100 = res.head(100)
    print(top100[top100.label != "normal"]["label"].value_counts())
    ```

    **Почему так.** На наших данных precision@20 = 1.00, precision@50 ≈ 0.96, precision@100 ≈ 0.68 при base rate ~0.052. Lift@100 ≈ 13x — детектор примерно на порядок эффективнее случайного отбора. Именно lift, а не абсолютный precision, продаёт систему бизнесу ("ваша команда находит фрод в N раз быстрее"). В разбивке топ-100 по типам уверенно доминирует multiaccount (~59), затем bonus_abuse (~7); часть bet_anomaly с умеренным `max_bet_share` тонет — это сигнал, что нужен дополнительный признак под эту схему.

!!! note "precision@k падает с ростом k — это нормально"

    На топ-150 точность уже ~0.49, на топ-200 ~0.39: чем глубже копаешь, тем больше нормы подмешивается. Поэтому порог k выбирают не "побольше", а по бюджету проверки и точке, где precision сваливается к base rate (см. шаг 9).

### Шаг 8: Ручной разбор топ-аномалий

**Зачем.** Score без объяснения операционная команда не примет — им нужно "почему этот". Для каждого кейса в топе генерируем **человеко-читаемое обоснование**: какие признаки сильнее всего отклонены от медианы популяции (в единицах робастного z-score через MAD). Это мост между ML и людьми.

**Задача.** Посчитай медиану `med` и MAD `mad` по `X`. Напиши функцию `explain(player_id, top_features=3)`, которая по игроку считает робастный z (`(row - med) / (1.4826 * mad)`), берёт топ-`top_features` признаков по `abs(z)` и собирает строку с направлением отклонения и фактическим значением. Прогони по топ-8 кейсов.

**Критерий шага:**

```python
r = explain(res.iloc[0]["player_id"])
assert isinstance(r, str) and "z=" in r, "обоснование должно быть текстом с z-значениями"
print("OK: обоснование генерируется ->", r)
```

??? success "Решение"

    ```python
    med = X.median()
    mad = (X - med).abs().median() + eps  # робастная мера разброса

    def explain(player_id, top_features=3):
        i = feat.index[feat["player_id"] == player_id][0]
        row = X.loc[i]
        z = (row - med) / (1.4826 * mad)  # робастный z через MAD
        top = z.abs().sort_values(ascending=False).head(top_features)
        parts = []
        for f in top.index:
            direction = "выше" if z[f] > 0 else "ниже"
            parts.append(f"{f} сильно {direction} нормы (z={z[f]:.1f}, факт={row[f]:.2f})")
        return "; ".join(parts)

    for pid in res.head(8)["player_id"]:
        lbl = res.loc[res.player_id == pid, "label"].values[0]
        print(f"[{pid}] (истина: {lbl})")
        print("   ", explain(pid), "\n")
    ```

    **Почему так.** По каждому игроку — топ-3 признака, загнавшие его в аномалии, с направлением и фактическим значением. Аналитик читает портрет за 5 секунд: высокий `bonus_to_deposit` + низкий `wagering_ratio` + свежий аккаунт = bonus abuser. Берём MAD/робастный z, а не обычный, потому что среднее и std сами поедены выбросами.

!!! warning "Подвох робастного z на почти-константных признаках"

    Если у признака MAD близок к нулю (например `device_per_ip` у нормы почти всегда ~1), знаменатель `1.4826 * mad` схлопывается к `eps`, и z раздувается до десятков тысяч — на топ-кейсах этого воркшопа ты увидишь именно такие гигантские числа. Это не "очень сильная аномалия", а артефакт деления на почти-ноль. В проде ставь пол на MAD (`mad = np.maximum(mad, 0.5 * std)`) или клиппируй z, иначе объяснение вводит аналитика в заблуждение. Хороший повод не доверять числу вслепую.

!!! example "Готовый артефакт"

    ```python
    res["reason"] = res["player_id"].map(lambda p: explain(p))
    handoff = res.head(100)[["player_id", "ensemble_score", "iso_rank", "lof_rank", "reason"]]
    handoff.insert(1, "priority", range(1, len(handoff) + 1))
    handoff.to_csv("suspicious_cases.csv", index=False)
    print("Сохранено", len(handoff), "кейсов для риск-команды")
    ```

    Это и есть тот файл, который уходит в операционную команду: приоритет, score, обоснование.

## Типичные ошибки

- **Утечка метки в признаки.** Оставили label (или прокси вроде "был ли возврат") в X — получили идеальные метрики и нерабочий прод. Собирай матрицу явно из поведенческих признаков.
- **Абсолютные суммы вместо отношений.** Скоринг по `deposit_sum` ловит богатых, а не мошенников. Всегда относительные/долевые признаки.
- **Забыли StandardScaler перед LOF.** LOF тогда смотрит только на признак с наибольшим масштабом. Iso Forest простит, LOF — нет.
- **Сложение сырых score двух моделей.** Разные шкалы → перекос. Усредняй ранги или нормированные score.
- **LOF с novelty=False для онлайна.** Он transductive, `predict` для новых точек нет. Для скоринга свежих транзакций — novelty=True и fit на чистой норме.
- **Контаминацию приняли за порог ранжирования.** contamination двигает бинарную метку, не score. Топ-k по score устойчив к её значению.
- **MAD-z на почти-константном признаке.** Если MAD≈0, z раздувается на пустом месте. Ставь пол на MAD или клиппируй z, иначе объяснение врёт.
- **(senior) Train/serve skew в признаках.** Офлайн считаешь wagering_ratio в pandas, онлайн — в SQL с другой логикой NULL/деления. Расхождение убивает качество тихо. Один источник определения признака.
- **(senior) Гонка за precision@k вместо lift.** В сильно несбалансированных данных абсолютный precision вводит в заблуждение; lift над base rate честнее показывает ценность.
- **(senior) Нет петли обратной связи.** Без сбора вердиктов и мониторинга precision во времени система незаметно деградирует под адаптацию мошенников.
- **VIP в топе каждый день.** Легальные крупные игроки — стабильные аномалии. Без whitelist команда жжёт бюджет на одних и тех же ложных.

!!! tip "AI-копилот в этом воркшопе"

    Где нейросеть ускорит: генерация скелета feature-инженерии ("предложи 15 поведенческих признаков для bonus abuse"), быстрый разбор API sklearn (параметры LOF novelty), черновик explain-функции и текста хэндоффа для риск-команды, генерация вариантов синтетики.

    Где подведёт именно здесь: (1) copilot охотно оставит **утечку label** в матрице, если не указать явно; (2) предложит складывать сырые score без нормировки; (3) забудет про `novelty=True` для онлайна; (4) сгенерирует "правдоподобные" precision@k числа, которых у тебя нет — всегда считай метрики сам на своих данных, не доверяй сгенерированным цифрам; (5) не заметит MAD-схлопывание в explain. И главное: AI не знает доменных схем фрода твоей платформы — карту "схема → признак" строишь ты.
