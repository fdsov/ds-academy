# M20.13 · Агенты и tool use кратко

### Интуиция

Обычная LLM умеет ровно одно: получив текст, продолжить его текстом. Она не видит твою базу, не знает сегодняшнюю дату, не умеет нажимать кнопки. Всё, что у неё есть, — это веса, замороженные на момент обучения. Поэтому на вопрос "сколько FTD вчера на рынке UZ" честная модель может только сочинить правдоподобное число — у неё нет доступа к данным.

Агент ломает эту стену. Агент — это та же LLM, которой выдали инструменты (функции) и право самой решать, когда их дёрнуть. Житейская картинка: переводчик, который не знает ответа, но знает, кому позвонить. Спросили про FTD — агент не выдумывает, а вызывает функцию `run_sql`, получает реальное число и вставляет его в ответ. Текстовый генератор превращается в исполнителя задач, который опирается на внешний мир, а не на свою память.

Главная мысль: ум модели остаётся в её голове (рассуждение на естественном языке), но руки и глаза ей дают инструменты. LLM решает что сделать, твой код делает это и возвращает результат, LLM решает что дальше. Цикл повторяется, пока цель не достигнута.

### Формально

Tool use (function calling) — это протокол, в котором ты заранее описываешь модели доступные функции: имя, назначение, параметры в виде JSON-схемы (типы, обязательность, описания). При генерации модель может вместо обычного текста вернуть структурированный объект — запрос на вызов функции с конкретными аргументами. Твой исполняющий код (orchestrator) исполняет функцию, возвращает результат модели как новое сообщение, и модель продолжает генерацию уже с этим результатом в контексте.

Агент формально — это управляющий цикл вокруг модели:

$$
s_{t+1} = \text{Model}(s_t),\quad a_t \in \{\text{call tool } k,\ \text{finish}\}
$$

где $s_t$ — текущее состояние (история сообщений, результаты прошлых вызовов), а на каждом шаге модель выбирает действие $a_t$: вызвать инструмент или завершить. Популярный паттерн — ReAct (Reasoning + Acting): модель чередует шаг рассуждения ("чтобы ответить, нужны данные за вчера") и шаг действия (вызов `run_sql`). Цикл: думает → выбирает инструмент → получает результат → думает дальше — до достижения цели или лимита шагов.

К 2026 подключение инструментов стандартизировано протоколом MCP (Model Context Protocol): он унифицирует, как модель видит инструменты и источники данных, чтобы один и тот же сервер инструментов работал с разными клиентами и моделями. Рядом вырос A2A (Agent2Agent) — слой для общения агентов между собой. Разделение простое: MCP связывает агента с инструментами, A2A — агентов друг с другом в multi-agent системах.

Ключевые ограничения, которые надо держать в голове:

- стоимость и латентность растут линейно по числу шагов: каждый шаг — отдельный вызов LLM с растущим контекстом;
- ошибки накапливаются: если на каждом из $n$ шагов вероятность верного действия $p$, то вероятность пройти всю цепочку без сбоя $\approx p^n$;
- инструменты с побочными эффектами (запись в БД, отправка денег, рассылка) опасны: галлюцинированный аргумент превращается в реальное действие.

#### Разбор на пальцах: цена шага и почему длинные цепочки рассыпаются

Посчитаем агента, который отвечает на вопрос продуктового аналитика про депозиты, делая в среднем 5 шагов (4 вызова инструментов + финальный ответ). Контекст растёт от шага к шагу, потому что в него дописываются результаты прошлых вызовов.

Грубая прикидка по токенам на запрос (вход растёт, выход почти постоянный):

- шаг 1: вход 2 000, выход 200
- шаг 2: вход 3 000, выход 200
- шаг 3: вход 4 500, выход 200
- шаг 4: вход 6 500, выход 200
- шаг 5: вход 9 000, выход 300

Суммарно вход $\approx 25\,000$ токенов, выход $\approx 1\,100$. При условной цене $\$3$ за 1M входных и $\$15$ за 1M выходных токенов:

$$
\text{вход} = \frac{25\,000}{10^6}\times\$3 = \$0.075,\quad \text{выход} = \frac{1\,100}{10^6}\times\$15 = \$0.0165
$$

Итого $\approx \$0.092$ за один диалог. Кажется копейками, но если такой агент обслуживает 50 000 запросов в месяц: $50\,000 \times \$0.092 = \$4\,600$ в месяц. Сравни с одиночным вызовом без агентного цикла (1 шаг, ~2 200 токенов, $\approx \$0.01$) — агент дороже примерно в 9 раз. Поэтому "обернуть всё в агента" — это осознанное денежное решение, а не дефолт.

Теперь надёжность. Пусть каждый шаг успешен с вероятностью $p = 0.95$ (модель верно выбрала инструмент и аргументы). Для цепочки из 5 шагов:

$$
P(\text{успех}) = 0.95^5 \approx 0.774
$$

То есть примерно каждый четвёртый диалог где-то сломается, хотя отдельный шаг выглядит почти безупречным (95\%). Растянем до 10 шагов — $0.95^{10}\approx 0.599$, уже почти монетка. Отсюда практический вывод: сокращай число шагов, давай меньше инструментов, добавляй проверки и ретраи на критичных вызовах.

!!! warning "Типичная ошибка"

    Новичок думает, что function calling — это когда модель сама исполняет функцию. Нет: модель только возвращает структурированный запрос "вызови `run_sql` с такими аргументами". Исполняет функцию твой код, и именно он отвечает за безопасность. Если слепо исполнять любой запрос модели на инструмент с побочными эффектами (`delete_user`, `send_payout`), галлюцинированный аргумент станет реальным удалением или выплатой. Инструменты, меняющие боевые системы, всегда заворачивают в guardrails: валидация аргументов, белые списки, подтверждение, dry-run.

!!! question "Проверь себя"

    1. Почему обычная LLM не может надёжно ответить "сколько FTD было вчера", а агент — может?
    2. Что конкретно возвращает модель при function calling — и кто исполняет функцию?
    3. Если каждый шаг успешен с вероятностью 0.95, какова примерно вероятность пройти цепочку из 5 шагов без сбоя? Какой практический вывод?
    4. В чём разница назначения протоколов MCP и A2A?

??? success "Ответы"

    1. У обычной LLM нет доступа к актуальным данным — веса заморожены на момент обучения, и вчерашних цифр там нет; она может только сочинить число. Агент вызывает инструмент (`run_sql`), получает реальное значение из базы и подставляет его.
    2. Модель возвращает не текст, а структурированный запрос на вызов функции — имя и аргументы по JSON-схеме. Исполняет функцию твой код (orchestrator), а затем возвращает результат модели.
    3. $0.95^5 \approx 0.77$, то есть примерно каждый четвёртый диалог сломается. Вывод: минимизировать число шагов и инструментов, добавлять валидацию и ретраи на критичных вызовах.
    4. MCP связывает агента с инструментами и источниками данных (унифицированное подключение), A2A — агентов друг с другом в multi-agent системах.

!!! example "Мини-задание"

    Спроектируй на бумаге агента, который отвечает на вопрос "какой средний депозит у игроков из UZ за последние 7 дней". За 3-5 минут: (а) перечисли инструменты, которые ему нужны, и пометь, какие из них read-only, а какие с побочными эффектами; (б) прикинь число шагов и при $p=0.9$ за шаг оцени вероятность успеха цепочки; (в) предложи одно изменение, которое сократит число шагов.
