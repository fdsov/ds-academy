Семантический поиск и эмбеддинг-классификатор на эмбеддингах.

```python
import numpy as np
from openai import OpenAI
from sklearn.linear_model import LogisticRegression

client = OpenAI()

def embed(texts: list[str]) -> np.ndarray:
    resp = client.embeddings.create(model="text-embedding-3-large", input=texts)
    return np.array([d.embedding for d in resp.data])

# Эмбеддинг-классификатор обращений: дёшево и без fine-tuning
tickets = ["не приходит вывод уже сутки", "как отыграть бонус", "не открывается сайт",
           "куда ввести промокод", "списали депозит дважды"]
labels  = ["withdrawal", "bonus", "tech", "bonus", "payment"]

X = embed(tickets)
clf = LogisticRegression(max_iter=1000).fit(X, labels)

new = embed(["деньги списались, а на счёт не зачислились"])
print(clf.predict(new))  # ожидаем payment

# Семантический поиск по косинусной близости
def cosine_topk(query: str, corpus: list[str], k: int = 3):
    q = embed([query])[0]
    M = embed(corpus)
    sims = M @ q / (np.linalg.norm(M, axis=1) * np.linalg.norm(q))
    idx = np.argsort(-sims)[:k]
    return [(corpus[i], float(sims[i])) for i in idx]
```

Минимальный RAG над внутренней документацией с векторной БД (pgvector через Qdrant-альтернативу показан на Qdrant для краткости).

```python
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct
from openai import OpenAI

client = OpenAI()
qd = QdrantClient(":memory:")  # в проде: host/port или Qdrant Cloud
COLL, DIM = "kb", 3072

qd.recreate_collection(COLL, vectors_config=VectorParams(size=DIM, distance=Distance.COSINE))

def embed_one(text: str) -> list[float]:
    return client.embeddings.create(model="text-embedding-3-large", input=[text]).data[0].embedding

# Индексация базы знаний (chunks — заранее нарезанные куски регламентов)
chunks = [
    "Вывод средств обрабатывается до 24 часов после прохождения верификации KYC.",
    "Бонус за депозит требует отыгрыша x35 в течение 7 дней.",
    "Для рынка UZ доступны локальные платёжные методы и фикса-блок комиссии.",
]
qd.upsert(COLL, points=[
    PointStruct(id=i, vector=embed_one(c), payload={"text": c}) for i, c in enumerate(chunks)
])

def rag_answer(question: str, k: int = 3) -> str:
    hits = qd.search(COLL, query_vector=embed_one(question), limit=k)
    context = "\n".join(f"[{h.id}] {h.payload['text']}" for h in hits)
    prompt = (
        "Ответь на вопрос ТОЛЬКО на основе контекста. "
        "Если ответа в контексте нет — скажи 'нет данных'. Указывай номер источника.\n\n"
        f"Контекст:\n{context}\n\nВопрос: {question}"
    )
    resp = client.chat.completions.create(
        model="gpt-5-mini", temperature=0,
        messages=[{"role": "user", "content": prompt}],
    )
    return resp.choices[0].message.content

print(rag_answer("Сколько времени идёт вывод?"))
```

Структурированное извлечение из свободного текста (structured output, гарантированный JSON).

```python
from pydantic import BaseModel
from openai import OpenAI

client = OpenAI()

class TicketInfo(BaseModel):
    category: str        # withdrawal | bonus | tech | payment | kyc
    sentiment: str       # negative | neutral | positive
    amount_mentioned: float | None

resp = client.beta.chat.completions.parse(
    model="gpt-5-mini", temperature=0,
    messages=[{"role": "user",
               "content": "Списали 50 USDT дважды при пополнении, верните деньги!"}],
    response_format=TicketInfo,
)
print(resp.choices[0].message.parsed)  # валидный объект, не строка
```

В 2026 OpenAI продвигает новый Responses API (`client.responses.parse(...)`) как основной интерфейс; показанный `beta.chat.completions.parse` — переходный вариант, проверяй актуальную сигнатуру в доке провайдера.

!!! tip "AI-копилот: где нейросети помогают"

    LLM-ассистенты (Claude в IDE, Cursor, Copilot) сильно ускоряют именно эту тему: они хорошо генерируют код пайплайнов эмбеддингов/RAG, объясняют параметры API, накидывают eval-наборы и помогают подбирать chunking-стратегию.

    Пример рабочего промпта:

    "Вот моя RAG-функция [код]. Retrieval часто промахивается на коротких вопросах про KYC. Предложи 3 улучшения retrieval (chunking, reranking, гибридный поиск), покажи минимальный diff под Qdrant и метрику recall@5, которой я померю эффект до/после."

    Где врут и что перепроверять: модель уверенно выдаёт устаревшие имена моделей и сигнатуры API (особенно embeddings/structured outputs меняются быстро) — всегда сверяй с актуальной докладной провайдера. Она выдумывает несуществующие параметры и библиотечные функции — проверяй автодополнением и запуском. Она занижает важность evals и data quality — это твоя зона ответственности, не её. И она почти никогда сама не предложит сначала попробовать эмбеддинги вместо fine-tuning, если ты не спросишь, — архитектурное решение остаётся за тобой.

!!! note "Стек 2026"

    Актуально: pgvector и Qdrant как дефолтные векторные хранилища; structured outputs / tool calling на уровне API вместо вымаливания JSON в промпте; reasoning-модели с test-time compute для сложных рассуждений; LoRA/DPO как стандарт дообучения (не full fine-tune, не классический RLHF); MCP как протокол подключения инструментов; prompt caching как обязательная статья экономии; eval-first разработка с RAGAS/Langfuse/LangSmith; reranking как почти обязательная ступень серьёзного RAG.

    Устарело относительно курсов 2022-2023: word2vec/GloVe как основной способ эмбеддингов (заменены на контекстные модели); ручной парсинг JSON из текста регулярками; вера, что fine-tuning — главный путь кастомизации (теперь сначала RAG); представление, что большое контекстное окно отменяет RAG (нет — дорого, "lost in the middle"); классический RLHF как единственный способ выравнивания.

!!! warning "Ловушки"

    - Fine-tuning вместо RAG для фактов. Дообучили модель на регламентах, через неделю регламент изменился — модель врёт. Свежие знания = RAG, не дообучение.
    - Игнор retrieval-качества. Вся энергия в промпт генерации, при том что top-k подаёт нерелевантные куски. Меряй retrieval отдельно.
    - Нет evals. Меняешь промпт "на глаз", улучшил одно — сломал другое, и не заметил. Без eval-набора ты не инженер.
    - Высокая температура на фактах. Креативная температура на классификации/извлечении даёт нестабильность и галлюцинации. Для аналитики держи около нуля.
    - Доверие LLM-as-a-judge без калибровки. Судья предвзят (любит длинные ответы, свой стиль). Сверь с человеческой разметкой на выборке.
    - Сеньорская ловушка: токены в русском. Оценил длину контекста "по словам" по-английски, а на русском токенов в полтора-два раза больше — упёрся в лимит окна и в бюджет неожиданно.
    - Сеньорская ловушка: chunking по символам вслепую. Режешь документ каждые N символов, рвёшь таблицы и предложения пополам — retrieval деградирует. Режь по смысловым границам с перекрытием.
    - Агент с боевыми write-инструментами без guardrails. LLM ошибётся на шаге и выполнит запись в прод. Write-операции — только по явной валидации, никогда не автономно на критичном.
