# M20.7 · Prompt engineering для DS-задач

### Интуиция

Промпт — это инструкция модели. Качество ответа на 80% определяется тем, насколько чётко ты сформулировал задачу, дал контекст и примеры. Хороший промпт похож на хорошее ТЗ джуниору: роль, задача, ограничения, формат ответа, примеры.

### Практические приёмы

- Роль и контекст: "Ты аналитик антифрода. Вот описание транзакции...". Задаёт регистр и фокус.
- Few-shot (несколько примеров): покажи 2-5 примеров "вход — желаемый выход". Резко повышает стабильность на классификации и извлечении.
- Chain-of-thought: для задач с рассуждением попроси "думай пошагово". Для reasoning-моделей 2026 это уже встроено, явно просить не нужно.
- Structured output: требуй строгий JSON по схеме. В 2026 это делается не молитвами в промпте, а параметром API (structured outputs / JSON schema / tool calling), который гарантирует валидный JSON на уровне декодирования.
- Декомпозиция: сложную задачу разбей на цепочку простых вызовов, а не один гигантский промпт.

Для DS особенно ценно: извлечение структурированных полей из свободного текста (из жалобы вытащить категорию, тональность, упомянутую сумму), нормализация грязных данных, генерация SQL по описанию, разметка обучающих данных (LLM как дешёвый разметчик с последующей валидацией человеком).

??? success "Ответы"

    1. Чтобы показать формат "вход — желаемый выход"; резко повышает стабильность на классификации и извлечении структурированных полей.
    2. Через параметр API (structured outputs / JSON schema / tool calling), который гарантирует валидный JSON на уровне декодирования, а не за счёт уговоров в тексте.
    3. Chain-of-thought у них встроен — они и так рассуждают дольше перед ответом.
