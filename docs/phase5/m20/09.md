# M20.9 · RAG: retrieval-augmented generation

### Интуиция

У LLM две проблемы: она не знает твоих внутренних данных (регламентов казино, актуальных цифр) и она выдумывает (галлюцинирует), когда не знает. RAG решает обе разом простой идеей: перед тем как просить модель ответить, сначала найди релевантные куски из твоей базы знаний и положи их в промпт. Модель отвечает не из памяти, а опираясь на поданные факты. Это как разница между "ответь по памяти" и "ответь, открыв учебник на нужной странице".

### Формально и архитектурно

RAG-пайплайн состоит из двух фаз.

Индексация (офлайн, один раз): документы режутся на куски (chunking), каждый кусок превращается в эмбеддинг, всё складывается в векторную БД.

Запрос (онлайн, на каждый вопрос):

1. Эмбеддинг вопроса пользователя.
2. Retrieval: поиск top-$k$ релевантных кусков по векторной близости (часто плюс гибрид с ключевым поиском BM25).
3. Augmentation: найденные куски вставляются в промпт как контекст.
4. Generation: LLM генерирует ответ строго на основе поданного контекста, в идеале со ссылками на источники.

Векторные БД в 2026 — стандарт инфраструктуры. Специализированные (Qdrant, Weaviate, Milvus, Pinecone) и расширения обычных БД (pgvector для PostgreSQL — частый дефолт, если данных не миллиарды). Они хранят векторы и делают быстрый ANN-поиск с фильтрами по метаданным (например, искать только в документах для рынка UZ).

Качество RAG почти целиком решается на этапе retrieval, а не генерации. Если ты подал нерелевантные куски — модель честно ответит ерунду по ерунде. Поэтому продвинутый RAG в 2026 включает: разумный chunking (по смысловым границам, с перекрытием), reranking (после грубого поиска top-50 пропустить через точную cross-encoder модель и оставить top-5), гибридный поиск (вектор + ключевые слова), и query rewriting (переформулировать вопрос пользователя перед поиском).

### Когда RAG, когда fine-tuning, когда просто промпт

Это самое частое архитектурное решение, и его путают. Разведём по тому, какую проблему решаем.

- Не хватает знаний / факты часто меняются → RAG. Регламенты, документация, актуальные данные. Обновить базу = переиндексировать, без переобучения.
- Не хватает поведения / стиля / формата → fine-tuning. Модель должна отвечать в строго определённом тоне, формате, на узком жаргоне, который промптом не объяснить за разумное число токенов.
- Задача простая и общая → просто хороший промпт (плюс few-shot). Начинай всегда отсюда.

RAG отвечает на вопрос "что модель знает", fine-tuning — "как модель себя ведёт". Их часто комбинируют, но решают они разное. Дефолтный порядок усложнения: промпт → few-shot → RAG → fine-tuning. Поднимайся по лестнице только когда предыдущая ступень доказанно не вывозит на твоих evals.

!!! warning "RAG не лечит плохие данные"

    Если твоя база знаний противоречива, устарела или содержит дубли с разными цифрами GGR — RAG добросовестно подаст модели мусор, и она уверенно ответит мусором. Data quality важнее всякой модели. RAG усиливает качество твоей базы знаний, как хорошее, так и плохое.

??? success "Ответы"

    1. Незнание внутренних данных и галлюцинации; идея — сначала найти релевантные куски базы знаний и положить их в промпт, чтобы модель отвечала по фактам, а не по памяти.
