# M20.14 · Стоимость и выбор модели

### Интуиция

Не существует "лучшей модели" — есть лучшая под задачу и бюджет. Гонять флагман на простой классификации тикетов — как возить пиццу на гоночном болиде. Выбор — это инженерный компромисс качество/цена/скорость.

### Практически

Платишь за токены, отдельно за вход (prompt) и выход (completion), выход обычно в несколько раз дороже. Стоимость запроса:

$$
\text{Cost} = \frac{N_{\text{in}}}{10^6}\cdot p_{\text{in}} + \frac{N_{\text{out}}}{10^6}\cdot p_{\text{out}}
$$

где $p$ — цена за миллион токенов. Рычаги экономии:

- Routing по сложности: дешёвая/малая модель на простое, флагман только на сложное. Часто 90% запросов закрывает маленькая модель.
- Prompt caching: если большой кусок промпта (системная инструкция, документация) повторяется, провайдеры кешируют его и берут за повторное чтение во много раз меньше. Огромная экономия для RAG и агентов.
- Контроль длины: лаконичные промпты, ограничение `max_tokens`, отсев лишнего контекста.
- Батчинг: офлайн-задачи через batch API дешевле.
- Маленькие open-source модели на своём железе при больших стабильных объёмах (но считай полную стоимость владения GPU, а не только цену токена).

Принцип llm-cost-optimizer: сначала измерь (откуда уходят токены), потом оптимизируй, потом мониторь постоянно.
