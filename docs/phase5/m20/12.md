# M20.12 · Оценка LLM-систем: evals

### Интуиция

Без измерения качества ты не инженер, а гадалка. "Вроде отвечает нормально" — не метрика. Evals (оценочные наборы) — это твой "тестовый датасет" для LLM-системы: набор вопросов с эталонными ответами или критериями, по которым ты численно меряешь, стало лучше или хуже после каждого изменения промпта, модели, chunking-стратегии.

### Формально

Eval — это (датасет входов, способ оценки выхода, агрегированная метрика). Способы оценки:

- Эталонные ответы: для классификации/извлечения — точность, F1, как в обычном ML.
- Программные проверки: валиден ли JSON, есть ли запрещённые слова, попал ли в источник.
- LLM-as-a-judge: другая (сильная) модель оценивает ответ по рубрике (релевантность, фактологичность, тон) по шкале. Дёшево и масштабируемо, но судья сам шумит и предвзят (любит длинные ответы, свой стиль) — калибруй его на размеченной человеком выборке.
- Человеческая оценка: золотой стандарт для финальной валидации, дорого.

Для RAG отдельно меряют retrieval (precision@k, recall@k, попал ли нужный кусок в top-k) и generation (faithfulness — не противоречит ли ответ источникам; answer relevance). Фреймворки 2026: RAGAS, и трейсинг-платформы (LangSmith, Langfuse, Arize Phoenix) для логирования и оценки в проде. Принцип: eval-first. Сначала собери хотя бы 30-50 показательных примеров с эталоном, потом крути систему. Иначе оптимизируешь вслепую.
