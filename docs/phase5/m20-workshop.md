## Что отрабатываем

Модуль M20 говорит: прежде чем дообучать модель на текстовой задаче, попробуй «эмбеддинг плюс лёгкий классификатор» (M20.4, M20.8), а для свежих фактов бери RAG, а не fine-tuning (M20.9, M20.10). Здесь ты собираешь это руками на потоке обращений игроков казино.

Отрабатываем понятия: sentence/document-эмбеддинги, косинусная близость, эмбеддинг-классификатор поверх логрегрессии, семантический поиск top-$k$, RAG-пайплайн (индексация → retrieval → augmentation → generation), eval-first (retrieval-метрика recall@k и generation-проверка faithfulness), заземление с разрешением сказать «нет данных» против галлюцинаций, и сравнение подхода с fine-tuning по стоимости.

Артефакт на выходе:

- рабочий эмбеддинг-классификатор тикетов по 5 категориям с мини-eval (accuracy, F1);
- минимальный RAG над регламентами с eval-набором и метрикой recall@k;
- табличка стоимости: эмбеддинги против fine-tuning на твоём объёме.

Весь код локальный, без платных API-ключей: эмбеддер — многоязычная модель E5 (сильна на русском, как и советует M20.4). Генерацию RAG-ответа делаем без вызова облака — извлечённый контекст уже самодостаточен, а место под облачный вызов помечено явно.

## Ход работы

### Шаг 1: эмбеддер и косинусная близость

Зачем: M20.4 — превратить текст в вектор и проверить руками, что близкое по смыслу лежит рядом. Берём многоязычную E5, для запросов и документов у неё есть префиксы `query:` / `passage:`.

```python
from sentence_transformers import SentenceTransformer
model = SentenceTransformer("intfloat/multilingual-e5-small")

def embed(texts, kind="passage"):
    pref = [f"{kind}: {t}" for t in texts]
    return model.encode(pref, normalize_embeddings=True)  # косинус = скалярное произведение

probe = ["внёс депозит", "пополнил счёт", "забыл пароль"]
V = embed(probe)
cos = V @ V.T
print(np.round(cos, 3))
```

Что получилось: матрица косинусов, где «внёс депозит» и «пополнил счёт» ближе друг к другу, чем любая из пар с «забыл пароль». Векторы уже нормированы, поэтому косинус — это просто `V @ V.T`.

!!! question "Проверь себя"

    1. Почему здесь косинус совпадает со скалярным произведением?
    2. Зачем E5 разделяет префиксы `query:` и `passage:`?

??? success "Ответы"

    1. Векторы нормированы (`normalize_embeddings=True`), длина равна 1, и в формуле косинуса знаменатель становится единицей — остаётся только числитель $u\cdot v$.
    2. Модель обучена асимметрично: вопрос и документ кодируются по-разному, префиксы переключают режим и поднимают качество retrieval по сравнению с одинаковым кодированием.

### Шаг 2: эмбеддинг-классификатор обращений

Зачем: ядро M20.8 — лёгкий классификатор поверх векторов вместо fine-tuning. Эмбеддинг несёт «понимание языка», логрегрессия учится за секунды.

```python
import time
from sklearn.model_selection import train_test_split
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import accuracy_score, f1_score, classification_report

X = embed(df.text.tolist())
Xtr, Xte, ytr, yte = train_test_split(X, df.label, test_size=0.3, random_state=20, stratify=df.label)

t0 = time.perf_counter()
clf = LogisticRegression(max_iter=1000, C=10).fit(Xtr, ytr)
fit_s = time.perf_counter() - t0

pred = clf.predict(Xte)
print(f"обучение: {fit_s:.2f} c")
print(f"accuracy={accuracy_score(yte, pred):.3f}  f1_macro={f1_score(yte, pred, average='macro'):.3f}")
print(classification_report(yte, pred))

for q in ["деньги списались, а на счёт не зачислились", "не могу отыграть фриспины"]:
    print(q, "->", clf.predict(embed([q]))[0])
```

Что получилось: accuracy и macro-F1 заметно выше 0.9 на синтетике, обучение — доли секунды, GPU не нужен. Это и есть «эмбеддинг плюс линейный классификатор» из M20.4.

### Шаг 3: мини-eval против zero-shot и сравнение цены с fine-tuning

Зачем: M20.10 и M20.14 — обосновать выбор подхода цифрами, а не модой. Сравниваем три пути: наш эмбеддинг-классификатор, zero-shot через генеративную LLM и fine-tuning. Eval уже есть (Шаг 2); добавляем эмуляцию zero-shot baseline через ближайший центроид категории (kNN-логика без облака) и считаем стоимость.

```python
# Zero-shot имитация: классификация по близости к усреднённому вектору меток-описаний
desc = {"withdrawal": "вывод средств и выплаты", "bonus": "бонусы промокоды фриспины вейджер",
        "tech": "технический сбой сайта приложения", "payment": "пополнение оплата списание депозита",
        "kyc": "верификация документов проверка аккаунта"}
labels = list(desc)
C = embed([desc[k] for k in labels], kind="query")
zs_pred = [labels[i] for i in (Xte @ C.T).argmax(1)]
print(f"zero-shot accuracy={accuracy_score(yte, zs_pred):.3f}")
print(f"trained   accuracy={accuracy_score(yte, pred):.3f}")
```

```python
# Стоимость на 100k тикетов/мес. Цены за 1M токенов — подставь актуальные из доки провайдера.
N = 100_000
tok_in = 60  # средний тикет ~60 токенов (русский дороже, см. M20.6)
def cost(price_in_1m, tok=tok_in, n=N, price_out_1m=0.0, tok_out=0):
    return n * tok / 1e6 * price_in_1m + n * tok_out / 1e6 * price_out_1m

emb_infer = cost(0.02)                       # дешёвый embeddings endpoint
genllm    = cost(0.15, price_out_1m=0.60, tok_out=15)  # generative zero-shot на каждый тикет
ft_train  = 25.0                             # разовый LoRA-прогон (грубо)
ft_infer  = cost(0.15, price_out_1m=0.60, tok_out=5)   # инференс дообученной модели
print(f"embeddings+logreg: ~${emb_infer:.2f}/мес (+0 обучение, локально вообще $0)")
print(f"generative zero-shot: ~${genllm:.2f}/мес")
print(f"fine-tuning: ${ft_train:.0f} разово + ~${ft_infer:.2f}/мес инференс + разметка и переобучение")
```

Что получилось: эмбеддинг-классификатор на порядок дешевле генеративного zero-shot и не требует разовых затрат на дообучение, при сопоставимом или лучшем качестве. Это прямая иллюстрация лестницы M20.9: промпт → few-shot → RAG → fine-tuning, и вывода M20.10 «fine-tuning не нужен, если эмбеддинги дают качество на evals».

### Шаг 4: индексация базы знаний и retrieval

Зачем: M20.9 — фаза индексации (chunk → эмбеддинг → хранилище) и онлайн-retrieval top-$k$ по косинусу. Векторную БД заменяем numpy-матрицей: на 10-1000 кусков точный перебор быстрее любой ANN, а интерфейс тот же.

```python
KB_VEC = embed(KB, kind="passage")  # индекс: один вектор на кусок

def retrieve(question, k=3):
    qv = embed([question], kind="query")[0]
    sims = KB_VEC @ qv
    idx = np.argsort(-sims)[:k]
    return [(int(i), KB[i], float(sims[i])) for i in idx]

for i, txt, s in retrieve("сколько идёт вывод?"):
    print(f"[{i}] {s:.3f}  {txt}")
```

Что получилось: на вопрос про вывод в top-3 поднимаются куски про 24 часа и про зависший вывод. Это retrieval-ядро RAG; качество системы решается именно здесь (M20.9).

### Шаг 5: RAG-ответ с заземлением и eval

Зачем: M20.9 (augmentation + generation с разрешением «нет данных») и M20.12 (eval-first: retrieval-метрика recall@k и generation-проверка). Эталоны для eval делаем как множество id релевантных кусков на вопрос.

```python
EVAL = [
    ("за сколько часов приходит вывод?", {0, 1}),
    ("какой вейджер у бонуса за депозит?", {2}),
    ("куда вводить промокод?", {4}),
    ("вернут ли двойное списание?", {5}),
    ("сколько идёт проверка документов?", {7}),
    ("какие платёжные методы для UZ?", {8}),
]

def recall_at_k(k=3):
    hits = []
    for q, gold in EVAL:
        got = {i for i, _, _ in retrieve(q, k)}
        hits.append(len(got & gold) / len(gold))
    return float(np.mean(hits))

for k in (1, 3, 5):
    print(f"recall@{k} = {recall_at_k(k):.3f}")

def rag_answer(question, k=3):
    ctx = retrieve(question, k)
    context = "\n".join(f"[{i}] {t}" for i, t, _ in ctx)
    prompt = ("Ответь ТОЛЬКО на основе контекста. Если ответа нет — скажи 'нет данных'. "
              f"Указывай номер источника.\n\nКонтекст:\n{context}\n\nВопрос: {question}")
    # Место под облачный вызов (M20.15): client.responses.create(model=..., temperature=0, input=prompt)
    # Локальная проверка faithfulness: ответ должен опираться на топовый кусок
    top_id, top_text, top_sim = ctx[0]
    grounded = top_sim > 0.80
    return {"source": top_id, "text": top_text, "grounded": grounded, "prompt": prompt}

print(rag_answer("за сколько часов приходит вывод?"))
print(rag_answer("какой лимит ставки на покер?"))  # нет в базе -> grounded=False -> "нет данных"
```

Что получилось: recall@3 высокий, на вопрос вне базы топовый косинус низкий — это сигнал ответить «нет данных» вместо галлюцинации (M20.11). Промпт собран с явным заземлением и нумерацией источников; подставив строку вызова провайдера из M20.15 при `temperature=0`, получаешь полноценный RAG-ответ с цитатами.

## Бизнес-вывод

Технический результат (accuracy, F1, recall@k, табличка стоимости) сам по себе не решение. Переведи его в рекомендацию, которую Head of Support сможет отнести на планёрку. Зафиксируй:

- [ ] **Рекомендация одним предложением:** что делаем — например, «внедряем эмбеддинг-классификатор + RAG-подсказку, генеративную LLM подключаем только на хвост сложных тикетов», без жаргона про логрегрессию.
- [ ] **Эффект в деньгах и метриках:** на сколько падает время первого ответа и сколько ручных часов высвобождается; сравни три сценария по стоимости (локальные эмбеддинги почти $0 против генеративного zero-shot и против разового дообучения) и переведи в ₽/мес экономии.
- [ ] **Риски и допущения:** качество померено на синтетике, нужен замер на реальных тикетах; классы могут дрейфовать (новые акции, рынки); RAG отвечает «нет данных» вместо галлюцинации — это плюс для compliance, но часть вопросов уйдёт оператору.
- [ ] **Следующий шаг:** пилот на N% живого трафика с замером recall@k и доли «нет данных», ручная разметка ~200 ответов для калибровки, ретро через 2 недели.
- [ ] **Как подать стейкхолдеру:** язык решений, не метрик — «снижаем нагрузку на операторов и время ответа при near-zero стоимости инференса, дообучение не окупается», а accuracy/F1 держи в приложении как доказательство.
