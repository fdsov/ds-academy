# M21.6 · Промпт-паттерны для DS

Качество ответа сильно зависит от промпта. Несколько паттернов, проверенных на практике (теория промптинга — в модуле M20).

### Контекст плюс схема плюс ограничения

Плохо: «напиши код для retention». Хорошо: дай роль, схему таблицы, определение метрики, версии библиотек и формат вывода.

```text
Ты senior data scientist. Стек: Python 3.12, polars 1.x.
Таблица events со столбцами: player_id (str), event_date (date),
event_type (enum: deposit, bet, login). Нужна функция, считающая
classic N-day retention для когорт по дате первого депозита.
Retention day N = доля игроков когорты, у которых был хотя бы один
login ровно на день N после первого депозита. Верни polars-код,
векторизованный, без apply. Добавь docstring и обработку пустого ввода.
```

Чем точнее определение метрики в промпте, тем меньше шанс, что модель подставит своё.

### Объясни, потом напиши (chain-of-thought)

Попроси сначала описать план словами, потом код. Это и улучшает результат, и даёт тебе точку контроля: если план неверен, останавливаешь до генерации кода.

### Сгенерируй тесты к своему коду

«Напиши pytest-тесты, включая edge cases: пустой ввод, один игрок, дубликаты событий, игрок без login после депозита». Тесты ты всё равно проверяешь, но это заставляет модель подумать о граничных случаях, которые она сама же нарушила.

### Критика собственного вывода

«Теперь выступи в роли скептика: найди в этом коде и в этом статистическом выводе три потенциальные ошибки». Модель неплохо находит свои же ляпы, когда её просят сменить роль. Не панацея, но дешёвый дополнительный фильтр.

### Anti-pattern: не проси выводов из данных, которых ты не дал

«Почему у нас упал GGR в марте» без данных — это приглашение к галлюцинации. Модель сочинит правдоподобную историю. Либо дай агрегаты, либо проси список гипотез для проверки, а не готовый вывод.

!!! question "Проверь себя"

    1. Из каких элементов состоит паттерн «контекст плюс схема плюс ограничения» и зачем в нём точное определение метрики?
    2. Чем паттерн «объясни, потом напиши» даёт тебе точку контроля?
    3. Почему просьба «почему у нас упал GGR в марте» без данных — это anti-pattern?
    4. Зачем просить модель сгенерировать тесты к собственному коду, если ты всё равно их проверяешь?

??? success "Ответы"

    1. Роль, схема таблицы (колонки, типы, пример строк без реальных данных), точное определение метрики, версии библиотек, формат вывода. Точное определение метрики мешает модели подставить своё понимание retention/LTV.
    2. Если план словами неверен, ты останавливаешь процесс до генерации кода — дешевле, чем ревьюить готовый неверный код.
    3. Без данных модель сочинит правдоподобную историю-галлюцинацию. Нужно либо дать агрегаты, либо просить список гипотез для проверки, а не готовый вывод.
    4. Это заставляет модель подумать о граничных случаях, которые она сама же нарушила в коде; ты получаешь дополнительный фильтр, хотя проверка тестов остаётся за тобой.

!!! example "Мини-задание"

    За 3-5 минут напиши промпт по паттерну «контекст плюс схема плюс ограничения» для одной своей реальной задачи: укажи роль, схему таблицы (без реальных данных), точное определение целевой метрики, версии библиотек и формат вывода. Проверь, что определение метрики не допускает двойного толкования.
