# M21.6 · Промпт-паттерны для DS

### Интуиция

На пальцах: LLM — это не оракул, который знает твой проект, а очень начитанный, но контекстно-слепой стажёр. Он не видит твою схему таблиц, не знает, как именно у вас считается retention, не в курсе версии polars в репозитории. Если ты не сказал — он догадается, а догадка превращается в правдоподобный, но неверный код или в выдуманный вывод из данных.

Промпт-паттерн — это дисциплина передачи контекста. Хороший промпт не "просит магию", а сужает пространство допустимых ответов до одного правильного: даёт роль, схему, точное определение метрики, ограничения и формат вывода. Чем уже воронка, тем меньше шансов, что модель подставит своё понимание вместо твоего.

Вторая половина интуиции — про контроль. Модель ошибается, и это нормально. Задача аналитика — встроить дешёвые точки проверки: сначала план словами (можно остановить до генерации кода), потом тесты на граничные случаи, потом критика собственного вывода. Каждый паттерн — это либо способ дать контекст, либо способ поймать ошибку дёшево, до того как она доехала до прода и до денег.

### Формально

Промпт-паттерн — воспроизводимый шаблон формулировки запроса к LLM, снижающий дисперсию и систематическую ошибку ответа за счёт явной передачи контекста и встроенных проверок. Базовые паттерны для DS-работы:

1. Контекст плюс схема плюс ограничения. Структура промпта: роль + схема данных (колонки, типы, пример строки без реальных данных) + точное определение целевой метрики + версии библиотек + формат вывода.

```text
Ты senior data scientist. Стек: Python 3.12, polars 1.x.
Таблица events со столбцами: player_id (str), event_date (date),
event_type (enum: deposit, bet, login). Нужна функция, считающая
classic N-day retention для когорт по дате первого депозита.
Retention day N = доля игроков когорты, у которых был хотя бы один
login ровно на день N после первого депозита. Верни polars-код,
векторизованный, без apply. Добавь docstring и обработку пустого ввода.
```

2. Объясни, потом напиши (chain-of-thought). Просьба сначала описать план словами, затем код. Даёт точку контроля до генерации.
3. Сгенерируй тесты к коду. Явный список edge cases: пустой ввод, один игрок, дубликаты событий, игрок без login после депозита.
4. Критика собственного вывода (self-critique). Смена роли на скептика: "найди три потенциальные ошибки в коде и статистическом выводе".
5. Anti-pattern: вывод из неданных данных. Запрос "почему упал GGR в марте" без агрегатов provоцирует галлюцинацию — модель сочинит причинную историю.

Формально полезность паттерна можно описать через ожидаемую стоимость ошибки. Пусть $p$ — вероятность дефектного ответа, $C$ — цена пропущенной в прод ошибки, $c$ — стоимость встроенной проверки (план, тесты). Без проверки ожидаемые потери:

$$
E_0 = p \cdot C.
$$

Точка контроля ловит долю $r$ дефектов до прода ценой $c$, оставляя:

$$
E_1 = c + p \,(1 - r)\, C.
$$

Проверка окупается, когда $E_1 < E_0$, то есть когда

$$
c < p \cdot r \cdot C.
$$

Если $C$ велико (ошибка в метрике дорогая), даже неидеальный фильтр ($r$ далеко от 1) оправдан.

#### Разбор на пальцах: сколько стоит размытое определение метрики

Возьмём промпт без точного определения retention: "напиши код для N-day retention". Модель с равной вероятностью реализует одну из двух распространённых трактовок:

- classic retention: login ровно на день $N$;
- rolling/range retention: хотя бы один login в интервале $[1, N]$.

Это разные метрики. На реальных данных rolling всегда выше classic, потому что считает по накопленному окну. Допустим, истинный classic 7-day retention когорты $R_{\text{classic}} = 18\%$, а rolling даёт $R_{\text{rolling}} = 31\%$.

Пусть когорта — 40 000 новых FTD-игроков в месяц, и команда планирует CRM-бюджет на удержание исходя из числа "ещё активных на день 7". По неверной метрике "активных" получится $0.31 \times 40\,000 = 12\,400$, по верной — $0.18 \times 40\,000 = 7\,200$. Разница 5 200 игроков.

Если на каждого "активного" закладывается бонус-стоимость $8, то завышенная метрика раздувает план на $5\,200 \times \$8 = \$41\,600$ в месяц. Проверим арифметику: $12\,400 - 7\,200 = 5\,200$; $5\,200 \times 8 = 41\,600$. То есть одно пропущенное слово в определении метрики стоит порядка $42\,000 в месяц мимо бюджета.

Теперь подставим в формулу окупаемости. Цена ошибки $C \approx \$41\,600$, стоимость точки контроля (дописать определение метрики и прочитать план модели) $c \approx \$20$ рабочего времени, вероятность что модель угадает не ту трактовку $p \approx 0.5$, доля отлова $r \approx 0.9$. Условие $c < p r C$ даёт $20 < 0.5 \times 0.9 \times 41\,600 = 18\,720$. Неравенство выполняется с запасом в три порядка — точное определение метрики окупается почти всегда.

!!! warning "Типичная ошибка"

    Новичок относится к LLM как к коллеге, который "и так знает контекст проекта". Модель не знает ни вашей схемы, ни вашего определения метрики, ни версий библиотек — у неё нет доступа к репозиторию, если вы его не дали в промпт. Самая дорогая ошибка — принять сгенерированный код за корректный, потому что он "выглядит правильно": синтаксически чистый код может считать не ту метрику. Всегда фиксируй определение целевой величины так, чтобы оно не допускало двойного толкования.

!!! info "Почему chain-of-thought помогает, а не магия"

    Просьба "сначала план словами" работает не потому, что модель "думает лучше", а потому что переносит ошибку на дешёвый для проверки артефакт. План на естественном языке ты ревьюишь за 30 секунд; неверный код в 80 строк — за 20 минут, и часть багов всё равно пропустишь. Это инженерный приём сдвига точки контроля влево, а не свойство интеллекта модели.

!!! question "Проверь себя"

    1. Из каких элементов состоит паттерн "контекст плюс схема плюс ограничения" и зачем в нём точное определение метрики?
    2. Чем паттерн "объясни, потом напиши" даёт тебе точку контроля?
    3. Почему просьба "почему у нас упал GGR в марте" без данных — это anti-pattern?
    4. Зачем просить модель сгенерировать тесты к собственному коду, если ты всё равно их проверяешь?

??? success "Ответы"

    1. Роль, схема таблицы (колонки, типы, пример строк без реальных данных), точное определение метрики, версии библиотек, формат вывода. Точное определение метрики мешает модели подставить своё понимание retention/LTV — иначе она с заметной вероятностью реализует не ту трактовку (classic vs rolling).
    2. Если план словами неверен, ты останавливаешь процесс до генерации кода — проверка дешёвого артефакта (план) вместо дорогого (80 строк кода).
    3. Без данных модель сочинит правдоподобную причинную историю-галлюцинацию. Нужно либо дать агрегаты, либо просить список гипотез для проверки, а не готовый вывод.
    4. Это заставляет модель явно обработать граничные случаи, которые она сама же нарушила в коде; ты получаешь дополнительный дешёвый фильтр, хотя финальная проверка тестов остаётся за тобой.

!!! example "Мини-задание"

    За 3-5 минут напиши промпт по паттерну "контекст плюс схема плюс ограничения" для одной своей реальной задачи: укажи роль, схему таблицы (без реальных данных), точное определение целевой метрики, версии библиотек и формат вывода. Затем проверь определение метрики на двойное толкование: попробуй сам придумать вторую трактовку — если получилось, определение надо ужесточить.
