# M9.9 · Bootstrap — рабочий инструмент 2026

### Интуиция

Формула стандартной ошибки $\sigma/\sqrt{n}$ существует для среднего. А для медианы депозита? Для 90-го перцентиля времени сессии? Для retention как доли с хитрым знаменателем? Для отношения двух метрик? Аналитических формул либо нет, либо они уродливы.

Bootstrap решает это грубой силой. Идея почти наглая: твоя выборка — лучшая доступная модель популяции. Так давай будем многократно "пересобирать" выборки из самой выборки (с возвращением), каждый раз пересчитывать метрику и смотреть, как она гуляет. Этот разброс и есть оценка неопределённости — без единой формулы про распределение.

Аналогия: у тебя один срез игроков. Ты как будто множество раз "переигрываешь" эту неделю, каждый раз случайно вытягивая игроков из имеющихся (некоторые попадут дважды, некоторые ни разу), и смотришь, насколько прыгает retention. Реальную популяцию ты заменил эмпирической.

Почему это вообще работает (интуиция чуть глубже). Настоящая неопределённость метрики — это разброс, который ты увидел бы, повтори ты сбор данных заново много раз из истинной популяции. Но populяция тебе недоступна — у тебя один-единственный сэмпл. Bootstrap делает подмену: вместо истинного распределения $F$ он подставляет эмпирическое распределение $\hat{F}$ — то есть "равные веса $1/n$ на каждом наблюдении". Ресэмплинг с возвращением — это и есть генерация новых выборок из $\hat{F}$. Чем больше $n$, тем ближе $\hat{F}$ к $F$ и тем честнее bootstrap. На $n = 30$ он шумит, на $n = 50\,000$ депозитов он практически безупречен.

Важная тонкость про "с возвращением". Если бы ты тянул без возвращения $n$ из $n$, ты бы каждый раз получал ровно ту же выборку — разброса ноль, информации ноль. Возвращение создаёт вариативность состава: в среднем в каждой bootstrap-выборке оказывается около $63\%$ уникальных исходных наблюдений ($1 - e^{-1} \approx 0.632$), остальное — дубли. Именно эта перетасовка весов и имитирует "а что, если бы выборка вышла чуть другой".

### Формально

Пусть выборка $x_1, \dots, x_n$. Bootstrap-процедура для статистики $\hat\theta$:

1. Сформируй bootstrap-выборку: $n$ элементов, выбранных из исходной с возвращением.
2. Посчитай на ней статистику $\hat\theta^{*(b)}$.
3. Повтори шаги 1-2 $B$ раз (обычно $B = 1000 \dots 10000$).
4. Разброс набора $\{\hat\theta^{*(1)}, \dots, \hat\theta^{*(B)}\}$ оценивает неопределённость $\hat\theta$.

Стандартная ошибка bootstrap — это стандартное отклонение bootstrap-распределения. Доверительный интервал — percentile-метод: берёшь 2.5-й и 97.5-й перцентили набора $\hat\theta^{*}$.

Ключевой момент, который часто путают: на каждом шаге ресэмплинга размер bootstrap-выборки равен исходному $n$, не меньше и не больше. Если у тебя 800 игроков в когорте, каждая из $B$ псевдо-выборок тоже содержит 800 элементов (с повторами). Уменьшишь размер — раздуешь SE искусственно; это уже не bootstrap, а другая процедура.

### Разобранный пример: CI для медианы GGR

Пусть у когорты есть GGR на игрока, и нас интересует доверительный интервал медианы. Аналитической формулы для SE медианы на тяжёлом хвосте по сути нет — идеальный кейс для bootstrap.

```python
import numpy as np

rng = np.random.default_rng(42)
# реальные данные подгрузил бы из БД; тут синтетика с тяжёлым правым хвостом
ggr = rng.lognormal(mean=3.0, sigma=1.2, size=800)  # GGR в $ на игрока

B = 10_000
n = len(ggr)
boot_medians = np.empty(B)
for b in range(B):
    sample = rng.choice(ggr, size=n, replace=True)  # ровно n, с возвращением
    boot_medians[b] = np.median(sample)

point = np.median(ggr)
se = boot_medians.std(ddof=1)
ci_low, ci_high = np.percentile(boot_medians, [2.5, 97.5])

print(f"медиана GGR = ${point:,.2f}")
print(f"bootstrap SE = ${se:,.2f}")
print(f"95% percentile-CI = [${ci_low:,.2f}, ${ci_high:,.2f}]")
```

Что читать в выводе: точечная оценка — медиана по исходным данным; SE — насколько медиана "гуляет"; CI — диапазон, в который ты веришь на $95\%$. Если интервал, скажем, $[\$18.4, \$23.1]$, а у конкурирующей когорты медиана $\$25$ и лежит вне интервала — различие правдоподобно реально, а не шум выборки. Векторизованная версия (`rng.choice(ggr, size=(B, n))` + `np.median(axis=1)`) считает то же самое в разы быстрее, без цикла.

### Когда bootstrap лучше формул

- Метрика без простой формулы SE: медиана, перцентили, GGR на игрока, отношение метрик, коэффициент Джини, AUC.
- Сильно несимметричное распределение (депозиты — классический тяжёлый правый хвост), где нормальное приближение из CLT хромает на разумных $n$.
- Когда хочется честный интервал, не предполагая нормальности.

### Ограничения

Bootstrap не творит данные из ничего. Если исходная выборка смещена — bootstrap воспроизведёт смещение, а не вылечит его. Плохо работает для оценки экстремальных квантилей (максимум) и при сильной зависимости наблюдений (временные ряды требуют block bootstrap). И он стоит вычислений, хотя на данных 2026 это секунды.

### Типичные ошибки на практике

- Ресэмплинг неправильного размера. Тянуть надо ровно $n$ с возвращением. "Возьму половину для скорости" — занижает разброс состава и завышает SE.
- Маленькое $B$. При $B = 100$ перцентили CI сами по себе шумные. Для точечной SE хватает $\sim 1000$, для аккуратных хвостов CI бери $5000\text{--}10000$.
- Зависимые наблюдения. Если в выборке несколько строк на одного игрока (несколько сессий), наивный bootstrap по строкам недооценит неопределённость — ресэмплить надо кластеры (игроков целиком), а не строки. Для временных рядов — block bootstrap.
- Вера, что bootstrap чинит смещение. Опрос только активных игроков останется опросом только активных игроков, сколько ни ресэмпли. Bootstrap меряет неопределённость, не убирает selection bias.
- Экстремальные статистики. Максимум депозита в bootstrap-выборках упирается в максимум исходной выборки — распределение вырождается, интервал бессмысленный.
