## Что отрабатываем

Ты прогоняешь руками один эксперимент целиком — ровно тот кейс, что тянулся через всю теорию: новый онбординг должен поднять конверсию в первый депозит. По шагам:

- формулировка H0/H1 и MDE до старта (M10.3, M10.6);
- power-анализ и расчёт размера выборки на группу (M10.7);
- генерация данных эксперимента с фиксированным seed (M10.9);
- проверка санитарии — SRM на размеры групп (M10.9);
- тест пропорций плюс доверительный интервал на эффект (M10.5, M10.8);
- симуляция peeking и измерение раздутого alpha (M10.11);
- итоговое решение go/no-go с числами.

Артефакт на выходе — короткое решение по тесту: «катим / не катим», с lift, CI, p-value, проверкой SRM и явной оговоркой про peeking. Это та самая фраза «эффект реальный», которую нельзя оспорить интуицией.

Это фокусный воркшоп: один эксперимент, частотный подход, целевая метрика — конверсия. CUPED и байес из теории здесь не трогаем (они в разделе «Развитие»).

## Бизнес-кейс

!!! example "Ситуация"

    Ты — data scientist в iGaming-продукте. К тебе приходит продакт-оунер онбординга: команда полгода пилила новый сценарий первого входа и хочет раскатать его на 100% новых игроков. Он уверен, что «стало заметно лучше», и торопит с запуском. Твоя задача — не дать раскатать на ощущениях, а вынести честный go/no-go по A/B-тесту.

    - Базовая конверсия нового игрока в первый депозит — около 8%. Новый онбординг по плану должен дать минимум +0.5 п.п. — меньший прирост не окупает уже потраченную разработку и поддержку фичи.
    - Иллюстративно: при ~5000 новых игроков в день и среднем первом депозите порядка 2000 ₽ каждые +0.5 п.п. конверсии — это примерно +5000 × 0.005 × 2000 ≈ 50 000 ₽ выручки в день, около 1.5 млн ₽ в месяц только на FTD. Цена ошибки в обе стороны крупная.
    - Что на кону: раскатаешь пустышку (статзначимо, но ниже MDE) — заморозишь ресурс команды на поддержку фичи без отдачи; зарубишь рабочую фичу из-за peeking или битого сплита — упустишь реальный прирост и доверие команды.
    - Ограничение: решение нужно на основе одного корректно посчитанного теста, без права «подсматривать и катить раньше срока»; данных, кроме исходов A/B, нет — истинный эффект тебе неизвестен.

```python
import numpy as np
from scipy import stats
from statsmodels.stats.proportion import proportions_ztest, proportion_effectsize, confint_proportions_2indep
from statsmodels.stats.power import NormalIndPower

rng = np.random.default_rng(10)

BASELINE = 0.080      # текущая конверсия в депозит
TRUE_LIFT = 0.006     # истинный эффект онбординга (+0.6 п.п.), нам он "неизвестен"
P_CONTROL = BASELINE
P_TEST = BASELINE + TRUE_LIFT
```

Мы специально берём истинный эффект (+0.6 п.п.) чуть выше MDE (+0.5 п.п.) — так тест при корректной мощности должен его поймать. В реальности этого числа у тебя нет, поэтому ниже мы ни разу им не пользуемся при анализе, только при генерации.

## Ход работы

### Шаг 1: Гипотезы и MDE до старта

Зачем. Из M10.3 и M10.6: выбор альтернативы и порог практической пользы фиксируются ДО данных, иначе это подгонка. Новый онбординг логически может как помочь, так и навредить (агрессивный онбординг иногда отпугивает), поэтому берём двустороннюю альтернативу — это честнее и заодно ловит ухудшение.

$$
H_0: p_{test} - p_{control} = 0 \qquad H_1: p_{test} - p_{control} \neq 0
$$

MDE задаёт бизнес: прирост меньше +0.5 п.п. конверсии не окупает разработку, значит детектировать более мелкое нам не нужно. Уровень $\alpha = 0.05$, мощность $0.8$ — индустриальный дефолт.

```python
ALPHA = 0.05
POWER = 0.80
MDE_ABS = 0.005       # минимально интересный прирост: +0.5 п.п.
```

Что получилось. Зафиксированы: двусторонний тест, primary-метрика — конверсия в депозит, MDE = +0.5 п.п., $\alpha=0.05$, power $=0.8$. Это контракт эксперимента, менять его по ходу нельзя.

### Шаг 2: Power-анализ и размер выборки

Зачем. M10.7: сначала настраиваем микроскоп, потом смотрим. Считаем, сколько игроков на группу нужно, чтобы с мощностью 0.8 различить эффект MDE. Используем Cohen's h через `proportion_effectsize` — то же, что в коде теории.

```python
effect = proportion_effectsize(BASELINE + MDE_ABS, BASELINE)
n_per_group = NormalIndPower().solve_power(
    effect_size=effect, alpha=ALPHA, power=POWER, alternative="two-sided"
)
n_per_group = int(np.ceil(n_per_group))
print(f"Нужно на группу: {n_per_group}")

players_per_day = 5000          # на обе группы суммарно
days = np.ceil(2 * n_per_group / players_per_day)
print(f"Срок теста: {days:.0f} дней")
```

Что получилось. Порядка 49 000 игроков на группу, то есть около 20 дней при 5000 новых игроков в день на обе группы. Сверь с приближённой формулой из M10.7 руками — числа должны сойтись по порядку. Запомни: захочешь ловить вдвое меньший эффект — выборка вырастет вчетверо.

!!! question "Проверь себя"

    1. Почему здесь двусторонняя альтернатива, а не односторонняя «онбординг только улучшает»?
    2. Что произойдёт с требуемым $n$, если поднять мощность до 0.9?
    3. Можно ли уменьшить MDE до +0.1 п.п. «чтобы точнее»? Чем это обернётся?

??? success "Ответы"

    1. Агрессивный онбординг может и навредить; двусторонний тест ловит эффект в обе стороны и не маскирует ухудшение. Односторонний берут, только если ухудшение бизнесу безразлично.
    2. $n$ вырастет: $z_{1-\beta}$ увеличивается с 0.84 до примерно 1.28, числитель формулы растёт, выборка на группу становится больше.
    3. Формально можно, но $n$ растёт квадратично при уменьшении эффекта: ловить +0.1 п.п. вместо +0.5 п.п. — это примерно в 25 раз больше игроков и месяцы теста. MDE задаётся пользой, а не желанием точности.

### Шаг 3: Генерация данных эксперимента

Зачем. M10.9: рандомизируем по стабильному user_id (единица рандомизации = единица анализа). Симулируем биномиальные исходы «задепозитил / нет» в каждой группе при найденном размере выборки.

```python
n = n_per_group
dep_control = rng.binomial(n, P_CONTROL)
dep_test    = rng.binomial(n, P_TEST)

conv = np.array([dep_control, dep_test])
nobs = np.array([n, n])
p_c, p_t = conv / nobs
print(f"Контроль: {dep_control}/{n} = {p_c:.4f}")
print(f"Тест:     {dep_test}/{n} = {p_t:.4f}")
```

Что получилось. Две группы по ~49 000 игроков с наблюдёнными конверсиями около 8.0% и 8.6%. Дальше работаем только с `conv` и `nobs` — притворяемся, что `TRUE_LIFT` нам неизвестен.

### Шаг 4: Проверка санитарии (SRM)

Зачем. M10.9: до интерпретации эффекта проверь, что эксперимент валиден. Sample Ratio Mismatch — расхождение фактического деления с плановым 50/50. Проверяем хи-квадратом на размеры групп. Здесь мы делили поровну сами, поэтому SRM пройдёт — но смоделируем и битый случай, чтобы увидеть, как он ловится.

```python
def srm_check(n_a, n_b, expected_ratio=0.5):
    total = n_a + n_b
    exp_a = total * expected_ratio
    exp_b = total * (1 - expected_ratio)
    chi2 = (n_a - exp_a) ** 2 / exp_a + (n_b - exp_b) ** 2 / exp_b
    p = stats.chi2.sf(chi2, df=1)
    return p

print(f"SRM p (наш тест 50/50): {srm_check(n, n):.3f}")          # большой p — ок
print(f"SRM p (битый 50/47):    {srm_check(50000, 47000):.6f}")  # крошечный p — баг
```

Что получилось. У нашего эксперимента SRM-p большой (деление честное, тест валиден). У искусственного 50000/47000 — p практически ноль: это не случайность, а баг рандомизации, при котором весь анализ эффекта невалиден, как бы красиво ни выглядел uplift.

!!! question "Проверь себя"

    1. Почему при пробитом SRM нельзя смотреть на lift и p-value эффекта?
    2. SRM-p = 0.5 — это хорошо или плохо?

??? success "Ответы"

    1. SRM сигналит о баге рандомизации или логирования: группы несопоставимы, и разница в метрике может объясняться кривым сплитом, а не онбордингом. Сначала чинишь сплит, потом анализируешь.
    2. Хорошо. Большой p означает, что фактическое деление совместимо с плановым 50/50 — оснований подозревать баг нет. Тревожит именно маленький p.

### Шаг 5: Тест пропорций и доверительный интервал

Зачем. M10.5 и M10.8: конверсия — доля, значит z-тест пропорций. Но p-value без размера эффекта бесполезно, поэтому сразу считаем lift и 95% CI на разницу долей. CI говорит больше, чем p: направление, масштаб, неопределённость.

```python
zstat, pval = proportions_ztest(conv, nobs)
lift = p_t - p_c

ci_low, ci_high = confint_proportions_2indep(
    dep_test, n, dep_control, n, method="wald", compare="diff"
)
print(f"lift = {lift:+.4f} ({lift*100:+.2f} п.п.)")
print(f"z = {zstat:.2f}, p = {pval:.4f}")
print(f"95% CI на разницу: [{ci_low:+.4f}, {ci_high:+.4f}]")
print(f"Весь CI выше MDE {MDE_ABS}? {ci_low > MDE_ABS}")
```

Что получилось. Lift около +0.6 п.п., p заметно ниже 0.05 — эффект статзначим. Ключевой бизнес-вопрос из M10.6: где лежит CI относительно MDE. Если нижняя граница CI выше +0.5 п.п. — эффект и значим, и практически полезен. Если CI зацепляет ноль или весь ниже MDE — статзначимость есть, а пользы нет.

!!! question "Проверь себя"

    1. Чем CI на разницу полезнее, чем одно p-value?
    2. Сформулируй p-value этого теста строго одним предложением.
    3. Получили p = 0.001. Корректно ли сказать «эффект на 99.9% реален»?

??? success "Ответы"

    1. CI показывает направление, масштаб и неопределённость эффекта сразу, и его можно сопоставить с MDE; p-value лишь говорит «совместимо ли с H0», ничего не сообщая о размере.
    2. Вероятность получить разницу конверсий как минимум настолько же экстремальную, как наблюдённая, при условии, что истинной разницы нет ($P(|T|\ge|T_{obs}|\mid H_0)$).
    3. Нет. p-value — это $P(\text{данные}\mid H_0)$, а не вероятность правоты гипотезы; для второго нужны Байес и априоры. И о воспроизводимости оно ничего не говорит.

### Шаг 6: Симуляция peeking

Зачем. M10.11: показать руками, что подглядывание раздувает реальный alpha. Прогоняем много A/A-тестов (эффекта нет вообще), в каждом подсматриваем несколько раз и останавливаемся на первом $p < 0.05$. Считаем фактическую долю «побед».

```python
def aa_peeking(n_looks=10, batch=4000, alpha=0.05, n_sims=2000, p=BASELINE, seed=0):
    g = np.random.default_rng(seed)
    false_pos = 0
    for _ in range(n_sims):
        a = g.binomial(batch, p, size=n_looks).cumsum()
        b = g.binomial(batch, p, size=n_looks).cumsum()
        nn = np.arange(1, n_looks + 1) * batch
        hit = False
        for k in range(n_looks):
            _, pv = proportions_ztest([a[k], b[k]], [nn[k], nn[k]])
            if pv < alpha:
                hit = True
                break
        false_pos += hit
    return false_pos / n_sims

print(f"Реальный alpha при 10 подглядываниях: {aa_peeking():.3f}")
```

Что получилось. При полном отсутствии эффекта доля «значимых» результатов оказывается не 5%, а порядка 15-20% — ровно та катастрофа из теории. Вывод для решения: либо держим срок из Шага 2 и смотрим один раз в конце, либо переходим на sequential/always-valid дизайн. Промежуточные «уже значимо, катим!» — невалидны.

### Шаг 7: Решение go/no-go

Зачем. Собрать всё в одно решение с числами — это и есть артефакт воркшопа. Правило раскатки: SRM чист И эффект статзначим И весь CI выше MDE И тест не останавливали раньше срока.

```python
go = (srm_check(n, n) > 0.001) and (pval < ALPHA) and (ci_low > MDE_ABS)
verdict = "GO — катим на 100%" if go else "NO-GO — не раскатываем"
print(f"""
РЕШЕНИЕ ПО ТЕСТУ ОНБОРДИНГА
  SRM:     p={srm_check(n, n):.3f} (чисто)
  Эффект:  {lift*100:+.2f} п.п., 95% CI [{ci_low*100:+.2f}, {ci_high*100:+.2f}] п.п.
  Значим:  p={pval:.4f}
  MDE:     +{MDE_ABS*100:.1f} п.п. — CI выше MDE: {ci_low > MDE_ABS}
  Peeking: тест держали полный срок ({days:.0f} дней), один анализ в конце
  ВЕРДИКТ: {verdict}
""")
```

Что получилось. Готовое решение с обоснованием в числах. Если у тебя выпал GO — поздравляю, но честный отчёт обязан включать guardrail-проверку (см. ниже). Если NO-GO из-за CI ниже MDE — это правильный отказ от практически бесполезной фичи, а не провал.

## Развитие

1. Добавь CUPED (M10.12): сгенерируй preperiod-выручку с корреляцией $\rho=0.6$, скорректируй метрику и покажи, что нужный $n$ падает на 30-40% при той же мощности.
2. Добавь guardrail-метрики (M10.9): retention_d7 и долю chargeback, проверь их non-inferiority и примени BH-FDR на пачку p-value (M10.10). Раскатывай, только если ни один guardrail не пробит.
3. Замени фиксированный дизайн на sequential: реализуй границы Pocock или доверительные последовательности и повтори Шаг 6 — убедись, что always-valid тест держит alpha при подглядывании.
4. Переведи решение в байес (M10.13): бета-биномиал, посчитай $P(\text{test лучше})$ и expected loss, сравни вердикт с частотным go/no-go.
