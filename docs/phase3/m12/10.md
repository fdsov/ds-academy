# M12.10 · Дашборды: коротко

Когда график нужен не разово, а как живой интерфейс к обновляемым данным, — это дашборд.

## График против дашборда — в чём разница на пальцах

График — это снимок: ты построил его в ноутбуке, вставил в отчёт, и он замёрз во времени. Дашборд — это труба: данные текут снизу (из базы, из выгрузки), а сверху их видит человек, который сам выбирает фильтр, период, сегмент. Разница не в красоте, а в **жизненном цикле**: график живёт одну презентацию, дашборд живёт месяцами и обновляется без твоего участия.

Практический критерий, когда пора делать дашборд, а не очередной график:

- один и тот же вопрос задают второй раз («а покажи ретеншн ещё и за май»);
- ответ зависит от свежих данных, которые меняются каждый день (DAU, GGR, депозиты);
- смотреть будет кто-то кроме тебя, и желательно без твоего участия в 23:00.

Если все три пункта — да, ты строишь дашборд. Если хотя бы график нужен один раз для слайда — оставайся в ноутбуке, не плоди инфраструктуру.

## Три инструмента — и под какую задачу каждый

- **Streamlit** — собираешь дата-приложение на чистом Python за часы; идеален для внутренних инструментов аналитика и быстрых демо моделей. Не BI-платформа, а способ обернуть свой скрипт в UI.
- **Apache Superset** — полноценная open-source BI поверх SQL-источников: дашборды, SQL Lab, разграничение доступа. Для продуктовой аналитики на уровне команды/компании.
- **Metabase** — самый низкий порог входа, «вопросы к данным» без SQL, быстрый старт для нетехнических коллег.

Ключевое различие — **кто источник данных и кто потребитель**. Streamlit крутится вокруг твоего Python-кода: если внутри дашборда нужна ML-модель, кастомная логика, scikit-learn прямо в рантайме — это Streamlit, потому что виджет вызывает твою функцию. Superset и Metabase крутятся вокруг **SQL поверх готовой базы**: они не запускают твой Python, они шлют запрос в Postgres/ClickHouse и рисуют результат. Поэтому для «150 метрик продукта с разграничением доступа по отделам» берут Superset, а не Streamlit — иначе ты будешь руками писать то, что в BI уже есть из коробки (расписание обновлений, права, кэш, алёрты).

Грубая шкала «порог входа → потолок возможностей»:

| Инструмент | Кто пишет | Источник | Сильная сторона | Когда жмёт |
|---|---|---|---|---|
| Metabase | нетехнический коллега | SQL/визард | старт за минуты, вопросы без SQL | сложная логика, кастомные графики |
| Superset | аналитик/BI | SQL | права, масштаб, SQL Lab | нужен Python в рантайме |
| Streamlit | аналитик-питонист | любой (Python) | ML, кастом, гибкость | многопользовательский BI, права |

## Минимальный Streamlit — чтобы было не абстрактно

Дашборд на Streamlit — это обычный Python-скрипт, который запускается командой `streamlit run app.py`. Каждое движение пользователя (сдвинул слайдер, выбрал дату) перезапускает скрипт сверху вниз, а `st.cache_data` не даёт каждый раз ходить в базу.

```python
import streamlit as st
import pandas as pd

st.title("GGR по дням")

@st.cache_data(ttl=600)  # кэш на 10 минут, чтобы не дёргать базу на каждый клик

df = load_data()

period = st.slider("Дней назад", 7, 90, 30)      # виджет-фильтр
sub = df.tail(period)

st.metric("GGR за период", f"${sub['ggr'].sum():,.0f}")
st.line_chart(sub.set_index("day")["ggr"])
```

Здесь `st.metric` показывает одно число-KPI, `st.line_chart` — тренд, `st.slider` — фильтр периода. Это и есть дашборд: данные + фильтр + ответ на вопрос «как менялся GGR».

## Главное правило: каждый виджет отвечает на конкретный вопрос конкретной аудитории

Правило дашборда то же, что и графика: каждый виджет отвечает на конкретный вопрос конкретной аудитории. Дашборд из 30 графиков «на всякий случай» — не инструмент, а свалка, в которую никто не смотрит.

Как это выглядит на гемблинг-проекте. Дашборд для **head of retention** — это 4-5 виджетов: ретеншн по когортам, доля реактиваций, средний срок до второго депозита, churn-rate VIP-сегмента. Дашборд для **CFO** — другие 4-5 виджетов: GGR, NGR, бонусная нагрузка, payback по каналам. Один дашборд не должен обслуживать обоих — у них разные вопросы. Если ты пытаешься впихнуть всё в одну страницу «чтобы было», получается ровно та свалка из 30 графиков: каждый отдельно смотрит мимо, а вместе они не складываются в решение.

Типичные ошибки, которые превращают дашборд в свалку:

- **«А давай ещё добавим»** — каждый новый виджет без явного вопроса размывает фокус. Перед добавлением спроси: на какой вопрос он отвечает и кто его задаёт.
- **Vanity-метрики** — большие красивые числа (всего регистраций за всё время), на которые нельзя повлиять решением. KPI должен быть actionable.
- **Нет владельца** — дашборд, который никто не открывает неделями, удаляют, а не «оставляют на всякий случай».
- **Микс аудиторий** — операционные метрики саппорта и стратегические метрики борда на одной странице: ни один из двоих не найдёт своё за 5 секунд.

!!! tip "Тест на полезность виджета"

    Перед тем как добавить график на дашборд, сформулируй вслух: «Этот виджет отвечает на вопрос ___, его задаёт ___, и по ответу он сделает ___». Если хотя бы один пропуск не заполняется — виджет лишний.
