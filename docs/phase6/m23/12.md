# M23.12 · Качество данных, идемпотентность, backfill

**На пальцах.** Пайплайн должен не только перевозить данные, но и проверять их в пути, уметь безопасно перезапускаться и уметь пересчитывать прошлое.

**Формально, три ключевых понятия.**

**Качество данных** встраивается в пайплайн как тесты (dbt tests, Great Expectations) на свежесть, полноту, уникальность, диапазоны. Измеряют по осям: completeness (нет ли дыр), validity (значения в допустимых границах — депозит не отрицательный), consistency (GGR в витрине сходится с сырьём), timeliness (данные свежие), uniqueness (нет дублей ставок). Проверки должны **падать громко** и блокировать публикацию плохих данных, а не молча пропускать.

**Идемпотентность** — свойство операции давать один и тот же результат при повторном запуске. Если пайплайн за 15 июня запустился дважды (упал и перезапустился), в витрине не должно появиться удвоенного GGR. Достигается через паттерн «delete-insert / merge по партиции»: перезапуск за дату сначала удаляет данные этой партиции, потом вставляет заново. Append без удаления — главный источник задвоений.

**Backfill** — пересчёт исторических данных по новой или исправленной логике. Понадобился новый разрез витрины или нашли баг — нужно прогнать пайплайн по всей истории. Backfill возможен только если: сырьё сохранено (привет, ELT), пайплайн идемпотентен и параметризован датой. Поэтому идемпотентность и backfill — две стороны одной монеты.

!!! info "Современный data stack: общая схема"

    Источники (продуктовая OLTP-база, Kafka, внешние API)
    → загрузка (Fivetran / Airbyte / свои продюсеры)
    → хранилище-сырьё (Iceberg/Parquet на S3, или Snowflake/BigQuery)
    → трансформации (dbt: staging → intermediate → marts)
    → оркестрация всего этого (Dagster / Airflow / Prefect)
    → потребление (BI-дашборды, DuckDB-аналитика, ML feature store)
    → поверх всего: качество данных, каталог/lineage, мониторинг.

!!! question "Проверь себя"

    1. По каким осям измеряют качество данных и приведи пример проверки для депозита.
    2. Что такое идемпотентность пайплайна и через какой паттерн она достигается?
    3. Почему backfill возможен только при идемпотентности, параметризации датой и сохранённом сырье?
    4. Почему `append` без удаления партиции — главный источник задвоений?

??? success "Ответы"

    1. Completeness (нет дыр), validity (значения в границах — депозит не отрицательный), consistency (витрина сходится с сырьём), timeliness (свежесть), uniqueness (нет дублей ставок); пример — проверка, что депозит не меньше нуля.
    2. Свойство давать тот же результат при повторном запуске; достигается паттерном delete-insert / merge по партиции (перезапуск за дату сначала удаляет данные партиции, потом вставляет заново).
    3. Backfill — пересчёт истории по новой/исправленной логике: нужно из чего пересчитывать (сырьё, привет ELT), безопасный повторный прогон (идемпотентность) и возможность гнать по конкретной дате (параметризация).
    4. При повторном запуске дня append добавляет те же строки ещё раз, и GGR в витрине задваивается — появляется «рост», которого нет.
