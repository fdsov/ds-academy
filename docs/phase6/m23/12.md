# M23.12 · Качество данных, идемпотентность, backfill

### Интуиция

На пальцах: пайплайн данных — это не просто труба, которая перекачивает события из продуктовой базы в витрину. Это труба с тремя обязательными свойствами, без которых ей нельзя доверять.

Первое — данные надо проверять прямо в пути. Представь конвейер на заводе: между станками стоят контролёры, которые отбраковывают детали с дефектом, не дожидаясь, пока бракованный продукт уедет к клиенту. В данных то же самое: депозит не может быть отрицательным, GGR в витрине обязан сходиться с сырьём, ставка не должна дублироваться. Если проверка не встроена в трубу, брак доедет до дашборда, и команда будет принимать решения на основе мусора, не зная об этом.

Второе — труба обязана уметь безопасно перезапускаться. Реальность такая: пайплайны падают. Сеть моргнула, кончилась память, упал источник. Оркестратор перезапускает упавший шаг. И вот ключевой вопрос: что будет с витриной, если расчёт за 15 июня отработает дважды? Если правильно — ничего, результат тот же. Если неправильно — GGR за этот день задвоится, и в отчёте появится «рост», которого в реальности нет.

Третье — труба должна уметь пересчитывать прошлое. Нашли баг в логике расчёта retention за последние полгода, или продукт попросил новый разрез витрины. Нужно прогнать пайплайн по всей истории заново. Это возможно только если сырьё сохранено и труба умеет считать любой конкретный день по требованию.

Эти три свойства — качество, идемпотентность, backfill — не роскошь, а минимальный контракт надёжного дата-пайплайна.

### Формально

Разберём три понятия строго.

**Качество данных** встраивается в пайплайн как автоматические тесты (dbt tests, Great Expectations, Soda) и измеряется по нескольким независимым осям:

- completeness (полнота) — нет ли пропусков и дыр в данных;
- validity (валидность) — значения в допустимых границах: депозит $d \ge 0$, валюта из справочника, дата не из будущего;
- consistency (согласованность) — агрегаты в витрине сходятся с сырьём, $GGR_{\text{витрина}} = GGR_{\text{сырьё}}$;
- timeliness (свежесть) — данные пришли вовремя, лаг не превышает SLA;
- uniqueness (уникальность) — нет дублей по ключу (одна ставка — одна строка).

Проверки должны **падать громко** и блокировать публикацию плохих данных вниз по пайплайну, а не молча логировать предупреждение. Это разница между circuit breaker (остановить и не отравить потребителей) и пассивным логом, который никто не читает.

**Идемпотентность** — свойство операции давать один и тот же результат при повторном применении. Формально операция $f$ идемпотентна, если

$$
f(f(x)) = f(x).
$$

Для пайплайна это значит: запуск расчёта за дату $D$ один раз и $k$ раз подряд даёт идентичное состояние витрины. Достигается паттерном **delete-insert / merge по партиции**: перезапуск за дату сначала удаляет все строки этой партиции, затем вставляет заново.

$$
\text{state} = (\text{DELETE WHERE } dt = D) \;\rightarrow\; (\text{INSERT } \text{rows}(D)).
$$

Противоположность — голый `append` (INSERT без удаления): он не идемпотентен, повторный запуск добавляет те же строки второй раз.

**Backfill** — пересчёт исторических данных по новой или исправленной логике. Backfill возможен только при одновременном выполнении трёх условий:

1. сырьё сохранено (это даёт парадигма ELT: грузим сырое, трансформируем потом, всегда можем пересчитать);
2. пайплайн идемпотентен (повторный прогон по уже существующим датам безопасен);
3. пайплайн параметризован датой (можно запустить расчёт ровно за нужный диапазон $[D_1, D_2]$).

Поэтому идемпотентность и backfill — две стороны одной монеты: без первого второе либо невозможно, либо задваивает историю.

#### Разбор на пальцах: во что обходится неидемпотентный append

Витрина считает дневной GGR казино. 15 июня пайплайн стартовал, обработал часть данных, упал на середине из-за таймаута к источнику, оркестратор перезапустил таск с нуля. Логика — голый `append`.

Положим, реальный GGR за 15 июня:

$$
GGR_{15.06} = \text{ставки} - \text{выплаты} = \$420\,000 - \$340\,000 = \$80\,000.
$$

Первый (упавший) прогон успел записать часть строк, второй прогон записал полный день. Если упавший прогон закоммитил, скажем, 60\% строк до падения, то в витрине окажется:

$$
GGR_{\text{витрина}} = 0.6 \times \$80\,000 + 1.0 \times \$80\,000 = \$48\,000 + \$80\,000 = \$128\,000.
$$

То есть витрина показывает $128\,000 вместо $80\,000 — завышение на $48\,000, или $+60\%$ «роста из воздуха». Аналитик утром видит всплеск, маркетинг рапортует об успехе кампании, финансы строят прогноз на завышенной базе.

Теперь то же самое с delete-insert. Перезапуск сначала выполняет `DELETE WHERE dt = '2026-06-15'` (сносит 60\% частичных строк), затем `INSERT` полного дня:

$$
GGR_{\text{витрина}} = 0 + \$80\,000 = \$80\,000.
$$

Результат корректен независимо от того, сколько раз таск перезапускался. Это и есть практическая ценность идемпотентности: цена ошибки в этом примере — $48\,000 искажения в одной метрике за один день, и такие ошибки имеют свойство копиться незаметно.

#### Пограничные случаи

- **Идемпотентность ключа, а не операции.** `MERGE`/`UPSERT` идемпотентен только если ключ дедупликации выбран корректно. Если ключ неуникален (например, мержим по `user_id` там, где у юзера несколько ставок в день), merge либо потеряет строки, либо всё равно задвоит.
- **Партиция должна совпадать с гранулярностью перезапуска.** Если данные партиционированы по месяцу, а перезапускаем по дню, то delete по дню внутри месячной партиции может быть дорогим или невозможным full-scan'ом.
- **Backfill и сторонние эффекты.** Если пайплайн при расчёте дёргает внешний API, шлёт нотификации или пишет в очередь, backfill повторит эти side effects по всей истории. Идемпотентна должна быть запись в витрину, но не обязательно безопасны побочные действия — их при backfill надо отключать.
- **Late-arriving data.** События могут прийти с опозданием на несколько дней. Если витрина за вчера уже «закрыта» и не пересчитывается, опоздавшие данные потеряются. Идемпотентный delete-insert с окном пересчёта (rolling window) решает это.

!!! warning "Типичная ошибка"

    Новичок пишет загрузку через `INSERT INTO ... SELECT ...` (голый append), тестирует на одном чистом прогоне, видит правильные числа и катит в прод. Проблема всплывает позже: при первом же ретрае упавшего таска или при ручном перезапуске дня витрина тихо задваивает данные. Симптом — необъяснимый «рост» метрики ровно в дни, когда были инциденты с пайплайном. Правило: любая загрузка в витрину должна быть идемпотентной по умолчанию (delete-insert или merge по партиции), даже если «сейчас ретраев нет» — они появятся.

!!! info "Современный data stack: общая схема"

    Источники (продуктовая OLTP-база, Kafka, внешние API)
    → загрузка (Fivetran / Airbyte / свои продюсеры)
    → хранилище-сырьё (Iceberg/Parquet на S3, или Snowflake/BigQuery)
    → трансформации (dbt: staging → intermediate → marts)
    → оркестрация всего этого (Dagster / Airflow / Prefect)
    → потребление (BI-дашборды, DuckDB-аналитика, ML feature store)
    → поверх всего: качество данных, каталог/lineage, мониторинг.

!!! question "Проверь себя"

    1. По каким осям измеряют качество данных и приведи пример проверки для депозита.
    2. Что такое идемпотентность пайплайна и через какой паттерн она достигается?
    3. Почему backfill возможен только при идемпотентности, параметризации датой и сохранённом сырье?
    4. Почему `append` без удаления партиции — главный источник задвоений?

??? success "Ответы"

    1. Completeness (нет дыр), validity (значения в границах — депозит не отрицательный), consistency (витрина сходится с сырьём), timeliness (свежесть), uniqueness (нет дублей ставок); пример — проверка, что депозит не меньше нуля.
    2. Свойство давать тот же результат при повторном запуске ($f(f(x)) = f(x)$); достигается паттерном delete-insert / merge по партиции (перезапуск за дату сначала удаляет данные партиции, потом вставляет заново).
    3. Backfill — пересчёт истории по новой/исправленной логике: нужно из чего пересчитывать (сырьё, привет ELT), безопасный повторный прогон (идемпотентность) и возможность гнать по конкретной дате (параметризация).
    4. При повторном запуске дня append добавляет те же строки ещё раз, и GGR в витрине задваивается — появляется «рост», которого нет.

!!! example "Мини-задание"

    Возьми загрузку дневного GGR, написанную как `INSERT INTO mart_ggr SELECT ... FROM staging WHERE dt = '{{ ds }}'`. За 2-5 минут: (1) объясни, что произойдёт с витриной, если Airflow перезапустит этот таск после частичного падения; (2) перепиши загрузку так, чтобы она стала идемпотентной по партиции `dt`; (3) назови один side effect, который нельзя повторять при backfill этого пайплайна по году истории.
