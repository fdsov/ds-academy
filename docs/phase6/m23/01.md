# M23.1 · Зачем этот модуль

Представь, что ты обучил блестящую churn-модель на выгрузке из CSV, которую тебе скинул коллега. Метрики прекрасны, презентация готова. А через месяц выясняется: выгрузка была сделана только по одному гео, депозиты считались в копейках, а не в рублях, и треть событий ставок потерялась, потому что Kafka-консьюмер падал по ночам. Модель в проде выдаёт мусор. Виноват не алгоритм — виноват **пайплайн данных**, которого ты не видел и не контролировал.

Data engineering — это слой, на котором стоит вся аналитика и весь ML. Если он кривой, то любая твоя регрессия, любой A/B-тест и любой дашборд по GGR будут считать неправильно — но молча, без ошибок, что хуже всего. Data scientist, не понимающий, откуда берутся данные, как они трансформируются и где теряются, обречён доверять чужим выгрузкам и расследовать «аномалии», которые на деле — баги ETL.

Этот модуль даёт тебе инженерный фундамент: ты научишься читать архитектуру данных компании, строить воспроизводимые пайплайны, понимать, почему витрина по ставкам считается именно так, и спорить с инженерами на их языке. Без этого знания ты остаёшься потребителем данных. С ним — становишься человеком, который может сам поднять источник истины.
