# M23.8 · dbt как стандарт трансформаций

**На пальцах.** dbt — это инструмент, который позволяет описывать трансформации данных обычными SELECT-запросами, но добавляет к ним то, что делает инженер: версионирование в git, зависимости между моделями, автотесты на данные и автодокументацию. Ты пишешь SQL, а dbt превращает его в управляемый, тестируемый, документированный слой трансформаций.

**Формально.** dbt (data build tool) реализует слой **T** в ELT. Основные сущности:

- **Модель (model)** — один `.sql` файл с `SELECT`. dbt оборачивает его в `CREATE TABLE/VIEW` и материализует в хранилище. Модели ссылаются друг на друга через `ref()`, из чего dbt строит DAG автоматически.
- **Тесты (tests)** — декларативные проверки данных: `unique`, `not_null`, `accepted_values`, `relationships`, плюс кастомные SQL-тесты. Прогоняются в пайплайне; падение теста ловит проблему до того, как её увидит аналитик.
- **Документация и lineage** — dbt генерирует сайт с описанием моделей, колонок и **графом происхождения** (откуда что берётся). Бесценно, когда нужно понять, как считается GGR.
- **Слои моделирования**: `staging` (1:1 чистка сырья), `intermediate` (промежуточная логика), `marts` (бизнес-витрины: depositов, retention, GGR).

!!! note "dbt в 2026: Fusion, Core v2.0, слияние с Fivetran"

    Крупные сдвиги 2026 года: dbt Labs выпустила **dbt Fusion** — новый движок на Rust (до 30x быстрее парсинга/компиляции, column-level lineage из коробки) и **dbt Core v2.0** (1 июня 2026). В тот же день закрылось **слияние dbt Labs и Fivetran**, объединившее загрузку (E/L) и трансформацию (T) под одной крышей. Суть для тебя не меняется: dbt остаётся стандартом слоя трансформаций — обновились движок и владелец.

```python
# models/staging/stg_bets.sql
with raw as (
    select * from {{ source('events', 'bets_raw') }}
)
select
    bet_id,
    player_id,
    cast(amount_cents as bigint) / 100.0 as bet_amount,
    cast(payout_cents as bigint) / 100.0 as payout,
    geo,
    cast(event_ts as timestamp) as bet_ts
from raw
where bet_id is not null
```

```python
# models/marts/fct_ggr_daily.sql
with bets as (
    select * from {{ ref('stg_bets') }}
)
select
    date_trunc('day', bet_ts) as bet_date,
    geo,
    sum(bet_amount) as total_bets,
    sum(payout) as total_payout,
    sum(bet_amount - payout) as ggr   -- GGR = ставки минус выплаты
from bets
group by 1, 2
```

```yaml
# models/marts/schema.yml — тесты и документация
models:
  - name: fct_ggr_daily
    description: "Дневной GGR по гео из сырых событий ставок"
    columns:
      - name: ggr
        description: "Gross Gaming Revenue = сумма ставок минус выплаты"
        tests:
          - not_null
      - name: geo
        tests:
          - not_null
          - accepted_values:
              values: ['RU', 'UZ', 'KZ', 'BR']
```

!!! question "Проверь себя"

    1. Что такое модель в dbt и как dbt строит DAG между моделями?
    2. Зачем нужны dbt-тесты и что они ловят?
    3. За что отвечают слои `staging`, `intermediate`, `marts`?

??? success "Ответы"

    1. Модель — один `.sql` файл с `SELECT`, который dbt оборачивает в `CREATE TABLE/VIEW`; модели ссылаются друг на друга через `ref()`, и из этих ссылок dbt автоматически строит DAG зависимостей.
    2. Декларативные проверки данных (`unique`, `not_null`, `accepted_values`, `relationships`, кастомные SQL); ловят проблему в пайплайне до того, как её увидит аналитик.
    3. `staging` — 1:1 чистка сырья, `intermediate` — промежуточная логика, `marts` — бизнес-витрины (GGR, retention, депозиты).
