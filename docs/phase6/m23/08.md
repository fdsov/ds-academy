# M23.8 · dbt как стандарт трансформаций

### Интуиция

Представь, что вся бизнес-логика аналитики живёт в десятках SQL-скриптов, разбросанных по ноутбукам, кронам и чьим-то локальным папкам. GGR в одном отчёте считается "ставки минус выплаты", в другом — "ставки минус выплаты минус бонусы". Никто не знает, какой вариант правильный, и при увольнении автора знание теряется. Это типичная боль аналитики без дисциплины.

dbt решает её простой идеей: трансформации данных — это код, и к нему нужно относиться как к коду. Ты по-прежнему пишешь обычные `SELECT`-запросы, но dbt добавляет вокруг них всё, что инженер ожидает от софта: версионирование в git, явные зависимости между шагами, автотесты на сами данные, автодокументацию и граф происхождения. Главное слово здесь — стандарт. dbt не делает ничего, что нельзя сделать руками, но он делает это одинаково для всей команды, воспроизводимо и проверяемо.

Житейская картинка: до dbt трансформации — это куча рукописных рецептов на салфетках, где у каждого повара свой "секретный" GGR. После dbt — это единая кулинарная книга в git, где каждый рецепт ссылается на ингредиенты из предыдущего, и на входе стоит дегустатор (тесты), который не пускает в зал блюдо с протухшими данными.

### Формально

dbt (data build tool) реализует слой **T** (Transform) в архитектуре **ELT**: сырьё сначала грузится в хранилище как есть (E + L), а трансформации выполняются уже внутри хранилища силами его движка (BigQuery, Snowflake, Postgres, DuckDB и т.д.). dbt сам не хранит и не двигает данные — он компилирует твой код в SQL и отдаёт его хранилищу на исполнение. Основные сущности:

- **Модель (model)** — один `.sql` файл с `SELECT`. dbt оборачивает его в `CREATE TABLE` или `CREATE VIEW` и материализует в хранилище. Модели ссылаются друг на друга через функцию `ref()`, из чего dbt автоматически строит ориентированный ациклический граф зависимостей (DAG) и определяет порядок выполнения.
- **Источники (sources)** — декларация сырых таблиц через `source()`. Позволяет тестировать свежесть сырья и отделять "вход" пайплайна от моделей.
- **Тесты (tests)** — декларативные проверки данных: `unique`, `not_null`, `accepted_values`, `relationships`, плюс кастомные SQL-тесты и пакеты вроде `dbt_utils`. Прогоняются командой `dbt test`; падение теста ловит проблему до того, как её увидит аналитик или дашборд.
- **Документация и lineage** — `dbt docs generate` строит сайт с описанием моделей, колонок и **графом происхождения** (lineage): откуда что берётся и что сломается выше по течению при изменении. Бесценно, когда нужно понять, как именно считается GGR.
- **Слои моделирования** (соглашение, не жёсткое правило): `staging` (1:1 чистка и переименование сырья), `intermediate` (промежуточная переиспользуемая логика), `marts` (бизнес-витрины: депозиты, retention, GGR, ARPU).
- **Материализации**: `view` (логика на лету), `table` (полный пересчёт), `incremental` (досчёт только новых строк — критично для больших event-таблиц вроде ставок), `ephemeral` (инлайнится как CTE, без объекта в БД).

Формально dbt отделяет *описание* трансформации (декларативный SQL + конфиг) от *исполнения* (скомпилированный SQL в движке хранилища). Это даёт воспроизводимость: один и тот же `dbt run` на одном коммите git даёт один и тот же набор объектов в БД.

!!! note "dbt в 2026: Fusion, Core v2.0, слияние с Fivetran"

    Крупные сдвиги 2026 года: dbt Labs выпустила **dbt Fusion** — новый движок на Rust (значительно быстрее парсинга и компиляции, column-level lineage из коробки, статический анализ SQL до запуска) и **dbt Core v2.0**. В том же периоде закрылось **слияние dbt Labs и Fivetran**, объединившее загрузку (E/L) и трансформацию (T) под одной крышей. Суть для тебя не меняется: dbt остаётся стандартом слоя трансформаций — обновились движок и владелец, а парадигма "SQL как версионируемый тестируемый код" та же.

```python
# models/staging/stg_bets.sql
with raw as (
    select * from {{ source('events', 'bets_raw') }}
)
select
    bet_id,
    player_id,
    cast(amount_cents as bigint) / 100.0 as bet_amount,
    cast(payout_cents as bigint) / 100.0 as payout,
    geo,
    cast(event_ts as timestamp) as bet_ts
from raw
where bet_id is not null
```

```python
# models/marts/fct_ggr_daily.sql
with bets as (
    select * from {{ ref('stg_bets') }}
)
select
    date_trunc('day', bet_ts) as bet_date,
    geo,
    sum(bet_amount) as total_bets,
    sum(payout) as total_payout,
    sum(bet_amount - payout) as ggr   -- GGR = ставки минус выплаты
from bets
group by 1, 2
```

```yaml
# models/marts/schema.yml — тесты и документация
models:
  - name: fct_ggr_daily
    description: "Дневной GGR по гео из сырых событий ставок"
    columns:
      - name: ggr
        description: "Gross Gaming Revenue = сумма ставок минус выплаты"
        tests:
          - not_null
      - name: geo
        tests:
          - not_null
          - accepted_values:
              values: ['RU', 'UZ', 'KZ', 'BR']
```

#### Разбор на пальцах: сколько стоит один пропущенный тест

Покажем в деньгах, зачем нужны декларативные тесты. Допустим, в сырьё `bets_raw` из-за бага в трекинге часть строк прилетает с `payout_cents`, ошибочно записанным в долларах, а не в центах. То есть для этих строк выплата завышена в 100 раз.

Возьмём конкретные числа за день по гео RU:

- честные ставки: $\text{total\_bets} = \$200\,000$;
- честные выплаты: $\text{total\_payout} = \$190\,000$;
- истинный дневной GGR: $200\,000 - 190\,000 = \$10\,000$.

Теперь предположим, что 0.5\% строк (битые) дали суммарную честную выплату $\$950$, но из-за бага она посчиталась как $950 \times 100 = \$95\,000$. Тогда:

$$
\text{payout}_{\text{битый}} = 190\,000 - 950 + 95\,000 = \$284\,050,
$$

$$
\text{GGR}_{\text{битый}} = 200\,000 - 284\,050 = -\$84\,050.
$$

Проверим арифметику: честная выплата без битых строк $190\,000 - 950 = 189\,050$; добавляем раздутую $189\,050 + 95\,000 = 284\,050$. Ставки не тронуты, $200\,000 - 284\,050 = -84\,050$. Дневной GGR из $+\$10\,000$ превратился в $-\$84\,050$ — расхождение почти $\$94\,000$ за один день из-за 0.5\% грязных строк.

Что ловит тест. Если бы в `schema.yml` стоял кастомный тест "`payout` не может превышать, скажем, 50-кратную ставку по строке" или `dbt_utils.accepted_range` на разумный диапазон выплат, `dbt test` упал бы в пайплайне, и битые данные не дошли бы до витрины GGR и до дашборда финансов. Стоимость теста — несколько строк YAML. Стоимость его отсутствия — финдиректор видит убыточный день, останавливает закупку трафика и теряет дневной приток депозитов. Тест на данные здесь дешевле инцидента на несколько порядков.

#### Пограничные случаи и частые заблуждения

- **dbt не оркестратор.** Сам по себе он не запускается по расписанию и не следит за внешними зависимостями. Его запускает Airflow, Dagster, dbt Cloud scheduler или GitHub Actions. dbt отвечает за порядок моделей *внутри* проекта (DAG из `ref`), но не за "когда вообще запускать".
- **dbt не грузит данные.** Это слой T, а не E/L. Загрузку делают Fivetran, Airbyte, кастомные скрипты. Распространённая путаница после слияния с Fivetran: продукт стал шире, но движок dbt по-прежнему только трансформирует.
- **`ref()` обязателен между моделями.** Если в `fct_ggr_daily` написать `from stg_bets` напрямую вместо `{{ ref('stg_bets') }}`, dbt не увидит зависимость, сломает порядок выполнения и lineage, и модель может выполниться раньше, чем построится upstream.
- **Тесты проверяют данные, а не SQL.** Зелёные тесты не гарантируют, что формула GGR бизнес-корректна — только что в колонке нет null или дублей. Логическую корректность даёт ревью кода и доменная экспертиза, а не `dbt test`.
- **`incremental` требует осторожности.** Неправильный `unique_key` или фильтр окна приводит к дублям или потере поздно прилетевших событий (late-arriving data) — частая боль на event-таблицах ставок.

!!! warning "Типичная ошибка"

    Новичок пишет в модели `from analytics.stg_bets` (хардкод имени таблицы) вместо `from {{ ref('stg_bets') }}`. SQL вроде бы работает локально, но dbt при этом не строит ребро зависимости в DAG. В результате `fct_ggr_daily` может запуститься раньше, чем пересоберётся `stg_bets`, и витрина посчитается на вчерашних или вовсе несуществующих данных. Правило простое: между своими моделями — всегда `ref()`, к сырью — всегда `source()`, и никогда не обращайся к таблице по сырому имени.

!!! question "Проверь себя"

    1. Что такое модель в dbt и как dbt строит DAG между моделями?
    2. Зачем нужны dbt-тесты и что именно они ловят (а что не ловят)?
    3. За что отвечают слои `staging`, `intermediate`, `marts`?
    4. dbt относится к E, L или T в ELT, и кто отвечает за остальные буквы и за расписание запусков?

??? success "Ответы"

    1. Модель — один `.sql` файл с `SELECT`, который dbt оборачивает в `CREATE TABLE/VIEW`; модели ссылаются друг на друга через `ref()`, и из этих ссылок dbt автоматически строит DAG зависимостей и порядок выполнения.
    2. Декларативные проверки данных (`unique`, `not_null`, `accepted_values`, `relationships`, кастомные SQL); ловят проблемы данных (null, дубли, недопустимые значения) в пайплайне до того, как их увидит аналитик. Не ловят логические ошибки в формулах — это задача ревью и домена.
    3. `staging` — 1:1 чистка и приведение сырья, `intermediate` — промежуточная переиспользуемая логика, `marts` — бизнес-витрины (GGR, retention, депозиты, ARPU).
    4. dbt — это слой **T** (Transform). Загрузку (E/L) делают Fivetran/Airbyte/скрипты, а расписанием запусков управляет оркестратор (Airflow, Dagster, dbt Cloud, CI). Сам dbt порядок задаёт только внутри проекта через DAG из `ref`.
