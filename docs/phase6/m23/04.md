# M23.4 · Источники и хранилища: OLTP против OLAP

**На пальцах.** Есть два мира. Первый — это база, которая обслуживает живой продукт: игрок нажал «поставить», транзакция записалась, баланс уменьшился. Тут важна скорость одной маленькой операции и её надёжность. Второй мир — это аналитика: «посчитай суммарный GGR по всем игрокам за квартал в разрезе гео». Тут одна операция читает миллионы строк. Это разные задачи, и для них нужны разные базы.

**Формально.**

- **OLTP (Online Transaction Processing)** — системы оперативных транзакций. Оптимизированы под множество мелких операций чтения/записи отдельных строк (insert/update по ключу), строки хранятся целиком (row-oriented), сильная нормализация. Примеры: PostgreSQL, MySQL — это операционная БД продукта, где живут депозиты, ставки, сессии в реальном времени.
- **OLAP (Online Analytical Processing)** — системы аналитической обработки. Оптимизированы под агрегации по большому числу строк и небольшому числу колонок. Хранение колоночное (column-oriented), что позволяет читать только нужные столбцы и сильно сжимать данные. Примеры: ClickHouse, BigQuery, Snowflake, DuckDB.

!!! info "Почему колоночное хранение быстрее для аналитики"

    Запрос `SELECT geo, SUM(ggr) FROM bets GROUP BY geo` по таблице из 200 колонок в row-store вынужден читать все 200 колонок каждой строки. В column-store он читает только `geo` и `ggr` — два столбца. Плюс одинаковые значения в столбце лежат рядом и сжимаются в разы лучше. Отсюда выигрыш на порядки по объёму чтения и стоимости.

Ключевое правило: **не гоняй тяжёлую аналитику по продакшн-OLTP**. Тяжёлый `GROUP BY` по таблице ставок может положить базу, обслуживающую живых игроков. Данные сначала переливают в аналитическое хранилище.

!!! question "Проверь себя"

    1. Под какой тип нагрузки оптимизирован OLTP, а под какой — OLAP?
    2. Почему колоночное хранение быстрее для запроса `SELECT geo, SUM(ggr) ... GROUP BY geo`?
    3. Почему нельзя гонять тяжёлый `GROUP BY` по продакшн-OLTP-базе?

??? success "Ответы"

    1. OLTP — множество мелких операций чтения/записи отдельных строк по ключу (живой продукт); OLAP — агрегации по большому числу строк и небольшому числу колонок (аналитика).
    2. Column-store читает только нужные столбцы (`geo`, `ggr`), а не все 200 колонок строки, плюс одинаковые значения в столбце лежат рядом и сильно сжимаются — выигрыш на порядки по объёму чтения.
    3. Такой запрос читает миллионы строк и может положить базу, обслуживающую живых игроков; аналитику делают на отдельной OLAP-копии.

!!! example "Мини-задание"

    Выпиши 5 запросов к гемблинг-данным и реши, OLTP это или OLAP: (а) списать ставку и уменьшить баланс игрока; (б) средний депозит по гео за квартал; (в) показать баланс одного игрока по его id; (г) топ-100 игроков по GGR за год; (д) записать новое событие регистрации. Для каждого объясни, по какому признаку отнёс (одна строка по ключу vs агрегация по многим строкам). Займёт 3-4 минуты.
