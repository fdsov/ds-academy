# M23.13 · Код: стек 2026

Сквозной мини-пайплайн: сырые события ставок → проверка качества → витрина GGR. Идемпотентно, на DuckDB, с параметром даты для backfill.

```python
import duckdb
import sys

def build_ggr_mart(run_date: str, db_path: str = "warehouse.duckdb"):
    con = duckdb.connect(db_path)

    # 1. Загрузка сырья за конкретную дату (партиция)
    con.execute(f"""
        create or replace temp table raw_bets as
        select * from read_parquet(
            's3://lake/bets/bet_date={run_date}/**/*.parquet'
        )
    """)

    # 2. Проверки качества — падаем громко до записи в витрину
    nulls = con.execute("""
        select count(*) from raw_bets
        where bet_id is null or player_id is null or amount_cents is null
    """).fetchone()[0]
    if nulls > 0:
        raise ValueError(f"DQ fail: {nulls} строк с NULL в ключевых полях за {run_date}")

    negatives = con.execute(
        "select count(*) from raw_bets where amount_cents < 0"
    ).fetchone()[0]
    if negatives > 0:
        raise ValueError(f"DQ fail: {negatives} отрицательных ставок за {run_date}")

    # 3. Идемпотентность: сносим партицию за дату перед вставкой
    con.execute("""
        create table if not exists fct_ggr_daily (
            bet_date date, geo varchar, total_bets double,
            total_payout double, ggr double, players bigint
        )
    """)
    con.execute("delete from fct_ggr_daily where bet_date = ?", [run_date])

    # 4. Трансформация и запись витрины
    con.execute("""
        insert into fct_ggr_daily
        select
            cast(event_ts as date) as bet_date,
            geo,
            sum(amount_cents) / 100.0 as total_bets,
            sum(payout_cents) / 100.0 as total_payout,
            sum(amount_cents - payout_cents) / 100.0 as ggr,
            count(distinct player_id) as players
        from raw_bets
        group by 1, 2
    """)
    con.close()

if __name__ == "__main__":
    # Запуск за дату; цикл по датам = backfill
    build_ggr_mart(sys.argv[1])
```

```python
# Оркестрация на Dagster (asset-центрично) — для prod вместо ручного запуска
from dagster import asset, AssetExecutionContext

@asset
def ggr_daily_mart(context: AssetExecutionContext):
    run_date = context.partition_key  # дата из дневной партиции Dagster
    build_ggr_mart(run_date)
    context.log.info(f"GGR-витрина построена за {run_date}")
```

!!! tip "AI-копилот: где нейросети помогают"

    LLM (Claude в Cursor, Copilot) сильны в data engineering на рутине: сгенерировать dbt-модель из описания витрины, написать boilerplate теста, перевести SQL между диалектами (Postgres → DuckDB → BigQuery), накидать Dagster/Airflow DAG по структуре, объяснить чужой запутанный запрос.

    Пример промпта: «Вот сырая таблица событий ставок (схема ниже). Напиши dbt staging-модель с приведением типов и slug-чисткой, mart-модель дневного GGR по гео, и schema.yml с тестами not_null, accepted_values по гео и кастомным тестом, что ggr не отрицателен. Диалект — DuckDB.»

    Где врут и что перепроверять: (1) **бизнес-логику метрик** — LLM не знает, что в твоей компании GGR считается за вычетом бонусов или что депозиты в копейках; формулу всегда сверяй с метрик-каталогом. (2) **Идемпотентность** — модели часто генерят `INSERT`/append без delete партиции, что даёт задвоение при перезапуске; требуй явно merge/delete-insert. (3) **Диалектные функции** — `date_trunc`, работа с временными зонами и JSON различаются между движками; гоняй на реальных данных. (4) Качество и edge-cases (NULL, дубли, поздние события) LLM почти всегда упускает — добавляй сам.

!!! note "Стек 2026"

    Актуально: ELT поверх lakehouse, **Apache Iceberg** как ведущий (но не единственный) открытый формат, разделение storage/compute. **dbt** — индустриальный стандарт трансформаций. **DuckDB** для small/medium data вместо Spark-кластеров. **Dagster** набирает долю у новых команд за asset-центричность и нативный dbt. Загрузка — Fivetran/Airbyte или managed-коннекторы.

    Устарело относительно старых курсов: ETL «трансформируй до загрузки» как дефолт; Hadoop/MapReduce как база (Spark и его вытеснил, а DuckDB теснит сам Spark на средних данных); «любая аналитика = поднимаем кластер»; ручные cron-скрипты без оркестрации и lineage; data lake без табличного слоя (превращается в болото).

!!! warning "Ловушки"

    - **Тяжёлая аналитика по продакшн-OLTP.** `GROUP BY` по таблице ставок на живой базе кладёт продукт. Аналитику — только в OLAP-копии.
    - **Append без идемпотентности.** Перезапуск дня задваивает GGR, и ты «находишь рост», которого нет. Всегда delete-insert / merge по партиции.
    - **Нет сырья — нет backfill.** Если хранишь только агрегаты (ETL-наследие), пересчитать историю по новой логике невозможно. Храни сырьё.
    - **Small files problem.** Переусердствовал с партиционированием — миллионы мелких файлов, и чтение метаданных съедает весь выигрыш.
    - **Молчаливая деградация качества.** Источник тихо переименовал поле или сменил единицы (рубли → копейки), тестов нет — модель считает мусор без единой ошибки. Тесты на свежесть, диапазоны и схему обязательны.
    - **Рассинхрон офлайн/онлайн фич (training-serving skew).** Фича в обучении считается одним кодом, в проде — другим. Модель деградирует. На этом горят сеньоры. Лекарство — единый источник определения фич.
    - **Преждевременный Spark/Kafka.** Кластер и стриминг там, где хватило бы DuckDB и батча — дорого, хрупко, медленно в разработке.
