# M23.14 · Практика

### Быстрый квиз

Выбери вариант — ответ подсветится сразу.

```quiz
Чем OLTP отличается от OLAP по типу нагрузки?
[x] OLTP — много мелких операций по ключу, OLAP — агрегации по множеству строк
[ ] OLTP — аналитика, OLAP — живые транзакции продукта
[ ] OLTP колоночный, OLAP строковый
[ ] Между ними нет разницы, это синонимы
> OLTP оптимизирован под множество мелких чтений/записей отдельных строк, OLAP — под агрегации по большому числу строк и малому числу колонок.
---
Почему индустрия сдвинулась от ETL к ELT?
[ ] Потому что SQL запретили использовать на серверах трансформации
[x] Хранилище и вычисления стали дёшевы, выгоднее грузить сырьё и трансформировать внутри
[ ] Потому что ELT не требует сохранять сырые данные
[ ] Потому что ELT работает без облачных хранилищ
> Дешёвые и масштабируемые облачные хранилища сделали выгодным грузить сырьё как есть и трансформировать его SQL внутри: воспроизводимость, скорость, версионирование в git.
---
Что lakehouse добавляет поверх файлов в data lake?
[ ] Только более дешёвое объектное хранилище
[ ] Обязательную схему, заданную при записи, как в warehouse
[x] Транзакционный табличный слой: ACID, версионирование, time travel
[ ] Запрет на хранение сырых файлов
> Lakehouse через открытые форматы (Iceberg, Delta Lake, Hudi) добавляет ACID-транзакции, версионирование, эволюцию схемы и time travel поверх файлов в озере.
---
Что такое модель в dbt и как строится DAG между моделями?
[x] Один .sql файл с SELECT; DAG строится автоматически из ссылок ref()
[ ] Python-класс, а зависимости прописываются вручную в конфиге
[ ] YAML-файл с описанием таблицы без SQL
[ ] Готовая витрина, которую dbt только материализует без трансформаций
> Модель — это .sql файл с SELECT, который dbt оборачивает в CREATE TABLE/VIEW; ссылки через ref() дают dbt граф зависимостей автоматически.
---
Зачем нужна идемпотентность пайплайна и как она достигается?
[ ] Чтобы пайплайн запускался быстрее за счёт кеша
[x] Чтобы повторный запуск давал тот же результат; через delete-insert/merge по партиции
[ ] Чтобы можно было удалить сырьё после загрузки
[ ] Чтобы append добавлял строки без проверок
> Идемпотентность — тот же результат при повторном запуске; достигается delete-insert/merge по партиции, иначе повторный прогон дня задваивает GGR.
---
Когда оправдан распределённый движок (Spark) вместо DuckDB?
[ ] Всегда, когда данных больше нескольких гигабайт
[ ] Когда нужно читать Parquet с S3
[x] Когда данные реально не влезают на одну машину (терабайты) или нужен распределённый ML
[ ] Когда нужно версионировать трансформации в git
> Правило 2026: сначала спроси «влезает ли в DuckDB на одной машине?». Spark берут только когда данные не влезают или нужен распределённый ML; преждевременный Spark — дорогая ошибка.
---
Почему преждевременный стриминг на Kafka — частая ошибка?
[ ] Kafka не умеет хранить события для повторного чтения
[ ] Батч в принципе не справляется с продуктовой аналитикой
[x] Стриминг дороже и сложнее в отладке, а батча хватает для ~80% задач
[ ] Kafka работает только с OLTP-базами
> Стриминг оправдан там, где задержка стоит денег (антифрод, риск, real-time персонализация); для большинства витрин дневных/часовых обновлений достаточно — не строй Kafka «потому что модно».
```

**Разминка.** Возьми CSV с событиями ставок (или сгенерируй синтетику: bet_id, player_id, amount_cents, payout_cents, geo, event_ts). Сконвертируй в Parquet с партиционированием по дате через DuckDB. Сравни размер файлов и время запроса `SUM(ggr) WHERE bet_date = X` для CSV и партиционированного Parquet.

**Основное.** Подними dbt-проект на DuckDB. Сделай слои: `stg_bets` (чистка типов, отсев NULL), `fct_ggr_daily` (витрина GGR по дню и гео), `dim_players` (первый депозит, гео). Добавь тесты: `not_null`, `unique` на bet_id, `accepted_values` на гео, кастомный тест «ggr не NULL и payout не больше суммы ставок на разумном горизонте». Сгенерируй dbt docs и посмотри граф lineage.

**Челлендж.** Сделай скрипт пайплайна идемпотентным и параметризованным датой (как в коде выше). Прогони backfill за 30 дней циклом. Затем намеренно запусти один день дважды и докажи запросом, что GGR не задвоился. Добавь «битый» день (отрицательные ставки) и убедись, что DQ-проверка падает и не пускает мусор в витрину.

**Мини-проект.** Собери end-to-end пайплайн: генератор синтетических событий ставок → Parquet-партиции в локальном «lake» → dbt-трансформации на DuckDB (staging → marts: GGR, retention D1/D7, средний депозит) → оркестрация в Dagster с дневными партициями и backfill → набор dbt-тестов качества → README с диаграммой data flow и описанием, как считается каждая метрика витрины. Цель — чтобы другой человек мог по твоему репозиторию пересчитать любую дату с нуля.
