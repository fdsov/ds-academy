Обучение с трекингом и регистрацией в MLflow:

```python
import mlflow
import mlflow.sklearn
import joblib
from sklearn.ensemble import HistGradientBoostingClassifier
from sklearn.model_selection import train_test_split
from sklearn.metrics import roc_auc_score

mlflow.set_tracking_uri("http://localhost:5000")
mlflow.set_experiment("churn")

X_train, X_val, y_train, y_val = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)

with mlflow.start_run():
    params = {"max_iter": 300, "learning_rate": 0.05, "max_depth": 6}
    model = HistGradientBoostingClassifier(**params, random_state=42)
    model.fit(X_train, y_train)

    auc = roc_auc_score(y_val, model.predict_proba(X_val)[:, 1])

    mlflow.log_params(params)
    mlflow.log_metric("val_auc", auc)
    # registered_model_name создаёт/обновляет запись в Model Registry
    mlflow.sklearn.log_model(model, "model", registered_model_name="churn")
    joblib.dump(model, "churn_model.joblib")
```

Online-сервинг на FastAPI (стандарт 2026):

```python
from fastapi import FastAPI
from pydantic import BaseModel
import joblib
import time

app = FastAPI(title="Churn API")
model = joblib.load("churn_model.joblib")

class Player(BaseModel):
    deposits_30d: float
    bets_30d: int
    avg_bet: float
    days_since_last_session: int

@app.post("/predict")
def predict(p: Player):
    start = time.perf_counter()
    features = [[p.deposits_30d, p.bets_30d, p.avg_bet, p.days_since_last_session]]
    proba = float(model.predict_proba(features)[0, 1])
    latency_ms = (time.perf_counter() - start) * 1000
    # latency_ms и proba уходят в систему мониторинга (Prometheus/логи)
    return {"churn_proba": proba, "latency_ms": round(latency_ms, 2)}
```

Детекция дрейфа через PSI:

```python
import numpy as np

def psi(expected, actual, bins=10):
    # бины фиксируем по обучающему распределению
    edges = np.quantile(expected, np.linspace(0, 1, bins + 1))
    edges[0], edges[-1] = -np.inf, np.inf

    e = np.histogram(expected, edges)[0] / len(expected)
    a = np.histogram(actual, edges)[0] / len(actual)
    # сглаживание, чтобы избежать деления на ноль и log(0)
    e, a = np.clip(e, 1e-6, None), np.clip(a, 1e-6, None)
    return float(np.sum((a - e) * np.log(a / e)))

drift = psi(X_train[:, 0], X_prod[:, 0])

```

В продакшене дрейф обычно считают не руками, а через специализированные библиотеки (Evidently, NannyML) — они дают готовые отчёты и интеграцию с мониторингом. Но понимать, что считается под капотом (PSI, KS, метрики качества с отложенными labels), обязательно.

!!! tip "AI-копилот: где нейросети помогают"

    LLM (Claude в Cursor, Copilot) сильны в рутине MLOps: сгенерировать каркас FastAPI-сервиса, Dockerfile, обвязку логирования в MLflow, заготовку Airflow-DAG для переобучения, функции статистических тестов дрейфа.

    Пример промпта: «Напиши FastAPI-эндпоинт для online-инференса joblib-модели оттока. Вход валидируй через Pydantic, добавь измерение латентности и структурированный лог каждого запроса в формате JSON с полями timestamp, features, proba, latency_ms. Объясни, где в этой схеме место для мониторинга дрейфа».

    Где врут и что перепроверять: LLM любят предлагать `pickle.load` на недоверенных данных (дыра в безопасности) и игнорировать совместимость версий библиотек — проверяй сам. Часто опускают перцентили латентности (дают среднее) и point-in-time correctness в признаках. Сгенерированный код мониторинга дрейфа нужно проверять на корректность статистики (правильные ли пороги, как обрабатываются пустые бины). И главное — LLM не знает твоего бизнес-контекста: какой порог дрейфа критичен и как часто переобучать — решаешь ты по данным, а не по дефолтам из ответа модели.

!!! warning "Ловушки"

    - Деплой без мониторинга. Модель тихо деградирует через дрейф, бизнес теряет деньги, никто не знает. Мониторинг — не опция, а часть деплоя.
    - Training-serving skew. Признаки в обучении и в проде считаются разным кодом. Предсказания портятся незаметно. Лечится единым источником признаков.
    - Средняя латентность вместо перцентилей. p99 в хвосте — реальная боль пользователей, среднее её прячет. На этом горят сеньоры.
    - pickle из недоверенного источника. Исполнение произвольного кода при загрузке. Плюс несовместимость версий библиотек — тихо неверные предсказания.
    - Data leakage через признаки без point-in-time. Признак «подглядывает» в будущее относительно момента события, на трейне отличный AUC, в проде провал.
    - Дрейф признака приняли за деградацию модели (или наоборот). Связывай мониторинг дрейфа с качеством и важностью признаков, не реагируй на дрейф неважных признаков.
    - «Дрейф», который на самом деле сломанный ETL. Перед переобучением проверь пайплайн данных: чаще виноват баг в источнике, а не изменившийся мир.
    - Автодеплой без gate-проверки качества. Continuous Training будет исправно выкатывать в прод модели хуже текущей. Всегда сравнивай с продовым бейзлайном перед промоутом.
    - Невозможность отката. Нет версионированного registry и иммутабельных артефактов — нет быстрого rollback. Беда обнаруживается ровно тогда, когда откат критически нужен.
