# M22.11 · Feature store: согласованность train/serve

### Интуиция: один источник признаков для обучения и прода

Представь: при обучении ты считал признак «средний депозит за 30 дней» одним SQL-запросом по историческим данным, а в проде разработчик пересчитал его другим кодом на лету. Даже мелкое расхождение в логике (как считать окно, что делать с пропусками) приводит к тому, что модель в проде видит не те числа, на которых училась. Предсказания тихо портятся. Это называется training-serving skew.

### Формально: feature store и его роль

Feature store — централизованный слой для определения, хранения и выдачи признаков с единой логикой для обучения и инференса. У него обычно два «лица»:

- Offline store — историческое хранилище признаков для обучения (большие объёмы, точечная выборка на момент времени).
- Online store — низколатентное хранилище (key-value) для выдачи признаков в online-инференсе за миллисекунды.

Главная ценность — согласованность train/serve: признак определён один раз, и обучение, и прод берут его из одного источника по одной логике. Это устраняет training-serving skew — расхождение между признаками на обучении и в продакшене, одну из самых коварных причин тихой деградации.

Дополнительные плюсы: переиспользование признаков между моделями, point-in-time correctness (выдача значений признака именно на момент события, без подглядывания в будущее — защита от data leakage), управление свежестью данных.

!!! warning "Feature store — не для всех с первого дня"

    Полноценный feature store (Feast и аналоги) — серьёзная инфраструктура. На раннем этапе с одной batch-моделью он избыточен — переусложнение убьёт скорость. Но как только появляется online-инференс и несколько моделей делят признаки, согласованность train/serve становится критичной, и feature store окупается. Решай по числу моделей и наличию online-сервинга, а не по моде.

!!! question "Проверь себя"

    1. Что такое training-serving skew и из-за чего он возникает?
    2. Как feature store устраняет эту проблему?
    3. Когда feature store ещё не нужен, а когда окупается?

??? success "Ответы"

    1. Расхождение признаков на обучении и в проде из-за того, что они считаются разным кодом или логикой; предсказания тихо портятся.
    2. Признак определён один раз, и обучение, и прод берут его из единого источника по одной логике — расхождение исчезает.
    3. На раннем этапе с одной batch-моделью он избыточен; окупается, когда появляется online-инференс и несколько моделей делят общие признаки.
