# M22.13 · Практика

### Быстрый квиз

Выбери вариант — ответ подсветится сразу.

```quiz
Почему ML-жизненный цикл в продакшене называют циклическим, а не линейным?
[ ] Потому что код приходится переписывать после каждого деплоя
[x] Потому что модель деградирует сама из-за изменения мира, и за деплоем следуют мониторинг и переобучение
[ ] Потому что обучение всегда нужно повторять минимум дважды
[ ] Потому что данные собираются по кругу из одних и тех же источников
> ML-система портится без правок кода — меняется распределение данных, поэтому цикл замыкается через мониторинг и переобучение.
---
Какие три оси изменчивости в ML-системе нужно версионировать отдельно?
[ ] Код, тесты и документацию
[x] Код, модель (веса) и данные (распределение входов)
[ ] Гиперпараметры, метрики и логи
[ ] Признаки, лейблы и предсказания
> В обычном софте меняется только код, а в ML независимо меняются код, веса модели и данные — иначе нельзя воспроизвести и откатить состояние.
---
Чем joblib предпочтительнее pickle для scikit-learn моделей?
[ ] Он безопасен и не исполняет произвольный код при загрузке
[ ] Он независим от версий Python и библиотек
[x] Он оптимизирован под большие numpy-массивы — сохраняет быстрее и компактнее
[ ] Он конвертирует модель в граф вычислений вне Python
> joblib — надстройка над pickle под большие массивы весов; ограничения по безопасности и совместимости версий у них общие.
---
Какой один вопрос помогает выбрать между batch- и online-сервингом?
[ ] Сколько признаков у модели?
[x] Сколько у бизнеса времени на ответ?
[ ] Какой фреймворк используется для обучения?
[ ] Сколько стоит инфраструктура?
> Секунды или миллисекунды по одному объекту — online (антифрод на депозите); раз в день по всей базе — batch (утренний список на бонус удержания).
---
Почему латентность мониторят по перцентилям p95/p99, а не по среднему?
[ ] Перцентили считаются быстрее среднего
[ ] Среднее нельзя посчитать в реальном времени
[x] Среднее маскирует хвост — p99 показывает, что каждый сотый ждёт почти секунду
[ ] Перцентили не зависят от выбросов
> Боль бизнеса именно в хвосте: при среднем 50 мс p99 может быть 800 мс, и среднее это прячет.
---
Чем data drift отличается от concept drift в терминах распределений?
[x] Data drift — меняется P(X) при неизменном P(y|X); concept drift — меняется само P(y|X)
[ ] Data drift меняет P(y|X), а concept drift меняет только P(X)
[ ] Оба означают одно и то же — изменение распределения входов
[ ] Data drift про лейблы, concept drift про латентность
> При data drift сдвигаются входы, при concept drift меняется сама связь вход-ответ (те же входы, другой правильный ответ).
---
Что в первую очередь проверить при резком скачке «дрейфа»?
[ ] Сразу запустить переобучение модели
[ ] Увеличить порог PSI, чтобы алерт не срабатывал
[x] Не сломан ли пайплайн данных — источник, схема, ETL
[ ] Откатить модель на предыдущую версию
> Очень часто «дрейф» оказывается багом в данных (сломанный ETL, изменившаяся схема), а не изменившимся миром.
```

Разминка:

1. Обучи любую табличную модель (можно churn на синтетике), сохрани её тремя способами: pickle, joblib, ONNX. Сравни размер файлов и время загрузки. Загрузи каждую и убедись, что предсказания совпадают.
2. Залогируй три прогона обучения с разными гиперпараметрами в MLflow. Найди лучший по val_auc через UI.

Основное:

3. Подними churn-модель как FastAPI-сервис с эндпоинтом `/predict`, валидацией входов через Pydantic и измерением латентности. Прогони 1000 запросов и посчитай p50/p95/p99 латентности — убедись, что среднее их прячет.
4. Упакуй сервис в Docker, собери образ, запусти контейнер, проверь, что эндпоинт отвечает. Зафиксируй версии зависимостей в lock-файле.
5. Зарегистрируй модель в MLflow Model Registry и присвой ей алиас `@champion`. Обучи вторую версию, переведи `@champion` на неё, затем сделай rollback на первую переназначением алиаса обратно.

Челлендж:

6. Реализуй детекцию дрейфа: возьми обучающее распределение как референс, искусственно сдвинь продовое (сымитируй маркетинговую кампанию — сдвинь распределение депозитов), посчитай PSI по каждому признаку. Настрой порог, при котором логируется алерт.
7. Сравни свою ручную реализацию PSI с отчётом Evidently на тех же данных. Объясни расхождения, если они есть.

Мини-проект:

Собери end-to-end MLOps-пайплайн churn-модели: версионирование данных через DVC, обучение с трекингом в MLflow и регистрацией в registry, сериализация, online-сервинг на FastAPI в Docker, и скрипт мониторинга дрейфа (PSI по ключевым признакам + распределение предсказаний) с алертом. Опиши в README, как делается переобучение и rollback. Главное требование: любой человек должен по твоему репозиторию воспроизвести продовую модель и откатить её на предыдущую версию.
