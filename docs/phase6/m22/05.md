# M22.5 · Сервинг: batch против online

### Интуиция: посылка по почте или звонок по телефону

Есть два способа отдавать предсказания. Batch — это как разослать всем игрокам письма раз в сутки: ночью прогоняешь всю базу через модель, складываешь результаты в таблицу, и продукт читает их утром. Online — это как телефонный звонок: запрос приходит прямо сейчас, и ответ нужен за десятки миллисекунд (например, показывать ли игроку удерживающий бонус в момент, когда он собирается уйти).

### Формально: два режима сервинга

Batch-инференс: предсказания считаются пакетом по расписанию для большого набора объектов, результаты сохраняются в хранилище (таблица, key-value store). Подходит, когда решение не требуется мгновенно: ежедневный скоринг риска оттока, сегментация игроков для рассылок, расчёт LTV-прогноза. Дёшево, просто, легко масштабируется. Минус — данные на момент чтения уже устарели (latency измеряется часами).

Online-инференс: предсказание считается по одному запросу в реальном времени, обычно через REST/gRPC API. Подходит, когда решение нужно здесь и сейчас: антифрод в момент депозита, персонализация в реальном времени, динамические лимиты. Требует низкой латентности, высокой доступности, продуманного масштабирования. Дороже и сложнее.

REST API на FastAPI — стандарт 2026 года для online-сервинга в Python-экосистеме. FastAPI даёт асинхронность, автоматическую валидацию входов через Pydantic и автогенерацию документации. Под капотом в проде — ASGI-сервер (uvicorn/gunicorn), а тяжёлый трафик заворачивают за балансировщик. Для нагруженного ML-инференса поверх или вместо голого FastAPI берут специализированные серверы — BentoML, KServe, Ray Serve: они из коробки дают батчинг запросов, автоскейлинг и управление несколькими моделями.

!!! tip "Как выбрать режим"

    Задай один вопрос: «сколько у бизнеса времени на ответ?». Если ответ нужен в течение секунд или миллисекунд по одному объекту — online. Если решение принимается раз в день/час по всей базе — batch. Антифрод на депозите — online. Утренний список «кому слать бонус удержания» — batch. Многие зрелые системы используют оба режима одновременно.

!!! question "Проверь себя"

    1. Какой один вопрос отличает batch-сервинг от online?
    2. Приведи по одному примеру задачи из гемблинга для batch и для online.
    3. Почему FastAPI стал стандартом online-сервинга в Python к 2026?

??? success "Ответы"

    1. «Сколько у бизнеса времени на ответ?» Секунды или миллисекунды по одному объекту — online; раз в день или час по всей базе — batch.
    2. Batch — утренний список «кому слать бонус удержания», ежедневный скоринг оттока. Online — антифрод в момент депозита, динамические лимиты.
    3. Асинхронность, автоматическая валидация входов через Pydantic и автогенерация документации.
