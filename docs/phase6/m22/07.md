# M22.7 · Версионирование данных, моделей и экспериментов

### Интуиция

Представь, что через полгода после деплоя тебя спрашивают: «Модель скоринга игроков, которая сейчас крутит выдачу бонусов, — на каких данных её обучали и каким кодом?» Если ты можешь ответить точно и воспроизвести ту самую модель байт-в-байт — у тебя есть версионирование. Если ты разводишь руками — у тебя нет ничего, кроме надежды.

Код ты уже версионируешь в git: каждый коммит — снимок состояния, к которому можно вернуться. Но в ML «состояние системы» — это не только код. Это три оси одновременно: какой код, какие данные, какая модель. Git отлично справляется с первой осью и плохо — с двумя другими, потому что датасет на 10 ГБ и веса модели на 2 ГБ в git класть нельзя (он создан для текстовых диффов, а не для больших бинарников: репозиторий распухает, операции тормозят).

Житейская картинка: git — это журнал правок текста рецепта. Но чтобы повторить блюдо, мало рецепта — нужны те же продукты (данные) и тот же готовый результат на пробу (модель-артефакт). Версионирование данных и моделей — это способ привязать к каждому коммиту кода ещё и «те самые продукты» и «ту самую готовую модель», не таская их физически внутри журнала.

Почему это критично именно в проде: без воспроизводимости ты не сможешь отладить деградацию («модель стала хуже предсказывать отток — это данные поехали или код поменялся?»), не сможешь честно сравнить две версии и не сможешь сделать быстрый откат, когда новая модель ломает метрики в продакшене.

### Формально

Воспроизводимость ML-системы держится на трёх осях версионирования:

- **Код** — git.
- **Данные** — DVC или аналог.
- **Эксперименты и модели** — трекинг + model registry (MLflow).

**DVC (Data Version Control)** — система версионирования данных и моделей поверх git. Механика: сам большой файл лежит в удалённом хранилище (S3, GCS, Azure Blob), а в git коммитится маленький текстовый указатель — `.dvc`-файл с хэшем содержимого и метаданными. Связка `git checkout <commit>` + `dvc checkout` восстанавливает согласованное состояние «код + данные» на любой момент. Адресация по хэшу содержимого (content-addressable storage) означает: одинаковые данные не дублируются, а изменение хоть одного байта даёт новый хэш и новую версию.

**Трекинг экспериментов** — систематическое логирование каждого запуска обучения: гиперпараметры, метрики, версия данных, версия кода (git commit hash), артефакт модели. Формально каждый run — это запись

$$
\text{run} = (\text{params},\ \text{metrics},\ \text{data\_version},\ \text{code\_commit},\ \text{artifact}).
$$

Без этой записи нельзя ответить, чем модель v7 лучше v6 и за счёт чего.

**MLflow** — де-факто стандарт на 2026 год. Четыре компонента:

- **Tracking** — логирование params/metrics/artifacts по каждому run. Отвечает на вопрос «какой набор гиперпараметров дал лучший AUC».
- **Models** — единый формат упаковки модели с метаданными окружения (flavor).
- **Model Registry** — реестр обученных моделей с версиями. Продовую версию помечают **алиасом** (например `@champion`), кандидата на замену — `@challenger`. Раньше использовали стадии (Staging/Production/Archived), но они объявлены устаревшими (deprecated): современный MLflow управляет продом через алиасы и теги, а не стадии.
- **Projects** — упаковка кода для воспроизводимого запуска.

**Rollback** в этой схеме — это переназначение алиаса. Деплой новой версии: присвоить ей `@champion`. Откат при проблеме: вернуть `@champion` на предыдущую версию. Сам артефакт остаётся в registry, меняется только указатель — поэтому откат быстрый и не требует переобучения.

#### Разбор на пальцах: воспроизводимость модели оттока игроков

Команда обучает модель прогноза оттока (churn) VIP-игроков, чтобы за неделю до предполагаемого ухода присылать ретеншн-бонус. Датасет — выгрузка поведения за квартал, 4 ГБ. Проследим, как версионирование спасает деньги.

Запуск 1 (понедельник):

- Данные: снимок `data/players.parquet`, хэш `a1b2c3`, зафиксирован в DVC.
- Код: коммит git `e4f5`.
- Гиперпараметры: `n_estimators=300, max_depth=6`.
- Метрика на отложенной выборке: $\text{AUC} = 0.78$, recall на топ-10\% риска $= 0.62$.
- MLflow логирует это как run #1, артефакт регистрируется как модель `churn` версии 5, ей дают `@champion` и деплоят.

Запуск 2 (пятница): дата-инженер пересобрал витрину, добавив фичу «дни с последнего депозита». Новый снимок данных, хэш `f9e8`, новый коммит `c7d8`, обучение даёт версию 6: $\text{AUC} = 0.81$. Ей дают `@challenger` и гоняют в shadow-режиме.

Через две недели модель v5 в проде начала проседать: recall на топ-риске упал с 0.62 до 0.49. Вопрос — данные поехали или что-то ещё? Благодаря трекингу видно: код и веса v5 не менялись, а распределение входной фичи «средний депозит» сместилось (data drift после смены платёжного провайдера). Без версионирования данных доказать это было бы нельзя — пришлось бы гадать.

Решение: переключить `@champion` с v5 на v6 (которая уже валидирована на свежих данных). Это одна операция смены алиаса, без переобучения, отрабатывает за секунды.

Посчитаем эффект в деньгах. Допустим, в топ-10\% риска каждую неделю попадает 500 VIP-игроков, истинных «уходящих» среди них — по recall. Ретеншн-бонус возвращает игрока с вероятностью 30\%, а удержанный VIP приносит в среднем \$200 маржи за следующий месяц.

- При recall 0.62 ловим $500 \times 0.62 = 310$ настоящих churn-кандидатов; возвращаем $310 \times 0.30 = 93$; маржа $93 \times \$200 = \$18\,600$ в неделю.
- При recall 0.49 (просевшая v5) ловим $500 \times 0.49 = 245$; возвращаем $245 \times 0.30 = 73.5$; маржа $73.5 \times \$200 = \$14\,700$ в неделю.

Разница $\$18\,600 - \$14\,700 = \$3\,900$ в неделю, около \$15\,600 в месяц. Быстрый откат на v6 через смену алиаса возвращает этот разрыв за минуту вместо дней переобучения вслепую. Арифметика: проверка $500 \times 0.62 = 310$, $310 \times 0.3 = 93$, $93 \times 200 = 18\,600$ — сходится.

#### Частые заблуждения и пограничные случаи

- **«DVC хранит файлы внутри git»** — нет. В git только указатель; данные в отдельном remote. Поэтому клонировать репозиторий быстро, а `dvc pull` тянет ровно нужную версию данных.
- **Версионировать каждый эфемерный эксперимент в registry не нужно.** Tracking логирует все runs (их сотни), а в Model Registry попадают только модели-кандидаты, которые реально рассматривают для прода.
- **Версионирование не заменяет мониторинг.** Оно даёт воспроизводимость и откат, но не скажет само, что в проде начался дрейф — для этого нужен отдельный мониторинг (тема соседних шагов).
- **Lineage важнее, чем просто «сохранить файл».** Ценность не в том, что у тебя лежит `model_v6.pkl`, а в том, что к нему привязаны точная версия данных и коммит кода. Артефакт без происхождения почти бесполезен для отладки.

!!! warning "Типичная ошибка"

    Новичок логирует в MLflow только метрики и гиперпараметры, но не фиксирует версию данных и git commit. Когда через месяц «лучшая» модель внезапно не воспроизводится, оказывается, что обучали её на витрине, которую с тех пор пересобрали, а кода той ветки уже нет. Метрика 0.81 в трекинге есть, а воспроизвести модель нельзя — значит, трекинг был бесполезен. Правило: каждый run должен жёстко привязывать и data_version (DVC-хэш), и code_commit (git hash), иначе это не воспроизводимость, а красивый журнал чисел.

!!! question "Проверь себя"

    1. Почему датасет на 10 ГБ и веса модели нельзя версионировать прямо в git, и как это решает DVC?
    2. За что отвечает Model Registry в MLflow и почему стадии Staging/Production заменили на алиасы?
    3. Как технически делается rollback через model registry и почему он быстрый?
    4. Какие три оси версионирования нужно зафиксировать, чтобы run был по-настоящему воспроизводим?

??? success "Ответы"

    1. git создан для текстовых диффов и плохо хранит большие бинарники (репозиторий распухает, операции тормозят). DVC кладёт в git только маленький текстовый указатель (`.dvc`-файл с хэшем и метаданными), а сам файл — в удалённое хранилище (S3/GCS), связывая версию кода с версией данных через `git checkout` + `dvc checkout`.
    2. Хранит обученные модели с версиями; продовую версию помечают алиасом (`@champion`), кандидата — `@challenger`. Алиасы заменили стадии, потому что дают гибкое именование указателей и позволяют управлять продом через теги/алиасы, а не через фиксированный набор стадий (Staging/Production deprecated).
    3. Переназначением алиаса `@champion` на нужную (предыдущую) версию. Быстро, потому что артефакт уже лежит в registry — меняется только указатель, переобучение не требуется.
    4. Код (git commit), данные (DVC-хэш версии) и сам артефакт модели с гиперпараметрами/метриками. Без привязки данных и коммита run невоспроизводим.

!!! example "Мини-задание"

    Ты обучил модель скоринга FTD-конверсии и хочешь, чтобы коллега смог воспроизвести её через месяц. За 2-5 минут выпиши минимальный чек-лист того, что нужно залогировать в одном MLflow run, чтобы воспроизводимость была гарантирована. Отдельно отметь, какой пункт чаще всего забывают (подсказка: он связан не с моделью, а с входом).
