# M22.7 · Версионирование данных, моделей и экспериментов

### Интуиция: git для данных и моделей

Код ты версионируешь в git. Но git плохо хранит большие файлы — датасет на 10 ГБ и веса модели туда складывать нельзя. А версионировать их нужно: чтобы через полгода ответить на вопрос «на каких данных и каким кодом получена модель, которая сейчас в проде». Для этого есть отдельные инструменты.

### Формально: DVC, MLflow, model registry

DVC (Data Version Control) — система версионирования данных и моделей поверх git. В git хранится маленький текстовый указатель (хэш и метаданные), а сам большой файл лежит в удалённом хранилище (S3, GCS). DVC связывает версию кода с версией данных: `git checkout` + `dvc checkout` восстанавливают согласованное состояние. Это даёт воспроизводимость пайплайна.

MLflow — стандарт 2026 для трекинга экспериментов и управления моделями. Четыре компонента:

- Tracking — логирование параметров, метрик и артефактов каждого запуска обучения. Отвечает на вопрос «какой набор гиперпараметров дал лучший AUC».
- Models — единый формат упаковки модели с метаданными окружения.
- Model Registry — реестр моделей с версиями. Какая версия сейчас в проде, помечают алиасом (например `@champion`), а кандидата на замену — `@challenger`. Раньше для этого использовали стадии (Staging/Production/Archived), но они объявлены устаревшими: современный MLflow управляет продовой версией через алиасы и теги, а не стадии.
- Projects — упаковка кода для воспроизводимого запуска.

Трекинг экспериментов — это систематическое логирование каждого обучения: параметры, метрики, версия данных, версия кода, артефакт модели. Без него ты не сможешь сказать, чем модель v7 лучше v6 и почему.

Model registry — централизованный каталог моделей с версионированием. Деплой и rollback делаются через переназначение алиаса: новой версии присваивают `@champion`, при проблеме — возвращают `@champion` на предыдущую версию. Это инфраструктурная основа отката.

!!! note "Связка инструментов"

    Типичный стек 2026: git (код) + DVC (данные) + MLflow (эксперименты, registry). Они отвечают на три части одного вопроса воспроизводимости — какой код, какие данные, какая модель. Версионируешь все три оси — можешь точно восстановить и откатить любое состояние системы.

!!! question "Проверь себя"

    1. Почему датасет на 10 ГБ и веса модели нельзя версионировать прямо в git, и как это решает DVC?
    2. За что отвечает Model Registry в MLflow?
    3. Как технически делается rollback через model registry?

??? success "Ответы"

    1. git плохо хранит большие бинарные файлы; DVC кладёт в git маленький текстовый указатель (хэш и метаданные), а сам файл — в удалённое хранилище (S3/GCS), связывая версию кода с версией данных.
    2. Хранит модели с версиями; продовую версию помечают алиасом (`@champion`) — он отвечает на вопрос «какая версия сейчас в проде» (стадии Staging/Production устарели).
    3. Переназначением алиаса `@champion` на нужную версию (или обратно на предыдущую) — это и есть откат.
