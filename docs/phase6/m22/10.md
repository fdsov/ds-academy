# M22.10 · Переобучение по расписанию и CI/CD для ML

### Интуиция: модель нужно регулярно «освежать»

Раз мир уезжает, модель нужно периодически переобучать на свежих данных. Это можно делать по расписанию (раз в неделю) или по триггеру (когда мониторинг поймал дрейф/деградацию). А чтобы это происходило надёжно и без ручного героизма — нужен ML-конвейер автоматизации.

### Формально: расписание, триггеры, CI/CD

Переобучение по расписанию (scheduled retraining) — оркестратор (Airflow, Prefect, Dagster) по cron-расписанию запускает пайплайн: подтянуть свежие данные, пересчитать признаки, обучить, сравнить с текущей продовой моделью на отложенной выборке, и при улучшении — зарегистрировать новую версию в registry. Расписание выбирают из скорости дрейфа: где поведение меняется быстро — чаще.

Триггерное переобучение (triggered retraining) — пайплайн запускается не по времени, а по событию от мониторинга: PSI превысил порог или метрика качества упала ниже SLA. Экономнее и реактивнее, но требует настроенного мониторинга как источника триггеров.

CI/CD для ML расширяет классический CI/CD тремя уровнями автоматизации:

- Continuous Integration — тесты не только кода, но и данных (валидация схемы, диапазонов) и модели (метрика не ниже порога, нет регрессии на ключевых сегментах).
- Continuous Delivery — автоматическая сборка и доставка модели как сервиса (контейнер + деплой).
- Continuous Training (CT) — уникальная для ML стадия: автоматический перезапуск обучения по расписанию или триггеру.

Обязательное правило перед автоматическим деплоем модели: gate-проверка. Новая версия выкатывается, только если прошла валидацию качества на свежем отложенном наборе и не уступает текущей продовой. Иначе автоматизация будет исправно деплоить деградировавшие модели.

### Воспроизводимость и rollback

Воспроизводимость — способность точно восстановить любой прошлый результат: при фиксированных версиях кода, данных и сидов повторный запуск даёт ту же модель. Это требует версионирования всех трёх осей (код/данные/модель) и фиксации случайных сидов и версий библиотек.

Rollback (откат) — быстрый возврат к предыдущей рабочей версии при проблеме с новой. Технически это смена указателя продовой версии в model registry (или редеплой предыдущего образа). Дешёвый и быстрый rollback возможен только когда у тебя есть версионированный registry и иммутабельные артефакты. Стратегии безопасного выката — canary (новая версия на малую долю трафика) и shadow (новая версия считает параллельно, но её ответы не идут пользователю, только сравниваются) — снижают риск ещё до полного переключения.
