### Шаг 3: проверяем утечку данных руками

Зачем: M25.6 называет утечку через препроцессинг до сплита самой частой и незаметной. Прежде чем критиковать чужой протокол, убедись, что сам видишь эффект. Воспроизводим классический механизм Kapoor — отбор признаков на всех данных до разбиения.

```python
import numpy as np
from sklearn.feature_selection import SelectKBest, f_classif
from sklearn.model_selection import train_test_split, StratifiedKFold, cross_val_score
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import Pipeline
from sklearn.metrics import roc_auc_score

rng = np.random.RandomState(SEED)
# Чистый шум: признаки никак не связаны с таргетом
Xn = rng.randn(500, 4000)
yn = rng.randint(0, 2, size=500)

# ЛЕЧЕНИЕ УТЕЧКИ: отбор на ВСЕХ данных до сплита (так делать нельзя)
sel = SelectKBest(f_classif, k=30).fit(Xn, yn)
Xs = sel.transform(Xn)
Xtr, Xte, ytr, yte = train_test_split(Xs, yn, test_size=0.3, random_state=SEED)
clf = LogisticRegression(max_iter=1000).fit(Xtr, ytr)
leaky = roc_auc_score(yte, clf.predict_proba(Xte)[:, 1])

# ЧЕСТНО: тот же отбор внутри Pipeline => fit только на train-фолде
pipe = Pipeline([
    ("sel", SelectKBest(f_classif, k=30)),
    ("clf", LogisticRegression(max_iter=1000)),
])
cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=SEED)
honest = cross_val_score(pipe, Xn, yn, cv=cv, scoring="roc_auc").mean()

print(f"leaky AUC = {leaky:.3f}   honest AUC = {honest:.3f}")
# leaky AUC ~ 0.65+   honest AUC ~ 0.50
```

Что получилось: на чистом шуме утечка рисует AUC заметно выше 0.5 — модель «нашла» сигнал, которого нет. Честный pipeline возвращает ~0.5. Теперь у тебя есть рабочее понимание: если в статье не описан протокол сплита и порядок препроцессинга — это не придирка, это реальный риск завышенных чисел.

!!! question "Проверь себя"

    1. Почему отбор признаков на чистом шуме даёт AUC > 0.5, если сигнала нет?
    2. Что именно в Pipeline закрывает эту утечку?

??? success "Ответы"

    1. SelectKBest, видя test, выбирает признаки, которые случайно скоррелированы с таргетом на всей выборке; на тесте эта случайная корреляция частично сохраняется и завышает метрику.
    2. Внутри cross_val_score Pipeline вызывает fit только на train-фолде, поэтому отбор признаков не видит test — порядок «split → fit препроцессинга → transform» соблюдён автоматически.

### Шаг 4: проверяем эффект бейзлайна и считаем CI

Зачем: M25.6 — слабый бейзлайн раздувает преимущество нового метода. Воспроизведём на нашей синтетике сам механизм: дефолтный бустинг против тюненного. Если разница большая, значит «победа над дефолтным бейзлайном» в чужой статье может быть артефактом, а не вкладом метода.

```python
import lightgbm as lgb
from scipy import stats
from sklearn.model_selection import cross_val_score, StratifiedKFold

def report_metric(scores, confidence=0.95):
    a = np.asarray(scores, float)
    h = stats.sem(a) * stats.t.ppf((1 + confidence) / 2.0, df=len(a) - 1)
    return f"{a.mean():.4f} ± {h:.4f} (95% CI, n={len(a)})"

cv = StratifiedKFold(n_splits=5, shuffle=True, random_state=SEED)

weak = lgb.LGBMClassifier(n_estimators=20, num_leaves=3, verbose=-1, random_state=SEED)
strong = lgb.LGBMClassifier(
    n_estimators=600, num_leaves=31, learning_rate=0.03,
    subsample=0.8, colsample_bytree=0.8, verbose=-1, random_state=SEED,
)

s_weak = cross_val_score(weak, X, y, cv=cv, scoring="roc_auc")
s_strong = cross_val_score(strong, X, y, cv=cv, scoring="roc_auc")
print("слабый бейзлайн :", report_metric(s_weak))
print("честный бейзлайн:", report_metric(s_strong))
```

Что получилось: тюненный LightGBM заметно обгоняет «специально ослабленный». Перенеси вывод на статью: ровно так в TabNet-критике (arXiv:2106.03253) показали, что дефолтные или слабо настроенные GBDT-бейзлайны создавали иллюзию превосходства DL. Заодно ты получил формат отчёта `mean ± CI` — то, чего часто нет в самих статьях.
