# M25.7 · Воспроизведение результатов

Воспроизвести статью — лучший способ её по-настоящему понять и проверить. План действий.

1. **Код**. Ищи официальную реализацию (ссылка в статье, Papers with Code, GitHub авторов). Нет официальной — ищи независимую. Нет никакой — это уже сильный сигнал о воспроизводимости.
2. **Данные**. Тот же датасет, та же версия, тот же сплит. Расхождение в препроцессинге — частая причина несовпадения чисел.
3. **Сиды и детерминизм**. Фиксируй все источники случайности: Python, NumPy, фреймворк, и по возможности отключай недетерминированные CUDA-ядра.
4. **Окружение**. Версии библиотек важны: один и тот же код на разных версиях даёт разные числа. Фиксируй их (`requirements.txt`, lock-файл, контейнер).
5. **Сравнение**. Совпадение в пределах разброса по сидам — успех. Систематическое расхождение — повод искать утечку у них, ошибку у себя или недосказанность в статье.

!!! question "Проверь себя"

    1. Почему отсутствие любой реализации (официальной или независимой) — сильный сигнал ещё до запуска кода?
    2. Какие источники случайности нужно зафиксировать перед прогоном?
    3. Почему версии библиотек выносят в отдельный пункт плана воспроизведения?
    4. Как трактовать совпадение твоего числа с заявленным в пределах разброса по сидам?

??? success "Ответы"

    1. Если работу никто, включая авторов, не смог или не захотел оформить в воспроизводимый код, это намёк на проблемы с воспроизводимостью самого результата.
    2. Python, NumPy, фреймворк (например torch) и, по возможности, недетерминированные CUDA-ядра; плюс сплит данных.
    3. Один и тот же код на разных версиях библиотек даёт разные числа; без фиксации (requirements/lock/контейнер) расхождение не отличить от собственной ошибки.
    4. Это успех воспроизведения. Систематическое расхождение за пределами разброса — повод искать утечку у авторов, ошибку у себя или недосказанность в статье.
