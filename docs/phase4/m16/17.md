# M16.17 · Практика

### Быстрый квиз

Выбери вариант — ответ подсветится сразу.

```quiz
На какое слагаемое bias-variance разложения влияет feature engineering?
[x] на bias — делает истинную функцию проще выразимой в данном классе моделей
[ ] на variance — снижает разброс предсказаний модели
[ ] на неустранимый шум sigma^2
[ ] сразу на все три в равной мере
> Удачное преобразование признаков снижает bias без роста variance; неустранимый шум затронуть нельзя.
---
В чём состоит главная опасность target (mean) encoding?
[ ] он взрывает размерность на высококардинальных категориях
[x] утечка таргета: значение признака для объекта зависит от его собственного таргета
[ ] он навязывает категориям ложный порядок
[ ] он не работает с пропусками
> Наивный groupby(cat).y.mean() по всему трейну подсматривает ответ; лечится out-of-fold или ordered кодированием внутри пайплайна.
---
Почему масштабирование бесполезно для XGBoost и LightGBM?
[ ] деревья сами нормируют входы перед обучением
[ ] бустинг работает только с категориальными признаками
[x] дерево режет по порогу x < t, а монотонное масштабирование не меняет порядок значений и ни одного сплита
[ ] масштабирование ломает нативную обработку пропусков
> StandardScaler перед бустингом — потерянное время и маркер непонимания того, как дерево делает сплит.
---
Зачем час кодируют парой sin/cos вместо одного числа от 0 до 23?
[ ] чтобы уменьшить число признаков
[ ] чтобы убрать выбросы во времени
[x] время циклично: час 23 и час 0 соседние, sin/cos с периодом 24 кладёт их рядом на окружности
[ ] чтобы деревья лучше находили пороги по времени
> Пара sin/cos делает расстояние между 23 и 0 малым, и модель улавливает суточный ритм активности.
---
Почему встроенной feature importance деревьев нельзя слепо доверять?
[ ] она считается слишком медленно
[x] она смещена в сторону высококардинальных и непрерывных признаков
[ ] она доступна только для линейных моделей
[ ] она всегда обнуляет важные признаки
> Надёжнее permutation importance (перемешать колонку и замерить просадку метрики) или SHAP.
---
Чем байесовская оптимизация в Optuna принципиально отличается от grid и random search?
[ ] она перебирает все комбинации сетки до конца
[ ] она вообще не использует кросс-валидацию
[x] строит суррогатную модель метрики и выбирает следующую пробу там, где ожидается наибольший выигрыш
[ ] она подбирает не гиперпараметры, а сами признаки
> Grid и random «слепы» — каждая проба не учитывает прошлые; Optuna балансирует исследование и эксплуатацию через функцию приобретения.
---
Зачем нужна вложенная (nested) кросс-валидация при тюнинге?
[ ] чтобы ускорить подбор гиперпараметров
[x] лучший CV-скор после тюнинга оптимистичен; внешний цикл честно оценивает на данных, которых внутренний не видел
[ ] чтобы заменить train/test split
[ ] чтобы избежать масштабирования признаков
> Внутренний цикл подбирает гиперпараметры, внешний — честно меряет качество; иначе оценка завышена подгонкой под валидацию.
```

Разминка. Возьми датасет с категориальным признаком средней кардинальности (например, страна). Закодируй его четырьмя способами: one-hot, label, частотное, target (через sklearn `TargetEncoder`). Обучи логистическую регрессию и LightGBM на каждом варианте, сравни ROC-AUC на кросс-валидации. Объясни словами, почему для линейной модели и для дерева лучшие способы разные.

Основное. По логу ставок (или синтетическому: сгенерируй player_id, ts, bet_amount, deposit_amount) собери churn-датасет с фиксированной cutoff-датой. Построй минимум 15 признаков: recency, частота, монетарные суммы по окнам 7/14/30 дней, отношения окон, волатильность ставки, циклические признаки времени активности. Заверни весь препроцессинг в `Pipeline` + `ColumnTransformer`. Докажи отсутствие утечки: проверь, что ни один признак не использует данные после cutoff, и что энкодеры фитятся внутри CV.

Челлендж. Прогони отбор признаков тремя способами (mutual information фильтр, permutation importance, RFECV) на своём churn-датасете и сравни, какие признаки каждый метод считает топовыми. Затем настрой LightGBM через Optuna (80+ trials, с pruner). Сравни лучший скор Optuna с random search и grid search при равном бюджете обучений — построй график сходимости.

Мини-проект. Сквозной churn-пайплайн на гемблинг-логе: feature engineering на polars → пайплайн без утечки → отбор признаков → тюнинг Optuna → вложенная кросс-валидация для честной оценки качества → SHAP-разбор топ-10 признаков с бизнес-интерпретацией («почему отношение bet_7d/bet_30d так важно для оттока»). Оформи как воспроизводимый ноутбук с отчётом: какой признак дал наибольший прирост и почему.
