# M16.13 · Подбор гиперпараметров

Гиперпараметры — это настройки, которые не выучиваются из данных, а задаются до обучения: глубина дерева, learning rate, сила регуляризации. Их надо подбирать, и способ подбора эволюционировал.

### Grid search

Интуиция: задать сетку значений по каждому гиперпараметру и перебрать все комбинации. Просто и исчерпывающе.

Проблема — комбинаторный взрыв. Пять гиперпараметров по пять значений — это $5^5 = 3125$ обучений. С кросс-валидацией умножай ещё. И большая часть сетки тратится на заведомо плохие зоны.

### Random search

Интуиция: вместо полной сетки случайно сэмплировать комбинации из заданных распределений и взять лучшую за фиксированный бюджет.

Почему часто лучше grid: обычно лишь несколько гиперпараметров реально важны. Grid тратит бюджет, перебирая неважные оси на фиксированной сетке. Random search покрывает важные оси большим числом разных значений при том же бюджете. Это классический результат Бергстры и Бенжио: при равном числе итераций случайный поиск обычно находит лучшую точку.

### Байесовская оптимизация через Optuna — стандарт 2026

Интуиция: grid и random «слепы» — каждая следующая проба не учитывает результаты предыдущих. Байесовская оптимизация умная: она строит вероятностную модель того, как гиперпараметры влияют на метрику, и выбирает следующую пробу там, где ожидается наибольший выигрыш. Балансирует исследование новых зон и эксплуатацию перспективных.

Формально: строится суррогатная модель $p(\text{метрика} \mid \text{гиперпараметры})$ (в Optuna по умолчанию TPE — Tree-structured Parzen Estimator). На каждом шаге максимизируется функция приобретения (acquisition function, например Expected Improvement), которая указывает следующую точку:

$$
x_{\text{next}} = \arg\max_x \; \mathbb{E}\big[\max(0,\; f_{\text{best}} - f(x))\big]
$$

То есть выбираем точку с наибольшим ожидаемым улучшением над текущим лучшим.

Почему Optuna стал стандартом к 2026. define-by-run API (пространство поиска задаётся обычным Python-кодом, можно условные и динамические параметры). Встроенный pruning — обрывает заведомо проигрышные обучения досрочно, экономя бюджет. Параллелизм и распределённый запуск, персистентность испытаний в БД, готовая визуализация важности гиперпараметров. Нативная интеграция с XGBoost, LightGBM, CatBoost, sklearn, PyTorch. Practical bottom line: на том же бюджете Optuna обычно находит конфигурацию лучше random search и кратно лучше grid. Из альтернатив стоит знать Ray Tune (распределённый тюнинг с современными планировщиками вроде ASHA) и FLAML (быстрый cost-aware AutoML-поиск) — но Optuna остаётся самым ходовым дефолтом.

!!! question "Проверь себя"

    1. Почему random search при равном бюджете обычно бьёт grid search?
    2. В чём принципиальное отличие байесовской оптимизации от grid и random?
    3. Что такое pruning в Optuna и на чём он экономит бюджет?

??? success "Ответы"

    1. Реально важны обычно лишь несколько гиперпараметров. Grid тратит бюджет, перебирая неважные оси на фиксированной сетке; random покрывает важные оси большим числом разных значений при том же числе итераций (результат Бергстры и Бенжио).
    2. Grid и random «слепы» — каждая проба не учитывает прошлые. Байесовская оптимизация строит суррогатную модель $p(\text{метрика} \mid \text{гиперпараметры})$ и выбирает следующую точку по функции приобретения (Expected Improvement), балансируя исследование и эксплуатацию.
    3. Pruning досрочно обрывает заведомо проигрышные испытания (не доводит до конца обучение, которое уже отстаёт), освобождая бюджет на перспективные конфигурации.
