## Что отрабатываем

Этот воркшоп отрабатывает ядро модуля M15 на одной сквозной задаче: разбить игроков казино на сегменты, когда меток нет вообще. Ты пройдёшь весь честный пайплайн unsupervised из теории:

- RFM-признаки (Recency / Frequency / Monetary) и логарифмирование тяжелохвостых денег;
- масштабирование через `StandardScaler` — ошибка номер один, если его забыть;
- K-means с выбором $k$ по силуэту, а не по «красоте локтя»;
- сравнение с DBSCAN по плотности (полумесяцы против сфер, шум как «ничьи» точки);
- PCA: доля объяснённой дисперсии и осознанное число компонент;
- UMAP только для просмотра глазами, не для выводов;
- профилирование и нейминг сегментов — без него числа кластеров бессмысленны;
- валидацию без таргета через пересечение трёх условий: метрика + стабильность + бизнес-смысл.

Артефакт на выходе: таблица `players` с колонкой `segment`, график силуэта, UMAP-картинка, и набор карточек сегментов (имя, средние RFM, размер, оффер). Это прямой прообраз мини-проекта из M15.13.

## Бизнес-кейс

!!! example "Ситуация"

    Ты data scientist в команде iGaming-казино. К тебе приходит руководитель CRM / Head of Retention: маркетинг сейчас шлёт одинаковые бонусы всей базе — фиксированный кешбэк и фриспины «на всех». Месячный бюджет на бонусы — ориентировочно 6 млн ₽, и он явно сгорает неэффективно: киты получают мелкий бонус, который их не двигает, а спящих и новичков заливают офферами, на которые они не реагируют. CRM-команда хочет перейти к сегментным кампаниям, но в продукте нет ни одной готовой метки игроков — кто «кит», кто «спящий», кто «новичок». Эти ярлыки нужно достать из поведения.

    - Что зависит от ответа: на твоей сегментации CRM построит 4 разных оффер-трека и перераспределит тот же бюджет 6 млн ₽/мес. Реалистичная цель — поднять отдачу на бонус и вернуть часть спящих; даже +2 п.п. к удержанию платящих в этой базе — это десятки миллионов ₽ выручки в год. Цена ошибки — слить бюджет на «мусорные» сегменты или дать VIP-оффер тем, кто и так платит.
    - Ограничение по данным: размеченных меток нет вообще, есть только сырое поведение (recency / frequency / monetary). Это честная unsupervised-задача.
    - Срок: 1 спринт. На выходе CRM ждёт не «силуэт 0.6», а понятные сегменты с именами, размерами и конкретным оффером под каждый.

## Данные

Никаких внешних файлов. Генерируем синтетику с фиксированным seed: четыре скрытые группы игроков плюс немного шума-выбросов (будущие кандидаты в аномалии). Денежные величины делаем тяжелохвостыми через логнормаль — ровно тот случай, где киты растягивают распределение и сырой K-means ломается.

```python
import numpy as np
import pandas as pd

rng = np.random.default_rng(42)

def cohort(n, rec_lo, rec_hi, freq_mu, dep_mu, dep_sigma):
    return pd.DataFrame({
        "recency": rng.integers(rec_lo, rec_hi, n),
        "frequency": np.clip(rng.poisson(freq_mu, n), 1, None),
        "monetary": rng.lognormal(dep_mu, dep_sigma, n),
    })

players = pd.concat([
    cohort(1200, 0, 7,   25, 6.5, 0.5),   # активные киты: заходят часто, недавно, платят много
    cohort(1500, 1, 14,  12, 5.0, 0.5),   # активные минималисты: часто, но малые суммы
    cohort(900,  30, 90,  4, 5.8, 0.6),   # спящие середняки: давно не были
    cohort(1000, 0, 5,    2, 4.2, 0.4),   # новички-разведчики: свежие, мало действий
    cohort(100,  0, 120, 60, 8.0, 1.2),   # шум/выбросы: экстремальное поведение
], ignore_index=True)

players = players.sample(frac=1, random_state=42).reset_index(drop=True)
print(players.describe().round(1))
```

Названия когорт в генераторе — это «правда», которой в реальной задаче у тебя нет. Не подсматривай в них при кластеризации; используй только в самом конце, чтобы проверить, насколько алгоритм восстановил структуру.

## Ход работы

### Шаг 1: подготовка RFM и логарифмирование денег

Зачем. Теория (ловушки M15.12) предупреждает: кластеризовать на сырых RFM нельзя — распределение депозитов тяжелохвостое, киты растягивают всё, и евклидово расстояние будет почти полностью определяться `monetary`. Лог сжимает хвост в нормальную форму.

```python
feat = pd.DataFrame({
    "recency": players["recency"],
    "frequency": players["frequency"],
    "monetary_log": np.log1p(players["monetary"]),
})
print(feat.describe().round(2))
print("\nasymmetry monetary до/после лога:",
      round(players["monetary"].skew(), 2),
      round(feat["monetary_log"].skew(), 2))
```

Что получилось. Асимметрия `monetary` падает с большого положительного значения почти до нуля. Теперь все три признака сопоставимы по форме, но всё ещё в разных единицах — это лечит следующий шаг.

### Шаг 2: масштабирование и наглядная демонстрация, зачем оно

Зачем. K-means считает $\lVert x_i - \mu_j \rVert^2$ в евклидовой метрике. Без `StandardScaler` признак с большим разбросом доминирует. Покажем это эмпирически: кластеризуем дважды — на сырых и на масштабированных признаках — и сравним.

```python
from sklearn.preprocessing import StandardScaler
from sklearn.cluster import KMeans

X = StandardScaler().fit_transform(feat)

raw_labels = KMeans(n_clusters=4, n_init=10, random_state=42).fit_predict(feat.values)
scaled_labels = KMeans(n_clusters=4, n_init=10, random_state=42).fit_predict(X)

def driver(labels):
    tmp = feat.copy()
    tmp["c"] = labels
    return tmp.groupby("c").mean().round(2)

print("БЕЗ масштабирования (кластеры почти только по monetary):")
print(driver(raw_labels))
print("\nС масштабированием:")
print(driver(scaled_labels))
```

Что получилось. В варианте без масштабирования центры кластеров различаются почти исключительно по `monetary_log`, а `recency`/`frequency` внутри кластеров перемешаны — депозит захватил расстояние. После масштабирования кластеры различаются по всем трём осям. Дальше работаем только с `X`.

!!! question "Проверь себя"

    1. Почему именно `monetary` доминировал в сыром K-means, а не `recency`?
    2. Лог уже сжал депозиты — зачем после него ещё и `StandardScaler`?
    3. Решающее дерево из M13 не требует масштабирования. Чем оно отличается от K-means в этом смысле?

??? success "Ответы"

    1. У `monetary` самый большой абсолютный разброс значений; евклидово расстояние складывает квадраты разностей по осям, и ось с наибольшим разбросом перебивает остальные.
    2. Лог чинит форму распределения (тяжёлый хвост), но не выравнивает единицы измерения: дни и логарифм рублей всё равно живут в разных масштабах. Scaler приводит каждый признак к среднему 0 и дисперсии 1.
    3. Дерево делит пространство порогами по одному признаку за раз и инвариантно к монотонным преобразованиям шкалы; K-means опирается на общую метрику расстояния, где масштабы признаков складываются.

### Шаг 3: выбор k по силуэту (не по локтю)

Зачем. M15.4: локоть субъективен, силуэт даёт сравнимое число $s \in [-1,1]$. Считаем обе кривые, но решение принимаем по силуэту.

```python
from sklearn.metrics import silhouette_score

rows = []
for k in range(2, 9):
    km = KMeans(n_clusters=k, n_init=10, random_state=42).fit(X)
    rows.append({
        "k": k,
        "inertia": round(km.inertia_, 1),
        "silhouette": round(silhouette_score(X, km.labels_), 3),
    })
scan = pd.DataFrame(rows)
print(scan)

best_k = int(scan.loc[scan["silhouette"].idxmax(), "k"])
print("k по максимуму силуэта:", best_k)
```

```python
import matplotlib.pyplot as plt

fig, ax = plt.subplots(1, 2, figsize=(11, 4))
ax[0].plot(scan["k"], scan["inertia"], "o-"); ax[0].set_title("Локоть (inertia)")
ax[1].plot(scan["k"], scan["silhouette"], "o-"); ax[1].set_title("Силуэт")
for a in ax: a.set_xlabel("k")
plt.tight_layout(); plt.savefig("k_selection.png", dpi=120)
```

Что получилось. Inertia падает монотонно и гладко — явного локтя нет, как и предупреждала теория. Силуэт даёт чёткий максимум (ожидаемо около $k=4$, поскольку мы заложили 4 когорты). Фиксируем `best_k` числом, а не на глаз.

### Шаг 4: финальная сегментация K-means

Зачем. M15.4 + правило про `random_state`: без фиксации seed кластеры «прыгают», а на этом нельзя строить продуктовые решения.

```python
km = KMeans(n_clusters=best_k, n_init=10, random_state=42)
players["segment"] = km.fit_predict(X)
print(players["segment"].value_counts().sort_index())
```

Что получилось. Колонка `segment` — это половина артефакта. Размеры кластеров не должны быть вырожденными (один сегмент на 5 человек, остальные на тысячи — повод пересмотреть `k` или признаки).

### Шаг 5: сравнение с DBSCAN по плотности

Зачем. M15.6: K-means пихает каждую точку в кластер, DBSCAN честно помечает выбросы как шум (`-1`). Сравним, как два разных, но валидных алгоритма видят одни данные. $\varepsilon$ подбираем по k-distance графику.

```python
import numpy as np
from sklearn.neighbors import NearestNeighbors
from sklearn.cluster import DBSCAN

min_samples = 8
nn = NearestNeighbors(n_neighbors=min_samples).fit(X)
kdist = np.sort(nn.kneighbors(X)[0][:, -1])
plt.figure(figsize=(6, 4))
plt.plot(kdist); plt.ylabel(f"расстояние до {min_samples}-го соседа")
plt.title("k-distance: ищем колено для eps")
plt.savefig("kdistance.png", dpi=120)

eps = round(float(np.quantile(kdist, 0.92)), 2)   # колено грубо у верхнего перегиба
db = DBSCAN(eps=eps, min_samples=min_samples).fit(X)
players["dbscan"] = db.labels_
print("eps =", eps)
print(pd.Series(db.labels_).value_counts())
print("доля шума:", round((db.labels_ == -1).mean(), 3))
```

Что получилось. DBSCAN выделяет несколько плотных ядер и помечает часть точек как шум (`-1`) — это в основном наши заложенные выбросы. Число кластеров он вывел сам, не получая `k`. Сравни состав: точки-шум у DBSCAN — это кандидаты в аномалии, которых K-means насильно распихал по ближайшим центрам.

!!! question "Проверь себя"

    1. Почему у DBSCAN появилась метка `-1`, а у K-means такой быть не может?
    2. Что произойдёт с разбиением DBSCAN, если взять слишком большой `eps`?
    3. На каких данных DBSCAN обыграл бы K-means принципиально, а не косметически?

??? success "Ответы"

    1. `-1` — это шум: точки, не попавшие ни в одну плотную область. K-means приписывает каждую точку к ближайшему из $k$ центров и понятия «ничья точка» не имеет.
    2. Кластеры начнут сливаться в один большой — при чрезмерном `eps` почти все точки оказываются взаимно достижимы по плотности, и структура исчезает.
    3. На кластерах сложной невыпуклой формы (два полумесяца, кольца) и при наличии выбросов: K-means режет по сферам вокруг центроидов и форму не проследит, DBSCAN идёт по связной плотности.

### Шаг 6: PCA — сколько компонент реально нужно

Зачем. M15.7: доля объяснённой дисперсии $\text{EVR}(m)$ говорит, какую часть разброса сохраняем при $m$ компонентах. На трёх RFM-признаках это игрушечный, но показательный расчёт — та же логика масштабируется на 200 признаков.

```python
from sklearn.decomposition import PCA

pca = PCA().fit(X)
evr = pca.explained_variance_ratio_
cum = np.cumsum(evr)
print("EVR по компонентам:", evr.round(3))
print("Накопленная:", cum.round(3))

n_comp = int(np.searchsorted(cum, 0.90)) + 1
print(f"Для 90% дисперсии нужно {n_comp} из {X.shape[1]} компонент")
```

Что получилось. Видно, сколько компонент покрывает 90% разброса. На реальных 200 признаках здесь обычно остаётся 10-30 компонент. Помни предупреждение теории: для антифрода PCA опасен — редкий сигнал фрода живёт в направлении малой дисперсии, и PCA(0.95) его убьёт. Здесь PCA — диагностика и препроцессинг визуализации, не препроцессинг детектора аномалий.

### Шаг 7: UMAP — только посмотреть глазами

Зачем. M15.8: UMAP — инструмент разведки, не доказательство. Кластеризацию мы уже сделали в исходном пространстве (`X`), а UMAP используем строго чтобы раскрасить точки по `segment` и оценить разделимость визуально.

```python
import umap

emb = umap.UMAP(n_neighbors=15, min_dist=0.1, random_state=42).fit_transform(X)

plt.figure(figsize=(7, 6))
plt.scatter(emb[:, 0], emb[:, 1], c=players["segment"], s=6, cmap="tab10")
plt.title("UMAP projection · раскраска по K-means сегментам")
plt.savefig("umap_segments.png", dpi=120)
```

Что получилось. Сегменты должны лечь относительно обособленными «островами». Но не делай выводов из расстояний между островами, их размеров и плотности — всё это искажено. UMAP подтверждает «структура есть / структуры нет», а не «сегмент 1 ближе к сегменту 2».

!!! question "Проверь себя"

    1. Почему кластеризацию делаем на `X`, а UMAP только для картинки?
    2. Три вещи на UMAP-картинке, которым нельзя доверять?
    3. Сменили `random_state` UMAP и острова переехали — это баг пайплайна?

??? success "Ответы"

    1. UMAP — нелинейная проекция, искажающая геометрию ради локальной близости; кластеры в ней визуально хороши, но метрики и расстояния недостоверны. В исходном (или PCA) пространстве расстояния честные.
    2. Расстояния между «островами», размеры кластеров и их плотность. Плюс зависимость картинки от random seed.
    3. Нет, это нормальное свойство метода: абсолютное расположение островов не несёт смысла и меняется с seed. Багом было бы, если бы при этом разваливалась сама кластеризация на `X`.

### Шаг 8: проверка стабильности

Зачем. M15.11: хорошая кластеризация устойчива. Если при другом seed игроки массово перетекают между кластерами — структура ненадёжна. Меряем совпадение разбиений через Adjusted Rand Index.

```python
from sklearn.metrics import adjusted_rand_score

base = players["segment"].values
aris = []
for seed in [1, 7, 13, 99]:
    lab = KMeans(n_clusters=best_k, n_init=10, random_state=seed).fit_predict(X)
    aris.append(adjusted_rand_score(base, lab))
print("ARI к базовому разбиению:", [round(a, 3) for a in aris])
```

Что получилось. ARI близко к 1.0 — разбиение стабильно при смене инициализации, структуре можно доверять. Если бы ARI плавало в районе 0.5-0.7, это сигнал, что `k` завышен или признаки слабые, и интерпретировать сегменты рано.

### Шаг 9: профилирование и карточки сегментов

Зачем. Кульминация валидации (M15.11): без бизнес-смысла высокий силуэт ничего не стоит. Считаем средние RFM по сегментам в исходных единицах (не масштабированных), даём имена и оффер.

```python
profile = players.groupby("segment").agg(
    n=("segment", "size"),
    recency=("recency", "mean"),
    frequency=("frequency", "mean"),
    monetary=("monetary", "median"),
).round(1).sort_values("monetary", ascending=False)
print(profile)
```

```python
def name_segment(r):
    if r.monetary > players["monetary"].quantile(0.75) and r.recency < 14:
        return "Активные киты"
    if r.frequency > players["frequency"].median() and r.monetary < players["monetary"].median():
        return "Активные минималисты"
    if r.recency > 30:
        return "Спящие"
    return "Новички-разведчики"

profile["name"] = [name_segment(r) for r in profile.itertuples()]
profile["offer"] = profile["name"].map({
    "Активные киты": "VIP-менеджер + персональный кешбэк",
    "Активные минималисты": "турниры с низким бай-ином, геймификация",
    "Спящие": "реактивация: бонус за возврат, A/B-тест размера",
    "Новички-разведчики": "онбординг-серия, фриспины на 2-3 депозит",
})
print(profile[["name", "n", "recency", "frequency", "monetary", "offer"]].to_string())
profile.to_csv("segment_cards.csv")
```

Что получилось. Готовый артефакт: карточки сегментов с именем, размером, средними RFM и конкретным оффером. Каждое имя обосновано числами из `profile` — ровно то, чего требует защита мини-проекта в M15.13. Пороговая логика нейминга здесь упрощённая; на реальных данных правь её под фактические центры кластеров.

!!! tip "Где подключить AI-копилота"

    Отдай LLM готовую таблицу `profile` и попроси предложить имена и маркетинговые гипотезы по сегментам, отметив, какие выводы требуют A/B-теста. Но не проси модель назвать «оптимальное k» по описанию данных — это галлюцинация из теории M15.12; `k` выбирается только силуэтом на твоих числах. И перепроверь, что во всех ветках есть `StandardScaler` — модель любит его терять.

Технический артефакт готов — но руководитель CRM не примет на вход `segment_cards.csv` и значение ARI. Переведи результат в решение, которое можно унести на планёрку и запустить в кампанию.
