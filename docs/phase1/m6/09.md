# M6.9 · merge, join, concat: соединяем таблицы

### Интуиция

Данные почти никогда не лежат в одной таблице. Регистрации в одной, депозиты в другой, ставки в третьей. `merge` (он же join из мира SQL) соединяет таблицы по общему ключу — например, по `player_id`. `concat` — это просто склейка таблиц встык: либо одна под другой (добавить строки), либо рядом (добавить столбцы).

Аналогия: `merge` — это как сопоставить два списка по общему ID (досье игрока + его транзакции). `concat` — это сложить две одинаковые по структуре пачки строк в одну.

Чтобы решить, что тебе нужно, задай один вопрос: **«я добавляю новые столбцы к тем же объектам или новые строки к той же схеме?»**

- Добавляю столбцы (к игроку подтягиваю его депозит, его сегмент, его LTV) — это `merge`. Объекты те же, признаков становится больше. Таблица растёт **вширь**.
- Добавляю строки (к мартовским ставкам добавляю апрельские) — это `concat` по строкам. Схема та же, объектов становится больше. Таблица растёт **вглубь**.

Ещё одна метафора из жизни: `merge` — это VLOOKUP в Excel, только без боли и без протягивания формулы; `concat(axis=0)` — это «скопировал диапазон и вставил снизу».

### Формально

```python
# inner: только совпавшие ключи
m = registrations.merge(deposits, on="player_id", how="inner")

# left: все игроки слева, депозиты где есть, иначе NaN
m = registrations.merge(deposits, on="player_id", how="left")

# проверка количества строк — обязательна
m = registrations.merge(deposits, on="player_id", how="left", validate="one_to_many")
```

Типы join: `inner` (пересечение ключей), `left` (все слева), `right` (все справа), `outer` (объединение). В аналитике чаще всего нужен `left`: "взять всех игроков и подтянуть к ним депозиты, не теряя тех, кто не депозитил".

```python
# concat по строкам (вертикально)
all_bets = pd.concat([bets_march, bets_april], ignore_index=True)

# concat по столбцам (горизонтально, по индексу)
wide = pd.concat([features_a, features_b], axis=1)
```

### Четыре типа join на пальцах

Представь два множества ключей — слева регистрации (R), справа депозиты (D). Join решает, какие ключи попадут в результат:

| how | Какие строки остаются | Когда использовать в аналитике |
|------|----------------------|-------------------------------|
| `inner` | только ключи, что есть И слева, И справа | «только депозитившие игроки» — когортный анализ платящих |
| `left` | все слева; справа — что нашлось, иначе `NaN` | дефолт: «все игроки, депозиты где были» |
| `right` | все справа; слева — что нашлось | редко; обычно проще поменять таблицы местами и взять `left` |
| `outer` | объединение всех ключей с обеих сторон | сверка двух источников: кто есть тут, но нет там |

Ключевая мысль: `inner` **молча выкидывает** игроков без депозита. Если ты считаешь конверсию в депозит и случайно сделал `inner` вместо `left`, знаменатель схлопнется до числителя — и конверсия превратится в 100%. Это классическая ошибка, которая не падает с ошибкой, а тихо врёт.

### Разобранный численный пример

```python
import pandas as pd

registrations = pd.DataFrame({
    "player_id": [1, 2, 3],
    "country":   ["UZ", "KZ", "UZ"],
})

# у игрока 1 — ДВА депозита, у игрока 3 — ни одного
deposits = pd.DataFrame({
    "player_id": [1, 1, 2],
    "amount":    [50, 30, 100],
})

left = registrations.merge(deposits, on="player_id", how="left")
print(left)
#    player_id country  amount
# 0          1      UZ    50.0
# 1          1      UZ    30.0   <- строка игрока 1 размножилась
# 2          2      KZ   100.0
# 3          3      UZ     NaN   <- игрок без депозита, amount = NaN

print(len(registrations), "->", len(left))   # 3 -> 4
print(left["amount"].sum())                   # 180.0
```

Обрати внимание на две вещи. Во-первых, строк стало 4, а не 3 — игрок 1 «раздвоился», потому что у него два депозита. Во-вторых, игрок 3 остался в таблице с `NaN` в `amount` — это и есть смысл `left`. Если бы ты сделал `inner`, игрок 3 бы исчез, и `len` стал бы 3 (только записи с депозитами).

Теперь представь, что это не учебные 3 игрока, а продакшен. Ты хочешь «число уникальных игроков и сумму депозитов». Если после такого merge ты напишешь `left["player_id"].nunique()` — получишь корректные 3 (nunique схлопывает дубли). Но если напишешь `len(left)` как «число игроков» — получишь 4. А `amount.sum()` тут как раз корректен (180 = 50+30+100), потому что суммируем сами депозиты. Опасность приходит, когда **размноженным оказывается то, что нельзя суммировать дважды** — см. ниже.

### Когда дубликаты реально портят цифры

Опасность не в самом размножении строк, а в том, что после merge ты агрегируешь столбец **из левой таблицы**, который теперь продублировался.

```python
registrations = pd.DataFrame({
    "player_id": [1, 2],
    "bonus_credited": [10, 10],   # бонус начислен один раз на игрока
})
deposits = pd.DataFrame({
    "player_id": [1, 1, 1, 2],
    "amount":    [50, 30, 20, 100],
})

m = registrations.merge(deposits, on="player_id", how="left")
print(m["bonus_credited"].sum())   # 40, а не 20!
```

Игрок 1 встретился 3 раза, поэтому его бонус `10` посчитался трижды. Реальная сумма выданных бонусов — `20` (по 10 на двух игроков), а отчёт показывает `40`. Это и есть «утроённый GGR» из ворнинга ниже, только в миниатюре. Лекарство то же: либо агрегируй депозиты **до** merge (свернуть к одной строке на игрока), либо считай метрики на уровне правильной гранулярности (`drop_duplicates` по `player_id` перед суммой того, что относится к игроку).

### Типичные ошибки

- **`inner` вместо `left`** — молча теряешь объекты без пары справа. Знаменатель конверсии схлопывается, доля растёт искусственно.
- **Ключ есть, но разного типа** — `player_id` строкой `"123"` слева и числом `123` справа не совпадут, получишь все `NaN` и пустой результат. Проверь `df.dtypes` по обе стороны.
- **Разный регистр / пробелы в строковом ключе** — `"UZ "` и `"UZ"` это разные ключи. Чисти ключ (`.str.strip()`) до merge.
- **Одноимённые столбцы не из ключа** — если в обеих таблицах есть `created_at`, pandas добавит суффиксы `_x` / `_y`. Управляй через `suffixes=("_reg", "_dep")`, иначе запутаешься.
- **`concat(axis=1)` при несовпадающих индексах** — горизонтальный concat склеивает **по индексу**, а не по порядку строк. Если индексы разные, получишь дыры с `NaN`. Для соединения по смыслу почти всегда лучше `merge` по явному ключу, чем `concat(axis=1)`.
- **Забыл `ignore_index=True` в `concat(axis=0)`** — индексы двух кусков склеятся как есть, получишь дубли индекса (две строки с индексом 0), и последующий `.loc[0]` вернёт обе.

!!! warning "Главный тихий баг аналитики — разъезд строк после join"

    Если ключ в правой таблице не уникален (на одного игрока несколько строк), `left join` **размножит** строки слева. Был миллион игроков — стало три миллиона, и твоя сумма GGR утроилась. Лекарство: всегда указывай `validate=` (`"one_to_one"`, `"one_to_many"`, `"many_to_one"`) — pandas упадёт с ошибкой, если ожидание не сошлось. И всегда сверяй `len(df)` до и после merge.

!!! tip "Как читать validate"

    `validate=` описывает кардинальность ключа «слева_к_справа». `"one_to_one"` — ключ уникален с обеих сторон (досье ↔ профиль). `"one_to_many"` — слева уникален, справа может повторяться (игрок → его депозиты). `"many_to_one"` — наоборот (депозиты → справочник стран). Если реальность не сошлась с ожиданием, pandas бросит `MergeError` ещё до того, как кривые цифры уйдут в дашборд. Это дешёвая страховка — пиши её всегда, даже когда «уверен».

??? success "Ответы"

    1. `merge` соединяет таблицы по общему ключу (как SQL join); `concat` склеивает встык — строки под строки или столбцы рядом, без сопоставления по ключу.
    2. `left` — взять всех слева (например, всех игроков) и подтянуть данные справа, не теряя тех, у кого справа совпадения нет.
    3. Если на один ключ слева приходится несколько строк справа, `left join` размножает строки слева; суммы (например GGR) раздуваются кратно.
    4. Указывать `validate=` (`"one_to_one"`, `"one_to_many"`, `"many_to_one"`) — pandas упадёт при нарушении; и сверять `len(df)` до и после merge.
