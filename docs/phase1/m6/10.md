# M6.10 · pivot и melt: широкий и длинный формат

### Интуиция

Одни и те же данные можно хранить "широко" и "длинно". Широкий формат — это таблица для человека: строки — игроки, столбцы — месяцы, в ячейках — депозиты. Длинный формат — таблица для машины: три столбца `игрок | месяц | депозит`, по строке на каждое наблюдение.

`pivot` (и `pivot_table`) превращает длинный формат в широкий — раскладывает значения категории по столбцам. `melt` делает обратное — "расплавляет" широкую таблицу в длинную. Длинный формат — родной для группировок, агрегаций и графиков; широкий — для отчётов и матриц.

Полезная мысленная метафора: длинный формат — это "журнал событий" (каждая строка = одно наблюдение, как запись в логе), а широкий — это "сводная матрица" (перекрёстная таблица, как в Excel pivot table). Машина любит журнал: `groupby`, `merge`, фильтры, seaborn-графики — всё это работает построчно и ждёт, что каждая переменная живёт в своём столбце ("tidy data" по Хэдли Уикхему: одна переменная — один столбец, одно наблюдение — одна строка). Человек любит матрицу: глазами проще читать "игрок × месяц", чем простыню из тысяч строк.

Практическое правило: **храни и считай в длинном, показывай в широком**. Сначала собираешь длинный tidy-датасет, агрегируешь его через `groupby`, и только финальный результат разворачиваешь `pivot_table` для отчёта или heatmap.

### Формально

```python
# длинный -> широкий
wide = df.pivot_table(
    index="player_id", columns="month", values="deposit", aggfunc="sum", fill_value=0
)

# широкий -> длинный
long = wide.reset_index().melt(
    id_vars="player_id", var_name="month", value_name="deposit"
)
```

Разбор параметров `pivot_table`:

- `index` — что станет строками (ключ наблюдения).
- `columns` — категория, чьи значения разъедутся по столбцам.
- `values` — какую числовую колонку класть в ячейки.
- `aggfunc` — как схлопывать дубли (`"sum"`, `"mean"`, `"count"`, можно список или dict разных функций на разные колонки).
- `fill_value` — чем заполнить пустые сочетания (например игрок не платил в этом месяце → `0`, а не `NaN`).

`pivot_table` отличается от `pivot` тем, что умеет агрегировать при дублях ключей (через `aggfunc`), тогда как голый `pivot` требует уникальности и падает на дублях. На практике почти всегда нужен `pivot_table`. Это тот самый инструмент, которым строится cohort-матрица retention (строки — когорта, столбцы — недели жизни).

### Разбор на числах

Пусть есть длинный лог депозитов (заметьте: у игрока 1 в январе две транзакции — это дубль ключа):

```python
import pandas as pd

df = pd.DataFrame({
    "player_id": [1, 1, 1, 2, 2],
    "month":     ["2026-01", "2026-01", "2026-02", "2026-01", "2026-03"],
    "deposit":   [100, 50, 200, 300, 80],
})

wide = df.pivot_table(
    index="player_id", columns="month",
    values="deposit", aggfunc="sum", fill_value=0
)
print(wide)
```

```text
month      2026-01  2026-02  2026-03
player_id
1              150      200        0
2              300        0       80
```

Что произошло: для игрока 1 в `2026-01` две строки (100 и 50) схлопнулись в `150` суммой — это и есть работа `aggfunc`. Голый `df.pivot(index="player_id", columns="month", values="deposit")` на этих данных упал бы с `ValueError: Index contains duplicate entries, cannot reshape`. Пустые сочетания (игрок 1 в марте, игрок 2 в феврале) стали `0` благодаря `fill_value`.

Обратная операция возвращает нас в длинный формат (но уже агрегированный — исходные две январские строки игрока 1 не восстановятся, информация о разбивке потеряна при свёртке):

```python
long = wide.reset_index().melt(
    id_vars="player_id", var_name="month", value_name="deposit"
)
```

### Гемблинг-пример: cohort retention за один pivot

Классическая задача — матрица удержания. Длинный датасет: на каждого игрока строка с когортой регистрации и неделей активности.

```python
cohort = pd.pivot_table(
    activity,
    index="reg_week",        # когорта по неделе регистрации
    columns="week_number",   # 0, 1, 2, ... недель с момента регистрации
    values="player_id",
    aggfunc="nunique",       # уникальные активные игроки
    fill_value=0,
)

# доля от размера когорты (нулевая неделя = 100%)
retention = cohort.divide(cohort[0], axis=0)
```

Строки — когорты, столбцы — возраст в неделях, ячейки — сколько игроков из когорты были активны на N-й неделе. Делением на нулевой столбец получаем проценты удержания — готовый материал для heatmap. Тот же приём в продуктовой аналитике: `index` — дата установки приложения, `columns` — day_n, `values` — DAU.

### Пограничные случаи и типичные ошибки

- **`pivot` падает на дублях.** Самая частая ошибка. Если не уверен в уникальности `index × columns` — бери `pivot_table`. Голый `pivot` только переформировывает, не агрегирует.
- **`NaN` против `0`.** Без `fill_value` отсутствующие сочетания станут `NaN`. Для денег и счётчиков почти всегда нужен `0`, иначе сломается последующая арифметика и суммы. Но осторожно: `0` и "нет данных" — это семантически разные вещи; не заполняй нулём там, где пропуск значимый.
- **MultiIndex в колонках.** Если передать список в `values` или несколько `aggfunc`, столбцы станут иерархическими (MultiIndex) — потом неудобно обращаться. Часто после pivot делают `.reset_index()` и/или `.columns = [...]` для плоских имён.
- **`melt` и широкие столбцы.** При `melt` укажи `id_vars` (что оставить как есть) — всё остальное расплавится. Забудешь `reset_index()` перед `melt` на результате pivot — потеряешь `player_id`, который сидит в индексе.
- **`pivot_table` тихо отбрасывает строки с `NaN` в ключах** (`index`/`columns`). Если в `month` есть пропуски — эти наблюдения исчезнут из матрицы. Проверяй `df["month"].isna().sum()` заранее.
- **Порядок столбцов.** После pivot столбцы сортируются по значению категории. Для "недель жизни" это ок (0,1,2,...), но для строковых меток вроде месяцев "2026-01" сортировка лексикографическая — совпадает с хронологией только при ISO-формате дат.
