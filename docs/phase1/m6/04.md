# M6.4 · Чтение данных: CSV, Parquet, SQL

### Интуиция

Данные приходят в разных форматах. CSV — это текст, который читает человек и любой инструмент, но он медленный, не хранит типы и раздут по размеру. Parquet — бинарный колоночный формат: он сжат, помнит типы, и читается в разы быстрее, потому что можно вытащить только нужные столбцы, не читая всю таблицу. SQL — это когда данные лежат в базе, и ты тащишь их запросом.

Правило 2026 года простое: **для обмена с людьми — CSV, для хранения и работы — Parquet, для боевых данных — тащи из базы или из колоночного хранилища.**

Чтобы прочувствовать разницу, представь таблицу ставок казино на 50 млн строк и 40 столбцов (player_id, amount, game_id, bet_ts, и ещё 36 полей). Тебе для отчёта по GGR нужны три столбца: `player_id`, `amount`, `bet_ts`. Что происходит физически:

- **CSV**: диск читает все 50 млн × 40 значений как текст, парсер прогоняет каждый байт, угадывает где число, где дата, где строка. На ноутбуке это легко 30–90 секунд и несколько гигабайт оперативки — и 90% прочитанного ты сразу выбросишь.
- **Parquet**: движок открывает метаданные файла, видит схему, читает с диска только три нужные колонки в уже бинарном виде. Это секунды и сотни мегабайт.

Разница не косметическая — это разница между «запустил и пошёл за кофе» и «получил ответ сразу». На реальных продуктовых данных именно выбор формата чаще всего определяет, тормозит твой пайплайн или летает.

### Формально

```python
import pandas as pd

df_csv = pd.read_csv("bets.csv", parse_dates=["bet_ts"])
df_parquet = pd.read_parquet("bets.parquet")  # типы и схема внутри файла

from sqlalchemy import create_engine
engine = create_engine("postgresql+psycopg://user:pass@host:5432/yohoho")
df_sql = pd.read_sql("SELECT player_id, amount, bet_ts FROM bets WHERE bet_ts >= '2026-03-01'", engine)
```

Почему Parquet быстрее CSV на порядок: CSV-парсер обязан прочитать каждый байт, угадать типы, распарсить числа из текста. Parquet хранит данные уже в бинарном колоночном виде со сжатием (обычно Snappy или Zstd) и метаданными о схеме. Колоночность означает, что `read_parquet(..., columns=["player_id", "amount"])` физически не читает остальные столбцы с диска.

### CSV: где он молча портит данные

CSV не хранит типы, поэтому при чтении pandas каждый раз заново их *угадывает* — и угадывает плохо. Типичные грабли на боевых данных:

- **Даты остаются строками.** Без `parse_dates=["bet_ts"]` колонка `bet_ts` придёт как `object` (строка), и любая фильтрация `>= '2026-03-01'` будет работать как лексикографическое сравнение текста, а не как сравнение дат. Поэтому в примере выше `parse_dates` указан явно.
- **ID-шники превращаются в числа и теряют нули.** `player_id = "00731"` прочитается как целое `731`. Лечится `dtype={"player_id": "string"}`.
- **Смешанные типы в колонке.** Если в столбце amount где-то затесалась строка `"н/д"`, вся колонка станет `object`, и арифметика сломается. Лечится `na_values=["н/д", "", "NULL"]`.
- **Тихое переполнение памяти.** `read_csv` читает всё целиком. Для больших файлов есть `chunksize=...` (читать кусками-итератором) или `usecols=[...]` (взять только нужные столбцы ещё на этапе парсинга).

```python
df = pd.read_csv(
    "bets.csv",
    parse_dates=["bet_ts"],
    dtype={"player_id": "string"},
    na_values=["н/д", "NULL"],
    usecols=["player_id", "amount", "bet_ts"],  # не парсим лишние 37 столбцов
)
```

Главная мысль: **каждое чтение CSV — это акт доверия к угадайке.** Поэтому CSV хорош для разовой передачи человеку и плох как формат хранения пайплайна.

!!! tip "Predicate и projection pushdown"

    "Pushdown" — это когда фильтр и выбор столбцов "проталкиваются" вниз, к самому чтению с диска, чтобы не грузить лишнее в память. Polars и DuckDB делают это автоматически: если ты в ленивом запросе написал `filter(amount > 0).select("player_id")`, движок прочитает с диска только нужные столбцы и только нужные группы строк. Pandas так не умеет — он сначала читает всё в память, потом фильтрует.

### Pushdown на пальцах: projection vs predicate

Два слова в названии — два разных среза экономии:

- **Projection pushdown (по столбцам)** — «не читай колонки, которые мне не нужны». Это умеет даже pandas через `columns=` / `usecols=`, а Polars/DuckDB делают сами по плану запроса.
- **Predicate pushdown (по строкам)** — «не читай группы строк, которые не пройдут фильтр». Parquet хранит файл блоками (row groups), и в метаданных каждой группы лежит min/max по колонкам. Если ты фильтруешь `bet_ts >= '2026-03-01'`, а у row group max-дата `'2026-02-15'`, движок *пропускает весь блок целиком*, даже не распаковывая его. Это называется row-group skipping и работает тем лучше, чем сильнее данные отсортированы по колонке фильтра.

Пример на Polars — ленивое чтение, где оба pushdown'а срабатывают автоматически:

```python
import polars as pl

ggr = (
    pl.scan_parquet("bets.parquet")          # ленивый план, файл ещё не читается
      .filter(pl.col("bet_ts") >= pl.date(2026, 3, 1))  # predicate pushdown
      .select(["player_id", "amount"])        # projection pushdown
      .group_by("player_id")
      .agg(pl.col("amount").sum().alias("ggr"))
      .collect()                              # только здесь происходит реальное чтение
)
```

`scan_parquet` строит план, а не читает данные; `collect()` запускает его, и Polars читает с диска только две колонки и только те row groups, что проходят по дате. На CSV `scan_csv` тоже умеет projection pushdown, но не умеет row-group skipping — у текстового файла нет блочных метаданных.

### SQL: фильтруй на сервере, а не в Python

Когда данные в базе, ключевое правило то же самое — **не тащи лишнее**. У `read_sql` есть антипаттерн-ловушка:

```python
# ПЛОХО: тащим всю таблицу в память, потом фильтруем в pandas
df = pd.read_sql("SELECT * FROM bets", engine)
df = df[df["bet_ts"] >= "2026-03-01"]

# ХОРОШО: фильтр и выбор столбцов делает Postgres, по сети идёт только нужное
df = pd.read_sql(
    "SELECT player_id, amount, bet_ts FROM bets WHERE bet_ts >= '2026-03-01'",
    engine,
)
```

`SELECT *` + фильтр в pandas — это ручной отказ от pushdown: база отдаёт миллионы строк по сети, а ты выбрасываешь 90% уже в Python. Сервер БД почти всегда отфильтрует быстрее (у него индексы), и по сети поедут килобайты вместо гигабайт. По сути SQL `WHERE`/`SELECT` — это и есть predicate/projection pushdown, только выраженный языком запросов.

Ещё две частые ошибки:

- **Хардкод значений в строку запроса** вместо параметров — это и SQL-инъекция, и потеря типов. Используй параметры: `pd.read_sql("... WHERE bet_ts >= %(d)s", engine, params={"d": "2026-03-01"})`.
- **Чтение огромного результата целиком** — у `read_sql` тоже есть `chunksize=`, чтобы обрабатывать результат итеративно.

### Пограничные случаи и типичные ошибки

- **Parquet «забыл» типы?** Этого не бывает — схема внутри файла. Если типы поехали, значит файл *записали* из уже испорченного DataFrame. Чини на этапе записи, а не чтения.
- **«Parquet всегда быстрее» — нет.** На крошечном файле (тысячи строк) разница незаметна, а накладные расходы на распаковку и метаданные могут сделать его даже чуть медленнее. Parquet выигрывает на масштабе и при выборке части столбцов; если ты всё равно читаешь все 40 колонок целиком, преимущество projection исчезает.
- **CSV открывается в Excel, Parquet — нет.** Если файл уходит коллеге без Python/BI-инструмента, CSV остаётся правильным выбором, несмотря на все минусы. Это и есть смысл правила «для людей — CSV».
- **Кодировка и разделитель.** Российские выгрузки часто приходят в `cp1251` и с `;` вместо `,`: `pd.read_csv(..., encoding="cp1251", sep=";")`. Parquet от этого избавлен полностью — там нет ни кодировки текста, ни разделителей.
