# M6.14 · Method chaining: стиль 2026

### Интуиция

Method chaining — это когда весь пайплайн пишется одной цепочкой операций, каждая на своей строке, без промежуточных переменных `df1`, `df2`, `df_tmp`. Читается сверху вниз как рецепт: "взять данные, отфильтровать, добавить столбец, сгруппировать, посчитать". Никаких разбросанных по коду мутаций — одно состояние течёт через цепочку.

Сравни два стиля на одной и той же задаче — "GGR по каналам за март":

```python
# Стиль "лесенка из временных переменных" — как НЕ надо
df1 = df[df["bet_ts"] >= "2026-03-01"]
df1["ggr"] = df1["bets"] - df1["payouts"]          # SettingWithCopyWarning
df2 = df1.groupby("channel", as_index=False)
df3 = df2.agg(players=("player_id", "nunique"), ggr=("ggr", "sum"))
report = df3.sort_values("ggr", ascending=False)
```

```python
# Method chaining — то же самое одной цепочкой
report = (
    df
    .query("bet_ts >= '2026-03-01'")
    .assign(ggr=lambda d: d["bets"] - d["payouts"])
    .groupby("channel", as_index=False)
    .agg(players=("player_id", "nunique"), ggr=("ggr", "sum"))
    .sort_values("ggr", ascending=False)
)
```

Главная мысль: в первом варианте у тебя в памяти болтаются `df1`, `df2`, `df3`, которые больше нигде не нужны, плюс прямое присваивание `df1["ggr"] = ...` режет срез исходного фрейма и провоцирует `SettingWithCopyWarning`. Во втором варианте промежуточных имён нет вообще — каждый метод возвращает новый фрейм, который тут же передаётся следующему методу.

### Почему работает: каждый метод возвращает DataFrame

Цепочка возможна ровно потому, что большинство методов pandas (`query`, `assign`, `groupby().agg`, `sort_values`, `rename`, `merge`, `drop`, `fillna`) возвращают новый объект DataFrame, а не `None`. Поэтому к результату сразу можно "прицепить" следующий вызов через точку.

Ловушка: методы с `inplace=True` возвращают `None` и ломают цепочку.

```python
report = df.sort_values("ggr", inplace=True).head(10)
# AttributeError: 'NoneType' object has no attribute 'head'
# потому что sort_values(inplace=True) вернул None
```

Правило 2026: `inplace=True` не использовать — он не даёт выигрыша по памяти (это миф), мешает chaining и помечен на удаление в pandas. Всегда работай через присваивание результата.

### Формально

```python
report = (
    df
    .query("bet_ts >= '2026-03-01'")
    .assign(ggr=lambda d: d["bets"] - d["payouts"])
    .groupby("channel", as_index=False)
    .agg(players=("player_id", "nunique"), ggr=("ggr", "sum"))
    .sort_values("ggr", ascending=False)
)
```

Почему это стиль 2026: цепочка не мутирует исходные данные (важно при Copy-on-Write в pandas 3.0), легко комментируется построчно, легко модифицируется (закомментировал шаг — увидел промежуток), и она структурно идентична тому, как пишется Polars и SQL. Минус — отладка середины цепочки требует метода `.pipe()` или временного разрыва. Для отладки удобен `.pipe(lambda d: (print(d.shape), d)[1])`.

### Скобки и точки — механика синтаксиса

Зачем внешние круглые скобки `( ... )`? В Python нельзя просто перенести `.method()` на новую строку — интерпретатор подумает, что выражение закончилось. Скобки включают "неявное продолжение строки": всё внутри `()` читается как одно выражение, и точку можно ставить в начале каждой строки.

```python
# Не сработает — SyntaxError на второй строке
report = df
    .query("bet_ts >= '2026-03-01'")

# Работает — всё обёрнуто в скобки
report = (
    df
    .query("bet_ts >= '2026-03-01'")
)
```

Стилевая конвенция: точка в начале строки (а не в конце предыдущей). Так каждая строка читается как "что делаем", и закомментировать любой шаг — это закомментировать ровно одну строку.

### `assign` и `lambda d`: почему именно так

В `assign` новый столбец часто задают через `lambda d: ...`, где `d` — это фрейм в текущем состоянии цепочки. Это критично: если в цепочке выше уже был `query`, то `lambda d` увидит уже отфильтрованный фрейм, а не исходный `df`.

```python
report = (
    df
    .query("country == 'BR'")
    .assign(ggr=lambda d: d["bets"] - d["payouts"])   # d — это уже только BR
    .assign(margin=lambda d: d["ggr"] / d["bets"])     # d — фрейм с уже посчитанным ggr
)
```

Типичная ошибка новичка — сослаться на исходную переменную `df` вместо `d`:

```python
# ПЛОХО: df["bets"] берётся из НЕотфильтрованного фрейма,
# длины не совпадут после query -> мусор или ошибка выравнивания индексов
.assign(ggr=lambda d: df["bets"] - df["payouts"])
```

Через `lambda d` цепочка остаётся самодостаточной: переименуешь `df` или вставишь шаг выше — ничего не сломается, потому что каждый шаг ссылается только на "вход этого шага".

### Отладка середины цепочки

Главный практический минус chaining: нельзя поставить точку останова "между" методами и посмотреть промежуточный фрейм. Решения:

```python
def peek(d, label=""):
    print(label, d.shape)
    return d

report = (
    df
    .query("bet_ts >= '2026-03-01'")
    .pipe(peek, "после фильтра")        # печатает форму, возвращает d дальше
    .assign(ggr=lambda d: d["bets"] - d["payouts"])
    .pipe(peek, "после ggr")
    .groupby("channel", as_index=False)
    .agg(ggr=("ggr", "sum"))
)
```

`.pipe(func)` передаёт текущий фрейм в `func` и продолжает цепочку результатом. Ключевое требование — функция внутри `pipe` должна **вернуть DataFrame**, иначе цепочка дальше сломается. Поэтому `peek` обязательно делает `return d`, а не просто печатает.

### Пример из продуктовой аналитики (в долларах)

Допустим, нужен отчёт: топ-каналы по марже, только там где было хотя бы 50 игроков, с конвертацией GGR в доллары по курсу.

```python
RATE = 0.018  # рублей -> долларов, грубо

report = (
    bets
    .query("bet_ts >= '2026-03-01' and status == 'settled'")
    .assign(ggr_rub=lambda d: d["stake"] - d["win"])
    .groupby("channel", as_index=False)
    .agg(
        players=("player_id", "nunique"),
        ggr_rub=("ggr_rub", "sum"),
    )
    .query("players >= 50")                              # фильтр уже по агрегату
    .assign(ggr_usd=lambda d: (d["ggr_rub"] * RATE).round(0))
    .sort_values("ggr_usd", ascending=False)
    .head(10)
)
```

Здесь видно силу стиля: первый `query` фильтрует строки ставок, второй `query` фильтрует уже сгруппированный результат (каналы с <50 игроков). Это два разных уровня агрегации в одной читаемой цепочке — на "лесенке" из временных переменных это превратилось бы в `df5`, `df6` без всякого смысла.

!!! warning "Где chaining реально мешает"

    Цепочка хороша для линейного потока. Если результат шага нужен **дважды** в разных ветках (например, и общий итог, и разбивка), цепочку придётся разорвать — храни промежуточный фрейм в осмысленной переменной `monthly` и строй две цепочки от неё. Не растягивай цепочку искусственно ради "одной строки".

### Чек-лист стиля 2026

- Внешние скобки `( ... )`, точка в начале каждой строки.
- Никаких `inplace=True` — всегда присваивание результата.
- Новые столбцы через `.assign(col=lambda d: ...)`, ссылаться на `d`, не на исходный `df`.
- Промежуточные переменные — только когда фрейм реально нужен повторно.
- Отладка через `.pipe(peek)`, функция обязана вернуть DataFrame.

!!! note "Проверь себя"

    Почему `df.sort_values("x", inplace=True).head()` падает с ошибкой, а `df.sort_values("x").head()` — нет? Подсказка: что возвращает метод с `inplace=True`.
