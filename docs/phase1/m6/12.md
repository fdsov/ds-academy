# M6.12 · Пропуски: fillna, dropna

### Интуиция

Реальные данные дырявые: игрок зарегистрировался, но не задепозитил — значит депозит "отсутствует". Пропуск (`NaN`/`null`) — это не ноль и не ошибка, это "значения нет". Что с ним делать — зависит от смысла. Иногда `NaN` депозита логично заменить на 0 (не платил — значит ноль). Иногда строку с пропуском в ключевом поле надо удалить. А иногда `NaN` нужно сохранить, чтобы он не искажал среднее.

Ключевая мысль, которую новички упускают: **пропуск — это информация, а не дефект**. Само наличие `NaN` часто говорит о механизме, который его породил. Три классических механизма из статистики (важно различать, потому что от них зависит правильное действие):

- **MCAR** (Missing Completely At Random) — пропуск не зависит ни от чего. Например, у части строк потерялась запись из-за сбоя выгрузки. Удаление таких строк не вносит смещение.
- **MAR** (Missing At Random) — пропуск зависит от других наблюдаемых полей. Например, `deposit` чаще пропущен у игроков из стран, где платёжный шлюз не пишет лог. Зная `country`, мы можем восстановить осмысленно.
- **MNAR** (Missing Not At Random) — пропуск зависит от самого скрытого значения. Например, игрок не указал возраст именно потому, что он несовершеннолетний. Здесь любое заполнение опасно — пропуск сам по себе предиктор.

На пальцах: прежде чем нажать `fillna` или `dropna`, спроси себя — "почему здесь дырка?". Ответ диктует действие.

### Формально

```python
df["deposit"].isna().sum()           # сколько пропусков
df["deposit"] = df["deposit"].fillna(0)   # заменить осмысленным нулём
df = df.dropna(subset=["player_id"])      # удалить строки без ключа
df["country"] = df["country"].fillna("unknown")
```

Тонкость: `NaN` в арифметике "заразен" — `5 + NaN = NaN`, но агрегаты pandas по умолчанию его игнорируют (`mean` считает по непустым). Это значит, что заменив `NaN` на 0 перед усреднением, ты исказишь среднее (ноль попадёт в знаменатель). Решай на уровне смысла, а не механически. Современный nullable-тип (`Int64`, Arrow-бэкенд) отличает "настоящий пропуск" от NaN с плавающей точкой, что чище.

### Разбор на числах: почему 0 ломает среднее

Пусть у нас четыре игрока, депозиты: `[100, 200, NaN, NaN]` (два задепозитили, два — нет).

- `mean()` по умолчанию: `(100 + 200) / 2 = 150`. Pandas считает только по двум непустым — это "средний депозит среди плативших" (ARPPU-логика).
- После `fillna(0)`: `[100, 200, 0, 0]`, `mean = 300 / 4 = 75`. Теперь это "средний депозит на всех зарегистрированных" (ARPU-логика).

Оба числа верны — но отвечают на **разные вопросы**. 150 — это ARPPU (среднее на платящего), 75 — ARPU (среднее на пользователя). Ошибка новичка — механически сделать `fillna(0)`, а потом отчитаться "средний депозит игрока 75" в контексте, где менеджер ждал 150. Один вызов `fillna` молча подменил метрику.

```python
import pandas as pd
import numpy as np

s = pd.Series([100, 200, np.nan, np.nan])
print(s.mean())              # 150.0  — среднее по плативших (ARPPU)
print(s.fillna(0).mean())    # 75.0   — среднее по всем (ARPU)
print(s.isna().mean())       # 0.5    — доля пропусков (50%)
```

`s.isna().mean()` — удобный приём: доля пропусков в колонке. Если она высокая (скажем, >50%), это сигнал разобраться, а не молча затыкать.

### Когда что применять

- **`fillna(0)`** — когда пропуск семантически равен нулю. Депозит отсутствует = игрок не платил = 0. Количество ставок отсутствует = 0 ставок. Это безопасно, если NaN действительно означает "событие не произошло".
- **`fillna("unknown")` / категорию-заглушку** — для категориальных полей (`country`, `traffic_source`). Не выкидывай строку из-за неизвестной страны — заведи явную категорию "unknown", она сама по себе аналитически полезна.
- **`fillna(median/mean)`** — для непрерывных признаков, где нужно сохранить строку для модели, а пропуск случаен (MCAR/MAR). Медиана устойчивее к выбросам, чем среднее. Важно: считать медиану нужно **только по train-части**, иначе утечка (data leakage) — тестовая статистика просочится в обучение.
- **`dropna(subset=[...])`** — когда без поля строка бессмысленна. Нет `player_id` — невозможно ничего посчитать, строка — мусор.
- **Оставить `NaN`** — когда заполнение исказит метрику (см. пример с ARPPU выше) или когда сам факт пропуска — признак (MNAR).

### Тонкости и подводные камни

`dropna` без аргументов удаляет строку, где пропущено **хоть одно** поле — на широкой витрине с десятками колонок это легко выкосит 90% данных. Почти всегда нужен явный `subset`:

```python
df.dropna()                          # удалит строку, если NaN в ЛЮБОЙ колонке — опасно
df.dropna(subset=["player_id"])      # удалит только если нет ключа — то, что нужно
df.dropna(thresh=5)                  # оставит строки, где есть хотя бы 5 непустых значений
df.dropna(axis=1)                    # удаляет КОЛОНКИ с пропусками, не строки
```

Ещё ловушки:

- **`fillna` не меняет DataFrame на месте по умолчанию** — возвращает новый объект. Нужно присвоить обратно: `df["x"] = df["x"].fillna(0)`. Просто вызвать `df["x"].fillna(0)` без присваивания — изменения потеряются.
- **`NaN != NaN`** — сравнение `df["x"] == np.nan` всегда даёт `False`. Для проверки пропусков только `isna()` / `notna()`, никогда `== np.nan`.
- **Пустая строка `""` и пробелы — это не `NaN`**. Если в выгрузке пропуск пришёл как `""` или `"NULL"`-строкой, `isna()` его не увидит. Сначала привести: `df["x"].replace(["", "NULL"], np.nan)`.
- **`groupby` молча выбрасывает строки с `NaN` в ключе группировки** (по умолчанию `dropna=True`). Если группируешь по `country`, а у части игроков страна пустая — они исчезнут из отчёта, и сумма по группам не сойдётся с общей.

!!! warning "Заполнил — задокументируй"

    Любое заполнение пропусков меняет смысл метрики. После `fillna` всегда фиксируй в коде или комментарии, что именно сделал и почему — иначе через месяц никто (включая тебя) не вспомнит, почему "средний депозит" вдруг просел вдвое.

### Проверь себя

У тебя витрина игроков, в колонке `deposit` половина значений `NaN`. Менеджер просит "средний депозит игрока". Какое одно уточнение нужно задать перед тем, как считать?

Ответ: на платящего или на всех зарегистрированных. От этого зависит — игнорировать `NaN` (ARPPU, `mean()` напрямую) или заменить на 0 (ARPU, `fillna(0).mean()`). Это разные числа и разные ответы.
