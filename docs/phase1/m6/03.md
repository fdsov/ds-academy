# M6.3 · Series и DataFrame: атомы pandas

### Интуиция

`Series` — это один столбец с подписями. Представь колонку в Excel, но у каждой ячейки есть имя (метка строки), а не только номер. `DataFrame` — это таблица: набор `Series`, склеенных по общему индексу строк. Если ты держишь в голове "Series = столбец, DataFrame = таблица из столбцов с общим индексом" — ты уже понял базу.

Ключевое отличие от Excel и от обычного списка Python: у строк есть **индекс** — отдельная ось меток. Это не просто номера 0, 1, 2. Индексом может быть дата, ID игрока, что угодно. Индекс — главная фишка pandas и одновременно главный источник путаницы для новичков.

Чтобы прочувствовать, зачем индекс нужен, представь две выгрузки из разных систем гемблинг-проекта. Первая — депозиты по игрокам, вторая — их ставки (bets). В обеих игроки идут в **разном порядке**, и в каждой свой набор пользователей. Если бы это были просто списки Python, тебе пришлось бы вручную сопоставлять, где чей депозит и чья ставка. В pandas обе выгрузки получают индекс по `player_id`, и когда ты делаешь `deposits + bets`, библиотека сама **выравнивает строки по меткам** — депозит игрока `u2` складывается именно со ставкой `u2`, независимо от порядка. Это называется alignment, и это то, ради чего вообще существует индекс.

### Формально

`Series` — это одномерный размеченный массив: пара (значения, индекс), где значения хранятся в типизированном массиве (поверх NumPy или, в новых версиях, Apache Arrow), а индекс — это отдельный объект `Index`, отображающий метки в позиции. `DataFrame` — двумерная размеченная структура: словарь из `Series` с общим объектом-индексом строк и объектом-индексом столбцов.

```python
import pandas as pd

deposits = pd.Series([100, 250, 50], index=["u1", "u2", "u3"], name="deposit")
deposits["u2"]  # доступ по метке -> 250
deposits.iloc[1]  # доступ по позиции -> 250
```

Важно сразу зафиксировать: метка и позиция — это разные вещи. `deposits["u2"]` ищет по метке, `deposits.iloc[1]` — по порядковому номеру. Когда метки — это случайно числа (0, 1, 2), новички их путают, и это порождает классические баги.

### Alignment на пальцах: численный пример

Это центральный механизм, ради которого стоит разобраться с индексом. Сложим два `Series` с пересекающимися, но не одинаковыми метками.

```python
import pandas as pd

deposits = pd.Series([100, 250, 50], index=["u1", "u2", "u3"], name="deposit")
bets     = pd.Series([300, 80, 40],  index=["u2", "u3", "u4"], name="bet")

deposits + bets
```

Результат:

```
u1      NaN
u2    550.0
u3     90.0
u4      NaN
dtype: float64
```

Разбор по шагам:

- `u2` есть в обоих: `250 + 300 = 550`. `u3` есть в обоих: `50 + 40 = 90`.
- `u1` есть только в депозитах, `u4` — только в ставках. Для них пара не нашлась, и pandas ставит `NaN` (не ошибку!) — результат стал `float64`, потому что `NaN` это float.

Это и есть выравнивание по индексу: pandas **не складывает позицию с позицией**, он складывает метку с меткой. Если бы это были обычные списки, `[100,250,50] + [300,80,40]` дали бы конкатенацию, а не поэлементную сумму, и уж точно проигнорировали бы, кто есть кто. Если хочешь, чтобы непересёкшиеся игроки давали не `NaN`, а, скажем, 0 (например, "ставки нет = 0"), используй `deposits.add(bets, fill_value=0)` — тогда `u1` станет `100`, а `u4` — `40`.

Типичная ошибка новичка здесь: он берёт две выгрузки, у которых строки в разном порядке, и делает `df1["x"] + df2["y"]`, ожидая "по строкам сверху вниз". А pandas выравнивает по индексу — и если индексы не совпадают, получаются массовые `NaN` "из ниоткуда". Лечится либо `reset_index(drop=True)` (когда ты осознанно хочешь позиционное сложение), либо приведением обоих к общему индексу.

### loc против iloc: подробно

`.loc[...]` — выбор по **меткам**, `.iloc[...]` — по **позициям** (как в обычном Python-списке). Две ловушки, на которых спотыкаются почти все:

- **Срезы по меткам включают правую границу.** `s.loc["u1":"u3"]` вернёт строки `u1, u2, u3` — конец включён. А `s.iloc[0:3]` вернёт позиции 0, 1, 2 — конец, как в Python, не включён.
- **Когда индекс числовой, `s[0]` неоднозначен.** Поэтому правило для продакшен-кода: явно пиши `.loc` для меток и `.iloc` для позиций, не полагайся на "голые" квадратные скобки.

```python
gp = pd.Series([0, 1200, 1850], index=[10, 20, 30])  # индекс — числа, но это МЕТКИ
gp.loc[20]   # 1200 — по метке 20
gp.iloc[1]   # 1200 — по позиции 1
gp.loc[10:30]  # метки 10..30 включительно -> все три строки
```

!!! note "dtype — это не мелочь"

    У каждого столбца есть тип (`int64`, `float64`, `object`, `category`, `datetime64[ns]`, `boolean`). Тип определяет и скорость, и корректность. В pandas 2.x столбец строк по умолчанию хранился как `object` — медленный питоновский тип, а Arrow был опцией; в pandas 3.0 дефолтом стал выделенный строковый dtype на PyArrow, так что строки сразу быстрее и компактнее. Для повторяющихся строк (канал, страна, статус) почти всегда лучше `category`. Бэкенд Arrow (`dtype_backend="pyarrow"`) даёт честные nullable-типы и заметно быстрее старого NumPy-бэкенда.

!!! note "category на реальных данных: почему это экономит память"

    Допустим, у тебя 2 млн строк событий и колонка `country` с 30 уникальными значениями ("RU", "KZ", "BR"...). Как `object` каждая ячейка — отдельная Python-строка, это сотни мегабайт. Как `category` pandas хранит словарь из 30 уникальных строк **один раз**, а в самих 2 млн ячеек держит маленькие целочисленные коды (`int8`/`int16`), указывающие в этот словарь. Память падает в десятки раз, а `groupby("country")` работает быстрее, потому что группировка идёт по целым числам, а не по строкам. Правило: повторяющаяся строка с небольшим числом уникальных значений (канал, страна, статус, валюта) — кандидат в `category`.

!!! question "Проверь себя"

    1. Чем `Series` отличается от `DataFrame` одним предложением?
    2. Что вернёт `deposits["u2"]` и что `deposits.iloc[1]` — по чему ищет каждый из них?
    3. Почему столбец строк лучше держать как `category`, а не как `object`?
    4. Что такое индекс в pandas и чем он отличается от обычной нумерации строк?
    5. Что вернёт `pd.Series([100,250], index=["u1","u2"]) + pd.Series([1,2], index=["u2","u3"])` и почему появляются `NaN`?

??? success "Ответы"

    1. `Series` — один размеченный столбец (значения + индекс); `DataFrame` — таблица из нескольких `Series` с общим индексом строк.
    2. `deposits["u2"]` ищет по метке, `deposits.iloc[1]` — по позиции; в данном примере оба вернут 250.
    3. `object` хранит полноценные питоновские строки (медленно и тяжело); `category` хранит словарь уникальных значений один раз, а в ячейках держит компактные целочисленные коды — меньше памяти, быстрее группировки.
    4. Индекс — отдельная ось меток строк (дата, ID игрока, что угодно), а не просто номера 0,1,2; по нему работает выравнивание и доступ через `loc`.
    5. Совпадает только метка `u2`: `250 + 1 = 251`. `u1` и `u3` есть лишь в одном из `Series`, для них пара не нашлась — pandas ставит `NaN` (выравнивание идёт по меткам, не по позициям).

!!! example "Мини-задание"

    Создай `Series` из 5 депозитов с индексом-строками вида `"u1"..."u5"`. Достань один элемент по метке и тот же по позиции через `.iloc`. Затем сделай из словаря двух таких `Series` один `DataFrame` и выведи `.dtypes`. Проверь, что у обоих столбцов общий индекс. 3 минуты.

!!! example "Мини-задание 2: почувствуй alignment"

    Создай `deposits` с индексом `["u1","u2","u3"]` и `bets` с индексом `["u2","u3","u4"]`. Сложи их обычным `+` и посмотри, где появились `NaN`. Затем повтори через `deposits.add(bets, fill_value=0)` и сравни результат. Объясни себе, почему в первом случае `u1` и `u4` дали `NaN`, а во втором — реальные числа. 3 минуты.

```quiz
Что произойдёт при сложении двух Series с частично разными метками индекса?
[ ] pandas сложит значения по позициям сверху вниз, игнорируя метки
[x] pandas выровняет по меткам; непересёкшиеся метки дадут NaN
[ ] pandas выбросит ошибку из-за несовпадения длины
> Сложение в pandas идёт по выравниванию меток (alignment), а не по позициям; метки без пары дают NaN.
---
Чем `.loc` отличается от `.iloc`?
[x] `.loc` выбирает по меткам индекса, `.iloc` — по позициям
[ ] `.loc` для столбцов, `.iloc` для строк
[ ] `.loc` быстрее, `.iloc` точнее, в остальном одинаковы
> `.loc` работает с метками (включая правую границу в срезах), `.iloc` — с целочисленными позициями, как в обычном списке.
---
Почему для колонки `country` с 30 уникальными значениями на 2 млн строк лучше `category`, чем `object`?
[ ] `category` округляет значения и теряет точность, экономя место
[x] `category` хранит уникальные строки один раз, а в ячейках держит компактные целочисленные коды
[ ] `category` автоматически удаляет дубликаты строк из таблицы
> `category` = словарь уникальных значений + целочисленные коды в ячейках: меньше памяти и быстрее группировки, данные не теряются.
---
Что вернёт `deposits.add(bets, fill_value=0)`, если метка `u4` есть только в `bets`?
[ ] NaN, потому что у `u4` нет пары в deposits
[x] значение `u4` из bets, так как отсутствующая сторона подставляется как 0
[ ] ошибку, потому что fill_value работает только с числовым индексом
> `fill_value=0` подставляет 0 для отсутствующей стороны перед сложением, поэтому `u4` даёт своё значение из bets, а не NaN.
---
Чем индекс в pandas отличается от обычной нумерации строк 0,1,2?

[ ] Ничем, индекс — это всегда подряд идущие целые числа
[ ] Индекс хранит только типы столбцов, а не метки строк
> Индекс — самостоятельная ось меток произвольной природы; именно по нему pandas выравнивает данные и ищет через `.loc`.
```
