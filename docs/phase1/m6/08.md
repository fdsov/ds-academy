# M6.8 · groupby: split-apply-combine во всю глубину

### Интуиция

`groupby` — самая важная операция в аналитике. Идея в трёх словах: **разбей, примени, собери** (split-apply-combine). Разбей таблицу на группы (например, по игроку или по каналу), к каждой группе примени функцию (сумма, среднее, своя логика), собери результаты обратно в таблицу.

"Сколько каждый игрок задепозитил?" — группируем по `player_id`, к каждой группе применяем сумму `deposit`, собираем. "Какой средний чек по каналам?" — группируем по `channel`, применяем среднее. Это паттерн, который покрывает 80% аналитических задач.

На пальцах: представь стопку чеков из кассы. Ты раскладываешь их в кучки — по кассиру, по дню, по способу оплаты (это split). Потом по каждой кучке считаешь итог калькулятором (apply). Потом записываешь итоги в одну табличку "кассир → выручка" (combine). `groupby` — это та же самая сортировка чеков по кучкам, только машина делает её за миллисекунды на миллионе строк.

Важно сразу заметить: ключ группировки становится **индексом** результата. После `df.groupby("channel")["deposit"].sum()` ты получаешь `Series`, где индекс — это названия каналов, а не привычный `0, 1, 2, ...`. Если индекс мешает (например, перед записью в БД или мержем), верни ключ обратно в столбец через `as_index=False` или `.reset_index()`:

```python
# индекс = channel
df.groupby("channel")["deposit"].sum()

# channel остаётся обычным столбцом
df.groupby("channel", as_index=False)["deposit"].sum()
```

### Формально

Парадигму split-apply-combine формализовал Хэдли Уикхем (2011). Pandas `groupby` реализует её буквально: объект `GroupBy` — это ленивое отображение "ключ группы → набор позиций строк". Реальное вычисление происходит, когда ты вызываешь агрегацию.

```python
# базовая агрегация
g = df.groupby("channel")["deposit"].sum()

# несколько агрегатов сразу, с именами столбцов (named aggregation)
report = df.groupby("channel").agg(
    players=("player_id", "nunique"),
    total_dep=("deposit", "sum"),
    avg_dep=("deposit", "mean"),
    ggr=("ggr", "sum"),
)

# группировка по нескольким ключам
df.groupby(["country", "channel"])["ggr"].sum()
```

Named aggregation (`agg(имя=(столбец, функция))`) — предпочтительный стиль для отчётов: ты сразу задаёшь читаемые имена столбцов, и порядок гарантирован. Сравни с устаревшим словарным синтаксисом `agg({"deposit": "sum"})`, который даёт менее предсказуемые имена и хуже читается в код-ревью.

Разберём числовой пример. Пусть лог депозитов такой:

| player_id | channel | deposit | ggr |
|-----------|---------|---------|-----|
| 1 | seo | 100 | 40 |
| 2 | seo | 50 | 20 |
| 1 | ppc | 200 | 90 |
| 3 | ppc | 100 | 10 |

Тогда `df.groupby("channel").agg(players=("player_id","nunique"), total_dep=("deposit","sum"), ggr=("ggr","sum"))` даст:

| channel | players | total_dep | ggr |
|---------|---------|-----------|-----|
| ppc | 2 | 300 | 100 |
| seo | 2 | 150 | 60 |

Обрати внимание на `nunique` для `players`: по каналу `ppc` две строки, но и два **разных** игрока (1 и 3), поэтому `players=2`. Если бы игрок 1 депнул в `ppc` дважды, `count` дал бы 3, а `nunique` всё равно 2. На практике путаница `count` против `nunique` — источник завышенного числа "уникальных" игроков в отчётах: `count` считает строки (события), `nunique` — уникальные значения.

### Три режима применения: agg, transform, apply

Здесь начинается то, что отличает джуна от мидла. У `groupby` есть три способа "применить" функцию, и они дают результаты разной формы.

`agg` (агрегация) — сворачивает каждую группу в **одно число**. На входе группа из N строк, на выходе одна строка. Результат короче исходной таблицы.

`transform` — применяет функцию к группе, но возвращает результат **той же длины, что группа**. Используется, когда нужно вернуть групповую характеристику обратно к каждой строке. Классика — "доля депозита игрока от суммы по его стране":

```python
df["country_total"] = df.groupby("country")["deposit"].transform("sum")
df["share_in_country"] = df["deposit"] / df["country_total"]
```

Здесь `transform("sum")` посчитал сумму по стране и "размножил" её обратно на каждую строку этой страны. Через `agg` так нельзя — он схлопнул бы таблицу.

Ключевое свойство `transform`: результат **выровнен по исходному индексу**, поэтому его можно напрямую присвоить как новый столбец `df["..."] = ...` без всякого мержа. Это главная причина, почему `transform` так удобен для фичеинжиниринга. Сравни два пути для той же задачи "доля депозита в стране":

```python
# путь через transform — одна строка, выравнивание автоматом
df["share"] = df["deposit"] / df.groupby("country")["deposit"].transform("sum")

# путь через agg + merge — длиннее и легче ошибиться
totals = df.groupby("country")["deposit"].sum().rename("country_total")
df = df.merge(totals, on="country")
df["share"] = df["deposit"] / df["country_total"]
```

`transform` принимает и встроенные имена (`"sum"`, `"mean"`, `"max"`), и свои функции. Частый продуктовый кейс — z-score внутри группы (нормировать метрику игрока относительно его страны), или ранг депозита внутри когорты:

```python
# нормировка ставки игрока относительно средней по его VIP-уровню
g = df.groupby("vip_level")["bet"]
df["bet_z"] = (df["bet"] - g.transform("mean")) / g.transform("std")
```

Подвох: функция внутри `transform` должна возвращать либо скаляр (тогда он размножается), либо объект ровно той же длины, что группа. Если вернуть что-то другой длины — будет ошибка выравнивания.

`apply` — самый гибкий и самый опасный. Принимает произвольную функцию, получающую целый sub-DataFrame группы, и может вернуть что угодно. Мощно, но медленно.

```python
def second_purchase_gap(g):
    g = g.sort_values("ts")
    if len(g) < 2:
        return pd.NA
    return (g["ts"].iloc[1] - g["ts"].iloc[0]).days

df.groupby("player_id").apply(second_purchase_gap)
```

Когда `apply` действительно оправдан: логика затрагивает **несколько столбцов сразу и зависит от порядка строк внутри группы** — например, разница между первым и вторым депозитом, длина серии проигрышей, сложный фильтр "оставить только игроков, у которых GGR вырос между сессиями". Такое не выражается одним встроенным агрегатом, потому что нужен доступ к целому sub-DataFrame.

Типичная ошибка с `apply` — менять переданный sub-DataFrame на месте. Внутри функции работай с копией (как `g = g.sort_values(...)`, которая уже возвращает новый объект), иначе рискуешь поймать `SettingWithCopyWarning` или испортить исходные данные. И ещё нюанс свежих версий pandas: ключ группировки теперь по умолчанию не попадает внутрь `g` повторно как столбец-дубль — поведение `group_keys` стало строже, поэтому не закладывайся на старые привычки.

### Почему apply бывает медленным

`apply` вызывает питоновскую функцию для **каждой группы** по отдельности — это интерпретируемый цикл под капотом, теряющий всю векторизацию. Если групп миллион, ты делаешь миллион питоновских вызовов. Встроенные агрегаты (`sum`, `mean`, `count`, `nunique`) реализованы в C/Cython и работают на порядки быстрее.

Грубая интуиция масштаба: на логе в 5 млн строк с 1 млн игроков `groupby("player_id")["deposit"].sum()` отрабатывает за доли секунды, потому что весь проход по группам идёт в скомпилированном коде. Тот же расчёт через `apply(lambda g: g["deposit"].sum())` запускает миллион отдельных питоновских вызовов и легко уходит в десятки секунд или минуты — отсюда и заявленная разница до 100x. На дашборде, который пересчитывается каждые 5 минут, это разница между "успели" и "запрос отвалился по таймауту".

Есть промежуточный приём между `apply` и встроенным агрегатом: если своя функция всё же нужна, попробуй сначала свернуть данные встроенными средствами до маленького числа групп, и только потом применять Python-логику к уже агрегированному результату. Чем меньше итераций питоновского цикла — тем быстрее.

!!! warning "Правило производительности"

    Прежде чем писать `apply`, спроси: можно ли выразить это через встроенный агрегат, `transform`, или векторную операцию? В 90% случаев — да. `apply` — последнее средство, а не первое. На больших данных разница между `agg("sum")` и `apply(lambda g: g.sum())` достигает 100x.

!!! question "Проверь себя"

    1. Расшифруй split-apply-combine на примере "средний депозит по каналам".
    2. Чем `agg` отличается от `transform` по форме результата?
    3. Почему "доля депозита игрока от суммы по его стране" делается через `transform`, а не через `agg`?
    4. Почему `apply` на большом числе групп медленный, и что пробовать перед ним?

??? success "Ответы"

    1. Split — разбить таблицу по `channel`; apply — к каждой группе применить `mean` депозита; combine — собрать результаты в таблицу "канал → средний депозит".
    2. `agg` сворачивает группу в одно число (результат короче исходной таблицы); `transform` возвращает результат той же длины, что группа, — групповая характеристика "размножается" обратно на каждую строку.
    3. Нужно вернуть сумму по стране к каждой строке этой страны (длина сохраняется); `agg` схлопнул бы таблицу до одной строки на страну.
    4. `apply` вызывает питоновскую функцию для каждой группы — интерпретируемый цикл без векторизации. Сначала ищи встроенный агрегат (`sum`, `mean`, `nunique`), `transform` или векторную операцию.
