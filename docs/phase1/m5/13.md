# M5.13 · Случайность: современный Generator API

### Что устарело

Старый интерфейс `np.random.seed(42)` плюс `np.random.rand()` — это глобальное состояние. Оно считается legacy с NumPy 1.17 (2019) и категорически не рекомендуется в 2026: глобальный seed создаёт скрытые зависимости между частями кода, плохо работает в параллельных процессах и затрудняет воспроизводимость.

Разберём, почему «глобальное состояние» — это реальная боль, а не теоретическая придирка. Когда ты пишешь `np.random.seed(42)`, ты настраиваешь один общий генератор, спрятанный внутри модуля `numpy.random`. Любой вызов `np.random.rand()`, `np.random.normal()`, любой `df.sample()` из pandas, любая сторонняя библиотека (sklearn, scipy) — все тянут числа из этого же единственного источника и сдвигают его состояние.

На пальцах: представь общий блокнот, куда все пишут по очереди. Ты записал «начать со страницы 42», но если коллега между твоими вызовами тоже что-то записал — твоя следующая страница уже не та, которую ты ждал. Итог — два классических бага:

- **Скрытая зависимость от порядка.** Добавил в начало скрипта один лишний `np.random.rand()` для дебага — и всё разбиение игроков на A/B поехало, хотя ты «ничего важного не менял».
- **Параллелизм ломается.** Запустил симуляцию в 8 процессах через `multiprocessing` — каждый процесс наследует одно и то же глобальное состояние и генерирует **одинаковые** «случайные» числа. Симуляция Монте-Карло по сути считает одно и то же 8 раз.

### Современный подход

`np.random.default_rng(seed)` создаёт изолированный объект-генератор `Generator` на базе алгоритма PCG64. Вся случайность идёт через него. Никакого глобального состояния — каждый генератор независим.

```python
rng = np.random.default_rng(seed=42)

rng.random(5)                       # равномерно [0,1)
rng.integers(0, 100, size=10)       # целые в [0,100)
rng.normal(loc=0, scale=1, size=5)  # нормальное
rng.exponential(scale=100, size=5)  # экспоненциальное (хорошо для сумм ставок)
rng.choice(["A", "B"], size=10, p=[0.7, 0.3])  # категории с вероятностями
rng.shuffle(arr)                    # перемешать на месте

# воспроизводимое A/B-разбиение игроков
players = np.arange(10_000)
rng = np.random.default_rng(2026)
groups = rng.integers(0, 2, size=players.size)  # 0=control, 1=treatment
```

Главная идея: вместо одного скрытого генератора ты явно держишь `rng` в руках и передаёшь его туда, где нужна случайность. Это делает код самодокументируемым — видно, что именно зависит от seed.

### Численный пример: симуляция оборота ставок

Допустим, средняя ставка на слотах — 50 рублей, распределена экспоненциально (много мелких, редко крупные). Смоделируем оборот 1000 ставок и убедимся в воспроизводимости.

```python
import numpy as np

rng = np.random.default_rng(seed=7)
bets = rng.exponential(scale=50, size=1000)   # средняя ≈ 50
print(round(bets.sum()))     # детерминированный итог при seed=7
print(round(bets.mean(), 1)) # ≈ 50, но не ровно — это выборка

# повторный запуск с тем же seed даёт ровно те же числа
rng2 = np.random.default_rng(seed=7)
bets2 = rng2.exponential(scale=50, size=1000)
print(np.array_equal(bets, bets2))   # True
```

Экспоненциальное распределение здесь не случайно: для денежных величин (сумма ставки, депозит, время до следующего захода) оно реалистичнее нормального, потому что не уходит в минус и имеет «тяжёлый правый хвост» — редкие крупные значения. У него $\text{mean} = \text{scale}$, поэтому `scale=50` задаёт среднюю ставку $50.

### Независимые потоки: spawn вместо ручных seed

Когда нужно много **независимых** генераторов (например, для параллельной симуляции), не выдумывай руками seed=1, seed=2, seed=3 — близкие seed могут давать скоррелированные потоки. Используй штатный механизм `spawn`:

```python
parent = np.random.default_rng(2026)
children = parent.spawn(8)            # 8 статистически независимых генераторов
totals = [c.exponential(50, 1000).sum() for c in children]
```

Это и есть правильный способ накормить случайностью 8 процессов Монте-Карло — каждый получает свой независимый поток, а весь набор остаётся воспроизводимым от одного родительского seed.

### Типичные ошибки

- **Создавать `rng` внутри цикла.** `for ...: rng = np.random.default_rng(42)` — каждая итерация заново инициализирует генератор и выдаёт одно и то же число. Создавай `rng` один раз снаружи.
- **Смешивать старый и новый API.** `np.random.seed(0)` никак не влияет на объект `default_rng` — это разные источники. Не жди, что глобальный seed «починит» воспроизводимость кода на `Generator`.
- **Звать `rng.shuffle` ожидая возврат.** `shuffle` перемешивает на месте и возвращает `None`. Для нового массива нужен `rng.permutation(arr)`.
- **Путать `integers(0, 100)` с включением 100.** Верхняя граница по умолчанию исключается: диапазон $[0, 100)$. Нужен включающий — `endpoint=True`.

!!! warning "Воспроизводимость экспериментов"

    В дизайне A/B (см. модуль про эксперименты) seed фиксирует разбиение и симуляции. Но seed гарантирует одинаковость только в рамках одной версии NumPy и одной платформы — не считай его криптографической гарантией. Для продакшн-сплита пользователей используй детерминированный хэш user_id, а не RNG.

!!! note "Почему хэш, а не RNG, для боевого сплита"

    RNG раздаёт группы по порядку обращения: первый игрок — первое число, второй — второе. Но в проде игроки приходят асинхронно, через разные сервера и реплики, и «порядок» не определён. Детерминированный хэш `hash(user_id + salt) % 100 < 50` присваивает группу как чистую функцию от самого `user_id`: один и тот же игрок всегда попадёт в одну и ту же группу, независимо от порядка, сервера и перезапусков. Это и есть свойство, которого RNG в распределённой системе не даёт.

!!! question "Проверь себя"

    1. Чем `np.random.default_rng` лучше старого `np.random.seed` + `np.random.rand`?
    2. Какой алгоритм лежит в основе `Generator` и почему важна изолированность генератора?
    3. Почему для продакшн-сплита пользователей в A/B лучше детерминированный хэш `user_id`, а не RNG?

??? success "Ответы"

    1. Нет глобального состояния: каждый генератор изолирован, нет скрытых зависимостей между частями кода, надёжнее работа в параллельных процессах и воспроизводимость.
    2. PCG64. Изолированность означает, что случайность в одной части кода не влияет на другую — нет общего глобального seed.
    3. seed гарантирует одинаковость лишь в рамках одной версии NumPy и одной платформы; хэш `user_id` даёт стабильное и переносимое разбиение, не зависящее от порядка вызовов RNG.
