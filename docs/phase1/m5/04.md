# M5.4 · ndarray, dtype, shape

Три атрибута полностью описывают любой NumPy-массив: **dtype** (тип одного элемента), **shape** (размер по каждой оси) и сам буфер данных. Если ты держишь эти три вещи в голове, поведение массива перестаёт быть магией — broadcasting, ошибки переполнения, расход памяти становятся предсказуемыми. Разберём каждый атрибут глубоко, через деньги и реальные ошибки аналитика.

### Что такое ndarray «на пальцах»

`ndarray` (N-dimensional array) — это не список Python. Это **один непрерывный блок памяти** плюс метаданные, которые объясняют, как этот блок читать. Метаданные: `dtype` (сколько байт занимает элемент и как его интерпретировать), `shape` (логическая форма), `strides` (на сколько байт шагнуть, чтобы перейти к следующему элементу по каждой оси).

Ключевое следствие: все элементы массива имеют **один и тот же тип**. Список Python `[1, 2.5, "x"]` хранит разнотипные объекты по ссылкам; ndarray хранит однородные значения подряд. Именно однородность даёт скорость (CPU читает память линейно, векторизованные операции) и компактность (нет overhead на объекты Python — у списка каждый `int` это полноценный объект ~28 байт, в массиве `int64` — ровно 8 байт).

```python
import numpy as np

py_list = [1, 2, 3, 4, 5]
arr = np.array(py_list)
print(type(arr))        # <class 'numpy.ndarray'>
print(arr.dtype)        # int64 — NumPy сам вывел тип
print(arr.strides)      # (8,) — шаг 8 байт между соседними int64
```

### dtype: почему это важно для денег

dtype определяет не только скорость, но и корректность. В гемблинг-аналитике это критично: депозиты, ставки, балансы — это деньги, и ошибка типа здесь стоит дорого.

`float64` (он же `double`) даёт примерно 15-16 значащих десятичных цифр. Для сумм до миллионов это надёжно. Но `float32` экономит память вдвое ценой всего ~7 цифр точности — складывая миллионы мелких ставок в `float32`, ты накопишь ощутимую ошибку округления. Для матриц признаков в ML `float32` часто оправдан (память и скорость GPU важнее), для финансового свода — нет.

Целочисленные типы могут переполняться молча. `int32` держит значения примерно до $2.1 \times 10^9$. Если ты считаешь суммарный объём ставок (bet volume) в копейках за год по крупному казино, ты легко это превысишь, и сумма «завернётся» в отрицательное число без всякого предупреждения.

```python
import numpy as np

a = np.array([2_000_000_000], dtype=np.int32)
print(a + a)  # [-294967296] — тихое переполнение

b = np.array([2_000_000_000], dtype=np.int64)
print(b + b)  # [4000000000] — корректно
```

#### Почему float теряет точность: разобранный пример

Проблема `float32` не в «больших числах», а в **сумме большого и маленького**. У float фиксировано число значащих цифр, поэтому когда к большому накопленному итогу прибавляют маленькую ставку, младшие разряды просто не помещаются и отбрасываются.

```python
import numpy as np

# Накопили 10 млн оборота, прибавляем ставку $0.01
total32 = np.float32(10_000_000.0)
print(total32 + np.float32(0.01))   # 10000000.0 — копейка ПОТЕРЯНА

total64 = np.float64(10_000_000.0)
print(total64 + np.float64(0.01))   # 10000000.01 — сохранилась

# Складываем миллион мелких ставок в float32 — копится дрейф
bets = np.full(1_000_000, 0.07, dtype=np.float32)
print(bets.sum())                   # ≈ 69995.7 вместо 70000.0
print(bets.astype(np.float64).sum())# ≈ 70000.0
```

Практический вывод для финансового свода: либо считай деньги в целых единицах (копейки/центы как `int64`), либо суммируй в `float64`. `int64` держит до $\approx 9.2 \times 10^{18}$ — это $\approx 92$ квадриллиона рублей в копейках, переполнить такое в гемблинге невозможно.

#### Типичные ошибки с dtype

- **Молчаливый downcast при создании.** `np.array([1, 2, 3])` даст `int64`, но `np.array([1, 2, 3], dtype=np.int8)` тихо уложит всё в один байт — и `100 + 100` там даст $-56$.
- **Деление целых.** `np.array([5], dtype=np.int64) / 2` всегда возвращает `float64` (`2.5`), а `// 2` — целочисленное (`2`). Не путай при расчёте средней ставки.
- **`astype` округляет к нулю, а не математически.** `np.array([2.9]).astype(np.int64)` даёт `2`, а не `3`. Для денег используй `np.round` перед приведением.
- **NaN живёт только во float.** `np.nan` нельзя положить в целочисленный массив — пропуски в депозитах требуют `float64`, либо nullable-типов pandas.

Проверить и сменить тип:

```python
deposits = np.array([100, 250, 5000])     # int64 по умолчанию
print(deposits.dtype)                      # int64
deposits_money = deposits.astype(np.float64)  # явное приведение
print(deposits_money.dtype)                # float64
```

### shape и количество осей

`shape` — кортеж. Скаляр имеет shape `()`, вектор `(n,)`, матрица `(rows, cols)`, тензор `(d0, d1, d2)` и так далее. Число осей называется `ndim`. Важно различать `(n,)` (одномерный вектор) и `(n, 1)` или `(1, n)` (двумерные «столбец» и «строка») — они ведут себя по-разному в broadcasting и матричном умножении, и это источник вечной путаницы.

```python
x = np.array([1.5, 2.0, 3.0])
print(x.shape, x.dtype, x.ndim)   # (3,) float64 1
print(x.size, x.nbytes)           # 3 24  — 3 элемента по 8 байт
```

#### Как читать shape в аналитике

Самая частая форма в работе аналитика — `(n_rows, n_cols)`: строки это игроки/события, столбцы это признаки. Первая ось (`axis=0`) идёт «вниз по строкам», вторая (`axis=1`) — «вдоль признаков». Отсюда и логика агрегаций: `arr.sum(axis=0)` суммирует по игрокам (итог на каждый признак), `arr.sum(axis=1)` — по признакам (итог на каждого игрока).

```python
# 3 игрока, 2 метрики: [депозит, оборот]
m = np.array([[100,  900],
              [250, 1800],
              [ 50,  120]], dtype=np.float64)
print(m.shape)            # (3, 2) — 3 строки, 2 столбца
print(m.sum(axis=0))      # [400. 2820.] — суммарный депозит и оборот
print(m.sum(axis=1))      # [1000. 2050.  170.] — итог на каждого игрока
```

#### Почему `(n,)` ≠ `(n, 1)` — пограничный случай

Это рушит broadcasting у новичков чаще всего. Одномерный `(3,)` и столбец `(3, 1)` при сложении дают **матрицу 3×3**, а не вектор — NumPy «растягивает» оси по правилам broadcasting.

```python
col = np.array([[1], [2], [3]])   # shape (3, 1) — столбец
row = np.array([10, 20, 30])      # shape (3,)   — вектор

print((col + row).shape)          # (3, 3) — неожиданно для многих!
print(col + row)
# [[11 21 31]
#  [12 22 32]
#  [13 23 33]]
```

Если хотел поэлементное сложение трёх чисел — оба операнда должны быть `(3,)`. Превратить столбец обратно в вектор: `col.ravel()` или `col.reshape(-1)` (форма `(3,)`). Превратить вектор в столбец: `row.reshape(-1, 1)` или `row[:, None]`. Привычка явно печатать `.shape` перед операцией экономит часы отладки.
