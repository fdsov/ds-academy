# M5.7 · Векторизация: почему циклы медленны

### Интуиция

Цикл Python выполняется интерпретатором: на каждой итерации он разбирает байткод, проверяет типы, управляет объектами. Векторизованная операция спускает весь цикл в скомпилированный код C, который работает с плотной памятью. Разница принципиальна — это не «оптимизация», а смена слоя исполнения.

Разберём «на пальцах», за что именно платит Python-цикл на каждой из миллиона итераций. Когда ты пишешь `bets[i] - payouts[i]`, интерпретатор делает примерно следующее:

1. Берёт `i`, упакованное в полноценный объект `int` (а не в «голое» число).
2. Вызывает `bets.__getitem__(i)` — это поиск метода, проверка границ, создание нового Python-объекта `float` под результат (в куче, с заголовком и счётчиком ссылок).
3. То же самое для `payouts[i]`.
4. Вызывает `float.__sub__`, снова проверяет типы обоих операндов, создаёт ещё один объект-результат.
5. Кладёт результат в список, обновляет счётчики ссылок, при необходимости запускает сборщик мусора.

Каждый из этих шагов — десятки машинных инструкций. NumPy же знает заранее: «весь массив — это `float64`, лежащий непрерывным блоком в памяти». Поэтому он один раз спускается в C и делает миллион вычитаний в тесном цикле, где на итерацию приходится буквально пара инструкций процессора, без создания объектов и без проверок типов. Плюс современный CPU применяет SIMD (одна инструкция обрабатывает сразу 4-8 чисел) — этого Python-цикл не умеет в принципе.

Ключевая мысль: дело не в том, что «C быстрее Python». Дело в том, что векторизация **убирает накладные расходы на каждую отдельную операцию**, выполняя их пакетом над однородными данными.

### Бенчмарк

Посчитаем GGR как «ставки минус выплаты» по миллиону записей двумя способами.

```python
import numpy as np
import time

n = 1_000_000
rng = np.random.default_rng(42)
bets = rng.exponential(100, n)
payouts = rng.exponential(95, n)

# Способ 1: чистый Python-цикл
t0 = time.perf_counter()
ggr_loop = [bets[i] - payouts[i] for i in range(n)]
t_loop = time.perf_counter() - t0

# Способ 2: векторизация NumPy
t0 = time.perf_counter()
ggr_vec = bets - payouts
t_vec = time.perf_counter() - t0

print(f"loop: {t_loop*1000:.1f} ms")
print(f"vec:  {t_vec*1000:.1f} ms")
print(f"ускорение: {t_loop / t_vec:.0f}x")
```

На типичной машине цикл занимает ~150-300 мс, векторизация — ~2-5 мс. Ускорение в 50-100 раз. И это простейшая операция; на сложных выражениях разрыв ещё больше, потому что промежуточные результаты тоже остаются в C.

### Что это значит в деньгах и масштабе

Цифры из бенчмарка кажутся абстрактными, пока их не привязать к работе. Допустим, ты считаешь GGR не один раз, а внутри ежедневного ETL, который гоняет 200 млн строк транзакций (типичный объём для гемблинг-проекта средней руки):

- Цикл: ~200-300 мс на миллион строк → на 200 млн строк это **40-60 секунд только на одно вычитание**. А таких операций в пайплайне десятки (комиссия, бонусы, налог, маржа). Легко набегает 20-30 минут CPU-времени на ровном месте.
- Векторизация: те же 200 млн строк за **0.5-1 секунду**. Весь пайплайн укладывается в секунды.

В облаке это прямые деньги: воркер, который крутится 30 минут вместо 30 секунд, стоит в 60 раз дороже за тот же отчёт. На дашборде, который пересчитывается каждые 5 минут, разница между «успели» и «очередь растёт» — это и есть граница между рабочим продуктом и сломанным.

### Разобранный пример: считаем NGR векторизованно

Возьмём более реалистичную формулу. NGR (Net Gaming Revenue) = ставки − выплаты − бонусы − комиссия провайдера. Сравним «наивный» цикл и векторную версию.

```python
import numpy as np

n = 1_000_000
rng = np.random.default_rng(0)
bets = rng.exponential(100, n)
payouts = rng.exponential(95, n)
bonuses = rng.exponential(5, n)
provider_fee_rate = 0.15  # 15% комиссии с GGR

# Векторизованный NGR одной строкой
ggr = bets - payouts
ngr = ggr - bonuses - np.maximum(ggr, 0) * provider_fee_rate

print(f"суммарный NGR: ${ngr.sum():,.0f}")
print(f"средний NGR на игрока: ${ngr.mean():.2f}")
```

Обрати внимание на `np.maximum(ggr, 0)` — комиссия берётся только с положительного GGR (когда казино в плюсе по сессии). Это типичная бизнес-логика «если..., то...», которую новички пытаются записать через `if` внутри цикла. Векторно она выражается через `np.maximum`, `np.where` или булевы маски — без единого Python-цикла. Все промежуточные массивы (`ggr`, произведение) живут в C, и финальный `.sum()` тоже считается в C.

!!! tip "Правило большого пальца"

    Если ты пишешь `for` по элементам NumPy-массива — почти наверняка ты делаешь что-то не так. Спроси себя: можно ли выразить это через поэлементные операции, маски, агрегации или broadcasting? В 95% случаев — да.

### Пограничные случаи и типичные ошибки

Векторизация — не серебряная пуля. Где она не помогает или вредит:

- **Истинно последовательные вычисления.** Если значение шага зависит от предыдущего (накопительный баланс, скользящее состояние, рекуррентная формула), наивно векторизовать нельзя — нужны специальные функции вроде `np.cumsum`, `np.cumprod` или `np.frompyfunc`. Обычный поэлементный трюк тут не сработает, потому что нет независимости итераций.
- **`np.vectorize` — это НЕ векторизация.** Распространённая ловушка: люди видят функцию `np.vectorize` и думают, что она ускоряет код. На самом деле это просто удобная обёртка над Python-циклом — она почти не даёт прироста скорости. Настоящая векторизация — это операции над массивами целиком (`bets - payouts`), а не «обёрнутая поэлементная функция».
- **Память против скорости.** Векторизация создаёт промежуточные массивы целиком в памяти. Для 200 млн `float64` один такой массив — это ~1.6 ГБ. Цепочка из пяти операций может временно занять 8 ГБ. На больших данных это упирается в RAM, и тогда нужны чанки, `numexpr` или переход на Polars/Dask. Цикл медленный, но «бедный» по памяти; вектор быстрый, но «прожорливый».
- **Маленькие массивы.** На 10-100 элементах разница незаметна, а иногда цикл даже быстрее из-за накладных расходов на создание NumPy-объектов. Выигрыш проявляется на десятках тысяч элементов и больше.
- **Смешанные/object-типы.** Если в массиве лежат строки, словари или `dtype=object`, NumPy не может спустить операцию в C — он всё равно будет дёргать Python для каждого элемента, и преимущество исчезает. Векторизация раскрывается на однородных числовых типах (`int64`, `float64`).
