# M5.7 · Векторизация: почему циклы медленны

### Интуиция

Цикл Python выполняется интерпретатором: на каждой итерации он разбирает байткод, проверяет типы, управляет объектами. Векторизованная операция спускает весь цикл в скомпилированный код C, который работает с плотной памятью. Разница принципиальна — это не «оптимизация», а смена слоя исполнения.

### Бенчмарк

Посчитаем GGR как «ставки минус выплаты» по миллиону записей двумя способами.

```python
import numpy as np
import time

n = 1_000_000
rng = np.random.default_rng(42)
bets = rng.exponential(100, n)
payouts = rng.exponential(95, n)

# Способ 1: чистый Python-цикл
t0 = time.perf_counter()
ggr_loop = [bets[i] - payouts[i] for i in range(n)]
t_loop = time.perf_counter() - t0

# Способ 2: векторизация NumPy
t0 = time.perf_counter()
ggr_vec = bets - payouts
t_vec = time.perf_counter() - t0

print(f"loop: {t_loop*1000:.1f} ms")
print(f"vec:  {t_vec*1000:.1f} ms")
print(f"ускорение: {t_loop / t_vec:.0f}x")
```

На типичной машине цикл занимает ~150-300 мс, векторизация — ~2-5 мс. Ускорение в 50-100 раз. И это простейшая операция; на сложных выражениях разрыв ещё больше, потому что промежуточные результаты тоже остаются в C.

!!! tip "Правило большого пальца"

    Если ты пишешь `for` по элементам NumPy-массива — почти наверняка ты делаешь что-то не так. Спроси себя: можно ли выразить это через поэлементные операции, маски, агрегации или broadcasting? В 95% случаев — да.
