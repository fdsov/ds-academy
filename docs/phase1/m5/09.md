# M5.9 · Агрегации по осям

### Интуиция

`axis` отвечает на вопрос «вдоль какой оси схлопываем». Это самое контринтуитивное место для новичков. Запомни так: `axis` указывает ось, которая **исчезает**. `axis=0` схлопывает строки — остаётся результат по столбцам. `axis=1` схлопывает столбцы — остаётся по строкам.

Почему именно «исчезает», а не «остаётся»? Потому что агрегация — это сжатие. Когда ты складываешь числа вдоль какой-то оси, по этой оси у тебя перестаёт быть массив значений: вместо «семи дней» появляется «одно число — сумма за неделю». Значит, размерность вдоль `axis` стягивается до 1 и затем выкидывается. Именно поэтому форма результата — это форма исходного массива **без** того измерения, которое ты указал в `axis`.

Ещё одна опора для памяти: индекс оси совпадает с позицией в `shape`. У массива `(4, 7)` нулевая позиция в `shape` — это `4` (игроки), первая — `7` (дни). `axis=0` убирает позицию 0 (игроков), остаётся `(7,)`. `axis=1` убирает позицию 1 (дни), остаётся `(4,)`. Никакой магии — просто «вырежи указанный элемент из кортежа `shape`».

### Пример: таблица ставок

Пусть строки — игроки, столбцы — дни недели, значения — суммы ставок.

```python
bets = rng.exponential(100, size=(4, 7))   # 4 игрока, 7 дней

bets.sum()           # один скаляр — всего по всем
bets.sum(axis=0)     # shape (7,) — сумма по дням (игроки схлопнуты)
bets.sum(axis=1)     # shape (4,) — сумма по игрокам (дни схлопнуты)
bets.max(axis=1)     # макс ставка каждого игрока
bets.argmax(axis=1)  # в какой день у каждого был максимум
```

Доступны `sum`, `mean`, `std`, `var`, `min`, `max`, `argmin`, `argmax`, `prod`, `cumsum`, `median`, `percentile` и другие. У всех единая логика осей.

### Разбор форм на конкретных числах

Возьмём маленькую матрицу 2 игрока на 3 дня, чтобы всё посчитать в уме:

```python
import numpy as np

bets = np.array([
    [10, 20, 30],   # игрок 0
    [40, 50, 60],   # игрок 1
])
# shape == (2, 3)

bets.sum(axis=0)   # [50, 70, 90]   -> shape (3,): по каждому дню сложили обоих игроков
bets.sum(axis=1)   # [60, 150]      -> shape (2,): по каждому игроку сложили все дни
bets.sum()         # 210            -> скаляр: всё сразу
```

Проверь руками: `axis=0` идёт «сверху вниз» по столбцу — `10+40=50`, `20+50=70`, `30+60=90`. `axis=1` идёт «слева направо» по строке — `10+20+30=60`, `40+50+60=150`. Стрелка движения агрегации направлена **вдоль** исчезающей оси, а не поперёк неё — это частый источник путаницы.

### keepdims — когда ось не выкидывается, а становится единичной

Иногда форму удобно сохранить, чтобы потом сделать broadcasting обратно к исходному массиву. Для этого есть `keepdims=True`:

```python
day_totals = bets.sum(axis=0, keepdims=True)   # shape (1, 3), не (3,)
share = bets / day_totals                       # доля каждого игрока в дневном обороте
```

Без `keepdims` форма `(3,)` тоже сбродкастится с `(2, 3)`, но при `axis=1` форма `(2,)` уже **не** сбродкастится с `(2, 3)` напрямую — нужна `(2, 1)`. Поэтому при делении «нормируем на сумму по строкам» почти всегда пишут `keepdims=True`, иначе словишь `ValueError: operands could not be broadcast`. Это типичная ошибка при расчёте долей и вероятностей.

### Продуктовая иллюстрация

Матрица `bets` формы `(игроки, дни)` — это привычный срез данных в гемблинг-аналитике. Несколько метрик одной строкой:

```python
bets.sum(axis=1)              # суммарный недельный оборот каждого игрока (bet volume)
bets.mean(axis=0)             # средний дневной оборот по всей когорте (профиль недели)
bets.sum(axis=0).argmax()     # самый «горячий» день недели по обороту
(bets > 0).sum(axis=1)        # сколько активных дней у каждого игрока (частота сессий)
np.percentile(bets, 95, axis=1)  # P95 ставки каждого игрока — детектор хайроллеров
```

В долларах: если `bets.sum()` за неделю даёт \$84 000, а `bets.sum(axis=0)` показывает, что на пятницу приходится \$28 000 из них — треть недельного оборота в один день. Это сигнал, куда ставить бонусную механику. А `argmax(axis=1)` подскажет, у каких игроков пик пришёлся именно на пятницу — основа для таргетинга.

### Пограничные случаи и типичные ошибки

- **`argmax(axis=1)` возвращает индекс, а не значение.** Получишь номер дня (0–6), а не саму максимальную ставку. Чтобы достать значение — `bets.max(axis=1)`. Путать эти две функции — классика.
- **`argmax` при ничьей** возвращает индекс **первого** максимума, молча. Если два дня с одинаковым пиком — второй проигнорируется. Для аналитики дубликатов это важно помнить.
- **`mean`/`std` по пустой оси** дают `nan` и предупреждение `RuntimeWarning`, а `sum` по пустому — `0`. Если в данных есть пропуски (`np.nan`), обычный `sum` вернёт `nan` для всей строки — используй `np.nansum`, `np.nanmean` и т.п.
- **Отрицательный axis работает с конца.** `axis=-1` — это последняя ось (для `(4, 7)` то же, что `axis=1`). Удобно для кода, не зависящего от числа измерений.
- **`std` по умолчанию делит на N (ddof=0), а не N−1.** Для выборочного стандартного отклонения нужен `bets.std(axis=1, ddof=1)`. Если сравниваешь с pandas (там по умолчанию `ddof=1`) — получишь разные числа, и это не баг.
