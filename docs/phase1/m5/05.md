# M5.5 · Создание массивов

Прежде чем считать что-то быстро, массив надо создать. В NumPy есть целое семейство конструкторов — и выбор правильного экономит и память, и время, и нервы (особенно когда форма массива должна совпасть с другим массивом). Разберём базовый набор.

```python
np.array([1, 2, 3])                 # из списка
np.zeros((3, 4))                    # нули, форма (3,4)
np.ones(5)                          # единицы
np.full((2, 2), 7.0)                # заполнить значением
np.arange(0, 10, 2)                 # [0 2 4 6 8] — как range, но массив
np.linspace(0, 1, 5)                # 5 точек равномерно от 0 до 1 включительно
np.eye(3)                           # единичная матрица 3x3
np.empty((2, 3))                    # без инициализации (быстро, но мусор внутри)
```

## Как это читать «на пальцах»

Все конструкторы делятся на две группы по тому, что им передают:

- **Форма (shape)** — кортеж размеров: `(3, 4)` значит «3 строки, 4 столбца». Сюда относятся `zeros`, `ones`, `full`, `empty`, `eye`. Ты говоришь «дай мне болванку такого-то размера», а значения уже фиксированы (0, 1, или указанное).
- **Диапазон** — границы и сетка значений. Сюда `arange` (задаёшь шаг) и `linspace` (задаёшь количество точек). Ты говоришь «пройди от A до B вот так-то».

Важная тонкость: для одномерного массива форму можно писать просто числом — `np.ones(5)`, а для многомерного нужен кортеж — `np.zeros((3, 4))`. Частая ошибка новичка — `np.zeros(3, 4)` без внутренних скобок: NumPy воспримет второй аргумент как `dtype` и упадёт с ошибкой.

## arange против linspace — главный выбор

Это две разные ментальные модели одной задачи «дай сетку чисел на отрезке»:

- `np.arange(start, stop, step)` — «иди от `start` с шагом `step`, пока не дойдёшь до `stop`». Правый конец **не включается** (как у обычного `range`). Ты контролируешь *шаг*, но не знаешь заранее точное число элементов.
- `np.linspace(start, stop, num)` — «положи ровно `num` точек, равномерно, от `start` до `stop` **включительно**». Ты контролируешь *количество*, а шаг NumPy высчитает сам.

Проверим на числах:

```python
np.arange(0, 1, 0.25)     # [0.   0.25 0.5  0.75]      — 4 элемента, 1.0 НЕ вошла
np.linspace(0, 1, 5)      # [0.   0.25 0.5  0.75 1.0 ] — 5 элементов, 1.0 вошла
```

Обрати внимание: чтобы покрыть отрезок `[0, 1]` с шагом 0.25 точками *включительно*, нужно 5 точек, а не 4. Отсюда мнемоника: количество точек = количество интервалов + 1. У `linspace(0, 1, 5)` — 4 интервала по 0.25.

!!! warning "arange и дробный шаг"

    `np.arange` с дробным шагом (`np.arange(0, 1, 0.1)`) может из-за накопления ошибок float дать на один элемент больше или меньше, чем ты ждёшь. Когда нужно точное число точек на отрезке — используй `np.linspace`, где ты задаёшь количество, а не шаг.

Почему именно дробный шаг опасен: `0.1` не представляется точно в double, и после десятка сложений накопленная ошибка может сделать последнее значение чуть меньше `stop` — тогда оно «проскочит» внутрь и массив окажется длиннее на 1. Для целого шага (`np.arange(0, 10, 2)`) этой проблемы нет — целые складываются точно.

## empty — быстро, но с подвохом

`np.empty` не записывает в память никаких значений — просто выделяет блок и отдаёт как есть. Внутри окажется то, что в этой памяти лежало раньше (любой «мусор»). Это быстрее `zeros`, но использовать `empty` стоит только когда ты **сразу** заполнишь массив целиком сам:

```python
arr = np.empty((3, 1000))
for i in range(3):
    arr[i] = some_calculation(i)   # перезаписываем каждую строку — ок
```

Если же ты собираешься *накапливать* (например `+=`), бери `np.zeros` — иначе прибавишь к мусору. Типичный баг: завести аккумулятор через `empty`, начать суммировать и получить гигантские случайные числа.

## dtype — тип элементов задаётся при создании

У каждого массива есть единый тип элементов. По умолчанию из целых литералов получается `int64`, из дробных — `float64`. Часто тип лучше задать явно:

```python
np.zeros(5, dtype=np.int32)        # целочисленные нули
np.array([1, 2, 3], dtype=float)   # [1. 2. 3.] — приведёт к float64
np.full((2, 2), 7)                 # int, потому что 7 — целое
np.full((2, 2), 7.0)               # float, потому что 7.0 — дробное
```

Зачем это в аналитике: `float32` вместо `float64` вдвое сокращает память — на матрице фичей 10 млн строк × 50 столбцов это разница примерно 4 ГБ против 2 ГБ. Когда точности `float32` достаточно (а для большинства ML-фичей достаточно), это реальная экономия RAM.

## Пример из продуктовой аналитики

Допустим, мы моделируем кривую удержания (retention) когорты игроков по дням 1–30 и хотим сетку «ставок» бонуса для перебора в A/B:

```python
days = np.arange(1, 31)                 # дни 1..30 (31 НЕ включается) → 30 чисел
bonus_grid = np.linspace(0, 50, 11)     # бонусы $0, $5, $10, ... $50 — ровно 11 точек
payout = np.zeros((len(days), len(bonus_grid)))   # болванка под результат, заполним позже
```

Здесь видны обе модели сразу: `arange` для целочисленных дней (шаг 1, правый край режем сознательно), `linspace` для денежной сетки (нужно ровно 11 равномерных уровней от \$0 до \$50 включительно — шаг \$5), и `zeros` как заранее размеченный контейнер под будущие выплаты, форма которого выводится из длин двух осей.

## Частые ошибки

- `np.zeros(3, 4)` вместо `np.zeros((3, 4))` — забыли кортеж, второй аргумент уходит в dtype, ошибка.
- Ждать, что `np.arange(0, 10, 2)` включит 10 — правый конец не входит, последний элемент 8.
- Считать, что `np.linspace(0, 1, 5)` даёт шаг 0.2 «потому что 5» — на самом деле шаг 0.25: точек 5, а интервалов между ними 4.
- Использовать `np.empty` под аккумулятор с `+=` — внутри мусор, результат испорчен.
