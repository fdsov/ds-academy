# M5.11 · Универсальные функции (ufuncs)

ufunc — это векторизованная функция, применяемая поэлементно в скомпилированном коде: `np.exp`, `np.log`, `np.sqrt`, `np.sin`, `np.abs`, `np.maximum`, `np.where` и десятки других. Они и есть носители скорости NumPy.

```python
ltv = np.array([120, 0, 450, 30])
np.log1p(ltv)                      # log(1+x), безопасно для нуля
np.sqrt(ltv)
np.clip(ltv, 0, 200)               # обрезать выбросы сверху
np.where(ltv > 100, "vip", "reg")  # векторный if-else
np.maximum(ltv, 50)                # поэлементный максимум с порогом
```

`np.where(condition, a, b)` — векторная замена `if`: где условие True берётся `a`, иначе `b`. Это рабочая лошадь для создания сегментов и флагов без цикла.

## Что значит «универсальная» и почему это быстро

Слово «universal» означает, что одна и та же функция применяется к каждому элементу массива независимо, и весь обход выполняется внутри скомпилированного C-кода, а не в интерпретаторе Python. Когда вы пишете `np.sqrt(ltv)`, Python вызывает ufunc ровно один раз; цикл по 4 элементам (или по 40 миллионам) крутится уже на уровне C. Сравните с эквивалентом «руками»:

```python
# Медленно: цикл в Python, на каждый элемент — оверхед интерпретатора
result = [math.sqrt(x) for x in ltv]

# Быстро: один вызов ufunc, цикл внутри C
result = np.sqrt(ltv)
```

На больших массивах разница — десятки и сотни раз. Интуиция «на пальцах»: каждый шаг Python-цикла — это распаковка объекта, проверка типов, вызов функции; ufunc делает это один раз для всего массива, плюс работает с плотной непрерывной памятью, что дружит с кэшем процессора.

## Поэлементность и broadcasting

ufunc с двумя аргументами (`np.maximum`, `np.add`, `np.where`) применяется попарно к соответствующим элементам. Если формы не совпадают, включается broadcasting — скаляр или меньший массив «растягивается» до общей формы:

```python
deposits = np.array([100, 0, 500, 50])
np.maximum(deposits, 50)     # скаляр 50 broadcast'ится: [100, 50, 500, 50]
np.add(deposits, 10)         # +10 к каждому: [110, 10, 510, 60]
```

То есть `np.maximum(deposits, 50)` ставит «пол» в 50 каждому депозиту, а `np.minimum(deposits, 500)` поставил бы «потолок». `np.clip(x, lo, hi)` — это сразу и пол, и потолок в одном вызове, удобно для отрезания выбросов с двух сторон.

## Разобранный пример: подготовка фичей из депозитов

Типичная задача в гемблинг-аналитике — превратить сырые суммы депозитов в признаки для модели или сегментации. Депозиты сильно скошены вправо (несколько китов на десятки тысяч долларов, основная масса — единицы и десятки), поэтому логарифмируют.

```python
import numpy as np

dep = np.array([0, 25, 100, 4500, 30000], dtype=float)  # депозиты, $

# 1. Логарифм для сжатия скошенного распределения.
#    log1p = log(1+x): нужен потому что log(0) = -inf, а депозит 0 валиден.
dep_log = np.log1p(dep)            # [0., 3.26, 4.62, 8.41, 10.31]

# 2. Обрезать выбросы сверху на $10 000, чтобы кит не доминировал.
dep_capped = np.clip(dep, 0, 10000)  # [0, 25, 100, 4500, 10000]

# 3. Бинарный флаг депозитора.
is_depositor = np.where(dep > 0, 1, 0)  # [0, 1, 1, 1, 1]

# 4. Текстовый сегмент по порогам.
seg = np.where(dep >= 1000, "high",
        np.where(dep >= 100, "mid", "low"))  # ['low','low','mid','high','high']
```

Обратите внимание на пункт 4: вложенный `np.where` — это векторный аналог `if / elif / else`. Внешний `where` выбирает «high» там, где `dep >= 1000`, а во всех остальных позициях подставляет результат внутреннего `where`, который уже различает «mid» и «low». Так строят многоуровневые сегменты без единого Python-цикла.

## Пограничные случаи и типичные ошибки

!!! warning "log(0), деление и предупреждения"

    `np.log(0)` возвращает `-inf`, а `np.log(-1)` — `nan` (с RuntimeWarning), не исключение. Если в данных есть нули или отрицательные значения, используйте `np.log1p` (для нуля) или заранее фильтруйте/clipьте. Деление `np.array([1, 0]) / np.array([0, 0])` даст `[inf, nan]` тоже без падения — NumPy по умолчанию не бросает ошибку, а пишет warning и кладёт `inf`/`nan`.

Ещё частые грабли:

- `np.where(cond, a, b)` вычисляет **обе** ветки `a` и `b` целиком, а потом выбирает. Поэтому `np.where(x > 0, np.log(x), 0)` всё равно посчитает `np.log(x)` для всех элементов, включая `x <= 0`, и выдаст warning. Чтобы избежать — clip аргумент: `np.where(x > 0, np.log(np.clip(x, 1e-9, None)), 0)`.
- Тип результата `np.where` со строками — фиксированная длина (`<U4`). Если ветки дают строки разной длины, длинные обрежутся. Для безопасности приводите к `object` или используйте pandas.
- `np.maximum` (поэлементный максимум двух массивов) — это НЕ `np.max` (максимум по всему массиву / по оси). Перепутать легко: `np.maximum(a, b)` сравнивает попарно, `np.max(a)` сворачивает в одно число.

!!! note "in-place и память"

    Многие ufunc принимают `out=` для записи результата в существующий массив без аллокации: `np.sqrt(x, out=x)`. На больших массивах это экономит память. Также есть `np.add.reduce`, `np.add.accumulate` — у ufunc'ов есть методы для свёрток (это и стоит за `np.sum`, `np.cumsum`).

## Мини-бенчмарк скорости

```python
import numpy as np, time
x = np.random.rand(10_000_000)

t = time.perf_counter()
y1 = [v ** 0.5 for v in x]          # чистый Python
print("loop:", time.perf_counter() - t)

t = time.perf_counter()
y2 = np.sqrt(x)                      # ufunc
print("ufunc:", time.perf_counter() - t)
```

На типичной машине цикл займёт секунды, а `np.sqrt` — десятки миллисекунд. Это и есть причина, по которой в продуктовой и quant-аналитике любую поэлементную трансформацию миллионов строк делают через ufunc, а не циклом.

!!! tip "Правило большого пальца"

    Если пишете `for` по элементам массива, чтобы посчитать что-то поэлементно, — почти наверняка есть ufunc, который сделает это в одну строку и в сотни раз быстрее. Сначала ищите готовую функцию (`np.<tab>`), потом думайте про цикл.
