# M5.3 · Почему NumPy быстрый: модель памяти

### Интуиция

Обычный Python-список — это коробка с указателями. Когда ты пишешь `[1, 2, 3]`, в памяти лежат не сами числа, а три «адреса», по которым разбросаны объекты-числа. Каждое число — это полноценный Python-объект с заголовком, типом, счётчиком ссылок. Чтобы сложить два списка поэлементно, интерпретатор для каждой пары проходит путь: достать указатель, найти объект, проверить его тип, вызвать метод сложения, создать новый объект. Это десятки машинных операций на одно сложение.

NumPy-массив — это плотная непрерывная лента байтов. Миллион чисел `float64` лежат подряд, ровно по 8 байт каждое, без заголовков и указателей. Тип у всех один и записан один раз в метаданных массива. Поэтому сложение двух массивов превращается в тугой цикл на C, который процессор может ещё и векторизовать на уровне SIMD-инструкций (одна инструкция обрабатывает 4-8 чисел сразу). Никакой проверки типов в цикле, никакого создания объектов.

#### На пальцах: почему «разбросанность» так дорого стоит

Представь, что числа списка — это люди, живущие в разных квартирах по всему городу, а массив — это очередь, где все стоят плечом к плечу. Чтобы «обойти всех» в списке, процессор каждый раз прыгает по случайным адресам памяти. Современный CPU читает память не по одному байту, а блоками-«кэш-линиями» по 64 байта. Когда числа лежат подряд (массив), одно чтение из RAM притаскивает в быстрый кэш сразу 8 соседних `float64` — следующие 7 сложений берутся почти бесплатно. Когда числа разбросаны (список), каждое обращение — это потенциальный промах кэша (cache miss), и процессор простаивает сотни тактов, ожидая данные из медленной RAM. Это явление называется **локальностью данных (data locality)**, и оно часто важнее, чем «цикл на C vs цикл на Python».

#### Конкретные цифры

Прикинем расход памяти на миллион чисел:

- NumPy `float64`: $10^6 \times 8 = 8{\,}000{\,}000$ байт $\approx 7.6$ МБ — ровно полезные данные.
- Python `list` из `int`: сам список хранит $10^6$ указателей по 8 байт ($\approx 7.6$ МБ), плюс каждый объект `int` в CPython занимает 28 байт. Итого $\approx 7.6 + 26.7 \approx 34$ МБ — в 4-5 раз больше, и это ещё без учёта фрагментации.

По скорости разница того же порядка драматичнее:

```python
import numpy as np

n = 1_000_000
py_list = list(range(n))
np_arr = np.arange(n)

# Python-список: интерпретируемый цикл + создание объектов
%timeit [x * 2 for x in py_list]      # ~50-80 мс

# NumPy: один вызов в C, векторизованный
%timeit np_arr * 2                    # ~1-2 мс
```

Разрыв в 30-100 раз — это и есть совокупный эффект единого dtype, отсутствия проверок типов в цикле, локальности данных и SIMD.

### Формально

`ndarray` (N-dimensional array) — это объект, описывающий непрерывный блок памяти плюс набор метаданных:

- `dtype` — тип элемента (например `int64`, `float64`, `bool`), фиксирован для всего массива;
- `shape` — кортеж размеров по каждой оси, например `(1000, 3)` для 1000 строк и 3 столбцов;
- `strides` — сколько байт перешагнуть в памяти, чтобы сдвинуться на единицу вдоль каждой оси.

Эта тройка позволяет интерпретировать одну и ту же ленту байтов как массив любой формы. Адрес элемента $A[i, j]$ вычисляется арифметически:

$$
\text{offset}(i, j) = i \cdot s_0 + j \cdot s_1
$$

где $s_0, s_1$ — strides. Именно благодаря strides многие операции (транспонирование, срезы) не копируют данные, а лишь меняют метаданные — об этом ниже в разделе про views.

#### Разбор strides на числах

Возьмём массив формы `(2, 3)` типа `int64` (8 байт на элемент), лежащий по строкам (C-порядок, row-major):

```python
import numpy as np

A = np.array([[10, 20, 30],
              [40, 50, 60]], dtype=np.int64)

A.shape      # (2, 3)
A.strides    # (24, 8)
```

Читаем `strides = (24, 8)` так: чтобы сдвинуться на одну строку вниз, нужно перешагнуть $3 \times 8 = 24$ байта (целая строка из 3 чисел); чтобы сдвинуться на один столбец вправо — $8$ байт (одно число). Проверим адрес `A[1, 2]` (число 60) относительно начала ленты:

$$
\text{offset}(1, 2) = 1 \cdot 24 + 2 \cdot 8 = 40 \text{ байт}
$$

40 байт от начала — это шестой `int64` (байты 40-47), то есть пятый по счёту элемент после нуля: $10, 20, 30, 40, 50, \mathbf{60}$. Сходится.

Теперь транспонируем. NumPy **не трогает байты** — он просто меняет местами `shape` и `strides`:

```python
B = A.T
B.shape      # (3, 2)
B.strides    # (8, 24)  — оси поменялись местами
```

Та же лента `10 20 30 40 50 60`, но теперь «шаг вниз» равен 8 байт, а «шаг вправо» — 24. Поэтому `A.T` — это O(1) операция, не зависящая от размера массива: переставить два числа в кортеже strides одинаково дёшево что для матрицы 2×3, что для 10000×10000.

!!! info "Цена однородности"

    Фиксированный dtype — это и сила, и ограничение. Массив не может одновременно хранить числа и строки эффективно. Если положить разнородные данные, NumPy либо приведёт всё к общему типу, либо создаст массив `dtype=object` — и тогда ты теряешь всю производительность, получая по сути обёрнутый список указателей.

!!! warning "Типичные ошибки и пограничные случаи"

    - **`dtype=object` крадётся незаметно.** `np.array([1, 2, "3"])` молча приведёт всё к строкам, а `np.array([[1, 2], [3, 4, 5]])` (рваные строки) на свежих версиях NumPy упадёт с ошибкой, а на старых создаст `object`-массив списков. Всегда проверяй `arr.dtype` после создания — если там `object` или неожиданный `<U...` (строка), производительность уже потеряна.
    - **Переполнение целых.** У `int64` фиксированный диапазон. `np.array([2**63], dtype=np.int64)` переполнится и даст отрицательное число без предупреждения — потому что цикл на C не проверяет границы, как делал бы Python с его «бесконечными» int.
    - **View vs copy — источник коварных багов.** Раз срез `arr[1:5]` возвращает view на ту же память, запись `view[0] = 999` изменит исходный массив. Если нужна независимая копия — явно вызывай `.copy()`.
    - **Смешивание типов в арифметике.** `int_array + 0.5` тихо повысит результат до `float64` (upcasting) и выделит новый массив — следи за dtype результата, если важна память.

!!! question "Проверь себя"

    1. Почему миллион чисел в NumPy-массиве занимает меньше памяти и обрабатывается быстрее, чем тот же список Python?
    2. Какие три метаданных вместе со ссылкой на блок памяти описывают `ndarray`?
    3. Зачем нужны `strides` и почему благодаря им транспонирование не копирует данные?
    4. Что произойдёт с производительностью, если положить в массив разнородные данные?

??? success "Ответы"

    1. В списке лежат указатели на отдельные объекты-числа с заголовками; в массиве числа лежат подряд плотной лентой с единым dtype, и цикл идёт на C с SIMD без проверки типов в цикле.
    2. `dtype`, `shape`, `strides`.
    3. `strides` — число байт для сдвига на единицу вдоль каждой оси; адрес элемента вычисляется арифметически, поэтому смена формы или порядка осей меняет лишь метаданные, не копируя байты.
    4. NumPy приведёт всё к общему типу либо создаст `dtype=object` (массив указателей), и производительность падает до уровня обычного списка.

!!! example "Зачем это в аналитике"

    Представь, что ты считаешь дневной GGR (gross gaming revenue) по 50 миллионам ставок: `ggr = bets - payouts`. В чистом Python цикл по 50 млн пар чисел займёт секунды и зажуёт сотни мегабайт на объекты-`int`. В NumPy `ggr = bets - payouts` — это один векторизованный вычет двух `float64`-лент: десятки миллисекунд и ровно $50\text{ млн} \times 8 = 400$ МБ полезных данных, без накладных расходов. Когда такие пересчёты крутятся в ETL каждые 5 минут по всем игрокам, разница «секунды vs миллисекунды» решает, успевает ли пайплайн в свой интервал. Та же логика лежит под `pandas` и под расчётом метрик в quant-бэктестах: считаешь не «по строчке», а целыми колонками-массивами.
