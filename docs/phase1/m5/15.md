# M5.15 · Views против copies и память

### Интуиция

Это место, где горят даже опытные. View — это «окно» в чужую память: другой объект-массив, но та же лента байтов. Изменишь view — изменится оригинал. Copy — независимый дубликат. Базовые срезы, `reshape`, `transpose` обычно дают view (быстро, без аллокации). Fancy indexing и boolean-маски дают copy. Перепутаешь — либо тихо испортишь исходные данные, либо удивишься, что твоё изменение «не применилось».

На пальцах: представь, что массив — это длинная физическая лента в памяти, а сам объект `ndarray` — это всего лишь «инструкция чтения»: с какого байта начинать (offset), сколько шагов делать между элементами (strides) и какой формы получится результат (shape). View — это новая инструкция чтения той же самой ленты. Copy — это новая лента, в которую байты физически перекопированы. Поэтому view создаётся мгновенно и без памяти (только новый маленький объект-обёртка), а copy стоит O(n) по времени и памяти.

### Формально

View возможен, когда новый массив выражается через те же данные сдвигом offset и пересчётом strides. Копия неизбежна, когда требуемые элементы нельзя описать регулярным шагом в памяти (именно так с fancy indexing и масками).

Ключевая мысль: «можно ли выразить выборку одним регулярным шагом?». Срез `a[2:9:2]` — это offset=2, stride=2 элемента, можно. А выборка `a[[2, 0, 7]]` идёт вразнобой — никакой единый шаг её не опишет, поэтому NumPy вынужден собрать новый буфер. Boolean-маска `a[a > 5]` — то же самое: какие именно индексы пройдут, заранее непредсказуемо, регулярного шага нет, значит copy.

```python
a = np.arange(10)
s = a[2:5]            # view
s[0] = 999
a                     # [0 1 999 3 4 5 6 7 8 9] — оригинал изменился!

f = a[[2, 3, 4]]      # fancy indexing -> copy
f[0] = -1
a                     # оригинал НЕ изменился

# проверить, делит ли память
np.shares_memory(a, s)   # True
np.shares_memory(a, f)   # False

# явная копия, когда нужна независимость
clean = a[2:5].copy()
```

### Как отличить view от copy на практике

Помимо `np.shares_memory`, у каждого массива есть атрибут `.base`: у view он указывает на исходный массив (или его данные), у copy он `None`.

```python
a = np.arange(12).reshape(3, 4)

v = a[:, 1:3]            # базовый срез по столбцам -> view
v.base is not None       # True (это окно в a)
np.shares_memory(a, v)   # True

c = a[a % 2 == 0]        # boolean-маска -> copy
c.base                   # None
np.shares_memory(a, c)   # False

t = a.T                  # transpose -> view (меняются только strides)
np.shares_memory(a, t)   # True
```

Тонкость про `reshape`: он отдаёт view, только если данные лежат непрерывно и совместимо с новой формой. Если массив получен, например, транспонированием и стал non-contiguous, `reshape` может вернуть copy — NumPy решает сам. Поэтому не полагайся на «reshape всегда view»; если важно знать — проверь `np.shares_memory`.

### Разобранный численный пример

Пусть `a = np.arange(10)`, делаем `s = a[2:5]`. Что значат strides и offset на этой «ленте»:

- `a.strides` = `(8,)` — между соседними int64 ровно 8 байт.
- `s` начинается с offset = 2 элемента = 16 байт от начала ленты `a`, имеет shape `(3,)` и тот же stride 8.
- Запись `s[0] = 999` пишет по адресу «начало `a` + 16 байт» — это ровно та же ячейка, что и `a[2]`. Отсюда и эффект: `a[2]` стал 999.

Для `f = a[[2, 3, 4]]` NumPy сначала аллоцирует новый буфер на 3 элемента, копирует туда `a[2], a[3], a[4]`, и `f` больше не связан с `a`. Запись `f[0] = -1` меняет только новый буфер.

### Границы и типичные ошибки

- **Chained indexing на присваивании.** `df`-стиль `a[mask][0] = 5` (или `a[a>5][0] = 0`) почти всегда баг: `a[mask]` уже вернул copy, и присваивание в copy теряется — оригинал не меняется, ошибки нет, данные «не применились». Правильно: `a[mask] = ...` одним выражением, тогда NumPy пишет на месте.
- **`reshape` ≠ гарантированный view.** См. выше про non-contiguous.
- **`np.asarray` vs `np.array`.** `np.array(x)` по умолчанию копирует, `np.asarray(x)` старается вернуть view, если dtype/форма совпадают. Это важно в горячем коде.
- **Скрытая copy при смене dtype.** `a.astype(np.float64)` всегда copy. Арифметика, повышающая тип (`int_array + 0.5`), тоже создаёт новый массив.
- **View держит оригинал в памяти.** Если ты взял маленький `view` от гигабайтного массива и оригинал тебе больше не нужен — пока жив view, весь большой буфер не освободится. В таком случае `view.copy()` отвяжет маленький кусок и позволит сборщику освободить гигабайт.

### Зачем это в продуктовой и гемблинг-аналитике

Представь датафрейм-подобный массив с ежедневными ставками игроков, скажем матрица `bets` на 50 млн строк (около 400 МБ для float64). Ты считаешь корректировку: «для VIP-сегмента пересчитать GGR с учётом рейка».

- Если делаешь `vip = bets[vip_mask]` (маска -> copy) и правишь `vip`, оригинал `bets` остаётся нетронутым — это безопасно, но ты заплатил памятью за копию подмножества.
- Если по ошибке думал, что `vip` — это view, и ждал, что правки попадут в `bets`, — получишь тихий баг: отчёт по полному датасету не изменится, а ты будешь искать причину часами.
- Наоборот, при нормализации больших колонок (`bets[:, col] *= 0.95` для применения 5% рейка) — это базовый срез + in-place, никакой лишней аллокации на 400 МБ, отчёт строится в разы быстрее. Для джоба, который крутится каждые 5 минут на проде, это разница между «успели до следующего тика» и «копим лаг».

Практический совет: если ты собираешься модифицировать срез исходного датасета и не хочешь трогать оригинал — вызови `.copy()` явно. И наоборот, если работаешь с гигабайтами и хочешь избежать лишних аллокаций — пользуйся views и in-place операциями (`arr += 1`, `out=`-аргумент ufuncs).
