# M5.12 · Линейная алгебра

### Интуиция

Скалярное произведение, матричное умножение — основа любого ML. Логистическая регрессия для скоринга оттока, эмбеддинги, ковариационные матрицы — всё это матричные операции. NumPy даёт оператор `@` специально для матричного умножения.

Самая полезная картинка «на пальцах»: линейная модель — это таблица объектов, умноженная на вектор весов. У тебя есть `X` — матрица «строка = игрок, столбец = признак» (например `deposits_count`, `avg_bet`, `days_since_reg`, `bonus_ratio`, `sessions_7d`). И есть вектор весов `w` — насколько каждый признак влияет на итоговый скор. Матричное умножение `X @ w` за одну операцию считает скор для всех игроков сразу, без единого Python-цикла. Вместо `for i in range(100_000): score[i] = sum(X[i] * w)` — один `@`, который внутри уходит в оптимизированный BLAS на C. Это и быстрее в десятки раз, и читается как формула.

Скалярное произведение двух векторов `a @ b = Σ aᵢ·bᵢ` — это «насколько они сонаправлены». Если поделить на длины, получится косинус угла между ними — основа поиска похожих эмбеддингов («какие игроки похожи на этого по поведению»).

### Формально

Для матриц $A$ размера $(m, k)$ и $B$ размера $(k, n)$ произведение $C = A B$ имеет размер $(m, n)$:

$$
C_{ij} = \sum_{p=1}^{k} A_{ip} B_{pj}
$$

Внутреннее измерение должно совпадать ($k = k$). Запомни правило размерностей как «цепочку домино»: $(m, \underline{k}) \cdot (\underline{k}, n) \to (m, n)$ — соприкасающиеся числа должны быть равны и «сокращаются», а внешние дают форму результата. Если они не совпадают, NumPy кинет `ValueError: matmul: ... mismatch`. Это причина ошибки номер один у новичков: перепутан порядок множителей или забыт `.T` (транспонирование).

```python
A = rng.normal(size=(3, 4))
B = rng.normal(size=(4, 2))
A @ B                 # (3,2) — оператор matmul, предпочтительный
np.dot(A, B)          # то же
np.matmul(A, B)       # то же

v = np.array([1.0, 2.0, 3.0])
v @ v                 # 14.0 — скалярное произведение (sum квадратов)

# линейная модель: предсказание = X @ weights + bias
X = rng.normal(size=(100, 5))
w = rng.normal(size=5)
preds = X @ w + 0.5   # (100,) предсказания для 100 объектов
```

Разберём `v @ v` руками, чтобы число не казалось магией: $v = [1, 2, 3]$, тогда $v \cdot v = 1\cdot1 + 2\cdot2 + 3\cdot3 = 1 + 4 + 9 = 14$. Скалярное произведение вектора с самим собой — это сумма квадратов, то есть квадрат его длины: $\lVert v \rVert = \sqrt{14} \approx 3.74$, что совпадает с `np.linalg.norm(v)`.

Разберём и форму `X @ w + 0.5`: `X` имеет форму `(100, 5)`, `w` — `(5,)`. По правилу домино $(100, \underline{5}) \cdot (\underline{5},) \to (100,)$ — получаем 100 чисел, по одному скору на игрока. Прибавление скаляра `0.5` (свободный член, bias) идёт по броадкастингу ко всем 100 значениям. Это буквально предсказание линейной модели до применения сигмоиды.

!!! warning "Транспонирование и порядок множителей"

    `A @ B` и `B @ A` — в общем случае разные вещи (умножение матриц некоммутативно), а часто `B @ A` вообще не определено из-за размерностей. Если поймал `shapes not aligned`, первым делом проверь: не нужен ли `.T`. Пример: чтобы получить матрицу попарных скалярных произведений признаков (грубая ковариация) из `X` формы `(n, d)`, нужно `X.T @ X` — форма `(d, d)`, а не `X @ X`, которое упадёт при `n != d`.

Модуль `np.linalg` содержит `inv` (обратная), `solve` (решение системы), `det`, `eig` (собственные значения), `svd` (сингулярное разложение), `norm`. Для решения систем всегда предпочитай `np.linalg.solve(A, b)` вместо `np.linalg.inv(A) @ b` — это точнее и быстрее.

Почему `solve` лучше `inv`? Явное обращение матрицы накапливает численную ошибку и делает лишнюю работу: `inv` фактически решает систему `n` раз (по разу на столбец единичной матрицы), а потом ты ещё раз умножаешь на `b`. `solve` идёт к ответу напрямую через разложение, без построения обратной матрицы. На плохо обусловленных матрицах (близких к вырожденным — например, когда два признака почти линейно зависимы, как `total_bets` и `total_bets * 1.01`) разница в точности становится критичной. Правило для практики: если видишь в коде `np.linalg.inv(A) @ b` — это почти всегда повод заменить на `np.linalg.solve(A, b)`.

```python
# Нормальное уравнение МНК: w = (XᵀX)⁻¹ Xᵀy — но решаем без inv
XtX = X.T @ X            # (5, 5)
Xty = X.T @ (X @ w)      # синтетический таргет для примера, (5,)
w_hat = np.linalg.solve(XtX, Xty)   # предпочтительно
# w_hat ≈ исходные веса w
```

!!! note "Типичные ошибки"

    - Использовать `*` вместо `@`: `A * B` — это поэлементное умножение (Hadamard), требует одинаковой формы, а не матричное произведение. Молчаливо даст неверный результат, если формы случайно броадкастятся.
    - Считать, что `np.dot` и `@` всегда идентичны: для 1-D и 2-D они совпадают, но для тензоров выше 2-D поведение `np.dot` и `np.matmul` различается. В ML-коде используй `@`.
    - Звать `inv` на вырожденной матрице: получишь `LinAlgError: Singular matrix` или огромные числа из-за почти-нулевого детерминанта.
