## Series и DataFrame: атомы pandas

### Интуиция

`Series` — это один столбец с подписями. Представь колонку в Excel, но у каждой ячейки есть имя (метка строки), а не только номер. `DataFrame` — это таблица: набор `Series`, склеенных по общему индексу строк. Если ты держишь в голове "Series = столбец, DataFrame = таблица из столбцов с общим индексом" — ты уже понял базу.

Ключевое отличие от Excel и от обычного списка Python: у строк есть **индекс** — отдельная ось меток. Это не просто номера 0, 1, 2. Индексом может быть дата, ID игрока, что угодно. Индекс — главная фишка pandas и одновременно главный источник путаницы для новичков.

### Формально

`Series` — это одномерный размеченный массив: пара (значения, индекс), где значения хранятся в типизированном массиве (поверх NumPy или, в новых версиях, Apache Arrow), а индекс — это отдельный объект `Index`, отображающий метки в позиции. `DataFrame` — двумерная размеченная структура: словарь из `Series` с общим объектом-индексом строк и объектом-индексом столбцов.

```python
import pandas as pd

deposits = pd.Series([100, 250, 50], index=["u1", "u2", "u3"], name="deposit")
deposits["u2"]  # доступ по метке -> 250
deposits.iloc[1]  # доступ по позиции -> 250
```

Важно сразу зафиксировать: метка и позиция — это разные вещи. `deposits["u2"]` ищет по метке, `deposits.iloc[1]` — по порядковому номеру. Когда метки — это случайно числа (0, 1, 2), новички их путают, и это порождает классические баги.

!!! note "dtype — это не мелочь"

    У каждого столбца есть тип (`int64`, `float64`, `object`, `category`, `datetime64[ns]`, `boolean`). Тип определяет и скорость, и корректность. Столбец строк хранится как `object` — это медленный питоновский тип; для повторяющихся строк (канал, страна, статус) почти всегда лучше `category`. В pandas 2.x появился бэкенд Arrow (`dtype_backend="pyarrow"`), который даёт честные nullable-типы и заметно быстрее старого NumPy-бэкенда.

## Чтение данных: CSV, Parquet, SQL

### Интуиция

Данные приходят в разных форматах. CSV — это текст, который читает человек и любой инструмент, но он медленный, не хранит типы и раздут по размеру. Parquet — бинарный колоночный формат: он сжат, помнит типы, и читается в разы быстрее, потому что можно вытащить только нужные столбцы, не читая всю таблицу. SQL — это когда данные лежат в базе, и ты тащишь их запросом.

Правило 2026 года простое: **для обмена с людьми — CSV, для хранения и работы — Parquet, для боевых данных — тащи из базы или из колоночного хранилища.**

### Формально

```python
import pandas as pd

df_csv = pd.read_csv("bets.csv", parse_dates=["bet_ts"])
df_parquet = pd.read_parquet("bets.parquet")  # типы и схема внутри файла

from sqlalchemy import create_engine
engine = create_engine("postgresql+psycopg://user:pass@host:5432/yohoho")
df_sql = pd.read_sql("SELECT player_id, amount, bet_ts FROM bets WHERE bet_ts >= '2026-03-01'", engine)
```

Почему Parquet быстрее CSV на порядок: CSV-парсер обязан прочитать каждый байт, угадать типы, распарсить числа из текста. Parquet хранит данные уже в бинарном колоночном виде со сжатием (обычно Snappy или Zstd) и метаданными о схеме. Колоночность означает, что `read_parquet(..., columns=["player_id", "amount"])` физически не читает остальные столбцы с диска.

!!! tip "Predicate и projection pushdown"

    "Pushdown" — это когда фильтр и выбор столбцов "проталкиваются" вниз, к самому чтению с диска, чтобы не грузить лишнее в память. Polars и DuckDB делают это автоматически: если ты в ленивом запросе написал `filter(amount > 0).select("player_id")`, движок прочитает с диска только нужные столбцы и только нужные группы строк. Pandas так не умеет — он сначала читает всё в память, потом фильтрует.

## Индексы, loc и iloc

### Интуиция

`loc` — доступ по меткам (по именам). `iloc` — доступ по позициям (по номерам, как в обычном списке). Если запомнить, что **l** в `loc` это "label" (метка), а **i** в `iloc` это "integer position" (целочисленная позиция), путаницы не будет.

Индекс DataFrame можно представить как "телефонную книгу": `loc` ищет по имени абонента, `iloc` — по номеру строки в книге. Когда именем абонента случайно оказывается число, легко перепутать, и pandas честно отдаст разные результаты.

### Формально

```python
df = df.set_index("player_id")

df.loc["u100"]              # строка с меткой "u100"
df.loc["u100", "amount"]   # ячейка: метка строки + имя столбца
df.iloc[0]                 # первая строка по позиции
df.iloc[0:5, [0, 2]]       # срез позиций строк и столбцов

df.loc[df["amount"] > 1000]  # loc принимает булеву маску
```

Семантика срезов различается, и это важно. `df.iloc[0:5]` берёт строки 0,1,2,3,4 — правый край НЕ включён, как везде в Python. А `df.loc["2026-03-01":"2026-03-31"]` включает оба конца — потому что для меток "конец среза" интуитивно тоже нужный элемент.

!!! warning "Chained indexing и SettingWithCopyWarning"

    Запись `df[df.amount > 0]["bonus"] = 0` — это chained indexing (две операции индексации подряд). Pandas не гарантирует, изменит он оригинал или копию, и кидает `SettingWithCopyWarning`. Всегда пиши через один `loc`: `df.loc[df.amount > 0, "bonus"] = 0`. В pandas 3.0 (Copy-on-Write стал дефолтом) поведение стало строже и предсказуемее, но привычка писать через единый `loc` — обязательна.

## Фильтрация

### Интуиция

Фильтрация — это "оставь только строки, удовлетворяющие условию". Ты строишь маску — столбец из True/False той же длины, что и таблица, — и pandas оставляет строки, где True. Думай об этом как о включённой/выключенной лампочке напротив каждой строки.

### Формально

```python
mask = (df["channel"] == "facebook") & (df["amount"] >= 50)
fb = df[mask]

# несколько значений
df[df["country"].isin(["UZ", "KZ", "RU"])]

# диапазон
df[df["amount"].between(10, 1000)]

# текст
df[df["promo"].str.startswith("WELCOME")]

# современный способ — query, читается как SQL
df.query("channel == 'facebook' and amount >= 50")
```

Критично: логические операторы — это `&`, `|`, `~` (не `and`, `or`, `not`), и каждое условие обязательно в скобках из-за приоритета операций. `df[df.a > 0 & df.b > 0]` без скобок интерпретируется неверно и упадёт.

## Добавление и изменение колонок, assign

### Интуиция

Новый столбец — это просто результат операции над существующими. "GGR = ставки минус выплаты", "это VIP, если депозит больше 10000". Векторно: операция применяется ко всему столбцу сразу, без циклов.

### Формально

```python
df["ggr"] = df["bets"] - df["payouts"]
df["is_vip"] = df["total_deposit"] > 10_000

import numpy as np
df["segment"] = np.where(df["total_deposit"] > 10_000, "vip", "regular")

# несколько условий
df["tier"] = pd.cut(df["total_deposit"], bins=[0, 100, 1000, 10_000, np.inf],
                    labels=["bronze", "silver", "gold", "vip"])
```

`assign` — это способ добавить столбцы, **возвращая новый DataFrame**, что идеально для method chaining (см. ниже):

```python
df2 = df.assign(
    ggr=lambda d: d["bets"] - d["payouts"],
    margin=lambda d: (d["bets"] - d["payouts"]) / d["bets"],
)
```

Лямбда внутри `assign` получает текущий DataFrame, поэтому можно ссылаться на столбцы, созданные в той же цепочке. Это делает код декларативным: видно, что каждый шаг порождает новое состояние, ничего не мутируется по дороге.

## groupby: split-apply-combine во всю глубину

### Интуиция

`groupby` — самая важная операция в аналитике. Идея в трёх словах: **разбей, примени, собери** (split-apply-combine). Разбей таблицу на группы (например, по игроку или по каналу), к каждой группе примени функцию (сумма, среднее, своя логика), собери результаты обратно в таблицу.

"Сколько каждый игрок задепозитил?" — группируем по `player_id`, к каждой группе применяем сумму `deposit`, собираем. "Какой средний чек по каналам?" — группируем по `channel`, применяем среднее. Это паттерн, который покрывает 80% аналитических задач.

### Формально

Парадигму split-apply-combine формализовал Хэдли Уикхем (2011). Pandas `groupby` реализует её буквально: объект `GroupBy` — это ленивое отображение "ключ группы → набор позиций строк". Реальное вычисление происходит, когда ты вызываешь агрегацию.

```python
# базовая агрегация
g = df.groupby("channel")["deposit"].sum()

# несколько агрегатов сразу, с именами столбцов (named aggregation)
report = df.groupby("channel").agg(
    players=("player_id", "nunique"),
    total_dep=("deposit", "sum"),
    avg_dep=("deposit", "mean"),
    ggr=("ggr", "sum"),
)

# группировка по нескольким ключам
df.groupby(["country", "channel"])["ggr"].sum()
```

### Три режима применения: agg, transform, apply

Здесь начинается то, что отличает джуна от мидла. У `groupby` есть три способа "применить" функцию, и они дают результаты разной формы.

`agg` (агрегация) — сворачивает каждую группу в **одно число**. На входе группа из N строк, на выходе одна строка. Результат короче исходной таблицы.

`transform` — применяет функцию к группе, но возвращает результат **той же длины, что группа**. Используется, когда нужно вернуть групповую характеристику обратно к каждой строке. Классика — "доля депозита игрока от суммы по его стране":

```python
df["country_total"] = df.groupby("country")["deposit"].transform("sum")
df["share_in_country"] = df["deposit"] / df["country_total"]
```

Здесь `transform("sum")` посчитал сумму по стране и "размножил" её обратно на каждую строку этой страны. Через `agg` так нельзя — он схлопнул бы таблицу.

`apply` — самый гибкий и самый опасный. Принимает произвольную функцию, получающую целый sub-DataFrame группы, и может вернуть что угодно. Мощно, но медленно.

```python
def second_purchase_gap(g):
    g = g.sort_values("ts")
    if len(g) < 2:
        return pd.NA
    return (g["ts"].iloc[1] - g["ts"].iloc[0]).days

df.groupby("player_id").apply(second_purchase_gap)
```

### Почему apply бывает медленным

`apply` вызывает питоновскую функцию для **каждой группы** по отдельности — это интерпретируемый цикл под капотом, теряющий всю векторизацию. Если групп миллион, ты делаешь миллион питоновских вызовов. Встроенные агрегаты (`sum`, `mean`, `count`, `nunique`) реализованы в C/Cython и работают на порядки быстрее.

!!! warning "Правило производительности"

    Прежде чем писать `apply`, спроси: можно ли выразить это через встроенный агрегат, `transform`, или векторную операцию? В 90% случаев — да. `apply` — последнее средство, а не первое. На больших данных разница между `agg("sum")` и `apply(lambda g: g.sum())` достигает 100x.

## merge, join, concat: соединяем таблицы

### Интуиция

Данные почти никогда не лежат в одной таблице. Регистрации в одной, депозиты в другой, ставки в третьей. `merge` (он же join из мира SQL) соединяет таблицы по общему ключу — например, по `player_id`. `concat` — это просто склейка таблиц встык: либо одна под другой (добавить строки), либо рядом (добавить столбцы).

Аналогия: `merge` — это как сопоставить два списка по общему ID (досье игрока + его транзакции). `concat` — это сложить две одинаковые по структуре пачки строк в одну.

### Формально

```python
# inner: только совпавшие ключи
m = registrations.merge(deposits, on="player_id", how="inner")

# left: все игроки слева, депозиты где есть, иначе NaN
m = registrations.merge(deposits, on="player_id", how="left")

# проверка количества строк — обязательна
m = registrations.merge(deposits, on="player_id", how="left", validate="one_to_many")
```

Типы join: `inner` (пересечение ключей), `left` (все слева), `right` (все справа), `outer` (объединение). В аналитике чаще всего нужен `left`: "взять всех игроков и подтянуть к ним депозиты, не теряя тех, кто не депозитил".

```python
# concat по строкам (вертикально)
all_bets = pd.concat([bets_march, bets_april], ignore_index=True)

# concat по столбцам (горизонтально, по индексу)
wide = pd.concat([features_a, features_b], axis=1)
```

!!! warning "Главный тихий баг аналитики — разъезд строк после join"

    Если ключ в правой таблице не уникален (на одного игрока несколько строк), `left join` **размножит** строки слева. Был миллион игроков — стало три миллиона, и твоя сумма GGR утроилась. Лекарство: всегда указывай `validate=` (`"one_to_one"`, `"one_to_many"`, `"many_to_one"`) — pandas упадёт с ошибкой, если ожидание не сошлось. И всегда сверяй `len(df)` до и после merge.

## pivot и melt: широкий и длинный формат

### Интуиция

Одни и те же данные можно хранить "широко" и "длинно". Широкий формат — это таблица для человека: строки — игроки, столбцы — месяцы, в ячейках — депозиты. Длинный формат — таблица для машины: три столбца `игрок | месяц | депозит`, по строке на каждое наблюдение.

`pivot` (и `pivot_table`) превращает длинный формат в широкий — раскладывает значения категории по столбцам. `melt` делает обратное — "расплавляет" широкую таблицу в длинную. Длинный формат — родной для группировок, агрегаций и графиков; широкий — для отчётов и матриц.

### Формально

```python
# длинный -> широкий
wide = df.pivot_table(
    index="player_id", columns="month", values="deposit", aggfunc="sum", fill_value=0
)

# широкий -> длинный
long = wide.reset_index().melt(
    id_vars="player_id", var_name="month", value_name="deposit"
)
```

`pivot_table` отличается от `pivot` тем, что умеет агрегировать при дублях ключей (через `aggfunc`), тогда как голый `pivot` требует уникальности и падает на дублях. На практике почти всегда нужен `pivot_table`. Это тот самый инструмент, которым строится cohort-матрица retention (строки — когорта, столбцы — недели жизни).

## Даты и временные ряды: resample, rolling

### Интуиция

Время — особый тип данных. Если столбец дат хранится как `datetime`, открываются мощные операции: сгруппировать по неделям (`resample`), посчитать скользящее среднее за 7 дней (`rolling`), сдвинуть значения на день назад (`shift`) для сравнения "сегодня против вчера".

`resample` — это "группировка по времени": бьём временную ось на корзины (день, неделя, месяц) и агрегируем. `rolling` — это "окно, которое едет по времени": для каждой точки берём окно из последних N значений и считаем по нему статистику. Скользящее среднее сглаживает шум и показывает тренд.

### Формально

```python
ts = df.set_index("bet_ts").sort_index()

# дневной GGR
daily = ts["ggr"].resample("D").sum()

# скользящее среднее за 7 дней (сглаживание DAU/GGR)
daily_smooth = daily.rolling(window=7, min_periods=1).mean()

# рост день-к-дню
daily_growth = daily / daily.shift(1) - 1
```

Под капотом `datetime64[ns]` хранит время как количество наносекунд от эпохи — это просто целое число, поэтому арифметика дат векторна и быстра. `resample("W-MON")` группирует по неделям с началом в понедельник; `rolling` с `min_periods` контролирует, сколько точек минимум нужно, чтобы посчитать окно (иначе в начале ряда будут `NaN`).

!!! info "Часовые пояса — источник боли в гемблинге"

    Игроки по всему миру, сервер в UTC, отчётность в местном времени. Если смешать tz-aware и tz-naive даты, pandas упадёт или, хуже, молча сдвинет сутки и испортит дневные когорты. Решение: храни всё в UTC (`tz_localize("UTC")`), переводи в локальную зону только на финальном отображении.

## Пропуски: fillna, dropna

### Интуиция

Реальные данные дырявые: игрок зарегистрировался, но не задепозитил — значит депозит "отсутствует". Пропуск (`NaN`/`null`) — это не ноль и не ошибка, это "значения нет". Что с ним делать — зависит от смысла. Иногда `NaN` депозита логично заменить на 0 (не платил — значит ноль). Иногда строку с пропуском в ключевом поле надо удалить. А иногда `NaN` нужно сохранить, чтобы он не искажал среднее.

### Формально

```python
df["deposit"].isna().sum()           # сколько пропусков
df["deposit"] = df["deposit"].fillna(0)   # заменить осмысленным нулём
df = df.dropna(subset=["player_id"])      # удалить строки без ключа
df["country"] = df["country"].fillna("unknown")
```

Тонкость: `NaN` в арифметике "заразен" — `5 + NaN = NaN`, но агрегаты pandas по умолчанию его игнорируют (`mean` считает по непустым). Это значит, что заменив `NaN` на 0 перед усреднением, ты исказишь среднее (ноль попадёт в знаменатель). Решай на уровне смысла, а не механически. Современный nullable-тип (`Int64`, Arrow-бэкенд) отличает "настоящий пропуск" от NaN с плавающей точкой, что чище.

## Категориальные типы

### Интуиция

Столбцы вроде `channel`, `country`, `segment` содержат мало уникальных значений, повторяющихся миллионы раз. Хранить их как строки — расточительно: каждая ячейка тащит полноценную питоновскую строку. Категориальный тип хранит словарь уникальных значений один раз, а в ячейках — компактные целочисленные коды. Это как сноски: вместо повторения текста ставим номер.

### Формально

```python
df["channel"] = df["channel"].astype("category")
```

Выигрыш двойной: память падает в разы (особенно при низкой кардинальности), а группировки и сравнения ускоряются, потому что работают с целыми кодами вместо строк. Для упорядоченных категорий (`bronze < silver < gold < vip`) можно задать порядок и использовать сравнения `<`/`>`. Минус: добавление значения вне словаря требует расширения категорий, поэтому для активно мутирующих столбцов категории не всегда удобны.

## Method chaining: стиль 2026

### Интуиция

Method chaining — это когда весь пайплайн пишется одной цепочкой операций, каждая на своей строке, без промежуточных переменных `df1`, `df2`, `df_tmp`. Читается сверху вниз как рецепт: "взять данные, отфильтровать, добавить столбец, сгруппировать, посчитать". Никаких разбросанных по коду мутаций — одно состояние течёт через цепочку.

### Формально

```python
report = (
    df
    .query("bet_ts >= '2026-03-01'")
    .assign(ggr=lambda d: d["bets"] - d["payouts"])
    .groupby("channel", as_index=False)
    .agg(players=("player_id", "nunique"), ggr=("ggr", "sum"))
    .sort_values("ggr", ascending=False)
)
```

Почему это стиль 2026: цепочка не мутирует исходные данные (важно при Copy-on-Write в pandas 3.0), легко комментируется построчно, легко модифицируется (закомментировал шаг — увидел промежуток), и она структурно идентична тому, как пишется Polars и SQL. Минус — отладка середины цепочки требует метода `.pipe()` или временного разрыва. Для отладки удобен `.pipe(lambda d: (print(d.shape), d)[1])`.

## Почему Polars и DuckDB вытесняют pandas

### Интуиция

Pandas написан 15 лет назад, во времена одноядерных задач и небольших данных. Он однопоточный (использует одно ядро процессора из, скажем, шестнадцати), жадный до памяти (часто держит несколько копий данных), и **энергичный** (eager): выполняет каждую строку кода сразу, не видя всего пайплайна целиком, а значит не может его оптимизировать.

Polars — это переосмысление с нуля на Rust. Он многопоточный (грузит все ядра), экономный по памяти (Arrow-формат, ноль лишних копий), и, главное, умеет **ленивый** (lazy) режим: ты описываешь весь план запроса, а движок сначала его оптимизирует (выкидывает ненужные столбцы, проталкивает фильтры к чтению, переставляет операции) и только потом исполняет. Это как разница между "делаю каждый шаг как сказали" и "сначала прочитал весь рецепт, понял что половину ингредиентов можно не доставать, и сделал оптимально".

DuckDB — это встроенная аналитическая SQL-база (как SQLite, но колоночная и для аналитики). Она живёт в том же процессе, читает Parquet/CSV напрямую, и для многих "тяжёлых join + group by" побеждает и pandas, и Polars, потому что у неё зрелый SQL-оптимизатор и она не боится данных больше оперативной памяти.

### Формально: в чём конкретно разница

**Eager vs lazy.** Pandas исполняет немедленно. Polars в lazy-режиме (`pl.scan_parquet`, `.lazy()`) строит граф вычислений (logical plan), прогоняет его через оптимизатор (predicate pushdown, projection pushdown, common subexpression elimination, переупорядочивание join), и материализует только при `.collect()`.

**Выражения (expressions).** В Polars `pl.col("amount").sum()` — это объект-выражение, который движок может анализировать, переставлять и параллелить. Это принципиально мощнее питоновских лямбд pandas: выражение не "чёрный ящик", а декларативное описание, которое оптимизатор понимает.

**Параллелизм.** Polars распараллеливает по ядрам автоматически, без усилий с твоей стороны. Pandas — однопоточный по своей природе.

**Память.** Polars строится на Apache Arrow и использует zero-copy где возможно; pandas (особенно на NumPy-бэкенде) часто копирует. На границе оперативной памяти Polars и DuckDB живут, pandas умирает с `MemoryError`.

### Сравнительная таблица

| Критерий | pandas | Polars | DuckDB |
|---|---|---|---|
| Модель | eager | eager + lazy | SQL, lazy-оптимизатор |
| Параллелизм | нет (1 ядро) | да (все ядра) | да |
| Память | жадная, копии | Arrow, экономная | out-of-core |
| Язык | методы Python | выражения Polars | SQL |
| Больше RAM | падает | стримит частично | да, спиллит на диск |
| Зрелость экосистемы | максимальная | растущая | растущая |
| Лучшее применение | малые данные, legacy, совместимость | новый код, средние/большие данные | тяжёлый SQL, join больших таблиц |

### Когда что выбирать

- **Данные до ~1 ГБ, прототип, нужна совместимость с scikit-learn/легаси** — pandas остаётся удобным, его знают все.
- **Новый ETL/аналитика, данные от сотен МБ до десятков ГБ, важна скорость** — Polars (пиши сразу в lazy).
- **Тяжёлые join и агрегации, данные больше RAM, команда знает SQL** — DuckDB, можно прямо по Parquet-файлам без загрузки.
- На практике 2026-го это часто **комбо**: DuckDB тащит и джойнит сырьё из Parquet, отдаёт результат в Polars/pandas для финальной доводки и подачи в ML.

### Пример миграции pandas → Polars

Pandas-версия отчёта по каналам:

```python
import pandas as pd

df = pd.read_parquet("bets.parquet")
report = (
    df[df["bet_ts"] >= "2026-03-01"]
    .assign(ggr=lambda d: d["bets"] - d["payouts"])
    .groupby("channel", as_index=False)
    .agg(players=("player_id", "nunique"), ggr=("ggr", "sum"))
    .sort_values("ggr", ascending=False)
)
```

Тот же отчёт на Polars в ленивом режиме:

```python
import polars as pl

report = (
    pl.scan_parquet("bets.parquet")              # не читает файл сразу
    .filter(pl.col("bet_ts") >= pl.date(2026, 3, 1))
    .with_columns((pl.col("bets") - pl.col("payouts")).alias("ggr"))
    .group_by("channel")
    .agg(
        pl.col("player_id").n_unique().alias("players"),
        pl.col("ggr").sum().alias("ggr"),
    )
    .sort("ggr", descending=True)
    .collect()                                   # здесь происходит вся работа
)
```

Структура почти идентична — поэтому переход дёшев. Но за `scan_parquet` + `collect` Polars: прочитает с диска только `bet_ts`, `bets`, `payouts`, `player_id`, `channel` (projection pushdown), отфильтрует мартовские строки на этапе чтения (predicate pushdown), и сделает всё параллельно. На 40 млн строк это разница между 0.5 и 15 секундами.

!!! tip "Самое важное про Polars-выражения"

    В Polars нет индекса как в pandas (и это сознательно — индекс был источником багов). Думай столбцами и выражениями. Контексты: `select` (выбрать/вычислить столбцы), `filter` (строки), `with_columns` (добавить столбцы), `group_by().agg()` (агрегация). Внутри любого контекста живут выражения `pl.col(...)`. Освоив этот ментальный паттерн, ты пишешь Polars свободно.

## Код: стек 2026

Полный пример: когортный retention и агрегаты по игрокам — то, что реально просит продакт. Сначала Polars (рекомендуемый для нового кода 2026), затем тот же расчёт на pandas.

```python
import polars as pl

# bets: player_id, bet_ts, bets, payouts, channel
# reg:  player_id, reg_ts, country, channel

events = pl.scan_parquet("bets.parquet")
reg = pl.scan_parquet("registrations.parquet")

# 1. агрегаты по игрокам (LTV-подобный профиль)
player_stats = (
    events
    .with_columns((pl.col("bets") - pl.col("payouts")).alias("ggr"))
    .group_by("player_id")
    .agg(
        total_ggr=pl.col("ggr").sum(),
        bet_count=pl.len(),
        active_days=pl.col("bet_ts").dt.date().n_unique(),
        last_bet=pl.col("bet_ts").max(),
    )
)

# 2. когортный retention по неделям
cohort = (
    events
    .join(reg.select("player_id", "reg_ts"), on="player_id", how="inner")
    .with_columns(
        cohort_week=pl.col("reg_ts").dt.truncate("1w"),
        # сколько недель прошло от регистрации до ставки
        week_offset=((pl.col("bet_ts") - pl.col("reg_ts")).dt.total_days() // 7),
    )
    .group_by("cohort_week", "week_offset")
    .agg(active_players=pl.col("player_id").n_unique())
    .sort("cohort_week", "week_offset")
    .collect()
)

# матрица retention: строки — когорта, столбцы — неделя жизни
retention_matrix = cohort.pivot(
    values="active_players", index="cohort_week", on="week_offset"
)
```

Тот же retention на pandas (для совместимости и чтения чужого кода):

```python
import pandas as pd

events = pd.read_parquet("bets.parquet")
reg = pd.read_parquet("registrations.parquet")[["player_id", "reg_ts"]]

m = events.merge(reg, on="player_id", how="inner", validate="many_to_one")
m["cohort_week"] = m["reg_ts"].dt.to_period("W").dt.start_time
m["week_offset"] = ((m["bet_ts"] - m["reg_ts"]).dt.days // 7)

cohort = (
    m.groupby(["cohort_week", "week_offset"])["player_id"]
     .nunique()
     .reset_index(name="active_players")
)
retention_matrix = cohort.pivot_table(
    index="cohort_week", columns="week_offset", values="active_players", fill_value=0
)

# retention в процентах от нулевой недели когорты
size0 = retention_matrix[0]
retention_pct = retention_matrix.div(size0, axis=0).round(3)
```

```python
# DuckDB-вариант той же когорты — прямо по Parquet, без загрузки в память
import duckdb

retention = duckdb.sql("""
    WITH joined AS (
        SELECT b.player_id, b.bet_ts, r.reg_ts,
               date_trunc('week', r.reg_ts) AS cohort_week,
               (date_diff('day', r.reg_ts, b.bet_ts)) / 7 AS week_offset
        FROM 'bets.parquet' b
        JOIN 'registrations.parquet' r USING (player_id)
    )
    SELECT cohort_week, week_offset, COUNT(DISTINCT player_id) AS active_players
    FROM joined
    GROUP BY 1, 2
    ORDER BY 1, 2
""").pl()  # .pl() -> Polars DataFrame, .df() -> pandas
```

!!! tip "AI-копилот: где нейросети помогают"

    LLM (Claude, Cursor, Copilot) сильны в трёх вещах на табличных данных: (1) перевод между pandas/Polars/SQL — "перепиши этот pandas-пайплайн на ленивый Polars", (2) генерация шаблонного groupby/pivot по описанию, (3) объяснение чужого кода и непонятных ошибок.

    Конкретный промпт, который работает:

    "Вот схема двух таблиц: bets(player_id, bet_ts, bets, payouts, channel) и registrations(player_id, reg_ts, country). Напиши Polars lazy-пайплайн, считающий недельный retention-матрицу по когортам регистрации. Используй scan_parquet и collect в конце. Объясни, какие оптимизации (pushdown) тут сработают."

    Где LLM врут и что обязательно перепроверять: модель часто путает API-версии (предлагает устаревший `df.append()`, удалённый в pandas 2.0; `groupby(...).apply` со старой сигнатурой; в Polars — старый `groupby` вместо `group_by`, `pl.count()` вместо `pl.len()`). Она уверенно galлюцинирует несуществующие аргументы. И самое опасное — она не видит твои данные: предложит `how="left"` там, где ключ не уникален, и молча размножит строки. Всегда сам проверяй `validate=` в merge, сверяй `len()` до/после join, и прогоняй на маленьком срезе перед полным запуском. AI пишет код, корректность бизнес-логики проверяешь ты.

!!! note "Стек 2026"

    Актуально: pandas 2.x/3.0 с Copy-on-Write по умолчанию и Arrow-бэкендом (`dtype_backend="pyarrow"`); Polars как основной выбор для нового кода (стабильный API, lazy-движок); DuckDB для тяжёлого SQL и данных больше RAM; Parquet как формат хранения по умолчанию; method chaining как стандарт стиля.

    Устарело относительно старых курсов: `df.append()` (удалён — используй `pd.concat`); `df.ix[]` (удалён давно — `loc`/`iloc`); чтение всего в CSV и обработка циклами `for`; `inplace=True` (антипаттерн, мешает chaining и Copy-on-Write); хранение строковых категорий как `object` вместо `category`; вера в то, что "pandas — это весь Python-data-stack". В 2026 джун, не знающий Polars и DuckDB, выглядит как джун из 2019.

!!! warning "Ловушки"

    - **Дубли после merge.** Неуникальный ключ справа размножает строки слева, суммы раздуваются. Лекарство: `validate=`, сверка `len()`.
    - **Chained indexing.** `df[mask]["col"] = x` пишет в копию. Только `df.loc[mask, "col"] = x`.
    - **`apply` вместо векторизации.** Питоновский цикл под капотом, до 100x медленнее. Сначала ищи встроенный агрегат/transform.
    - **`&`/`|` без скобок.** `df[df.a > 0 & df.b > 0]` падает из-за приоритета. Каждое условие в скобках.
    - **NaN в среднем.** Заменил `NaN` на 0 перед `mean()` — исказил среднее. Решай по смыслу.
    - **Таймзоны.** Смешал tz-aware и tz-naive — сдвиг суток, сломанные когорты. Храни UTC.
    - **На чём горят сеньоры:** `groupby(...).apply` с побочными эффектами и неявным сортированием; молчаливое приведение `int64` к `float64` при появлении одного NaN (ломает join по ID — теперь `1` и `1.0`); забытый `reset_index()` после groupby, ломающий последующий merge; в Polars — ожидание pandas-индекса, которого нет.
