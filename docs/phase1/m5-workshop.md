## Что отрабатываем

Этот воркшоп берёт три самых рабочих идеи модуля M5 и доводит их до мышечной памяти на одном датасете:

- **Векторизация** — переписать Python-цикл в поэлементные операции `ndarray` и доказать выигрыш бенчмарком (M5.7).
- **Broadcasting** — «растянуть» массивы разной формы без копирования и без циклов (M5.8), плюс агрегации по `axis` с `keepdims` (M5.9).
- **Монте-Карло** — оценить вероятность и интеграл через выборки из `Generator` API (`np.random.default_rng`, M5.13), опираясь на маски и `.mean()` доли True (M5.6).

Попутно закрываем ловушки модуля: тихое переполнение `int`, view против copy, NaN-агрегации.

**Артефакт на выходе** — один скрипт `m5_workshop.py`, который:

1. печатает таблицу «до/после» с временами цикла и вектора и фактором ускорения по нескольким задачам;
2. выводит Монте-Карло оценку числа $\pi$ и вероятности события с доверительным интервалом;
3. сохраняет результаты бенчмарка в `bench_results.csv`.

Правило приёмки простое: **ни одного Python-цикла по элементам данных в финальных версиях**. Цикл допустим только там, где мы намеренно показываем «как было».

## Бизнес-кейс

Векторизация и Монте-Карло — это не «академический NumPy», а два инструмента, которые в iGaming напрямую конвертируются в деньги и в скорость реакции: первый позволяет пересчитывать антифрод и риск-метрики по всему логу ставок за секунды, второй — отвечать на вопросы о вероятности убытка там, где аналитическую формулу не вывести из-за тяжёлого хвоста выплат.

!!! example "Ситуация"

    Ты — аналитик в команде iGaming-продукта. К тебе приходит **руководитель риск-менеджмента** с двумя сцепленными вопросами по текущей настройке выплат.

    - **Риск убытка.** После недавнего повышения множителя выплат дом ушёл в лёгкий минус по матожиданию на части слотов. Вопрос: какова вероятность, что дневной GGR казино уйдёт в минус? Иллюстративно — если эта вероятность окажется выше 15-20%, кассовый буфер придётся увеличивать, а настройку — откатывать; цена ошибки в одну сторону это просадка ликвидности, в другую — недополученный оборот на десятки млн ₽ в квартал.
    - **Скорость пересчёта.** Текущий ночной скрипт, который сканирует ~40 млн ставок на подозрительные паттерны, написан на Python-циклах и идёт несколько часов, не укладываясь в утренний SLA — фрод-кейсы попадают к команде с задержкой в сутки. Нужно показать, что та же логика на векторизации укладывается в секунды.

    От твоего ответа зависит решение: **крутить ли ручку выплат назад** и **переписывать ли антифрод-пайплайн на векторизацию**. Бюджет на кону — буфер ликвидности и стоимость пропущенного фрода.

    Ограничение: ответ нужен за один день, доступен только синтетический лог ставок (реальную выгрузку дадут позже) — поэтому риск считаем симуляцией Монте-Карло, а не на исторических данных.

## Данные

Базовый генератор, на котором держится весь воркшоп (положи в начало `m5_workshop.py`):

```python
import time
import numpy as np

RNG = np.random.default_rng(2026)

def make_log(n: int) -> dict[str, np.ndarray]:
    return {
        "player_id": RNG.integers(0, 50_000, size=n),
        "bet": RNG.exponential(100.0, size=n),
        "win": None,  # заполним ниже, зависит от bet
    }

N = 1_000_000
log = make_log(N)
log["win"] = log["bet"] * RNG.random(N) * 2.0   # выплата как доля ставки
bet, win = log["bet"], log["win"]
print("сгенерировано ставок:", N, "| dtype bet:", bet.dtype)
```

`exponential` выбран осознанно: суммы ставок в гемблинге имеют тяжёлый правый хвост, и это пригодится, когда будем ловить «крупные» ставки маской.

## Ход работы

### Шаг 1: цикл против вектора, честный замер

**Зачем.** M5.7 говорит: цикл Python исполняется интерпретатором, вектор спускается в C. Мы не верим на слово — меряем. Считаем GGR (`bet - win`) двумя способами и сравниваем через `time.perf_counter`.

```python
def ggr_loop(bet, win):
    out = []
    for i in range(len(bet)):
        out.append(bet[i] - win[i])
    return np.array(out)

def ggr_vec(bet, win):
    return bet - win

def bench(fn, *args, repeats=3):
    best = float("inf")
    for _ in range(repeats):
        t0 = time.perf_counter()
        res = fn(*args)
        best = min(best, time.perf_counter() - t0)
    return best, res

t_loop, a = bench(ggr_loop, bet, win)
t_vec, b = bench(ggr_vec, bet, win)

assert np.allclose(a, b)
print(f"loop: {t_loop*1000:8.1f} ms")
print(f"vec:  {t_vec*1000:8.1f} ms")
print(f"ускорение: {t_loop / t_vec:6.0f}x")
```

**Что получилось.** Цикл — порядка 150-300 мс, вектор — единицы мс, ускорение в 50-150 раз. `assert np.allclose` доказывает, что результаты идентичны: ускорение не куплено ценой правильности. Берём минимум из нескольких прогонов (`repeats`) — это устойчивее к шуму ОС, чем один замер.

!!! tip "Почему min, а не среднее"

    Время выполнения зашумлено сверху (планировщик, GC, другие процессы), но не снизу — быстрее «физического минимума» машина не станет. Поэтому для микробенчмарков берут минимум, а не среднее: оно ближе к истинной стоимости операции.

!!! question "Проверь себя"

    1. Почему мы вызываем `np.allclose`, а не `a == b` для сравнения двух float-результатов?
    2. Что сломается в бенчмарке, если `ggr_loop` будет возвращать список, а не `np.array`, и мы сравним его с вектором?

??? success "Ответы"

    1. Float-арифметика в разном порядке даёт расхождения в последних битах; `==` вернёт массив False там, где числа совпадают «по смыслу». `np.allclose` сравнивает с допуском.
    2. `np.allclose(list, ndarray)` отработает (список приведётся к массиву), но если бы мы сравнивали через `is` или по идентичности типов — упали бы. Главное — сравнивать значения, а не контейнеры.

### Шаг 2: переписать антифрод-эвристику из цикла в маски

**Зачем.** Реальная задача аналитика — не вычесть два массива, а отфильтровать по нескольким условиям. Отрабатываем boolean-маски и операторы `&`/`|` в скобках (M5.6). Ищем подозрительные ставки: крупная (топ-1% по объёму) И выплата больше 1.9 от ставки.

```python
def suspicious_loop(bet, win):
    thr = np.percentile(bet, 99)
    flags = []
    for i in range(len(bet)):
        flags.append(bet[i] > thr and win[i] > bet[i] * 1.9)
    return np.array(flags)

def suspicious_vec(bet, win):
    thr = np.percentile(bet, 99)
    return (bet > thr) & (win > bet * 1.9)

t_loop, m1 = bench(suspicious_loop, bet, win)
t_vec, m2 = bench(suspicious_vec, bet, win)

assert np.array_equal(m1, m2)
print("подозрительных:", m2.sum(), "| доля:", round(m2.mean(), 5))
print(f"ускорение масок: {t_loop / t_vec:.0f}x")
```

**Что получилось.** Та же логика, но без цикла и без `and` (для массивов работают только `&`/`|` в скобках — приоритет операторов из M5.6). `m2.sum()` даёт количество, `m2.mean()` — долю True одной строкой. Ускорение снова десятки раз.

!!! warning "Скобки обязательны"

    `bet > thr & win > bet * 1.9` без скобок распарсится как `bet > (thr & win) > ...` из-за приоритета `&` и упадёт с `TypeError`. Всегда `(...) & (...)`.

### Шаг 3: broadcasting вместо цикла по столбцам

**Зачем.** Строим матрицу признаков на игрока и нормируем z-score по столбцам — это ровно пример из M5.8. Цель — прочувствовать, что `(50000, 3) - (3,)` растягивается по строкам сам, а центрирование по строкам требует `keepdims=True`.

```python
ggr = bet - win
n_players = 50_000
pid = log["player_id"]

vol  = np.bincount(pid, weights=bet, minlength=n_players)
cnt  = np.bincount(pid, minlength=n_players).astype(np.float64)
gsum = np.bincount(pid, weights=ggr, minlength=n_players)

metrics = np.stack([vol, cnt, gsum], axis=1)   # (50000, 3)

mu = metrics.mean(axis=0)                       # (3,)
sigma = metrics.std(axis=0)                     # (3,)
metrics_z = (metrics - mu) / np.where(sigma == 0, 1.0, sigma)

print("форма признаков:", metrics_z.shape)
print("среднее по столбцам ~0:", metrics_z.mean(axis=0).round(3))
print("std по столбцам ~1:", metrics_z.std(axis=0).round(3))

# аномалийный скор = евклидова норма z-вектора игрока (broadcast + axis)
score = np.linalg.norm(metrics_z, axis=1)       # (50000,)
top = np.argsort(score)[-10:][::-1]
print("топ-10 по аномалийному скору:", top)
```

**Что получилось.** Ноль циклов по данным. `np.bincount` с `weights` группирует по целочисленному `player_id` быстрее любого Python-словаря (идиома из M5.16). `np.where(sigma == 0, 1, sigma)` страхует от деления на ноль для мёртвых столбцов. `np.linalg.norm(..., axis=1)` схлопывает ось признаков — `axis` это та ось, что исчезает (M5.9).

!!! question "Проверь себя"

    1. Какую форму даст `metrics - mu`, где `metrics` это `(50000, 3)`, а `mu` это `(3,)`? Почему broadcasting срабатывает?
    2. Если бы мы центрировали по строкам через `metrics.mean(axis=1)`, какой формы был бы результат и почему он сломал бы вычитание?

??? success "Ответы"

    1. `(50000, 3)`. Выравнивая справа налево: `3 == 3` совпадает, недостающая левая ось у `mu` достраивается как 1 и растягивается до 50000.
    2. `(50000,)`. При вычитании она выровняется справа к оси столбцов (3 против 50000) — формы несовместимы, ошибка или неверный результат. Нужен `keepdims=True` для `(50000, 1)`.

### Шаг 4: Монте-Карло — оценка числа π через выборки

**Зачем.** Монте-Карло — это «считаем долю попаданий в событие по случайным выборкам». Отрабатываем `Generator` API (M5.13), маски и `.mean()` как долю True (M5.6). Классика: бросаем точки в квадрат $[-1,1]^2$, доля попавших в круг радиуса 1 равна $\pi/4$.

$$
\frac{\text{площадь круга}}{\text{площадь квадрата}} = \frac{\pi r^2}{(2r)^2} = \frac{\pi}{4}
\;\Rightarrow\;
\pi \approx 4 \cdot \frac{\#\{x^2 + y^2 \le 1\}}{N}
$$

```python
def estimate_pi(n: int, rng: np.random.Generator) -> tuple[float, float]:
    xy = rng.uniform(-1.0, 1.0, size=(n, 2))
    inside = (xy[:, 0] ** 2 + xy[:, 1] ** 2) <= 1.0   # маска
    p = inside.mean()                                  # доля попаданий
    pi_hat = 4.0 * p
    # стандартная ошибка доли: sqrt(p(1-p)/n), масштабируем на 4
    se = 4.0 * np.sqrt(p * (1.0 - p) / n)
    return pi_hat, se

rng = np.random.default_rng(777)
for n in [10_000, 1_000_000, 50_000_000]:
    pi_hat, se = estimate_pi(n, rng)
    lo, hi = pi_hat - 1.96 * se, pi_hat + 1.96 * se
    print(f"N={n:>10}: pi≈{pi_hat:.5f}  95% CI=[{lo:.5f}, {hi:.5f}]")
```

**Что получилось.** Оценка сходится к 3.14159 по мере роста $N$, а доверительный интервал сужается примерно как $1/\sqrt{N}$ — это фундаментальная скорость сходимости Монте-Карло (увеличил выборку в 100 раз — точность выросла в 10). Ни одного цикла: вся симуляция в `rng.uniform`, маске и `.mean()`. Обрати внимание — генератор `rng` локальный, не глобальный `np.random.seed`.

!!! tip "Память против циклов"

    `size=(50_000_000, 2)` это ~800 МБ float64. Монте-Карло легко упирается в память при наивном «всё разом». Если не влезает — разбей на батчи в обычном `for` по батчам (не по элементам!) и накапливай счётчик попаданий. Это легальный цикл: он идёт по блокам, внутри каждого блока — векторизация.

!!! question "Проверь себя"

    1. Во сколько раз нужно увеличить $N$, чтобы сузить доверительный интервал Монте-Карло вдвое?
    2. Почему `inside.mean()` корректно даёт долю попаданий, хотя `inside` это массив из True/False?

??? success "Ответы"

    1. В 4 раза: ошибка падает как $1/\sqrt{N}$, поэтому для деления ошибки на 2 нужно $N \times 4$.
    2. Для boolean-массива True трактуется как 1, False как 0, поэтому среднее равно доле True (идиома M5.6).

### Шаг 5: Монте-Карло прикладной — вероятность отрицательного дня казино

**Зачем.** Переносим Монте-Карло на бизнес-вопрос модуля. Симулируем много «дней» по 10 000 ставок и оцениваем вероятность, что дневной GGR казино уйдёт в минус. Здесь Монте-Карло заменяет аналитическую формулу, которую вывести тяжело из-за тяжёлого хвоста выплат.

```python
def prob_negative_day(n_days: int, bets_per_day: int,
                      rng: np.random.Generator) -> tuple[float, float]:
    b = rng.exponential(100.0, size=(n_days, bets_per_day))
    w = b * rng.random((n_days, bets_per_day)) * 2.05   # дом чуть в минус по матожиданию
    daily_ggr = (b - w).sum(axis=1)                     # схлопываем ставки -> (n_days,)
    neg = (daily_ggr < 0).mean()
    se = np.sqrt(neg * (1 - neg) / n_days)
    return neg, se

rng = np.random.default_rng(99)
p, se = prob_negative_day(200_000, 10_000, rng)
print(f"P(минусовой день) ≈ {p:.4f}  ±{1.96*se:.4f}")
```

**Что получилось.** `b - w` векторно, `.sum(axis=1)` схлопывает ставки внутри дня (M5.9: `axis=1` исчезает, остаётся вектор по дням), `(daily_ggr < 0).mean()` — снова доля True. Множитель 2.05 искусственно даёт дому отрицательное матожидание, и симуляция показывает, что доля минусовых дней высокая. Меняй множитель и смотри, как сдвигается вероятность — это и есть сила Монте-Карло: подставил параметр, получил вероятность без вывода формул.

### Шаг 6: собрать артефакт — таблица до/после и CSV

**Зачем.** Финальный артефакт модуля: сводим бенчмарки в одну таблицу и сохраняем. Заодно отрабатываем `np.savetxt` и осознанный выбор dtype для целочисленного ключа (ловушка переполнения `int32` из M5.4).

```python
rows = [
    ("ggr",        *bench(ggr_loop, bet, win)[:1], bench(ggr_vec, bet, win)[0]),
    ("suspicious", bench(suspicious_loop, bet, win)[0], bench(suspicious_vec, bet, win)[0]),
]

print(f"\n{'task':<12}{'loop_ms':>10}{'vec_ms':>10}{'speedup':>10}")
table = []
for name, t_loop, t_vec in rows:
    sp = t_loop / t_vec
    print(f"{name:<12}{t_loop*1e3:>10.1f}{t_vec*1e3:>10.2f}{sp:>9.0f}x")
    table.append((t_loop * 1e3, t_vec * 1e3, sp))

np.savetxt(
    "bench_results.csv",
    np.array(table),
    delimiter=",",
    header="loop_ms,vec_ms,speedup",
    comments="",
    fmt="%.3f",
)

# контроль dtype для ключа: суммарный объём в копейках за год переполнит int32
kopecks = (bet * 100).astype(np.int64)   # int64 осознанно
print("\nсумма в копейках (int64):", kopecks.sum())
print("в int32 было бы переполнение:", np.int32(kopecks.sum()))
```

**Что получилось.** На диске лежит `bench_results.csv` с временами и фактором ускорения — это и есть «до/после» артефакт. Последние строки демонстрируют, что та же сумма в `int32` «заворачивается» в мусорное число, а в `int64` корректна (ловушка денег из M5.4).

## Бизнес-вывод

## Развитие

1. **Батчевый Монте-Карло для π на $10^9$ точек.** Перепиши `estimate_pi` так, чтобы он шёл циклом по батчам (не по точкам), накапливал счётчик попаданий в `int64` и не держал всю матрицу в памяти. Сравни итоговую точность с однопроходной версией и замерь пиковую память.

2. **Попарные расстояния через broadcasting.** Реализуй наивную (двойной цикл) и векторную ($\lVert a-b\rVert^2 = \lVert a\rVert^2 + \lVert b\rVert^2 - 2ab$) версии матрицы расстояний для `X` формы `(n, d)`. Проверь совпадение через `np.allclose` и оцени, при каком `n` промежуточная матрица `(n, n)` вызовет OOM (ловушка «сеньорской» матрицы из M5.16).

3. **Антифрод-скор и ложные срабатывания.** Доведи Шаг 3 до топ-20 подозрительных и проверь, не являются ли они просто хайроллерами (vip). Добавь признак «доля выигрышных ставок» и подумай, как z-score штрафует легитимных крупных игроков — это типичный bias скоринга по норме.

4. **timeit вместо ручного perf_counter.** Перенеси бенчмарк на `timeit.repeat` с `number`/`repeat` и сравни стабильность результатов с ручным замером. Добавь третий конкурент — list comprehension против явного `for.append` — и посмотри, даёт ли comprehension заметную разницу (обычно нет, оба упираются в интерпретатор).
