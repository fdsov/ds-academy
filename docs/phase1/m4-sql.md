## M4.1 · Зачем этот модуль

Данные в реальной компании не лежат в CSV-файле у тебя на столе. Они живут в базе данных: депозиты игроков, ставки, события входа в игру, история бонусов, выплаты. Эти таблицы измеряются десятками и сотнями миллионов строк. Питон с pandas не вытянет 200 миллионов ставок в память твоего ноутбука, а если и вытянет — ты будешь ждать минут двадцать вместо двух секунд.

SQL (Structured Query Language) — это язык, на котором ты разговариваешь с базой и просишь её посчитать ровно то, что нужно, прямо там, где лежат данные. База оптимизирована под это десятилетиями. Аналитик, который не умеет в SQL, — это переводчик без словаря: он зависит от других людей, которые "выгрузят ему данные". А значит, он не может проверить гипотезу за пять минут, не может построить когорту retention без чужой помощи, не может ответить на вопрос продакта "сколько у нас вчера было FTD из Узбекистана" немедленно.

Без SQL ломается вся работа data scientist'а на ранних этапах: сбор данных, EDA, построение признаков для модели, проверка качества данных, ad-hoc аналитика для бизнеса. Это базовый навык, такой же фундаментальный, как умение читать. И парадокс: SQL легко начать (за час напишешь первый SELECT), но по-настоящему глубоко им владеют единицы — оконные функции, планы запросов и логику NULL понимают плохо даже многие senior-аналитики.

## M4.3 · Реляционная модель: как устроены данные

Интуиция. Представь набор связанных таблиц Excel. Одна таблица — игроки: по строке на человека. Другая — депозиты: по строке на каждое пополнение счёта. Чтобы понять, кто сделал депозит, в таблице депозитов хранится не имя игрока (оно может повторяться, меняться), а его уникальный номер — `player_id`. Этот номер — мостик между таблицами.

Формально. Реляционная модель (Эдгар Кодд, 1970) представляет данные как набор отношений (relations) — таблиц. Каждая таблица состоит из строк (кортежей) и столбцов (атрибутов) с фиксированным типом. Ключевые понятия:

- Первичный ключ (primary key) — столбец или набор столбцов, однозначно идентифицирующий строку. В таблице `players` это `player_id`. Двух строк с одинаковым primary key быть не может.
- Внешний ключ (foreign key) — столбец, ссылающийся на первичный ключ другой таблицы. В `deposits.player_id` хранится ссылка на `players.player_id`. Это и есть связь.
- Связь один-ко-многим — у одного игрока много депозитов, но каждый депозит принадлежит ровно одному игроку. Это самый частый тип связи в продуктовых базах.

Нормализация — принцип "каждый факт хранится в одном месте". Имя игрока лежит только в `players`. Если игрок сменил ник, ты меняешь одну строку, а не миллион строк в депозитах. Это защищает данные от рассогласования.

!!! info "Наши учебные таблицы"

    Дальше во всех примерах используем три таблицы гемблинг-продукта:

    `players(player_id, signup_date, country, source)` — игроки.

    `deposits(deposit_id, player_id, amount, deposit_ts)` — пополнения счёта.

    `bets(bet_id, player_id, stake, payout, bet_ts)` — ставки. `stake` — сумма ставки, `payout` — выплата игроку.

!!! question "Проверь себя"

    1. Чем первичный ключ отличается от внешнего? Приведи пример каждого в наших таблицах.
    2. Почему в `deposits` хранят `player_id`, а не имя игрока?
    3. Что означает связь "один-ко-многим" между `players` и `deposits` и в какой таблице лежит внешний ключ?
    4. Какую проблему решает нормализация при смене ника игрока?

??? success "Ответы"

    1. Первичный ключ однозначно идентифицирует строку своей таблицы (`players.player_id`); внешний ключ ссылается на первичный ключ другой таблицы (`deposits.player_id`).
    2. Имя может повторяться и меняться, а `player_id` уникален и стабилен — он надёжный мостик между таблицами.
    3. У одного игрока много депозитов, но каждый депозит принадлежит ровно одному игроку. Внешний ключ лежит на стороне "многих" — в `deposits`.
    4. Имя хранится в одном месте (`players`), поэтому при смене ника правится одна строка, а не миллион строк в депозитах — данные не рассогласовываются.

## M4.4 · SELECT, WHERE, ORDER BY, LIMIT — чтение данных

Интуиция. SELECT — это "покажи мне". WHERE — "только те строки, которые подходят под условие". ORDER BY — "отсортируй". LIMIT — "и хватит первых N штук".

Формально. Базовый запрос на выборку:

```sql
SELECT player_id, amount, deposit_ts
FROM deposits
WHERE amount >= 100
ORDER BY deposit_ts DESC
LIMIT 10;
```

Это значит: из таблицы `deposits` взять столбцы `player_id, amount, deposit_ts`, оставить только строки где сумма не меньше 100, отсортировать по времени от новых к старым (`DESC` — descending), показать первые 10.

Критично понимать порядок выполнения, потому что он не совпадает с порядком написания. SQL читается сверху вниз, но выполняется так:

1. `FROM` — берём источник
2. `WHERE` — фильтруем строки
3. `GROUP BY` — группируем
4. `HAVING` — фильтруем группы
5. `SELECT` — вычисляем столбцы и алиасы
6. `ORDER BY` — сортируем
7. `LIMIT` — отрезаем

Именно поэтому в `WHERE` нельзя сослаться на алиас из `SELECT` — на момент фильтрации строк его ещё не существует. Это первая вещь, которая отличает понимающего от зубрилы.

!!! question "Проверь себя"

    1. В каком порядке выполняются `FROM`, `WHERE`, `SELECT`, `ORDER BY`, `LIMIT`?
    2. Почему в `WHERE` нельзя сослаться на алиас, заданный в `SELECT`?
    3. Что делает `DESC` в `ORDER BY` и для чего нужен `LIMIT`?

??? success "Ответы"

    1. `FROM` → `WHERE` → `GROUP BY` → `HAVING` → `SELECT` → `ORDER BY` → `LIMIT`.
    2. На момент фильтрации строк в `WHERE` шаг `SELECT` ещё не выполнялся, поэтому алиаса физически не существует.
    3. `DESC` сортирует по убыванию (от больших к меньшим); `LIMIT N` оставляет только первые N строк результата.

## M4.5 · Агрегаты, GROUP BY, HAVING — свёртка данных

Интуиция. Агрегат — это когда из множества строк получается одно число. "Сколько всего депозитов" — COUNT. "На какую сумму" — SUM. GROUP BY — "посчитай это отдельно для каждой страны / каждого дня / каждого игрока".

Формально. Пять базовых агрегатных функций: `COUNT` (количество), `SUM` (сумма), `AVG` (среднее), `MIN` / `MAX` (минимум / максимум). С `GROUP BY` они считаются по группам:

```sql
SELECT country,
       COUNT(*)              AS deposits_cnt,
       COUNT(DISTINCT player_id) AS depositors,
       SUM(amount)           AS total_amount,
       AVG(amount)           AS avg_check
FROM deposits
GROUP BY country
HAVING SUM(amount) > 10000
ORDER BY total_amount DESC;
```

Тонкости, которые многие не знают:

- `COUNT(*)` считает все строки. `COUNT(column)` считает только строки где `column` не NULL. `COUNT(DISTINCT player_id)` — число уникальных игроков. Это три разных числа.
- `HAVING` — это `WHERE` для групп. `WHERE` фильтрует строки до группировки, `HAVING` — группы после. Хочешь "страны с суммой депозитов больше 10000" — это HAVING. Хочешь "только депозиты больше 100, потом сгруппировать" — это WHERE.
- `AVG` игнорирует NULL. Среднее по `[100, NULL, 200]` равно 150, а не 100. Это часто не то, что ты думаешь.

!!! question "Проверь себя"

    1. Чем отличаются `COUNT(*)`, `COUNT(column)` и `COUNT(DISTINCT player_id)`?
    2. В чём разница между `WHERE` и `HAVING` и почему "страны с суммой депозитов больше 10000" — это `HAVING`?
    3. Чему равно `AVG` по значениям `[100, NULL, 200]` и почему?

??? success "Ответы"

    1. `COUNT(*)` считает все строки; `COUNT(column)` — только строки, где `column` не NULL; `COUNT(DISTINCT player_id)` — число уникальных значений.
    2. `WHERE` фильтрует строки до группировки, `HAVING` — группы после агрегации. Сумма по стране известна только после `GROUP BY`, поэтому условие на неё ставится в `HAVING`.
    3. 150: `AVG` игнорирует NULL и делит сумму 300 на 2 непустых значения, а не на 3.

## M4.6 · JOIN — соединение таблиц

Это сердце SQL. Интуиция: JOIN склеивает строки из двух таблиц по совпадению ключа. У тебя есть депозиты с `player_id`, и ты хочешь подтянуть страну игрока из `players` — JOIN по `player_id`.

Формально, типы соединений:

- INNER JOIN — только строки, где совпадение есть в обеих таблицах. Депозит без игрока (битый `player_id`) выпадет. Игрок без депозитов выпадет.
- LEFT JOIN — все строки левой таблицы плюс совпадения из правой. Где совпадения нет — правые столбцы будут NULL. Это рабочая лошадь аналитика: "все игроки, и их депозиты, если есть".
- RIGHT JOIN — зеркало LEFT: все строки правой таблицы. На практике почти не используется — проще поменять таблицы местами и написать LEFT.
- FULL JOIN (FULL OUTER) — все строки обеих таблиц, NULL там, где пары нет. Нужен для сверки двух источников: "что есть слева, чего нет справа, и наоборот".
- CROSS JOIN — декартово произведение: каждая строка слева с каждой справа. 1000 на 1000 даст миллион строк. Используется редко и осознанно — например, чтобы сгенерировать сетку "каждый день × каждая страна" для заполнения пропусков.

```sql
SELECT p.country,
       COUNT(DISTINCT p.player_id)  AS players,
       COALESCE(SUM(d.amount), 0)   AS revenue
FROM players p
LEFT JOIN deposits d ON d.player_id = p.player_id
GROUP BY p.country;
```

LEFT JOIN здесь критичен: мы хотим увидеть и страны, где игроки есть, но никто не задепозитил (revenue = 0). С INNER JOIN такие страны просто исчезли бы, и ты бы сделал неверный вывод, что в них всё хорошо.

!!! warning "Раздувание строк при JOIN"

    Если соединить `deposits` с `bets` по `player_id`, и у игрока 5 депозитов и 10 ставок, то JOIN выдаст 5 × 10 = 50 строк. Любой `SUM(amount)` после этого посчитает каждый депозит по 10 раз. Это самая частая причина, по которой "выручка в отчёте завышена в разы". Правило: агрегируй каждую таблицу до соединения (в CTE или подзапросе), а потом соединяй уже свёрнутые результаты по `player_id`.

!!! question "Проверь себя"

    1. Чем INNER JOIN отличается от LEFT JOIN на примере "страны без депозитов"?
    2. Зачем в запросе с LEFT JOIN на сумму депозитов нужен `COALESCE(SUM(d.amount), 0)`?
    3. Почему RIGHT JOIN на практике почти не пишут?
    4. Откуда берётся раздувание строк при JOIN и как его избежать?

??? success "Ответы"

    1. INNER оставит только страны, где есть и игроки, и депозиты, — страна без депозитов исчезнет. LEFT сохранит все страны игроков, подставив NULL/0 для депозитов.
    2. Без депозитов `SUM` вернёт NULL; `COALESCE` превращает его в 0, чтобы revenue читался как ноль, а не пустота.
    3. Это зеркало LEFT — проще поменять таблицы местами и написать LEFT JOIN, читается понятнее.
    4. При соединении двух таблиц "многие-ко-многим" по ключу строки перемножаются (5 депозитов × 10 ставок = 50). Решение: агрегировать каждую таблицу до соединения в CTE/подзапросе, потом джойнить свёрнутое.

Затем набросай альтернативу: сначала свернуть депозиты в CTE `SUM(amount) GROUP BY player_id`, потом джойнить.

## M4.7 · Подзапросы и CTE

Интуиция. Подзапрос — это запрос внутри запроса: сначала посчитай вот это, потом используй результат. CTE (Common Table Expression, конструкция `WITH`) — то же самое, но вынесенное наверх и названное человеческим именем. Это как разбить длинную формулу на промежуточные переменные.

Формально. Подзапрос можно поставить в `FROM`, в `WHERE`, даже в `SELECT`:

```sql
SELECT player_id, amount
FROM deposits
WHERE amount > (SELECT AVG(amount) FROM deposits);
```

Здесь подзапрос считает среднее, а внешний запрос берёт депозиты выше среднего. Тот же запрос через CTE становится читаемее, когда логика растёт:

```sql
WITH player_revenue AS (
    SELECT player_id, SUM(amount) AS revenue
    FROM deposits
    GROUP BY player_id
),
big_spenders AS (
    SELECT player_id
    FROM player_revenue
    WHERE revenue > 1000
)
SELECT p.country, COUNT(*) AS whales
FROM big_spenders b
JOIN players p ON p.player_id = b.player_id
GROUP BY p.country;
```

CTE — не про производительность (хотя иногда влияет), а про читаемость и поддержку. Запрос на 80 строк без CTE невозможно проверить. Разбитый на именованные шаги — читается как абзацы текста. Правило профессионала: если запрос длиннее экрана — почти наверняка он просится в CTE.

## M4.8 · Оконные функции — самое важное для аналитики

Это водораздел между джуном и сильным аналитиком. Если освоишь оконные функции — сможешь считать когорты, retention, нарастающие итоги, ранги и "разницу с предыдущим днём" без боли.

Интуиция. Обычный GROUP BY схлопывает группу строк в одну. Оконная функция считает агрегат по группе, но не схлопывает строки — она дописывает результат к каждой строке как новый столбец. Представь: ты смотришь на каждую ставку и одновременно видишь "а какая по счёту эта ставка у этого игрока" и "сколько он суммарно поставил к этому моменту". Строки остаются на месте, рядом появляется контекст.

Формально. Синтаксис: `функция() OVER (PARTITION BY ... ORDER BY ...)`.

- `PARTITION BY` — на какие группы (окна) бить данные. Аналог GROUP BY, но без схлопывания.
- `ORDER BY` внутри OVER — порядок внутри окна, важен для ранжирования и нарастающих итогов.

Ключевые функции:

- `ROW_NUMBER()` — порядковый номер строки в окне: 1, 2, 3, 4. Уникальный всегда.
- `RANK()` — ранг с пропусками при равенстве: 1, 2, 2, 4.
- `DENSE_RANK()` — ранг без пропусков: 1, 2, 2, 3.
- `LAG(x)` / `LEAD(x)` — значение из предыдущей / следующей строки окна. Незаменимы для "разница со вчера".
- `SUM(x) OVER (... ORDER BY ...)` — нарастающий итог (running total).

```sql
SELECT player_id,
       deposit_ts,
       amount,
       ROW_NUMBER() OVER (PARTITION BY player_id ORDER BY deposit_ts) AS deposit_no,
       SUM(amount)  OVER (PARTITION BY player_id ORDER BY deposit_ts) AS cumulative_amount,
       LAG(deposit_ts) OVER (PARTITION BY player_id ORDER BY deposit_ts) AS prev_deposit_ts
FROM deposits;
```

В одном проходе мы получили: номер депозита у игрока (`deposit_no = 1` — это FTD, первый депозит!), накопленную сумму к этому моменту и время предыдущего депозита (чтобы посчитать интервал между ними). Без оконных функций каждое из этих чисел требовало бы отдельного самосоединения таблицы — медленно и нечитаемо.

??? example "Retention через оконные функции и self-join по дням"

    Классическая задача: D1/D7/D30 retention. Сначала находим дату регистрации (день 0) и все дни активности игрока, затем считаем разницу в днях и смотрим, есть ли активность на нужном лаге.

    ```sql
    WITH activity AS (
        SELECT player_id,
               bet_ts::date AS active_day
        FROM bets
        GROUP BY player_id, bet_ts::date
    ),
    cohort AS (
        SELECT player_id, MIN(active_day) AS cohort_day
        FROM activity
        GROUP BY player_id
    )
    SELECT c.cohort_day,
           COUNT(DISTINCT c.player_id) AS cohort_size,
           COUNT(DISTINCT CASE WHEN a.active_day - c.cohort_day = 1
                               THEN a.player_id END) AS d1_retained,
           COUNT(DISTINCT CASE WHEN a.active_day - c.cohort_day = 7
                               THEN a.player_id END) AS d7_retained
    FROM cohort c
    LEFT JOIN activity a ON a.player_id = c.player_id
    GROUP BY c.cohort_day
    ORDER BY c.cohort_day;
    ```

    D1 retention для когорты — это `d1_retained / cohort_size`. Делим в внешнем слое или в питоне.

!!! question "Проверь себя"

    1. Чем оконная функция отличается от обычного `GROUP BY` по поведению со строками?
    2. За что отвечают `PARTITION BY` и `ORDER BY` внутри `OVER (...)`?
    3. Чем различаются `ROW_NUMBER()`, `RANK()` и `DENSE_RANK()` на наборе равных значений?
    4. Как с помощью `ROW_NUMBER()` найти первый депозит (FTD) каждого игрока?

??? success "Ответы"

    1. `GROUP BY` схлопывает группу строк в одну; оконная функция считает агрегат по окну, но строки сохраняет, дописывая результат отдельным столбцом.
    2. `PARTITION BY` задаёт окна (на какие группы бить данные), `ORDER BY` внутри `OVER` — порядок строк в окне, нужный для рангов и нарастающих итогов.
    3. На значениях-дубликатах: `ROW_NUMBER` даёт 1,2,3,4 (всегда уникально); `RANK` — 1,2,2,4 (с пропуском); `DENSE_RANK` — 1,2,2,3 (без пропуска).
    4. `ROW_NUMBER() OVER (PARTITION BY player_id ORDER BY deposit_ts)` и взять строки с номером 1.

!!! example "Мини-задание"

    Напиши 3-5 строк SQL: с помощью `ROW_NUMBER() OVER (PARTITION BY player_id ORDER BY deposit_ts)` пометь номер каждого депозита и оставь только первый депозит каждого игрока (`deposit_no = 1`). Проверь, что получилась ровно одна строка на игрока.

## M4.9 · CASE WHEN — условная логика

Интуиция. Это `if/else` прямо внутри SQL. "Если депозит больше 500 — это 'high', иначе 'low'". Незаменим для сегментации и для так называемой условной агрегации.

Формально:

```sql
SELECT player_id,
       SUM(CASE WHEN amount >= 500 THEN amount ELSE 0 END) AS high_amount,
       COUNT(CASE WHEN amount < 50 THEN 1 END)             AS small_deposits,
       AVG(CASE WHEN country = 'UZ' THEN amount END)        AS avg_uz_check
FROM deposits
GROUP BY player_id;
```

Приём `SUM(CASE WHEN ... THEN x ELSE 0 END)` называется условная агрегация и решает огромный класс задач: посчитать несколько метрик с разными фильтрами в одном проходе по таблице, без многократного сканирования. Это и быстрее, и читаемее, чем три отдельных запроса с JOIN.

## M4.10 · Работа с датами

Интуиция. Время — основа продуктовой аналитики: всё считается "по дням", "за последние 7 дней", "по месяцам регистрации". Нужно уметь обрезать timestamp до даты, считать разницу и группировать по периодам.

Формально (PostgreSQL и DuckDB во многом совпадают):

```sql
SELECT date_trunc('day', bet_ts)      AS day,        -- обрезать до дня
       date_trunc('month', bet_ts)    AS month,      -- до месяца
       bet_ts::date                   AS day_cast,    -- то же короче
       deposit_ts - signup_ts         AS time_to_ftd, -- интервал
       EXTRACT(dow FROM bet_ts)        AS weekday     -- день недели 0-6
FROM bets;
```

Для DAU (Daily Active Users) ключевое — обрезать timestamp до дня и считать уникальных игроков:

```sql
SELECT bet_ts::date AS day,
       COUNT(DISTINCT player_id) AS dau
FROM bets
GROUP BY bet_ts::date
ORDER BY day;
```

!!! warning "Часовой пояс — тихий убийца метрик"

    Если `bet_ts` хранится в UTC, а бизнес считает день по местному времени, твой "день" будет сдвинут на несколько часов. Игрок, поставивший ставку в 1 ночи по локальному, попадёт во "вчера" по UTC. DAU и retention поедут. Всегда уточняй таймзону хранения и приводи к нужной: `bet_ts AT TIME ZONE 'UTC' AT TIME ZONE 'Asia/Tashkent'`.

## M4.11 · Логика NULL и её ловушки

NULL — это не ноль и не пустая строка. Это "значение неизвестно". Из этого следуют три ловушки, на которых горят даже опытные.

Ловушка 1 — сравнение. `NULL = NULL` даёт не TRUE, а NULL (то есть "неизвестно"). Любое сравнение с NULL через `=` проваливается. Проверять надо через `IS NULL` / `IS NOT NULL`, никогда через `= NULL`.

Ловушка 2 — NOT IN с NULL. Запрос `WHERE player_id NOT IN (SELECT player_id FROM blacklist)` вернёт ноль строк, если в `blacklist` есть хоть один NULL. Потому что `x NOT IN (1, 2, NULL)` логически разворачивается в `x<>1 AND x<>2 AND x<>NULL`, а последнее — NULL, что губит всё условие. Решение: `NOT EXISTS` или фильтрация NULL в подзапросе.

Ловушка 3 — агрегаты и фильтры пропускают NULL. `COUNT(column)` не считает NULL. `SUM`, `AVG` игнорируют их. А условие `WHERE country <> 'UZ'` НЕ вернёт строки где `country IS NULL` — хотя интуитивно "не UZ" должно их включать. Для NULL "не равно UZ" — это "неизвестно", а не TRUE.

```sql
-- Безопасная замена NULL на значение по умолчанию
SELECT player_id, COALESCE(country, 'unknown') AS country
FROM players;
```

`COALESCE(a, b, c)` возвращает первый не-NULL аргумент. Это твой главный инструмент защиты от NULL в выводе.

!!! question "Проверь себя"

    1. Почему `NULL = NULL` не даёт TRUE и как тогда проверять на NULL?
    2. Почему `WHERE player_id NOT IN (SELECT player_id FROM blacklist)` может вернуть ноль строк, и чем это лечится?
    3. Вернёт ли `WHERE country <> 'UZ'` строки, где `country IS NULL`, и почему?
    4. Что делает `COALESCE(country, 'unknown')`?

??? success "Ответы"

    1. Для SQL NULL — "неизвестно", и сравнение через `=` даёт NULL, а не TRUE. Проверять надо через `IS NULL` / `IS NOT NULL`.
    2. Если в `blacklist` есть NULL, условие `x <> NULL` внутри `NOT IN` становится NULL и обнуляет всё. Лечится через `NOT EXISTS` или фильтрацию NULL в подзапросе.
    3. Нет: для NULL "не равно UZ" — это "неизвестно", а не TRUE, поэтому такие строки выпадут. Нужно добавить `OR country IS NULL`.
    4. Возвращает первый не-NULL аргумент — здесь подставляет `'unknown'` вместо отсутствующей страны.

## M4.12 · Индексы и почему запрос медленный

Интуиция. Индекс — это как алфавитный указатель в конце книги. Без него, чтобы найти все депозиты игрока 12345, база читает всю таблицу подряд (full scan) — все 200 миллионов строк. С индексом по `player_id` она прыгает сразу к нужным строкам, как ты открываешь словарь на нужной букве.

Формально. Индекс — это вспомогательная структура данных (чаще всего B-дерево), которая хранит отсортированные значения столбца и указатели на строки. Создаётся так:

```sql
CREATE INDEX idx_deposits_player ON deposits (player_id);
```

Когда индекс помогает: фильтрация (`WHERE player_id = ...`), соединения (`JOIN ON player_id`), сортировка по индексированному столбцу. Когда не помогает или вредит: на маленьких таблицах (full scan и так дёшев), при записи (каждый INSERT обновляет все индексы — это замедляет вставку), при низкой селективности (индекс по столбцу `is_active` с двумя значениями почти бесполезен).

Почему запрос медленный, типичные причины: нет индекса на столбце фильтра/джойна; функция на индексированном столбце (`WHERE date_trunc('day', ts) = ...` отключает индекс по `ts` — пиши `ts >= ... AND ts < ...`); JOIN раздувает строки; `SELECT *` тащит лишние столбцы.

## M4.13 · EXPLAIN — читаем план запроса

Интуиция. Прежде чем база выполнит запрос, она строит план — пошаговую стратегию "как именно я это посчитаю". `EXPLAIN` показывает этот план, не выполняя запрос. `EXPLAIN ANALYZE` — выполняет и показывает реальное время.

Формально:

```sql
EXPLAIN ANALYZE
SELECT * FROM deposits WHERE player_id = 12345;
```

Что искать в выводе:

- `Seq Scan` (sequential scan) — база читает таблицу целиком. На большой таблице с фильтром это красный флаг: вероятно, не хватает индекса.
- `Index Scan` — база использует индекс. То, что нужно.
- `Nested Loop` / `Hash Join` / `Merge Join` — алгоритмы соединения. Hash Join хорош для больших таблиц без сортировки, Nested Loop — для маленьких.
- Числа `rows` и `actual time` — оценка и реальность. Если планировщик ждал 100 строк, а пришёл миллион, статистика устарела — стоит выполнить `ANALYZE table`.

Умение читать EXPLAIN — это переход от "запрос работает" к "запрос работает за 50 мс вместо 30 секунд". На больших данных это разница между рабочим дашбордом и таймаутом.

!!! question "Проверь себя"

    1. На что в выводе `EXPLAIN` указывает `Seq Scan` на большой таблице с фильтром и почему это красный флаг?
    2. Чем `EXPLAIN` отличается от `EXPLAIN ANALYZE`?
    3. Почему `WHERE date_trunc('day', ts) = ...` мешает использовать индекс по `ts` и как переписать условие?
    4. Что значит сильное расхождение оценки `rows` и реального числа строк, и что с этим делать?

??? success "Ответы"

    1. `Seq Scan` — чтение всей таблицы подряд; при точечном фильтре на больших данных это признак отсутствия подходящего индекса.
    2. `EXPLAIN` показывает план без выполнения; `EXPLAIN ANALYZE` реально выполняет запрос и показывает фактическое время и число строк.
    3. Функция оборачивает столбец, и индекс по сырому `ts` не применяется. Переписать на диапазон: `ts >= ... AND ts < ...` (или сделать функциональный индекс).
    4. Статистика планировщика устарела — стоит выполнить `ANALYZE table`, чтобы план строился по актуальному распределению.

## M4.14 · Диалекты: PostgreSQL и DuckDB

PostgreSQL — стандарт продакшен-баз: транзакционная (OLTP) и аналитическая работа, надёжность, богатый SQL. Это та база, где живут данные приложения. Скорее всего, твои продуктовые таблицы лежат именно в Postgres или его форке.

DuckDB — стандарт локальной аналитики 2026 года. Это встраиваемая аналитическая (OLAP) база, "SQLite для аналитики". Она колоночная, что делает её в десятки раз быстрее на агрегациях, и работает прямо из питона без сервера. DuckDB читает Parquet и CSV напрямую как таблицы и даже умеет выполнять SQL по pandas-датафреймам в памяти. Для EDA, фичеринга и ad-hoc аналитики на данных до сотен гигабайт DuckDB — лучший выбор на одной машине.

Диалекты на 95% совпадают по синтаксису, который мы прошли. Различия в деталях: функции дат, приведение типов, специфичные расширения. Профессионал пишет переносимый SQL и держит в голове, где диалекты расходятся.

## M4.15 · Код: стек 2026

```python
import duckdb
import pandas as pd

con = duckdb.connect()

# DuckDB читает Parquet/CSV напрямую как таблицу — без загрузки в память
dau = con.execute("""
    SELECT bet_ts::date AS day,
           COUNT(DISTINCT player_id) AS dau,
           SUM(stake) AS total_stake,
           SUM(stake - payout) AS ggr   -- gross gaming revenue
    FROM 'bets.parquet'
    GROUP BY bet_ts::date
    ORDER BY day
""").df()   # .df() возвращает pandas DataFrame

# Депозитные когорты: месяц первого депозита и LTV-кривая
cohorts = con.execute("""
    WITH first_dep AS (
        SELECT player_id,
               date_trunc('month', MIN(deposit_ts)) AS cohort_month,
               MIN(deposit_ts) AS ftd_ts
        FROM 'deposits.parquet'
        GROUP BY player_id
    )
    SELECT f.cohort_month,
           COUNT(DISTINCT f.player_id) AS cohort_size,
           SUM(d.amount) AS revenue,
           SUM(d.amount) / COUNT(DISTINCT f.player_id) AS arpu
    FROM first_dep f
    JOIN 'deposits.parquet' d ON d.player_id = f.player_id
    GROUP BY f.cohort_month
    ORDER BY f.cohort_month
""").df()
```

```python
# Тот же SQL, но против боевого PostgreSQL через SQLAlchemy
from sqlalchemy import create_engine

engine = create_engine("postgresql+psycopg://user:pwd@host:5432/yohoho")
df = pd.read_sql("""
    SELECT country, COUNT(DISTINCT player_id) AS depositors,
           SUM(amount) AS total
    FROM deposits d
    JOIN players p USING (player_id)
    WHERE deposit_ts >= now() - interval '30 days'
    GROUP BY country
    ORDER BY total DESC
""", engine)
```

!!! tip "AI-копилот: где нейросети помогают"

    LLM (Claude в Cursor, Copilot) сильны в SQL: они отлично генерируют скелет запроса по описанию, объясняют чужой запрос на 100 строк, переписывают коррелированный подзапрос в оконную функцию, ловят забытый GROUP BY.

    Хороший промпт даёт схему и точную задачу: "Вот DDL таблиц players(player_id, signup_date, country) и deposits(deposit_id, player_id, amount, deposit_ts). Напиши PostgreSQL-запрос для D7 retention по когортам месяца регистрации. Активность = наличие депозита. Объясни логику оконных функций."

    Где они врут и что перепроверять обязательно: LLM регулярно путает диалекты (предлагает MySQL-функцию `DATE()` для Postgres, или `TOP` вместо `LIMIT`). Не учитывает раздувание строк при JOIN — спокойно выдаст `SUM` по соединению, завышающий метрику в разы. Игнорирует логику NULL в `NOT IN`. И главное: модель не знает реального распределения твоих данных — корректность бизнес-логики (что считать "активным", какая таймзона, что есть FTD) проверяешь только ты. Любой сгенерированный запрос гоняй на маленькой выборке и сверяй итоги вручную.

!!! note "Стек 2026"

    Актуально: DuckDB — де-факто стандарт локальной и embedded-аналитики, читает Parquet/CSV/JSON напрямую и работает по pandas/Polars-датафреймам в памяти. PostgreSQL остаётся королём продакшен-баз. dbt — стандарт для трансформаций SQL в дата-команде (версионирование, тесты, документация моделей). Оконные функции и CTE — обязательная база, без них на интервью не берут.

    Устарело относительно старых курсов: учить MySQL как первый диалект — менее релевантно, аналитика ушла в Postgres/DuckDB. Курсоры и хранимые процедуры для аналитики почти не нужны. Выгрузка всей таблицы в pandas для агрегации, которую можно сделать в SQL, — антипаттерн: считай в базе, тащи в питон уже агрегат.

!!! warning "Ловушки"

    Раздувание строк при JOIN многих таблиц — `SUM` считает значения по несколько раз. Агрегируй до соединения.

    `NOT IN (подзапрос с NULL)` молча возвращает ноль строк. Используй `NOT EXISTS`.

    Функция на столбце в WHERE (`WHERE lower(country) = 'uz'`, `date_trunc(...)`) отключает индекс — переписывай на диапазон или функциональный индекс.

    `WHERE col <> 'x'` теряет строки с `col IS NULL`. Если они нужны — добавь `OR col IS NULL`.

    `COUNT(*)` против `COUNT(DISTINCT player_id)` — путаница даёт неверный DAU. Уникальные пользователи всегда через DISTINCT.

    `SELECT *` в продакшен-запросах и дашбордах — тащит лишние столбцы, ломается при изменении схемы. Перечисляй явно.

    Senior-ловушка: доверять плану на тестовых данных. На 1000 строках Seq Scan быстрее Index Scan, и план врёт о том, что будет на проде с 200 миллионами. Анализируй на реалистичном объёме.

## M4.16 · Практика

Разминка.

1. Выбери топ-20 депозитов за последние 7 дней, отсортированных по убыванию суммы.
2. Посчитай число игроков и средний чек депозита по каждой стране.
3. Найди игроков, у которых суммарный депозит превышает средний по всей базе (подзапрос).

Основное.

4. Построй DAU по дням за месяц, считая активным игрока, сделавшего хотя бы одну ставку.
5. Через `ROW_NUMBER()` найди для каждого игрока его первый депозит (FTD) — сумму и дату.
6. Посчитай GGR (`SUM(stake - payout)`) по дням и нарастающий итог GGR через оконную `SUM(...) OVER (ORDER BY day)`.
7. Сделай LEFT JOIN `players` и `deposits` так, чтобы увидеть страны без единого депозита (revenue = 0 через COALESCE).

Челлендж.

8. Построй таблицу D1/D7/D30 retention по когортам недели регистрации. Активность = ставка.
9. Через `LAG()` посчитай среднее время между первым и вторым депозитом игрока по странам.
10. Возьми медленный запрос из задания 8, прогони `EXPLAIN ANALYZE`, найди Seq Scan, добавь индекс и покажи разницу во времени.

Мини-проект. Собери на DuckDB cohort-retention-отчёт: входные данные — два Parquet-файла (`players`, `bets`), выход — pandas DataFrame с колонками cohort_week, cohort_size, retention_d1, retention_d7, retention_d30. Весь расчёт — на чистом SQL внутри DuckDB, питон только запускает запрос и отдаёт результат. Построй тепловую карту retention по когортам (cohort_week по строкам, лаг по столбцам).
