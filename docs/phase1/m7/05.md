# M7.5 · REST API: забор данных правильно

API отдаёт данные по HTTP. Вы шлёте запрос на endpoint, получаете ответ (обычно JSON). Проблема в том, что данных много, а за один запрос отдают порцию, сервер вас ограничивает по частоте, и требует доказать, что вам можно.

Разберём на пальцах, как устроен один цикл. Вы как аналитик хотите выгрузить все депозиты за месяц из платёжного провайдера. Вы делаете `GET /v1/transactions?from=2026-05-01&to=2026-05-31`. Провайдер не присылает 2 миллиона строк — он присылает первые 100 и говорит «дальше бери по курсору». Если вы будете долбить его в цикле без пауз — на 101-м запросе получите `429` и блок на минуту. А если ваш ключ лежит в коде и репозиторий случайно стал публичным — провайдер увидит чужие запросы с вашим ключом и счёт за чужой трафик. Все четыре темы ниже — про то, как пройти этот цикл без потерь.

### Библиотеки: httpx как дефолт 2026

Классика — библиотека `requests`, она жива и прекрасно работает для простых синхронных задач. Но стандарт 2026 для нового кода — `httpx`: тот же знакомый API, плюс встроенная поддержка асинхронности (можно лить сотни запросов параллельно), HTTP/2 и таймаутов из коробки. Для тяжёлого параллельного забора это решает. Учтите: HTTP/2 не входит в базовую установку — для `http2=True` нужен extra-пакет `httpx[http2]` (он подтянет зависимость `h2`).

Один нюанс, на котором спотыкаются почти все новички: `requests` по умолчанию ждёт ответа **бесконечно**. Если сервер завис, ваш скрипт зависнет вместе с ним — и забор данных в полночь молча встанет до утра. `httpx` ставит дефолтный таймаут 5 секунд, но в реальном коде лучше задавать его явно.

```python
import httpx

# Явный таймаут на connect/read/write/pool — обязательная гигиена
client = httpx.Client(timeout=httpx.Timeout(10.0, connect=5.0))

resp = client.get(
    "https://api.example.com/v1/transactions",
    params={"from": "2026-05-01", "to": "2026-05-31", "limit": 100},
    headers={"Authorization": "Bearer <token>"},
)
resp.raise_for_status()  # бросит исключение на 4xx/5xx — не глотаем ошибку молча
data = resp.json()
```

Привычка `raise_for_status()` важна: без неё `resp.json()` на ответе `404` или `500` либо упадёт с невнятной ошибкой парсинга, либо (хуже) вернёт тело с описанием ошибки, и вы запишете в витрину мусор вместо данных.

### Пагинация

Сервер не отдаст вам миллион транзакций одним ответом. Он режет их на страницы. Три основных схемы.

Offset/limit: вы просите «дай 100 строк начиная с 200-й» (`?limit=100&offset=200`). Просто, но ломается, если данные меняются во время обхода (вставили строку — сдвинулись все offset, получите дубли или пропуски).

Page-based: `?page=3&per_page=100`. То же самое другими словами.

Cursor-based (курсорная): сервер в каждом ответе отдаёт указатель на следующую порцию (`next_cursor`), вы передаёте его в следующий запрос. Устойчива к изменениям данных, потому что курсор привязан к позиции в стабильном порядке, а не к номеру. Это предпочтительная схема для больших и живых данных.

Почему offset реально ломается — на конкретном примере. Вы выгружаете игроков, отсортированных по дате регистрации, страницами по 100. Прошли первую страницу (offset 0–99). Пока вы её обрабатывали, зарегистрировались 3 новых игрока — и поскольку сортировка по дате по убыванию, они встали **в начало** списка. Теперь строки, которые были на позициях 97, 98, 99, сдвинулись на 100, 101, 102. Вы запрашиваете `offset=100` и получаете заново те же три записи — **дубли**. При удалении было бы наоборот: строки сдвинулись бы вверх, и вы бы **пропустили** ровно столько записей, сколько удалили. Курсор такой проблемы не создаёт: `next_cursor` обычно кодирует «последний виденный id/timestamp», и следующий запрос значит «дай всё, что строго после этого значения» — независимо от того, сколько строк вставили или удалили выше.

Типовой скелет обхода курсором:

```python
def fetch_all(client, url, params):
    cursor = None
    while True:
        page_params = {**params, "limit": 100}
        if cursor:
            page_params["cursor"] = cursor
        resp = client.get(url, params=page_params)
        resp.raise_for_status()
        body = resp.json()
        yield from body["data"]
        cursor = body.get("next_cursor")
        if not cursor:          # сервер перестал отдавать курсор — конец данных
            break
```

Типичная ошибка здесь — условие выхода. Не завязывайтесь на «пришло меньше 100 строк, значит конец»: некоторые API на последней полной странице всё равно отдают курсор, а пустую страницу — следующим запросом. Надёжный признак конца — **отсутствие `next_cursor`**, а не размер порции.

### Rate limits

Сервер защищается от перегрузки и ограничивает вас: например, 100 запросов в минуту. Превысите — получите HTTP 429 Too Many Requests. Грамотный клиент это уважает: читает заголовки (`Retry-After`, `X-RateLimit-Remaining`) и при 429 ждёт. Стандартная стратегия повтора — экспоненциальная выдержка с джиттером: каждая следующая попытка ждёт вдвое дольше плюс случайная добавка, чтобы тысяча клиентов не ломанулись одновременно.

$$
t_{\text{ожидания}} = \min\left(t_{\max},\; b \cdot 2^{\,n}\right) + U(0, j)
$$

где $n$ — номер попытки, $b$ — базовая задержка, $t_{\max}$ — потолок, $U(0,j)$ — случайный джиттер.

Посчитаем на числах. Пусть базовая задержка $b = 1$ секунда, потолок $t_{\max} = 30$ секунд, джиттер до $j = 1$ секунды. Тогда «голая» выдержка $b \cdot 2^n$ по попыткам: 1, 2, 4, 8, 16, затем срабатывает потолок и дальше всегда 30. Реальная пауза — это плюс случайная добавка от 0 до 1 секунды. Без потолка $2^n$ улетел бы в космос (на 10-й попытке это уже ~17 минут), поэтому $t_{\max}$ обязателен. А джиттер критичен вот почему: если в момент сбоя API 1000 ваших воркеров одновременно получили `429`, то без джиттера все они подождут ровно 2 секунды и ударят по серверу синхронно — получится «thundering herd», новый всплеск, новый `429`, и так по кругу. Случайный разброс размазывает повторы по времени.

Важный приоритет: если сервер прислал заголовок `Retry-After` (в секундах или как HTTP-дата), он главнее вашей формулы — ждите именно столько, сколько просит сервер. Формула — это fallback, когда сервер ничего конкретного не сказал. И различайте коды: `429` и `503` имеет смысл повторять с выдержкой, а `400`/`401`/`404` повторять бессмысленно — это ошибка вашего запроса, и тысяча повторов её не исправит.

```python
import random, time

def get_with_retry(client, url, params, b=1.0, t_max=30.0, j=1.0, max_tries=6):
    for n in range(max_tries):
        resp = client.get(url, params=params)
        if resp.status_code != 429:
            resp.raise_for_status()
            return resp
        retry_after = resp.headers.get("Retry-After")
        if retry_after:
            wait = float(retry_after)            # сервер сказал явно — слушаем его
        else:
            wait = min(t_max, b * 2 ** n) + random.uniform(0, j)
        time.sleep(wait)
    raise RuntimeError("rate limit: исчерпаны попытки")
```

### Авторизация

Сервер должен знать, что вы — это вы. Основные способы: API-ключ в заголовке (`Authorization: Bearer <token>`), Basic Auth (логин/пароль), OAuth 2.0 (токен с ограниченным сроком, который надо обновлять). Железное правило: ключи и токены никогда не в коде и не в git. Только переменные окружения или менеджер секретов.

На практике это выглядит так: секрет лежит в `.env` (который добавлен в `.gitignore` и **не** коммитится), а код читает его из окружения.

```python
import os

token = os.environ["PAYMENTS_API_TOKEN"]   # KeyError, если забыли задать — это хорошо, заметно сразу
headers = {"Authorization": f"Bearer {token}"}
```

Разница между способами на пальцах: API-ключ и Basic Auth — это «долгоживущий пароль», который вы шлёте в каждом запросе; если он утёк — доступ у злоумышленника, пока вы вручную не отзовёте ключ. OAuth 2.0 безопаснее тем, что отдаёт **короткоживущий** access-токен (минуты-часы) плюс refresh-токен для обновления: даже если access-токен утёк, окно злоупотребления узкое. Поэтому крупные провайдеры (платёжные шлюзы, рекламные кабинеты, Google/Meta API для маркетинг-аналитики) почти всегда на OAuth 2.0, а не на голом ключе.

!!! warning "Секреты в коде"

    Закоммиченный API-ключ — это инцидент безопасности, даже если репозиторий приватный. История git помнит всё; удалить ключ из последнего коммита недостаточно. Если ключ утёк — его надо отозвать и перевыпустить, а не «подчистить».

!!! question "Проверь себя"

    1. Почему cursor-пагинация устойчивее offset/limit при обходе живых, меняющихся данных?
    2. Что означает HTTP 429 и как грамотный клиент на него реагирует?
    3. Зачем в стратегии повторов добавляют случайный джиттер, а не просто удваивают задержку?

??? success "Ответы"

    1. Курсор привязан к позиции в стабильном порядке, а не к номеру строки: вставка/удаление данных во время обхода не сдвигает уже пройденное, поэтому нет дублей и пропусков.
    2. Too Many Requests — превышен лимит частоты; клиент читает `Retry-After`/`X-RateLimit-Remaining` и ждёт перед повтором, а не долбит сервер.
    3. Чтобы тысяча клиентов, упавших на лимит одновременно, не повторили запрос в один и тот же момент и не создали новый всплеск нагрузки (thundering herd).
