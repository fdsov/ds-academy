# M7.3 · Откуда берутся данные

Прежде чем чистить, надо достать. Источников немного, и важно понимать характер каждого, потому что он определяет, какие именно загрязнения вы получите. Принцип, который стоит запомнить на всю карьеру: **тип источника предсказывает тип грязи**. Если вы знаете, откуда пришли данные, вы уже знаете, что в них искать, и не потратите час на поиск проблемы, которую можно было ожидать с порога. Базы дают сравнительно чистые типы, но размазанную бизнес-логику; API — сетевые сбои; файлы — испорченные значения; события — дубли и кривое время. Дальше — по каждому источнику подробно.

### Базы данных

Транзакционные СУБД (PostgreSQL, MySQL) — это первоисточник для большинства продуктовых данных: депозиты, ставки, пользователи, сессии. Данные здесь обычно нормализованы и относительно чисты на уровне типов, потому что схема их принуждает. Но нормализация означает, что бизнес-сущность размазана по десятку таблиц, и собрать её — отдельная работа (джойны, см. модуль про SQL). Аналитические хранилища (ClickHouse, BigQuery, Snowflake, DuckDB локально) — это денормализованные витрины, заточенные под чтение больших объёмов. В 2026 году типичный путь данных такой: продуктовая Postgres -> ETL/ELT -> аналитическое хранилище -> ваш ноутбук.

«Схема принуждает корректность» на пальцах: если в Postgres колонка `amount` объявлена как `numeric(12,2)`, туда физически нельзя записать строку `"много"` или дату — база отклонит вставку. Поэтому из транзакционной базы вы почти никогда не получите текст в числовой колонке. Зато получите другое: одна ставка живёт в `bets`, игрок в `users`, его страна в `geo`, валюта депозита в `payments` — и чтобы посчитать «средний депозит игроков из Бразилии», нужно аккуратно соединить четыре таблицы, не размножив и не потеряв строки на джойнах.

!!! info "Интуиция: транзакционная база против аналитической"

    Транзакционная база — это касса в магазине: задача записать каждую покупку быстро и без ошибки, по одной за раз. Аналитическая база — это склад с отчётами: задача быстро ответить «сколько всего продали красных кроссовок за квартал», читая миллионы строк разом. Формально: первые оптимизированы под OLTP (много мелких записей, строковое хранение), вторые под OLAP (массовое чтение, колоночное хранение).

Почему колоночное хранение быстрее для аналитики — конкретно. Запрос `SELECT avg(deposit_amount) FROM payments` должен прочитать одну колонку из, скажем, 40. В строковом хранении (OLTP) строки лежат целиком подряд, поэтому чтобы добраться до `deposit_amount`, движок всё равно тянет с диска все 40 колонок каждой строки — 39/40 работы впустую. В колоночном хранении (OLAP) значения `deposit_amount` лежат отдельным непрерывным блоком: читаем только его, плюс одинаковые значения в колонке отлично сжимаются. Отсюда правило: тяжёлый аналитический `GROUP BY` по миллионам строк гоните в ClickHouse/BigQuery, а не в боевую Postgres — иначе вы нагрузите кассу, по которой в этот момент идут реальные платежи игроков.

### API

Когда данных нет в вашей базе, их отдают по API: платёжные провайдеры, антифрод-сервисы, биржевые котировки, рекламные кабинеты. Это сетевой источник, и потому самый ненадёжный: таймауты, частичные ответы, лимиты запросов, изменения схемы без предупреждения. Разберём ниже отдельно.

Главная ловушка API — **молчаливая частичность**. Сетевой вызов может вернуть код 200 OK и при этом отдать неполные данные: пагинация оборвалась на третьей странице из десяти, провайдер по тихому лимиту вернул только 1000 транзакций вместо 5000. Если вы не проверяете полноту, вы недосчитаете депозитов и решите, что день был слабый, хотя слабым был не день, а ответ API.

```python
import time
import requests

def fetch_all_deposits(base_url, params, max_retries=3):
    results, page = [], 1
    while True:
        for attempt in range(max_retries):
            r = requests.get(base_url, params={**params, "page": page}, timeout=10)
            if r.status_code == 429:          # rate limit — ждём и повторяем
                time.sleep(2 ** attempt)       # экспоненциальный backoff
                continue
            r.raise_for_status()
            break
        else:
            raise RuntimeError(f"страница {page} не загрузилась после {max_retries} попыток")
        batch = r.json()["data"]
        results.extend(batch)
        if not batch:                          # пустая страница — данные кончились
            break
        page += 1
    return results
```

Два приёма здесь обязательны: **backoff на 429** (не долбить сервер сразу же, а ждать всё дольше) и **явное условие конца пагинации**. Типичная ошибка новичка — взять первую страницу, увидеть 1000 записей и решить, что это все данные.

### Файлы

Выгрузки, которые присылают коллеги или партнёры: CSV, Excel, JSON. Самый грязный источник в принципе, потому что файл — это снимок без схемы и без гарантий. Excel особенно коварен: он молча превращает `00123` в число `123`, дату ставки в серийный номер, а большое число — в экспоненциальную запись.

Разберём конкретный случай, который ломает аналитику тихо. Партнёрский ID игрока `00123` в базе — это строка с ведущими нулями. Excel при открытии CSV видит «число» и превращает его в `123`. Теперь при джойне выгрузки с вашей базой по полю `player_id` строка `"00123"` не совпадёт с `"123"` — игрок «потеряется», и в отчёте по партнёру окажется недосчёт. Аналогично дата `15.03.2026` может стать серийным числом `46096` (число дней от эпохи Excel), а длинный ID транзакции `1234567890123456` — превратиться в `1.23457E+15`, безвозвратно потеряв младшие цифры.

```python
import pandas as pd

# Плохо: pandas сам угадает типы и тоже срежет ведущие нули
df_bad = pd.read_csv("partners.csv")

# Хорошо: критичные ID читаем строго как строки
df = pd.read_csv(
    "partners.csv",
    dtype={"player_id": str, "transaction_id": str},
    keep_default_na=False,   # чтобы "NA"/"" не превратились молча в NaN
)
```

Правило-рефлекс: любые идентификаторы (player_id, transaction_id, телефон, промокод) читать как `str`, а не доверять автоопределению типов. Один потерянный ведущий ноль — это разъехавшийся джойн и неверная цифра в отчёте, которую потом ищут полдня.

### Скрапинг

Когда у источника нет API, данные вытягивают со страниц. Это хрупкий способ (вёрстка меняется), юридически серый (проверяйте Terms of Service и robots.txt) и требует уважения к источнику (rate limiting, чтобы не положить чужой сервер). В DS-практике применяется для конкурентной разведки, сбора коэффициентов, парсинга публичных реестров.

Конкретный пример из гемблинга: мониторинг линий конкурентов. Вы каждые несколько минут снимаете коэффициенты с публичных страниц нескольких букмекеров, чтобы видеть, где ваша маржа выбивается из рынка. Три вещи, которые отличают аккуратный скрапинг от безответственного: задержка между запросами (например, 1-2 секунды, чтобы не имитировать DDoS), `User-Agent`, по которому вас можно опознать, и устойчивость к изменению вёрстки — селектор `.odds-value` может исчезнуть после редизайна, и парсер должен это заметить и упасть громко, а не молча писать `None` в коэффициенты.

### Продуктовые события

Самый важный для продуктовой аналитики источник — event tracking. Каждое значимое действие пользователя (открыл приложение, сделал ставку, нажал «пополнить») шлётся событием в систему сбора (в 2026 это чаще всего что-то из стека Snowplow, Segment-подобных CDP, или собственный event-collector, льющий в Kafka). Из этих сырых событий потом собираются метрики: DAU, конверсия воронки, retention-кривые. Загрязнения здесь специфические: дубли из-за ретраев клиента, события из будущего из-за кривых часов на устройстве, потерянные события из-за блокировщиков.

Почему именно эти три загрязнения — разберём механику каждого, потому что лечатся они по-разному:

- **Дубли из-за ретраев.** Клиент отправил событие `bet_placed`, но из-за плохой мобильной сети не получил подтверждения и отправил ещё раз. Сервер получил два одинаковых события об одной ставке. Если не дедуплицировать (обычно по уникальному `event_id`, который клиент генерирует один раз), вы насчитаете в 1.5-2 раза больше ставок, чем было.
- **События «из будущего».** Время события часто проставляет само устройство пользователя. У части игроков часы выставлены неверно или в чужом часовом поясе, поэтому прилетает ставка с меткой времени `2027-01-01`. Это ломает дневные графики и retention-окна. Лекарство — опираться на серверное время приёма (`server_timestamp`), а клиентское использовать с осторожностью.
- **Потерянные события из-за блокировщиков.** Часть пользователей с AdBlock/anti-tracking режет запросы к коллектору. Их действия просто не доезжают, и вы систематически недооцениваете аудиторию — причём не случайно, а со смещением (теряете именно технически продвинутых юзеров).

!!! warning "Типичная ошибка: считать событийный поток полным и точным"

    Сырые события — это не истина в последней инстанции, а зашумлённый сигнал. Прежде чем строить на них метрику, всегда проверяйте: дедуплицированы ли события по `event_id`, на какое время вы опираетесь (серверное или клиентское) и нет ли систематической недодоставки. Метрика DAU, посчитанная по недедуплицированным событиям с клиентским временем, может разойтись с реальностью на десятки процентов — и вы будете принимать решения по фантому.

!!! question "Проверь себя"

    1. Чем транзакционная СУБД (OLTP) отличается от аналитического хранилища (OLAP) по задаче и способу хранения?
    2. Почему файловые выгрузки (CSV, Excel) считаются самым грязным источником, а не базы данных?
    3. Какие три специфических загрязнения характерны именно для потока продуктовых событий?

??? success "Ответы"

    1. OLTP оптимизирована под множество мелких записей по одной (касса), хранение чаще строковое; OLAP — под массовое чтение больших объёмов (склад отчётов), хранение колоночное.
    2. Файл — это снимок без схемы и без гарантий на типы; никакая система не принуждает корректность, плюс Excel молча портит значения (`00123` -> `123`, даты в серийники).
    3. Дубли из-за клиентских ретраев, события «из будущего» из-за кривых часов на устройстве, потерянные события из-за блокировщиков.
