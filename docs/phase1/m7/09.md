# M7.9 · Валидация данных как стандарт 2026

Ключевой сдвиг профессионального уровня: вы не чистите данные руками каждый раз, а декларативно описываете, какими данные должны быть, и проверка становится частью пайплайна. Если данные не соответствуют контракту — пайплайн громко падает, а не тихо считает мусор.

Почему это критично именно в аналитике: ошибки данных не «крашат» программу — они дают правдоподобный, но неверный ответ. Скрипт посчитает средний депозит, отчёт уйдёт продакту, продакт примет решение, и только через неделю кто-то заметит, что в выгрузку попали тестовые аккаунты с депозитом 999 999. Тихая ошибка дороже громкой: громкая падает сразу и стоит 5 минут, тихая всплывает в решении и стоит недели. Контракт на данные превращает тихие ошибки в громкие — на входе, а не в выводе.

### Pydantic

Pydantic валидирует отдельные записи (например, каждый объект из API-ответа) на уровне типов и правил Python. Идеален на границе между внешним миром и вашим кодом: распарсил JSON — сразу проверил.

Интуиция «на пальцах»: Pydantic работает построчно — один объект за раз. Вы описываете класс-модель, и каждый словарь из API превращается в типизированный объект; если поле не того типа или нарушает правило — летит понятное исключение с указанием, какое именно поле и почему сломалось.

```python
from pydantic import BaseModel, field_validator

class Deposit(BaseModel):
    user_id: int
    amount: float
    currency: str

    @field_validator("amount")
    @classmethod
    def amount_positive(cls, v: float) -> float:
        if v <= 0:
            raise ValueError("amount must be positive")
        return v

# приходит из API
raw = {"user_id": "1024", "amount": "49.90", "currency": "USD"}
dep = Deposit(**raw)   # "1024" -> 1024, "49.90" -> 49.9 (приведение типов)
print(dep.amount)      # 49.9

bad = {"user_id": "1024", "amount": "-5", "currency": "USD"}
Deposit(**bad)         # ValidationError: amount must be positive
```

Обратите внимание: Pydantic не просто проверяет, но и приводит типы — строка `"1024"` стала `int`, `"49.90"` стала `float`. Это и есть «граница»: грязный внешний JSON входит, чистый типизированный объект выходит.

### Pandera

Pandera валидирует таблицы (DataFrame целиком): типы колонок, диапазоны, уникальность, отсутствие пропусков, межколоночные правила. Это то, что нужно для DataFrame-пайплайнов на pandas и Polars. В 2026 Pandera поддерживает оба бэкенда.

Интуиция «на пальцах»: если Pydantic смотрит на одну запись, то Pandera смотрит на колонку целиком и на отношения между колонками. Вопросы, на которые отвечает Pandera: «все ли значения в `amount` ≥ 0?», «уникален ли `user_id`?», «нет ли пропусков в `currency`?», «всегда ли `withdraw_amount <= deposit_amount`?».

```python
import pandas as pd
import pandera.pandas as pa
from pandera import Column, Check

schema = pa.DataFrameSchema({
    "user_id":  Column(int, unique=True, nullable=False),
    "amount":   Column(float, Check.in_range(0, 1_000_000), nullable=False),
    "currency": Column(str, Check.isin(["USD", "EUR", "RUB"])),
})

df = pd.DataFrame({
    "user_id":  [1, 2, 2],            # дубликат -> упадёт на unique
    "amount":   [49.9, 12.0, 5.0],
    "currency": ["USD", "EUR", "USD"],
})

schema.validate(df, lazy=True)   # lazy=True соберёт ВСЕ ошибки, а не первую
```

`lazy=True` — важная деталь: без него валидация падает на первой же ошибке, с ним собирает полный отчёт обо всех нарушениях сразу. Для разбора качества выгрузки это удобнее: видишь все проблемы за один прогон.

Межколоночное правило — то, чего нельзя выразить через типы:

```python
schema = pa.DataFrameSchema(
    {
        "deposit":  Column(float, Check.ge(0)),
        "withdraw": Column(float, Check.ge(0)),
    },
    checks=Check(lambda d: d["withdraw"] <= d["deposit"],
                 error="withdraw exceeds deposit"),
)
```

### Great Expectations

Great Expectations — тяжёлый промышленный инструмент для data quality на уровне всей платформы: декларативные «ожидания», автогенерация документации качества (Data Docs), интеграция с оркестраторами. Для отдельного пайплайна избыточен, для платформенного контроля качества — стандарт.

Чем он отличается от Pandera по сути: Pandera — это библиотека-проверка внутри вашего кода (схема живёт в репозитории рядом с пайплайном). Great Expectations — это система с собственным хранилищем ожиданий, версионированием, HTML-документацией качества (Data Docs), которую можно показать нетехническому стейкхолдеру, и встройкой в Airflow/Dagster. Он окупается, когда проверок много, источников много и за качество отвечает команда, а не один человек.

!!! tip "Интуиция: контракт на данные"

    Раньше проверки данных писали как разрозненные `assert` по тексту скрипта, и они расползались и устаревали. Современный подход — собрать все ожидания в одну схему-контракт рядом с данными. Это как типизация в коде: вместо «надеюсь, тут число» вы объявляете «тут число от 0 до 1 000 000, без пропусков», и нарушение ловится автоматически на входе, а не всплывает в отчёте через неделю.

!!! example "Когда что выбирать: связка на одном пайплайне"

    Эти три инструмента не конкурируют — они стоят на разных рубежах. Типичный продакшен-поток выгрузки депозитов:

    - на границе с API каждую запись чистит и приводит к типам **Pydantic**;
    - собранный из записей DataFrame целиком проверяет **Pandera** (уникальность `user_id`, диапазон `amount`, межколоночное `withdraw <= deposit`);
    - на уровне платформы качество всех таблиц мониторит и документирует **Great Expectations** с Data Docs для стейкхолдеров.

    Правило выбора: одна запись на границе — Pydantic; таблица в пайплайне — Pandera; платформенный контроль качества и документация — Great Expectations.

!!! warning "Типичные ошибки"

    - Ставить валидацию в конце пайплайна, после всех преобразований. Контракт должен стоять на **входе** — чтобы мусор не дошёл до расчётов.
    - Проверять `amount` только на тип `float`, но не на диапазон. Тип пропустит и `-5`, и `999_999_999` — нужен `Check.in_range`.
    - Использовать Pandera без `lazy=True` при разборе качества: увидите только первую ошибку и будете чинить выгрузку по одной проблеме за прогон.
    - Тянуть Great Expectations на один скрипт «чтобы по-взрослому» — это месяцы настройки ради проверки, которую Pandera делает в 10 строк.

!!! question "Проверь себя"

    1. Что валидирует Pydantic и что — Pandera, и где проходит граница их применения?
    2. Чем декларативный контракт-схема лучше россыпи `assert` по тексту скрипта?
    3. Когда оправдан тяжёлый Great Expectations, а когда он избыточен?

??? success "Ответы"

    1. Pydantic проверяет отдельные записи (объекты из API на границе с внешним миром); Pandera — таблицы целиком (типы колонок, диапазоны, уникальность, межколоночные правила) в DataFrame-пайплайнах.
    2. Контракт собирает все ожидания в одном месте рядом с данными, не расползается и не устаревает по скрипту; нарушение ловится автоматически на входе, а не всплывает в отчёте через неделю — это типизация для данных.
    3. Great Expectations оправдан для data quality на уровне всей платформы (Data Docs, интеграция с оркестраторами); для одного пайплайна он избыточен — там достаточно Pandera/Pydantic.
