# M7.4 · Форматы данных

Формат — это то, в чём данные физически лежат на диске и ездят между системами. Выбор формата влияет на скорость, объём и на то, сохранятся ли типы.

Важно сразу различать два разных вопроса, которые часто путают. Первый: *как данные закодированы на диске* — текст или бинарь, построчно или поколоночно (это и есть формат: CSV, JSON, Parquet). Второй: *как данные лежат в оперативной памяти, пока их обрабатывает инструмент* (это представление: например, Arrow). Один и тот же датасет может прочитаться из Parquet-файла и развернуться в Arrow-память — это две стороны одного пайплайна, а не конкурирующие варианты.

### CSV и JSON

CSV — текст, разделённый запятыми. Прост, читается глазом, открывается в Excel. Минусы фундаментальны: нет типов (всё строки, парсер угадывает), нет схемы, проблемы с разделителями внутри значений, проблемы с кодировками. Для обмена с людьми сойдёт, для пайплайна — нет.

JSON хорош для вложенных структур и ответов API, человекочитаем, но многословен и медленно парсится на больших объёмах. Для построчной потоковой обработки используют JSON Lines (NDJSON) — один JSON-объект на строку.

!!! warning "Чем «нет типов» бьёт на практике"

    Это не теоретическая придирка. Типичные грабли при чтении CSV в гемблинг-аналитике:

    - `user_id` вида `0071234` парсер прочитает как число `71234` — ведущие нули потеряны, джойн с другой таблицей развалится.
    - `amount` с запятой как десятичным разделителем (`1,50`) в CSV с разделителем-запятой расколется на две колонки.
    - `timestamp` прочитается как строка, и сортировка пойдёт лексикографически: `"2026-06-09"` окажется «больше» `"2026-06-10"`, если форматы поехали.
    - Пустая ячейка — это `NULL`, пустая строка или текст `"NaN"`? CSV не знает, угадывает каждый инструмент по-своему.

    В Parquet таких сюрпризов нет: тип записан в файле, и `user_id` останется строкой `"0071234"`, потому что так было при записи.

### Parquet и Arrow как стандарт 2026

В 2026 году дефолт для аналитических данных — Apache Parquet на диске и Apache Arrow в памяти. Это надо понять, а не заучить.

!!! tip "Интуиция: строки против колонок"

    Представьте таблицу ставок: колонки `user_id`, `amount`, `game`, `timestamp`. CSV хранит её построчно: сначала вся первая ставка, потом вся вторая. Чтобы посчитать сумму депозитов, придётся прочитать с диска всё, включая ненужные `game` и `timestamp`. Parquet хранит поколоночно: все `amount` лежат подряд. Нужна сумма — читаем только колонку `amount`, остальное не трогаем. Плюс одинаковые значения в колонке отлично жмутся.

Формально, Parquet — это бинарный колоночный формат со встроенной схемой (типы хранятся внутри файла), сжатием на уровне колонок и статистикой по блокам (min/max), позволяющей пропускать целые куски файла при фильтрации (predicate pushdown). На реальных аналитических данных Parquet даёт в 5-20 раз меньший размер и кратно более быстрое чтение, чем CSV, и при этом не теряет типы: число останется числом, дата — датой.

!!! note "Почему колонка жмётся, а строка — нет"

    Сжатие работает тем лучше, чем более похожи соседние байты. В строке рядом лежат разнотипные значения: `571`, `12.50`, `"poker"`, `"2026-06-16T20:01"` — общего мало, жать почти нечего. В колонке рядом лежат однотипные значения одного смысла: подряд идут суммы `12.50, 9.99, 12.50, 25.00…` или подряд имена игр `"poker", "slots", "poker"…`. Parquet применяет к таким колонкам словарное кодирование (заменяет повторяющиеся строки на короткие коды) и run-length encoding (серию одинаковых значений хранит как «значение × количество»). Именно поэтому колонка `game` с десятком уникальных игр на миллионе строк сожмётся в десятки раз, а колонка-UUID почти не сожмётся — в ней каждое значение уникально.

Predicate pushdown стоит разобрать на числах. Допустим, файл со ставками за месяц разбит на блоки (row groups), и для каждого блока Parquet хранит min и max по колонке `timestamp`. Вы пишете фильтр `WHERE timestamp >= '2026-06-15'`. Движок смотрит статистику блоков: если у блока `max(timestamp) = '2026-06-10'`, весь блок пропускается, не читая ни байта данных. Из 30 блоков-дней под фильтр последних двух недель прочитаются ~15 — половина диска не тронута. С CSV так нельзя в принципе: чтобы узнать, какие строки подходят, их сначала надо все прочитать и распарсить.

Arrow — это спецификация представления тех же колоночных данных в оперативной памяти, общая для разных инструментов (pandas, Polars, DuckDB, Spark). Её ценность в нулевой стоимости передачи: инструменты обмениваются данными без сериализации и копирования. Практический вывод для вас: промежуточные и итоговые датасеты сохраняйте в Parquet, а не в CSV. CSV оставьте для обмена с не-инженерами.

!!! tip "Что значит «zero-copy» на пальцах"

    Обычная передача данных между двумя инструментами — это сериализация (превратить таблицу в поток байт по своим правилам), запись, чтение, десериализация (собрать таблицу обратно по другим правилам). На гигабайтах это дорого и по времени, и по памяти. Arrow задаёт единый бинарный layout колонок в памяти. Если Polars и DuckDB оба говорят на Arrow, DuckDB читает ровно те же байты, что Polars уже держит в RAM, — никакой конвертации, передаётся лишь указатель на область памяти. Отсюда термин zero-copy: данные физически не копируются и не переупаковываются.

$$
\text{Размер}_{\text{Parquet}} \approx \frac{\text{Размер}_{\text{CSV}}}{C}, \quad C \in [5, 20] \text{ на типичных аналитических данных}
$$

!!! info "Когда какой формат выбрать — шпаргалка"

    - Отдать отчёт менеджеру, открыть в Excel, посмотреть глазами → **CSV**.
    - Ответ API, конфиг, вложенные структуры (массив ставок внутри сессии) → **JSON / NDJSON**.
    - Промежуточный или итоговый аналитический датасет, который будут читать инструменты (pandas, Polars, DuckDB) → **Parquet**.
    - Передача данных между инструментами в рамках одного процесса/пайплайна без записи на диск → **Arrow** (in-memory).

    Частая ошибка джуна: гонять многогигабайтные промежуточные датасеты пайплайна через CSV «потому что привычно». Это и медленно (парсинг текста), и опасно (теряются типы между шагами).

!!! question "Проверь себя"

    1. Почему для подсчёта суммы по одной колонке Parquet читает с диска меньше данных, чем CSV?
    2. Что Parquet хранит внутри файла такого, чего принципиально нет у CSV?
    3. В чём ценность Arrow при передаче данных между инструментами (pandas, Polars, DuckDB)?

??? success "Ответы"

    1. Parquet колоночный: все значения одной колонки лежат подряд, поэтому читается только нужная колонка, а не все строки целиком, как в построчном CSV.
    2. Встроенную схему (типы), сжатие на уровне колонок и блочную статистику (min/max) для predicate pushdown — пропуска кусков файла при фильтрации.
    3. Общее представление в памяти даёт передачу с нулевой стоимостью: без сериализации и копирования данных между инструментами.

!!! example "Мини-задание"

    Возьмите любой свой DataFrame на 100k+ строк (или сгенерируйте). Сохраните его и в CSV, и в Parquet, затем сравните размеры файлов и время чтения:

    ```python
    import polars as pl, os, time

    df.write_csv("data.csv")
    df.write_parquet("data.parquet")
    print("CSV  MB:", os.path.getsize("data.csv") / 1e6)
    print("PARQ MB:", os.path.getsize("data.parquet") / 1e6)
    ```

    Посчитайте коэффициент сжатия $C$. Попадает ли он в обещанный диапазон $[5, 20]$ на ваших данных? Если нет — подумайте, почему (много текста с высокой кардинальностью жмётся хуже).

    Затем сравните и время чтения — именно ради него всё и затевалось:

    ```python
    t = time.time(); pl.read_csv("data.csv");     print("CSV  read s:", time.time() - t)
    t = time.time(); pl.read_parquet("data.parquet"); print("PARQ read s:", time.time() - t)
    ```

    Бонус: прочитайте из Parquet только одну колонку — `pl.read_parquet("data.parquet", columns=["amount"])` — и убедитесь, что это быстрее, чем читать весь файл. С CSV такой трюк не сработает: чтобы достать одну колонку, файл всё равно придётся распарсить целиком.

### Быстрый квиз

Выбери вариант — ответ подсветится сразу.

```quiz
Почему Parquet при подсчёте суммы по колонке amount читает с диска меньше данных, чем CSV?
[ ] Parquet всегда сжимает данные сильнее, поэтому файл меньше целиком
[x] Parquet колоночный — значения одной колонки лежат подряд, и читается только нужная колонка
[ ] Parquet хранит заранее посчитанную сумму в заголовке файла
> CSV построчный, поэтому ради одной колонки приходится прочитать все строки целиком. Parquet читает только колонку amount.
---
Что из перечисленного есть у Parquet, но принципиально отсутствует у CSV?
[x] Встроенная схема с типами и блочная статистика min/max
[ ] Возможность открыть файл в текстовом редакторе
[ ] Разделение значений запятыми
> Parquet хранит типы внутри файла и статистику по блокам для predicate pushdown. CSV — это просто текст без схемы.
---
В чём ценность Arrow при обмене данными между Polars и DuckDB в одном процессе?
[ ] Arrow сильнее сжимает файлы на диске, чем Parquet
[x] Общий layout в памяти даёт zero-copy: данные не сериализуются и не копируются
[ ] Arrow автоматически переводит данные в CSV для совместимости
> Arrow — это представление в памяти. Если оба инструмента говорят на Arrow, передаётся указатель на ту же область RAM, без конвертации.
---
Колонка с каким содержимым сожмётся в Parquet ХУЖЕ всего?
[ ] game с десятком уникальных названий игр на миллионе строк
[ ] amount, где много повторяющихся сумм подряд
[x] user_uuid, где каждое значение уникально
> Сжатие (словарное, RLE) выигрывает на повторах. Уникальные UUID не повторяются — жать почти нечего.
---
Что такое predicate pushdown в Parquet?
[x] Пропуск целых блоков файла по статистике min/max, если они не проходят фильтр
[ ] Перенос вычислений с CPU на GPU при чтении
[ ] Автоматическое добавление недостающих колонок при записи
> По min/max каждого row group движок понимает, что блок не содержит подходящих строк, и не читает его вовсе.
```
