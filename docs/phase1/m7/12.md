Забор данных из API с пагинацией, обработкой 429 и авторизацией через переменные окружения.

```python
import os
import time
import random
import httpx

def fetch_all_events(base_url: str) -> list[dict]:
    token = os.environ["API_TOKEN"]  # секрет только из окружения
    headers = {"Authorization": f"Bearer {token}"}
    events: list[dict] = []
    cursor: str | None = None

    with httpx.Client(http2=True, timeout=30.0, headers=headers) as client:
        while True:
            params = {"limit": 1000}
            if cursor:
                params["cursor"] = cursor

            for attempt in range(6):
                resp = client.get(f"{base_url}/events", params=params)
                if resp.status_code == 429:
                    # уважаем сервер: Retry-After или экспонента с джиттером
                    wait = float(resp.headers.get("Retry-After", 2 ** attempt))
                    time.sleep(min(wait, 60) + random.uniform(0, 1))
                    continue
                resp.raise_for_status()
                break
            else:
                raise RuntimeError("rate limit не отпустил после 6 попыток")

            page = resp.json()
            events.extend(page["data"])
            cursor = page.get("next_cursor")
            if not cursor:
                break

    return events
```

class BetsSchema(pa.DataFrameModel):
    event_id: Series[str] = pa.Field(unique=True)
    user_id: Series[str] = pa.Field(nullable=False)
    amount: Series[float] = pa.Field(ge=0, le=1_000_000)
    currency: Series[str] = pa.Field(isin=["USD", "EUR", "RUB"])
    event_type: Series[str] = pa.Field(isin=["bet", "deposit", "withdrawal"])
    ts: Series[pl.Datetime] = pa.Field(nullable=False)

    class Config:
        strict = True  # лишние колонки запрещены

Сам пайплайн очистки: детерминированные шаги, валидация на выходе, сохранение в Parquet.

```python
import polars as pl

def clean_bets(raw: pl.DataFrame) -> pl.DataFrame:
    df = (
        raw
        # 1. типы: строки сумм с запятой-разделителем -> float
        .with_columns(
            pl.col("amount")
            .str.replace_all(" ", "")
            .str.replace(",", ".")
            .cast(pl.Float64, strict=False),
            pl.col("ts").str.to_datetime(strict=False),
            pl.col("currency").str.to_uppercase(),
        )
        # 2. полные дубли от ретраев загрузки
        .unique(subset=["event_id"], keep="first")
        # 3. согласованность: события из будущего = битые часы клиента
        .filter(pl.col("ts") <= pl.lit(pl.datetime(2100, 1, 1)))
        # 4. флаг пропуска суммы вместо молчаливого fillna(0)
        .with_columns(pl.col("amount").is_null().alias("amount_was_missing"))
        # 5. винзоризация хвоста, а не удаление: гасим артефакты, храним строку
        .with_columns(
            pl.col("amount").clip(
                upper_bound=pl.col("amount").quantile(0.999)
            )
        )
    )

    # контракт на выходе: если не прошли — падаем громко
    BetsSchema.validate(df, lazy=True)
    return df


raw = pl.read_parquet("bronze/bets_raw.parquet")
clean = clean_bets(raw)
clean.write_parquet("silver/bets_clean.parquet")  # Parquet, не CSV
```

def quality_report(df: pl.DataFrame) -> dict:
    n = df.height
    return {
        "rows": n,
        "completeness_amount": 1 - df["amount"].null_count() / n,
        "uniqueness_event_id": df["event_id"].n_unique() / n,

!!! tip "AI-копилот: где нейросети помогают"

    LLM (Claude, Cursor, Copilot) сильны на этой теме в трёх вещах: генерация boilerplate (схемы Pandera/Pydantic по примеру данных), разбор грязных строк (написать регулярку под «1 234,50 руб.»), и подсказка диагностики («какие проверки качества стоит добавить для лога транзакций»).

    Пример хорошего промпта: «Вот первые 20 строк сырого лога депозитов (CSV ниже). Опиши все обнаруженные аномалии по типам: пропуски, дубли, несогласованные типы, выбросы, нарушения согласованности. Для каждой предложи проверку на Pandera (polars-бэкенд) и объясни выбор стратегии. Не заполняй пропуски нулями. Колонки: user_id, amount, currency, ts.»

    Где врут и что обязательно перепроверять: модель уверенно предлагает `fillna(0)` и `dropna()` без вопроса о механизме пропуска — это всегда проверяйте сами через MCAR/MAR/MNAR. Она может выдумать несуществующие параметры функций (особенно для свежих версий Polars/Pandera 2026) — сверяйтесь с докой. Она не знает вашей доменной семантики (что для гемблинга `amount=0` у депозита невозможен, а у бонуса возможен) — бизнес-правила задаёте вы. И LLM не видит распределения данных целиком по 20 строкам, поэтому пороги выбросов на её слово не берите.

!!! note "Стек 2026"

    Актуально: Parquet/Arrow как дефолт хранения и обмена; Polars как быстрая альтернатива pandas на средних и больших данных (ленивые планы, многопоточность); httpx вместо requests для нового кода; Pandera и Pydantic v2 как контракты на данные; DuckDB для аналитики прямо по Parquet-файлам без БД; pandas 2.x с Arrow-бэкендом.

    Устарело относительно старых курсов: хранить промежуточные данные в CSV; `requests` как единственный выбор; чистка россыпью `assert` без единой схемы; `df.fillna(df.mean())` как универсальный рецепт; вера, что pandas — единственный DataFrame-инструмент. Excel как формат обмена в пайплайне был и остаётся источником боли.

!!! warning "Ловушки"

    Тихий джойн-промах: `user_id` строкой в одной таблице и числом в другой — джойн молча теряет строки без ошибки. Всегда приводите ключи к одному типу и проверяйте размер результата до и после.

    Пропуски, замаскированные под значения: `-1`, `0`, `999`, `"unknown"`, `1970-01-01` (нулевой Unix timestamp). Среднее по такой колонке врёт. Ищите подозрительные «круглые» и «предельные» значения в распределении.

    `fillna(0)` для денег: нулевой депозит — это реальное число, оно занизит средние и сломает LTV. Сеньоры на этом горят при спешке.

    Удаление выбросов перед тем, как понять их природу: выкинули «аномальные» крупные ставки — и заодно выкинули весь сегмент хайроллеров или всю мошенническую активность, которую и надо было найти.

    Утечка из будущего при чистке: импутация или нормализация по статистикам, посчитанным на всём датасете (включая тест), завышает метрики модели. Статистики чистки считаются только на train (см. модули по ML).

    Часовые пояса: смешали naive и aware datetime, или собрали данные из источников в разных TZ без приведения к UTC — получили события «из будущего» и кривой retention по дням.

    Float для денег в финансовых расчётах: `0.1 + 0.2 != 0.3`. Для точных сумм используйте Decimal или целые копейки/центы, особенно в quant/finance.
