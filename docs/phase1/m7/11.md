# M7.11 · Воспроизводимый пайплайн очистки

Разовая чистка в ноутбуке — это технический долг. Завтра придут новые данные, и всё надо повторять руками, забывая шаги. Профессиональный артефакт — пайплайн: набор чистых детерминированных функций, прогоняющих сырые данные через фиксированную последовательность шагов с валидацией на входе и выходе.

## Зачем это вообще нужно — на пальцах

Представь, что ты аналитик гемблинг-проекта. В понедельник ты выгрузил депозиты за неделю, в ноутбуке руками удалил тестовые аккаунты, привёл валюты к евро, выкинул отрицательные суммы и построил отчёт по FTD (first-time deposit). Менеджер доволен. В следующий понедельник приходит новая выгрузка — и ты должен вспомнить **все** шаги, в **том же порядке**, с **теми же порогами**. Через месяц ты уже не помнишь, почему отрезал депозиты больше \$50 000 и удалял ли ты дубликаты до или после конвертации валют. Отчёты «плывут», числа в соседних дашбордах не сходятся, и доверие к аналитике падает.

Пайплайн решает это так: однажды описанные шаги фиксируются в коде, и любой прогон — твой, коллеги или ночного планировщика — даёт идентичный результат. Ноутбук становится местом для исследования, а пайплайн — местом для продакшена.

## Принципы

Идемпотентность: повторный прогон на тех же данных даёт тот же результат (важно для дозагрузок). Детерминизм: никаких «почистил на глаз», только воспроизводимые правила. Никогда не мутировать сырьё: сырые данные неприкосновенны (raw zone), чистка пишет новый слой. Логирование: каждый шаг сообщает, сколько строк затронул. Валидация на границах: контракт на входе и на выходе. Версионирование: и код пайплайна, и сами данные (DVC, lakeFS или просто партиционирование по дате загрузки).

Разберём два принципа, которые чаще всего понимают неправильно.

**Идемпотентность ≠ детерминизм.** Детерминизм — про то, что функция от одного и того же входа даёт один и тот же выход. Идемпотентность — про то, что применить операцию один раз и применить её дважды подряд даёт один и тот же результат: $f(f(x)) = f(x)$. Классический пример неидемпотентного шага — `df = df.append(reference_rows)`: каждый прогон добавляет справочные строки заново, и после трёх дозагрузок они утроятся. Идемпотентная версия — сначала удалить старые справочные строки, потом вставить актуальные, либо использовать `merge`/`upsert` по ключу. Проверка простая: прогони пайплайн на одних и тех же данных дважды и сравни выход хешем — `hashlib.md5(df.to_csv().encode()).hexdigest()` должен совпасть.

**Не мутировать сырьё.** Если ты перезаписал исходный CSV «очищенной» версией, ты потерял возможность переочистить данные по новым правилам. Завтра выяснится, что тестовые аккаунты надо было оставлять (для отдельного отчёта по QA), а ты их уже стёр из единственной копии. Поэтому raw zone — read-only, чистка читает из неё и пишет в новый слой.

## Численный пример: контроль через row counts

Логирование числа затронутых строк — это не украшение, а ранний детектор поломок. Допустим, пайплайн депозитов обычно выглядит так:

```
load raw:            1 000 000 строк
drop test accounts:  -12 000  -> 988 000
drop negative amt:   -340     -> 987 660
dedup by tx_id:      -1 200   -> 986 460
convert currency:     0        -> 986 460  (строки не теряются)
WRITE silver:        986 460 строк
```

Если завтра шаг `drop test accounts` вдруг убрал не 12 000, а 300 000 строк — это сигнал, что в выгрузке сломался флаг `is_test` (например, стал `NULL` и условие отфильтровало всех). Без логов ты бы увидел только странно низкий FTD в отчёте через неделю. С логами — поймал в момент прогона. Хорошая практика: писать не только абсолютное число, но и долю отброшенного, и падать (или хотя бы громко предупреждать), если доля вышла за исторический диапазон.

## Скелет пайплайна на Python

```python
import logging

logging.basicConfig(level=logging.INFO)
log = logging.getLogger("cleaning")

def drop_test_accounts(df):
    before = len(df)
    out = df[df["is_test"] == False].copy()
    log.info("drop_test_accounts: %d -> %d (-%d)", before, len(out), before - len(out))
    return out

def drop_negative_amount(df):
    before = len(df)
    out = df[df["amount"] >= 0].copy()
    log.info("drop_negative_amount: -%d", before - len(out))
    return out

def dedup_by_tx(df):
    before = len(df)
    out = df.drop_duplicates(subset=["tx_id"]).copy()
    log.info("dedup_by_tx: -%d", before - len(out))
    return out

def clean(df_raw):
    # контракт на входе
    assert {"tx_id", "amount", "is_test"}.issubset(df_raw.columns), "missing required columns"
    df = (
        df_raw
        .pipe(drop_test_accounts)
        .pipe(drop_negative_amount)
        .pipe(dedup_by_tx)
    )
    # контракт на выходе
    assert df["tx_id"].is_unique, "tx_id not unique after dedup"
    assert (df["amount"] >= 0).all(), "negative amount leaked through"
    return df
```

Обрати внимание: сырьё `df_raw` не мутируется — каждый шаг делает `.copy()` и возвращает новый DataFrame. `.pipe()` выстраивает фиксированную последовательность, которую легко читать и тестировать по одной функции. Ассерты на границах — это и есть «контракт на входе и выходе»: пайплайн падает громко и сразу, а не выдаёт тихо испорченные данные дальше по конвейеру.

!!! warning "Типичные ошибки"

    Чистый код без логов — половина пайплайна: при поломке в проде ты не поймёшь, на каком шаге потерялись строки. Конвертация валют по «курсу на сегодня» вместо «курса на дату транзакции» ломает детерминизм — вчерашний прогон не воспроизведётся. Удаление дубликатов до конвертации валют может оставить две «разные» по сумме записи одной транзакции, если суммы в разных валютах, — порядок шагов важен. И самое опасное — `inplace=True` или перезапись raw-файла: это нарушает неприкосновенность сырья.

!!! note "Medallion-архитектура"

    Индустриальный паттерн организации слоёв — bronze/silver/gold. Bronze — сырьё как пришло, неизменное. Silver — очищенное, типизированное, провалидированное. Gold — агрегированные витрины под конкретные задачи (например, таблица retention по когортам). Ваш пайплайн очистки — это переход bronze -> silver.

```quiz

[ ] Пайплайн можно запускать только один раз
> Идемпотентность: $f(f(x)) = f(x)$, повторное применение ничего не меняет. Это критично для безопасных дозагрузок.

Шаг `df = df.append(reference_rows)` при каждом прогоне дозагрузки. В чём проблема?

[ ] Он нарушает неприкосновенность сырья

---
Что в medallion-архитектуре соответствует переходу bronze -> silver?
[ ] Построение агрегированной витрины retention по когортам
```
