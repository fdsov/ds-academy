## Что построишь

В модуле M7 был тезис: грязные данные почти никогда не падают с исключением — они дают правдоподобный, но неверный ответ. Этот воркшоп переводит тезис в навык. Ты возьмёшь сырой лог ставок и депозитов гемблинг-продукта, в котором сидят все классы загрязнений из теории (пропуски, полные дубли, выбросы, кривые типы, события из будущего, рассогласование регистра валют), и построишь из него профессиональный артефакт.

Что именно отрабатываем из M7:

- M7.6 — классификация загрязнений и диагностика профилированием.
- M7.7 — выбор стратегии для пропусков через механизм MCAR/MAR/MNAR, а не наугад. Принцип «никогда не `fillna(0)` для денег».
- M7.8 — поиск выбросов методом Тьюки (IQR) и винзоризация вместо удаления.
- M7.9 — контракт на данные через Pandera (polars-бэкенд), падение пайплайна на нарушении.
- M7.10 — шесть измерений качества как числа с SLA.
- M7.11 — идемпотентный детерминированный пайплайн bronze → silver, сырьё неизменно.

Артефакт на выходе: чистая функция `clean_bets()` плюс контракт `BetsSchema` плюс отчёт о качестве данных до/после. Всё запускается одной командой у любого, кто склонировал репозиторий.

!!! info "Как устроен этот воркшоп"

    Это не лекция с готовым кодом, а задачник. Каждый содержательный шаг построен по схеме **Задача → Критерий → Решение**:

    - **Задача** — что именно сделать руками, с явными входами и именами выходных переменных, которые проверяет критерий.
    - **Критерий шага** — кусок кода с `assert` (или чек-лист самопроверки), который ты запускаешь после своего решения. Зелёный прогон без ошибки = шаг сдан. Это твой локальный авто-грейдер.
    - **Решение** спрятано под спойлер `Решение`. Открывай только после своей попытки — чтобы сверить подход, а не списать.
    - **Числовые подзадачи** проверяются прямо на странице: посчитай число, впиши в поле, нажми «Проверить».

    Все `assert` рассчитаны на сгенерированные данные с `seed=42` (и `seed=7` для дублей). Не меняй seed, иначе числа поплывут.

## Бизнес-кейс

!!! example "Ситуация"

    Ты — аналитик в команде iGaming-продукта. К тебе приходит **CFO**: финансовая отчётность за полгода строится на ежедневной выгрузке лога ставок и депозитов от партнёра, и цифры перестали сходиться с эквайрингом. Дашборд показывает депозитный оборот за период около **312 млн ₽**, а банк подтверждает примерно **305 млн ₽** — расхождение порядка **7 млн ₽**, и никто не может объяснить, откуда оно.

    Ты заглядываешь в сырьё и видишь типичный «партнёрский CSV»: суммы строкой с запятой и пробелами, депозиты с пропущенной суммой, события, датированные 2099 годом, и дубли от ретраев загрузки. Плюс десяток транзакций по 10 000 000,00 — явно тестовые прогоны, которые раздувают оборот.

    - **Что зависит от ответа:** на этих цифрах CFO защищает квартальный P&L перед советом и считает маркетинговый бюджет на следующий квартал. Завышенный на 2-3% оборот = переоценённая база для плана закупки трафика и риск не выполнить заложенные цели.
    - **Цена ошибки:** один процент фантомного оборота на этих объёмах — около 3 млн ₽ некорректной строки в отчёте; тестовые транзакции и дубли вместе дают как раз порядок наблюдаемого расхождения.
    - **Ограничение:** доступа к исходной системе партнёра нет, чинить на их стороне нельзя — у тебя только ежедневный CSV «как пришёл». Нужен воспроизводимый пайплайн, который любой в команде запустит одной командой и получит тот же результат, до конца недели.

## Данные

Никаких внешних загрузок. Генерируем синтетический лог с фиксированным seed, чтобы у всех байт-в-байт совпадало, и тут же портим его всеми способами из M7.6. Это и есть твоя bronze-зона.

Это **setup-код, а не задача** — скопируй и запусти как есть. Он создаёт `bronze/bets_raw.csv` (сырьё как пришло от партнёра).

!!! warning "Зачем фиксировать seed"

    Без `seed` каждый запуск даст другие числа, и ты не сможешь сверить результат с воркшопом или воспроизвести баг. Все критерии-`assert` ниже рассчитаны на `seed=42` (данные) и `seed=7` (дубли).

```python
# gen_data.py
import polars as pl
import numpy as np
from pathlib import Path
from datetime import datetime

rng = np.random.default_rng(42)
N = 10_000
Path("bronze").mkdir(exist_ok=True)

user_ids = [f"u{rng.integers(1, 1500):05d}" for _ in range(N)]
amounts = np.round(rng.lognormal(mean=4.0, sigma=1.0, size=N), 2)  # логнормаль ~ как реальные суммы
currency = rng.choice(["USD", "usd", "EUR", "eur", "RUB", "rub"], size=N)
country = rng.choice(["US", "DE", "RU"], size=N)
etype = rng.choice(["bet", "deposit", "withdrawal"], size=N, p=[0.6, 0.25, 0.15])
# интервал подобран так, чтобы точек хватило на N строк (с запасом), затем срез [:N]
ts = pl.datetime_range(
    pl.datetime(2026, 1, 1), pl.datetime(2026, 6, 1),
    interval="21m", eager=True
)[:N]

df = pl.DataFrame({
    "event_id": [f"e{i:07d}" for i in range(N)],
    "user_id": user_ids,
    "amount": amounts,
    "currency": currency,
    "country": country,
    "event_type": etype,
    "ts": ts,
})

# --- порча по M7.6 ---
# 1. суммы как строки с запятой-разделителем и пробелами (формат партнёра)
df = df.with_columns(
    pl.col("amount").map_elements(lambda x: f"{x:,.2f}".replace(",", " ").replace(".", ","),
                                  return_dtype=pl.Utf8)
)
# 2. ~5% пропусков в amount — причём чаще у deposit (потом проверим механизм)
miss_mask = (rng.random(N) < np.where(df["event_type"] == "deposit", 0.15, 0.02))
df = df.with_columns(
    pl.when(pl.Series(miss_mask)).then(None).otherwise(pl.col("amount")).alias("amount")
)
# 3. десяток сумм-артефактов на миллион (тестовые транзакции)
art_idx = rng.choice(N, size=12, replace=False)
amt = df["amount"].to_list()
for i in art_idx:
    amt[i] = "10 000 000,00"
df = df.with_columns(pl.Series("amount", amt))
# 4. события из будущего — битые часы клиента
fut_idx = rng.choice(N, size=20, replace=False)
ts2 = df["ts"].to_list()
for i in fut_idx:
    ts2[i] = datetime(2099, 1, 1)
df = df.with_columns(pl.Series("ts", ts2).cast(pl.Datetime))
# ts в строку формата партнёра DD.MM.YYYY HH:MM
df = df.with_columns(pl.col("ts").dt.strftime("%d.%m.%Y %H:%M"))

# 5. 2% полных дублей от ретраев загрузки
dup = df.sample(fraction=0.02, seed=7)
df = pl.concat([df, dup])

df.write_csv("bronze/bets_raw.csv")
print("bronze rows:", df.height)  # 10200
```

Колонки сырья: `event_id`, `user_id`, `amount` (строка `"1 234,50"`, есть `null`), `currency` (разный регистр), `country`, `event_type`, `ts` (строка `DD.MM.YYYY HH:MM`, есть из будущего). Дубли и артефакты внутри.

!!! note "Почему именно polars + pandera"

    `polars` — быстрый колоночный DataFrame с ленивым API; `pandera` даёт декларативный контракт данных, который умеет работать на polars-бэкенде. Связка покрывает весь путь от сырого CSV до валидированного silver-Parquet без ухода в pandas там, где он не нужен.

## Ход работы

### Шаг 1: Профилирование сырья

**Зачем.** Прежде чем чистить, надо знать, с чем имеешь дело — это дисциплина недоверия из M7.1. Считаем шесть измерений качества как числа, чтобы потом сравнить до/после. Без точки отсчёта ты не докажешь CFO, что именно починил.

**Задача.** Напиши функцию `quality_report(df, amount_col="amount_num")`, возвращающую словарь метрик: `rows`, `completeness_amount` (доля заполненных `amount`), `uniqueness_event_id` (доля уникальных `event_id`), `validity_currency` (доля валют из справочника после `to_uppercase`), `future_ts` (число событий с `ts` позже 2026-12-31), `outliers_amount` (число сумм > 1 млн, если колонка уже числовая, иначе `None`). Прочитай `bronze/bets_raw.csv` в `raw` и положи отчёт в `before = quality_report(raw)`.

**Критерий шага:**

```python
assert before["rows"] == 10200, "bronze = 10000 + 200 дублей"
assert 0.94 <= before["completeness_amount"] <= 0.95
assert 0.97 <= before["uniqueness_event_id"] <= 0.99, "около 0.98 из-за 2% дублей"
assert before["validity_currency"] == 1.0, "после to_uppercase все валюты в справочнике"
assert before["future_ts"] == 20, "ровно 20 событий из будущего заложено"
print("OK: профиль сырья снят —", before)
```

??? success "Решение"

    ```python
    # pipeline.py
    import polars as pl

    def quality_report(df: pl.DataFrame, amount_col: str = "amount_num") -> dict:
        n = df.height
        has_num = amount_col in df.columns
        return {
            "rows": n,
            "completeness_amount": round(1 - df["amount"].null_count() / n, 4),
            "uniqueness_event_id": round(df["event_id"].n_unique() / n, 4),
            "validity_currency": round(
                df.filter(pl.col("currency").str.to_uppercase().is_in(["USD", "EUR", "RUB"])).height / n, 4
            ),
            "future_ts": df.filter(
                pl.col("ts").str.strptime(pl.Datetime, "%d.%m.%Y %H:%M", strict=False)
                > pl.datetime(2026, 12, 31)
            ).height if df["ts"].dtype == pl.Utf8 else
            df.filter(pl.col("ts") > pl.datetime(2026, 12, 31)).height,
            "outliers_amount": (
                df.filter(pl.col(amount_col) > 1_000_000).height if has_num else None
            ),
        }

    raw = pl.read_csv("bronze/bets_raw.csv")
    before = quality_report(raw)
    print("BEFORE:", before)
    ```

    **Почему так.** `completeness_amount` около 0.95, `uniqueness_event_id` около 0.98 (2% дублей), `validity_currency` = 1.0 **только** потому, что метрика сама применяет `to_uppercase` — это и есть сигнал, что регистр в сырье кривой. `future_ts` = 20. На сыром `amount` (ещё строка) `outliers_amount` вернёт `None`. Это твоя точка отсчёта.

Проверь понимание:

```text
Q: Почему uniqueness_event_id меньше 1.0 ещё до всякой чистки?
[ ] Часть amount пропущена, поэтому строки считаются неполными
[x] В сырьё добавлены 2% полных дублей от ретраев — n_unique/n < 1 ловит повтор event_id
[ ] event_id сгенерированы неуникально в генераторе
> Полнота тут ни при чём - строки есть, они просто дублируются. uniqueness ловит именно класс загрязнения "дубли" (M7.6).
---
Q: Поле currency заполнено в 100% строк. Почему это не делает его валидным?
[ ] Заполненность и валидность - это одно и то же
[x] Полнота и валидность - разные измерения: "usd" заполнено, но не входит в справочник ISO в верхнем регистре
[ ] currency валидно, метрика считает с ошибкой
> Значение есть (полнота ок), но формат нарушен (регистр). M7.10 разводит шесть измерений качества: заполненная ячейка может быть невалидной.
```

### Шаг 2: Диагностика механизма пропуска

**Зачем.** Главная ошибка новичка — `fillna(0)` или `dropna()` без вопроса «почему пропущено». Механизм MCAR/MAR/MNAR определяет, смещены ли выводы. Если пропуск зависит от типа события, удаление строк перекосит выборку.

**Задача.** Проверь гипотезу: зависит ли пропуск `amount` от `event_type`? Сгруппируй `raw` по `event_type`, посчитай долю пропусков (`amount.is_null().mean()`) в каждой группе, отсортируй по убыванию и положи результат в `miss_by_type`.

**Критерий шага:**

```python
mr = dict(zip(miss_by_type["event_type"].to_list(), miss_by_type["miss_rate"].to_list()))
assert mr["deposit"] > 0.10, "у депозитов пропусков заметно больше"
assert mr["bet"] < 0.05 and mr["withdrawal"] < 0.05, "у ставок/выводов мало"
assert mr["deposit"] > mr["bet"], "пропуск зависит от event_type — это MAR, не MCAR"
print("OK: пропуск зависит от event_type (MAR) —", mr)
```

??? success "Решение"

    ```python
    miss_by_type = (
        raw.with_columns(pl.col("amount").is_null().alias("is_miss"))
           .group_by("event_type")
           .agg(pl.col("is_miss").mean().alias("miss_rate"))
           .sort("miss_rate", descending=True)
    )
    print(miss_by_type)
    ```

    **Почему так.** У `deposit` доля пропусков ~0.15, у `bet`/`withdrawal` ~0.02. Пропуск зависит от наблюдаемой переменной `event_type`, но (по построению) не от самой скрытой суммы внутри группы — это **MAR**, не MNAR. Вывод для стратегии: listwise deletion выкинул бы непропорционально много депозитов и сместил выборку, поэтому строки НЕ удаляем. Берём решение из M7.7: флаг `amount_was_missing` плюс честное сохранение факта отсутствия, а не `fillna(0)`.

!!! tip "Различить MAR и MNAR кодом нельзя"

    MNAR был бы, если бы пропущены оказывались именно крупные суммы (хайроллеры прячут депозит) — тогда ни удаление, ни медиана не спасают. Здесь пропуск объясняется наблюдаемым `event_type`, значит MAR. Но различить MAR и MNAR можно только пониманием механизма генерации, не тестом на самих данных.

Проверь понимание:

```text
Q: Пропуск amount чаще у deposit, чем у bet. Какой это механизм и что из него следует?
[ ] MCAR — пропуск случаен, можно спокойно dropna()
[x] MAR — пропуск зависит от наблюдаемого event_type; удалять строки нельзя, иначе сместим долю депозитов
[ ] MNAR — пропущены именно крупные суммы, спасения нет
> Зависимость от наблюдаемой колонки = MAR. Listwise deletion выкинет непропорционально много депозитов и сместит выборку. Помечаем флагом, а не удаляем.
---
Q: Почему для денежной суммы опасен fillna(0)?
[ ] Ноль ломает типизацию колонки
[x] Ноль — это конкретное ложное значение: он занизит средний депозит и внесёт ложь в оборот, скрыв сам факт пропуска
[ ] fillna(0) всегда корректен для чисел
> 0 у денег - не "пусто", а "ноль рублей". Это искажает агрегаты и стирает сигнал о пропуске. Честно: флаг amount_was_missing + сохранение null.
```

### Шаг 3: Контракт Pandera на выходе

**Зачем.** Профессиональный сдвиг из M7.9 — не чистить руками каждый раз, а декларативно описать, какими данные должны быть. Контракт собирает все ожидания в одном месте; нарушение ловится на входе валидации, а не всплывает в отчёте через неделю.

**Задача.** Опиши класс-контракт `BetsSchema` (Pandera, polars-бэкенд) для silver-таблицы: `event_id` (str, unique), `user_id` (str, not null), `amount` (float, `ge=0`, `le=1_000_000`, nullable), `amount_was_missing` (bool), `currency` (str, isin `["USD","EUR","RUB"]`), `country` (str, isin `["US","DE","RU"]`), `event_type` (str, isin `["bet","deposit","withdrawal"]`), `ts` (Datetime, not null). Включи `strict = True` (лишние колонки запрещены). Сам `validate` ты вызовешь в шаге 4.

**Критерий шага (self-check).** Кода с числами тут нет — проверь декларацию по чек-листу:

- [ ] `event_id` помечен `unique=True` (дубли обязаны быть сбиты до валидации);
- [ ] у `amount` стоит `le=1_000_000` — значит артефакты на 10 млн должны быть погашены ДО `validate`;
- [ ] `amount` остаётся `nullable=True` — мы сознательно не заполняем пропуски (MAR из шага 2);
- [ ] `currency`/`country`/`event_type` ограничены справочниками через `isin`;
- [ ] `Config.strict = True` — схема упадёт на лишней колонке.

??? success "Решение"

    ```python
    import pandera.polars as pa
    from pandera.typing.polars import Series

    class BetsSchema(pa.DataFrameModel):
        event_id: Series[str] = pa.Field(unique=True)
        user_id: Series[str] = pa.Field(nullable=False)
        amount: Series[float] = pa.Field(ge=0, le=1_000_000, nullable=True)
        amount_was_missing: Series[bool] = pa.Field()
        currency: Series[str] = pa.Field(isin=["USD", "EUR", "RUB"])
        country: Series[str] = pa.Field(isin=["US", "DE", "RU"])
        event_type: Series[str] = pa.Field(isin=["bet", "deposit", "withdrawal"])
        ts: Series[pl.Datetime] = pa.Field(nullable=False)

        class Config:
            strict = True  # лишние колонки запрещены
    ```

    **Почему так.** Это формальное описание silver-таблицы: `amount` неотрицательна и не больше миллиона (артефакты на 10 млн обязаны быть погашены до валидации), валюта и страна из справочников, `event_id` уникален. Если `clean_bets` не дочистит — `validate` упадёт громко, и битый файл не попадёт в отчёт. Контракт — это исполняемая спецификация, а не комментарий.

### Шаг 4: Чистая функция clean_bets

**Зачем.** Собираем детерминированный идемпотентный пайплайн bronze → silver. Каждый шаг — воспроизводимое правило, сырьё не мутируем, на выходе валидация. Ключевое решение M7.8: выброс-артефакт **винзоризуем** (гасим влияние), а доказанный артефакт (будущая дата) **удаляем**.

**Задача.** Напиши `clean_bets(raw)`, которая возвращает новый DataFrame и: (1) парсит `amount` `"1 234,50" → 1234.50`, приводит `currency` к верхнему регистру, парсит `ts`; (2) бьёт полные дубли через `unique(subset=["event_id"], keep="first")`; (3) удаляет события из будущего (`ts > 2026-12-31`); (4) добавляет флаг `amount_was_missing`; (5) винзоризует `amount` по верхней границе Тьюки `Q3 + 1.5*IQR` (`clip`, не удаление); (6) в конце вызывает `BetsSchema.validate(df, lazy=True)`. Положи результат в `clean = clean_bets(raw)`.

**Критерий шага:**

```python

assert clean["event_id"].n_unique() == clean.height, "event_id уникальны"
assert "amount_was_missing" in clean.columns
assert clean.filter(pl.col("amount") > 1_000_000).height == 0, "артефакты погашены винзоризацией"
assert 220 <= clean["amount"].max() <= 230, "верхний ус Тьюки ~225"
assert raw["amount"].dtype == pl.Utf8, "сырьё не мутировано — amount всё ещё строка"
assert clean_bets(raw).height == clean.height, "идемпотентность: повторный прогон даёт ту же высоту"
print("OK: clean_bets отработала, строк:", clean.height)
```

??? tip "Подсказка"

    Границу Тьюки считай ПОСЛЕ парсинга `amount` в float, по непустым значениям: `q1 = df["amount"].quantile(0.25)`, `q3 = df["amount"].quantile(0.75)`, `upper = q3 + 1.5*(q3 - q1)`. Клипуй только непустые: `pl.when(col.is_not_null()).then(col.clip(upper_bound=upper)).otherwise(None)`.

??? success "Решение"

    ```python
    def clean_bets(raw: pl.DataFrame) -> pl.DataFrame:
        before = raw.height
        df = (
            raw
            # 1. типы: "1 234,50" -> 1234.50, нормализация валюты, парс даты
            .with_columns(
                pl.col("amount").str.replace_all(" ", "").str.replace(",", ".")
                  .cast(pl.Float64, strict=False).alias("amount"),
                pl.col("currency").str.to_uppercase(),
                pl.col("ts").str.strptime(pl.Datetime, "%d.%m.%Y %H:%M", strict=False),
            )
            # 2. полные дубли от ретраев -> по event_id, keep first
            .unique(subset=["event_id"], keep="first")
            # 3. согласованность: события из будущего = битые часы клиента
            .filter(pl.col("ts") <= pl.datetime(2026, 12, 31))
            # 4. флаг пропуска вместо молчаливого fillna(0) (MAR, M7.7)
            .with_columns(pl.col("amount").is_null().alias("amount_was_missing"))
        )
        # 5. винзоризация по верхней границе Тьюки, а не удаление (M7.8)
        q1 = df["amount"].quantile(0.25)
        q3 = df["amount"].quantile(0.75)
        upper = q3 + 1.5 * (q3 - q1)
        df = df.with_columns(
            pl.when(pl.col("amount").is_not_null())
              .then(pl.col("amount").clip(upper_bound=upper))
              .otherwise(None).alias("amount")
        )
        print(f"[clean] {before} -> {df.height} строк (верхний ус Тьюки: {upper:.2f})")

        BetsSchema.validate(df, lazy=True)  # контракт на выходе
        return df
    ```

    **Почему так.** Функция логирует переход `10200 -> 9980` (200 дублей + 20 будущих дат), приводит типы, помечает пропуски флагом и винзоризует хвост по границе Тьюки ~225. Артефакты на 10 млн прижаты к верхнему усу, поэтому проходят `le=1_000_000` в контракте. Сырьё `raw` не изменено — `clean_bets` возвращает новый DataFrame, и потому повторный прогон даёт байт-в-байт тот же результат (идемпотентность).

## Типичные ошибки

- **`fillna(0)` для денег.** Ноль у депозита — это не «пусто», а «ноль рублей». Занижает оборот и стирает сигнал о пропуске. Диагностируй механизм (MCAR/MAR/MNAR), помечай флагом.
- **`dropna()` при MAR.** Удаление строк с пропуском перекашивает выборку (выкинет непропорционально много депозитов). Удалять можно только при MCAR и осознанно.
- **Удаление выбросов вместо винзоризации.** Крупная сумма может быть реальным хайроллером. Молча удалять нельзя — гаси влияние `clip` по границе Тьюки, а строку сохраняй (M7.8).
- **Путаница полноты и валидности.** Заполненная ячейка `"usd"` невалидна (нет в справочнике). Это разные измерения качества (M7.10).
- **Мутация сырья.** Если чистить `raw` на месте, теряешь bronze-зону и идемпотентность. Сырьё неизменно, `clean_bets` возвращает новый DataFrame (M7.11).
- **Чистка без контракта.** Без `BetsSchema.validate` битый файл от партнёра тихо протечёт в отчёт. Контракт ловит нарушение на входе, а не через неделю в P&L (M7.9).
- **Одна финальная цифра вместо до/после.** Без дельты не докажешь, что починил, и не заметишь регрессию (искусственно подтянутую полноту).

## Развитие

- **Честная импутация по группе.** Замени простой флаг на импутацию `amount` медианой внутри `event_type` (групповая медиана в Polars или sklearn `IterativeImputer`), сохранив флаг. Сравни распределение `amount` до/после — заметь, как импутация занижает дисперсию (M7.7).
- **Контракт на входе через Pydantic.** Распарси каждую сырую запись как модель на границе и отсей структурно битые строки до DataFrame-чистки (M7.9). Сравни, что ловит Pydantic построчно, а что Pandera таблично.
- **Межколоночное правило.** Реализуй проверку согласованности `country`/`currency` как `@pa.dataframe_check`: запрети `country=US` при `currency=RUB` и измерь долю нарушений как седьмую кастомную метрику качества.
- **Medallion до gold.** Вынеси `gen_data` в bronze неизменным сырьём, `clean_bets` в silver, добавь gold-шаг — агрегат `deposit`-сумм по дням, читающий silver-Parquet через DuckDB прямо по файлу без загрузки в память (M7.11, M7.12).
