# M4.4 · SELECT, WHERE, ORDER BY, LIMIT — чтение данных

Интуиция. SELECT — это "покажи мне". WHERE — "только те строки, которые подходят под условие". ORDER BY — "отсортируй". LIMIT — "и хватит первых N штук".

На пальцах представь стопку карточек, где каждая карточка — это депозит игрока. `FROM` — это сама стопка. `WHERE` — ты быстро перебираешь карточки и откладываешь в сторону те, что не подходят (например, мелкие депозиты). `ORDER BY` — оставшиеся раскладываешь в нужном порядке. `LIMIT` — берёшь сверху ровно N карточек, остальное не трогаешь. Никакой магии: четыре простых действия над набором строк.

Формально. Базовый запрос на выборку:

```sql
SELECT player_id, amount, deposit_ts
FROM deposits
WHERE amount >= 100
ORDER BY deposit_ts DESC
LIMIT 10;
```

Это значит: из таблицы `deposits` взять столбцы `player_id, amount, deposit_ts`, оставить только строки где сумма не меньше 100, отсортировать по времени от новых к старым (`DESC` — descending), показать первые 10.

## Порядок выполнения — главное, что отличает понимающего

Критично понимать порядок выполнения, потому что он не совпадает с порядком написания. SQL читается сверху вниз, но выполняется так:

1. `FROM` — берём источник
2. `WHERE` — фильтруем строки
3. `GROUP BY` — группируем
4. `HAVING` — фильтруем группы
5. `SELECT` — вычисляем столбцы и алиасы
6. `ORDER BY` — сортируем
7. `LIMIT` — отрезаем

Именно поэтому в `WHERE` нельзя сослаться на алиас из `SELECT` — на момент фильтрации строк его ещё не существует. Это первая вещь, которая отличает понимающего от зубрилы.

Из этого порядка вытекает несколько практических следствий, которые на собеседованиях спрашивают чаще всего:

- В `ORDER BY` алиас из `SELECT` использовать **можно** — потому что сортировка идёт уже после `SELECT`. То есть `SELECT amount * 0.95 AS net ... ORDER BY net DESC` — валидно, а `WHERE net > 100` — нет.
- `LIMIT` применяется в самом конце. Поэтому `LIMIT 10` без `ORDER BY` возвращает "какие-то" 10 строк — порядок не гарантирован движком и может меняться от запуска к запуску. Для отчёта это ловушка: цифры "плавают" без видимой причины.
- `WHERE` работает с физическими строками, `HAVING` — с уже посчитанными агрегатами групп. Их нельзя путать (детальнее — в шаге про `GROUP BY`).

## Разбор на конкретных данных

Пусть в `deposits` лежит вот такая стопка:

| player_id | amount | deposit_ts          |
|-----------|--------|---------------------|
| 7         | 50     | 2026-06-15 10:00    |
| 7         | 250    | 2026-06-16 12:30    |
| 12        | 100    | 2026-06-14 09:15    |
| 12        | 80     | 2026-06-16 20:00    |
| 31        | 500    | 2026-06-16 08:45    |

Прогоняем запрос по шагам:

1. `FROM deposits` — на входе 5 строк.
2. `WHERE amount >= 100` — выкидываем строки с `amount` 50 и 80. Осталось 3 строки: (7, 250), (12, 100), (31, 500).
3. `SELECT player_id, amount, deposit_ts` — оставляем нужные столбцы.
4. `ORDER BY deposit_ts DESC` — сортируем по времени, новые сверху: (7, 250, 06-16 12:30), (31, 500, 06-16 08:45), (12, 100, 06-14 09:15).
5. `LIMIT 10` — строк всего 3, это меньше 10, поэтому отдаём все три.

Обрати внимание: депозит на 500 (самый крупный) оказался **вторым**, а не первым, потому что сортируем по времени, а не по сумме. Перепутать "сортировку по дате" с "топ по сумме" — классическая ошибка в продуктовых отчётах: попросили "последние крупные депозиты", а отдали "самые большие".

## Где это в гемблинг-аналитике

Эти четыре оператора — рабочая лошадка ежедневных проверок. Несколько типовых задач:

```sql
-- Топ-20 самых крупных депозитов за сегодня (здесь сортируем уже по сумме)
SELECT player_id, amount, deposit_ts
FROM deposits
WHERE deposit_ts >= '2026-06-17'
ORDER BY amount DESC
LIMIT 20;
```

```sql
-- Последние действия конкретного игрока при разборе жалобы или подозрении на фрод
SELECT amount, deposit_ts
FROM deposits
WHERE player_id = 12
ORDER BY deposit_ts DESC
LIMIT 50;
```

В деньгах логика такая же: `WHERE amount >= 100` отсекает мелкие пополнения, чтобы аналитик смотрел только на платежи, влияющие на GGR. А `LIMIT 20 ... ORDER BY amount DESC` — это буквально "покажи 20 китов за период", с которых начинается разбор аномального дня.

## Пограничные случаи и типичные ошибки

- **NULL в `WHERE`.** Условие `amount >= 100` отбрасывает строки, где `amount IS NULL`, — NULL не равен и не больше ничего, сравнение даёт "неизвестно", а не "истина". Если в данных есть незаполненные суммы, их легко молча потерять. Для NULL используют `IS NULL` / `IS NOT NULL`, а не `= NULL`.
- **`LIMIT` без `ORDER BY`.** Как сказано выше — недетерминированный результат. В отчёте, который смотрит начальство, это означает "цифры меняются при каждом обновлении".
- **`ORDER BY` по тексту вместо числа.** Если время или сумма хранятся как строка, сортировка пойдёт лексикографически ('100' < '20'), и топ будет неверным. Проверяй тип столбца.
- **Алиас в `WHERE`.** `SELECT amount*0.95 AS net FROM deposits WHERE net > 100` — упадёт с ошибкой "column net does not exist". Нужно либо повторить выражение в `WHERE` (`WHERE amount*0.95 > 100`), либо обернуть в подзапрос.
- **`LIMIT` для оценки объёма данных.** Когда таблица огромная, привычка ставить `LIMIT 100` при разведке экономит время и нагрузку на базу — сначала посмотри 100 строк, потом считай всё.

!!! question "Проверь себя"

    1. В каком порядке выполняются `FROM`, `WHERE`, `SELECT`, `ORDER BY`, `LIMIT`?
    2. Почему в `WHERE` нельзя сослаться на алиас, заданный в `SELECT`?
    3. Что делает `DESC` в `ORDER BY` и для чего нужен `LIMIT`?

??? success "Ответы"

    1. `FROM` → `WHERE` → `GROUP BY` → `HAVING` → `SELECT` → `ORDER BY` → `LIMIT`.
    2. На момент фильтрации строк в `WHERE` шаг `SELECT` ещё не выполнялся, поэтому алиаса физически не существует.
    3. `DESC` сортирует по убыванию (от больших к меньшим); `LIMIT N` оставляет только первые N строк результата.

!!! example "Мини-задание"

    Возьми запрос из примера выше. Не запуская его, выпиши на бумаге порядок шагов выполнения для этого конкретного запроса и отметь, на каком шаге отсекаются строки с `amount < 100`, а на каком остаются только 10 строк. Затем проверь себя по списку порядка выполнения.
