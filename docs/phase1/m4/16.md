# M4.16 · Практика

### Быстрый квиз

Выбери вариант — ответ подсветится сразу.

```quiz
В каком порядке SQL реально выполняет запрос?
[ ] SELECT → FROM → WHERE → GROUP BY → ORDER BY
[x] FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
[ ] WHERE → FROM → SELECT → HAVING → ORDER BY
> Источник берётся в FROM, строки фильтруются в WHERE, и только потом вычисляется SELECT — поэтому на алиас из SELECT в WHERE сослаться нельзя.
---
Чем отличается LEFT JOIN от INNER JOIN на примере "страны без депозитов"?
[ ] Разницы нет, оба вернут одинаковый результат
[x] LEFT сохранит все страны игроков (revenue 0), INNER уберёт страны без депозитов
[ ] INNER сохранит страны без депозитов, а LEFT их уберёт
> INNER оставляет только совпадения в обеих таблицах, поэтому страна без депозитов исчезнет; LEFT сохранит её, подставив NULL/0.
---
Чему равно AVG по значениям [100, NULL, 200]?
[ ] 100
[x] 150
[ ] 300
> AVG игнорирует NULL: делит сумму 300 на 2 непустых значения, а не на 3.
---
Почему оконная функция отличается от обычного GROUP BY?
[x] Она считает агрегат по окну, но не схлопывает строки, а дописывает результат столбцом
[ ] Она всегда работает быстрее любого GROUP BY
[ ] Она схлопывает строки сильнее, чем GROUP BY
> GROUP BY сворачивает группу в одну строку; оконная функция через OVER (...) сохраняет все строки и добавляет агрегат рядом.
---
Почему WHERE player_id NOT IN (SELECT player_id FROM blacklist) может вернуть ноль строк?
[ ] NOT IN не поддерживается в PostgreSQL
[x] Если в blacklist есть NULL, условие x <> NULL обнуляет всё выражение
[ ] Подзапросы запрещены внутри WHERE
> NOT IN разворачивается в цепочку x<>... AND x<>NULL; последнее даёт NULL и губит условие. Лечится через NOT EXISTS или фильтрацию NULL.
---
Что в выводе EXPLAIN на большой таблице с точечным фильтром является красным флагом?
[ ] Index Scan
[x] Seq Scan
[ ] Hash Join
> Seq Scan читает всю таблицу подряд; при фильтре по одному значению на больших данных это обычно признак отсутствия подходящего индекса.
---
Откуда берётся раздувание строк при JOIN и как его избежать?
[ ] Из-за сортировки ORDER BY, помогает LIMIT
[x] Из перемножения связей многие-ко-многим; агрегировать каждую таблицу до соединения в CTE
[ ] Из-за COALESCE; нужно убрать его из запроса
> 5 депозитов × 10 ставок дают 50 строк, и SUM считает депозит по 10 раз. Решение — свернуть таблицы по player_id до JOIN.
```

Разминка.

1. Выбери топ-20 депозитов за последние 7 дней, отсортированных по убыванию суммы.
2. Посчитай число игроков и средний чек депозита по каждой стране.
3. Найди игроков, у которых суммарный депозит превышает средний по всей базе (подзапрос).

Основное.

4. Построй DAU по дням за месяц, считая активным игрока, сделавшего хотя бы одну ставку.
5. Через `ROW_NUMBER()` найди для каждого игрока его первый депозит (FTD) — сумму и дату.
6. Посчитай GGR (`SUM(stake - payout)`) по дням и нарастающий итог GGR через оконную `SUM(...) OVER (ORDER BY day)`.
7. Сделай LEFT JOIN `players` и `deposits` так, чтобы увидеть страны без единого депозита (revenue = 0 через COALESCE).

Челлендж.

8. Построй таблицу D1/D7/D30 retention по когортам недели регистрации. Активность = ставка.
9. Через `LAG()` посчитай среднее время между первым и вторым депозитом игрока по странам.
10. Возьми медленный запрос из задания 8, прогони `EXPLAIN ANALYZE`, найди Seq Scan, добавь индекс и покажи разницу во времени.

Мини-проект. Собери на DuckDB cohort-retention-отчёт: входные данные — два Parquet-файла (`players`, `bets`), выход — pandas DataFrame с колонками cohort_week, cohort_size, retention_d1, retention_d7, retention_d30. Весь расчёт — на чистом SQL внутри DuckDB, питон только запускает запрос и отдаёт результат. Построй тепловую карту retention по когортам (cohort_week по строкам, лаг по столбцам).
