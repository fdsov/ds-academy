# M4.11 · Логика NULL и её ловушки

NULL — это не ноль и не пустая строка. Это "значение неизвестно". Из этого следуют три ловушки, на которых горят даже опытные.

## Интуиция на пальцах: трёхзначная логика

В обычном программировании логика двузначная: условие либо TRUE, либо FALSE. В SQL логика **трёхзначная**: TRUE, FALSE и UNKNOWN (его обычно и называют NULL). Любая операция сравнения, где участвует NULL, возвращает не TRUE и не FALSE, а UNKNOWN — потому что "сравнить с неизвестным" нельзя.

Представь, что в анкете игрока поле "страна" пусто. Вопрос "его страна равна UZ?" — честный ответ не "да" и не "нет", а "я не знаю". Именно это и есть UNKNOWN. И вот ключевое правило, из которого растут все три ловушки:

> `WHERE` пропускает строку, только если условие дало именно **TRUE**. Строки с UNKNOWN и FALSE отбрасываются одинаково — обе "не прошли".

Таблицы истинности, которые стоит держать в голове (`U` = UNKNOWN):

```text
AND    | T   F   U          OR     | T   F   U
-------+----------          -------+----------
  T    | T   F   U            T    | T   T   T
  F    | F   F   F            F    | T   F   U
  U    | U   F   U            U    | T   U   U

NOT T = F     NOT F = T     NOT U = U
```

Обрати внимание на две строки: `F AND U = F` (FALSE "побеждает" в AND), но `T AND U = U` и `T OR U = T` (TRUE "побеждает" в OR). Эти правила объясняют, почему один NULL в `NOT IN` рушит весь запрос, а в обычном `OR` — нет.

## Ловушка 1 — сравнение

`NULL = NULL` даёт не TRUE, а NULL (то есть "неизвестно"). Любое сравнение с NULL через `=` проваливается. Проверять надо через `IS NULL` / `IS NOT NULL`, никогда через `= NULL`.

То же касается `<`, `>`, `<>`, `LIKE` и даже `NULL = NULL`: все они дают UNKNOWN. `IS NULL` и `IS NOT NULL` — единственные операторы, которые гарантированно возвращают TRUE/FALSE и никогда UNKNOWN.

```sql
-- Неверно: вернёт 0 строк, потому что условие всегда UNKNOWN
SELECT * FROM players WHERE country = NULL;

-- Верно
SELECT * FROM players WHERE country IS NULL;
```

Типичная ошибка джуна: написать `WHERE deposit != NULL`, не дождаться ни одной строки и решить, что "NULL-ов в таблице нет". На самом деле условие просто никогда не бывает TRUE.

## Ловушка 2 — NOT IN с NULL

Запрос `WHERE player_id NOT IN (SELECT player_id FROM blacklist)` вернёт ноль строк, если в `blacklist` есть хоть один NULL. Потому что `x NOT IN (1, 2, NULL)` логически разворачивается в `x<>1 AND x<>2 AND x<>NULL`, а последнее — NULL, что губит всё условие. Решение: `NOT EXISTS` или фильтрация NULL в подзапросе.

Разберём по таблице истинности на конкретном `player_id = 7`, которого в чёрном списке нет:

```text
7 NOT IN (1, 2, NULL)
= 7<>1 AND 7<>2 AND 7<>NULL
=   T   AND   T   AND   U
=   U                        → строка отброшена
```

Даже игрок, которого заведомо нет в blacklist, не проходит фильтр — из-за `T AND U = U`. И так для каждой строки, поэтому результат — пустой. Коварство в том, что без NULL-а запрос работал месяцами, а потом кто-то вставил в blacklist строку с пустым `player_id`, и витрина "легальных игроков" внезапно опустела. Метрика DAU обнулилась не из-за бизнеса, а из-за одного NULL.

Три рабочих лечения:

```sql
-- Вариант 1: NOT EXISTS — не страдает от NULL, читается яснее
SELECT p.*
FROM players p
WHERE NOT EXISTS (
    SELECT 1 FROM blacklist b WHERE b.player_id = p.player_id
);

-- Вариант 2: отфильтровать NULL прямо в подзапросе
SELECT * FROM players
WHERE player_id NOT IN (
    SELECT player_id FROM blacklist WHERE player_id IS NOT NULL
);
```

Контраст: `IN` (без `NOT`) с NULL менее опасен — `7 IN (1, 2, NULL)` даёт `F OR F OR U = U`, тоже не вернёт лишнего, но и не обнулит всё разом. Опасен именно `NOT IN`, потому что там UNKNOWN протаскивается через `AND`.

## Ловушка 3 — агрегаты и фильтры пропускают NULL

`COUNT(column)` не считает NULL. `SUM`, `AVG` игнорируют их. А условие `WHERE country <> 'UZ'` НЕ вернёт строки где `country IS NULL` — хотя интуитивно "не UZ" должно их включать. Для NULL "не равно UZ" — это "неизвестно", а не TRUE.

Здесь скрыта частая ошибка в аналитике. Разберём на депозитах:

```text
player_id | deposit
--------- | -------
    1     |  100
    2     |  NULL
    3     |   50

COUNT(*)         = 3      -- считает все строки
COUNT(deposit)   = 2      -- NULL пропущен
SUM(deposit)     = 150    -- NULL пропущен
AVG(deposit)     = 75     -- = 150 / 2, а НЕ 150 / 3 = 50
```

Главная западня — `AVG`. Он делит сумму на число **не-NULL** значений. Если ты считаешь "средний депозит на игрока", а часть игроков ещё не вносила депозит (NULL), то `AVG` посчитает средний депозит только среди плативших и завысит метрику. Когда бизнесу нужен средний по всем игрокам — заменяй NULL на 0 заранее: `AVG(COALESCE(deposit, 0))` даст 50, а не 75. Это разные метрики, и выбор зависит от вопроса.

Та же логика с фильтрами по странам. Если в выгрузке "все игроки кроме Узбекистана" ты пишешь `WHERE country <> 'UZ'`, то игроки с неизвестной страной молча выпадут из отчёта — и сумма по сегментам не сойдётся с тотал:

```sql
-- Потеряет строки с country IS NULL
WHERE country <> 'UZ'

-- Включит и неизвестную страну
WHERE country <> 'UZ' OR country IS NULL

-- Эквивалентно и короче
WHERE COALESCE(country, 'unknown') <> 'UZ'
```

```sql
-- Безопасная замена NULL на значение по умолчанию
SELECT player_id, COALESCE(country, 'unknown') AS country
FROM players;
```

`COALESCE(a, b, c)` возвращает первый не-NULL аргумент. Это твой главный инструмент защиты от NULL в выводе.

Полезные родственники `COALESCE`:

- `NULLIF(a, b)` — возвращает NULL, если `a = b`, иначе `a`. Классика — защита от деления на ноль: `revenue / NULLIF(bets, 0)` вместо ошибки вернёт NULL, когда ставок не было.
- `COALESCE` принимает много аргументов и берёт первый не-NULL: `COALESCE(phone, email, 'нет контакта')`.

Ещё одна тонкость, о которой забывают: NULL ломает арифметику и конкатенацию. `100 + NULL = NULL`, `'A' || NULL = NULL`. Поэтому в вычисляемых полях оборачивай потенциально пустые столбцы в `COALESCE` до арифметики, а не после.

!!! warning "Где это бьёт в проде"

    Самые дорогие баги с NULL — тихие: запрос не падает, а возвращает технически валидный, но смыслово неверный результат. Пустая витрина из-за `NOT IN`, завышенный `AVG`, сегменты, не сходящиеся с тоталом. Поэтому при любой агрегации спрашивай себя: "а есть ли в этом столбце NULL, и что я хочу, чтобы с ними произошло?"

!!! question "Проверь себя"

    1. Почему `NULL = NULL` не даёт TRUE и как тогда проверять на NULL?
    2. Почему `WHERE player_id NOT IN (SELECT player_id FROM blacklist)` может вернуть ноль строк, и чем это лечится?
    3. Вернёт ли `WHERE country <> 'UZ'` строки, где `country IS NULL`, и почему?
    4. Что делает `COALESCE(country, 'unknown')`?

??? success "Ответы"

    1. Для SQL NULL — "неизвестно", и сравнение через `=` даёт NULL, а не TRUE. Проверять надо через `IS NULL` / `IS NOT NULL`.
    2. Если в `blacklist` есть NULL, условие `x <> NULL` внутри `NOT IN` становится NULL и обнуляет всё. Лечится через `NOT EXISTS` или фильтрацию NULL в подзапросе.
    3. Нет: для NULL "не равно UZ" — это "неизвестно", а не TRUE, поэтому такие строки выпадут. Нужно добавить `OR country IS NULL`.
    4. Возвращает первый не-NULL аргумент — здесь подставляет `'unknown'` вместо отсутствующей страны.

### Быстрый квиз

Выбери вариант — ответ подсветится сразу.

```quiz
Чему равно выражение WHERE country = NULL для строки, где country IS NULL?
[ ] TRUE — строка попадёт в результат
[x] UNKNOWN — строка не попадёт в результат
[ ] FALSE — но строка всё равно попадёт
> Сравнение через = с NULL всегда даёт UNKNOWN, а WHERE пропускает только TRUE. Нужен IS NULL.
---
Почему player_id NOT IN (SELECT player_id FROM blacklist) вернёт ноль строк при наличии NULL в blacklist?
[ ] Потому что NOT IN всегда медленнее IN и таймаутится
[x] Потому что x <> NULL даёт UNKNOWN, и через AND это обнуляет всё условие
[ ] Потому что NULL считается равным любому player_id
> NOT IN разворачивается в цепочку x<>v через AND; один NULL делает всё выражение UNKNOWN. Лечится NOT EXISTS.
---
В таблице депозитов значения 100, NULL, 50. Чему равен AVG(deposit)?
[ ] 50 — сумма 150 делится на 3 строки
[x] 75 — сумма 150 делится на 2 не-NULL значения
[ ] NULL — потому что один из элементов NULL
> AVG игнорирует NULL и делит на число не-NULL значений: 150 / 2 = 75. Чтобы делить на 3, нужен COALESCE(deposit, 0).
---
Как корректно посчитать revenue / bets, если bets может быть нулём?
[ ] revenue / COALESCE(bets, 0)
[x] revenue / NULLIF(bets, 0)
[ ] revenue / IS NULL(bets)
> NULLIF(bets, 0) превращает ноль в NULL, и деление вернёт NULL вместо ошибки деления на ноль.
---
Какое условие включит в выборку игроков с любой страной, кроме UZ, в том числе с неизвестной страной?
[ ] WHERE country <> 'UZ'
[x] WHERE country <> 'UZ' OR country IS NULL
[ ] WHERE country != 'UZ' AND country IS NULL
> Для NULL условие country <> 'UZ' даёт UNKNOWN и строка выпадает; нужно явно добавить OR country IS NULL.
```
