# M4.5 · Агрегаты, GROUP BY, HAVING — свёртка данных

Интуиция. Агрегат — это когда из множества строк получается одно число. "Сколько всего депозитов" — COUNT. "На какую сумму" — SUM. GROUP BY — "посчитай это отдельно для каждой страны / каждого дня / каждого игрока".

Представь стопку из 10 000 чеков на столе. Без агрегата ты просто смотришь на каждый чек по очереди. Агрегат — это когда ты сворачиваешь всю стопку в один итог ("общая выручка — 4,2 млн рублей"). GROUP BY — это когда ты сначала раскладываешь чеки по кучкам (отдельно Россия, отдельно Казахстан, отдельно Узбекистан) и сворачиваешь каждую кучку в свой итог. На выходе строк ровно столько, сколько кучек, — одна строка на группу.

Формально. Пять базовых агрегатных функций: `COUNT` (количество), `SUM` (сумма), `AVG` (среднее), `MIN` / `MAX` (минимум / максимум). С `GROUP BY` они считаются по группам:

```sql
SELECT country,
       COUNT(*)              AS deposits_cnt,
       COUNT(DISTINCT player_id) AS depositors,
       SUM(amount)           AS total_amount,
       AVG(amount)           AS avg_check
FROM deposits
GROUP BY country
HAVING SUM(amount) > 10000
ORDER BY total_amount DESC;
```

## Разбор на реальных данных

Возьмём маленькую таблицу `deposits` и руками посчитаем то, что выдаст запрос. Так видно, откуда берётся каждое число.

| player_id | country | amount |
|-----------|---------|--------|
| 1 | RU | 100 |
| 1 | RU | 200 |
| 2 | RU | NULL |
| 3 | KZ | 5000 |
| 4 | KZ | 6000 |
| 5 | UZ | 300 |

Группируем по `country` и считаем:

| country | `COUNT(*)` | `COUNT(amount)` | `COUNT(DISTINCT player_id)` | `SUM(amount)` | `AVG(amount)` |
|---------|-----------|-----------------|------------------------------|---------------|---------------|
| RU | 3 | 2 | 2 | 300 | 150 |
| KZ | 2 | 2 | 2 | 11000 | 5500 |
| UZ | 1 | 1 | 1 | 300 | 300 |

Обрати внимание на строку RU: `COUNT(*)` равно 3 (три строки), но `COUNT(amount)` равно 2 (одна строка с NULL не считается), а уникальных игроков всего 2 (игрок 1 положил два депозита). И `AVG` по RU равно 150, а не 100: NULL выкинут, сумма 300 делится на 2.

После `HAVING SUM(amount) > 10000` останется только KZ (11000 > 10000). RU (300) и UZ (300) отсеются. Это и есть смысл `HAVING` — фильтр по уже посчитанному итогу группы.

## Тонкости, которые многие не знают

- `COUNT(*)` считает все строки. `COUNT(column)` считает только строки где `column` не NULL. `COUNT(DISTINCT player_id)` — число уникальных игроков. Это три разных числа.
- `HAVING` — это `WHERE` для групп. `WHERE` фильтрует строки до группировки, `HAVING` — группы после. Хочешь "страны с суммой депозитов больше 10000" — это HAVING. Хочешь "только депозиты больше 100, потом сгруппировать" — это WHERE.
- `AVG` игнорирует NULL. Среднее по `[100, NULL, 200]` равно 150, а не 100. Это часто не то, что ты думаешь.

## Порядок выполнения — почему его важно понимать

SQL пишется в одном порядке, а выполняется в другом. Логический порядок такой:

`FROM` → `WHERE` → `GROUP BY` → `HAVING` → `SELECT` → `ORDER BY`

Из этого следуют два практических вывода:

- `WHERE` отрабатывает **до** `GROUP BY`, поэтому в нём нельзя ссылаться на агрегат: запись `WHERE SUM(amount) > 10000` — ошибка, на этом этапе сумма ещё не посчитана. Условие на агрегат идёт только в `HAVING`.
- `SELECT` (а значит и алиасы вроде `total_amount`) вычисляется **после** `GROUP BY` и `HAVING`. В большинстве СУБД нельзя использовать алиас `total_amount` внутри `HAVING` — нужно повторить выражение `SUM(amount)`. А вот в `ORDER BY` алиас уже доступен, потому что `ORDER BY` идёт последним.

Если хочешь и отфильтровать строки до свёртки, и группы после — используешь оба фильтра вместе:

```sql
SELECT country, SUM(amount) AS total_amount
FROM deposits
WHERE amount > 100          -- режем мелкие депозиты ДО группировки
GROUP BY country
HAVING SUM(amount) > 10000  -- режем "мелкие" страны ПОСЛЕ группировки
ORDER BY total_amount DESC;
```

## Типичные ошибки

- **Несгруппированный столбец в `SELECT`.** Правило: каждый столбец в `SELECT` должен быть либо в `GROUP BY`, либо завёрнут в агрегат. `SELECT player_id, country, SUM(amount) ... GROUP BY country` — ошибка (или, хуже, в MySQL "тихо" вернёт случайный `player_id` из группы и собьёт тебя с толку).
- **`AVG` по сумме как "средний чек на игрока".** `AVG(amount)` — это средний депозит, а не средний доход с игрока. Если игрок положил пять депозитов, он попадёт в среднее пять раз. Средний чек на игрока — это `SUM(amount) / COUNT(DISTINCT player_id)`.
- **`COUNT(column)` там, где нужен `COUNT(*)`.** Если в столбце есть NULL, числа разойдутся, и метрика "сколько всего депозитов" окажется занижена.
- **Фильтр по агрегату в `WHERE`.** Хочется написать `WHERE SUM(amount) > 10000` — не сработает, только `HAVING`.

## Зачем это в продуктовой аналитике

Почти любая базовая метрика гемблинга — это агрегат с группировкой:

- `SUM(amount)` по дате — дневной депозитный объём (deposit volume).
- `COUNT(DISTINCT player_id)` по дате — DAU (дневная активная аудитория) или число платящих.
- `SUM(amount) / COUNT(DISTINCT player_id)` по стране — ARPPU (средний доход с платящего) в этой стране.
- `HAVING COUNT(DISTINCT player_id) >= 30` — частый приём, чтобы отбросить мелкие сегменты, где статистика шумная и метрике нельзя доверять.

!!! question "Проверь себя"

    1. Чем отличаются `COUNT(*)`, `COUNT(column)` и `COUNT(DISTINCT player_id)`?
    2. В чём разница между `WHERE` и `HAVING` и почему "страны с суммой депозитов больше 10000" — это `HAVING`?
    3. Чему равно `AVG` по значениям `[100, NULL, 200]` и почему?

??? success "Ответы"

    1. `COUNT(*)` считает все строки; `COUNT(column)` — только строки, где `column` не NULL; `COUNT(DISTINCT player_id)` — число уникальных значений.
    2. `WHERE` фильтрует строки до группировки, `HAVING` — группы после агрегации. Сумма по стране известна только после `GROUP BY`, поэтому условие на неё ставится в `HAVING`.
    3. 150: `AVG` игнорирует NULL и делит сумму 300 на 2 непустых значения, а не на 3.
