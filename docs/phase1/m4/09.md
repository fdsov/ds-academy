# M4.9 · CASE WHEN — условная логика

Интуиция. Это `if/else` прямо внутри SQL. "Если депозит больше 500 — это 'high', иначе 'low'". Незаменим для сегментации и для так называемой условной агрегации.

На пальцах. `CASE WHEN` пробегает условия сверху вниз и возвращает значение из первого `THEN`, чьё `WHEN` оказалось истинным. Как только нашёл совпадение — дальше не смотрит. Если ни одно условие не сработало и нет `ELSE` — возвращается `NULL`. Это ключевой момент: `CASE` всегда возвращает ровно одно скалярное значение для каждой строки.

Две формы записи. Есть "поисковая" (searched) форма с полноценными условиями и "простая" (simple) форма со сравнением одного выражения:

```sql
-- searched CASE: любые условия
CASE WHEN amount >= 500 THEN 'high'
     WHEN amount >= 100 THEN 'mid'
     ELSE 'low'
END

-- simple CASE: сравнение одного выражения с набором значений
CASE country
     WHEN 'UZ' THEN 'Узбекистан'
     WHEN 'KZ' THEN 'Казахстан'
     ELSE 'Прочее'
END
```

Важно про порядок: условия проверяются последовательно. Если поставить `WHEN amount >= 100` выше, чем `WHEN amount >= 500`, то депозит на 700 попадёт в `'mid'`, потому что первое условие уже истинно. Всегда располагай более узкие (строгие) условия выше.

Формально:

```sql
SELECT player_id,
       SUM(CASE WHEN amount >= 500 THEN amount ELSE 0 END) AS high_amount,
       COUNT(CASE WHEN amount < 50 THEN 1 END)             AS small_deposits,
       AVG(CASE WHEN country = 'UZ' THEN amount END)        AS avg_uz_check
FROM deposits
GROUP BY player_id;
```

Приём `SUM(CASE WHEN ... THEN x ELSE 0 END)` называется условная агрегация и решает огромный класс задач: посчитать несколько метрик с разными фильтрами в одном проходе по таблице, без многократного сканирования. Это и быстрее, и читаемее, чем три отдельных запроса с JOIN.

## Почему COUNT и AVG ведут себя по-разному

Тонкость, которая ломает половину запросов у новичков, — взаимодействие `CASE` с `NULL` и агрегатами.

- `COUNT(CASE WHEN amount < 50 THEN 1 END)` — здесь нет `ELSE`, поэтому для строк с `amount >= 50` выражение даёт `NULL`. А `COUNT` по выражению **не считает `NULL`**. Значит, мы получаем именно количество мелких депозитов. Если бы мы написали `COUNT(CASE WHEN amount < 50 THEN 1 ELSE 0 END)`, то для крупных депозитов было бы `0` (не `NULL`), и `COUNT` посчитал бы **все** строки — метрика сломалась бы.
- `AVG(CASE WHEN country = 'UZ' THEN amount END)` — снова без `ELSE`. Для не-UZ строк значение `NULL`, и `AVG` его игнорирует. Поэтому в знаменателе среднего окажутся только депозиты из UZ. Если добавить `ELSE 0`, то нули из других стран попадут в среднее и занизят результат.

Правило простое:

- для `SUM` пиши `ELSE 0` — чтобы складывать только нужное, а остальное обнулять;
- для `COUNT` и `AVG` оставляй без `ELSE` — чтобы лишние строки превращались в `NULL` и выпадали из подсчёта.

## Разбор на числах

Пусть у игрока такие депозиты:

| amount | country |
|--------|---------|
| 700    | UZ      |
| 40     | UZ      |
| 200    | KZ      |
| 30     | KZ      |

Считаем три метрики из запроса выше:

- `high_amount = SUM(CASE WHEN amount >= 500 THEN amount ELSE 0 END)` = `700 + 0 + 0 + 0 = 700`.
- `small_deposits = COUNT(CASE WHEN amount < 50 THEN 1 END)` = считаем строки 40 и 30, остальные `NULL` → `2`.
- `avg_uz_check = AVG(CASE WHEN country = 'UZ' THEN amount END)` = берём только UZ-строки `700` и `40`, среднее `(700 + 40) / 2 = 370`. KZ-строки стали `NULL` и в знаменатель не попали.

Заметь: если бы в `avg_uz_check` стоял `ELSE 0`, среднее было бы `(700 + 40 + 0 + 0) / 4 = 185` — совсем другой и неверный для задачи ответ.

## Продуктовый пример: доля FTD в один проход

Классическая аналитическая задача — посчитать конверсию или долю сегмента без подзапросов. Условная агрегация делает это элегантно:

```sql
SELECT
    country,
    COUNT(*)                                                   AS players,
    SUM(CASE WHEN is_ftd THEN 1 ELSE 0 END)                    AS ftd_players,
    ROUND(
        100.0 * SUM(CASE WHEN is_ftd THEN 1 ELSE 0 END) / COUNT(*),
    2)                                                         AS ftd_rate_pct,
    SUM(CASE WHEN is_ftd THEN deposit_usd ELSE 0 END)          AS ftd_revenue_usd
FROM players
GROUP BY country;
```

Здесь в одном проходе мы получаем и количество игроков, и число сделавших первый депозит (FTD), и их долю в процентах, и сумму первых депозитов в долларах. Множитель `100.0` нужен, чтобы деление было дробным, а не целочисленным.

## Типичные ошибки

- Целочисленное деление: `SUM(...)/COUNT(*)` на целых типах даст `0` для долей меньше единицы. Лечится умножением на `100.0` или приведением `::numeric`.
- Перепутанный порядок условий — широкое условие выше узкого "съедает" строки (см. пример с `mid`/`high`).
- `ELSE 0` там, где нужен `NULL` (внутри `AVG`/`COUNT`) — занижает или завышает метрику.
- Забытый `ELSE` в `SUM` обычно безвреден (несовпавшее = `NULL`, `SUM` его игнорирует), но `ELSE 0` делает намерение явным и читается лучше.
- `CASE` нельзя ставить там, где СУБД ждёт имя столбца на этапе планирования (например, нельзя `GROUP BY` по алиасу `CASE` в некоторых диалектах — приходится повторять выражение или оборачивать в подзапрос).

!!! warning "ELSE и NULL"

    Отсутствие `ELSE` — это не ошибка, а инструмент. Несовпавшие строки становятся `NULL`, и агрегаты `SUM`, `AVG`, `COUNT(expr)` их пропускают. Осознанно выбирай между `ELSE 0` (когда нужен ноль в сумме) и отсутствием `ELSE` (когда строка должна выпасть из расчёта).

!!! note "Проверь себя"

    Чему равно `AVG(CASE WHEN country = 'UZ' THEN amount END)` для четырёх строк из таблицы выше — и как изменится ответ, если дописать `ELSE 0`? Ответ: `370` без `ELSE` и `185` с `ELSE 0`, потому что во втором случае нули из KZ попадают в знаменатель.
