# M4.14 · Диалекты: PostgreSQL и DuckDB

PostgreSQL — стандарт продакшен-баз: транзакционная (OLTP) и аналитическая работа, надёжность, богатый SQL. Это та база, где живут данные приложения. Скорее всего, твои продуктовые таблицы лежат именно в Postgres или его форке.

DuckDB — стандарт локальной аналитики 2026 года. Это встраиваемая аналитическая (OLAP) база, "SQLite для аналитики". Она колоночная, что делает её в десятки раз быстрее на агрегациях, и работает прямо из питона без сервера. DuckDB читает Parquet и CSV напрямую как таблицы и даже умеет выполнять SQL по pandas-датафреймам в памяти. Для EDA, фичеринга и ad-hoc аналитики на данных до сотен гигабайт DuckDB — лучший выбор на одной машине.

Диалекты на 95% совпадают по синтаксису, который мы прошли. Различия в деталях: функции дат, приведение типов, специфичные расширения. Профессионал пишет переносимый SQL и держит в голове, где диалекты расходятся.

## OLTP против OLAP — почему вообще две базы

Главное различие — не синтаксис, а то, как данные лежат на диске, и под какую нагрузку база заточена.

- **OLTP (Postgres)** — Online Transaction Processing. Хранит данные **построчно** (row-store). Оптимизирован под много мелких операций: "вставь одну ставку", "обнови баланс игрока", "найди игрока по id". Когда тебе нужна одна полная строка целиком — row-store идеален: вся строка лежит рядом, читается одним движением.
- **OLAP (DuckDB)** — Online Analytical Processing. Хранит данные **по колонкам** (column-store). Оптимизирован под "посчитай сумму депозитов по всем 50 млн строк за год". При такой агрегации тебе нужна одна-две колонки из сорока. Колоночная база читает с диска **только** нужные колонки, а не всю таблицу, и сжимает их лучше (в колонке однотипные значения). Отсюда десятикратное ускорение на агрегациях.

На пальцах: представь таблицу `bets` на 50 млн строк и 40 колонок. Запрос `SELECT SUM(bet_amount) FROM bets`. Postgres физически вынужден пройтись по всем строкам со всеми 40 колонками (если нет покрывающего индекса). DuckDB прочитает с диска только колонку `bet_amount` — это ~1/40 объёма данных, да ещё и в сжатом виде. Вот откуда разница в скорости.

!!! note "Когда какую базу брать"

    Постоянные данные приложения, конкурентные записи, транзакции, "истина в последней инстанции" — Postgres. Разовый разбор выгрузки, фичеринг для ML, тяжёлая агрегация на ноутбуке, чтение Parquet/CSV — DuckDB. В реальном пайплайне они работают в связке: продакшен-данные живут в Postgres, ты выгружаешь срез в Parquet и крутишь его в DuckDB локально.

## Где диалекты реально расходятся

Те самые 5% различий, на которых ломаются перенесённые запросы.

**Разница дат (главная боль переносимости).** В Postgres вычитание дат даёт `interval`, и чтобы получить число дней, явно берут `EXTRACT`:

```sql
-- PostgreSQL: разница в днях между регистрацией и первым депозитом
SELECT EXTRACT(DAY FROM (first_deposit_at - registered_at)) AS days_to_ftd
FROM players;
```

```sql
-- DuckDB: есть удобная функция date_diff с явной единицей
SELECT date_diff('day', registered_at, first_deposit_at) AS days_to_ftd
FROM players;
```

Функция `date_diff('day', ...)` есть и в DuckDB, и в современных версиях Postgres-совместимых движков, но сигнатуры и поведение на границах суток отличаются — всегда проверяй на своём движке.

**Текущее время.** `now()` и `current_timestamp` есть в обоих. Но "усечь до начала дня" — `date_trunc('day', ts)` работает одинаково в обоих диалектах, это безопасный переносимый вариант (лучше, чем парсить строки).

**Приведение типов.** Канонический ANSI-способ `CAST(x AS INTEGER)` работает везде. Постгресовый шорткат `x::integer` работает и в Postgres, и в DuckDB (DuckDB намеренно копирует синтаксис Postgres). А вот MySQL-стиль `CONVERT(...)` — уже не переносим.

```sql
-- работает и в Postgres, и в DuckDB
SELECT CAST(bet_amount AS DECIMAL(12,2)) AS bet_amount,
       bet_amount::INTEGER               AS bet_rounded
FROM bets;
```

**Строки.** Конкатенация через `||` — стандарт, работает в обоих. `string_agg(x, ',')` тоже общий. А `GROUP_CONCAT` (MySQL) — нет.

**Чтение файлов — суперсила DuckDB.** В Postgres, чтобы поднять CSV, нужен `COPY` в реальную таблицу или внешнее расширение. В DuckDB файл — это уже таблица:

```sql
-- DuckDB: Parquet/CSV прямо как источник, без загрузки в таблицу
SELECT country, SUM(deposit_amount) AS dep_vol
FROM 'data/deposits_2026.parquet'
WHERE deposit_amount > 0
GROUP BY country
ORDER BY dep_vol DESC;
```

## Разобранный пример: один запрос — два диалекта

Задача из гемблинг-аналитики: посчитать DepVol (сумму депозитов) и число уникальных платящих игроков по странам за июнь 2026.

```sql
-- PostgreSQL: данные уже в таблице deposits
SELECT country,
       SUM(amount)              AS dep_vol_usd,
       COUNT(DISTINCT player_id) AS payers
FROM deposits
WHERE created_at >= DATE '2026-06-01'
  AND created_at <  DATE '2026-07-01'
GROUP BY country
ORDER BY dep_vol_usd DESC;
```

```sql
-- DuckDB: ровно тот же запрос, но источник — Parquet-выгрузка
SELECT country,
       SUM(amount)              AS dep_vol_usd,
       COUNT(DISTINCT player_id) AS payers
FROM 'exports/deposits_june.parquet'
WHERE created_at >= DATE '2026-06-01'
  AND created_at <  DATE '2026-07-01'
GROUP BY country
ORDER BY dep_vol_usd DESC;
```

Заметь: тело запроса (`SELECT ... GROUP BY ... ORDER BY`, фильтр по полуинтервалу `>= ... AND < ...`, `COUNT(DISTINCT ...)`) идентично. Разошёлся только источник данных — таблица против файла. Именно так выглядит переносимый SQL на практике: ядро логики одно, отличается обвязка.

!!! tip "Полуинтервал по датам вместо BETWEEN"

    Фильтр `>= '2026-06-01' AND < '2026-07-01'` переносим и работает одинаково в обоих диалектах. Он надёжнее, чем `BETWEEN`, который включает обе границы и ломается на временах с долями секунды (`23:59:59.999`). Это правило справедливо и для Postgres, и для DuckDB.

## Типичные ошибки

- **Слепой перенос date-функций.** Запрос с `EXTRACT(DAY FROM a - b)` из Postgres не обязан вести себя так же в другом движке. Перенося запрос между диалектами — пересмотри именно работу с датами и интервалами, остальное обычно совпадает.
- **Использование диалект-специфичных шорткатов без нужды.** Если запрос будет жить и в Postgres, и в DuckDB — предпочитай ANSI-формы (`CAST`, `date_trunc`, `||`, `string_agg`) вместо экзотики.
- **Ожидание, что DuckDB заменит Postgres в проде.** DuckDB — встраиваемый одно-процессный движок без полноценной конкурентной многопользовательской записи. Это инструмент аналитика на его машине, а не замена транзакционной БД приложения.
- **`COUNT(DISTINCT ...)` на огромных данных в Postgres.** В row-store это дорого; в DuckDB та же агрегация на колонке отрабатывает заметно быстрее — ещё одна причина выгружать срез в DuckDB для тяжёлой аналитики.

!!! note "Проверь себя"

    Почему `SELECT SUM(amount) FROM bets` на таблице в 50 млн строк выполнится в DuckDB быстрее, чем в Postgres без подходящего индекса? Потому что DuckDB колоночный: он прочитает с диска только сжатую колонку `amount`, а не все строки целиком, как построчный Postgres.
