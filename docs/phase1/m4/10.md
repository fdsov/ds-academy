# M4.10 · Работа с датами

Интуиция. Время — основа продуктовой аналитики: всё считается "по дням", "за последние 7 дней", "по месяцам регистрации". Нужно уметь обрезать timestamp до даты, считать разницу и группировать по периодам.

Чтобы не путаться, держи в голове три разных типа данных:

- `timestamp` — точный момент: дата + время до секунд/миллисекунд (`2026-06-17 01:34:22`). Это то, что обычно пишет бэкенд в момент события (ставка, депозит, логин).
- `date` — только календарная дата без времени (`2026-06-17`). Получается обрезкой timestamp.
- `interval` — длительность, разница между двумя моментами (`2 days 03:15:00`). Получается вычитанием.

Большинство ошибок новичка — это путаница между этими тремя: например, попытка сгруппировать DAU прямо по `timestamp` (тогда каждая строка с разными секундами станет отдельной "группой" и группировки фактически не будет).

Формально (PostgreSQL и DuckDB во многом совпадают):

```sql
SELECT date_trunc('day', bet_ts)      AS day,        -- обрезать до дня
       date_trunc('month', bet_ts)    AS month,      -- до месяца
       bet_ts::date                   AS day_cast,    -- то же короче
       deposit_ts - signup_ts         AS time_to_ftd, -- интервал
       EXTRACT(dow FROM bet_ts)        AS weekday     -- день недели 0-6
FROM bets;
```

Разберём ключевую тонкость: `date_trunc('day', bet_ts)` и `bet_ts::date` дают логически "тот же день", но это РАЗНЫЕ типы. `date_trunc` возвращает `timestamp` с занулённым временем (`2026-06-17 00:00:00`), а `::date` возвращает чистый `date` (`2026-06-17`). Для группировки DAU оба работают, но для отчётов и джойнов с календарной таблицей чище `::date`.

`EXTRACT` вытаскивает из даты числовую "часть": `dow` (день недели, 0 = воскресенье ... 6 = суббота), `hour`, `month`, `year`, `epoch` (число секунд от 1970). Это незаменимо для анализа паттернов: например, "в какие часы суток идёт пик ставок" — `EXTRACT(hour FROM bet_ts)`.

Для DAU (Daily Active Users) ключевое — обрезать timestamp до дня и считать уникальных игроков:

```sql
SELECT bet_ts::date AS day,
       COUNT(DISTINCT player_id) AS dau
FROM bets
GROUP BY bet_ts::date
ORDER BY day;
```

### Разбор на пальцах: почему именно COUNT(DISTINCT)

Представь три строки за один день:

| player_id | bet_ts              |
|-----------|---------------------|
| 101       | 2026-06-17 10:05:00 |
| 101       | 2026-06-17 22:40:00 |
| 202       | 2026-06-17 13:00:00 |

Игрок 101 поставил две ставки. `COUNT(*)` дал бы 3 (число ставок), а DAU — это число уникальных активных игроков, поэтому нужен `COUNT(DISTINCT player_id)` = 2. Путаница `COUNT(*)` вместо `COUNT(DISTINCT player_id)` — классическая ошибка, из-за которой DAU оказывается завышенным в разы (он начинает считать активность, а не людей).

### Численный пример: интервалы и time-to-FTD

`time_to_ftd` (время от регистрации до первого депозита) — важная метрика онбординга. Если игрок зарегистрировался `2026-06-15 09:00:00`, а первый депозит сделал `2026-06-17 12:00:00`:

```sql
SELECT (DATE '2026-06-17' + TIME '12:00')
     - (DATE '2026-06-15' + TIME '09:00') AS time_to_ftd;
-- результат: 2 days 03:00:00 (интервал)
```

Чтобы превратить интервал в число для агрегаций (медиана, среднее), извлекают секунды/часы/дни:

```sql
SELECT EXTRACT(epoch FROM (deposit_ts - signup_ts)) / 3600.0 AS hours_to_ftd
FROM players;
```

`epoch` даёт длительность в секундах, делим на 3600 — получаем часы. По такому полю уже можно считать `percentile_cont(0.5)` (медианное время до депозита). Среднее здесь обманчиво: один игрок, который занёс депозит спустя 90 дней, утянет mean вверх, поэтому в продуктовой аналитике для time-to-event почти всегда смотрят на медиану.

### Полезные операции с датами

```sql
SELECT current_date,                              -- сегодня
       bet_ts::date - INTERVAL '7 days'           AS week_ago,   -- сдвиг назад
       bet_ts::date + 30                          AS plus_30d,   -- date + целое = +N дней
       date_trunc('week', bet_ts)                 AS week_start, -- начало недели (понедельник)
       age(deposit_ts, signup_ts)                 AS human_gap   -- "2 mons 5 days"
FROM bets;
```

Окно "за последние 7 дней" в фильтре пишут так:

```sql
WHERE bet_ts >= current_date - INTERVAL '7 days'
```

Важно сравнивать `bet_ts` (timestamp) с границей по timestamp/date корректно: если написать `bet_ts = current_date`, сравнение почти всегда даст false, потому что у `bet_ts` есть время, а у `current_date` — нет (оно как `00:00:00`). Поэтому для "событий сегодня" пишут `bet_ts::date = current_date`, а не `bet_ts = current_date`.

!!! warning "Часовой пояс — тихий убийца метрик"

    Если `bet_ts` хранится в UTC, а бизнес считает день по местному времени, твой "день" будет сдвинут на несколько часов. Игрок, поставивший ставку в 1 ночи по локальному, попадёт во "вчера" по UTC. DAU и retention поедут. Всегда уточняй таймзону хранения и приводи к нужной: `bet_ts AT TIME ZONE 'UTC' AT TIME ZONE 'Asia/Tashkent'`.

!!! danger "Дырки в днях ломают графики"

    `GROUP BY day` показывает только те дни, где БЫЛА хотя бы одна ставка. Если в какой-то день не было активности (праздник, инцидент, обнуление), этой строки просто не будет в результате — на графике появится "склейка" соседних дней, и тренд исказится. Чтобы DAU был непрерывным, генерируют полный календарь (`generate_series`) и делают `LEFT JOIN` с данными, подставляя 0 там, где активности не было.

### Частые ошибки

- `GROUP BY bet_ts` вместо `GROUP BY bet_ts::date` — группировка по секундам, каждая строка отдельно.
- `COUNT(*)` вместо `COUNT(DISTINCT player_id)` для DAU — считаются события, а не люди.
- `bet_ts = current_date` для "сегодня" — почти всегда false из-за компоненты времени.
- Игнор таймзоны — день сдвинут на UTC-offset, метрика тихо врёт.
- Среднее вместо медианы для time-to-FTD — длинный хвост перекашивает результат.
