```python
import duckdb
import pandas as pd

con = duckdb.connect()

# DuckDB читает Parquet/CSV напрямую как таблицу — без загрузки в память
dau = con.execute("""
    SELECT bet_ts::date AS day,
           COUNT(DISTINCT player_id) AS dau,
           SUM(stake) AS total_stake,
           SUM(stake - payout) AS ggr   -- gross gaming revenue
    FROM 'bets.parquet'
    GROUP BY bet_ts::date
    ORDER BY day
""").df()   # .df() возвращает pandas DataFrame

# Депозитные когорты: месяц первого депозита и LTV-кривая
cohorts = con.execute("""
    WITH first_dep AS (
        SELECT player_id,
               date_trunc('month', MIN(deposit_ts)) AS cohort_month,
               MIN(deposit_ts) AS ftd_ts
        FROM 'deposits.parquet'
        GROUP BY player_id
    )
    SELECT f.cohort_month,
           COUNT(DISTINCT f.player_id) AS cohort_size,
           SUM(d.amount) AS revenue,
           SUM(d.amount) / COUNT(DISTINCT f.player_id) AS arpu
    FROM first_dep f
    JOIN 'deposits.parquet' d ON d.player_id = f.player_id
    GROUP BY f.cohort_month
    ORDER BY f.cohort_month
""").df()
```

!!! tip "AI-копилот: где нейросети помогают"

    LLM (Claude в Cursor, Copilot) сильны в SQL: они отлично генерируют скелет запроса по описанию, объясняют чужой запрос на 100 строк, переписывают коррелированный подзапрос в оконную функцию, ловят забытый GROUP BY.

    Хороший промпт даёт схему и точную задачу: "Вот DDL таблиц players(player_id, signup_date, country) и deposits(deposit_id, player_id, amount, deposit_ts). Напиши PostgreSQL-запрос для D7 retention по когортам месяца регистрации. Активность = наличие депозита. Объясни логику оконных функций."

    Где они врут и что перепроверять обязательно: LLM регулярно путает диалекты (предлагает MySQL-функцию `DATE()` для Postgres, или `TOP` вместо `LIMIT`). Не учитывает раздувание строк при JOIN — спокойно выдаст `SUM` по соединению, завышающий метрику в разы. Игнорирует логику NULL в `NOT IN`. И главное: модель не знает реального распределения твоих данных — корректность бизнес-логики (что считать "активным", какая таймзона, что есть FTD) проверяешь только ты. Любой сгенерированный запрос гоняй на маленькой выборке и сверяй итоги вручную.

!!! note "Стек 2026"

    Актуально: DuckDB — де-факто стандарт локальной и embedded-аналитики, читает Parquet/CSV/JSON напрямую и работает по pandas/Polars-датафреймам в памяти. PostgreSQL остаётся королём продакшен-баз. dbt — стандарт для трансформаций SQL в дата-команде (версионирование, тесты, документация моделей). Оконные функции и CTE — обязательная база, без них на интервью не берут.

    Устарело относительно старых курсов: учить MySQL как первый диалект — менее релевантно, аналитика ушла в Postgres/DuckDB. Курсоры и хранимые процедуры для аналитики почти не нужны. Выгрузка всей таблицы в pandas для агрегации, которую можно сделать в SQL, — антипаттерн: считай в базе, тащи в питон уже агрегат.

!!! warning "Ловушки"

    Раздувание строк при JOIN многих таблиц — `SUM` считает значения по несколько раз. Агрегируй до соединения.

    `NOT IN (подзапрос с NULL)` молча возвращает ноль строк. Используй `NOT EXISTS`.

    Функция на столбце в WHERE (`WHERE lower(country) = 'uz'`, `date_trunc(...)`) отключает индекс — переписывай на диапазон или функциональный индекс.

    `WHERE col <> 'x'` теряет строки с `col IS NULL`. Если они нужны — добавь `OR col IS NULL`.

    `COUNT(*)` против `COUNT(DISTINCT player_id)` — путаница даёт неверный DAU. Уникальные пользователи всегда через DISTINCT.

    `SELECT *` в продакшен-запросах и дашбордах — тащит лишние столбцы, ломается при изменении схемы. Перечисляй явно.

    Senior-ловушка: доверять плану на тестовых данных. На 1000 строках Seq Scan быстрее Index Scan, и план врёт о том, что будет на проде с 200 миллионами. Анализируй на реалистичном объёме.
