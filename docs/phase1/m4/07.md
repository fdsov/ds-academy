# M4.7 · Подзапросы и CTE

Интуиция. Подзапрос — это запрос внутри запроса: сначала посчитай вот это, потом используй результат. CTE (Common Table Expression, конструкция `WITH`) — то же самое, но вынесенное наверх и названное человеческим именем. Это как разбить длинную формулу на промежуточные переменные.

На пальцах. Представь, что считаешь в уме среднюю ставку игрока за месяц, а потом ищешь, кто ставит выше неё. Ты не держишь обе мысли одновременно — ты сначала находишь одно число (среднее), запоминаешь его, и только потом сравниваешь. База данных делает ровно то же: подзапрос даёт промежуточный результат, внешний запрос им пользуется. CTE просто даёт этому промежуточному результату имя, чтобы не держать его «в голове» внутри скобок.

Формально. Подзапрос можно поставить в `FROM`, в `WHERE`, даже в `SELECT`:

```sql
SELECT player_id, amount
FROM deposits
WHERE amount > (SELECT AVG(amount) FROM deposits);
```

Здесь подзапрос считает среднее, а внешний запрос берёт депозиты выше среднего. Тот же запрос через CTE становится читаемее, когда логика растёт:

```sql
WITH player_revenue AS (
    SELECT player_id, SUM(amount) AS revenue
    FROM deposits
    GROUP BY player_id
),
big_spenders AS (
    SELECT player_id
    FROM player_revenue
    WHERE revenue > 1000
)
SELECT p.country, COUNT(*) AS whales
FROM big_spenders b
JOIN players p ON p.player_id = b.player_id
GROUP BY p.country;
```

CTE — не про производительность (хотя иногда влияет), а про читаемость и поддержку. Запрос на 80 строк без CTE невозможно проверить. Разбитый на именованные шаги — читается как абзацы текста. Правило профессионала: если запрос длиннее экрана — почти наверняка он просится в CTE.

## Три места, где живёт подзапрос

Подзапрос ведёт себя по-разному в зависимости от того, куда его поставили. Это важно различать, потому что от этого зависит, что он обязан вернуть.

- **В `WHERE` как скаляр** — подзапрос обязан вернуть ровно одно значение (одна строка, одна колонка). Пример выше с `AVG(amount)`. Если такой подзапрос внезапно вернёт две строки, СУБД упадёт с ошибкой `more than one row returned by a subquery`.
- **В `WHERE` со списком (`IN`)** — подзапрос возвращает колонку значений, а внешний запрос проверяет вхождение:

```sql
SELECT player_id, amount
FROM deposits
WHERE player_id IN (
    SELECT player_id FROM players WHERE country = 'BR'
);
```

- **В `FROM` как «таблица на лету»** (производная таблица, derived table) — подзапрос возвращает целый набор строк, и внешний запрос обращается к нему как к обычной таблице. Этому подзапросу обязательно нужен псевдоним:

```sql
SELECT country, AVG(revenue) AS avg_rev
FROM (
    SELECT player_id, SUM(amount) AS revenue
    FROM deposits
    GROUP BY player_id
) AS pr
JOIN players p ON p.player_id = pr.player_id
GROUP BY country;
```

## Разбор численного примера

Возьмём таблицу `deposits` из пяти строк и проследим, что происходит:

| player_id | amount |
|-----------|--------|
| 1         | 50     |
| 1         | 150    |
| 2         | 20     |
| 3         | 800    |
| 3         | 400    |

Запрос `WHERE amount > (SELECT AVG(amount) FROM deposits)`:

1. Подзапрос считает среднее: $(50 + 150 + 20 + 800 + 400) / 5 = 1420 / 5 = 284$.
2. Внешний запрос оставляет строки с `amount > 284`: это депозиты на 800 и 400 (оба игрока `3`).

Тонкость, которая ловит почти всех новичков: сравнение идёт построчно с **общим** средним, а не со средним внутри игрока. Депозит игрока 1 на 150 не проходит, хотя он выше личного среднего игрока 1. Если нужно «выше своего среднего» — это уже оконная функция (`AVG(amount) OVER (PARTITION BY player_id)`), а не обычный подзапрос.

## Коррелированный подзапрос

Особый случай — подзапрос, который ссылается на колонку из внешнего запроса. Он пересчитывается для каждой строки внешнего запроса, как тело цикла:

```sql
SELECT d.player_id, d.amount
FROM deposits d
WHERE d.amount > (
    SELECT AVG(amount)
    FROM deposits d2
    WHERE d2.player_id = d.player_id
);
```

Здесь для каждой строки `d` подзапрос считает среднее **именно этого** игрока (`d2.player_id = d.player_id`). Это и есть «выше своего среднего». Минус — цена: на миллионах строк коррелированный подзапрос может выполняться очень медленно, потому что концептуально это вложенный цикл. На больших объёмах его почти всегда переписывают через оконную функцию или `JOIN` с агрегированным CTE.

## Связь с гемблинг-аналитикой

Типичная боевая задача: посчитать ARPU (средний доход на игрока) по странам, но только среди «китов» — игроков с депозитом выше 1000 долларов. В лоб это не выразить одним `GROUP BY`, потому что сначала надо агрегировать по игроку, потом отфильтровать, и только потом группировать по стране. Ровно это делает второй пример с CTE наверху: `player_revenue` сворачивает депозиты в выручку на игрока, `big_spenders` фильтрует китов, финальный `SELECT` считает их по странам. Три понятных шага вместо нечитаемой матрёшки из скобок.

!!! warning "Типичные ошибки"

    - Скалярный подзапрос в `WHERE`, который вернул больше одной строки, — ошибка выполнения. Используй `IN` вместо `=`, если ожидаешь список.
    - Подзапрос в `FROM` без псевдонима — синтаксическая ошибка в большинстве СУБД (`subquery in FROM must have an alias`).
    - Путаница «общее среднее» против «среднее внутри группы»: обычный подзапрос даёт первое, коррелированный или оконная функция — второе.
    - `NOT IN` с подзапросом, где есть `NULL`, может молча вернуть пустой результат — коварный баг. На таких задачах безопаснее `NOT EXISTS`.

!!! note "Проверь себя"

    Чем коррелированный подзапрос отличается от обычного, и почему он может быть медленным? (Подсказка: он ссылается на колонку внешнего запроса и пересчитывается для каждой строки.)
