# M4.12 · Индексы и почему запрос медленный

Интуиция. Индекс — это как алфавитный указатель в конце книги. Без него, чтобы найти все депозиты игрока 12345, база читает всю таблицу подряд (full scan) — все 200 миллионов строк. С индексом по `player_id` она прыгает сразу к нужным строкам, как ты открываешь словарь на нужной букве.

Чтобы интуиция не оставалась метафорой, прикинем в числах. Допустим, в `deposits` 200 млн строк, и база читает примерно 1 млн строк в секунду при последовательном чтении. Full scan = около 200 секунд только на то, чтобы перебрать всё и выкинуть 99.99% строк. Теперь представь, что у игрока 12345 ровно 80 депозитов. B-дерево высотой 4-5 уровней позволяет найти первую подходящую строку за несколько обращений к диску, а дальше прочитать 80 строк. Итог — миллисекунды вместо минут. Именно поэтому в продуктовой аналитике отчёт «депозиты конкретного игрока» без индекса по `player_id` способен положить базу, а с индексом отвечает мгновенно.

Формально. Индекс — это вспомогательная структура данных (чаще всего B-дерево), которая хранит отсортированные значения столбца и указатели на строки. Создаётся так:

```sql
CREATE INDEX idx_deposits_player ON deposits (player_id);
```

Как устроено B-дерево «на пальцах». Это сбалансированное дерево: корень, промежуточные узлы и листья. В листьях лежат отсортированные значения `player_id` и ссылки на физическое расположение строк. Поиск идёт сверху вниз — на каждом уровне база выбирает нужную ветку, отбрасывая остальные. Поскольку дерево сбалансировано, глубина растёт логарифмически: при росте таблицы в 1000 раз глубина увеличивается всего на пару уровней. Отсюда ключевое свойство — индекс особенно выгоден на больших таблицах, а на маленьких выигрыша почти нет.

Когда индекс помогает: фильтрация (`WHERE player_id = ...`), соединения (`JOIN ON player_id`), сортировка по индексированному столбцу. Когда не помогает или вредит: на маленьких таблицах (full scan и так дёшев), при записи (каждый INSERT обновляет все индексы — это замедляет вставку), при низкой селективности (индекс по столбцу `is_active` с двумя значениями почти бесполезен).

Про селективность подробнее. Селективность — это насколько узко условие отсекает строки. Высокая селективность = условие оставляет малую долю таблицы (например, `player_id = 12345` оставляет 80 строк из 200 млн — это 0.00004%). Низкая селективность = условие оставляет большую долю (например, `is_active = true`, если активны 70% игроков). Оптимизатор базы сам решает использовать индекс или нет: если по индексу всё равно придётся вытащить половину таблицы, дешевле сделать честный full scan, чем прыгать индексом туда-сюда по диску. Поэтому индекс по булеву флагу или по полу игрока почти всегда бесполезен, а индекс по `player_id`, `email`, `transaction_id` — почти всегда полезен.

Цена записи в долларах смысла. Индексы не бесплатны. Каждый дополнительный индекс на таблице — это структура, которую нужно обновлять при каждом `INSERT`, `UPDATE` затронутого столбца и `DELETE`. Если на `deposits` висит 6 индексов, одна вставка депозита превращается в 1 запись строки + 6 обновлений деревьев. На потоке в тысячи депозитов в минуту это заметная нагрузка. Практический вывод: индексируй то, по чему реально фильтруешь и джойнишь в отчётах, и не плоди индексы «на всякий случай».

Почему запрос медленный, типичные причины: нет индекса на столбце фильтра/джойна; функция на индексированном столбце (`WHERE date_trunc('day', ts) = ...` отключает индекс по `ts` — пиши `ts >= ... AND ts < ...`); JOIN раздувает строки; `SELECT *` тащит лишние столбцы.

Разбор главной ловушки — функция на столбце. Это самая частая причина «у нас же есть индекс, почему медленно?». Индекс по `ts` хранит отсортированные сырые значения `ts`. Как только ты оборачиваешь столбец в функцию, база не может сопоставить её с тем, что лежит в индексе, и вынуждена вычислять функцию для каждой строки — то есть снова full scan.

```sql
-- Плохо: индекс по ts не используется, full scan
SELECT count(*) FROM deposits
WHERE date_trunc('day', ts) = DATE '2026-06-17';

-- Хорошо: индекс по ts работает, читается узкий диапазон
SELECT count(*) FROM deposits
WHERE ts >= TIMESTAMP '2026-06-17 00:00:00'
  AND ts <  TIMESTAMP '2026-06-18 00:00:00';
```

Тот же принцип ломает индекс в случаях `WHERE lower(email) = '...'` (функция на столбце), `WHERE player_id::text = '12345'` (приведение типа), `WHERE amount + bonus > 100` (арифметика на столбце). Правило: держи индексированный столбец «голым» в левой части условия, а все преобразования переноси на правую часть (на константу) или используй специальный функциональный индекс (`CREATE INDEX ... ON deposits (lower(email))`).

Как проверить, использовался ли индекс. Не гадай — спроси у базы через план выполнения:

```sql
EXPLAIN ANALYZE
SELECT * FROM deposits WHERE player_id = 12345;
```

В выводе ищи слова: `Index Scan` / `Index Only Scan` — индекс используется, хорошо; `Seq Scan` (Sequential Scan) на большой таблице с узким фильтром — индекс не используется, плохо, повод разобраться. `ANALYZE` реально выполняет запрос и показывает фактическое время и число прочитанных строк, поэтому это лучший инструмент для диагностики «почему медленно».

!!! warning "Типичные ошибки"

    - Навесить индекс на `is_active`, `gender`, `currency` и ждать ускорения — низкая селективность, индекс почти всегда бесполезен.
    - Создать десяток индексов «чтобы наверняка» — раздувается стоимость записи и место на диске.
    - Фильтровать по `date_trunc('day', ts)` / `cast` / `lower()` и удивляться, что индекс «не работает».
    - Решать, что индекс используется, по ощущениям, вместо `EXPLAIN ANALYZE`.

!!! note "Проверь себя"

    Отчёт по FTD (первым депозитам) за вчера тормозит. В `WHERE` стоит `date_trunc('day', deposit_ts) = DATE '2026-06-16'`, индекс по `deposit_ts` есть. Почему медленно и как починить? Ответ: функция `date_trunc` на индексированном столбце отключает индекс. Перепиши на диапазон `deposit_ts >= '2026-06-16' AND deposit_ts < '2026-06-17'` — индекс заработает.

### Быстрый квиз

Выбери вариант — ответ подсветится сразу.

```quiz
Почему `WHERE date_trunc('day', ts) = ...` мешает использовать индекс по `ts`?
[ ] date_trunc слишком медленная функция сама по себе
[x] функция на столбце не сопоставляется со значениями в индексе, и база считает её для каждой строки
[ ] индекс по timestamp в принципе не существует
> Индекс хранит сырые значения `ts`. Преобразование столбца функцией ломает сопоставление, поэтому нужен полный перебор. Решение — переписать условие на диапазон.
---
Для какого столбца индекс даст наибольший выигрыш на таблице в 200 млн строк?
[x] player_id (миллионы разных значений, высокая селективность)
[ ] is_active (два значения true/false)
[ ] currency (несколько валют)
> Индекс эффективен при высокой селективности, когда условие отсекает почти всю таблицу. Булевы и малозначные столбцы — низкая селективность, индекс почти бесполезен.
---
Что является обратной стороной (ценой) добавления индексов?
[ ] запросы на чтение становятся медленнее
[x] замедляется запись: каждый INSERT/UPDATE/DELETE обновляет все индексы
[ ] таблица перестаёт принимать новые строки
> Индекс ускоряет чтение, но его нужно поддерживать при записи. Много индексов = дорогие вставки и обновления, плюс место на диске.
---
Какой командой проверить, использовался ли индекс в запросе?
[ ] SHOW INDEX
[x] EXPLAIN ANALYZE
[ ] DESCRIBE
> `EXPLAIN ANALYZE` показывает план выполнения и фактическое время. `Index Scan` — индекс работает, `Seq Scan` на большой таблице с узким фильтром — нет.
---
На крошечной таблице-справочнике из 50 строк индекс по столбцу фильтра обычно...
[x] почти не даёт выигрыша — full scan и так дёшев
[ ] ускоряет запрос в тысячи раз
[ ] обязателен, иначе запрос не выполнится
> Выигрыш индекса растёт с размером таблицы (логарифмическая глубина дерева). На 50 строках перебрать всё дешевле, чем ходить по дереву.
```
