# M4.8 · Оконные функции — самое важное для аналитики

Это водораздел между джуном и сильным аналитиком. Если освоишь оконные функции — сможешь считать когорты, retention, нарастающие итоги, ранги и "разницу с предыдущим днём" без боли.

Интуиция. Обычный GROUP BY схлопывает группу строк в одну. Оконная функция считает агрегат по группе, но не схлопывает строки — она дописывает результат к каждой строке как новый столбец. Представь: ты смотришь на каждую ставку и одновременно видишь "а какая по счёту эта ставка у этого игрока" и "сколько он суммарно поставил к этому моменту". Строки остаются на месте, рядом появляется контекст.

Формально. Синтаксис: `функция() OVER (PARTITION BY ... ORDER BY ...)`.

- `PARTITION BY` — на какие группы (окна) бить данные. Аналог GROUP BY, но без схлопывания.
- `ORDER BY` внутри OVER — порядок внутри окна, важен для ранжирования и нарастающих итогов.

Ключевые функции:

- `ROW_NUMBER()` — порядковый номер строки в окне: 1, 2, 3, 4. Уникальный всегда.
- `RANK()` — ранг с пропусками при равенстве: 1, 2, 2, 4.
- `DENSE_RANK()` — ранг без пропусков: 1, 2, 2, 3.
- `LAG(x)` / `LEAD(x)` — значение из предыдущей / следующей строки окна. Незаменимы для "разница со вчера".
- `SUM(x) OVER (... ORDER BY ...)` — нарастающий итог (running total).

```sql
SELECT player_id,
       deposit_ts,
       amount,
       ROW_NUMBER() OVER (PARTITION BY player_id ORDER BY deposit_ts) AS deposit_no,
       SUM(amount)  OVER (PARTITION BY player_id ORDER BY deposit_ts) AS cumulative_amount,
       LAG(deposit_ts) OVER (PARTITION BY player_id ORDER BY deposit_ts) AS prev_deposit_ts
FROM deposits;
```

В одном проходе мы получили: номер депозита у игрока (`deposit_no = 1` — это FTD, первый депозит!), накопленную сумму к этому моменту и время предыдущего депозита (чтобы посчитать интервал между ними). Без оконных функций каждое из этих чисел требовало бы отдельного самосоединения таблицы — медленно и нечитаемо.

Численный пример «на пальцах». Возьмём одного игрока `player_id = 42` с тремя депозитами:

| deposit_ts | amount | deposit_no | cumulative_amount | prev_deposit_ts |
|------------|--------|------------|-------------------|-----------------|
| 2026-01-01 | \$50   | 1          | \$50              | NULL            |
| 2026-01-04 | \$30   | 2          | \$80              | 2026-01-01      |
| 2026-01-10 | \$120  | 3          | \$200             | 2026-01-04      |

Что важно заметить:

- `deposit_no = 1` — это и есть FTD. У первой строки `prev_deposit_ts = NULL`, потому что предыдущей строки в окне нет. `LAG` всегда возвращает `NULL` на границе окна (или значение по умолчанию, если указать `LAG(x, 1, default)`).
- `cumulative_amount` растёт строка за строкой именно потому, что внутри `OVER` стоит `ORDER BY deposit_ts`. Без `ORDER BY` функция `SUM() OVER (PARTITION BY player_id)` посчитала бы общую сумму по всему игроку (\$200 во всех трёх строках) — это типичная ошибка. Наличие `ORDER BY` превращает агрегат в нарастающий итог.
- Интервал между депозитами считается как `deposit_ts - prev_deposit_ts`: 3 дня, затем 6 дней. Это сырьё для метрики «среднее время между депозитами» и для anti-churn триггеров.

Где порядок вычисления в запросе. Оконные функции считаются на стадии `SELECT` — после `WHERE`, `GROUP BY` и `HAVING`, но до `ORDER BY` и `LIMIT` всего запроса. Практическое следствие: нельзя фильтровать по результату оконной функции в том же `WHERE` (`WHERE deposit_no = 1` упадёт с ошибкой — столбца ещё не существует). Решение — обернуть запрос в подзапрос или CTE и фильтровать снаружи:

```sql
WITH numbered AS (
    SELECT player_id, deposit_ts, amount,
           ROW_NUMBER() OVER (PARTITION BY player_id ORDER BY deposit_ts) AS deposit_no
    FROM deposits
)
SELECT * FROM numbered WHERE deposit_no = 1;   -- только FTD каждого игрока
```

Это самый частый паттерн «оставить одну строку на группу»: пронумеровал внутри окна, отфильтровал снаружи. Так же берут «последний депозит» (`ORDER BY deposit_ts DESC`, затем `deposit_no = 1`) или «топ-3 ставки игрока».

Типичные ошибки и пограничные случаи:

- Забыли `ORDER BY` внутри `OVER` у `SUM`/`ROW_NUMBER`/`LAG` — для ранжирования и running total порядок обязателен, иначе результат либо бессмысленный, либо недетерминированный.
- Поставили `ORDER BY` без уникального ключа: при равных `deposit_ts` порядок строк не гарантирован, и `ROW_NUMBER` может «прыгать» между запусками. Добавляй tie-breaker: `ORDER BY deposit_ts, deposit_id`.
- Перепутали `RANK` и `DENSE_RANK` при ограничении «топ-N»: если нужны ровно N строк — берут `ROW_NUMBER`; если «все, кто на призовых местах с учётом ничьих» — `RANK`/`DENSE_RANK`.
- `LAG`/`LEAD` без обработки `NULL` на границе: при вычислении `amount - LAG(amount)` первая строка даст `NULL`. Это нормально, но в агрегатах потом учитывай (`AVG` игнорирует `NULL`, а `SUM` тоже, но смысл проверь).

??? example "Retention через оконные функции и self-join по дням"

    Классическая задача: D1/D7/D30 retention. Сначала находим дату регистрации (день 0) и все дни активности игрока, затем считаем разницу в днях и смотрим, есть ли активность на нужном лаге.

    ```sql
    WITH activity AS (
        SELECT player_id,
               bet_ts::date AS active_day
        FROM bets
        GROUP BY player_id, bet_ts::date
    ),
    cohort AS (
        SELECT player_id, MIN(active_day) AS cohort_day
        FROM activity
        GROUP BY player_id
    )
    SELECT c.cohort_day,
           COUNT(DISTINCT c.player_id) AS cohort_size,
           COUNT(DISTINCT CASE WHEN a.active_day - c.cohort_day = 1
                               THEN a.player_id END) AS d1_retained,
           COUNT(DISTINCT CASE WHEN a.active_day - c.cohort_day = 7
                               THEN a.player_id END) AS d7_retained
    FROM cohort c
    LEFT JOIN activity a ON a.player_id = c.player_id
    GROUP BY c.cohort_day
    ORDER BY c.cohort_day;
    ```

    D1 retention для когорты — это `d1_retained / cohort_size`. Делим в внешнем слое или в питоне.

!!! question "Проверь себя"

    1. Чем оконная функция отличается от обычного `GROUP BY` по поведению со строками?
    2. За что отвечают `PARTITION BY` и `ORDER BY` внутри `OVER (...)`?
    3. Чем различаются `ROW_NUMBER()`, `RANK()` и `DENSE_RANK()` на наборе равных значений?
    4. Как с помощью `ROW_NUMBER()` найти первый депозит (FTD) каждого игрока?

??? success "Ответы"

    1. `GROUP BY` схлопывает группу строк в одну; оконная функция считает агрегат по окну, но строки сохраняет, дописывая результат отдельным столбцом.
    2. `PARTITION BY` задаёт окна (на какие группы бить данные), `ORDER BY` внутри `OVER` — порядок строк в окне, нужный для рангов и нарастающих итогов.
    3. На значениях-дубликатах: `ROW_NUMBER` даёт 1,2,3,4 (всегда уникально); `RANK` — 1,2,2,4 (с пропуском); `DENSE_RANK` — 1,2,2,3 (без пропуска).
    4. `ROW_NUMBER() OVER (PARTITION BY player_id ORDER BY deposit_ts)` и взять строки с номером 1.

!!! example "Мини-задание"

    Напиши 3-5 строк SQL: с помощью `ROW_NUMBER() OVER (PARTITION BY player_id ORDER BY deposit_ts)` пометь номер каждого депозита и оставь только первый депозит каждого игрока (`deposit_no = 1`). Проверь, что получилась ровно одна строка на игрока.

!!! note "Frame — продвинутый контроль окна"

    У оконного агрегата есть третья часть после `ORDER BY` — рамка (frame), которая задаёт, какие строки внутри окна реально участвуют в расчёте. По умолчанию для `SUM() OVER (... ORDER BY ...)` рамка равна `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` — то есть «от начала окна до текущей строки», поэтому и получается нарастающий итог.

    Рамку можно сузить. Например, скользящее среднее GGR по 7 дням:

    ```sql
    SELECT bet_day,
           daily_ggr,
           AVG(daily_ggr) OVER (
               ORDER BY bet_day
               ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
           ) AS ggr_ma7
    FROM daily_ggr;
    ```

    `ROWS BETWEEN 6 PRECEDING AND CURRENT ROW` берёт текущую строку плюс 6 предыдущих — окно из 7 строк, которое скользит вниз. Это рабочая лошадка для сглаживания «дёрганых» дневных метрик в дашбордах. Помни отличие `ROWS` (считает физические строки) от `RANGE` (считает строки с тем же значением `ORDER BY`) — на данных с дубликатами по дате они дают разный результат.

```quiz

> GROUP BY схлопывает группу в одну строку, а оконная функция оставляет все строки и добавляет результат рядом.

На наборе равных значений ROW_NUMBER, RANK и DENSE_RANK дают соответственно:
```
