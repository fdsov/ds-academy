Не понимаешь NumPy — не понимаешь, что происходит внутри.

Что сломается без этого знания. Ты будешь писать медленный код и не понимать, почему он медленный. Ты будешь получать загадочные ошибки `ValueError: operands could not be broadcast together`. ## M5.3 · Почему NumPy быстрый: модель памяти

### Интуиция

Обычный Python-список — это коробка с указателями. Когда ты пишешь `[1, 2, 3]`, в памяти лежат не сами числа, а три «адреса», по которым разбросаны объекты-числа. Каждое число — это полноценный Python-объект с заголовком, типом, счётчиком ссылок. Чтобы сложить два списка поэлементно, интерпретатор для каждой пары проходит путь: достать указатель, найти объект, проверить его тип, вызвать метод сложения, создать новый объект. Это десятки машинных операций на одно сложение.

NumPy-массив — это плотная непрерывная лента байтов. Миллион чисел `float64` лежат подряд, ровно по 8 байт каждое, без заголовков и указателей. Тип у всех один и записан один раз в метаданных массива. Поэтому сложение двух массивов превращается в тугой цикл на C, который процессор может ещё и векторизовать на уровне SIMD-инструкций (одна инструкция обрабатывает 4-8 чисел сразу). Никакой проверки типов в цикле, никакого создания объектов.

### Формально

`ndarray` (N-dimensional array) — это объект, описывающий непрерывный блок памяти плюс набор метаданных:

- `dtype` — тип элемента (например `int64`, `float64`, `bool`), фиксирован для всего массива;
- `shape` — кортеж размеров по каждой оси, например `(1000, 3)` для 1000 строк и 3 столбцов;
- `strides` — сколько байт перешагнуть в памяти, чтобы сдвинуться на единицу вдоль каждой оси.

Эта тройка позволяет интерпретировать одну и ту же ленту байтов как массив любой формы. Адрес элемента $A[i, j]$ вычисляется арифметически:

$$
\text{offset}(i, j) = i \cdot s_0 + j \cdot s_1
$$

где $s_0, s_1$ — strides. Именно благодаря strides многие операции (транспонирование, срезы) не копируют данные, а лишь меняют метаданные — об этом ниже в разделе про views.

!!! info "Цена однородности"

    Фиксированный dtype — это и сила, и ограничение. Массив не может одновременно хранить числа и строки эффективно. Если положить разнородные данные, NumPy либо приведёт всё к общему типу, либо создаст массив `dtype=object` — и тогда ты теряешь всю производительность, получая по сути обёрнутый список указателей.

!!! question "Проверь себя"

    1. Почему миллион чисел в NumPy-массиве занимает меньше памяти и обрабатывается быстрее, чем тот же список Python?
    2. Какие три метаданных вместе со ссылкой на блок памяти описывают `ndarray`?
    3. Зачем нужны `strides` и почему благодаря им транспонирование не копирует данные?
    4. Что произойдёт с производительностью, если положить в массив разнородные данные?

??? success "Ответы"

    1. В списке лежат указатели на отдельные объекты-числа с заголовками; в массиве числа лежат подряд плотной лентой с единым dtype, и цикл идёт на C с SIMD без проверки типов в цикле.
    2. `dtype`, `shape`, `strides`.
    3. `strides` — число байт для сдвига на единицу вдоль каждой оси; адрес элемента вычисляется арифметически, поэтому смена формы или порядка осей меняет лишь метаданные, не копируя байты.
    4. NumPy приведёт всё к общему типу либо создаст `dtype=object` (массив указателей), и производительность падает до уровня обычного списка.

## M5.4 · ndarray, dtype, shape

### dtype: почему это важно для денег

dtype определяет не только скорость, но и корректность. В гемблинг-аналитике это критично: депозиты, ставки, балансы — это деньги, и ошибка типа здесь стоит дорого.

`float64` (он же `double`) даёт примерно 15-16 значащих десятичных цифр. Для сумм до миллионов это надёжно. Но `float32` экономит память вдвое ценой всего ~7 цифр точности — складывая миллионы мелких ставок в `float32`, ты накопишь ощутимую ошибку округления. Для матриц признаков в ML `float32` часто оправдан (память и скорость GPU важнее), для финансового свода — нет.

Целочисленные типы могут переполняться молча. `int32` держит значения примерно до $2.1 \times 10^9$. Если ты считаешь суммарный объём ставок (bet volume) в копейках за год по крупному казино, ты легко это превысишь, и сумма «завернётся» в отрицательное число без всякого предупреждения.

```python
import numpy as np

a = np.array([2_000_000_000], dtype=np.int32)
print(a + a)  # [-294967296] — тихое переполнение

b = np.array([2_000_000_000], dtype=np.int64)
print(b + b)  # [4000000000] — корректно
```

### shape и количество осей

`shape` — кортеж. Скаляр имеет shape `()`, вектор `(n,)`, матрица `(rows, cols)`, тензор `(d0, d1, d2)` и так далее. Число осей называется `ndim`. Важно различать `(n,)` (одномерный вектор) и `(n, 1)` или `(1, n)` (двумерные «столбец» и «строка») — они ведут себя по-разному в broadcasting и матричном умножении, и это источник вечной путаницы.

```python
x = np.array([1.5, 2.0, 3.0])
print(x.shape, x.dtype, x.ndim)   # (3,) float64 1
print(x.size, x.nbytes)           # 3 24  — 3 элемента по 8 байт
```

!!! question "Проверь себя"

    1. Почему `float32` опасен для финансового свода, но приемлем для матрицы признаков в ML?
    2. Что происходит при переполнении `int32` и почему это особенно коварно?
    3. Чем различаются формы `(n,)`, `(n, 1)` и `(1, n)`?

??? success "Ответы"

    1. `float32` даёт лишь ~7 значащих цифр — при суммировании миллионов мелких сумм копится ошибка округления; в ML важнее память и скорость GPU, а такая точность не критична.
    2. Значение «заворачивается» в отрицательное без всякого предупреждения (тихое переполнение) — ошибку легко не заметить.
    3. `(n,)` — одномерный вектор; `(n, 1)` — двумерный столбец; `(1, n)` — двумерная строка. Они по-разному ведут себя в broadcasting и матричном умножении.

!!! example "Мини-задание"

## M5.5 · Создание массивов

```python
np.array([1, 2, 3])                 # из списка
np.zeros((3, 4))                    # нули, форма (3,4)
np.ones(5)                          # единицы
np.full((2, 2), 7.0)                # заполнить значением
np.arange(0, 10, 2)                 # [0 2 4 6 8] — как range, но массив
np.linspace(0, 1, 5)                # 5 точек равномерно от 0 до 1 включительно
np.eye(3)                           # единичная матрица 3x3
np.empty((2, 3))                    # без инициализации (быстро, но мусор внутри)
```

!!! warning "arange и дробный шаг"

    `np.arange` с дробным шагом (`np.arange(0, 1, 0.1)`) может из-за накопления ошибок float дать на один элемент больше или меньше, чем ты ждёшь. Когда нужно точное число точек на отрезке — используй `np.linspace`, где ты задаёшь количество, а не шаг.

## M5.6 · Индексация, срезы, маски

### Базовые срезы

Синтаксис как у списков, но по каждой оси через запятую. `A[i, j]` — элемент, `A[i]` — целая строка, `A[:, j]` — целый столбец.

```python
A = np.arange(12).reshape(3, 4)
A[1, 2]      # 6 — строка 1, столбец 2
A[0]         # [0 1 2 3] — первая строка
A[:, 1]      # [1 5 9] — второй столбец
A[0:2, 1:3]  # подматрица
```

Важнейший факт: базовый срез возвращает **view**, а не копию. Изменение среза меняет исходный массив. Это отличается от поведения списков Python, где срез копирует.

### Boolean-маски — главный инструмент аналитика

#### Интуиция

Маска — это массив из True/False той же длины, который говорит «этот элемент берём, этот нет». Сравнение массива с числом даёт сразу всю маску, без цикла. Это твой основной способ фильтровать данные: «все депозиты больше 10000», «все игроки с retention выше медианы».

#### Формально

Выражение `arr > 100` применяет оператор поэлементно и возвращает массив `dtype=bool` той же формы. Индексация массивом-маской `arr[mask]` возвращает одномерный массив из элементов, где маска True (всегда копия, не view). Условия комбинируются поэлементными операторами `&` (и), `|` (или), `~` (не) — обязательно в скобках, потому что приоритет операторов в Python отдаёт `&` ниже, чем сравнения.

```python
deposits = np.array([50, 15000, 300, 99000, 20])

big = deposits > 10_000           # [False True False True False]
deposits[big]                     # [15000 99000]

mask = (deposits > 100) & (deposits < 50_000)
deposits[mask]                    # [15000 300]

# подсчёт и доля без цикла
big.sum()                         # 2 — сколько крупных депозитов
big.mean()                        # 0.4 — их доля
```

Последняя строка — частый идиоматический приём: для boolean-массива `.mean()` даёт долю True, потому что True считается за 1, False за 0. «Какая доля игроков сделала повторный депозит» — это одна строка.

### Fancy indexing

Индексация массивом целочисленных индексов выбирает элементы в произвольном порядке и с повторами. Возвращает копию.

```python
prices = np.array([10, 20, 30, 40, 50])
idx = np.array([0, 2, 2, 4])
prices[idx]                       # [10 30 30 50]

# в двух измерениях — пары координат
A = np.arange(12).reshape(3, 4)
A[[0, 1, 2], [1, 2, 3]]           # [1 6 11] — элементы (0,1),(1,2),(2,3)
```

!!! question "Проверь себя"

    1. Базовый срез массива возвращает view или копию? Чем это отличается от срезов списков Python?
    2. Что вернёт `mask.mean()` для boolean-маски и почему?
    3. Почему условия в маске нужно оборачивать в скобки: `(a > 1) & (b < 2)`?
    4. Fancy indexing и boolean-маска возвращают view или копию?

??? success "Ответы"

    1. View — окно в ту же память; изменение среза меняет оригинал. Списки Python при срезе копируют.
    2. Долю элементов True: True считается за 1, False за 0, поэтому среднее равно доле True.

## M5.7 · Векторизация: почему циклы медленны

### Интуиция

Цикл Python выполняется интерпретатором: на каждой итерации он разбирает байткод, проверяет типы, управляет объектами. Векторизованная операция спускает весь цикл в скомпилированный код C, который работает с плотной памятью. Разница принципиальна — это не «оптимизация», а смена слоя исполнения.

### Бенчмарк

Посчитаем GGR как «ставки минус выплаты» по миллиону записей двумя способами.

```python
import numpy as np
import time

n = 1_000_000
rng = np.random.default_rng(42)
bets = rng.exponential(100, n)
payouts = rng.exponential(95, n)

# Способ 1: чистый Python-цикл
t0 = time.perf_counter()
ggr_loop = [bets[i] - payouts[i] for i in range(n)]
t_loop = time.perf_counter() - t0

# Способ 2: векторизация NumPy
t0 = time.perf_counter()
ggr_vec = bets - payouts
t_vec = time.perf_counter() - t0

print(f"loop: {t_loop*1000:.1f} ms")
print(f"vec:  {t_vec*1000:.1f} ms")
print(f"ускорение: {t_loop / t_vec:.0f}x")
```

На типичной машине цикл занимает ~150-300 мс, векторизация — ~2-5 мс. Ускорение в 50-100 раз. И это простейшая операция; на сложных выражениях разрыв ещё больше, потому что промежуточные результаты тоже остаются в C.

!!! tip "Правило большого пальца"

    Если ты пишешь `for` по элементам NumPy-массива — почти наверняка ты делаешь что-то не так. Спроси себя: можно ли выразить это через поэлементные операции, маски, агрегации или broadcasting? В 95% случаев — да.

## M5.8 · Broadcasting

### Интуиция

Broadcasting — это правила, по которым NumPy «растягивает» массивы разной формы, чтобы операция стала поэлементной, не копируя данные физически. Простейший пример: `array * 2` умножает каждый элемент на 2 — скаляр «растягивается» на всю форму. То же работает и между массивами: вектор может растянуться вдоль матрицы.

Практический смысл: нормировать матрицу признаков по столбцам, вычесть среднее из каждой строки, применить курс валют к таблице сумм — всё без циклов.

### Формально

Две формы совместимы, если, выравнивая их **справа налево**, по каждой оси выполняется хотя бы одно: размеры равны, либо один из них равен 1. Ось размера 1 «растягивается» до размера партнёра. Недостающие оси слева достраиваются как 1.

$$
(3, 4) \;\text{и}\; (4,) \rightarrow (3, 4) \;\text{и}\; (1, 4) \rightarrow (3, 4)
$$

$$
(3, 1) \;\text{и}\; (1, 4) \rightarrow (3, 4)
$$

Несовместимо, например, `(3,)` и `(4,)` — размеры не равны и ни один не 1.

```python
# нормировка z-score по столбцам матрицы признаков
X = rng.normal(size=(1000, 5))
mu = X.mean(axis=0)        # shape (5,) — среднее по каждому столбцу
sigma = X.std(axis=0)      # shape (5,)
X_norm = (X - mu) / sigma  # (1000,5) - (5,) -> broadcast по строкам

# вычесть среднее ИЗ КАЖДОЙ строки нужен столбец (n,1)
row_mean = X.mean(axis=1, keepdims=True)  # shape (1000,1)
X_centered = X - row_mean                 # broadcast по столбцам
```

Обрати внимание на `keepdims=True` — он сохраняет ось как размер 1 вместо её удаления, и тогда broadcasting срабатывает в нужном направлении. Без него `X.mean(axis=1)` даст shape `(1000,)`, который выровняется справа к столбцам и сломает логику.

!!! question "Проверь себя"

    1. Сформулируй правило совместимости форм при broadcasting.
    2. Совместимы ли `(3, 4)` и `(4,)`? А `(3,)` и `(4,)`?
    3. Зачем нужен `keepdims=True` при вычитании среднего из каждой строки матрицы?

??? success "Ответы"

    1. Выравнивая формы справа налево, по каждой оси либо размеры равны, либо один из них равен 1 (тогда он растягивается до размера партнёра); недостающие оси слева достраиваются как 1.
    2. `(3, 4)` и `(4,)` совместимы → `(3, 4)`. `(3,)` и `(4,)` несовместимы: размеры не равны и ни один не равен 1.
    3. Он сохраняет схлопнутую ось как размер 1: `X.mean(axis=1, keepdims=True)` даёт `(n, 1)`, который растягивается по столбцам. Без него результат `(n,)` выровняется справа к столбцам и посчитает не то.

!!! example "Мини-задание"

    Возьми `X = rng.normal(size=(5, 3))`. Сделай z-score нормировку по столбцам через broadcasting и проверь, что у результата по каждому столбцу среднее близко к 0, а std близко к 1 (`X_norm.mean(axis=0)`, `X_norm.std(axis=0)`). Затем центрируй матрицу по строкам с `keepdims=True` и сравни формы промежуточных массивов.

## M5.9 · Агрегации по осям

### Интуиция

`axis` отвечает на вопрос «вдоль какой оси схлопываем». Это самое контринтуитивное место для новичков. Запомни так: `axis` указывает ось, которая **исчезает**. `axis=0` схлопывает строки — остаётся результат по столбцам. `axis=1` схлопывает столбцы — остаётся по строкам.

### Пример: таблица ставок

Пусть строки — игроки, столбцы — дни недели, значения — суммы ставок.

```python
bets = rng.exponential(100, size=(4, 7))   # 4 игрока, 7 дней

bets.sum()           # один скаляр — всего по всем
bets.sum(axis=0)     # shape (7,) — сумма по дням (игроки схлопнуты)
bets.sum(axis=1)     # shape (4,) — сумма по игрокам (дни схлопнуты)
bets.max(axis=1)     # макс ставка каждого игрока
bets.argmax(axis=1)  # в какой день у каждого был максимум
```

Доступны `sum`, `mean`, `std`, `var`, `min`, `max`, `argmin`, `argmax`, `prod`, `cumsum`, `median`, `percentile` и другие. У всех единая логика осей.

!!! question "Проверь себя"

    1. Как одной фразой запомнить, что делает `axis`?
    2. Для матрицы `(4 игрока, 7 дней)` какой `axis` даёт сумму по дням, а какой — по игрокам? Какие формы у результатов?
    3. Что по смыслу вернёт `bets.argmax(axis=1)`?

??? success "Ответы"

    1. `axis` указывает ось, которая исчезает (схлопывается).
    2. `axis=0` схлопывает игроков → сумма по дням, форма `(7,)`. `axis=1` схлопывает дни → сумма по игрокам, форма `(4,)`.
    3. Для каждого игрока — индекс дня, в который у него была максимальная ставка.

## M5.10 · Изменение формы: reshape, transpose, concatenate, stack

```python
a = np.arange(12)
a.reshape(3, 4)       # та же память, новая форма (view)
a.reshape(3, -1)      # -1 = "посчитай сам" -> (3,4)
a.reshape(2, 2, 3)    # трёхмерный

M = np.arange(6).reshape(2, 3)
M.T                   # транспонирование (2,3)->(3,2), view, меняет strides

x = np.array([1, 2, 3])
y = np.array([4, 5, 6])
np.concatenate([x, y])            # [1 2 3 4 5 6]
np.stack([x, y])                  # (2,3) новая ось
np.stack([x, y], axis=1)          # (3,2)
np.vstack([x, y])                 # вертикально -> (2,3)
np.hstack([x, y])                 # горизонтально -> (6,)
```

`reshape` требует, чтобы общее число элементов совпадало. `-1` по одной оси означает «вычисли автоматически». `concatenate` соединяет вдоль существующей оси, `stack` создаёт новую ось.

## M5.11 · Универсальные функции (ufuncs)

ufunc — это векторизованная функция, применяемая поэлементно в скомпилированном коде: `np.exp`, `np.log`, `np.sqrt`, `np.sin`, `np.abs`, `np.maximum`, `np.where` и десятки других. Они и есть носители скорости NumPy.

```python
ltv = np.array([120, 0, 450, 30])
np.log1p(ltv)                      # log(1+x), безопасно для нуля
np.sqrt(ltv)
np.clip(ltv, 0, 200)               # обрезать выбросы сверху
np.where(ltv > 100, "vip", "reg")  # векторный if-else
np.maximum(ltv, 50)                # поэлементный максимум с порогом
```

`np.where(condition, a, b)` — векторная замена `if`: где условие True берётся `a`, иначе `b`. Это рабочая лошадь для создания сегментов и флагов без цикла.

## M5.12 · Линейная алгебра

### Интуиция

Скалярное произведение, матричное умножение — основа любого ML. Логистическая регрессия для скоринга оттока, эмбеддинги, ковариационные матрицы — всё это матричные операции. NumPy даёт оператор `@` специально для матричного умножения.

### Формально

Для матриц $A$ размера $(m, k)$ и $B$ размера $(k, n)$ произведение $C = A B$ имеет размер $(m, n)$:

$$
C_{ij} = \sum_{p=1}^{k} A_{ip} B_{pj}
$$

Внутреннее измерение должно совпадать ($k = k$).

```python
A = rng.normal(size=(3, 4))
B = rng.normal(size=(4, 2))
A @ B                 # (3,2) — оператор matmul, предпочтительный
np.dot(A, B)          # то же
np.matmul(A, B)       # то же

v = np.array([1.0, 2.0, 3.0])
v @ v                 # 14.0 — скалярное произведение (sum квадратов)

# линейная модель: предсказание = X @ weights + bias
X = rng.normal(size=(100, 5))
w = rng.normal(size=5)
preds = X @ w + 0.5   # (100,) предсказания для 100 объектов
```

Модуль `np.linalg` содержит `inv` (обратная), `solve` (решение системы), `det`, `eig` (собственные значения), `svd` (сингулярное разложение), `norm`. Для решения систем всегда предпочитай `np.linalg.solve(A, b)` вместо `np.linalg.inv(A) @ b` — это точнее и быстрее.

## M5.13 · Случайность: современный Generator API

### Что устарело

Старый интерфейс `np.random.seed(42)` плюс `np.random.rand()` — это глобальное состояние. Оно считается legacy с NumPy 1.17 (2019) и категорически не рекомендуется в 2026: глобальный seed создаёт скрытые зависимости между частями кода, плохо работает в параллельных процессах и затрудняет воспроизводимость.

### Современный подход

`np.random.default_rng(seed)` создаёт изолированный объект-генератор `Generator` на базе алгоритма PCG64. Вся случайность идёт через него. Никакого глобального состояния — каждый генератор независим.

```python
rng = np.random.default_rng(seed=42)

rng.random(5)                       # равномерно [0,1)
rng.integers(0, 100, size=10)       # целые в [0,100)
rng.normal(loc=0, scale=1, size=5)  # нормальное
rng.exponential(scale=100, size=5)  # экспоненциальное (хорошо для сумм ставок)
rng.choice(["A", "B"], size=10, p=[0.7, 0.3])  # категории с вероятностями
rng.shuffle(arr)                    # перемешать на месте

# воспроизводимое A/B-разбиение игроков
players = np.arange(10_000)
rng = np.random.default_rng(2026)
groups = rng.integers(0, 2, size=players.size)  # 0=control, 1=treatment
```

!!! warning "Воспроизводимость экспериментов"

    В дизайне A/B (см. модуль про эксперименты) seed фиксирует разбиение и симуляции. Но seed гарантирует одинаковость только в рамках одной версии NumPy и одной платформы — не считай его криптографической гарантией. Для продакшн-сплита пользователей используй детерминированный хэш user_id, а не RNG.

!!! question "Проверь себя"

    1. Чем `np.random.default_rng` лучше старого `np.random.seed` + `np.random.rand`?
    2. Какой алгоритм лежит в основе `Generator` и почему важна изолированность генератора?
    3. Почему для продакшн-сплита пользователей в A/B лучше детерминированный хэш `user_id`, а не RNG?

??? success "Ответы"

    1. Нет глобального состояния: каждый генератор изолирован, нет скрытых зависимостей между частями кода, надёжнее работа в параллельных процессах и воспроизводимость.
    2. PCG64. Изолированность означает, что случайность в одной части кода не влияет на другую — нет общего глобального seed.
    3. seed гарантирует одинаковость лишь в рамках одной версии NumPy и одной платформы; хэш `user_id` даёт стабильное и переносимое разбиение, не зависящее от порядка вызовов RNG.

## M5.14 · Работа с NaN

### Интуиция

NaN (Not a Number) — это специальное значение float, означающее «пропуск/неопределённость». В реальных данных оно вездесуще: игрок не сделал депозит, метрика не посчиталась, джойн не нашёл пару. NaN «заразен»: любая арифметика с ним даёт NaN, и обычные `sum`/`mean` тоже вернут NaN, если в данных есть хоть один пропуск.

### Формально

NaN определён стандартом IEEE 754 и обладает странным свойством: он не равен сам себе. Поэтому `nan == nan` — это False, и проверять надо через `np.isnan`.

```python
x = np.array([10.0, np.nan, 30.0, np.nan, 50.0])

x.mean()                # nan — заражено
np.isnan(x)             # [False True False True False]
x[~np.isnan(x)]         # [10. 30. 50.] — отфильтровать пропуски

np.nanmean(x)           # 30.0 — среднее, игнорируя NaN
np.nansum(x)            # 90.0
np.nanstd(x), np.nanmax(x)

# заполнить пропуски значением
np.where(np.isnan(x), 0.0, x)   # [10. 0. 30. 0. 50.]
```

Существует семейство `nan`-агрегаций: `nanmean`, `nansum`, `nanstd`, `nanmax`, `nanmin`, `nanmedian`. Используй их, когда пропуски ожидаемы и их нужно игнорировать, но всегда осознанно — иногда NaN означает «надо разобраться, а не замолчать».

!!! note "NaN только во float"

    NaN существует только в типах с плавающей точкой. У целочисленных массивов NaN нет. Поэтому если в колонке возможны пропуски, она будет float (или object). pandas в M9 решает это через nullable-типы и `pd.NA`, но базовый NumPy NaN живёт лишь во float.

## M5.15 · Views против copies и память

### Интуиция

Это место, где горят даже опытные. View — это «окно» в чужую память: другой объект-массив, но та же лента байтов. Изменишь view — изменится оригинал. Copy — независимый дубликат. Базовые срезы, `reshape`, `transpose` обычно дают view (быстро, без аллокации). Fancy indexing и boolean-маски дают copy. Перепутаешь — либо тихо испортишь исходные данные, либо удивишься, что твоё изменение «не применилось».

### Формально

View возможен, когда новый массив выражается через те же данные сдвигом offset и пересчётом strides. Копия неизбежна, когда требуемые элементы нельзя описать регулярным шагом в памяти (именно так с fancy indexing и масками).

```python
a = np.arange(10)
s = a[2:5]            # view
s[0] = 999
a                     # [0 1 999 3 4 5 6 7 8 9] — оригинал изменился!

f = a[[2, 3, 4]]      # fancy indexing -> copy
f[0] = -1
a                     # оригинал НЕ изменился

# проверить, делит ли память
np.shares_memory(a, s)   # True
np.shares_memory(a, f)   # False

# явная копия, когда нужна независимость
clean = a[2:5].copy()
```

Практический совет: если ты собираешься модифицировать срез исходного датасета и не хочешь трогать оригинал — вызови `.copy()` явно. И наоборот, если работаешь с гигабайтами и хочешь избежать лишних аллокаций — пользуйся views и in-place операциями (`arr += 1`, `out=`-аргумент ufuncs).

!!! question "Проверь себя"

    1. Какие операции обычно дают view, а какие — копию?
    2. Как программно проверить, делят ли два массива одну память?
    3. Ты хочешь модифицировать срез датасета, не трогая оригинал. Что сделать?

??? success "Ответы"

    1. View: базовые срезы, `reshape`, `transpose`. Копия: fancy indexing и boolean-маски.
    2. `np.shares_memory(a, b)` — вернёт `True`, если массивы делят память.
    3. Сделать явную копию среза: `a[2:5].copy()`.

## M5.16 · Код: стек 2026

```python
import numpy as np

# NumPy 2.x — актуальная мажорная линейка на 2026.
# Изолированный генератор вместо legacy np.random.seed.
rng = np.random.default_rng(2026)

# Синтетический лог: 1 млн ставок
n = 1_000_000
player_id = rng.integers(0, 50_000, size=n)
bet = rng.exponential(100, size=n)
win = bet * rng.random(n) * 2          # выплата как доля ставки
is_fraud_flag = rng.random(n) < 0.001  # редкие фрод-метки

# GGR по каждой ставке (доход казино) — векторизованно
ggr = bet - win

# Антифрод-эвристика без цикла: крупная ставка + выигрыш > 1.9x
suspicious = (bet > np.percentile(bet, 99)) & (win > bet * 1.9)
print("подозрительных ставок:", suspicious.sum())

# GGR по игрокам через bincount (быстрая группировка по целому ключу)
ggr_by_player = np.bincount(player_id, weights=ggr, minlength=50_000)
top10 = np.argsort(ggr_by_player)[-10:][::-1]
print("топ-10 игроков по GGR:", top10)

# Доля прибыльных игроков
print("доля прибыльных:", (ggr_by_player > 0).mean().round(3))

# Z-score нормировка метрик игроков (broadcast)
metrics = np.stack([
    np.bincount(player_id, weights=bet, minlength=50_000),
    np.bincount(player_id, minlength=50_000).astype(float),
    ggr_by_player,
], axis=1)  # (50000, 3): объём, число ставок, GGR

mu = metrics.mean(axis=0)
sigma = metrics.std(axis=0)
metrics_z = (metrics - mu) / np.where(sigma == 0, 1, sigma)
print("форма матрицы признаков:", metrics_z.shape)
```

`np.bincount` с `weights` — это идиоматичный быстрый способ агрегировать по целочисленному ключу без pandas, идеален для группировки по player_id или дню.

!!! tip "AI-копилот: где нейросети помогают"

    NumPy — одна из тех областей, где LLM (Claude, Cursor, GitHub Copilot) сильны, потому что API стабилен и хорошо документирован. Реальная польза: быстро вспомнить нужную функцию, перевести медленный цикл в векторную форму, объяснить непонятную ошибку broadcasting.

    Пример хорошего промпта: «Вот функция на чистом Python с двойным циклом, считающая попарные расстояния между строками матрицы X формы (n, d). Перепиши её векторизованно на NumPy через broadcasting, без явных циклов, и объясни, какая форма получается на каждом промежуточном шаге.»

    Где LLM врут и что перепроверять обязательно:

    - **Axis-логика.** Модели регулярно путают `axis=0` и `axis=1`. Всегда проверяй форму результата руками на маленьком примере (`shape` до и после).
    - **View vs copy.** LLM может уверенно сказать, что операция возвращает копию, когда это view (или наоборот). Проверяй `np.shares_memory`, если от этого зависит корректность.
    - **Legacy random.** Модели по инерции пишут `np.random.seed`/`np.random.rand` из старых обучающих данных. Переписывай на `default_rng`.
    - **Тихие ошибки dtype.** LLM не подскажет про переполнение `int32` или потерю точности `float32` — это твоя ответственность, особенно с деньгами.
    - **Производительность.** Иногда «векторизованное» решение от LLM аллоцирует огромную промежуточную матрицу (например `(n, n)` для миллиона строк) и падает по памяти. Прикинь объём перед запуском.

!!! note "Стек 2026"

    Актуально:

    - **NumPy 2.x** — текущая мажорная линейка. В переходе с 1.x были breaking changes (изменения в правилах продвижения типов NEP 50, чистка публичного API, удаление устаревших алиасов вроде `np.float`, `np.int`). Если видишь старый код с `np.float` — это NumPy 1.x, в 2.x он удалён, пиши `float` или `np.float64`.
    - **`np.random.default_rng` / `Generator`** — единственный рекомендованный способ работы со случайностью.
    - **Оператор `@`** для матричного умножения — стандарт с Python 3.5, всегда предпочтительнее `np.dot` для читаемости.

    Устарело относительно старых курсов:

    - `np.random.seed` + функции верхнего уровня (`rand`, `randn`, `randint`) — legacy с 2019 года.
    - `np.matrix` — отдельный класс матриц, в статусе pending deprecation, никогда не используй, только `ndarray`.
    - Алиасы `np.int`, `np.float`, `np.bool` — удалены.

    Что рядом: для данных, которые не помещаются в память, смотри в сторону Dask и Polars; для GPU-ускорения той же NumPy-модели — CuPy и JAX. Но фундамент везде один — модель `ndarray`, которую ты учишь здесь.

!!! warning "Ловушки"

    - **Случайная порча данных через view.** Сделал срез, поменял его «для расчёта» — испортил исходный датасет. Если нужна независимость, `.copy()` явно.
    - **Путаница axis.** `axis` — это ось, которая исчезает, а не «по которой считаем в бытовом смысле». Проверяй форму результата.
    - **Broadcasting не туда.** `X - X.mean(axis=1)` падает или считает не то, потому что `(n,)` выравнивается к столбцам. Нужен `keepdims=True`, чтобы получить `(n, 1)`.
    - **Тихое переполнение int.** Суммы в копейках/центах за длинный период переполняют `int32`. Используй `int64` или работай во float с контролем точности.
    - **`==` для NaN.** `x == np.nan` всегда False. Только `np.isnan`.
    - **`&`/`|` без скобок.** `mask = a > 1 & b < 2` парсится как `a > (1 & b) < 2` из-за приоритета. Всегда `(a > 1) & (b < 2)`.
    - **Цикл по NumPy-массиву.** Если пишешь `for x in arr` для вычислений — почти всегда есть векторная замена в 50+ раз быстрее.
    - **Сеньорская ловушка: огромная промежуточная матрица.** «Элегантное» broadcasting-решение для попарных расстояний создаёт `(n, n)` — для $n = 10^6$ это $8 \times 10^{12}$ байт, мгновенный OOM. Векторизация не бесплатна по памяти, считай объём заранее.
    - **`float32` для денег.** Соблазн сэкономить память на финансовых суммах оборачивается ошибками округления при суммировании миллионов записей.

```quiz
Какие три метаданных вместе со ссылкой на блок памяти описывают ndarray?
[ ] size, ndim, nbytes
[x] dtype, shape, strides
[ ] index, columns, values
[ ] offset, length, capacity
> dtype задаёт тип элемента, shape — размеры по осям, strides — на сколько байт шагнуть вдоль каждой оси. Эта тройка позволяет интерпретировать одну ленту байтов как массив любой формы.
---
Почему float32 опасен для финансового свода, но приемлем для матрицы признаков в ML?
[ ] float32 не поддерживает отрицательные числа
[ ] float32 медленнее, чем float64, и поэтому копит ошибку
[x] float32 даёт лишь ~7 значащих цифр, и при суммировании миллионов сумм копится ошибка округления, а в ML важнее память и скорость GPU
[ ] float32 округляет все значения до целых
> float32 экономит память вдвое ценой точности (~7 цифр против ~15-16 у float64). Для финансовых сумм это накапливает ошибку, для матриц признаков в ML точность не критична, а память и скорость важны.
---
Что вернёт mask.mean() для boolean-маски и почему?
[x] Долю элементов True, потому что True считается за 1, а False за 0
[ ] Среднее значение исходного массива по элементам, где маска True
[ ] Всегда 0.5, так как маска состоит из True и False
[ ] Ошибку, потому что среднее не определено для bool
> Для boolean-массива True трактуется как 1, False как 0, поэтому среднее равно доле True. Это идиоматичный способ посчитать долю, например, повторных депозитов одной строкой.
---
Базовый срез NumPy-массива (например a[2:5]) возвращает view или копию?
[ ] Копию, как и срез списка Python
[x] View — окно в ту же память, изменение среза меняет оригинал
[ ] Зависит от dtype массива
[ ] View только для одномерных массивов, иначе копию
> Базовый срез возвращает view: новый объект-массив поверх той же ленты байтов. Изменение элементов среза меняет исходный массив — в отличие от среза списка Python, который копирует. Для независимости нужен .copy().
---
Зачем нужен keepdims=True при вычитании среднего из каждой строки матрицы?
[ ] Чтобы среднее считалось по столбцам, а не по строкам
[ ] Чтобы ускорить расчёт за счёт кэширования оси
[x] Чтобы сохранить схлопнутую ось как размер 1 и broadcasting сработал в нужном направлении
[ ] Чтобы результат остался целочисленным
> X.mean(axis=1, keepdims=True) даёт форму (n, 1), которая растягивается по столбцам. Без него результат (n,) выровняется справа к столбцам и посчитает не то.
---
Почему np.random.default_rng предпочтительнее старого np.random.seed + np.random.rand?
[ ] default_rng генерирует числа криптографически стойко
[ ] Старый интерфейс удалён в NumPy 2.x и больше не работает
[x] У default_rng нет глобального состояния: каждый генератор изолирован, нет скрытых зависимостей и надёжнее воспроизводимость
[ ] default_rng работает только с целыми числами
> Старый seed — это глобальное состояние, создающее скрытые зависимости между частями кода и проблемы в параллельных процессах. default_rng создаёт изолированный Generator (PCG64), каждый независим.
```
