## Что отрабатываем

Этот воркшоп закрывает разрыв между «прочитал формулу» и «вижу формулу за вызовом библиотеки». Модуль M3 утверждал четыре вещи, в которые легко поверить на словах, но которые становятся твоими, только когда ты подтвердил их числами:

- вектор как стрелка: норма, скалярное произведение, косинусная близость как мера схожести профилей игроков;
- матричное умножение «строка на столбец» как пакетный расчёт взвешенных сумм и как преобразование пространства (поворот обнуляет скалярное произведение);
- ранг и число обусловленности как диагностика мультиколлинеарности, из-за которой $X^\top X$ становится сингулярной;
- собственные значения и векторы ковариационной матрицы как сердце PCA;
- градиентный спуск $\mathbf{w}_{t+1} = \mathbf{w}_t - \eta\,\nabla L(\mathbf{w}_t)$, написанный руками, без `sklearn`, с правильным коэффициентом $2/n$ в градиенте MSE.

**Артефакт на выходе:** один скрипт (или ноутбук) `m3_workshop.py`, где каждый теоретический тезис M3 подтверждён численным экспериментом, а интуиция совпала с вычислением. Это твой личный «канонический» файл, к которому ты вернёшься, когда забудешь, почему косинус двух коллинеарных векторов равен единице.

!!! tip "Главный принцип воркшопа"

    Правило M3 из раздела про AI-копилота: любое математическое утверждение проверяй числами. Здесь мы делаем это системно — для каждой формулы рядом стоит `assert`, который падает, если ты ошибся в реализации. Если все ассерты прошли — твоя интуиция бьётся с математикой.

## Бизнес-кейс

Этот воркшоп фундаментальный — ты тренируешь не одну метрику, а математику, на которой стоит весь твой будущий ML: похожесть профилей, диагностику данных, обучение модели руками. Чтобы навык не повис в воздухе, привяжем его к конкретному решению, которое от этой математики зависит.

!!! example "Ситуация"

    Ты — data scientist в iGaming-продукте. **Продакт-оунер монетизации** просит модель, которая объясняет, что разгоняет GGR игрока (`депозиты`, `средняя_ставка`, `дней_активности`), чтобы решить, на какой драйвер давить в следующем спринте. Аналитик прислал линейную регрессию, но коэффициенты ведут себя странно: на мартовских данных вес «средней ставки» вышел +180, на апрельских −90 — на одних данных фактор «помогает», на других «вредит». На таких качелях нельзя строить решение.

    - **Что на кону.** По выводу модели продакт перераспределит ~3-4 ₽ млн квартального бюджета на удержание и бонусы между драйверами. Ошибка в знаке коэффициента = деньги уйдут не туда.
    - **Где зарыта проблема.** Признаки скоррелированы (больше дней активности → больше депозитов), $X^\top X$ почти сингулярна, число обусловленности гигантское — отсюда «скачущие» веса. Ровно то, что ты проверяешь числами в шагах 3 и 5.
    - **Зачем ещё косинус и PCA.** Тот же продакт хочет находить lookalike к топовым VIP по форме профиля, не по объёму (шаг 1), и схлопнуть скоррелированные признаки в независимые компоненты (шаг 4) — чтобы модель перестала «скакать».
    - **Ограничение.** Времени до планёрки спринта — два дня, своих данных не дают, тренируешься на синтетике с тем же поведением, что в проде.

    Пройдя воркшоп, ты сможешь не просто запустить `sklearn`, а объяснить продакту **почему** его текущей модели нельзя верить и что с этим делать.

Генератор данных — профили игроков и регрессионная выборка для GGR. Признаки в духе примеров модуля: `[депозитов, средняя_ставка, дней_активности]`.

```python
import numpy as np

rng = np.random.default_rng(42)

# Профили двух похожих по форме, но разных по масштабу игроков
player_a = np.array([5.0, 120.0, 14.0])   # обычный
player_b = np.array([10.0, 240.0, 28.0])  # ровно вдвое крупнее по всем осям

# Регрессионная выборка: GGR = X @ w_true + шум
n = 500
X = rng.normal(size=(n, 3))
w_true = np.array([2.0, -1.0, 0.5])
y = X @ w_true + rng.normal(scale=0.1, size=n)
```

## Ход работы

### Шаг 1: Векторы, нормы, косинус — схожесть по форме, а не по масштабу

**Зачем.** Модуль утверждает: косинусная близость убирает влияние длины и оставляет направление, поэтому два VIP-игрока с одинаковым паттерном, но разным объёмом ставок близки по косинусу, хотя далеки по евклидову расстоянию. `player_b` — это ровно `2 * player_a`, идеальный тест: косинус обязан быть единицей, а евклидово расстояние — большим.

```python
def cosine(u, v):
    return (u @ v) / (np.linalg.norm(u) * np.linalg.norm(v))

dot = player_a @ player_b
cos = cosine(player_a, player_b)
euclid = np.linalg.norm(player_a - player_b)

l2 = np.linalg.norm(player_a, ord=2)
l1 = np.linalg.norm(player_a, ord=1)

print(f"dot={dot:.1f}  cos={cos:.6f}  euclid={euclid:.1f}")
print(f"L2={l2:.3f}  L1={l1:.1f}")

assert np.isclose(cos, 1.0)                       # коллинеарны -> косинус ровно 1
assert euclid > 100                               # но по расстоянию далеки
assert np.isclose(l2, np.sqrt((player_a**2).sum()))
```

**Что получилось.** `cos = 1.000000`, потому что `player_b` лежит на том же луче, что `player_a` — направление идентично. При этом `euclid` огромное: масштаб разный. Ты численно подтвердил, почему рекомендации и retrieval строят на косинусе, а не на евклиде, когда важна «форма» профиля.

### Шаг 2: Матричное умножение как взвешенная сумма и как поворот

**Зачем.** Два взгляда M3 на матричное умножение. Первый: $\hat{\mathbf{y}} = X\mathbf{w}$ — это пакетный расчёт скалярных произведений строки весов на столбец признаков. Второй: матрица — преобразование пространства; поворот на 90° делает векторы перпендикулярными, а перпендикулярность означает нулевое скалярное произведение.

```python
# Взгляд 1: прогноз линейной модели руками == матричное умножение
Xb = np.array([[5, 120, 14],
               [6, 110, 16],
               [2,  30,  3]], dtype=float)
w = np.array([0.5, 0.01, 2.0])

y_hat = Xb @ w
y_hat_manual = np.array([row @ w for row in Xb])   # строка на столбец вручную
assert np.allclose(y_hat, y_hat_manual)

# Взгляд 2: матрица поворота на 90 градусов обнуляет скалярное произведение
theta = np.pi / 2
R = np.array([[np.cos(theta), -np.sin(theta)],
              [np.sin(theta),  np.cos(theta)]])
v = np.array([3.0, 1.0])
v_rot = R @ v
assert np.isclose(v @ v_rot, 0.0)                  # перпендикулярны -> dot = 0

# Некоммутативность: AB != BA
A = np.array([[1, 2], [0, 3]], dtype=float)
B = np.array([[1, 0], [4, 1]], dtype=float)
assert not np.allclose(A @ B, B @ A)
```

**Что получилось.** Ручная сумма произведений совпала с `@` — ты видишь, что под `X @ w` нет магии, только $\sum_l a_{il} w_l$. Поворот сделал `v @ v_rot == 0`: матрица реально повернула пространство. И `AB != BA` — некоммутативность не баг, а отражение того, что «повернуть, потом растянуть» не равно «растянуть, потом повернуть».

!!! question "Проверь себя"

    1. Почему `cos(player_a, player_b)` получился ровно 1, хотя векторы разные?
    2. Что должно совпасть в размерностях, чтобы `Xb @ w` не упало с ошибкой формы?
    3. Почему после поворота на 90° скалярное произведение исходного и повёрнутого вектора равно нулю?

??? success "Ответы"

    1. `player_b = 2 * player_a` — коллинеарны, направление идентично. Косинус смотрит только на направление (делит на нормы), масштаб 2 сокращается.
    2. Число столбцов `Xb` (3) должно равняться длине `w` (3). Результат — вектор длины, равной числу строк `Xb`.
    3. Поворот на 90° делает векторы перпендикулярными, а у перпендикулярных $\mathbf{a}\cdot\mathbf{b} = \|\mathbf{a}\|\|\mathbf{b}\|\cos 90° = 0$.

### Шаг 3: Ранг и число обусловленности — диагностика мультиколлинеарности

**Зачем.** M3: если третий признак — линейная комбинация первых двух, ранг падает ниже числа столбцов, $X^\top X$ сингулярна, регрессия теряет единственное решение. Воспроизведём это и посмотрим, как крошечный шум «чинит» ранг формально, но оставляет матрицу почти сингулярной (огромное `cond`).

```python
base = rng.normal(size=(5, 2))
col3 = base[:, 0] + base[:, 1]                     # ровно сумма первых двух
X_collinear = np.column_stack([base, col3])

print("rank:", np.linalg.matrix_rank(X_collinear)) # 2 < 3 -> зависимость
print("cond:", np.linalg.cond(X_collinear))        # огромное число

assert np.linalg.matrix_rank(X_collinear) == 2

# Добавим крошечный шум в третий столбец
X_noisy = X_collinear.copy()
X_noisy[:, 2] += rng.normal(scale=1e-6, size=5)

print("rank (noisy):", np.linalg.matrix_rank(X_noisy))  # формально 3
print("cond (noisy):", np.linalg.cond(X_noisy))         # всё ещё гигантское

assert np.linalg.matrix_rank(X_noisy) == 3
assert np.linalg.cond(X_noisy) > 1e5               # почти сингулярна
```

**Что получилось.** Без шума `rank = 2` при трёх столбцах — третий признак не несёт новой информации, как «депозиты в долларах и в евро по фиксу». С микрошумом ранг формально стал 3, но `cond` остался гигантским: матрица почти сингулярна. Вывод для регрессии: коэффициенты на таких данных «скачут» и бессмысленны, даже если формально решение существует. Число обусловленности — твой детектор беды задолго до краша.

### Шаг 4: Собственные значения и PCA с нуля

**Зачем.** M3: собственные векторы ковариационной матрицы — направления максимального разброса (главные компоненты), собственные значения — величина разброса. Соберём PCA руками через `eigh` и сверим с `sklearn`. Знаки компонент могут отличаться — это нормально, важна доля объяснённой дисперсии.

```python
from sklearn.decomposition import PCA

Xc = X - X.mean(axis=0)                             # центрирование обязательно
cov = (Xc.T @ Xc) / (len(X) - 1)                   # ковариационная матрица

eigvals, eigvecs = np.linalg.eigh(cov)             # eigh — для симметричных
order = np.argsort(eigvals)[::-1]                  # по убыванию дисперсии
eigvals, eigvecs = eigvals[order], eigvecs[:, order]

explained = eigvals / eigvals.sum()
print("доля объяснённой дисперсии (руками):", explained.round(4))

# Собственные векторы симметричной матрицы ортогональны
assert np.allclose(eigvecs.T @ eigvecs, np.eye(3), atol=1e-10)
# Av = lambda v для первой компоненты
assert np.allclose(cov @ eigvecs[:, 0], eigvals[0] * eigvecs[:, 0])

sk = PCA().fit(Xc)
assert np.allclose(np.sort(explained)[::-1],
                   np.sort(sk.explained_variance_ratio_)[::-1], atol=1e-6)
print("доля (sklearn):           ", sk.explained_variance_ratio_.round(4))
```

**Что получилось.** Доля объяснённой дисперсии руками совпала со `sklearn` до шестого знака — твой PCA «с нуля» эквивалентен библиотечному. Ассерт `cov @ v == lambda * v` буквально подтвердил определяющее равенство $A\mathbf{v} = \lambda\mathbf{v}$ на реальной матрице. Ортогональность собственных векторов (`eigvecs.T @ eigvecs == I`) — то самое свойство, благодаря которому главные компоненты независимы.

!!! question "Проверь себя"

    1. Почему перед расчётом ковариационной матрицы данные обязательно центрируют?
    2. Что означает прошедший ассерт `cov @ eigvecs[:,0] == eigvals[0] * eigvecs[:,0]`?
    3. Почему мы используем `np.linalg.eigh`, а не `eig`?

??? success "Ответы"

    1. Ковариация измеряет совместный разброс вокруг среднего. Без вычитания среднего матрица будет описывать разброс вокруг нуля и компоненты исказятся.
    2. Что первый собственный вектор — настоящий собственный вектор: матрица действует на него как умножение на число $\lambda$ (определение из M3), не сворачивая в сторону.
    3. `eigh` предназначена для симметричных матриц (ковариация симметрична), она быстрее и гарантирует вещественные собственные значения и ортогональные векторы.

### Шаг 5: Градиентный спуск для линейной регрессии руками, по шагам

**Зачем.** Ядро воркшопа. Реализуем $\mathbf{w}_{t+1} = \mathbf{w}_t - \eta\,\nabla L$ без `sklearn`. Градиент MSE по весам: $\nabla L = \frac{2}{n} X^\top (X\mathbf{w} - \mathbf{y})$. Тот самый коэффициент $2/n$, который LLM и новички теряют. Проверим градиент конечными разностями (как требует раздел «Ловушки» M3), потом сравним сошедшиеся веса с нормальным уравнением через `np.linalg.solve` (не через явную инверсию).

```python
def mse(w):
    r = X @ w - y
    return (r @ r) / len(X)

def grad_mse(w):
    return (2 / len(X)) * X.T @ (X @ w - y)

# Сверка аналитического градиента с конечными разностями
w0 = rng.normal(size=3)
eps = 1e-6
num_grad = np.array([
    (mse(w0 + eps*e) - mse(w0 - eps*e)) / (2*eps)
    for e in np.eye(3)
])
assert np.allclose(grad_mse(w0), num_grad, atol=1e-6)   # коэффициент 2/n верный

# Сам градиентный спуск
w = np.zeros(3)
eta = 0.1
history = []
for step in range(200):
    history.append(mse(w))
    w -= eta * grad_mse(w)

print("восстановленные веса:", w.round(3))              # ~ [2, -1, 0.5]
assert np.allclose(w, w_true, atol=0.05)
assert history[-1] < history[0]                         # лосс убывал

# Эталон: нормальное уравнение через solve (НЕ через inv)
w_closed = np.linalg.solve(X.T @ X, X.T @ y)
assert np.allclose(w, w_closed, atol=0.05)
```

**Что получилось.** Аналитический градиент совпал с конечными разностями — значит коэффициент $2/n$ и знак верны, модель не «учится наоборот». За 200 шагов веса сошлись к `[2, -1, 0.5]` и совпали с решением нормального уравнения через `solve`. Ты построил обучение «всего ML» из одной формулы шага.

### Шаг 6: Learning rate решает судьбу оптимизации

**Зачем.** M3 предупреждает: слишком большой $\eta$ — лосс взрывается в NaN, слишком маленький — «не учится». Проверим оба режима явно и нарисуем кривые убывания лосса.

```python
import matplotlib.pyplot as plt

def run_gd(eta, steps=100):
    w = np.zeros(3)
    hist = []
    for _ in range(steps):
        hist.append(mse(w))
        w = w - eta * grad_mse(w)
    return np.array(hist)

good = run_gd(0.1)
slow = run_gd(0.001)
blow = run_gd(1.5)

assert good[-1] < 1e-2                          # сошлось
assert slow[-1] > good[-1]                      # слишком медленно
assert np.isnan(blow).any() or blow[-1] > blow[0]  # разошлось

for name, h in [("eta=0.1 (good)", good), ("eta=0.001 (slow)", slow)]:
    plt.plot(h, label=name)
plt.yscale("log"); plt.xlabel("шаг"); plt.ylabel("MSE (log)")
plt.legend(); plt.title("Убывание лосса vs learning rate")
plt.savefig("loss_curves.png", dpi=120)
```

**Что получилось.** `eta=0.1` уводит лосс ниже `1e-2`, `eta=0.001` ползёт и за 100 шагов не доходит, `eta=1.5` расходится (NaN или рост). Картинка `loss_curves.png` — наглядное доказательство, что learning rate это первый гиперпараметр, который надо проверять до того, как менять модель. Это и есть выпуклость в действии: задача линейной регрессии выпукла, поэтому при адекватном $\eta$ спуск гарантированно приходит в глобальный минимум.

!!! question "Проверь себя"

    1. Что именно проверяет сверка аналитического градиента с конечными разностями?
    2. Почему мы решаем нормальное уравнение через `np.linalg.solve`, а не `np.linalg.inv(X.T @ X) @ ...`?
    3. Почему здесь градиентный спуск гарантированно находит глобальный минимум, а у нейросети нет?

??? success "Ответы"

    1. Что в формуле градиента не потерян коэффициент $2/n$ и знак верный — частая ошибка, из-за которой модель расходится или «учится наоборот».
    2. Явная инверсия численно нестабильна и медленна, особенно при плохой обусловленности; `solve` использует разложение и точнее. Прямой `inv` — признак наивного кода (предупреждение M3).
    3. Функция потерь линейной регрессии (MSE) выпукла — один минимум, он же глобальный. Нейросеть невыпукла, у неё много локальных минимумов, и результат зависит от инициализации.

## Критерий готовности

- [ ] `cos(player_a, player_b) == 1`, а евклидово расстояние большое — косинус ловит форму, не масштаб
- [ ] Ручная «строка на столбец» совпала с `@`, поворот обнулил скалярное произведение, `AB != BA`
- [ ] На коллинеарных данных `rank < n_cols`, `cond` гигантское; микрошум чинит ранг, но не обусловленность

## Развитие

- **L1 vs L2 руками.** Добавь к градиенту MSE слагаемое регуляризации: для Ridge — $2\lambda\mathbf{w}$, для Lasso — $\lambda\,\mathrm{sign}(\mathbf{w})$. Прогони на коллинеарных данных и покажи численно, что Lasso обнуляет лишний признак, а Ridge только прижимает — тот самый отбор признаков из M3.
- **Цепное правило и автоград.** Возьми вложенную функцию `y = sin(x**2)`, посчитай производную руками (`cos(x**2) * 2x`) и сверь с `torch.autograd`. Это мостик к backpropagation: убедись, что ручная цепочка совпала с автоградом до машинной точности.
- **PCA через SVD.** Замени `eigh(cov)` на `np.linalg.svd(Xc)` и покажи, что сингулярные значения дают ту же объяснённую дисперсию. Это численно устойчивый путь, который M3 рекомендует для больших данных вместо явной ковариации.
- **Стохастический градиентный спуск.** Перепиши шаг 5 так, чтобы градиент считался по мини-батчу из 32 объектов, а не по всей выборке. Сравни траекторию лосса с полнобатчевой: шумнее, но дешевле на шаг — основа обучения больших моделей.
