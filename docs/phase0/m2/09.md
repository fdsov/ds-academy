# M2.9 · ООП для Data Science: зачем аналитику классы

Тут нужна честность. В DS большая часть кода — функциональная: данные внутрь, преобразование, данные наружу. Но ООП незаменимо в трёх местах: кастомные трансформеры для пайплайнов scikit-learn, конфигурации с состоянием, и обёртки над моделями. Понимать ООП обязательно ещё и потому, что весь sklearn, PyTorch и pandas построены на классах.

Когда писать класс, а когда оставить функцию — простое правило: **класс нужен там, где данные и поведение должны жить вместе и переживать один вызов**. Если у тебя есть «состояние», которое надо помнить между вызовами (обученные параметры модели, накопленный буфер, конфиг), — это класс. Если ты просто гоняешь DataFrame через серию преобразований без состояния — оставь функцию, не плоди сущности. Типичная ошибка новичка после знакомства с ООП — заворачивать в класс то, что прекрасно работает как три строчки функции. Это усложняет код, а не упрощает.

Интуиция. Класс — это чертёж, объект — изделие по чертежу. Класс `Player` описывает, какие данные есть у игрока (атрибуты) и что с ним можно делать (методы). Создаёшь конкретного игрока — это экземпляр. Чертёж один, изделий по нему — сколько угодно, и каждое со своими значениями атрибутов.

Формально. Класс инкапсулирует данные (атрибуты) и поведение (методы). `__init__` — конструктор, вызывается при создании экземпляра, `self` — ссылка на сам экземпляр. Наследование позволяет переиспользовать и расширять поведение.

```python
class Player:
    def __init__(self, player_id: int, deposits: float, withdrawals: float):
        self.player_id = player_id
        self.deposits = deposits
        self.withdrawals = withdrawals

    def ltv(self) -> float:
        return self.deposits - self.withdrawals

    def __repr__(self) -> str:        # как объект печатается
        return f"Player(id={self.player_id}, ltv={self.ltv():.2f})"


class VIPPlayer(Player):              # наследование
    def __init__(self, player_id, deposits, withdrawals, manager: str):
        super().__init__(player_id, deposits, withdrawals)
        self.manager = manager

    def ltv(self) -> float:           # переопределение: VIP-бонус 10%
        return super().ltv() * 1.1
```

Разберём, что происходит на конкретных числах. Создадим обычного игрока и VIP-игрока с одинаковыми финансами и посмотрим на разницу:

```python
p = Player(player_id=1, deposits=1000.0, withdrawals=400.0)
print(p.ltv())        # 600.0  ->  1000 - 400
print(p)              # Player(id=1, ltv=600.00)  ->  сработал __repr__

v = VIPPlayer(player_id=2, deposits=1000.0, withdrawals=400.0, manager="Anna")
print(v.ltv())        # 660.0  ->  super().ltv() = 600, затем * 1.1
print(v.manager)      # Anna   ->  атрибут, которого нет у базового Player
print(v)              # Player(id=2, ltv=660.00)  ->  __repr__ унаследован, но видит новый ltv()
```

Обрати внимание на две тонкости. Первая: `__repr__` мы не переопределяли в `VIPPlayer`, он унаследован от `Player` — но внутри он вызывает `self.ltv()`, а `self` здесь VIP, поэтому печатается уже 660, а не 600. Это и есть полиморфизм: один и тот же код `self.ltv()` ведёт себя по-разному в зависимости от реального типа объекта. Вторая: `super().ltv()` в VIP вызывает именно версию родителя (600), а не сам себя — иначе была бы бесконечная рекурсия.

!!! warning "Типичная ошибка: атрибут класса вместо атрибута экземпляра"

    Изменяемый объект (`list`, `dict`), объявленный на уровне класса, а не внутри `__init__`, становится общим для всех экземпляров.

    ```python
    class Cohort:
        players = []                  # ОПАСНО: один список на ВЕСЬ класс

        def add(self, pid):
            self.players.append(pid)

    a, b = Cohort(), Cohort()
    a.add(1)
    print(b.players)                  # [1] — хотя в b ничего не клали!
    ```

    Правильно — создавать изменяемое состояние в `__init__`: `self.players = []`. Тогда у каждого экземпляра свой список.

### Dunder-методы

Интуиция. Это «магические» методы с двойными подчёркиваниями, которые позволяют твоему объекту вести себя как встроенный: складываться, печататься, сравниваться, попадать в множество.

Формально. Python вызывает их неявно. `__init__` — конструктор, `__repr__`/`__str__` — текстовое представление, `__eq__` — сравнение `==`, `__hash__` — хеш для использования в set/dict, `__len__` — длина, `__lt__` — оператор `<` для сортировки.

```python
class Cohort:
    def __init__(self, market, month):
        self.market = market
        self.month = month

    def __eq__(self, other):
        return (self.market, self.month) == (other.market, other.month)

    def __hash__(self):
        return hash((self.market, self.month))   # теперь Cohort можно класть в set
```

Зачем это нужно на практике. Допустим, ты собираешь когорты из разных источников (выгрузка по депозитам и выгрузка по ретеншену) и хочешь найти уникальные. Без `__eq__`/`__hash__` Python сравнивает объекты по адресу в памяти — два «одинаковых» по смыслу объекта будут считаться разными:

```python
без_dunder = {Cohort("RU", "2026-06"), Cohort("RU", "2026-06")}
# если бы __eq__/__hash__ не были заданы -> len == 2 (две разные ссылки)

с_dunder = {Cohort("RU", "2026-06"), Cohort("RU", "2026-06")}
print(len(с_dunder))            # 1 — Python понял, что это одна и та же когорта
```

Важное правило-ловушка: `__eq__` и `__hash__` **обязаны быть согласованы**. Если два объекта равны по `__eq__`, их хеши тоже должны совпадать (иначе set/dict сломаются). Поэтому в обоих методах используем один и тот же набор полей — кортеж `(market, month)`. И ещё: как только ты определяешь свой `__eq__`, Python автоматически выставляет `__hash__ = None` (объект становится нехешируемым), пока ты не зададишь `__hash__` явно. Именно поэтому здесь нужны оба метода.

### Современная идиома: dataclass

!!! tip "Современная идиома: dataclass вместо ручного __init__"

    Писать `__init__`, `__repr__`, `__eq__` руками — устаревший стиль. С Python 3.7+ есть `@dataclass`, который генерирует их автоматически.

    ```python
    from dataclasses import dataclass

    @dataclass(frozen=True)          # frozen=True делает объект неизменяемым и хешируемым
    class Cohort:
        market: str
        month: str

    c = Cohort("RU", "2026-06")      # __init__, __repr__, __eq__, __hash__ — бесплатно
    ```

Почему именно `frozen=True` даёт хеш. По умолчанию `@dataclass` генерирует `__eq__`, но **не** `__hash__` (объект остаётся изменяемым, а изменяемое нельзя безопасно хешировать — хеш «уплыл» бы при изменении полей). `frozen=True` запрещает менять поля после создания, и тогда dataclass добавляет `__hash__` автоматически. Попытка присвоить полю значение после создания упадёт:

```python
c.market = "KZ"     # FrozenInstanceError: cannot assign to field 'market'
```

Это ровно тот случай, когда меньше кода = меньше багов: четыре аккуратно согласованных dunder-метода ты получаешь одной строкой декоратора вместо двадцати строк ручной писанины, где легко забыть синхронизировать `__eq__` и `__hash__`.

    2. Что делают `__init__`, `self` и `__repr__`?
    3. Зачем переопределять `__eq__` и `__hash__`, чтобы класть свои объекты в `set`?
    4. Что генерирует `@dataclass` автоматически и чем полезен `frozen=True`?

??? success "Ответы"

    1. Кастомные трансформеры для пайплайнов sklearn, конфигурации с состоянием, обёртки над моделями. Плюс — sklearn/PyTorch/pandas сами построены на классах.
    2. `__init__` — конструктор (вызывается при создании экземпляра), `self` — ссылка на сам экземпляр, `__repr__` — текстовое представление объекта при печати.
    3. `__eq__` задаёт сравнение по значению, `__hash__` — стабильный хеш; без согласованной пары объект нельзя корректно хранить в set/использовать ключом dict.
    4. Генерирует `__init__`, `__repr__`, `__eq__` (и `__hash__` при frozen). `frozen=True` делает объект неизменяемым и хешируемым.
