# M2.10 · Декораторы

Интуиция. Декоратор — это обёртка вокруг функции, которая добавляет поведение, не трогая саму функцию. Как чехол на телефон: телефон тот же, но добавилась защита. Типичное применение: логирование, замер времени, кеширование, проверка прав.

Можно представить иначе. У вас есть готовая функция, которая что-то считает. Вам нужно, чтобы вокруг неё «до» и «после» происходило что-то ещё — например, печатался лог или замерялось время. Вы не хотите лезть внутрь функции и засорять её этим. Декоратор делает ровно это: оборачивает функцию во внешний слой, который сначала делает свою работу, потом зовёт оригинал, потом снова делает свою работу. Оригинал остаётся чистым.

Формально. Декоратор — это функция, которая принимает функцию и возвращает новую функцию. Синтаксис `@decorator` над определением — синтаксический сахар для `func = decorator(func)`.

То есть две записи ниже полностью эквивалентны:

```python
@timed
def compute_retention(events):
    ...

# то же самое без сахара:
def compute_retention(events):
    ...
compute_retention = timed(compute_retention)
```

Ключевая мысль: после декорирования имя `compute_retention` указывает уже не на исходную функцию, а на `wrapper`, который вернул декоратор. Поэтому всё, что вы кладёте в `wrapper`, выполняется при каждом вызове.

```python
import functools
import time

def timed(func):
    @functools.wraps(func)            # сохраняет имя и docstring оригинала
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        print(f"{func.__name__}: {time.perf_counter() - start:.3f}s")
        return result
    return wrapper

@timed
def compute_retention(events):
    ...                               # тяжёлый расчёт
    return result
```

Разберём по слоям. Внешняя функция `timed` принимает `func` (оригинал) и существует один раз — в момент декорирования. Внутренняя `wrapper` создаётся и возвращается; именно она зовётся при каждом вызове `compute_retention(...)`. Внутри `wrapper` важны три вещи: `*args, **kwargs` пробрасывают любые аргументы насквозь, `func(*args, **kwargs)` зовёт оригинал, а `return result` возвращает его результат наружу. Если забыть `return result`, декорированная функция начнёт возвращать `None` — классическая и очень обидная ошибка.

Зачем `functools.wraps`. Без него у обёрнутой функции `compute_retention.__name__` стал бы `"wrapper"`, а docstring потерялся бы. Это ломает логи, автодокументацию и отладку: вы зовёте `compute_retention`, а в трейсбеке и логах видите безликий `wrapper`. `@functools.wraps(func)` копирует метаданные оригинала на обёртку. Правило простое: пишешь декоратор — всегда ставь `wraps`.

Декоратор с аргументами. Иногда обёртке нужны собственные параметры (например, сколько раз повторять при ошибке). Тогда добавляется ещё один уровень вложенности: внешняя функция принимает параметр и возвращает уже сам декоратор.

```python
import functools

def retry(times):                          # принимает параметр декоратора
    def decorator(func):                   # принимает функцию
        @functools.wraps(func)
        def wrapper(*args, **kwargs):      # принимает аргументы вызова
            for attempt in range(times):
                try:
                    return func(*args, **kwargs)
                except Exception:
                    if attempt == times - 1:
                        raise
        return wrapper
    return decorator

@retry(times=3)                            # сначала вызвали retry(3) -> вернулся decorator
def fetch_ggr(market: str):
    return flaky_db_query(market)          # нестабильный запрос, до 3 попыток
```

Здесь `@retry(times=3)` сначала исполняется как обычный вызов `retry(3)`, который возвращает `decorator`, и уже `decorator` оборачивает `fetch_ggr`. Три уровня: параметры декоратора → функция → аргументы вызова.

Самый полезный встроенный декоратор для аналитика — `functools.lru_cache`. Он кеширует результаты функции: вызвал с теми же аргументами — мгновенно вернул запомненное. Это спасает при повторных дорогих запросах к одним и тем же когортам.

```python
from functools import lru_cache

@lru_cache(maxsize=None)
def cohort_size(market: str, month: str) -> int:
    return expensive_db_query(market, month)   # второй вызов с теми же аргументами — без запроса
```

Численный пример выгоды. Пусть запрос размера когорты к БД занимает 200 мс, а в отчёте по 50 рынкам вы дёргаете `cohort_size` для каждого рынка трижды (на трёх разных графиках). Без кеша это $150$ запросов $\times 0.2$ с $= 30$ с. С `lru_cache` уникальных аргументов всего 50, значит реальных запросов — 50, остальные 100 берутся из памяти почти мгновенно: $50 \times 0.2 = 10$ с. Втрое быстрее, и нагрузка на БД втрое меньше.

Подводные камни `lru_cache`, о которых забывают:

- Аргументы должны быть хешируемыми. `str`, `int`, `tuple` — можно. `list`, `dict`, `set`, `DataFrame` — нельзя, будет `TypeError: unhashable type`. Если передаёте список рынков, оберните его в `tuple`.
- Кеш живёт в памяти процесса и не сбрасывается сам. Если данные в БД за месяц изменились, функция всё равно вернёт старое значение — кеш про это не знает. Для свежести вызывайте `cohort_size.cache_clear()` или используйте версионный ключ (например, добавляйте дату загрузки в аргументы).
- `maxsize=None` означает безлимитный рост кеша — память может течь на больших ключевых пространствах. Для часто меняющихся ключей ставьте конечный `maxsize`, тогда работает вытеснение по принципу «давно не использованных» (LRU = Least Recently Used).
- Кеш по умолчанию различает позиционные и именованные аргументы: `cohort_size("UZ", "2026-05")` и `cohort_size(market="UZ", month="2026-05")` считаются разными ключами и дадут два запроса.

!!! warning "Типичная ошибка"

    Кешировать функцию, которая зависит не только от своих аргументов, а ещё и от «внешнего мира» (текущего времени, состояния БД, глобальной переменной). Кеш привязан строго к аргументам, поэтому при тех же аргументах вы навсегда заморозите первый результат. Кешируйте только чистые функции — те, чей вывод полностью определяется входами.

!!! note "Проверь себя"

    Что выведет код, если убрать строку `return result` из `wrapper` в декораторе `timed`, а затем напечатать `compute_retention(events)`? Ответ: `None` — обёртка ничего не возвращает наружу, и результат тяжёлого расчёта теряется, хотя сам расчёт отработал.

### Быстрый квиз

Выбери вариант — ответ подсветится сразу.

```quiz
Чему эквивалентна запись @timed над def compute_retention?
[ ] timed(compute_retention()) — вызывается сразу с результатом
[x] compute_retention = timed(compute_retention)
[ ] timed = compute_retention(timed)
> Декоратор — синтаксический сахар: имя переопределяется на результат decorator(func).
---
Зачем внутри декоратора нужен functools.wraps(func)?
[ ] Чтобы ускорить вызов обёрнутой функции
[x] Чтобы сохранить имя и docstring оригинала на обёртке
[ ] Чтобы автоматически кешировать результаты
> Без wraps __name__ станет "wrapper", метаданные потеряются — это ломает логи и отладку.
---
Что вернёт декорированная функция, если в wrapper забыть return result?
[ ] Исходный результат расчёта
[x] None
[ ] Ошибку TypeError при вызове
> wrapper отработает func, но ничего не вернёт наружу — наружу уходит None.
---
Почему нельзя передать в функцию под @lru_cache аргумент типа list?
[x] list не хешируемый, а ключ кеша должен быть хешируемым
[ ] list слишком большой для кеша
[ ] lru_cache работает только со строками
> Ключ кеша строится из аргументов и должен хешироваться; list даёт TypeError. Нужен tuple.
---
Когда lru_cache вернёт устаревший результат?
[ ] Никогда — он сам отслеживает изменения в БД
[x] Когда данные в источнике изменились, а аргументы вызова те же
[ ] Только если задан maxsize=None
> Кеш привязан к аргументам и не знает о внешних изменениях; нужен cache_clear() или версионный ключ.
```
