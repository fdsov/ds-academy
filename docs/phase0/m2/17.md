# M2.17 · Идиомы и читаемость (PEP 8)

Питоничный код — не про знание трюков, а про ясность и соглашения, которые знает всё сообщество. PEP 8 — стиль-гайд Python.

## Зачем вообще стиль-гайд

Код читают в разы чаще, чем пишут. В аналитике это особенно болезненно: SQL-запрос, который считает GGR, или скрипт, который строит когортную retention-таблицу, переживёт своего автора — его через полгода откроет коллега (или вы сами, забывший контекст) и должен понять за минуту, а не за час. PEP 8 убирает «лишнюю когнитивную нагрузку»: когда весь код в проекте выглядит одинаково, мозг не тратит силы на расшифровку чужих привычек и фокусируется на сути — логике расчёта.

Идея PEP 8 буквально из его текста: «A style guide is about consistency. Consistency with this style guide is important. Consistency within a project is more important.» То есть единообразие важнее буквы правила. Если в проекте уже принято иначе — следуй проекту.

## Ключевое

Отступы 4 пробела, имена переменных и функций `snake_case`, классов `CamelCase`, констант `UPPER_CASE`. Строки до 88-100 символов (раньше было 79). Пробелы вокруг операторов. Импорты сгруппированы: стандартная библиотека, сторонние, локальные.

Чуть подробнее по каждому пункту, с примерами из аналитики:

```python
# имена говорят сами за себя — это часть стиля
ftd_amount = 50.0           # snake_case для переменной
def calc_hold(ggr, bet_volume):   # snake_case для функции
    ...

class PlayerCohort:         # CamelCase (PascalCase) для класса
    ...

DEFAULT_HOLD = 0.05         # UPPER_CASE для константы модуля
MIN_DEPOSIT_USD = 10
```

Типичные ошибки именования, по которым видно новичка: `l`, `O`, `I` как однобуквенные имена (визуально путаются с `1` и `0`); венгерская нотация (`strName`, `dfPlayers`); `CamelCase` для обычных переменных (`PlayerLtv` вместо `player_ltv`). Однобуквенные имена уместны только для очень короткого scope — счётчик `i` в цикле, координаты `x, y`.

Пробелы — два частых антипаттерна:

```python

x = my_func( a,b )
hold=ggr/bet_volume

# хорошо
x = my_func(a, b)

# но в аргументах-ключевых словах пробелов вокруг = НЕТ
plot(data, color="red", alpha=0.5)     # не color = "red"
```

## Питоничные идиомы, по которым узнают уровень

```python
# вместо range(len(x)) — enumerate
for i, bet in enumerate(bets):
    ...

# параллельный обход — zip
for player, deposit in zip(players, deposits):
    ...

# распаковка вместо индексов
first, *rest = daily_ggr

# truthiness вместо len() == 0
if not bets:                          # пусто, а не if len(bets) == 0
    return

# тернарник для простого ветвления
status = "vip" if ltv > 10000 else "regular"

# словарь get с дефолтом вместо if-проверки
hold = config.get("hold", 0.05)
```

Почему `enumerate`, а не `range(len(...))`. Сравните «непитоничную» версию и идиому:

```python
# непитонично — индексируем вручную, легко ошибиться и шумно
for i in range(len(bets)):
    print(i, bets[i])

# питонично — индекс и элемент сразу, без bets[i]
for i, bet in enumerate(bets):
    print(i, bet)

# enumerate умеет начинать счёт не с нуля
for rank, player in enumerate(top_players, start=1):
    print(f"{rank}. {player}")    # 1. ..., 2. ...
```

Разбор `truthiness`. В Python «пустое» значение само по себе ложно: `[]`, `{}`, `""`, `0`, `0.0`, `None` — все falsy. Поэтому `if not bets` читается как «если ставок нет». Но здесь прячется частая ловушка в аналитике:

```python
# опасно: 0.0 — валидный hold, но truthiness посчитает его "пустым"
if not hold:               # сработает и при hold == 0.0!
    hold = DEFAULT_HOLD

# правильно, когда 0 — легитимное значение, а отсутствие — это None
if hold is None:
    hold = DEFAULT_HOLD
```

То есть `if not x` хорош для проверки «коллекция пуста», но опасен, когда ноль — это настоящие данные (нулевой депозит, нулевой GGR за день, нулевая ставка). Для «значение не задано» проверяйте именно `is None`.

Ещё одна идиома — `dict.get` с дефолтом. Численный пример: конфиг гео-настроек, где у части стран hold не прописан.

```python
geo_config = {"UZ": {"hold": 0.07}, "KZ": {}}

# непитонично и хрупко
if "hold" in geo_config["KZ"]:
    h = geo_config["KZ"]["hold"]
else:
    h = 0.05

# питонично
h = geo_config["KZ"].get("hold", 0.05)   # -> 0.05, ключа нет, берём дефолт
```

## Форматтер вместо ручного стиля

В 2026 стиль не правят руками — это делает форматтер. Стандарт де-факто — `ruff` (он же линтер, он же форматтер, написан на Rust, в десятки раз быстрее старых `black` + `flake8`).

На практике это две команды:

```bash
ruff format .     # переформатирует код по стилю (как black)
ruff check .      # ищет проблемы: неиспользованные импорты, идиомы, баги
ruff check --fix .  # чинит то, что можно починить автоматически
```

Важно различать роли. **Форматтер** трогает только внешний вид: расставляет пробелы, переносы, кавычки — он не меняет поведение программы. **Линтер** ищет смысловые проблемы: переменная объявлена и не использована, импорт лишний, сравнение `== None` вместо `is None`, потенциальный баг. Форматтер не заменяет линтер, и наоборот — ruff просто совмещает оба в одном инструменте.

Зачем это в команде: форматтер прогоняется автоматически (pre-commit hook или CI), поэтому споры «таб или пробел», «79 или 100 символов» исчезают как класс — стиль не предмет ревью, его обеспечивает машина. Ревьюер смотрит на логику, а не на запятые. Это и есть главная ценность PEP 8 на масштабе: не «правильные пробелы», а отсутствие дискуссий о пробелах.

!!! warning "Стиль не делает код корректным"

    Форматтер и линтер ловят оформление и часть мелких багов, но не проверяют логику. Идеально отформатированный скрипт может считать GGR по неверной формуле. PEP 8 — про читаемость, а не про правильность расчёта.
