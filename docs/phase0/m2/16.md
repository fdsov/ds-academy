# M2.16 · Стандартная библиотека для DS

Перед тем как ставить pandas, знай, что в стандартной библиотеке уже лежит много нужного. Это работает без зависимостей и часто быстрее на простых задачах.

Почему это важно на практике: типичный ETL-скрипт аналитика сначала **читает сырые логи** (json, построчно), **группирует и считает** (collections, itertools), **режет по датам** (datetime) и **раскладывает по файлам** (pathlib). На этом маршруте pandas нужен далеко не всегда — а лишняя зависимость замедляет старт скрипта, раздувает Docker-образ и усложняет деплой. Стандартная библиотека уже установлена везде, где есть Python.

### collections

`Counter` — подсчёт частот за одну строку. `defaultdict` — словарь с автозначением, избавляет от проверок «есть ли ключ». `deque` — очередь с быстрыми операциями с обоих концов.

```python
from collections import Counter, defaultdict

bets_by_game = Counter(b.game for b in bets)        # {"slots": 1200, "roulette": 340, ...}
top3 = bets_by_game.most_common(3)

deposits = defaultdict(list)
for d in all_deposits:
    deposits[d.player_id].append(d.amount)          # без проверки на существование ключа
```

**На пальцах: что именно убирает `defaultdict`.** Без него типичная группировка выглядит так:

```python
deposits = {}
for d in all_deposits:
    if d.player_id not in deposits:                 # ручная проверка на каждой итерации
        deposits[d.player_id] = []
    deposits[d.player_id].append(d.amount)
```

`defaultdict(list)` означает «если ключа нет — создай его со значением `list()`, то есть пустой список». Фабрика передаётся **без скобок** (`list`, а не `list()`): это функция, которую словарь сам вызовет при промахе. Частые фабрики: `defaultdict(int)` для счётчиков (`d[k] += 1`), `defaultdict(list)` для группировки, `defaultdict(set)` для уникальных значений.

**`Counter` — это специализированный `defaultdict(int)` с бонусами.** Он умеет складываться, вычитаться и сразу выдаёт топ:

```python
c1 = Counter(["slots", "slots", "roulette"])
c2 = Counter(["slots", "blackjack"])
c1 + c2                       # Counter({'slots': 3, 'roulette': 1, 'blackjack': 1})
c1.most_common(1)             # [('slots', 2)]
c1["poker"]                   # 0 — отсутствующий ключ даёт 0, не KeyError
```

Типичная ошибка: `most_common()` без аргумента возвращает **все** элементы, отсортированные по убыванию частоты — на миллионе уникальных игр это медленно и не нужно. Передавай число: `most_common(3)`.

**`deque` — когда нужна очередь.** Удаление из начала обычного списка (`list.pop(0)`) стоит $O(n)$, потому что все элементы сдвигаются. У `deque` оба конца — $O(1)$. Удобно для скользящего окна: «последние N событий игрока».

```python
from collections import deque

last5 = deque(maxlen=5)        # автоматически выкидывает самый старый при переполнении
for bet in player_bets:
    last5.append(bet.amount)   # всегда держим только 5 последних ставок
```

### itertools

Инструменты для эффективной работы с итераторами, всё ленивое.

```python
from itertools import groupby, chain, islice, accumulate

cumulative_ggr = list(accumulate(daily_ggr))        # нарастающий итог
first_100 = list(islice(huge_generator, 100))       # первые 100 без материализации всего
all_events = chain(deposits, withdrawals, bets)     # последовательная склейка потоков
```

**«Ленивое» — что это значит в деньгах памяти.** Если у тебя лог на 50 млн событий по 200 байт, материализовать его в список — это ~10 ГБ RAM. `islice(gen, 100)` пройдёт ровно 100 элементов и остановится, не тронув остальное. `chain` склеивает три потока, не создавая промежуточный объединённый список. Это разница между «скрипт упал по памяти» и «скрипт отработал на ноутбуке».

**Разобранный пример `accumulate`** — нарастающий GGR по дням:

```python
daily_ggr = [1200, -300, 800, 500]
list(accumulate(daily_ggr))    # [1200, 900, 1700, 2200]
```

Каждый элемент — сумма всех предыдущих включительно. Второй день минусовой (игроки выиграли), и накопленный итог просел с 1200 до 900, но потом восстановился. Можно передать свою операцию: `accumulate(values, max)` даст «текущий максимум» (полезно для drawdown-анализа), `accumulate(values, operator.mul)` — произведение (множители ставок).

**`groupby` — главная ловушка новичка.** Он группирует только **подряд идущие** одинаковые ключи, а не как SQL `GROUP BY`. Перед `groupby` данные нужно отсортировать по тому же ключу, иначе одна «слотс»-группа в начале и в конце станут двумя разными группами:

```python
from itertools import groupby

bets = sorted(bets, key=lambda b: b.game)           # ОБЯЗАТЕЛЬНО сначала сортировка
for game, group in groupby(bets, key=lambda b: b.game):
    total = sum(b.amount for b in group)            # group — одноразовый итератор!
    print(game, total)
```

Вторая ловушка: `group` — это итератор, его можно пройти **только один раз**. Если тебе нужны и сумма, и количество — материализуй: `items = list(group)`.

### datetime

Работа с датами и временем — основа когортного анализа.

```python
from datetime import datetime, timedelta, date

reg = datetime(2026, 6, 1)
day7 = reg + timedelta(days=7)                       # граница retention D7
age_days = (datetime.now() - reg).days
cohort_month = reg.strftime("%Y-%m")                 # "2026-06"
```

**Арифметика дат через `timedelta`.** Вычитание двух `datetime` даёт объект `timedelta`, у которого есть `.days` и `.total_seconds()`. Это и есть «возраст игрока» или «лаг между регистрацией и первым депозитом» (time-to-FTD — ключевая метрика воронки).

```python
ftd_lag = (first_deposit_dt - reg).total_seconds() / 3600   # в часах
```

**`strftime` / `strptime` — две стороны одной медали.** `strftime` (string **f**rom time) форматирует дату в строку, `strptime` (string **p**arse time) разбирает строку обратно в дату. Когортная метка месяца через `strftime("%Y-%m")` — стандартный способ свести игроков в месячные когорты.

```python
raw = "2026-06-01 14:30:00"
dt = datetime.strptime(raw, "%Y-%m-%d %H:%M:%S")     # строка из лога -> datetime
```

**Пограничные случаи, на которых горят аналитики:**

- **Naive vs aware.** `datetime.now()` без таймзоны — «наивная» дата. Если логи в UTC, а ты считаешь `now()` в локальной зоне сервера, retention D1 может «сдвинуться» на несколько часов и испортить когорту. Для серьёзных расчётов фиксируй таймзону: `datetime.now(timezone.utc)`.
- **`date` vs `datetime`.** Для когорт по дням обычно достаточно `date` (без времени). Сравнивать `date` и `datetime` напрямую нельзя — будет `TypeError`. Приводи к одному типу: `reg.date()`.
- **D7 — это граница, а не день.** `reg + timedelta(days=7)` даёт момент через 7×24 часа. «Retention D7» обычно считают как «был ли вход в день с номером 7», что требует аккуратности с границами суток.

### pathlib

Современная работа с путями — объектная, кроссплатформенная. Заменяет старый `os.path`.

```python
from pathlib import Path

data_dir = Path("data") / "raw"                      # склейка через /
for f in data_dir.glob("*.jsonl"):                   # все jsonl-файлы
    text = f.read_text()
Path("out").mkdir(exist_ok=True)                     # создать папку, не падать если есть
```

**Почему `/` вместо конкатенации строк.** На Windows разделитель `\`, на Linux/Mac — `/`. Если писать пути как `"data" + "/" + "raw"`, скрипт сломается при переносе. `Path("data") / "raw"` сам подставит правильный разделитель. Это и есть «кроссплатформенность».

**Полезные атрибуты и методы, которые экономят строки:**

```python
p = Path("data/raw/bets_2026-06.jsonl")
p.name        # 'bets_2026-06.jsonl'  — имя с расширением
p.stem        # 'bets_2026-06'        — имя без расширения
p.suffix      # '.jsonl'              — расширение
p.parent      # Path('data/raw')      — родительская папка
p.exists()    # True/False            — проверка существования
```

Ловушка: `mkdir()` без `exist_ok=True` падает с `FileExistsError`, если папка уже есть; а без `parents=True` — с `FileNotFoundError`, если нет промежуточных папок. Безопасный идиом для подготовки выходной директории: `Path("out/reports").mkdir(parents=True, exist_ok=True)`.

И `glob("*.jsonl")` берёт файлы только в текущей папке; для рекурсивного обхода всех подпапок используй `rglob("*.jsonl")` или `glob("**/*.jsonl")`.

### json

Сериализация в JSON и обратно — формат логов событий и API.

```python
import json

event = json.loads('{"type": "deposit", "amount": 500}')   # строка -> dict
line = json.dumps({"player": 101, "ltv": 1234.5}, ensure_ascii=False)
```

**Мнемоника `loads`/`dumps`.** Буква **s** на конце = **s**tring (работа со строкой в памяти). `loads` — **load s**tring (строка → объект), `dumps` — **dump s**tring (объект → строка). Без `s` (`json.load(file)` / `json.dump(obj, file)`) работают сразу с файловым объектом.

**Разбор реального лога событий (JSON Lines).** В гемблинге события часто пишут построчно — один JSON-объект на строку (`.jsonl`). Это удобно для стриминга: не нужно держать весь файл в памяти.

```python
from collections import Counter
import json

ggr_by_game = Counter()
with open("events.jsonl") as f:
    for line in f:                                  # ленивое построчное чтение
        e = json.loads(line)                        # каждая строка — отдельный JSON
        if e["type"] == "bet":
            ggr_by_game[e["game"]] += e["bet"] - e["payout"]
```

**Важные флаги и подводные камни:**

- `ensure_ascii=False` — без него кириллица и эмодзи кодируются в `\uXXXX` (`"слоты"` → `"сл..."`). Для читаемых логов всегда ставь `False`.
- `json.dumps` не умеет сериализовать `datetime`, `set`, `Decimal` — упадёт с `TypeError`. Приводи вручную (`dt.isoformat()`) или передавай `default=str`.
- JSON хранит числа как float. Деньги в float опасны (`0.1 + 0.2 != 0.3`); для точных сумм в проде используют целые «копейки/центы» или `Decimal`, а JSON — только как транспорт.
