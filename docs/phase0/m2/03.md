# M2.3 · Модель данных: имена, объекты и (не)изменяемость

Прежде чем говорить о типах, надо понять одну вещь, которую пропускают почти все самоучки и которая порождает половину их багов.

Интуиция. В Python переменная — это не «коробка, в которую кладут значение». Это ярлык (имя), приклеенный к объекту, который живёт где-то в памяти. Когда ты пишешь `a = [1, 2, 3]`, ты создал список-объект и приклеил к нему ярлык `a`. Если потом написать `b = a`, ты приклеил второй ярлык к тому же самому списку. Это не копия. Поменяешь через `a` — увидишь изменение и через `b`, потому что объект один.

Формально. Каждый объект имеет идентичность (адрес в памяти, его показывает `id()`), тип и значение. Присваивание связывает имя со ссылкой на объект, оно никогда не копирует объект. Оператор `is` сравнивает идентичность (тот же объект?), `==` сравнивает значение (равны ли по содержимому?).

```python
deposits = [100, 250, 500]
backup = deposits          # тот же объект, не копия
backup.append(999)
print(deposits)            # [100, 250, 500, 999] — изменилось через другой ярлык
print(deposits is backup)  # True

real_copy = deposits.copy()   # новый объект с тем же содержимым
print(deposits is real_copy)  # False
print(deposits == real_copy)  # True
```

Разбор примера по шагам. Строка `backup = deposits` не создаёт второй список — она вешает второе имя на тот же объект. Поэтому `backup.append(999)` достраивает единственный список, и `deposits` «видит» это изменение: ярлыков два, объект один. А `deposits.copy()` физически создаёт новый объект с теми же элементами — отсюда `is` даёт `False` (разные объекты), но `==` даёт `True` (одинаковое содержимое).

Тут вступает ключевое различие. Объекты делятся на изменяемые (mutable) и неизменяемые (immutable).

- Неизменяемые: `int`, `float`, `str`, `bool`, `tuple`, `frozenset`, `bytes`. Их нельзя поменять «на месте»; любая операция создаёт новый объект.
- Изменяемые: `list`, `dict`, `set`, и почти все пользовательские объекты. Их содержимое можно менять, идентичность при этом сохраняется.

Что значит «неизменяемый» на пальцах. Когда ты пишешь `s = "ru"; s = s + "_vip"`, кажется, что строка изменилась. На самом деле Python создал новую строку `"ru_vip"` и перевесил ярлык `s` на неё; старая `"ru"` осталась нетронутой (и будет удалена сборщиком мусора, если на неё больше никто не ссылается). То же с числами: `x = 100; x += 50` не меняет объект `100`, а создаёт объект `150` и переклеивает имя. Поэтому фраза «строки и числа неизменяемы» не мешает делать `x += 1` — меняется имя, а не объект.

```python
total = 1000          # объект int со значением 1000
print(id(total))      # некий адрес A
total += 250          # создаётся НОВЫЙ объект 1250
print(id(total))      # адрес B != A — это другой объект

geo = ("RU", "2026-06")
# geo[0] = "KZ"       # TypeError: 'tuple' object does not support item assignment
```

Почему это важно для аналитика. Неизменяемость — это гарантия безопасности. Кортеж координат `(lat, lon)` или ключ когорты `("RU", "2026-06")` нельзя случайно испортить. Поэтому только неизменяемые объекты можно класть в `set` или использовать ключами `dict` — система должна быть уверена, что хеш объекта не поменяется под ней.

Боевой пример: общий мутабельный объект как источник бага. Представь, что считаешь депозиты по когортам. Ты завёл «пустой» шаблон строки отчёта и переиспользуешь его:

```python
template = {"deposits": [], "ftd": 0}

ru = template          # НЕ копия — тот же объект
kz = template          # тоже тот же объект
ru["deposits"].append(100)
kz["deposits"].append(250)

print(ru["deposits"])  # [100, 250] — депозиты RU и KZ слиплись!
print(ru is kz)        # True
```

Оба «региона» делят один словарь, и цифры RU перемешиваются с KZ. Это классическая утечка состояния через изменяемый объект. Правильно — создавать новый объект на каждую когорту: `dict(deposits=[], ftd=0)` внутри цикла, либо глубокую копию через `copy.deepcopy(template)` (обычный `.copy()` здесь не спасёт — он копирует верхний словарь, но вложенный список `deposits` останется общим; это «поверхностная» vs «глубокая» копия).

Граничный случай: кортеж неизменяем, но может содержать изменяемое. `t = ([1, 2], 3)` — сам кортеж переприсвоить нельзя (`t[0] = ...` упадёт), но `t[0].append(99)` сработает, потому что внутри лежит живой список. Из-за этого такой кортеж нельзя положить в `set` или сделать ключом `dict`: `hash(t)` бросит `TypeError: unhashable type: 'list'`. Хешируемость требует, чтобы неизменяемым было всё «насквозь».

!!! warning "Главная ловушка новичка"

    `a == b` не значит `a is b`. И наоборот. Для маленьких целых чисел и коротких строк Python кеширует объекты, поэтому `x = 5; y = 5; x is y` даёт `True` — но это деталь реализации (CPython), а НЕ правило языка. Никогда не сравнивай числа или строки через `is`. `is` используй только для `None`, `True`, `False`.

!!! question "Проверь себя"

    1. Что именно делает присваивание `b = a` для списка — копирует объект или связывает второе имя с тем же объектом?
    2. Чем отличается `is` от `==`? Для каких трёх значений `is` уместен?
    3. Почему ключом `dict` или элементом `set` может быть только неизменяемый объект?
    4. Назови по три изменяемых и три неизменяемых типа.

??? success "Ответы"

    1. Связывает второе имя (ярлык) с тем же самым объектом. Это не копия: изменение через любое имя видно через оба. Копия — `a.copy()` или `a[:]`.
    2. `is` сравнивает идентичность (тот же ли это объект в памяти), `==` сравнивает значение (равно ли содержимое). `is` уместен только для `None`, `True`, `False`.
    3. Потому что коллекция полагается на хеш, а хеш не должен меняться под ней. У изменяемого объекта содержимое (и хеш) можно поменять — это сломало бы поиск.
    4. Изменяемые: `list`, `dict`, `set`. Неизменяемые: `int`, `str`, `tuple` (а также `float`, `bool`, `frozenset`, `bytes`).

!!! example "Мини-задание (3 минуты)"

    В интерпретаторе выполни:

    ```python
    a = [1, 2, 3]
    b = a
    c = a.copy()
    print(id(a) == id(b), id(a) == id(c))
    b.append(99)
    print(a, c)
    ```

    Предскажи вывод до запуска, затем сверь. Объясни, почему `a` изменился, а `c` — нет.
