# M2.14 · Аннотации типов (type hints)

Интуиция. Аннотации — это подписи к коду: «эта функция принимает список чисел и возвращает float». Python их не проверяет в рантайме, но они нужны людям и инструментам: IDE подсказывает, статический анализатор ловит ошибки до запуска.

Формально. Синтаксис `def f(x: int) -> str`. С Python 3.9+ встроенные дженерики пишутся как `list[int]`, `dict[str, float]`. Проверяются статически инструментами `mypy` или `pyright`, в рантайме игнорируются.

```python
def aggregate_ltv(players: list[Player], market: str | None = None) -> dict[str, float]:
    result: dict[str, float] = {}
    for p in players:
        if market is None or p.market == market:
            result[p.id] = p.ltv()
    return result
```

В DS аннотации особенно ценны на границах: сигнатуры функций, которые принимают DataFrame и что-то возвращают. Они документируют контракт лучше любого комментария и не устаревают незаметно, потому что их проверяет линтер.

## На пальцах: подпись, а не охрана

Главное недопонимание новичков: они думают, что аннотация — это «защита», которая не пустит неправильный аргумент. Это не так. Python — динамический язык, и в рантайме он полностью игнорирует то, что вы написали после двоеточия. Вот доказательство:

```python
def deposit_count(player_id: int) -> int:
    return player_id  # возвращаем что дали

print(deposit_count("не число"))   # выведет: не число
print(deposit_count([1, 2, 3]))    # выведет: [1, 2, 3]
```

Код выполнится без единой ошибки, хотя мы передали строку и список туда, где обещали `int`. Аннотация — это документация, читаемая машиной, а не проверка. Проверку делает отдельный инструмент (`mypy`/`pyright`), который читает ваш код глазами, но не запускает его.

Аналогия: аннотация — это надпись «осторожно, горячо» на чашке. Надпись не остудит кофе и не помешает вам обжечься. Но она предупреждает читателя и позволяет роботу-сортировщику (линтеру) отбраковать чашку до того, как её возьмут в руки.

## Что ловит статическая проверка

Запустим `mypy` на типичной ошибке:

```python
def hold_pct(ggr: float, bet_volume: float) -> float:
    return ggr / bet_volume

rate = hold_pct(1500, 0)        # ок по типам
report: str = hold_pct(1500, 50000)   # ошибка: float присвоен в str
```

`mypy` на последней строке скажет примерно: `error: Incompatible types in assignment (expression has type "float", variable has type "str")`. Это найдено **до запуска**, без единого деления, без тестовых данных. В аналитике это бесценно: функция, считающая GGR по когорте, может вызываться в десятке мест, и опечатка в типе всплыла бы только на проде через час расчёта.

Важная граница: `mypy` проверяет согласованность типов, но **не** проверяет значения. Деление на ноль во второй строке (`bet_volume = 0`) он не поймает — это рантайм-ошибка логики, а не типов. Type hints защищают от «передал не то», но не от «передал то, но плохое».

## Базовый словарь аннотаций

```python
from typing import Optional, Union, Callable, Any

x: int = 5
name: str = "vip_player"
ratio: float = 0.42
active: bool = True

ids: list[int] = [101, 102]
profile: dict[str, float] = {"ltv": 220.0, "ggr": 90.0}
pair: tuple[str, int] = ("UZ", 3)
markets: set[str] = {"UZ", "KZ"}

# "значение либо float, либо отсутствует"
deposit: float | None = None          # современный синтаксис (3.10+)
deposit2: Optional[float] = None       # старый синтаксис, то же самое

# "либо int, либо str"
key: int | str = "abc"                 # = Union[int, str]

# функция как аргумент
metric_fn: Callable[[float, float], float]  # принимает два float, возвращает float
```

Запомните эквивалентности: `X | None` ≡ `Optional[X]`, `A | B` ≡ `Union[A, B]`. Современный код предпочитает синтаксис с `|` (Python 3.10+), но в чужих кодовых базах вы постоянно встретите `Optional`/`Union` из модуля `typing`.

`Any` — это «отключить проверку для этого места». Полезно как заплатка, но злоупотребление `Any` превращает type hints в декорацию: проверять становится нечего.

## Численный пример из гемблинг-аналитики

Допустим, считаем средний депозит игрока в долларах с защитой от пустого списка:

```python
def avg_deposit(amounts: list[float]) -> float | None:
    if not amounts:
        return None
    return sum(amounts) / len(amounts)

vip = avg_deposit([120.0, 340.0, 80.0])   # 180.0
new = avg_deposit([])                       # None
```

Аннотация `-> float | None` честно сообщает вызывающему коду: «я могу вернуть None, обработай этот случай». Без неё легко написать `avg_deposit([]) * 30` и получить `TypeError` в рантайме. С аннотацией `mypy` заранее скажет: `error: Unsupported operand types for * ("None" and "int")` — потому что увидит, что результат может быть `None`. Это и есть главная ценность `Optional` в аналитике: он делает «пустую когорту» видимой на уровне типов.

## Типичные ошибки

- **Изменяемый дефолт.** `def f(items: list[int] = [])` — аннотация верна, но сам дефолт-список общий между вызовами (классическая ловушка Python). Используйте `items: list[int] | None = None` и `if items is None: items = []`.
- **`list` без параметра.** `def f(xs: list)` технически работает, но `mypy` не знает, что внутри. Пишите `list[int]`, иначе теряете половину пользы.
- **Путаница `Optional[X]` и «опциональный аргумент».** `Optional[float]` означает «значение может быть None», а не «аргумент можно не передавать». Опциональность аргумента задаётся дефолтом (`= None`), а `Optional` — только про тип значения. Часто идут вместе, но это разные вещи.
- **Вера в рантайм-защиту.** Повторим: без `mypy`/`pyright` аннотации не делают ничего. Если в проекте не настроен статический анализатор, type hints полезны только как документация для людей и IDE.
- **Слишком много `Any`.** Один `Any` на входе «заражает» всё, что от него зависит — проверка тихо отключается по цепочке.

!!! tip "Как включить проверку"

    Установите `mypy` (`pip install mypy`) и запустите `mypy your_file.py`. В IDE вроде VS Code с расширением Pylance проверка `pyright` работает на лету — подчёркивает несовпадения типов прямо в редакторе, без отдельного запуска.

!!! note "Проверь себя"

    Что выведет `print(deposit_count("hello"))` для функции `def deposit_count(player_id: int) -> int: return player_id`? Ответ: `hello` — рантайм игнорирует аннотацию, ошибку показал бы только `mypy`.
