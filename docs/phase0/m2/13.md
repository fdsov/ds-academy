# M2.13 · Обработка исключений

Интуиция. Исключение — это «что-то пошло не так, нормальный ход прерван». Вместо того чтобы программа упала, ты ловишь ошибку и решаешь, что делать: пропустить строку, подставить дефолт, залогировать.

На пальцах. Представь конвейер обработки выгрузки депозитов: миллион строк, и в 37-й попадается `amount = "—"` вместо числа. Без обработки исключений весь скрипт падает на этой строке, и ты теряешь обработку остального миллиона. С обработкой — ловишь ошибку именно на проблемной строке, помечаешь её как «битую», и конвейер едет дальше. Исключения превращают «всё или ничего» в «деградируй контролируемо».

Формально. `try` — рискованный код, `except КонкретныйТип` — обработка конкретной ошибки, `else` — если ошибки не было, `finally` — выполнится всегда. Лови конкретные исключения, не голый `except`.

```python
def safe_ratio(payout, stake):
    try:
        return payout / stake
    except ZeroDivisionError:
        return 0.0                    # ставка нулевая — осмысленный дефолт
    except TypeError:
        raise ValueError(f"нечисловые данные: {payout}, {stake}")
```

## Полный синтаксис: try / except / else / finally

Четыре блока работают вместе. Важно понимать, в какой момент срабатывает каждый.

```python
import logging

def parse_bet_amount(raw):
    try:
        value = float(raw)          # рискованная операция
    except ValueError:
        logging.warning("битая ставка: %r", raw)
        return None                 # обработка ошибки
    else:
        # выполнится ТОЛЬКО если try прошёл без исключения
        logging.info("ставка распознана: %.2f", value)
        return value
    finally:
        # выполнится ВСЕГДА: и при успехе, и при ошибке, и при return
        logging.debug("parse_bet_amount завершён для %r", raw)
```

Зачем нужен `else`, если код можно положить в конец `try`? Чтобы сузить зону риска. В `try` должна быть только та строка, которая реально может бросить исключение. Всё, что должно выполниться при успехе, идёт в `else` — тогда случайное исключение в «успешной» логике не будет ошибочно поймано тем же `except`. Это защищает от ложных срабатываний.

`finally` нужен для гарантированной уборки: закрыть файл, вернуть соединение в пул, снять блокировку. Он отрабатывает, даже если внутри `try` стоит `return` или летит непойманное исключение.

## Иерархия исключений и порядок except

Все исключения наследуются от `BaseException`, большинство — от `Exception`. `except` ловит указанный класс и всех его потомков. Поэтому порядок блоков важен: ставь специфичные раньше общих.

```python
try:
    risky()
except ZeroDivisionError:      # специфичный — первым
    handle_zero()
except ArithmeticError:        # ZeroDivisionError — потомок ArithmeticError
    handle_math()
except Exception:             # самый общий — последним
    handle_other()
```

Если поставить `except Exception` первым, он перехватит всё, и до `ZeroDivisionError` управление никогда не дойдёт — специфичная ветка станет мёртвым кодом.

## Численный пример: агрегируем GGR по строкам выгрузки

Покажем, как обработка исключений спасает массовый расчёт. Считаем суммарный GGR (gross gaming revenue = ставки минус выплаты) по списку игроков, где данные грязные.

```python
rows = [
    {"player": 1, "bets": "100", "payout": "60"},   # GGR = 40
    {"player": 2, "bets": "200", "payout": "—"},    # битый payout
    {"player": 3, "bets": "50",  "payout": "0"},    # GGR = 50
    {"player": 4, "bets": None,  "payout": "10"},   # bets = None
]

total_ggr = 0.0
bad_rows = []
for r in rows:
    try:
        ggr = float(r["bets"]) - float(r["payout"])
    except (ValueError, TypeError) as e:
        bad_rows.append((r["player"], str(e)))
        continue                  # пропускаем битую строку, не падаем
    total_ggr += ggr

print(total_ggr)   # 90.0  (40 + 50)
print(bad_rows)    # [(2, "could not convert..."), (4, "float() argument...")]
```

Без `try/except` скрипт упал бы на игроке 2, и ты не узнал бы GGR ни по кому. С обработкой — получил $90 по чистым строкам и отдельный список из двух битых, которые отдашь дата-инженеру. Контролируемая деградация вместо полной остановки.

Обрати внимание на `except (ValueError, TypeError)` — кортеж типов ловит несколько классов одним блоком, когда обработка для них одинаковая.

## raise и проброс: не теряй причину

`raise` бросает исключение. Внутри `except` можно перебросить дальше — либо тот же объект, либо новый, более осмысленный для вызывающего кода.

```python
def load_config(path):
    try:
        return read_file(path)
    except FileNotFoundError as e:
        # оборачиваем в доменную ошибку, сохраняя первопричину через "from"
        raise RuntimeError(f"конфиг не найден: {path}") from e
```

Конструкция `raise ... from e` сохраняет цепочку: в трейсбэке будет видно и твою `RuntimeError`, и исходную `FileNotFoundError`. Без `from` причина может потеряться или показаться «во время обработки одного исключения возникло другое», что путает при отладке.

!!! warning "Не глотай исключения молча"

    `except: pass` — это профессиональное преступление. Ты прячешь баг, который всплывёт в проде искажёнными цифрами. Лови конкретный тип, логируй, и если не знаешь, что делать — пробрасывай дальше через `raise`.

## Типичные ошибки

- Голый `except:` (или `except Exception:` ради «чтобы не падало») — ловит вообще всё, включая опечатки в именах переменных (`NameError`) и `KeyboardInterrupt`. Баг маскируется под «обработанную ошибку».
- `except: pass` в цикле обработки данных — самая дорогая ошибка в аналитике: цифры тихо становятся неверными, и ты узнаёшь об этом из претензии бизнеса, а не из лога.
- Слишком широкий `try` — десять строк в блоке, и ты не знаешь, какая из них бросила исключение. Держи в `try` минимум.
- Ловля исключения, с которым ты ничего не можешь сделать. Если не знаешь, как восстановиться, — не лови, дай упасть или пробрось наверх к тому, кто знает.
- `return` внутри `try`, а ресурс закрываешь в `try` после `return` — закрытие не выполнится. Используй `finally` или контекстный менеджер `with`.

!!! note "Связь с with"

    Для закрытия файлов и соединений чаще используют менеджер контекста `with` — он сам вызывает уборку, как `finally`, но короче и надёжнее. `try/finally` остаётся для случаев, где `with` не подходит.
