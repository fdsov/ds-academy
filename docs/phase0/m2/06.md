# M2.6 · Comprehensions: list, dict, set, generator

Это визитная карточка питоничного кода. Где новичок пишет цикл с `append`, профессионал пишет одну строку.

Интуиция. Comprehension — это «собери новую коллекцию из старой по правилу». Читается как фраза на английском: «возьми x для каждого x из данных, где выполнено условие».

На пальцах. Любой comprehension разворачивается в один и тот же скелет из трёх частей: что складываем (выражение), откуда берём (`for ... in ...`), что пропускаем (`if ...`). Сравни два варианта одного и того же:

```python
# обычный цикл — четыре строки, заводим пустой список вручную
wins = []
for b in bets:
    if b.outcome == "win":
        wins.append(b.amount)

# тот же смысл одной строкой
wins = [b.amount for b in bets if b.outcome == "win"]
```

Порядок чтения слева направо совпадает с порядком в цикле: сначала `for b in bets` (внешний цикл), потом `if b.outcome == "win"` (фильтр), и только то, что прошло фильтр, попадает в результат через `b.amount`. Выражение слева пишется первым, но вычисляется последним — это единственное место, где порядок «ломается».

Формально. Выражение `[выражение for элемент in итерабельное if условие]`. Эквивалентно циклу с накоплением, но быстрее (оптимизировано в интерпретаторе — не вызывается метод `.append` на каждой итерации) и читается одним взглядом.

```python
# list comprehension: суммы выигрышных ставок
wins = [b.amount for b in bets if b.outcome == "win"]

# dict comprehension: id игрока -> его LTV
ltv = {p.id: p.deposits - p.withdrawals for p in players}

# set comprehension: уникальные страны
countries = {p.country for p in players}

# generator expression: ленивая сумма, без построения списка в памяти
total_ggr = sum(b.stake - b.payout for b in all_bets)
```

Четыре скобки — четыре разных результата. Запомни по виду скобок:

| Синтаксис | Что строит | Тип результата |
|-----------|-----------|----------------|
| `[x for x in d]` | список | `list` |
| `{x for x in d}` | множество (уникальные) | `set` |
| `{k: v for x in d}` | словарь (есть `:`) | `dict` |
| `(x for x in d)` | генератор (ленивый) | `generator` |

Тонкость: фигурные скобки `{}` дают и set, и dict — различает их двоеточие. `{p.country for p in players}` — множество стран, а `{p.country: p.id for p in players}` — словарь. А пустые `{}` — это всегда dict, пустого set-литерала не существует, для него нужен `set()`.

Разница между скобками принципиальна. Квадратные `[]` строят список целиком в памяти. Круглые `()` создают генератор — он не хранит результат, а выдаёт элементы по одному по требованию. На миллиарде ставок `sum([... for ...])` съест всю память, а `sum(... for ...)` пройдёт потоком за константную память.

Численная прикидка в деньгах памяти. Допустим, у нас 1 млрд ставок и нам нужна сумма `stake`. Каждое число `float` в Python занимает ~24 байта, плюс список держит указатели по 8 байт на элемент. Список: $10^9 \times (24 + 8) \approx 32$ ГБ оперативки только чтобы потом сложить и выбросить. Генератор: десятки байт независимо от объёма — он держит в памяти ровно один текущий элемент и накопленную сумму. На больших выгрузках GGR это разница между «упало с MemoryError» и «посчиталось за минуту».

```python
import sys

# материализуем весь список — память растёт с числом элементов
big_list = [x for x in range(1_000_000)]
print(sys.getsizeof(big_list))   # ~8 МБ только на указатели

# генератор — постоянный крошечный размер
gen = (x for x in range(1_000_000))
print(sys.getsizeof(gen))        # ~200 байт, сколько бы ни было элементов
```

Когда генератор НЕ выручит. Генератор одноразовый: его можно пройти ровно один раз. После `sum(gen)` повторный `max(gen)` вернёт ошибку или пустоту — поток уже исчерпан. Если результат нужен дважды (и посчитать сумму, и максимум, и потом отфильтровать) — материализуй список один раз. Генератор хорош для «прочитал поток → свернул в одно число»; список — для «нужно переиспользовать или индексировать».

Типичные ошибки.

- Внутри генератора нельзя обращаться к нему по индексу: `gen[0]` упадёт, индексация есть только у списка.
- Двоеточие в фигурных скобках — это всегда dict; если хотел set уникальных пар, set из словарных пар не построишь, нужен другой подход.
- Переменная цикла не «утекает» наружу: в Python 3 `[i for i in range(5)]` не оставляет `i` в окружающем коде (в отличие от обычного цикла). Не рассчитывай на её значение после comprehension.

!!! warning "Не превращай comprehension в нечитаемое"

    Вложенные comprehensions с двумя-тремя условиями — это уже не питонично, а ребус. Если строка не читается за один взгляд, разверни в обычный цикл. Питоничность — про ясность, а не про «в одну строку любой ценой».

!!! question "Проверь себя"

    1. В чём принципиальная разница между `[x for x in data]` и `(x for x in data)` по памяти?
    2. Почему `sum(b.stake for b in all_bets)` на миллиарде ставок предпочтительнее, чем `sum([b.stake for b in all_bets])`?
    3. Когда comprehension стоит развернуть обратно в обычный цикл?

??? success "Ответы"

    1. Квадратные скобки строят список целиком в памяти (память = размер результата). Круглые создают генератор: элементы выдаются по одному по требованию, результат не хранится.
    2. Версия с генератором проходит данные потоком за константную память; версия со списком сначала материализует весь список и съест всю память.
    3. Когда читаемость страдает: вложенные comprehensions с двумя-тремя условиями превращаются в ребус. Питоничность — про ясность, а не «в одну строку любой ценой».
