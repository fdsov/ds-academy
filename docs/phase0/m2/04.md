# M2.4 · Коллекции: когда какую

Это вопрос, на котором сыпется большинство решений в DS-коде. Выбор коллекции — это выбор асимптотической сложности операций. Перепутал список с множеством — и отчёт, который должен считаться за секунду, считается полчаса. Разберём четыре основные.

Главный вопрос, который ты задаёшь себе при выборе: **что я буду делать с этими данными чаще всего?** Перебирать по порядку? Искать по ключу? Проверять «есть ли внутри»? Хранить уникальные? Ответ определяет структуру, а структура определяет скорость.

### list — упорядоченный изменяемый массив

Интуиция. Список — это пронумерованная полка: элементы лежат по порядку, к любому есть доступ по индексу, можно добавлять в конец, вставлять в середину. Это рабочая лошадка для последовательностей: список ставок игрока, список дневных депозитов.

Формально. Динамический массив. Доступ по индексу и добавление в конец — амортизированно $O(1)$. Вставка/удаление в начале или середине — $O(n)$, потому что элементы сдвигаются. Проверка `x in lst` — $O(n)$, перебор всех элементов.

```python
bets = [50, 120, 30, 200]
bets.append(75)        # O(1)
bets.insert(0, 10)     # O(n) — сдвиг всех
total = sum(bets)
```

Что значит «амортизированно $O(1)$». Под капотом список держит кусок памяти с запасом. Пока запас есть — `append` просто кладёт элемент в свободную ячейку за $O(1)$. Когда запас кончился — Python выделяет новый блок (обычно в ~2 раза больше), копирует туда всё старое за $O(n)$ и продолжает. Такое дорогое копирование случается редко, и если «размазать» его стоимость по всем `append`, в среднем выходит $O(1)$ на операцию. Отсюда слово «амортизированно» — отдельный `append` иногда дорогой, но в среднем дёшев.

Типичная ошибка. Накапливать результат через `result = result + [x]` в цикле — каждый раз создаётся новый список, это $O(n^2)$. Правильно — `result.append(x)`.

### tuple — упорядоченный неизменяемый кортеж

Интуиция. То же, что список, но «запаянный»: после создания не поменять. Используй, когда набор значений — это единое целое, которое не должно меняться: строка результата, ключ когорты, возврат нескольких значений из функции.

Формально. Неизменяемая последовательность. Чуть компактнее в памяти и быстрее в создании, чем list. Может быть ключом словаря и элементом множества (если все его элементы тоже неизменяемы).

```python
def cohort_stats(rows):
    return len(rows), sum(rows) / len(rows)   # возврат кортежа

count, avg = cohort_stats([100, 200, 300])    # распаковка
cohort_key = ("RU", "2026-06")                # годится в ключ dict
```

Почему именно неизменяемость даёт «ключ словаря». Словарь ищет значение по хешу ключа. Если бы ключ можно было поменять после вставки, его хеш изменился бы — и словарь «потерял» бы запись. Поэтому ключом может быть только то, что нельзя изменить: число, строка, tuple из неизменяемых элементов. List ключом быть не может — Python кинет `TypeError: unhashable type: 'list'`.

Гемблинг-пример. Когортный ключ `("RU", "2026-06", "mobile")` — гео, месяц регистрации, платформа. Это естественный составной ключ для словаря `revenue_by_cohort`. Сделать его списком нельзя — словарь его не примет.

Граничный случай. Кортеж из одного элемента пишется с запятой: `(5,)`, а не `(5)` — второе это просто число 5 в скобках.

### dict — хеш-таблица ключ→значение

Интуиция. Словарь — это телефонная книга: по ключу мгновенно находишь значение. Игрок по id → его профиль. Дата → суммарный GGR за день. Это самая важная структура в продуктовой аналитике после массива.

Формально. Хеш-таблица. Доступ, вставка, удаление по ключу — в среднем $O(1)$. Ключи должны быть хешируемыми (неизменяемыми). С Python 3.7+ словари гарантированно сохраняют порядок вставки.

```python
ggr_by_day = {}
for day, amount in transactions:
    ggr_by_day[day] = ggr_by_day.get(day, 0) + amount   # get с дефолтом — без KeyError
```

Разбор паттерна агрегации. Строка `ggr_by_day.get(day, 0) + amount` — это классический «group by» вручную. `get(day, 0)` возвращает текущую сумму по дню, а если дня ещё не было — `0` вместо падения с `KeyError`. Так за один проход по транзакциям ($O(n)$) мы получаем GGR по каждому дню. Альтернатива без `get`:

```python
from collections import defaultdict
ggr_by_day = defaultdict(int)        # отсутствующий ключ → 0 автоматически
for day, amount in transactions:
    ggr_by_day[day] += amount
```

Численный пример. Транзакции `[("06-01", 100), ("06-02", 50), ("06-01", 30)]` дадут `{"06-01": 130, "06-02": 50}`. Первая строка положила 100, вторая — 50 по новому дню, третья добавила 30 к уже существующим 130.

Типичная ошибка. Обращаться к `ggr_by_day[day]` напрямую, когда ключа ещё нет — это `KeyError`. Всегда либо `get` с дефолтом, либо `defaultdict`, либо проверка `if day in ggr_by_day`.

### set — множество уникальных элементов

Интуиция. Множество — это мешок без повторов, где главный вопрос «есть ли элемент внутри?». Идеально для дедупликации (уникальные игроки за день) и проверки членства (этот id в чёрном списке?).

Формально. Хеш-таблица без значений. Проверка `x in s`, добавление, удаление — в среднем $O(1)$. Поддерживает теоретико-множественные операции: объединение `|`, пересечение `&`, разность `-`.

```python
players_mon = {101, 102, 103}
players_tue = {102, 103, 104}
retained = players_mon & players_tue   # вернулись: {102, 103}
churned  = players_mon - players_tue   # ушли:     {101}
```

Дедупликация одной строкой. `unique_players = set(all_player_ids)` мгновенно убирает повторы — например, чтобы посчитать DAU из лога событий, где один игрок встречается много раз: `dau = len(set(player_ids_today))`.

Теоретико-множественные операции через метрики:

- `players_mon | players_tue` — все, кто заходил хотя бы в один из дней (для расчёта охвата).
- `players_mon & players_tue` — вернувшиеся (retention).
- `players_mon - players_tue` — отток за день (churn).
- `players_tue - players_mon` — новые во вторник (если в понедельник их не было).

Граничный случай. `{}` — это пустой **словарь**, а не множество. Пустое множество создаётся только через `set()`.

!!! tip "Правило выбора за 5 секунд"

    Нужен порядок и индексы, содержимое меняется → list. Фиксированная запись/ключ → tuple. Поиск по ключу → dict. Уникальность и «есть ли внутри» → set. Если ловишь себя на `x in some_list` внутри цикла по миллиону строк — это $O(n^2)$, замени list на set и получишь $O(n)$.

??? example "Сравнение сложности — почему set спасает retention-расчёт"

    Найти, сколько вчерашних игроков вернулись сегодня. Наивно через списки:

    ```python
    # O(n*m): для каждого вчерашнего ищем во всём сегодняшнем списке
    retained = [p for p in yesterday_list if p in today_list]
    ```

    На 500k игроков это часы. Через множество:

    ```python
    today_set = set(today_list)                    # O(m)
    retained = [p for p in yesterday_list if p in today_set]   # O(n), in — за O(1)
    ```

    Те же строки кода, но из $O(n \cdot m)$ получился $O(n + m)$. Это разница между «не дождался» и «мгновенно».

    Прикинем в числах. При $n = m = 500\,000$ наивный вариант делает $500\,000 \times 500\,000 = 2.5 \times 10^{11}$ проверок. Версия через set — около $10^6$ операций. Разница в ~250 000 раз: то, что считалось бы часами, считается за доли секунды. Стоимость подготовки — одна строка `set(today_list)`.

!!! question "Проверь себя"

    1. Какова средняя асимптотика проверки `x in coll` для list и для set? Почему они разные?
    2. Под какую задачу берут tuple, а не list? Назови минимум одно свойство tuple, которого нет у list.
    3. Какой структурой ты посчитаешь GGR по дням (день → сумма) и почему именно ей?
    4. Что вернут операции `players_mon & players_tue` и `players_mon - players_tue` по смыслу retention?

??? success "Ответы"

    1. Для list — $O(n)$ (линейный перебор), для set — в среднем $O(1)$ (хеш-таблица). Set хранит элементы по хешу и сразу проверяет нужную ячейку.
    2. tuple берут под фиксированную запись/ключ/возврат нескольких значений. Свойство: неизменяемость, поэтому tuple (из неизменяемых элементов) может быть ключом dict и элементом set.
    3. dict: ключ — день, значение — сумма. Доступ и обновление по ключу в среднем $O(1)$, порядок вставки сохраняется (3.7+).
    4. `&` (пересечение) — вернувшиеся игроки (были оба дня); `-` (разность) — ушедшие (были в понедельник, нет во вторник).

### Быстрый квиз

Выбери вариант — ответ подсветится сразу.

```quiz
Почему проверка `x in coll` для set в среднем $O(1)$, а для list — $O(n)$?
[ ] set всегда хранит меньше элементов, чем list
[x] set хранит элементы по хешу и сразу проверяет нужную ячейку, а list перебирает подряд
[ ] list не поддерживает оператор `in`
> Хеш-таблица вычисляет адрес ячейки из значения за константу; список вынужден сравнивать элемент за элементом.
---
Что произойдёт при попытке использовать список как ключ словаря: `d[[1, 2]] = 5`?
[ ] Сработает, список станет неизменяемым ключом
[x] Ошибка: список нехешируемый (unhashable type: 'list')
[ ] Список молча преобразуется в tuple
> Ключом может быть только неизменяемое: число, строка, tuple. Изменяемый список менял бы свой хеш и ломал словарь.
---
Какой структурой считать GGR по дням (день → сумма) за один проход по транзакциям?
[x] dict: ключ — день, значение — накопленная сумма, обновление за $O(1)$
[ ] list: добавлять суммы по индексу дня
[ ] set: класть туда суммы по дням
> Нужен поиск и обновление по ключу-дню — это ровно работа словаря. Set не хранит значений, list не индексируется датой.
---
Что вернёт `players_mon - players_tue` для `{101, 102, 103}` и `{102, 103, 104}`?
[ ] {104} — новые игроки вторника
[x] {101} — ушедшие: были в понедельник, нет во вторник
[ ] {102, 103} — вернувшиеся игроки
> Разность множеств оставляет элементы первого, которых нет во втором, — это отток за день.
---
Почему `append` в список называют амортизированно $O(1)$, а не строго $O(1)$?
[ ] Потому что append всегда копирует весь список
[x] Изредка при нехватке места список перевыделяется и копируется за $O(n)$, но в среднем по всем append выходит $O(1)$
[ ] Потому что append работает за $O(n)$ всегда
> Дорогое копирование случается редко; «размазав» его по всем вставкам, получаем константу в среднем.
```
