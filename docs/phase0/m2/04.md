# M2.4 · Коллекции: когда какую

Это вопрос, на котором сыпется большинство решений в DS-коде. Выбор коллекции — это выбор асимптотической сложности операций. Разберём четыре основные.

### list — упорядоченный изменяемый массив

Интуиция. Список — это пронумерованная полка: элементы лежат по порядку, к любому есть доступ по индексу, можно добавлять в конец, вставлять в середину. Это рабочая лошадка для последовательностей: список ставок игрока, список дневных депозитов.

Формально. Динамический массив. Доступ по индексу и добавление в конец — амортизированно $O(1)$. Вставка/удаление в начале или середине — $O(n)$, потому что элементы сдвигаются. Проверка `x in lst` — $O(n)$, перебор всех элементов.

```python
bets = [50, 120, 30, 200]
bets.append(75)        # O(1)
bets.insert(0, 10)     # O(n) — сдвиг всех
total = sum(bets)
```

### tuple — упорядоченный неизменяемый кортеж

Интуиция. То же, что список, но «запаянный»: после создания не поменять. Используй, когда набор значений — это единое целое, которое не должно меняться: строка результата, ключ когорты, возврат нескольких значений из функции.

Формально. Неизменяемая последовательность. Чуть компактнее в памяти и быстрее в создании, чем list. Может быть ключом словаря и элементом множества (если все его элементы тоже неизменяемы).

```python
def cohort_stats(rows):
    return len(rows), sum(rows) / len(rows)   # возврат кортежа

count, avg = cohort_stats([100, 200, 300])    # распаковка
cohort_key = ("RU", "2026-06")                # годится в ключ dict
```

### dict — хеш-таблица ключ→значение

Интуиция. Словарь — это телефонная книга: по ключу мгновенно находишь значение. Игрок по id → его профиль. Дата → суммарный GGR за день. Это самая важная структура в продуктовой аналитике после массива.

Формально. Хеш-таблица. Доступ, вставка, удаление по ключу — в среднем $O(1)$. Ключи должны быть хешируемыми (неизменяемыми). С Python 3.7+ словари гарантированно сохраняют порядок вставки.

```python
ggr_by_day = {}
for day, amount in transactions:
    ggr_by_day[day] = ggr_by_day.get(day, 0) + amount   # get с дефолтом — без KeyError
```

### set — множество уникальных элементов

Интуиция. Множество — это мешок без повторов, где главный вопрос «есть ли элемент внутри?». Идеально для дедупликации (уникальные игроки за день) и проверки членства (этот id в чёрном списке?).

Формально. Хеш-таблица без значений. Проверка `x in s`, добавление, удаление — в среднем $O(1)$. Поддерживает теоретико-множественные операции: объединение `|`, пересечение `&`, разность `-`.

```python
players_mon = {101, 102, 103}
players_tue = {102, 103, 104}
retained = players_mon & players_tue   # вернулись: {102, 103}
churned  = players_mon - players_tue   # ушли:     {101}
```

!!! tip "Правило выбора за 5 секунд"

    Нужен порядок и индексы, содержимое меняется → list. Фиксированная запись/ключ → tuple. Поиск по ключу → dict. Уникальность и «есть ли внутри» → set. Если ловишь себя на `x in some_list` внутри цикла по миллиону строк — это $O(n^2)$, замени list на set и получишь $O(n)$.

??? example "Сравнение сложности — почему set спасает retention-расчёт"

    Найти, сколько вчерашних игроков вернулись сегодня. Наивно через списки:

    ```python
    # O(n*m): для каждого вчерашнего ищем во всём сегодняшнем списке
    retained = [p for p in yesterday_list if p in today_list]
    ```

    На 500k игроков это часы. Через множество:

    ```python
    today_set = set(today_list)                    # O(m)
    retained = [p for p in yesterday_list if p in today_set]   # O(n), in — за O(1)
    ```

    Те же строки кода, но из $O(n \cdot m)$ получился $O(n + m)$. Это разница между «не дождался» и «мгновенно».

!!! question "Проверь себя"

    1. Какова средняя асимптотика проверки `x in coll` для list и для set? Почему они разные?
    2. Под какую задачу берут tuple, а не list? Назови минимум одно свойство tuple, которого нет у list.
    3. Какой структурой ты посчитаешь GGR по дням (день → сумма) и почему именно ей?
    4. Что вернут операции `players_mon & players_tue` и `players_mon - players_tue` по смыслу retention?

??? success "Ответы"

    1. Для list — $O(n)$ (линейный перебор), для set — в среднем $O(1)$ (хеш-таблица). Set хранит элементы по хешу и сразу проверяет нужную ячейку.
    2. tuple берут под фиксированную запись/ключ/возврат нескольких значений. Свойство: неизменяемость, поэтому tuple (из неизменяемых элементов) может быть ключом dict и элементом set.
    3. dict: ключ — день, значение — сумма. Доступ и обновление по ключу в среднем $O(1)$, порядок вставки сохраняется (3.7+).
    4. `&` (пересечение) — вернувшиеся игроки (были оба дня); `-` (разность) — ушедшие (были в понедельник, нет во вторник).
