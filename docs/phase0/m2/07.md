# M2.7 · Функции: args, kwargs, lambda

Функция — базовая единица переиспользования. Разберём то, что выходит за рамки «def с двумя аргументами».

### Позиционные, именованные, *args, **kwargs

Интуиция. `*args` — «сколько угодно безымянных аргументов, собери их в кортеж». `**kwargs` — «сколько угодно именованных, собери в словарь». Это нужно, когда заранее не знаешь, сколько параметров передадут.

Формально. При вызове `*` распаковывает итерабельное в позиционные аргументы, `**` распаковывает словарь в именованные. При определении функции они, наоборот, собирают лишние аргументы.

```python
def ggr(*stakes, hold=0.05, **meta):
    base = sum(stakes) * hold
    return base, meta

result, info = ggr(100, 200, 300, hold=0.04, market="UZ")
# stakes=(100,200,300), hold=0.04, meta={"market":"UZ"}
```

#### Порядок параметров — он строго фиксирован

В сигнатуре функции аргументы идут в одном и только одном порядке. Нарушишь — `SyntaxError` ещё до запуска:

```python
def f(pos, *args, kw_only, default=10, **kwargs): ...
#     │     │       │           │          └ всё остальное именованное → dict
#     │     │       │           └ именованный с дефолтом
#     │     │       └ keyword-only: после *args передаётся ТОЛЬКО по имени
#     │     └ лишние позиционные → tuple
#     └ обычный позиционный
```

Ключевой и неочевидный момент: всё, что объявлено **после** `*args`, становится **keyword-only** — такой параметр нельзя передать позиционно, только по имени. Именно поэтому в примере выше `hold` всегда задаётся как `hold=0.04`, а попытка `ggr(100, 200, 0.04)` молча отправит `0.04` в `stakes`, а не в `hold`. Это частая ловушка: думаешь, что переопределил hold, а на деле добавил ещё одну ставку.

#### Распаковка при вызове — обратная операция

`*` и `**` работают в обе стороны. При определении — собирают, при вызове — разбирают:

```python
stakes = [100, 200, 300]
config = {"hold": 0.04, "market": "UZ"}

ggr(*stakes, **config)   # эквивалент ggr(100, 200, 300, hold=0.04, market="UZ")
```

На пальцах. Это как «развернуть пакет»: список `stakes` рассыпается на три отдельных позиционных аргумента, словарь `config` — на два именованных. Очень удобно, когда параметры приходят из конфига или из JSON, а не зашиты в код.

#### Зачем это в аналитике

Типичный кейс — функция-обёртка, которая прозрачно прокидывает аргументы дальше, не зная их заранее. Например, тонкий враппер над `pandas.read_csv`, который добавляет логирование, но не хочет дублировать все 50 параметров pandas:

```python
def read_logged(path, *args, **kwargs):
    print(f"loading {path}")
    df = pd.read_csv(path, *args, **kwargs)
    print(f"{len(df)} rows")
    return df

# любой аргумент read_csv пройдёт насквозь:
read_logged("bets.csv", sep=";", parse_dates=["ts"], nrows=1000)
```

Без `*args, **kwargs` пришлось бы вручную перечислять каждый параметр `read_csv` — это и есть главная боль, которую они снимают.

### lambda — анонимная функция-однострочник

Интуиция. lambda — это функция без имени для одноразового использования, чаще всего как ключ сортировки или аргумент `map`/`filter`.

Формально. `lambda аргументы: выражение`. Возвращает значение выражения. Не может содержать операторов (только выражение), поэтому годится лишь для простой логики.

```python
players.sort(key=lambda p: p.ltv, reverse=True)   # сортировка по LTV убыванию
top_depositors = sorted(deposits, key=lambda d: d.amount)[-10:]
```

#### Что значит «ключ сортировки» на пальцах

`key=...` — это функция, которую Python вызывает для каждого элемента, чтобы получить значение, ПО которому сравнивать. Сам элемент при этом не меняется, сортируется он по результату функции.

```python
players = [
    {"id": 1, "ltv": 120.0, "country": "UZ"},
    {"id": 2, "ltv": 45.0,  "country": "KZ"},
    {"id": 3, "ltv": 300.0, "country": "UZ"},
]

# по убыванию LTV:
ranked = sorted(players, key=lambda p: p["ltv"], reverse=True)
# id=3 (300), id=1 (120), id=2 (45)

# сортировка по двум ключам сразу — кортеж: сначала страна, потом -LTV
ranked2 = sorted(players, key=lambda p: (p["country"], -p["ltv"]))
```

Трюк с `-p["ltv"]` внутри кортежа — стандартный приём, когда по одному полю нужно по возрастанию, а по другому по убыванию: `reverse=True` развернул бы оба, а минус разворачивает только числовое поле.

#### lambda против def — когда что

lambda не лучше и не хуже `def`, это просто компактная запись для случая «нужна функция прямо здесь и сейчас, одним выражением». Если выражение становится сложным или его хочется переиспользовать — пиши обычный `def` с именем: его проще читать, отлаживать и видеть в трейсбэке.

```python
# OK — короткий ключ:
sorted(deposits, key=lambda d: d.amount)

# ПЛОХО — нечитаемая lambda, лучше def:
key=lambda d: (d.amount * 0.95 if d.country == "UZ" else d.amount) - d.bonus

# лучше так:
def net_value(d):
    rate = 0.95 if d.country == "UZ" else 1.0
    return d.amount * rate - d.bonus

sorted(deposits, key=net_value)
```

!!! warning "lambda в цикле — поздняя привязка переменной"

    Lambda захватывает переменную по ИМЕНИ, а не её значение в момент создания. Все lambda, созданные в цикле, будут смотреть на финальное значение переменной.

    ```python
    funcs = [lambda: i for i in range(3)]
    [f() for f in funcs]   # [2, 2, 2] — НЕ [0, 1, 2]!
    ```

    Фикс — «заморозить» значение через аргумент по умолчанию:

    ```python
    funcs = [lambda i=i: i for i in range(3)]
    [f() for f in funcs]   # [0, 1, 2]
    ```

!!! warning "Изменяемый аргумент по умолчанию — классический баг сеньоров"

    Значение по умолчанию вычисляется ОДИН раз при определении функции, а не при каждом вызове. Поэтому изменяемый дефолт живёт между вызовами.

    ```python
    def add_bet(bet, history=[]):     # ОПАСНО
        history.append(bet)
        return history

    add_bet(10)   # [10]
    add_bet(20)   # [10, 20] — тот же список! не [20]
    ```

    Правильно — дефолт `None`, список создаём внутри:

    ```python
    def add_bet(bet, history=None):
        if history is None:
            history = []
        history.append(bet)
        return history
    ```

    Почему так. Дефолтные значения хранятся в атрибуте `add_bet.__defaults__` — это одни и те же объекты на всю жизнь функции. Список `[]` создаётся в момент `def`, и каждый вызов без аргумента переиспользует ровно его. Для неизменяемых дефолтов (`0`, `""`, `None`, кортеж) проблемы нет — их нельзя мутировать, поэтому общий объект безвреден. Опасны именно изменяемые: `list`, `dict`, `set`.

!!! question "Проверь себя"

    1. Что собирают `*args` и `**kwargs` при определении функции — в какие типы?
    2. Почему `def add_bet(bet, history=[])` опасен? Что происходит между вызовами?
    3. Как правильно задать изменяемый аргумент по умолчанию?
    4. Какое ограничение есть у `lambda` по сравнению с `def`?

??? success "Ответы"

    1. `*args` собирает лишние позиционные аргументы в кортеж (tuple), `**kwargs` — лишние именованные в словарь (dict).
    2. Значение по умолчанию вычисляется один раз при определении функции, а не при каждом вызове. Один и тот же список живёт между вызовами и накапливает данные.
    3. Дефолт `None`, а коллекцию создавать внутри: `if history is None: history = []`.
    4. `lambda` содержит только выражение (одно), в неё нельзя поместить операторы/несколько строк. Годится лишь для простой логики (ключ сортировки, аргумент map/filter).
