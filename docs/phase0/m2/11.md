# M2.11 · Генераторы, yield и ленивые вычисления

Это самая важная тема для работы с большими данными на чистом Python.

Интуиция. Обычная функция готовит всё блюдо и отдаёт целиком. Генератор — это конвейер: отдаёт по одному элементу, когда у него просят, и помнит, где остановился. Он не держит весь результат в памяти. На логе ставок в 50 ГБ это разница между «упало по памяти» и «обработалось потоком».

Формально. Функция с `yield` вместо `return` становится генератором. При вызове она не выполняется, а возвращает объект-генератор. Каждый `next()` выполняет код до следующего `yield`, отдаёт значение и замораживает состояние. Это ленивые вычисления: элемент производится только в момент запроса.

```python
def read_bets(path):
    with open(path) as f:
        for line in f:
            yield parse_bet(line)     # отдаёт по одной ставке, файл целиком в память не грузится

# обработка 50 ГБ за константную память
total = sum(b.stake for b in read_bets("bets.log") if b.stake > 0)
```

Сравни. `lines = f.readlines()` загрузит весь файл в список (память = размер файла). Генератор обрабатывает строку, выбрасывает, берёт следующую. Память почти не растёт.

На пальцах: что такое «состояние». Когда генератор доходит до `yield` и отдаёт значение, он не завершается, как при `return`, — он «засыпает» прямо на этой строке. Локальные переменные, позиция в цикле, открытый файл — всё сохраняется. На следующем `next()` он просыпается ровно там, где остановился, и идёт дальше до следующего `yield`. Когда `yield`-ов больше нет (функция дошла до конца), генератор бросает `StopIteration` — это сигнал, который `for` ловит молча и завершает цикл.

```python
def countdown(n):
    print("старт")
    while n > 0:
        yield n
        n -= 1

g = countdown(3)        # ничего не печатается — тело ещё не выполнялось
print(next(g))          # печатает "старт", потом 3
print(next(g))          # 2 (продолжил с замороженного состояния, "старт" НЕ печатается снова)
print(next(g))          # 1
# print(next(g))        # StopIteration — yield-ов больше нет
```

Численный пример экономии памяти. Лог ставок — 50 ГБ, 500 млн строк. Через `readlines()` нужно ~50 ГБ RAM плюс накладные расходы на список объektов — машина с 16 ГБ упадёт с `MemoryError`. Через генератор в памяти живёт одна распарсенная ставка (несколько сотен байт), и неважно, 50 ГБ файл или 5 ТБ — потребление константно. Это буквально граница между «задача решается на ноутбуке» и «нужен кластер».

Подводные камни. Генератор одноразовый: после того как ты прошёл по нему циклом, он исчерпан, повторный `for` даст ноль элементов.

```python
bets = read_bets("bets.log")
total = sum(b.stake for b in bets)      # прошли весь генератор
count = sum(1 for _ in bets)            # 0 — генератор уже исчерпан!
```

Если данные нужны дважды — либо создавай генератор заново (`read_bets(...)` ещё раз), либо материализуй в список (`list(...)`), но тогда теряешь экономию памяти. Вторая частая ошибка — `len(gen)`: у генератора нет длины, потому что он не знает, сколько элементов отдаст, пока не дойдёт до конца. Третья — попытка обратиться по индексу `gen[5]`: индексация не поддерживается, только последовательный обход.

??? example "Конвейер генераторов — потоковая обработка депозитов"

    Генераторы можно соединять в конвейер, и данные текут сквозь него по одному элементу, не материализуясь:

    ```python
    def parse(lines):
        for line in lines:
            yield json.loads(line)

    def only_deposits(events):
        for e in events:
            if e["type"] == "deposit":
                yield e

    def amounts(deposits):
        for d in deposits:
            yield d["amount"]

    with open("events.jsonl") as f:
        total = sum(amounts(only_deposits(parse(f))))
    # весь файл прошёл потоком: в памяти всегда одна запись
    ```

    Ключевая идея конвейера: `parse(f)` не парсит весь файл сразу. `sum` дёргает `amounts` за одним числом, тот дёргает `only_deposits` за одним событием, тот дёргает `parse` за одной распарсенной записью, тот читает одну строку из файла. Спрос идёт справа налево по одному элементу — как тяга воды по трубам. Поэтому добавить новый этап фильтрации (например, «только депозиты больше $50») — это ещё одна функция-генератор в цепочке, без роста памяти и без промежуточных списков.

??? tip "Генераторное выражение — короткая запись"

    Для простых случаев не нужна отдельная функция: генераторное выражение пишется как list comprehension, но в круглых скобках. Оно тоже ленивое.

    ```python
    # список: материализует ВСЕ суммы в память
    deposits_list = [e["amount"] for e in events if e["type"] == "deposit"]

    # генератор: ничего не материализует, считает потоком
    total = sum(e["amount"] for e in events if e["type"] == "deposit")
    ```

    Когда результат нужен только чтобы сразу его свернуть (`sum`, `max`, `any`, `min`), круглые скобки вместо квадратных экономят память бесплатно. Лишние внешние скобки можно опустить: `sum(... for ...)`.

!!! question "Проверь себя"

    1. Чем функция с `yield` отличается от функции с `return` при вызове?
    2. Что значит «ленивые вычисления» и почему генератор обрабатывает лог в 50 ГБ за константную память?
    3. Чем `f.readlines()` хуже генератора при чтении большого файла?
    4. Что такое конвейер генераторов и сколько записей лежит в памяти, когда данные текут сквозь него?

??? success "Ответы"

    1. Функция с `yield` при вызове не выполняется, а возвращает объект-генератор; код идёт до следующего `yield` только на каждом `next()`, замораживая состояние между ними.
    2. Элемент производится только в момент запроса. Генератор держит в памяти одну запись за раз, а не весь результат, поэтому размер входа почти не влияет на потребление памяти.
    3. `readlines()` загружает весь файл в список (память = размер файла). Генератор читает строку, обрабатывает, выбрасывает и берёт следующую.
    4. Цепочка генераторов, где выход одного — вход другого. Данные текут по одному элементу, ничего не материализуется; в памяти всегда одна запись.

```quiz

[x] вернётся объект-генератор, тело функции ещё не выполнялось
[ ] откроется файл и прочитается первая строка
> Тело генератора не запускается при вызове — только при первом next() (или начале for-цикла).
```
