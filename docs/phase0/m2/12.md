# M2.12 · Контекстные менеджеры (with)

Интуиция. `with` — это «гарантированная уборка»: что бы ни случилось внутри блока (даже ошибка), ресурс будет закрыт. Открыл файл — он точно закроется. Взял соединение с БД — оно точно вернётся.

Формально. Контекстный менеджер — объект с методами `__enter__` (вход в блок) и `__exit__` (выход, в том числе при исключении). `with` гарантирует вызов `__exit__` всегда. Свой контекстный менеджер удобно писать через `contextlib.contextmanager`.

```python
from contextlib import contextmanager

@contextmanager
def timer(label):
    start = time.perf_counter()
    try:
        yield                         # здесь выполняется тело with
    finally:
        print(f"{label}: {time.perf_counter() - start:.2f}s")

with timer("retention join"):
    result = heavy_join(df_a, df_b)   # время напечатается даже при ошибке
```

## Зачем это нужно: проблема, которую решает with

Без `with` уборку приходится писать руками — и легко забыть. Сравните два способа открыть файл:

```python
# Хрупко: если read_csv упадёт с ошибкой, f.close() не выполнится — файл «течёт»
f = open("players.csv")
data = process(f.read())   # упало здесь → f остаётся открытым
f.close()                  # эта строка не выполнится

# Надёжно: __exit__ закроет файл в любом случае
with open("players.csv") as f:
    data = process(f.read())
# здесь f уже закрыт, даже если process() кинул исключение
```

Ключевая мысль: `with` отвязывает «освобождение ресурса» от «нормального пути выполнения». Это и есть смысл `try/finally`, только без шума — контекстный менеджер прячет `finally` внутрь себя.

## Как это работает по шагам

Конструкция `with cm as x:` разворачивается примерно так:

```python
x = cm.__enter__()          # 1. вход: что вернул __enter__, попадает в x
try:
    ...                     # 2. тело блока with
finally:
    cm.__exit__(exc_type, exc_value, traceback)   # 3. выход — ВСЕГДА
```

Тонкость про `__exit__` и исключения. Метод получает три аргумента об ошибке (если ошибки не было — все три `None`). Если `__exit__` вернёт «истинное» значение (`True`), исключение считается обработанным и **подавляется** — оно не вылетит наружу. Если вернёт `None`/`False` — исключение пробросится дальше после уборки. По умолчанию `@contextmanager` исключение не глушит: оно «выстреливает» в точке `yield`, и блок `finally` всё равно отрабатывает.

В варианте через `@contextmanager` соответствие такое:

- код **до** `yield` = `__enter__`;
- значение, отданное в `yield` = то, что попадёт в `as x`;
- код в `finally` (или после `yield`) = `__exit__`.

## Разбор кода timer построчно

```python
import time
from contextlib import contextmanager

@contextmanager
def timer(label):
    start = time.perf_counter()       # __enter__: засекли старт
    try:
        yield                         # отдаём управление телу with (ничего не возвращаем в as)
    finally:
        dt = time.perf_counter() - start
        print(f"{label}: {dt:.2f}s")  # __exit__: печать времени — даже если внутри была ошибка

with timer("retention join"):
    result = heavy_join(df_a, df_b)
```

Почему `time.perf_counter()`, а не `time.time()`: `perf_counter` — монотонный таймер высокого разрешения, его не «дёргает» перевод системных часов или NTP-синхронизация. Для измерения длительности (а не «который сейчас час») это правильный выбор.

Почему `finally`, а не просто строка после `yield`: если внутри блока возникнет исключение, оно «прилетит» обратно в точку `yield`. Без `try/finally` строка с `print` не выполнится, и вы потеряете замер именно в тот момент, когда он нужнее всего — при падении.

## Несколько менеджеров и возврат значения

Можно открывать сразу несколько ресурсов в одном `with` через запятую — они закроются в обратном порядке (последний открыт — первый закрыт):

```python
with open("in.csv") as src, open("out.csv", "w") as dst:
    dst.write(src.read())
# dst закроется первым, затем src
```

А `__enter__` может что-то вернуть — например объект соединения:

```python
@contextmanager
def pg_connection(dsn):
    conn = psycopg2.connect(dsn)
    try:
        yield conn                    # это попадёт в `as conn`
    finally:
        conn.close()                  # вернули соединение в любом случае

with pg_connection(DSN) as conn:
    df = pd.read_sql("select * from bets where ts > now() - interval '1 day'", conn)
```

## Пример из продуктовой аналитики

Типичная задача: посчитать дневной GGR по когорте и не «забыть» открытое соединение и не оставить транзакцию висеть. Контекстный менеджер делает откат при ошибке и коммит при успехе:

```python
@contextmanager
def transaction(conn):
    try:
        yield conn
        conn.commit()                 # дошли до конца без ошибок — фиксируем
    except Exception:
        conn.rollback()               # ошибка внутри — откатываем всё
        raise                         # и пробрасываем дальше, чтобы не молчать
    finally:
        pass

with pg_connection(DSN) as conn, transaction(conn):
    update_player_segment(conn, player_id=42, segment="VIP")
    recalc_ltv(conn, player_id=42)    # если упадёт здесь — первый UPDATE откатится
```

Здесь видна вся сила паттерна: либо обе операции применятся вместе, либо ни одна — без ручного `rollback()` в каждой ветке.

## Пограничные случаи и типичные ошибки

- **Два `yield` в `@contextmanager`** — ошибка. Генератор-менеджер должен отдать управление ровно один раз. Если выполнение дойдёт до второго `yield`, Python кинет `RuntimeError`.
- **Исключение проглочено молча.** Если в `transaction` написать `except Exception: conn.rollback()` без `raise`, ошибка исчезнет, а данные окажутся не такими, как ждали. Логируйте и пробрасывайте, если не уверены.
- **`with` не делает повтор.** Контекстный менеджер гарантирует уборку, но не ретраи и не таймаут — это отдельная забота.
- **Возврат `True` из `__exit__` подавляет исключение.** Иногда это нужно (подавить конкретный класс ошибок), но по умолчанию опасно: легко спрятать баг. `@contextmanager` без явного перехвата исключение не глушит.
- **`with` для не-ресурсов.** Менеджеры годятся не только для файлов/соединений: временная смена настроек (`pd.option_context`), захват/освобождение блокировки (`with lock:`), подмена сидов и т. п. — всё, что нужно «вернуть как было».

!!! note "Готовые менеджеры из стандартной библиотеки"

    `contextlib.suppress(FileNotFoundError)` — тихо проглотить конкретное исключение; `contextlib.redirect_stdout(buf)` — перенаправить вывод; `contextlib.ExitStack` — складывать переменное число менеджеров в один блок. Часто не нужно писать свой класс — нужное уже есть.

## Проверь себя

Что напечатает `timer`, если `heavy_join` бросит исключение на середине? Подсказка: строка с временем всё равно выведется (это `finally`), а само исключение после этого вылетит наружу и остановит программу.
