# M2.8 · Модули и импорты

Интуиция. Модуль — это просто файл `.py`. Импорт — «дай мне доступ к коду из другого файла». Пакет — папка с модулями.

Формально. `import module` загружает модуль и даёт к нему доступ через пространство имён `module.func`. `from module import name` тянет конкретное имя в текущее пространство. Python ищет модули по путям из `sys.path`. Модуль выполняется один раз при первом импорте и кешируется.

```python
import numpy as np                      # алиас — идиома сообщества
from collections import Counter         # точечный импорт
from pathlib import Path

if __name__ == "__main__":              # код только при прямом запуске файла
    main()
```

Конструкция `if __name__ == "__main__"` — важная идиома. Она отделяет «код, который выполняется при запуске файла как скрипта» от «кода, который импортируется как библиотека». Без неё импорт твоего модуля случайно запустит весь анализ.

## Зачем вообще модули: одна история на пальцах

Представь, что ты пишешь аналитику депозитов для гемблинг-проекта. Сначала всё лежит в одном файле `analysis.py` на 800 строк: загрузка данных, очистка, расчёт FTD-конверсии, расчёт retention, построение графиков. Через месяц ты не можешь найти, где считается retention, а функция `clean()` называется одинаково в трёх местах. Модули решают ровно эту боль: ты раскладываешь код по файлам по смыслу.

```
project/
├── data_loader.py     # загрузка из БД
├── metrics.py         # FTD, retention, GGR
├── plots.py           # графики
└── main.py            # склеивает всё вместе
```

Теперь в `main.py`:

```python
from data_loader import load_deposits
from metrics import ftd_conversion, retention_d7

deposits = load_deposits("2026-06-01", "2026-06-16")
print(ftd_conversion(deposits))
print(retention_d7(deposits))
```

Каждый файл — отдельная, тестируемая, понятная единица. Это и есть главная ценность модулей: не «магия», а гигиена кода.

## Четыре формы импорта — и когда какую брать

```python
import math                 # 1. весь модуль, доступ через math.sqrt
import numpy as np          # 2. весь модуль под алиасом, np.array
from math import sqrt       # 3. одно имя прямо в namespace, sqrt(2)
from math import sqrt, pi   # тянем несколько имён сразу
```

- `import math` — самый явный: всегда видно, откуда `math.sqrt`. Безопасно от коллизий имён.
- `import numpy as np` — алиас для длинных или общепринятых имён. `np`, `pd`, `plt` — стандарт сообщества, отступать от него не стоит, иначе чужой код станет нечитаемым.
- `from math import sqrt` — короче в использовании, но теряется контекст «откуда оно».

Чего избегать — `from module import *` (импорт всего звёздочкой):

```python
from math import *          # тянет ВСЕ имена модуля в твой namespace
log = "лог пользователя"    # а потом math.log молча перезаписан строкой
print(log(100))             # TypeError: 'str' object is not callable
```

Звёздочка засоряет пространство имён и приводит к скрытым перезаписям. В продакшен-коде её практически не используют.

## Что на самом деле делает `__name__`

`__name__` — это автоматическая переменная, которую Python кладёт в каждый модуль. Её значение зависит от того, **как** модуль попал в программу:

- запустил файл напрямую (`python metrics.py`) → `__name__ == "__main__"`;
- импортировал его (`from metrics import retention_d7`) → `__name__ == "metrics"` (имя модуля).

Проверим руками. Файл `metrics.py`:

```python
def retention_d7(deposits):
    return 0.42  # заглушка

print("Этот print выполнится ВСЕГДА при импорте:", __name__)

if __name__ == "__main__":
    print("А этот — только при прямом запуске")
    print(retention_d7([]))
```

Запуск `python metrics.py` напечатает обе строки. А вот `from metrics import retention_d7` из другого файла напечатает только первую (`__name__` будет равен `"metrics"`), потому что блок под `if` пропустится. Отсюда практическое правило: **всё, что не должно срабатывать при импорте — тяжёлые расчёты, чтение БД, обучение модели — прячь под `if __name__ == "__main__"`.** Иначе простой `import metrics` запустит весь анализ как побочный эффект.

## Кеширование: модуль выполняется ровно один раз

Когда модуль импортируется первый раз, Python выполняет его сверху вниз и кладёт результат в словарь `sys.modules`. Повторные `import` того же модуля не выполняют его заново — берут готовое из кеша.

```python
# config.py
print("config загружается")
DB_URL = "postgres://yohoho"
```

```python
# main.py
import config   # печатает "config загружается"
import config   # ничего не печатает — взято из кеша
from config import DB_URL  # тоже из кеша, без повторного выполнения
```

Это важно для гемблинг-аналитики: если в `config.py` ты один раз открыл соединение с БД или прочитал секреты, повторные импорты не создадут второе соединение. Обратная сторона — если ты поправил код модуля во время работы интерактивной сессии (Jupyter), новый `import` его не подхватит; нужен `importlib.reload(config)` или перезапуск ядра. Это типичная ловушка новичка: «я же поправил функцию, почему старое поведение?» — потому что модуль закеширован.

## Пакеты и пути поиска

Пакет — папка с модулями. Исторически в ней лежал файл `__init__.py` (может быть пустым), который сообщал Python «это пакет». Импорт внутри пакета идёт через точку:

```python
from analytics.metrics import retention_d7   # файл analytics/metrics.py
from analytics import plots                   # модуль analytics/plots.py
```

Где Python вообще ищет модули — определяет список `sys.path`: сначала папка текущего скрипта, затем стандартная библиотека, затем установленные пакеты (`site-packages`).

```python
import sys
print(sys.path)   # список папок, по которым идёт поиск
```

Самая частая ошибка новичка — `ModuleNotFoundError: No module named 'metrics'`. Обычно причина в том, что ты запускаешь скрипт из другой директории, и папки с `metrics.py` нет в `sys.path`. Решение — запускать из корня проекта, либо оформить проект как устанавливаемый пакет.

!!! warning "Не называй свои файлы как стандартные библиотеки"

    Если создать файл `random.py` или `email.py` рядом со скриптом, он перекроет одноимённый модуль стандартной библиотеки: `import random` подтянет твой файл вместо настоящего. Симптом — странные `ImportError` или `AttributeError: module 'random' has no attribute 'randint'`. Проверь, нет ли в проекте файла-двойника.

## Типичные ошибки

- `from module import *` — скрытая перезапись имён, нечитаемость. Не используй.
- Тяжёлый код на верхнем уровне модуля без `if __name__ == "__main__"` — запустится при каждом импорте.
- Свой файл `random.py` / `json.py` / `string.py` — конфликт со стандартной библиотекой.
- Правка модуля в Jupyter без `reload` — старый код берётся из кеша.
- Циклический импорт: `a.py` импортирует `b.py`, а `b.py` импортирует `a.py` — обычно сигнал, что код плохо разложен.

!!! note "Проверь себя"

    Что напечатает `import config` второй раз подряд, если в `config.py` на верхнем уровне есть `print("loaded")`? Ничего — модуль уже в `sys.modules`, повторное выполнение не происходит. Печать `loaded` случилась только при первом импорте.
