## M1.1 · Зачем этот модуль

Представь: ты три недели строил модель, которая предсказывает отток игроков (churn). Метрики красивые, ROC-AUC 0.87, ты гордо отдаёшь ноутбук коллеге. Он запускает — и получает `ModuleNotFoundError`, потом другая версия библиотеки выдаёт другой результат, потом выясняется, что половина чисел в ноутбуке посчитана ячейками, которые ты запускал в случайном порядке и потом удалил. Воспроизвести нельзя. Работа есть, а результата нет.

Это не редкость, это норма для тех, кто пропустил настройку среды. Среда (environment) — это фундамент. На кривом фундаменте можно построить что угодно красивое, но оно рухнет в момент, когда понадобится кому-то это повторить: ревьюеру, коллеге, тебе самому через полгода, проду.

Простыми словами: правильная среда — это гарантия, что «у меня работает» означает «работает везде и всегда одинаково». В аналитике и в quant-исследованиях это не вопрос удобства, а вопрос доверия к выводам. Если расчёт LTV по когорте нельзя воспроизвести байт-в-байт, то решение «вливать ли ещё 200 тысяч в этот трафик-источник» принимается на основе числа, которому нельзя верить.

Формально: воспроизводимость (reproducibility) — свойство вычислительного результата, при котором повторный запуск того же кода на тех же данных с теми же зафиксированными версиями зависимостей даёт идентичный (или статистически эквивалентный при контролируемой случайности) результат. Без этого свойства результат не является научным фактом — это анекдот.

Без этого модуля сломается всё последующее. Ты не сможешь нормально сотрудничать (код не запустится у других), не сможешь возвращаться к старым экспериментам, не сможешь катить модель в прод, и будешь терять часы на отладку окружения вместо анализа данных.

## M1.3 · Воспроизводимость как принцип

Прежде чем трогать инструменты, нужно понять цель, ради которой они существуют. Все инструменты этого модуля решают одну задачу — сделать твою работу воспроизводимой. Если держать эту цель в голове, выбор инструмента становится очевидным.

Воспроизводимость распадается на слои, и каждый слой закрывается своим инструментом:

- Код — закрывается git (версионирование).
- Зависимости (какие библиотеки и каких версий) — закрывается `uv` и lockfile.
- Среда исполнения (какой Python, какая ОС) — закрывается виртуальным окружением, в пределе Docker.
- Данные — закрывается фиксацией версии датасета (хеши, DVC, snapshot в хранилище).
- Случайность — закрывается фиксацией random seed.

!!! info "Уровни строгости"

    Различают reproducibility (тот же код + те же данные дают тот же результат — минимальный обязательный уровень) и replicability (другой код или другие данные подтверждают тот же вывод — это уже про научную надёжность гипотезы). В прикладной аналитике сначала добиваются первого. Второе — признак зрелого исследования, когда вывод устойчив к деталям реализации.

Практический критерий: проект воспроизводим, если новый человек на чистой машине может выполнить три команды (склонировать, установить зависимости, запустить) и получить тот же результат, не задавая тебе вопросов. Всё, что мы дальше настраиваем, служит этому критерию.

!!! question "Проверь себя"

    1. Назови пять слоёв воспроизводимости и инструмент, который закрывает каждый.
    2. В чём разница между reproducibility и replicability?
    3. По какому практическому критерию проект считается воспроизводимым?

??? success "Ответы"

    1. Код — git; зависимости — uv и lockfile; среда исполнения — виртуальное окружение (в пределе Docker); данные — фиксация версии датасета (хеши, DVC, snapshot); случайность — фиксация random seed.
    2. Reproducibility — тот же код и те же данные дают тот же результат (минимальный обязательный уровень). Replicability — другой код или другие данные подтверждают тот же вывод (научная надёжность гипотезы).
    3. Новый человек на чистой машине выполняет три команды (склонировать, установить зависимости, запустить) и получает тот же результат, не задавая вопросов.

## M1.4 · Python 3.13 / 3.14

Python — основной язык DS, и версия имеет значение. На июнь 2026 актуальный рабочий выбор — Python 3.13 или 3.14 (3.14 вышел 7 октября 2025 и к лету 2026 уже зрелый). Эти релизы принесли две вещи, важные именно для нас.

Простыми словами: начиная с 3.13 Python научился (тогда в экспериментальном режиме) работать без «глобального замка», который раньше мешал по-настоящему параллельно использовать несколько ядер процессора в одном процессе, а в 3.14 эта сборка стала официально поддерживаемой. Плюс 3.13 принёс быстрее работающий интерактивный интерпретатор с подсветкой.

Формально: речь о GIL (Global Interpreter Lock) — мьютексе, который исторически разрешал исполнять байткод Python только одному потоку одновременно. Python 3.13 ввёл экспериментальную сборку free-threaded (`--disable-gil`, PEP 703), где GIL можно отключить, и режим экспериментального JIT-компилятора (PEP 744). В Python 3.14 эта сборка (`python3.14t`, PEP 779) перешла из экспериментальной в официально поддерживаемую — но по-прежнему не является дефолтной. Для DS это пока не «включил и стало в 4 раза быстрее» — большинство тяжёлых вычислений и так живут в NumPy/Polars, где работа идёт в нативном коде вне GIL. Но направление задано, и через пару лет free-threaded станет дефолтом.

!!! warning "Не бери самую свежую версию вслепую"

    Только что вышедший Python (свежий мажорный релиз в первые месяцы) часто опережает экосистему: бинарные колёса (wheels) для NumPy, PyTorch, scientific-стека под него ещё не собраны, и установка ломается или собирается из исходников часами. Практическое правило: бери предпоследний стабильный релиз. На июнь 2026 безопасный дефолт — 3.13, осознанный выбор для нового проекта — 3.14 (вышел 7 октября 2025 и к лету уже зрелый).

Важно: системный Python (тот, что идёт с macOS или Linux) трогать нельзя — от него зависит сама ОС. Версиями Python для проектов управляет `uv` (ниже), который умеет скачивать и ставить любую нужную версию изолированно.

!!! question "Проверь себя"

    1. Что такое GIL и почему отключение его во free-threaded-сборке пока слабо влияет на типичные DS-вычисления?
    2. Почему не стоит сразу ставить только что вышедшую версию Python в первые месяцы после релиза?
    3. Почему системный Python нельзя трогать, и кто управляет версиями Python для проектов?

??? success "Ответы"

    1. GIL (Global Interpreter Lock) — мьютекс, разрешавший исполнять байткод Python только одному потоку. Эффект отключения мал, потому что тяжёлые вычисления и так живут в NumPy/Polars, работающих в нативном коде вне GIL.
    2. Под свежий релиз ещё не собраны бинарные wheels для NumPy, PyTorch и scientific-стека — установка ломается или собирается из исходников часами. Правило: брать предпоследний стабильный релиз.
    3. От системного Python зависит сама ОС, его поломка ломает систему. Версиями Python для проектов управляет uv, скачивая и ставя нужную версию изолированно.

## M1.5 · Виртуальные окружения: зачем

Это, возможно, самая важная привычка во всём модуле. Без неё ничего не работает.

Простыми словами: представь, что для проекта А тебе нужна библиотека pandas версии 1.5, а для проекта Б — версии 2.2, и они несовместимы. Если ставить всё в одно общее место, проекты передерутся за версии, и один сломает другой. Виртуальное окружение — это отдельная коробка с библиотеками для каждого проекта. Что лежит в коробке проекта А, не влияет на проект Б.

Формально: виртуальное окружение (virtual environment) — изолированная директория с собственным интерпретатором Python и собственным набором установленных пакетов, не пересекающаяся с системным Python и с другими окружениями. Технически это поддиректория (обычно `.venv/`) со своим `bin/` (или `Scripts/` на Windows) и `site-packages/`. Активация окружения переключает, какой `python` и `pip` использует твоя оболочка.

Железное правило: один проект — одно окружение. Никогда не ставь пакеты в системный Python через `sudo pip install` — это прямой путь к нерабочей ОС и неразрешимым конфликтам версий.

!!! question "Проверь себя"

    1. Объясни на примере двух проектов с конфликтующими версиями библиотеки, зачем нужно виртуальное окружение.
    2. Что физически представляет собой виртуальное окружение на диске?
    3. Почему `sudo pip install` в системный Python — антипаттерн?

??? success "Ответы"

    1. Проекту А нужна pandas 1.5, проекту Б — несовместимая 2.2. В общем месте они передерутся за версии и сломают друг друга. Окружение — отдельная коробка библиотек на каждый проект, и содержимое одной не влияет на другую.
    2. Изолированная директория (обычно `.venv/`) со своим интерпретатором, `bin/` (или `Scripts/` на Windows) и `site-packages/`. Активация переключает, какой `python` и `pip` использует оболочка.
    3. Пакеты попадают в системный Python, от которого зависит ОС; это ведёт к нерабочей системе и неразрешимым конфликтам версий. Правило: один проект — одно окружение.

## M1.6 · Пакетные менеджеры: почему uv победил

До 2024 года ландшафт был зоопарком: `pip` ставил пакеты, `venv` создавал окружения, `conda` делал и то и другое (но медленно и со своей экосистемой), `poetry` и `pipenv` добавляли lockfile и управление зависимостями. Чтобы собрать проект, нужно было жонглировать тремя-четырьмя инструментами, и каждый решал кусок задачи.

`uv` (от создателей линтера ruff, написан на Rust) в 2024-2025 поглотил все эти роли в одном инструменте и сделал это в 10-100 раз быстрее. К июню 2026 `uv` — фактический стандарт для новых DS-проектов.

Простыми словами: `uv` — это один инструмент, который умеет: скачать нужную версию Python, создать коробку-окружение, поставить туда библиотеки, зафиксировать их точные версии и запустить твой код. Быстро, потому что написан на быстром языке и умно кеширует.

Формально: `uv` — это менеджер пакетов и проектов, совмещающий функции pip, pip-tools, virtualenv, pyenv и poetry. Он реализует resolver зависимостей (решает граф совместимости версий), управляет виртуальными окружениями, версиями интерпретатора и генерирует детерминированный lockfile.

### Основные команды

```python
# Это команды оболочки, не Python. Запускаются в терминале.

# Инициализировать новый проект (создаёт pyproject.toml, .venv, .git)
uv init churn-model

# Создать виртуальное окружение вручную (если нужно отдельно)
uv venv --python 3.14

# Добавить зависимость — ставит пакет И записывает его в pyproject.toml + lock
uv add polars scikit-learn

# Добавить dev-зависимость (нужна для разработки, не для прода)
uv add --dev pytest ruff

# Запустить скрипт в окружении проекта без ручной активации
uv run python train.py

# Установить всё из lockfile точь-в-точь (на новой машине, в CI, в проде)
uv sync

# Запустить инструмент разово, не устанавливая в проект (как npx)
uvx ruff check .
```

Разберём ключевые:

`uv venv` создаёт изолированное окружение. `uv add <пакет>` — главная команда: она не просто ставит пакет, а записывает его в `pyproject.toml` (объявленные зависимости) и обновляет lockfile (точные разрешённые версии). `uv run` запускает команду внутри окружения проекта — тебе не нужно вручную «активировать» окружение, `uv` сам подставит правильный Python. `uvx` (сокращение от `uv tool run`) запускает CLI-инструмент во временном окружении, не засоряя проект — удобно для разовых утилит.

### Что такое lockfile

Это центральное понятие воспроизводимости зависимостей, и его часто недопонимают.

Простыми словами: в `pyproject.toml` ты пишешь «мне нужен polars новее версии 1.0». Но «новее 1.0» — это и 1.2, и 1.9, и они могут вести себя по-разному. Lockfile (`uv.lock`) фиксирует: «в этом проекте используется ровно polars 1.31.0, и вот его криптографический хеш, чтобы никто не подменил». Это разница между «примерно вот эти ингредиенты» и «точный чек из магазина с граммами и штрихкодами».

Формально: lockfile — это полностью разрешённый, детерминированный снимок всего дерева зависимостей, включая транзитивные (зависимости твоих зависимостей), с точными версиями и хешами артефактов. `pyproject.toml` декларирует намерение (constraints), lockfile фиксирует решение (resolution). Когда коллега делает `uv sync`, он получает байт-в-байт те же версии, что у тебя.

!!! tip "Что коммитить в git"

    Коммить и `pyproject.toml`, и `uv.lock`. Первый описывает, что тебе нужно, второй гарантирует, что у всех будет одинаково. Не коммить `.venv/` — это тяжёлая локальная папка, которая пересоздаётся командой `uv sync` из lockfile за секунды.

!!! question "Проверь себя"

    1. Какие функции прежнего «зоопарка» (pip, venv, pyenv, poetry) совмещает в себе uv?
    2. Чем `uv add` отличается от обычной установки пакета и что он меняет в проекте?
    3. В чём разница между `pyproject.toml` и `uv.lock` и почему в git нужны оба?
    4. Что коммитить, а что нет: `pyproject.toml`, `uv.lock`, `.venv/`?

??? success "Ответы"

    1. Менеджер пакетов и проектов: pip, pip-tools, virtualenv, pyenv и poetry — resolver зависимостей, управление окружениями и версиями интерпретатора плюс детерминированный lockfile.
    2. `uv add` не просто ставит пакет, а записывает его в `pyproject.toml` (объявленные зависимости) и обновляет lockfile (точные разрешённые версии транзитивного дерева).
    3. `pyproject.toml` декларирует намерение (constraints вроде «polars новее 1.0»), `uv.lock` фиксирует решение — точные версии и хеши всего дерева зависимостей. Первый без второго не даёт воспроизводимости.
    4. Коммитить `pyproject.toml` и `uv.lock`. `.venv/` не коммитить — это тяжёлая локальная папка, пересоздаваемая `uv sync` за секунды.

!!! example "Мини-задание"

## M1.7 · Git и GitHub: версионирование кода и экспериментов

Git — система контроля версий. Без неё DS-работа неуправляема.

Простыми словами: git — это машина времени и журнал для твоего проекта. Каждый раз, когда ты делаешь осмысленный шаг, ты сохраняешь снимок (commit) с подписью «что и зачем». Можешь вернуться к любому снимку, посмотреть, что менялось, и кто это сделал. GitHub — облачное хранилище этих снимков плюс место для совместной работы.

Формально: git — распределённая система контроля версий, хранящая историю изменений как направленный ациклический граф коммитов. Каждый commit — иммутабельный снимок состояния дерева файлов с хешем-идентификатором, ссылкой на родителя, автором и сообщением.

### Базовый цикл

```python
# Команды оболочки

git init                          # создать репозиторий (uv init уже делает это)
git add train.py features.py      # пометить файлы для сохранения (staging)
git commit -m "Add churn features: recency, frequency, deposit_sum"
git branch experiment-xgboost     # создать ветку под эксперимент
git switch experiment-xgboost     # переключиться на неё
git push origin experiment-xgboost  # отправить ветку на GitHub
```

Ключевые понятия:

Commit — сохранённый снимок с сообщением. Хорошее сообщение объясняет «почему», а не «что» (диф и так показывает, что менялось). Плохо: «fix». Хорошо: «Fix LTV leak: exclude deposits after cohort cutoff date».

Branch (ветка) — параллельная линия разработки. Для DS это критично: каждый эксперимент или фича живёт в своей ветке, не ломая рабочую `main`. Хочешь попробовать другую модель — ответвись, поэкспериментируй, и если получилось — влей обратно.

Pull request (PR) — механизм GitHub: ты предлагаешь влить свою ветку в `main`, коллеги ревьюят диф, обсуждают, и только потом мерж. Это точка контроля качества и распространения знаний в команде.

### Зачем версионировать заметки об экспериментах

В DS код — это половина истории. Вторая половина — что ты пробовал и что вышло. «Поднял глубину дерева до 12 — ROC-AUC упал, переобучение». Если эти заметки не версионированы рядом с кодом, через месяц ты повторишь ту же ошибку.

Практика: держи в репозитории файл вроде `EXPERIMENTS.md` или лог в структурированном виде, и коммить его вместе с изменениями кода. Тогда конкретная версия модели, конкретный код и конкретный вывод связаны одним коммитом. Это и есть лабораторный журнал исследователя в цифровом виде.

!!! warning "Git — не для данных и не для моделей"

    Git хранит текст и плохо работает с большими бинарными файлами. Не коммить датасеты на гигабайты и обученные модели — репозиторий распухнет необратимо. Для версионирования данных и моделей есть DVC, git-lfs или хранение артефактов в облаке с фиксацией версии/хеша. В git — только код, конфиги и заметки.

!!! question "Проверь себя"

    1. Что хорошее commit-сообщение должно объяснять — «что» или «почему» — и почему именно так?
    2. Зачем в DS каждый эксперимент держать в отдельной ветке, а не коммитить всё в `main`?
    3. Почему датасеты на гигабайты и обученные модели нельзя коммитить в git, и чем их версионировать?

??? success "Ответы"

    1. «Почему»: диф и так показывает, что менялось, а мотивация изменения из кода не видна. Плохо: «fix». Хорошо: «Fix LTV leak: exclude deposits after cohort cutoff date».
    2. Ветка — параллельная линия разработки; эксперимент не ломает рабочую `main`. Получилось — мержишь через pull request с ревью; нет — выбрасываешь ветку.
    3. Git хранит текст и плохо работает с большими бинарниками — репозиторий распухает необратимо. Данные и модели версионируют через DVC, git-lfs или облако с фиксацией версии/хеша; в git — только код, конфиги и заметки.

## M1.8 · Ноутбуки: Jupyter, JupyterLab, marimo

Ноутбук — интерактивный документ, где код, его вывод, графики и текст живут вместе. Для исследования данных это незаменимо: ты видишь результат каждого шага сразу.

Простыми словами: ноутбук — это блокнот, где ты пишешь кусочек кода в ячейке, жмёшь запуск и тут же видишь под ней результат — таблицу, график, число. Идеально, чтобы крутить данные руками и думать.

### Jupyter Notebook против JupyterLab

Jupyter Notebook — классический интерфейс, одна вкладка с одним ноутбуком. JupyterLab — это среда вокруг той же технологии: вкладки, файловый менеджер, терминал, несколько ноутбуков рядом, расширения. На 2026 JupyterLab — стандартный выбор, если идёшь по экосистеме Jupyter. «Классический» Notebook 7 теперь сам построен на технологиях Lab.

### Проблема скрытого состояния

Это главная боль Jupyter, и на ней горят даже опытные люди. Понять её обязательно.

Простыми словами: в Jupyter ты можешь запускать ячейки в любом порядке и сколько угодно раз. Запустил ячейку 5, потом 2, потом снова 5, потом удалил ячейку 3, в которой определил переменную. На экране всё выглядит работающим — но это иллюзия. Переменная всё ещё живёт в памяти, хотя кода, который её создал, в ноутбуке уже нет. Дай этот ноутбук другому — у него ничего не запустится, потому что состояние было «спрятано» в памяти твоей сессии, а не в коде.

Формально: проблема hidden state — рассинхрон между видимым кодом ноутбука и фактическим состоянием ядра (kernel namespace). Ядро хранит все переменные от всех когда-либо выполненных ячеек, независимо от текущего содержимого и порядка ячеек на экране. Результат — ноутбук, который воспроизводится только в твоей конкретной живой сессии и нигде больше. Исследования воспроизводимости публичных Jupyter-ноутбуков показывают, что значительная их часть не исполняется сверху вниз без ошибок.

!!! tip "Дисциплина Jupyter"

    Если работаешь в Jupyter — регулярно делай Restart Kernel and Run All. Это перезапускает ядро с чистого листа и прогоняет все ячейки сверху вниз. Если так не запускается — твой ноутбук сломан, просто ты этого ещё не заметил. Делай это перед каждым коммитом.

### marimo: почему набирает в 2026

marimo — ноутбук нового поколения, который архитектурно убирает проблему скрытого состояния. К 2026 он быстро набирает популярность именно поэтому.

Простыми словами: marimo — это реактивный ноутбук. Он понимает, какая ячейка от какой зависит. Изменил переменную в одной ячейке — все ячейки, которые её используют, автоматически пересчитываются, а устаревшие помечаются. Удалил ячейку — переменная исчезает из памяти тоже. Состояние на экране всегда соответствует реальности. Плюс сам файл — это обычный `.py`, а не JSON, поэтому он нормально дружит с git (видно человеческие дифы).

Формально: marimo строит DAG (направленный ациклический граф) зависимостей между ячейками через статический анализ кода. При изменении ячейки он детерминированно перевыполняет все зависимые ячейки в топологическом порядке, гарантируя консистентность состояния. Запрещены циклические зависимости и повторное определение одной переменной в разных ячейках — это исключает классы багов Jupyter. Файл хранится как исполняемый Python-модуль, что даёт чистые git-дифы, импортируемость и запуск как обычного скрипта или как веб-приложения.

```python
# Пример ячеек marimo. Зависимости выводятся автоматически.
import marimo as mo
import polars as pl

df = pl.read_parquet("data/processed/deposits.parquet")

# Эта ячейка зависит от df. Изменишь df — она пересчитается сама.
ftd = df.filter(pl.col("is_first_deposit")).select(pl.col("amount").mean())
```

Прагматичный вывод на 2026: для быстрого личного исследования Jupyter/JupyterLab всё ещё повсеместны и это нормально. Но для воспроизводимых аналитических отчётов, для кода, который пойдёт в команду или в прод, и для дашбордов marimo — лучший выбор, и стоит освоить его сейчас, пока это даёт преимущество.

!!! question "Проверь себя"

    1. В чём суть проблемы скрытого состояния (hidden state) в Jupyter?
    2. Что делает Restart Kernel and Run All и зачем гонять его перед каждым коммитом?
    3. За счёт чего marimo архитектурно устраняет hidden state и почему его `.py`-файл дружит с git?

??? success "Ответы"

    1. Рассинхрон между видимым кодом и состоянием ядра: ядро хранит все переменные от всех когда-либо выполненных ячеек независимо от текущего содержимого и порядка. Ноутбук «работает» только в живой сессии и нигде больше.
    2. Перезапускает ядро с нуля и прогоняет все ячейки сверху вниз. Если падает — ноутбук на самом деле сломан; проверка ловит это до коммита.
    3. marimo строит DAG зависимостей между ячейками статическим анализом и детерминированно перевыполняет зависимые ячейки в топологическом порядке; запрещены циклы и повторное определение переменной. Файл — исполняемый Python-модуль, отсюда чистые git-дифы.

## M1.9 · Редакторы и AI-копилоты

Ноутбук хорош для исследования, но проект из ноутбуков не построить. Серьёзный код (модули, пайплайны, тесты) живёт в редакторе.

VS Code — индустриальный стандарт: бесплатный, расширяемый, отличная поддержка Python, git, ноутбуков прямо внутри. Cursor — форк VS Code, переосмысленный вокруг AI: он держит в голове весь проект и позволяет редактировать код диалогом с моделью. На 2026 оба — валидный выбор; Cursor популярен у тех, кто плотно работает с AI-ассистентом.

### AI-копилот в воркфлоу

К июню 2026 работа без AI-ассистента — это сознательный отказ от инструмента, который ускоряет рутину в разы. Но место его в воркфлоу нужно понимать точно.

Три типовых инструмента: GitHub Copilot (автодополнение прямо в редакторе, «продолжает мысль»), Cursor (агентное редактирование в рамках проекта), Claude Code (агент в терминале, который читает/пишет файлы, запускает команды, ведёт целые задачи).

Простыми словами: копилот — это очень быстрый джуниор, который мгновенно пишет шаблонный код, вспоминает синтаксис, объясняет чужой код и предлагает варианты. Но он не понимает твою бизнес-задачу и регулярно уверенно ошибается. Ты — старший, который ставит задачу, проверяет и отвечает за результат.

Правильное место в воркфлоу: копилот пишет черновик и рутину (boilerplate, парсинг, типовые трансформации, тесты), ты — задаёшь намерение, ревьюишь каждую строку и держишь архитектуру. Никогда не наоборот.

## M1.10 · Структура DS-проекта

Хаотичная папка с десятком `final_v2_REAL_final.ipynb` — признак непрофессионала. Стандартная структура экономит часы и делает проект понятным с первого взгляда.

Простыми словами: у всего своё место. Сырые данные отдельно, обработанные отдельно, код отдельно, ноутбуки-черновики отдельно. Любой человек (включая тебя через полгода) сразу понимает, где что искать.

Базовая структура, восходящая к шаблону cookiecutter-data-science:

```python
churn-model/
├── data/
│   ├── raw/            # сырые данные, неизменяемые, read-only
│   ├── interim/        # промежуточные результаты обработки
│   └── processed/      # финальные датасеты для моделей
├── notebooks/          # исследовательские ноутбуки (черновики)
├── src/                # переиспользуемый код: модули, функции
│   ├── features.py     # построение признаков
│   ├── train.py        # обучение модели
│   └── evaluate.py     # оценка
├── models/             # сохранённые обученные модели (в git не коммитим)
├── reports/            # отчёты, графики, результаты
├── tests/              # тесты кода из src
├── pyproject.toml      # зависимости и конфиг проекта
├── uv.lock             # зафиксированные версии
├── .env                # секреты — НИКОГДА не в git
├── .gitignore          # что git должен игнорировать
└── README.md           # что это, как запустить
```

Формально: cookiecutter-data-science — это шаблонизатор, генерирующий стандартизованную структуру DS-проекта одной командой. Ключевой принцип — data/raw неизменяем и read-only: сырые данные никогда не правятся на месте, любая обработка порождает новый файл в interim/ или processed/. Это гарантирует, что ты всегда можешь воспроизвести весь пайплайн с исходной точки.

Главное архитектурное правило: код, который повторяется или пойдёт в прод, переезжает из ноутбука в `src/` как функции, и ноутбук их импортирует. Ноутбук — для исследования и демонстрации, `src/` — для логики. Это и есть граница между «поиграл с данными» и «построил систему».

!!! question "Проверь себя"

    1. Почему `data/raw/` объявляют неизменяемым и read-only, и куда тогда попадают результаты обработки?
    2. Где проходит граница между ноутбуком и `src/`, и что это говорит о зрелости проекта?
    3. Что из структуры проекта осознанно не коммитят в git?

??? success "Ответы"

    1. Чтобы всегда можно было воспроизвести весь пайплайн с исходной точки. Любая обработка порождает новый файл в `interim/` или `processed/`, сырьё не правится на месте.
    2. Код, который повторяется или пойдёт в прод, переезжает из ноутбука в `src/` как функции, а ноутбук их импортирует. Ноутбук — для исследования и демонстрации, `src/` — для логики; наличие этой границы и есть признак зрелости.
    3. `.env` (секреты), `.venv/` (окружение), `data/` и `models/` (обученные модели) — они в `.gitignore`.

## M1.11 · Секреты и .env

Это вопрос безопасности, на котором горят регулярно и больно.

Простыми словами: пароли от базы данных, API-ключи, токены — это секреты. Их нельзя писать прямо в коде, потому что код уходит в git, а оттуда — на GitHub, где их могут увидеть. Секреты кладут в отдельный файл `.env`, который git'у запрещено трогать.

Формально: `.env` — текстовый файл вида `KEY=value`, хранящий конфигурацию и секреты вне исходного кода. Он обязательно вносится в `.gitignore`. В репозиторий кладут только `.env.example` — шаблон без значений, показывающий, какие переменные нужны. Принцип разделения кода и конфигурации восходит к методологии Twelve-Factor App: конфигурация живёт в окружении, не в коде.

```python
# .env (в .gitignore, не коммитится)
# DB_PASSWORD=super_secret_123
# YOHOHO_API_KEY=sk-...

# Чтение в коде через python-dotenv
import os
from dotenv import load_dotenv

load_dotenv()
db_password = os.environ["DB_PASSWORD"]  # упадёт, если переменной нет — это хорошо
```

!!! warning "Если секрет утёк в git — он скомпрометирован навсегда"

    Просто удалить ключ из файла и закоммитить недостаточно: он остаётся в истории git и доступен любому, кто склонирует репозиторий. Единственный правильный шаг — немедленно отозвать (revoke) и перевыпустить ключ на стороне сервиса. Чистка истории git (filter-repo) — вторична. Поэтому проще не допускать утечки: `.gitignore` настраивается до первого коммита.

!!! question "Проверь себя"

    1. Почему секреты держат в `.env`, а не прямо в коде, и что из этой пары кладут в репозиторий?
    2. Почему просто удалить утёкший ключ из файла и закоммитить недостаточно, и какой шаг правильный?
    3. На какую методологию опирается принцип «конфигурация живёт в окружении, не в коде»?

??? success "Ответы"

    1. Код уходит в git и на GitHub, где секреты могут увидеть. Секреты в `.env`, который внесён в `.gitignore`. В репозиторий кладут только `.env.example` — шаблон без значений.
    2. Ключ остаётся в истории git и доступен любому, кто склонирует репозиторий. Правильно — немедленно отозвать (revoke) и перевыпустить ключ на стороне сервиса; чистка истории вторична.
    3. Twelve-Factor App, фактор III «Config».

    В тестовом проекте создай `.env` со строкой вида `API_KEY=test123`, добавь `.env` в `.gitignore` и выполни `git status`. Убедись, что `.env` не отслеживается. Затем создай `.env.example` с `API_KEY=` без значения — именно его и коммитят.

Пример переезда логики из ноутбука в `src/features.py` — переиспользуемая функция признаков для модели оттока:

```python
import polars as pl

def build_rfm_features(bets: pl.DataFrame, cutoff: str) -> pl.DataFrame:
    """RFM-признаки игроков до даты cutoff (защита от утечки будущего)."""
    window = bets.filter(pl.col("bet_ts") < cutoff)
    return window.group_by("player_id").agg(
        recency_days=(pl.lit(cutoff).str.to_datetime() - pl.col("bet_ts").max()).dt.total_days(),
        frequency=pl.len(),
        monetary=pl.col("stake").sum(),
    )
```

```python
# tests/test_features.py — даже простой тест ловит регрессии
import polars as pl
from src.features import build_rfm_features

def test_cutoff_excludes_future_bets():
    bets = pl.DataFrame({
        "player_id": [1, 1],
        "bet_ts": ["2026-01-01", "2026-12-31"],
        "stake": [100.0, 999.0],
    }).with_columns(pl.col("bet_ts").str.to_datetime())
    out = build_rfm_features(bets, cutoff="2026-06-01")
    assert out["monetary"][0] == 100.0  # ставка из декабря не должна попасть
```

    "Собери стартовую структуру DS-проекта для модели оттока игроков: uv-проект на Python 3.13, зависимости polars/scikit-learn/marimo, dev-зависимости pytest/ruff. Создай data/{raw,interim,processed}, src/, notebooks/, tests/. Настрой .gitignore так, чтобы .env, .venv и data/ не попадали в git. Покажи команды и объясни каждую строку pyproject.toml."

    Где врут и что перепроверять обязательно: модели регулярно дают устаревшие команды (предлагают pip/conda/poetry вместо uv, потому что обучались на старых данных) — сверяй с актуальной документацией uv. Они путают флаги (`uv pip install` против `uv add` — это разные вещи) и иногда советуют закоммитить .env или .venv — это прямая дыра в безопасности, лови такое всегда. И они уверенно выдумывают несуществующие опции команд: если флага нет в `uv --help`, его нет, что бы ни говорил ассистент.

!!! note "Стек 2026"

    Актуально на июнь 2026: управление зависимостями — `uv` (один инструмент вместо pip+venv+conda+poetry). Python 3.13 как безопасный дефолт, 3.14 для нового кода (free-threaded официально поддерживается с 3.14 как `python3.14t`, но не дефолт). DataFrame — `polars` всё активнее в новых проектах за счёт скорости и ленивых вычислений, хотя `pandas` остаётся повсеместным легаси. Ноутбуки — `marimo` как воспроизводимая реактивная альтернатива, JupyterLab как мейнстрим. Редакторы — VS Code и Cursor, AI-ассистент (Claude Code/Copilot) — стандартная часть воркфлоу. Линтер и форматтер — `ruff` (заменил black, isort, flake8).

    Устарело относительно старых курсов: `conda`/`miniconda` как основной менеджер (медленно, тяжело, вытеснено uv для большинства задач); связка `pip` + `virtualenv` + `requirements.txt` вручную (uv делает это лучше и с честным lockfile); `pipenv` и в значительной мере `poetry` (uv быстрее и универсальнее); black + isort + flake8 по отдельности (всё в ruff). requirements.txt без хешей больше не считается воспроизводимым.

!!! warning "Ловушки"

    - `sudo pip install` в системный Python — ломает ОС и плодит конфликты. Никогда. Только окружение проекта.
    - Ноутбук, запускаемый ячейками вразнобой. Выглядит рабочим, не воспроизводится. Restart and Run All перед каждым коммитом, либо переходи на marimo.
    - Секрет в коде или в закоммиченном .env. Утёкший ключ компрометирован навсегда — отзывай и перевыпускай, а не просто удаляй из файла.
    - Коммит .venv/, data/ и обученных моделей в git. Репозиторий распухает необратимо. В git — только код, конфиги, заметки.
    - Отсутствие lockfile («работает с любой версией»). На чужой машине разрешатся другие версии, и результат поедет. Коммить uv.lock.
    - Сеньорская ловушка: вся логика осталась в ноутбуке, ничего не вынесено в src/. Невозможно тестировать, переиспользовать и катить в прод. Граница ноутбук/модуль — признак зрелости.
    - Незафиксированный random seed. Думаешь, что воспроизводимо, а каждый запуск даёт другие train/test split и другие метрики. Фиксируй seed везде, где есть случайность.

## M1.13 · Практика

Разминка:

1. Установи `uv`. Создай проект `uv init --python 3.14 sandbox`, добавь `polars`, посмотри, что появилось в `pyproject.toml` и `uv.lock`. Объясни словами разницу между этими двумя файлами.
2. Инициализируй git, сделай три осмысленных коммита, создай ветку `feature-x`, переключись на неё и обратно. Посмотри `git log --oneline --graph`.

Основное:

3. Развернись по структуре DS-проекта (data/{raw,interim,processed}, src/, notebooks/, tests/). Настрой `.gitignore` так, чтобы `.env`, `.venv/` и `data/` не попадали в git. Проверь через `git status`, что они действительно игнорируются.
4. Возьми любой публичный CSV (например, синтетические транзакции/ставки). Положи его в `data/raw/` как read-only. В marimo-ноутбуке посчитай простую метрику (средний депозит, конверсия). Затем вынеси расчёт признаков в `src/features.py` как функцию и импортируй её в ноутбук.
5. Напиши один pytest-тест к своей функции из src/. Запусти `uv run pytest` и `uvx ruff check .`.

Челлендж:

6. Возьми старый «грязный» Jupyter-ноутбук (свой или чужой с GitHub). Сделай Restart Kernel and Run All. Если упал — почини так, чтобы он исполнялся сверху вниз с нуля. Опиши в `EXPERIMENTS.md`, что было сломано (это и есть проблема скрытого состояния вживую).

Мини-проект:

7. Собери репозиторий `churn-starter`: воспроизводимое окружение (uv + lockfile), чистая структура, один marimo-ноутбук с разведкой данных, логика в src/, минимум один тест, README с тремя командами запуска (clone, `uv sync`, `uv run ...`), `.env.example` без секретов. Запушь на GitHub. Критерий готовности: ты удаляешь `.venv/`, делаешь `uv sync`, `uv run pytest` — и всё зелёное. Дай ссылку коллеге — у него должно завестись с тех же трёх команд.

## M1.15 · Ресурсы

- Официальная документация uv (docs.astral.sh/uv) — единственный источник истины по командам и lockfile, актуален на 2026.
- Документация marimo (docs.marimo.io) — концепция реактивных ноутбуков и проблема скрытого состояния, с примерами.
- Cookiecutter Data Science (cookiecutter-data-science.drivendata.org) — обоснование структуры DS-проекта и принципа неизменяемых сырых данных.
- Pro Git (Scott Chacon, git-scm.com/book, бесплатно) — фундаментальная книга по git, главы 1-3 обязательны.
- The Twelve-Factor App (12factor.net), фактор III «Config» — почему конфигурация и секреты живут в окружении, не в коде.
- Документация CPython по venv и What's New in Python 3.14 (docs.python.org) — про окружения и про официально поддерживаемый free-threaded-билд и JIT.
- Статьи о воспроизводимости Jupyter-ноутбуков (например, исследования Pimentel et al. о доле невоспроизводимых публичных ноутбуков) — эмпирическая база проблемы скрытого состояния.
- Документация ruff (docs.astral.sh/ruff) — современный линтер и форматтер, заменивший связку black/isort/flake8.
