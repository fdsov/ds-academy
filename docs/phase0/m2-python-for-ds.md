## M2.1 · Зачем этот модуль

Python — это не «ещё один язык программирования», а рабочий стол аналитика данных. Что конкретно сломается без этого знания. Ты напишешь цикл, который обрабатывает таблицу ставок построчно, и он будет работать 40 минут вместо 0.4 секунды, потому что ты не понял разницу между списком и генератором. Этот модуль — не справочник по синтаксису (его дёшево спросить у LLM). Это разбор того, как Python устроен внутри и как писать код, который читают и поддерживают сеньоры. Мы строим интуицию: почему так, а не «как набрать символы».

## M2.3 · Модель данных: имена, объекты и (не)изменяемость

Прежде чем говорить о типах, надо понять одну вещь, которую пропускают почти все самоучки и которая порождает половину их багов.

Интуиция. В Python переменная — это не «коробка, в которую кладут значение». Это ярлык (имя), приклеенный к объекту, который живёт где-то в памяти. Когда ты пишешь `a = [1, 2, 3]`, ты создал список-объект и приклеил к нему ярлык `a`. Если потом написать `b = a`, ты приклеил второй ярлык к тому же самому списку. Это не копия. Поменяешь через `a` — увидишь изменение и через `b`, потому что объект один.

Формально. Каждый объект имеет идентичность (адрес в памяти, его показывает `id()`), тип и значение. Присваивание связывает имя со ссылкой на объект, оно никогда не копирует объект. Оператор `is` сравнивает идентичность (тот же объект?), `==` сравнивает значение (равны ли по содержимому?).

```python
deposits = [100, 250, 500]
backup = deposits          # тот же объект, не копия
backup.append(999)
print(deposits)            # [100, 250, 500, 999] — изменилось через другой ярлык
print(deposits is backup)  # True

real_copy = deposits.copy()   # новый объект с тем же содержимым
print(deposits is real_copy)  # False
print(deposits == real_copy)  # True
```

Тут вступает ключевое различие. Объекты делятся на изменяемые (mutable) и неизменяемые (immutable).

- Неизменяемые: `int`, `float`, `str`, `bool`, `tuple`, `frozenset`, `bytes`. Их нельзя поменять «на месте»; любая операция создаёт новый объект.
- Изменяемые: `list`, `dict`, `set`, и почти все пользовательские объекты. Их содержимое можно менять, идентичность при этом сохраняется.

Почему это важно для аналитика. Неизменяемость — это гарантия безопасности. Кортеж координат `(lat, lon)` или ключ когорты `("RU", "2026-06")` нельзя случайно испортить. Поэтому только неизменяемые объекты можно класть в `set` или использовать ключами `dict` — система должна быть уверена, что хеш объекта не поменяется под ней.

!!! warning "Главная ловушка новичка"

    `a == b` не значит `a is b`. И наоборот. Для маленьких целых чисел и коротких строк Python кеширует объекты, поэтому `x = 5; y = 5; x is y` даёт `True` — но это деталь реализации (CPython), а НЕ правило языка. Никогда не сравнивай числа или строки через `is`. `is` используй только для `None`, `True`, `False`.

!!! question "Проверь себя"

    1. Что именно делает присваивание `b = a` для списка — копирует объект или связывает второе имя с тем же объектом?
    2. Чем отличается `is` от `==`? Для каких трёх значений `is` уместен?
    3. Почему ключом `dict` или элементом `set` может быть только неизменяемый объект?
    4. Назови по три изменяемых и три неизменяемых типа.

??? success "Ответы"

    1. Связывает второе имя (ярлык) с тем же самым объектом. Это не копия: изменение через любое имя видно через оба. Копия — `a.copy()` или `a[:]`.
    2. `is` сравнивает идентичность (тот же ли это объект в памяти), `==` сравнивает значение (равно ли содержимое). `is` уместен только для `None`, `True`, `False`.
    3. Потому что коллекция полагается на хеш, а хеш не должен меняться под ней. У изменяемого объекта содержимое (и хеш) можно поменять — это сломало бы поиск.
    4. Изменяемые: `list`, `dict`, `set`. Неизменяемые: `int`, `str`, `tuple` (а также `float`, `bool`, `frozenset`, `bytes`).

## M2.4 · Коллекции: когда какую

Это вопрос, на котором сыпется большинство решений в DS-коде. Выбор коллекции — это выбор асимптотической сложности операций. Разберём четыре основные.

### list — упорядоченный изменяемый массив

Интуиция. Список — это пронумерованная полка: элементы лежат по порядку, к любому есть доступ по индексу, можно добавлять в конец, вставлять в середину. Это рабочая лошадка для последовательностей: список ставок игрока, список дневных депозитов.

Формально. Динамический массив. Доступ по индексу и добавление в конец — амортизированно $O(1)$. Вставка/удаление в начале или середине — $O(n)$, потому что элементы сдвигаются. Проверка `x in lst` — $O(n)$, перебор всех элементов.

```python
bets = [50, 120, 30, 200]
bets.append(75)        # O(1)
bets.insert(0, 10)     # O(n) — сдвиг всех
total = sum(bets)
```

### tuple — упорядоченный неизменяемый кортеж

Интуиция. То же, что список, но «запаянный»: после создания не поменять. Используй, когда набор значений — это единое целое, которое не должно меняться: строка результата, ключ когорты, возврат нескольких значений из функции.

Формально. Неизменяемая последовательность. Чуть компактнее в памяти и быстрее в создании, чем list. Может быть ключом словаря и элементом множества (если все его элементы тоже неизменяемы).

```python
def cohort_stats(rows):
    return len(rows), sum(rows) / len(rows)   # возврат кортежа

count, avg = cohort_stats([100, 200, 300])    # распаковка
cohort_key = ("RU", "2026-06")                # годится в ключ dict
```

### dict — хеш-таблица ключ→значение

Интуиция. Словарь — это телефонная книга: по ключу мгновенно находишь значение. Игрок по id → его профиль. Дата → суммарный GGR за день. Это самая важная структура в продуктовой аналитике после массива.

Формально. Хеш-таблица. Доступ, вставка, удаление по ключу — в среднем $O(1)$. Ключи должны быть хешируемыми (неизменяемыми). С Python 3.7+ словари гарантированно сохраняют порядок вставки.

```python
ggr_by_day = {}
for day, amount in transactions:
    ggr_by_day[day] = ggr_by_day.get(day, 0) + amount   # get с дефолтом — без KeyError
```

### set — множество уникальных элементов

Интуиция. Множество — это мешок без повторов, где главный вопрос «есть ли элемент внутри?». Идеально для дедупликации (уникальные игроки за день) и проверки членства (этот id в чёрном списке?).

Формально. Хеш-таблица без значений. Проверка `x in s`, добавление, удаление — в среднем $O(1)$. Поддерживает теоретико-множественные операции: объединение `|`, пересечение `&`, разность `-`.

```python
players_mon = {101, 102, 103}
players_tue = {102, 103, 104}
retained = players_mon & players_tue   # вернулись: {102, 103}
churned  = players_mon - players_tue   # ушли:     {101}
```

!!! tip "Правило выбора за 5 секунд"

    Нужен порядок и индексы, содержимое меняется → list. Фиксированная запись/ключ → tuple. Поиск по ключу → dict. Уникальность и «есть ли внутри» → set. Если ловишь себя на `x in some_list` внутри цикла по миллиону строк — это $O(n^2)$, замени list на set и получишь $O(n)$.

??? example "Сравнение сложности — почему set спасает retention-расчёт"

    Найти, сколько вчерашних игроков вернулись сегодня. Наивно через списки:

    ```python
    # O(n*m): для каждого вчерашнего ищем во всём сегодняшнем списке
    retained = [p for p in yesterday_list if p in today_list]
    ```

    На 500k игроков это часы. Через множество:

    ```python
    today_set = set(today_list)                    # O(m)
    retained = [p for p in yesterday_list if p in today_set]   # O(n), in — за O(1)
    ```

    Те же строки кода, но из $O(n \cdot m)$ получился $O(n + m)$. Это разница между «не дождался» и «мгновенно».

!!! question "Проверь себя"

    1. Какова средняя асимптотика проверки `x in coll` для list и для set? Почему они разные?
    2. Под какую задачу берут tuple, а не list? Назови минимум одно свойство tuple, которого нет у list.
    3. Какой структурой ты посчитаешь GGR по дням (день → сумма) и почему именно ей?
    4. Что вернут операции `players_mon & players_tue` и `players_mon - players_tue` по смыслу retention?

??? success "Ответы"

    1. Для list — $O(n)$ (линейный перебор), для set — в среднем $O(1)$ (хеш-таблица). Set хранит элементы по хешу и сразу проверяет нужную ячейку.
    2. tuple берут под фиксированную запись/ключ/возврат нескольких значений. Свойство: неизменяемость, поэтому tuple (из неизменяемых элементов) может быть ключом dict и элементом set.
    3. dict: ключ — день, значение — сумма. Доступ и обновление по ключу в среднем $O(1)$, порядок вставки сохраняется (3.7+).
    4. `&` (пересечение) — вернувшиеся игроки (были оба дня); `-` (разность) — ушедшие (были в понедельник, нет во вторник).

## M2.5 · Срезы (slicing)

Интуиция. Срез — это «вырезать кусок последовательности». Как ножницами: с какого элемента начать, на каком закончить, с каким шагом.

Формально. Синтаксис `seq[start:stop:step]`. Включает `start`, исключает `stop` (полуинтервал). Отрицательные индексы считают с конца. Пропущенные части берут значения по умолчанию: начало, конец, шаг 1.

```python
bets = [10, 20, 30, 40, 50, 60]
bets[1:4]     # [20, 30, 40] — со 2-го по 4-й
bets[:3]      # [10, 20, 30] — первые три
bets[-2:]     # [50, 60] — последние два
bets[::2]     # [10, 30, 50] — каждый второй
bets[::-1]    # [60, 50, 40, 30, 20, 10] — разворот
```

Срез списка создаёт новый список (поверхностную копию). Это идиома для безопасного копирования: `copy = bets[:]`. На строках срезы работают так же, но возвращают новую строку (строки неизменяемы).

## M2.6 · Comprehensions: list, dict, set, generator

Это визитная карточка питоничного кода. Где новичок пишет цикл с `append`, профессионал пишет одну строку.

Интуиция. Comprehension — это «собери новую коллекцию из старой по правилу». Читается как фраза на английском: «возьми x для каждого x из данных, где выполнено условие».

Формально. Выражение `[выражение for элемент in итерабельное if условие]`. Эквивалентно циклу с накоплением, но быстрее (оптимизировано в интерпретаторе) и читается одним взглядом.

```python
# list comprehension: суммы выигрышных ставок
wins = [b.amount for b in bets if b.outcome == "win"]

# dict comprehension: id игрока -> его LTV
ltv = {p.id: p.deposits - p.withdrawals for p in players}

# set comprehension: уникальные страны
countries = {p.country for p in players}

# generator expression: ленивая сумма, без построения списка в памяти
total_ggr = sum(b.stake - b.payout for b in all_bets)
```

Разница между скобками принципиальна. Квадратные `[]` строят список целиком в памяти. Круглые `()` создают генератор — он не хранит результат, а выдаёт элементы по одному по требованию. На миллиарде ставок `sum([... for ...])` съест всю память, а `sum(... for ...)` пройдёт потоком за константную память.

!!! warning "Не превращай comprehension в нечитаемое"

    Вложенные comprehensions с двумя-тремя условиями — это уже не питонично, а ребус. Если строка не читается за один взгляд, разверни в обычный цикл. Питоничность — про ясность, а не про «в одну строку любой ценой».

!!! question "Проверь себя"

    1. В чём принципиальная разница между `[x for x in data]` и `(x for x in data)` по памяти?
    2. Почему `sum(b.stake for b in all_bets)` на миллиарде ставок предпочтительнее, чем `sum([b.stake for b in all_bets])`?
    3. Когда comprehension стоит развернуть обратно в обычный цикл?

??? success "Ответы"

    1. Квадратные скобки строят список целиком в памяти (память = размер результата). Круглые создают генератор: элементы выдаются по одному по требованию, результат не хранится.
    2. Версия с генератором проходит данные потоком за константную память; версия со списком сначала материализует весь список и съест всю память.
    3. Когда читаемость страдает: вложенные comprehensions с двумя-тремя условиями превращаются в ребус. Питоничность — про ясность, а не «в одну строку любой ценой».

## M2.7 · Функции: args, kwargs, lambda

Функция — базовая единица переиспользования. Разберём то, что выходит за рамки «def с двумя аргументами».

### Позиционные, именованные, *args, **kwargs

Интуиция. `*args` — «сколько угодно безымянных аргументов, собери их в кортеж». `**kwargs` — «сколько угодно именованных, собери в словарь». Это нужно, когда заранее не знаешь, сколько параметров передадут.

Формально. При вызове `*` распаковывает итерабельное в позиционные аргументы, `**` распаковывает словарь в именованные. При определении функции они, наоборот, собирают лишние аргументы.

```python
def ggr(*stakes, hold=0.05, **meta):
    base = sum(stakes) * hold
    return base, meta

result, info = ggr(100, 200, 300, hold=0.04, market="UZ")
# stakes=(100,200,300), hold=0.04, meta={"market":"UZ"}
```

### lambda — анонимная функция-однострочник

Интуиция. lambda — это функция без имени для одноразового использования, чаще всего как ключ сортировки или аргумент `map`/`filter`.

Формально. `lambda аргументы: выражение`. Возвращает значение выражения. Не может содержать операторов (только выражение), поэтому годится лишь для простой логики.

```python
players.sort(key=lambda p: p.ltv, reverse=True)   # сортировка по LTV убыванию
top_depositors = sorted(deposits, key=lambda d: d.amount)[-10:]
```

!!! warning "Изменяемый аргумент по умолчанию — классический баг сеньоров"

    Значение по умолчанию вычисляется ОДИН раз при определении функции, а не при каждом вызове. Поэтому изменяемый дефолт живёт между вызовами.

    ```python
    def add_bet(bet, history=[]):     # ОПАСНО
        history.append(bet)
        return history

    add_bet(10)   # [10]
    add_bet(20)   # [10, 20] — тот же список! не [20]
    ```

    Правильно — дефолт `None`, список создаём внутри:

    ```python
    def add_bet(bet, history=None):
        if history is None:
            history = []
        history.append(bet)
        return history
    ```

    2. Почему `def add_bet(bet, history=[])` опасен? Что происходит между вызовами?
    3. Как правильно задать изменяемый аргумент по умолчанию?
    4. Какое ограничение есть у `lambda` по сравнению с `def`?

??? success "Ответы"

    1. `*args` собирает лишние позиционные аргументы в кортеж (tuple), `**kwargs` — лишние именованные в словарь (dict).
    2. Значение по умолчанию вычисляется один раз при определении функции, а не при каждом вызове. Один и тот же список живёт между вызовами и накапливает данные.
    3. Дефолт `None`, а коллекцию создавать внутри: `if history is None: history = []`.
    4. `lambda` содержит только выражение (одно), в неё нельзя поместить операторы/несколько строк. Годится лишь для простой логики (ключ сортировки, аргумент map/filter).

## M2.8 · Модули и импорты

Интуиция. Модуль — это просто файл `.py`. Импорт — «дай мне доступ к коду из другого файла». Пакет — папка с модулями.

Формально. `import module` загружает модуль и даёт к нему доступ через пространство имён `module.func`. `from module import name` тянет конкретное имя в текущее пространство. Python ищет модули по путям из `sys.path`. Модуль выполняется один раз при первом импорте и кешируется.

```python
import numpy as np                      # алиас — идиома сообщества
from collections import Counter         # точечный импорт
from pathlib import Path

if __name__ == "__main__":              # код только при прямом запуске файла
    main()
```

Конструкция `if __name__ == "__main__"` — важная идиома. Она отделяет «код, который выполняется при запуске файла как скрипта» от «кода, который импортируется как библиотека». Без неё импорт твоего модуля случайно запустит весь анализ.

## M2.9 · ООП для Data Science: зачем аналитику классы

Тут нужна честность. В DS большая часть кода — функциональная: данные внутрь, преобразование, данные наружу. Но ООП незаменимо в трёх местах: кастомные трансформеры для пайплайнов scikit-learn, конфигурации с состоянием, и обёртки над моделями. Понимать ООП обязательно ещё и потому, что весь sklearn, PyTorch и pandas построены на классах.

Интуиция. Класс — это чертёж, объект — изделие по чертежу. Класс `Player` описывает, какие данные есть у игрока (атрибуты) и что с ним можно делать (методы). Создаёшь конкретного игрока — это экземпляр.

Формально. Класс инкапсулирует данные (атрибуты) и поведение (методы). `__init__` — конструктор, вызывается при создании экземпляра, `self` — ссылка на сам экземпляр. Наследование позволяет переиспользовать и расширять поведение.

```python
class Player:
    def __init__(self, player_id: int, deposits: float, withdrawals: float):
        self.player_id = player_id
        self.deposits = deposits
        self.withdrawals = withdrawals

    def ltv(self) -> float:
        return self.deposits - self.withdrawals

    def __repr__(self) -> str:        # как объект печатается
        return f"Player(id={self.player_id}, ltv={self.ltv():.2f})"


class VIPPlayer(Player):              # наследование
    def __init__(self, player_id, deposits, withdrawals, manager: str):
        super().__init__(player_id, deposits, withdrawals)
        self.manager = manager

    def ltv(self) -> float:           # переопределение: VIP-бонус 10%
        return super().ltv() * 1.1
```

### Dunder-методы

Интуиция. Это «магические» методы с двойными подчёркиваниями, которые позволяют твоему объекту вести себя как встроенный: складываться, печататься, сравниваться, попадать в множество.

Формально. Python вызывает их неявно. `__init__` — конструктор, `__repr__`/`__str__` — текстовое представление, `__eq__` — сравнение `==`, `__hash__` — хеш для использования в set/dict, `__len__` — длина, `__lt__` — оператор `<` для сортировки.

```python
class Cohort:
    def __init__(self, market, month):
        self.market = market
        self.month = month

    def __eq__(self, other):
        return (self.market, self.month) == (other.market, other.month)

    def __hash__(self):
        return hash((self.market, self.month))   # теперь Cohort можно класть в set
```

!!! tip "Современная идиома: dataclass вместо ручного __init__"

    Писать `__init__`, `__repr__`, `__eq__` руками — устаревший стиль. С Python 3.7+ есть `@dataclass`, который генерирует их автоматически.

    ```python
    from dataclasses import dataclass

    @dataclass(frozen=True)          # frozen=True делает объект неизменяемым и хешируемым
    class Cohort:
        market: str
        month: str

    c = Cohort("RU", "2026-06")      # __init__, __repr__, __eq__, __hash__ — бесплатно
    ```

!!! question "Проверь себя"

    1. В каких трёх местах DS-кода ООП реально оправдано, а не лишнее?
    2. Что делают `__init__`, `self` и `__repr__`?
    3. Зачем переопределять `__eq__` и `__hash__`, чтобы класть свои объекты в `set`?
    4. Что генерирует `@dataclass` автоматически и чем полезен `frozen=True`?

??? success "Ответы"

    1. Кастомные трансформеры для пайплайнов sklearn, конфигурации с состоянием, обёртки над моделями. Плюс — sklearn/PyTorch/pandas сами построены на классах.
    2. `__init__` — конструктор (вызывается при создании экземпляра), `self` — ссылка на сам экземпляр, `__repr__` — текстовое представление объекта при печати.
    3. `__eq__` задаёт сравнение по значению, `__hash__` — стабильный хеш; без согласованной пары объект нельзя корректно хранить в set/использовать ключом dict.
    4. Генерирует `__init__`, `__repr__`, `__eq__` (и `__hash__` при frozen). `frozen=True` делает объект неизменяемым и хешируемым.

## M2.10 · Декораторы

Интуиция. Декоратор — это обёртка вокруг функции, которая добавляет поведение, не трогая саму функцию. Как чехол на телефон: телефон тот же, но добавилась защита. Типичное применение: логирование, замер времени, кеширование, проверка прав.

Формально. Декоратор — это функция, которая принимает функцию и возвращает новую функцию. Синтаксис `@decorator` над определением — синтаксический сахар для `func = decorator(func)`.

```python
import functools
import time

def timed(func):
    @functools.wraps(func)            # сохраняет имя и docstring оригинала
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        print(f"{func.__name__}: {time.perf_counter() - start:.3f}s")
        return result
    return wrapper

@timed
def compute_retention(events):
    ...                               # тяжёлый расчёт
    return result
```

Самый полезный встроенный декоратор для аналитика — `functools.lru_cache`. Он кеширует результаты функции: вызвал с теми же аргументами — мгновенно вернул запомненное. Это спасает при повторных дорогих запросах к одним и тем же когортам.

```python
from functools import lru_cache

@lru_cache(maxsize=None)
def cohort_size(market: str, month: str) -> int:
    return expensive_db_query(market, month)   # второй вызов с теми же аргументами — без запроса
```

## M2.11 · Генераторы, yield и ленивые вычисления

Это самая важная тема для работы с большими данными на чистом Python.

Интуиция. Обычная функция готовит всё блюдо и отдаёт целиком. Генератор — это конвейер: отдаёт по одному элементу, когда у него просят, и помнит, где остановился. Он не держит весь результат в памяти. На логе ставок в 50 ГБ это разница между «упало по памяти» и «обработалось потоком».

Формально. Функция с `yield` вместо `return` становится генератором. При вызове она не выполняется, а возвращает объект-генератор. Каждый `next()` выполняет код до следующего `yield`, отдаёт значение и замораживает состояние. Это ленивые вычисления: элемент производится только в момент запроса.

```python
def read_bets(path):
    with open(path) as f:
        for line in f:
            yield parse_bet(line)     # отдаёт по одной ставке, файл целиком в память не грузится

# обработка 50 ГБ за константную память
total = sum(b.stake for b in read_bets("bets.log") if b.stake > 0)
```

Сравни. `lines = f.readlines()` загрузит весь файл в список (память = размер файла). Генератор обрабатывает строку, выбрасывает, берёт следующую. Память почти не растёт.

??? example "Конвейер генераторов — потоковая обработка депозитов"

    Генераторы можно соединять в конвейер, и данные текут сквозь него по одному элементу, не материализуясь:

    ```python
    def parse(lines):
        for line in lines:
            yield json.loads(line)

    def only_deposits(events):
        for e in events:
            if e["type"] == "deposit":
                yield e

    def amounts(deposits):
        for d in deposits:
            yield d["amount"]

    with open("events.jsonl") as f:
        total = sum(amounts(only_deposits(parse(f))))
    # весь файл прошёл потоком: в памяти всегда одна запись
    ```

!!! question "Проверь себя"

    1. Чем функция с `yield` отличается от функции с `return` при вызове?
    2. Что значит «ленивые вычисления» и почему генератор обрабатывает лог в 50 ГБ за константную память?
    3. Чем `f.readlines()` хуже генератора при чтении большого файла?
    4. Что такое конвейер генераторов и сколько записей лежит в памяти, когда данные текут сквозь него?

??? success "Ответы"

    1. Функция с `yield` при вызове не выполняется, а возвращает объект-генератор; код идёт до следующего `yield` только на каждом `next()`, замораживая состояние между ними.
    2. Элемент производится только в момент запроса. Генератор держит в памяти одну запись за раз, а не весь результат, поэтому размер входа почти не влияет на потребление памяти.
    3. `readlines()` загружает весь файл в список (память = размер файла). Генератор читает строку, обрабатывает, выбрасывает и берёт следующую.
    4. Цепочка генераторов, где выход одного — вход другого. Данные текут по одному элементу, ничего не материализуется; в памяти всегда одна запись.

## M2.12 · Контекстные менеджеры (with)

Интуиция. `with` — это «гарантированная уборка»: что бы ни случилось внутри блока (даже ошибка), ресурс будет закрыт. Открыл файл — он точно закроется. Взял соединение с БД — оно точно вернётся.

Формально. Контекстный менеджер — объект с методами `__enter__` (вход в блок) и `__exit__` (выход, в том числе при исключении). `with` гарантирует вызов `__exit__` всегда. Свой контекстный менеджер удобно писать через `contextlib.contextmanager`.

```python
from contextlib import contextmanager

@contextmanager
def timer(label):
    start = time.perf_counter()
    try:
        yield                         # здесь выполняется тело with
    finally:
        print(f"{label}: {time.perf_counter() - start:.2f}s")

with timer("retention join"):
    result = heavy_join(df_a, df_b)   # время напечатается даже при ошибке
```

## M2.13 · Обработка исключений

Интуиция. Исключение — это «что-то пошло не так, нормальный ход прерван». Вместо того чтобы программа упала, ты ловишь ошибку и решаешь, что делать: пропустить строку, подставить дефолт, залогировать.

Формально. `try` — рискованный код, `except КонкретныйТип` — обработка конкретной ошибки, `else` — если ошибки не было, `finally` — выполнится всегда. Лови конкретные исключения, не голый `except`.

```python
def safe_ratio(payout, stake):
    try:
        return payout / stake
    except ZeroDivisionError:
        return 0.0                    # ставка нулевая — осмысленный дефолт
    except TypeError:
        raise ValueError(f"нечисловые данные: {payout}, {stake}")
```

!!! warning "Не глотай исключения молча"

    `except: pass` — это профессиональное преступление. Ты прячешь баг, который всплывёт в проде искажёнными цифрами. Лови конкретный тип, логируй, и если не знаешь, что делать — пробрасывай дальше через `raise`.

## M2.14 · Аннотации типов (type hints)

Интуиция. Аннотации — это подписи к коду: «эта функция принимает список чисел и возвращает float». Python их не проверяет в рантайме, но они нужны людям и инструментам: IDE подсказывает, статический анализатор ловит ошибки до запуска.

Формально. Синтаксис `def f(x: int) -> str`. С Python 3.9+ встроенные дженерики пишутся как `list[int]`, `dict[str, float]`. Проверяются статически инструментами `mypy` или `pyright`, в рантайме игнорируются.

```python
def aggregate_ltv(players: list[Player], market: str | None = None) -> dict[str, float]:
    result: dict[str, float] = {}
    for p in players:
        if market is None or p.market == market:
            result[p.id] = p.ltv()
    return result
```

В DS аннотации особенно ценны на границах: сигнатуры функций, которые принимают DataFrame и что-то возвращают. Они документируют контракт лучше любого комментария и не устаревают незаметно, потому что их проверяет линтер.

## M2.15 · f-strings

Интуиция. f-строка — это шаблон, в который прямо внутри вставляются значения переменных. Современный, читаемый способ форматирования.

Формально. Строка с префиксом `f`, выражения внутри `{}` вычисляются и подставляются. Поддерживает форматные спецификаторы после `:` и режим отладки `=`.

```python
ltv = 1234.5678
market = "RU"
print(f"LTV={ltv:.2f} в {market}")       # LTV=1234.57 в RU
print(f"{ltv:,.0f}")                       # 1,235 — разделитель тысяч
print(f"{ltv=}")                           # ltv=1234.5678 — режим отладки
conv = 0.1834
print(f"конверсия {conv:.1%}")             # конверсия 18.3%
```

## M2.16 · Стандартная библиотека для DS

Перед тем как ставить pandas, знай, что в стандартной библиотеке уже лежит много нужного. Это работает без зависимостей и часто быстрее на простых задачах.

### collections

`Counter` — подсчёт частот за одну строку. `defaultdict` — словарь с автозначением, избавляет от проверок «есть ли ключ». `deque` — очередь с быстрыми операциями с обоих концов.

```python
from collections import Counter, defaultdict

bets_by_game = Counter(b.game for b in bets)        # {"slots": 1200, "roulette": 340, ...}
top3 = bets_by_game.most_common(3)

deposits = defaultdict(list)
for d in all_deposits:
    deposits[d.player_id].append(d.amount)          # без проверки на существование ключа
```

### itertools

Инструменты для эффективной работы с итераторами, всё ленивое.

```python
from itertools import groupby, chain, islice, accumulate

cumulative_ggr = list(accumulate(daily_ggr))        # нарастающий итог

```

Работа с датами и временем — основа когортного анализа.

Сериализация в JSON и обратно — формат логов событий и API.

Собирающий пример: потоковый расчёт D7-retention по логу событий, на чистом Python со стандартной библиотекой.

```python
from __future__ import annotations
import json
from collections import defaultdict
from datetime import datetime, timedelta
from pathlib import Path
from dataclasses import dataclass


@dataclass(frozen=True)
class Event:
    player_id: int
    ts: datetime
    kind: str


def read_events(path: Path):
    """Ленивое чтение лога: одна запись в памяти за раз."""
    with path.open() as f:
        for line in f:
            raw = json.loads(line)
            yield Event(
                player_id=raw["player_id"],
                ts=datetime.fromisoformat(raw["ts"]),
                kind=raw["type"],
            )


def d7_retention(path: Path) -> float:
    first_seen: dict[int, datetime] = {}
    active_days: dict[int, set] = defaultdict(set)

    for e in read_events(path):
        if e.player_id not in first_seen:
            first_seen[e.player_id] = e.ts
        active_days[e.player_id].add(e.ts.date())

    retained = 0
    for pid, reg in first_seen.items():
        target = (reg + timedelta(days=7)).date()
        if target in active_days[pid]:
            retained += 1

    cohort = len(first_seen)
    return retained / cohort if cohort else 0.0


if __name__ == "__main__":
    rate = d7_retention(Path("events.jsonl"))
    print(f"D7 retention: {rate:.1%}")
```

Здесь работает почти всё из модуля: dataclass, генератор для ленивого чтения, контекстный менеджер `with`, `defaultdict` и `set` для уникальных дней, аннотации типов, f-string, идиома `__main__`.

!!! tip "AI-копилот: где нейросети помогают"

    LLM (Claude, Cursor, Copilot) сильны в Python-рутине: сгенерировать comprehension, написать декоратор по описанию, объяснить чужой код, предложить рефакторинг цикла в векторную форму. Это ускоряет в разы.

    Пример хорошего промпта: «Вот функция, которая считает retention циклом по списку с проверкой `in` на 2 млн игроков и работает 20 минут. Перепиши, заменив списки на множества/словари, объясни итоговую асимптотику и где именно было $O(n^2)$. Сохрани аннотации типов и стиль PEP 8.»

    Где LLM врут и что перепроверять. Первое — асимптотика и память: модель уверенно скажет «оптимально», хотя оставит скрытый $O(n^2)$ или материализует список там, где нужен генератор; всегда сам проверяй сложность на горячем пути. Второе — изменяемые дефолты и побочные эффекты: модель спокойно сгенерирует `def f(x, acc=[])`; читай сигнатуры. Третье — устаревший стек: по обучающим данным LLM тянет `os.path`, `black`+`flake8`, `poetry` вместо `pathlib`, `ruff`, `uv`. Четвёртое — тонкости копирования (поверхностная против глубокой) и `is` vs `==`. Правило: LLM пишет черновик, ты держишь модель данных и сложность в голове.

!!! note "Стек 2026"

    Актуально сейчас: `uv` для окружений и пакетов (вытеснил pip/venv/poetry), `ruff` для линта и форматирования (вытеснил black + flake8 + isort), `pyright`/`mypy` для типов, dataclasses и `@dataclass(frozen=True)` вместо ручных конструкторов, `pathlib` вместо `os.path`, f-strings вместо `.format()` и `%`, встроенные дженерики `list[int]` и `str | None` вместо `typing.List`, `typing.Optional`. Polars всё активнее теснит pandas на больших данных, но pandas остаётся стандартом.

    Устарело относительно старых курсов: `%`-форматирование и `.format()`, `os.path.join`, ручное открытие/закрытие файлов без `with`, `from typing import List, Dict, Optional`, классы с написанными вручную `__init__`/`__repr__` там, где хватает dataclass, и совет «pip install в глобальное окружение».

!!! warning "Ловушки"

    - Изменяемый аргумент по умолчанию (`def f(x, acc=[])`) — живёт между вызовами. Дефолт `None`, коллекцию создавай внутри.
    - Сравнение чисел/строк через `is` — работает «случайно» из-за кеширования, ломается непредсказуемо. `is` только для `None`/`True`/`False`.
    - `b = a` для списка/словаря — это второй ярлык, не копия. Меняешь один — меняется оба. Нужна копия — `a.copy()`, для вложенных — `copy.deepcopy`.
    - `x in some_list` внутри цикла — $O(n^2)$. На больших данных переводи в set.
    - `[... for ...]` вместо `(... for ...)` там, где результат сразу агрегируется (`sum`, `any`, `max`) — лишняя материализация и память.
    - `except: pass` — глотает баги. Лови конкретный тип, логируй.
    - Поверхностная копия вложенной структуры: `outer.copy()` копирует только верхний уровень, внутренние списки общие.
    - Float-арифметика денег: `0.1 + 0.2 != 0.3`. Для финансов используй `decimal.Decimal` или считай в копейках целыми.
    - Мутация коллекции во время итерации по ней — приводит к пропускам и ошибкам. Итерируй по копии или строй новую.
