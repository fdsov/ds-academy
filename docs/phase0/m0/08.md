Покажем парадокс Симпсона на синтетических данных — почувствовать руками, как агрегат врёт. Стек актуален на июнь 2026: Python 3.12+, polars как основной DataFrame-движок для аналитики, pandas там, где нужна совместимость, numpy для расчётов.

```python
import polars as pl

# Синтетика: A/B по платёжной странице, конфаундер — тип устройства
df = pl.DataFrame({
    "variant":    ["A"]*1000 + ["B"]*1000,
    "device":     (["desktop"]*800 + ["mobile"]*200) + (["desktop"]*200 + ["mobile"]*800),
    # конверсия в депозит: B лучше внутри КАЖДОГО устройства,
    # но в B перекошен трафик в сторону mobile с низкой базовой конверсией
    "converted":  (
        [1]*240 + [0]*560 +   # A desktop: 240/800 = 0.30
        [1]*30  + [0]*170     # A mobile:  30/200  = 0.15
    ) + (
        [1]*70  + [0]*130 +   # B desktop: 70/200  = 0.35  (выше, чем A)
        [1]*144 + [0]*656     # B mobile:  144/800 = 0.18  (выше, чем A)
    ),
})

by_device = (
    df.group_by(["variant", "device"])
      .agg(pl.col("converted").mean().alias("conv"))
      .sort(["device", "variant"])
)

aggregate = (
    df.group_by("variant")
      .agg(pl.col("converted").mean().alias("conv"))
      .sort("variant")
)

print(by_device)   # внутри каждого устройства B > A
print(aggregate)   # в агрегате A > B — парадокс Симпсона
```

Вывод покажет: на desktop и на mobile по отдельности вариант B конвертит лучше, но суммарно A выигрывает — потому что в B загнали больше мобильного трафика с низкой базой. Это и есть тот случай, когда наивный агрегат приведёт к неверному решению "откатить B".

Полезная привычка с первого дня — сразу считать неопределённость, а не голую точку:

```python
import numpy as np
from scipy import stats

def conv_ci(successes: int, n: int, alpha: float = 0.05):
    p = successes / n
    z = stats.norm.ppf(1 - alpha / 2)
    half = z * np.sqrt(p * (1 - p) / n)   # нормальная аппроксимация; для малых n брать Wilson
    return p, (p - half, p + half)

print(conv_ci(214, 1000))  # точка + 95% доверительный интервал
```

Цифра без интервала — мнение. Цифра с интервалом — результат. Сделай это рефлексом.

!!! tip "AI-копилот: где нейросети помогают"

    LLM (Claude, Cursor, Copilot) в 2026 — мощный ускоритель именно на этом модуле, но как оппонент, а не как оракул.

    Сильные применения:

    - Сгенерировать список альтернативных объяснений вывода. Промпт: "Вот мой вывод: игроки с приложением приносят в 3 раза больше GGR. Сыграй роль скептика-статистика и перечисли все причины, кроме причинной связи 'приложение растит GGR', которые могли породить эту корреляцию. Для каждой предложи, как её проверить на данных."
    - Помочь подобрать корректный статистический метод под вопрос и дизайн данных.
    - Объяснить незнакомую ловушку на твоём конкретном примере и сгенерировать синтетику для проверки интуиции.

    Где LLM врёт и что перепроверять обязательно:

    - Может уверенно подтвердить причинность там, где её нет — модель склонна соглашаться с твоей формулировкой (sycophancy). Если ты в промпте уже намекнул на желаемый вывод, она его поддержит. Спрашивай нейтрально или прямо проси опровергнуть.
    - Путает поправки на множественность, формулы доверительных интервалов, условия применимости тестов. Любую формулу и порог сверяй с первоисточником.
    - Выдумывает несуществующие "правила" и ссылки. Проверяй факты, особенно численные пороги и названия методов.

    Правильная роль LLM здесь — твой персональный adversarial-reviewer, который помогает атаковать собственный вывод. Финальную ответственность за корректность несёшь ты.

!!! note "Стек 2026"

    Что актуально сейчас:

    - polars стал дефолтом для аналитики на одной машине: быстрее pandas, ленивые вычисления, честная работа с большими данными. pandas жив для legacy и узких API.
    - DuckDB — стандарт для аналитики "SQL прямо по файлам/датафреймам" без поднятия хранилища.
    - Эксперименты и причинность вышли из академии в продукт: причинный вывод (uplift-моделирование, синтетический контроль) — рабочий инструмент аналитика, а не экзотика.
    - LLM встроены в IDE и в рабочий процесс по умолчанию; навык не "знать ли про них", а "не верить им слепо".

    Что устарело относительно старых курсов:

    - Подача Data Science как "обучи нейросеть на Kaggle и ты в деле" — это узкая часть профессии, а не она сама.
    - Чистый pandas-only стек без polars/DuckDB на новых проектах.
    - Преподавание статистики и кода в отрыве от научного метода и причинности — именно так и производят выпускников danger zone.

!!! warning "Ловушки"

    - Путать корреляцию и причинность под давлением дедлайна: вывод "X влияет на Y" без эксперимента или причинного дизайна — почти всегда преждевременный. На этом горят и сеньоры, потому что красивый график убедителен.
    - Смотреть только на "выживших": считать LTV и lifetime по активным/VIP-игрокам, забывая входную когорту. Самая частая причина завышенного прогноза ценности.
    - Доверять агрегату без разбивки по ключевым конфаундерам (устройство, гео, канал) — парадокс Симпсона ждёт именно там.
    - Подглядывать в A/B-тест и останавливать его, как только увидел значимость. Это форма p-hacking, которая раздувает ложные победы. Размер выборки и метрику фиксируй до старта.
    - Резать данные на сегменты в поисках "хоть где-то значимого" эффекта и отчитываться о найденном. Чем больше срезов, тем выше вероятность фальшивки.
    - Сидеть в danger zone незаметно для себя: чем ты опытнее в домене и сильнее в коде, тем легче перестать перепроверять себя статистикой. Авторитет усиливает ущерб от ошибки.
    - Показывать точку без неопределённости. Вывод без доверительного интервала или без оценки риска ошибки — это не результат.
