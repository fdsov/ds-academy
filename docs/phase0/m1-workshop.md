!!! info "Как устроен этот воркшоп"

    Это не лекция с готовым кодом, а задачник. Каждый содержательный шаг построен так:

    - **Задача** — что именно сделать руками, с явными входами и именами выходных переменных.
    - **Критерий шага** — либо кусок кода с `assert`, который ты запускаешь после своего решения (зелёный прогон = шаг сдан, это твой локальный авто-грейдер), либо чек-лист `self-check` для шагов, где результат — это состояние файловой системы или внешних инструментов (`git`, `uv`, `marimo`), а не число.
    - **Решение** спрятано под спойлер `Решение`. Открывай его только после своей попытки — чтобы сверить подход, а не списать.
    - **Числовые проверки** считаются прямо на странице: получи число, впиши в поле, нажми «Проверить».

    Все числа и `assert` рассчитаны на сгенерированные данные с `SEED=42`. Не меняй seed, иначе контрольные суммы поплывут.

## Бизнес-кейс

Воспроизводимость кажется «инженерной гигиеной», но на деле это про доверие к цифре, на которой стоит бюджет. Этот воркшоп — не учебный репозиторий ради репозитория: ты собираешь скелет, на котором завтра будет жить настоящая churn/RFM-аналитика, и от его надёжности зависит, поверят ли твоему числу при разговоре о деньгах.

!!! example "Ситуация"

    Ты — аналитик в команде iGaming-продукта. Head of Retention готовит защиту бюджета на удержание и просит тебя: «Посчитай RFM-сегменты и дай churn-скор по когортам — на основе этого делю 4 000 000 ₽ в квартал между push-кампаниями и VIP-обзвоном». Полтора месяца назад похожий расчёт уже разошёлся: у тебя выходило, что в зоне риска ~18% депозитчиков, у коллеги на «той же» выгрузке — 12%. Разница в 6 п.п. — это ориентировочно 1 200 000 ₽ бюджета, уехавшего не в тот канал, и подорванное доверие к аналитике в целом.

    - **Что на кону.** Распределение 4 000 000 ₽/квартал и репутация числа, под которым ты подпишешься перед стейкхолдером.
    - **От чего разошлись результаты.** Разные версии библиотек, незафиксированный random seed, «подправленный руками» сырой файл, утечка будущих ставок в признаки.
    - **Ограничение.** До защиты бюджета 3 дня; внешних данных нет, работаешь на синтетической выгрузке ставок, которая должна воспроизводиться байт-в-байт у любого ревьюера.

    Твоя задача — закрыть саму причину расхождений: собрать проект так, чтобы тот же расчёт у Head of Retention, у коллеги и в CI давал идентичный результат с трёх команд. Без этого фундамента любая последующая модель удержания — это число, которое нельзя защитить.

## Данные

Никаких внешних загрузок: генерируем синтетические ставки игроков детерминированно с фиксированным seed. Это сразу закрывает слой «случайность» из M1.3 — у любого, кто запустит скрипт, получится байт-в-байт тот же CSV. Этот файл станет неизменяемым сырьём в `data/raw/`.

Это setup-код, а не задача — скопируй и используй как есть. Положишь его в `scripts/make_data.py` на шаге 4.

```python
# scripts/make_data.py — генератор сырых данных, кладётся ВНЕ data/raw
import numpy as np
import polars as pl

SEED = 42

def generate_bets(n_players: int = 500, n_bets: int = 20_000) -> pl.DataFrame:
    rng = np.random.default_rng(SEED)
    player_id = rng.integers(1, n_players + 1, size=n_bets)
    day_offset = rng.integers(0, 180, size=n_bets)
    bet_ts = (np.datetime64("2026-01-01") + day_offset.astype("timedelta64[D]"))
    stake = np.round(rng.gamma(shape=2.0, scale=50.0, size=n_bets), 2)
    is_first_deposit = rng.random(n_bets) < 0.03
    return pl.DataFrame({
        "player_id": player_id,
        "bet_ts": bet_ts.astype(str),
        "stake": stake,
        "is_first_deposit": is_first_deposit,
    }).sort(["player_id", "bet_ts"])

if __name__ == "__main__":
    df = generate_bets()
    df.write_csv("data/raw/bets.csv")
    print(f"wrote {df.height} rows, hash check:", df["stake"].sum())
```

Фиксированный seed плюс детерминированный генератор означают: повторный запуск даёт идентичный `bets.csv`. Контрольная сумма `stake` в конце — твой быстрый способ убедиться, что данные те же, что и у соседа. Прогони генератор и сверь контрольную сумму:

```text
TASK: Запусти генератор (или make_data.py) и посмотри строку hash check. Чему равна сумма всех stake в bets.csv? Округли до 0.1.
ANSWER: 1992845.5
TOL: 1.0
PLACEHOLDER: 0.0
EXPLAIN: Сумма по 20000 округлённым до 2 знаков ставкам из gamma(shape=2, scale=50) при SEED=42 равна 1992845.5. Это и есть "хеш" датасета: если у коллеги другое число - значит у него другой seed, другая версия numpy/polars или кто-то правил файл руками. Именно так ловится первая причина расхождения отчётов.
```

!!! tip "Чем заменить на реальные данные"

    Подойдёт любая реальная квартальная выгрузка ставок: колонки `player_id`, `bet_ts`, `stake`, флаг первого депозита. Структура та же — меняешь только источник в `make_data.py`, остальной пайплайн (RFM, тесты, проверка воспроизводимости) работает без правок. Принципиально важно: и на реальных данных сырьё в `data/raw/` остаётся неизменяемым, а версия выгрузки фиксируется (дата/хеш файла).

### Шаг 4: Генерируем данные и выносим логику в src/

**Зачем.** Отрабатываем границу «ноутбук против `src/`» из M1.10: переиспользуемая логика живёт функцией в `src/`, ноутбук её только импортирует. И заодно защиту от утечки будущего (cutoff), как в примере модуля.

Сохрани результат на cutoff `"2026-04-01"` в переменную `rfm` для проверки.

Сначала прикинь, сколько данных вообще попадёт в окно:

```text
TASK: При cutoff="2026-04-01" сколько ставок попадает в RFM-окно (строк с bet_ts строго раньше cutoff)? Ответ - целое число.
ANSWER: 9919
TOL: 0
PLACEHOLDER: целое число
EXPLAIN: Окно с 2026-01-01 по 2026-04-01 - это ~90 дней из 180-дневного диапазона генерации, поэтому в признаки попадает примерно половина: ровно 9919 из 20000 ставок. Остальные 10081 - это "будущее" относительно cutoff, и в RFM они не идут. Если бы ты случайно посчитал признаки по всем 20000 - это и была бы классическая утечка будущего в фичи.
```

??? tip "Подсказка"

    Отфильтруй ставки по `bet_ts < cutoff` (приведи `cutoff` к datetime), затем `group_by("player_id").agg(...)`. Recency — разница между `cutoff` и максимальным `bet_ts` игрока в днях; frequency — число ставок (`pl.len()`); monetary — сумма `stake`.

??? success "Решение"

    ```python
    # src/features.py
    import polars as pl

    def build_rfm_features(bets: pl.DataFrame, cutoff: str) -> pl.DataFrame:
        """RFM-признаки игроков строго до даты cutoff (защита от утечки будущего)."""
        window = bets.filter(pl.col("bet_ts") < pl.lit(cutoff).str.to_datetime())
        return window.group_by("player_id").agg(
            recency_days=(pl.lit(cutoff).str.to_datetime() - pl.col("bet_ts").max()).dt.total_days(),
            frequency=pl.len(),
            monetary=pl.col("stake").sum(),
        ).sort("player_id")
    ```

    **Почему так.** Фильтр `bet_ts < cutoff` — это и есть защита от утечки: в признаки попадают только события, известные на момент cutoff, а «будущие» ставки отбрасываются. Логика лежит импортируемой функцией в `src/`, а не заперта в ячейке ноутбука — это и есть признак зрелости из M1.10: один и тот же расчёт переиспользуют и ноутбук, и тест, и продовый пайплайн.

**Критерий шага.** Прогони после того, как написал функцию (`df` — результат `generate_bets()`, приведённый к datetime):


### Шаг 6: Тест на защиту от утечки

## Типичные ошибки

- **Коммит только `pyproject.toml` без `uv.lock`.** Намерение есть, точное дерево версий — нет. У коллеги разрешатся другие версии, и расчёт разойдётся. Коммить оба файла.
- **Секрет утёк в git.** `.env` попал в историю, потому что не был в `.gitignore`. Утёкший ключ компрометирован навсегда — его отзывают, а не «удаляют». Всегда `git check-ignore .env` до первого коммита.
- **`.get()` вместо `os.environ["KEY"]` для обязательного секрета.** Тихий `None` уводит ошибку вглубь, вместо громкого падения при старте.
- **Утечка будущего в признаки.** Расчёт RFM по всем ставкам, без фильтра `bet_ts < cutoff`, тащит в фичи события, неизвестные на момент решения. Тест на cutoff ловит это.
- **Правка сырья руками.** `data/raw/` неизменяемо. Любая ручная правка убивает воспроизводимость и контрольную сумму. Все преобразования — кодом, в `interim/processed`.
- **Незафиксированный seed.** Без `SEED=42` каждый запуск даёт другие данные — и спор «у меня 18%, у тебя 12%» не разрешить.
- **Логика заперта в ячейке ноутбука.** Расчёт, который нельзя импортировать и протестировать, нельзя и переиспользовать в проде. Выноси в `src/`.
- **«Работает у меня».** Проект ни разу не собирался из lockfile на чистом окружении. Пока не прошёл `rm -rf .venv && uv sync && uv run pytest`, воспроизводимость не доказана.
