# M1.7 · Git и GitHub: версионирование кода и экспериментов

Git — система контроля версий. Без неё DS-работа неуправляема.

Простыми словами: git — это машина времени и журнал для твоего проекта. Каждый раз, когда ты делаешь осмысленный шаг, ты сохраняешь снимок (commit) с подписью «что и зачем». Можешь вернуться к любому снимку, посмотреть, что менялось, и кто это сделал. GitHub — облачное хранилище этих снимков плюс место для совместной работы.

Формально: git — распределённая система контроля версий, хранящая историю изменений как направленный ациклический граф коммитов. Каждый commit — иммутабельный снимок состояния дерева файлов с хешем-идентификатором, ссылкой на родителя, автором и сообщением.

«Распределённая» — ключевое слово. У каждого разработчика на ноутбуке лежит полная копия всей истории, а не только последняя версия. Поэтому ты можешь смотреть историю, делать коммиты и переключать ветки без интернета — GitHub нужен только чтобы синхронизироваться с другими. Это отличие от старых централизованных систем (SVN), где история жила только на сервере.

### Три зоны: рабочая директория, staging, история

Чтобы понять `add` и `commit`, держи в голове три области, через которые проходит файл:

- **Working directory** — твои файлы на диске прямо сейчас, как ты их редактируешь.
- **Staging area (index)** — «корзина» того, что попадёт в следующий снимок. `git add` кладёт сюда.
- **Repository (история)** — зафиксированные коммиты. `git commit` берёт всё из staging и создаёт иммутабельный снимок.

Зачем промежуточный шаг staging, а не сразу коммит всего подряд? Чтобы собирать **осмысленные** коммиты. Ты поправил три разные вещи в одном файле — функцию фичей, опечатку в комментарии и эксперимент с гиперпараметром. Через `git add -p` (по кускам) можно положить в один коммит только связанные изменения, а остальное оставить на следующий. Один коммит = одна логически завершённая мысль.

### Базовый цикл

```python
# Команды оболочки

git init                          # создать репозиторий (uv init уже делает это)
git add train.py features.py      # пометить файлы для сохранения (staging)
git commit -m "Add churn features: recency, frequency, deposit_sum"
git branch experiment-xgboost     # создать ветку под эксперимент
git switch experiment-xgboost     # переключиться на неё
git push origin experiment-xgboost  # отправить ветку на GitHub
```

Команды, которые ты будешь смотреть каждый день и которых нет в цикле выше:

```python
git status                        # что изменено, что в staging, на какой ветке
git diff                          # что именно поменялось (но ещё не в staging)
git log --oneline --graph         # история коммитов компактно, с графом веток
git switch main                   # вернуться на основную ветку
git merge experiment-xgboost      # влить ветку эксперимента в текущую
git clone <url>                   # скачать чужой/свой репозиторий с GitHub
```

Ключевые понятия:

Commit — сохранённый снимок с сообщением. Хорошее сообщение объясняет «почему», а не «что» (диф и так показывает, что менялось). Плохо: «fix». Хорошо: «Fix LTV leak: exclude deposits after cohort cutoff date».

Branch (ветка) — параллельная линия разработки. Для DS это критично: каждый эксперимент или фича живёт в своей ветке, не ломая рабочую `main`. Хочешь попробовать другую модель — ответвись, поэкспериментируй, и если получилось — влей обратно.

Технически ветка — это просто подвижный указатель на коммит, а не копия файлов. Поэтому создание ветки мгновенно и почти бесплатно: git не копирует проект, он создаёт новую «закладку» в графе коммитов. `HEAD` — указатель на то, где ты сейчас находишься (на какой ветке/коммите).

Pull request (PR) — механизм GitHub: ты предлагаешь влить свою ветку в `main`, коллеги ревьюят диф, обсуждают, и только потом мерж. Это точка контроля качества и распространения знаний в команде.

### Разобранный пример: эксперимент от ветки до мержа

Допустим, на проекте churn-модели у тебя есть рабочая `main` с ROC-AUC 0.78. Ты хочешь попробовать XGBoost, но не уверен, что станет лучше:

```python
git switch main
git switch -c experiment-xgboost     # -c = создать и сразу переключиться
# ... правишь train.py, добавляешь модель ...
git add train.py EXPERIMENTS.md
git commit -m "Try XGBoost: ROC-AUC 0.81 on holdout, +0.03 vs baseline"
git push origin experiment-xgboost
# открываешь PR на GitHub, коллега смотрит диф и проверяет, нет ли leak
```

Если эксперимент удался — PR мержится в `main`, и улучшение становится частью основной модели. Если ROC-AUC упал или нашли утечку — ты просто удаляешь ветку (`git branch -D experiment-xgboost`), и `main` остаётся нетронутой. Главная идея: **провальный эксперимент не стоит ничего** — ты ничего не сломал, потому что работал в изолированной ветке.

### Зачем версионировать заметки об экспериментах

В DS код — это половина истории. Вторая половина — что ты пробовал и что вышло. «Поднял глубину дерева до 12 — ROC-AUC упал, переобучение». Если эти заметки не версионированы рядом с кодом, через месяц ты повторишь ту же ошибку.

Практика: держи в репозитории файл вроде `EXPERIMENTS.md` или лог в структурированном виде, и коммить его вместе с изменениями кода. Тогда конкретная версия модели, конкретный код и конкретный вывод связаны одним коммитом. Это и есть лабораторный журнал исследователя в цифровом виде.

Почему именно «одним коммитом» — это важно: коммит атомарно связывает три вещи, которые иначе разъезжаются. Через полгода ты смотришь `git log`, видишь коммит «XGBoost: ROC-AUC 0.81», и по его хешу можешь восстановить **ровно тот** код и **ровно те** заметки, которые дали этот результат. Без версионирования заметки живут в Notion, код в одной версии, число 0.81 в голове — и связь между ними теряется через неделю.

!!! note "Связь с воспроизводимостью"

    Версионирование заметок — частный случай воспроизводимости из M1.6. Чтобы результат был воспроизводим, нужно зафиксировать не только код и зависимости (`uv.lock`), но и какой именно эксперимент дал какое число. Коммит с кодом + `EXPERIMENTS.md` + зафиксированным `random_state` — это полный «отпечаток» эксперимента.

### Типичные ошибки новичка

- **Гигантский коммит раз в неделю** «весь прогресс за неделю». История становится бесполезной: нельзя откатить одну неудачную идею, не потеряв десять хороших. Коммить часто и мелко.
- **Сообщения `fix`, `update`, `wip`.** Через месяц `git log` превращается в стену из бессмысленных строк. Сообщение пишется для будущего себя.
- **Работа прямо в `main`.** Любая сырая идея сразу в основной ветке. Когда нужно срочно показать рабочую версию — а там полусломанный эксперимент.
- **Коммит секретов и данных.** Попавший в историю файл `.env` с паролем к проду или выгрузка игроков остаются в истории **навсегда**, даже если удалить файл следующим коммитом. Защита — файл `.gitignore`, куда заносят `.env`, `data/`, `*.csv`, `*.pkl`, `__pycache__/`.

```python
# .gitignore — что git должен игнорировать
.env
data/
*.csv
*.pkl
__pycache__/
```

!!! warning "Git — не для данных и не для моделей"

    Git хранит текст и плохо работает с большими бинарными файлами. Не коммить датасеты на гигабайты и обученные модели — репозиторий распухнет необратимо. Для версионирования данных и моделей есть DVC, git-lfs или хранение артефактов в облаке с фиксацией версии/хеша. В git — только код, конфиги и заметки.

    Почему «необратимо»: git хранит **всю историю**. Если ты закоммитил файл на 2 ГБ, а потом удалил его, снимок с этими 2 ГБ всё равно остаётся в истории — каждый, кто клонирует репозиторий, скачивает их заново. Вычистить такое можно только переписыванием истории (`git filter-repo`), что ломает копии у всех коллег.

!!! question "Проверь себя"

    1. Что хорошее commit-сообщение должно объяснять — «что» или «почему» — и почему именно так?
    2. Зачем в DS каждый эксперимент держать в отдельной ветке, а не коммитить всё в `main`?
    3. Почему датасеты на гигабайты и обученные модели нельзя коммитить в git, и чем их версионировать?

??? success "Ответы"

    1. «Почему»: диф и так показывает, что менялось, а мотивация изменения из кода не видна. Плохо: «fix». Хорошо: «Fix LTV leak: exclude deposits after cohort cutoff date».
    2. Ветка — параллельная линия разработки; эксперимент не ломает рабочую `main`. Получилось — мержишь через pull request с ревью; нет — выбрасываешь ветку.
    3. Git хранит текст и плохо работает с большими бинарниками — репозиторий распухает необратимо. Данные и модели версионируют через DVC, git-lfs или облако с фиксацией версии/хеша; в git — только код, конфиги и заметки.
