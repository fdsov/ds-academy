# M1.5 · Виртуальные окружения: зачем

Это, возможно, самая важная привычка во всём модуле. Без неё ничего не работает.

Простыми словами: представь, что для проекта А тебе нужна библиотека pandas версии 1.5, а для проекта Б — версии 2.2, и они несовместимы. Если ставить всё в одно общее место, проекты передерутся за версии, и один сломает другой. Виртуальное окружение — это отдельная коробка с библиотеками для каждого проекта. Что лежит в коробке проекта А, не влияет на проект Б.

Формально: виртуальное окружение (virtual environment) — изолированная директория с собственным интерпретатором Python и собственным набором установленных пакетов, не пересекающаяся с системным Python и с другими окружениями. Технически это поддиректория (обычно `.venv/`) со своим `bin/` (или `Scripts/` на Windows) и `site-packages/`. Активация окружения переключает, какой `python` и `pip` использует твоя оболочка.

Железное правило: один проект — одно окружение. Никогда не ставь пакеты в системный Python через `sudo pip install` — это прямой путь к нерабочей ОС и неразрешимым конфликтам версий.

## Почему общий Python ломается: интуиция на пальцах

Представь полку с инструментами, которой пользуются все жильцы дома. Один взял отвёртку и заточил её под свой шуруп, другой пришёл — а она уже не подходит. В Python ровно так же: `site-packages` — это общая полка. Когда два проекта ставят туда несовместимые версии одной библиотеки, последний установивший «затачивает» полку под себя, а остальные молча ломаются.

Ключевая проблема в том, что в `site-packages` физически **не может лежать две версии одного пакета одновременно**. `pip install pandas==2.2` поверх `pandas==1.5` просто сносит первую. Поэтому без изоляции даже не нужно злого умысла — достаточно работать над двумя проектами, и рано или поздно один сломает другой. Виртуальное окружение даёт каждому проекту свою отдельную полку.

## Аналогия из гемблинг-аналитики

Допустим, ты ведёшь два проекта. Первый — legacy-дашборд по retention, который писали два года назад на `pandas==1.3` и `numpy==1.21`; код использует старое поведение `df.append()`, которое в новых версиях удалили. Второй — свежая churn-модель на `pandas==2.2` и `scikit-learn==1.5`, которой старый numpy просто не хватает по API.

Без окружений это безвыходная ситуация: обновишь pandas ради модели — рухнет дашборд по retention, который смотрит руководство каждое утро. Откатишь ради дашборда — не соберётся модель. С окружениями оба живут параллельно: `~/retention-dash/.venv` с pandas 1.3 и `~/churn-model/.venv` с pandas 2.2, и они ничего не знают друг о друге. Это не теоретическая аккуратность, а возможность вообще не выбирать между «сломать прод» и «не сделать новую задачу».

## Что физически создаётся на диске

Когда ты выполняешь `python -m venv .venv`, появляется примерно такая структура:

```text
.venv/
├── bin/                 # (Scripts/ на Windows)
│   ├── python           # симлинк/копия интерпретатора
│   ├── pip
│   └── activate         # скрипт активации
├── lib/
│   └── python3.12/
│       └── site-packages/   # сюда ставятся пакеты ЭТОГО проекта
└── pyvenv.cfg           # конфиг: на какой базовый Python смотрит окружение
```

Активация (`source .venv/bin/activate`) не делает магии — она всего лишь дописывает `.venv/bin` в начало переменной `PATH`. После этого команда `python` находит сначала интерпретатор из окружения, а не системный. Проверить, что окружение активно, можно так:

```bash
source .venv/bin/activate
which python          # должно показать .../проект/.venv/bin/python
python -m pip install pandas
deactivate            # выйти из окружения, PATH вернётся как был
```

Привычка использовать `python -m pip install` вместо просто `pip install` страхует от ситуации, когда в PATH «не тот» pip: `python -m pip` всегда ставит пакет именно в то окружение, чей `python` сейчас активен.

## Численная иллюстрация: чем это окупается

Допустим, у тебя 4 активных проекта и каждый зависит от своего набора версий. Без окружений конфликт версий гарантирован — это не вопрос «если», а «когда». Прикинем стоимость одного такого инцидента: сломанный прод-дашборд чинится в среднем 3 часа, а аналитик стоит условные $40/час. Один инцидент = $120 прямых потерь плюс простой тех, кто ждал отчёт. Команда из 5 человек, по одному такому инциденту в месяц на каждого, даёт $600/мес выброшенных на проблему, которой просто не существует при правильной изоляции. `python -m venv` создаётся за секунды и стоит ноль — это один из самых выгодных по соотношению усилие/эффект приёмов во всей разработке.

## Пограничные случаи и типичные ошибки

- **`sudo pip install` в системный Python.** На macOS и Linux от системного Python зависит сама ОС и её утилиты. Перезапись его пакетов через `sudo` способна сломать загрузку или системные скрипты. Никогда так не делай — это и есть тот самый антипаттерн.
- **Глобальная установка без `sudo`, в user-site (`pip install --user`).** Лучше, чем `sudo`, но всё равно общая полка для всех проектов — конфликты вернутся. Это не замена окружению.
- **Закоммитить `.venv/` в git.** Окружение тяжёлое, привязано к ОС и абсолютным путям, его нельзя переносить между машинами. В репозиторий идёт не папка `.venv/`, а список зависимостей (`requirements.txt` или `pyproject.toml`), из которого окружение пересобирается. `.venv/` добавляют в `.gitignore`.
- **Забыл активировать окружение.** Классика: ставишь пакет, а он улетает в системный Python; потом «почему-то не импортируется». Признак — `which python` показывает `/usr/bin/python`, а не путь внутри проекта. Сначала активируй, потом ставь.
- **Перенос папки проекта вместе с `.venv`.** Из-за абсолютных путей в `pyvenv.cfg` и скриптах активации перемещённое окружение часто ломается. Правильный путь — удалить `.venv` и пересоздать из файла зависимостей.

!!! note "venv, virtualenv, conda, uv — это всё про одно"

    Инструментов несколько, но идея одна — изолированная коробка пакетов. `venv` встроен в Python и его достаточно для большинства задач. `uv` — современный быстрый менеджер, который делает то же самое в разы быстрее. `conda` популярен в data science, потому что умеет ставить ещё и не-Python зависимости. Какой бы ты ни выбрал — принцип «один проект, одно окружение» не меняется.
