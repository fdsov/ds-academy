# M1.13 · Практика

### Быстрый квиз

Выбери вариант — ответ подсветится сразу.

```quiz
Что означает воспроизводимость (reproducibility) в её минимальном обязательном уровне?
[ ] Другой код или другие данные подтверждают тот же вывод
[x] Тот же код на тех же данных с теми же версиями зависимостей даёт тот же результат
[ ] Модель показывает высокую метрику ROC-AUC при повторном обучении
[ ] Ноутбук открывается на любой машине без ошибок импорта
> Reproducibility — это тот же код плюс те же данные плюс зафиксированные версии дают идентичный результат; подтверждение вывода другим кодом или данными — это уже replicability.
---
Какой слой воспроизводимости закрывает lockfile (`uv.lock`)?
[ ] Среду исполнения (версию Python и ОС)
[x] Зависимости — точные версии всех библиотек и их транзитивного дерева
[ ] Случайность — фиксацию random seed
[ ] Версионирование кода
> Lockfile фиксирует полностью разрешённое дерево зависимостей с точными версиями и хешами; код закрывает git, случайность — seed, среду — виртуальное окружение.
---
В чём суть проблемы скрытого состояния (hidden state) в Jupyter?
[ ] Ноутбук хранится в формате JSON, и git не видит человеческих дифов
[ ] Ядро слишком медленно выполняет тяжёлые вычисления
[x] Ядро держит переменные от всех когда-либо выполненных ячеек, независимо от текущего кода и порядка
[ ] Ячейки нельзя запускать повторно без перезапуска ядра
> Ячейки можно гонять в любом порядке, и переменная живёт в памяти ядра даже после удаления её кода — ноутбук «работает» только в живой сессии и нигде больше.
---
Почему `uv add polars` лучше, чем просто установка пакета?
[ ] Он ставит пакет в системный Python, чтобы его видели все проекты
[x] Он записывает пакет в `pyproject.toml` и обновляет lockfile с точными версиями
[ ] Он автоматически коммитит изменения в git
[ ] Он скачивает датасеты, нужные для polars
> `uv add` не просто ставит пакет, а декларирует его в `pyproject.toml` и фиксирует разрешённое дерево версий в `uv.lock` — это и даёт воспроизводимость зависимостей.
---
Что из перечисленного НЕ нужно коммитить в git?
[ ] `pyproject.toml`
[ ] `uv.lock`
[x] `.venv/` и файл `.env` с секретами
[ ] `README.md`
> `.venv/` тяжёлая и пересоздаётся из lockfile командой `uv sync`, а `.env` содержит секреты; в git идут `pyproject.toml`, `uv.lock`, код и заметки.
---
Что делать, если API-ключ случайно утёк в историю git?
[ ] Удалить ключ из файла и сделать новый коммит — этого достаточно
[ ] Сделать репозиторий приватным
[x] Немедленно отозвать (revoke) и перевыпустить ключ на стороне сервиса
[ ] Просто добавить `.env` в `.gitignore`
> Ключ остаётся в истории git и доступен любому, кто склонирует репозиторий, поэтому единственный надёжный шаг — отозвать и перевыпустить его; чистка истории вторична.
---
Почему за версиями Python для проектов отвечает `uv`, а не системный Python?
[ ] Системный Python работает медленнее проектного
[x] От системного Python зависит сама ОС, и его поломка ломает систему
[ ] Системный Python не поддерживает виртуальные окружения
[ ] uv умеет работать только со своей сборкой Python
> Системный Python трогать нельзя — от него зависит ОС; uv изолированно скачивает и ставит любую нужную версию под проект, не задевая систему.
```

Разминка:

1. Установи `uv`. Создай проект `uv init --python 3.14 sandbox`, добавь `polars`, посмотри, что появилось в `pyproject.toml` и `uv.lock`. Объясни словами разницу между этими двумя файлами.
2. Инициализируй git, сделай три осмысленных коммита, создай ветку `feature-x`, переключись на неё и обратно. Посмотри `git log --oneline --graph`.

Основное:

3. Развернись по структуре DS-проекта (data/{raw,interim,processed}, src/, notebooks/, tests/). Настрой `.gitignore` так, чтобы `.env`, `.venv/` и `data/` не попадали в git. Проверь через `git status`, что они действительно игнорируются.
4. Возьми любой публичный CSV (например, синтетические транзакции/ставки). Положи его в `data/raw/` как read-only. В marimo-ноутбуке посчитай простую метрику (средний депозит, конверсия). Затем вынеси расчёт признаков в `src/features.py` как функцию и импортируй её в ноутбук.
5. Напиши один pytest-тест к своей функции из src/. Запусти `uv run pytest` и `uvx ruff check .`.

Челлендж:

6. Возьми старый «грязный» Jupyter-ноутбук (свой или чужой с GitHub). Сделай Restart Kernel and Run All. Если упал — почини так, чтобы он исполнялся сверху вниз с нуля. Опиши в `EXPERIMENTS.md`, что было сломано (это и есть проблема скрытого состояния вживую).

Мини-проект:

7. Собери репозиторий `churn-starter`: воспроизводимое окружение (uv + lockfile), чистая структура, один marimo-ноутбук с разведкой данных, логика в src/, минимум один тест, README с тремя командами запуска (clone, `uv sync`, `uv run ...`), `.env.example` без секретов. Запушь на GitHub. Критерий готовности: ты удаляешь `.venv/`, делаешь `uv sync`, `uv run pytest` — и всё зелёное. Дай ссылку коллеге — у него должно завестись с тех же трёх команд.
