# M1.10 · Структура DS-проекта

Хаотичная папка с десятком `final_v2_REAL_final.ipynb` — признак непрофессионала. Стандартная структура экономит часы и делает проект понятным с первого взгляда.

Простыми словами: у всего своё место. Сырые данные отдельно, обработанные отдельно, код отдельно, ноутбуки-черновики отдельно. Любой человек (включая тебя через полгода) сразу понимает, где что искать.

Базовая структура, восходящая к шаблону cookiecutter-data-science:

```python
churn-model/
├── data/
│   ├── raw/            # сырые данные, неизменяемые, read-only
│   ├── interim/        # промежуточные результаты обработки
│   └── processed/      # финальные датасеты для моделей
├── notebooks/          # исследовательские ноутбуки (черновики)
├── src/                # переиспользуемый код: модули, функции
│   ├── features.py     # построение признаков
│   ├── train.py        # обучение модели
│   └── evaluate.py     # оценка
├── models/             # сохранённые обученные модели (в git не коммитим)
├── reports/            # отчёты, графики, результаты
├── tests/              # тесты кода из src
├── pyproject.toml      # зависимости и конфиг проекта
├── uv.lock             # зафиксированные версии
├── .env                # секреты — НИКОГДА не в git
├── .gitignore          # что git должен игнорировать
└── README.md           # что это, как запустить
```

Формально: cookiecutter-data-science — это шаблонизатор, генерирующий стандартизованную структуру DS-проекта одной командой. Ключевой принцип — data/raw неизменяем и read-only: сырые данные никогда не правятся на месте, любая обработка порождает новый файл в interim/ или processed/. Это гарантирует, что ты всегда можешь воспроизвести весь пайплайн с исходной точки.

## Почему data-слои — это конвейер, а не просто папки

На пальцах: `data/raw → data/interim → data/processed` — это три стадии одной фабрики, а не три способа хранить «примерно одно и то же».

- `raw/` — то, что пришло из источника байт-в-байт. Выгрузка из БД, CSV от партнёра, дамп API. Эти файлы священны: их не пересохраняют, не чистят «по-быстрому» в Excel, не дописывают столбец руками. Если raw изменили — воспроизводимость потеряна, и никто уже не докажет, какие именно данные дали ту самую модель.
- `interim/` — полуфабрикат. Сджойнили три таблицы, привели типы, выкинули мусорные строки — но это ещё не финал. Сюда складывают то, что дорого пересчитывать каждый раз, но что само по себе модель не кормит.
- `processed/` — готовый к обучению датасет: уже с признаками, в нужном формате (часто `.parquet`), детерминированно собранный скриптом из raw.

Гемблинг-пример. Из прод-БД выгрузили `data/raw/players_2026_q1.csv` — депозиты, ставки, логины по игрокам. Скрипт `src/features.py` читает этот файл, считает на игрока `deposits_30d`, `bets_30d`, `days_since_last_login`, метку `churned` (не заходил 30 дней) и пишет `data/processed/churn_train.parquet`. Если завтра аналитик усомнится в цифре оттока — он перезапускает скрипт на том же неизменном raw и получает байт-в-байт тот же результат. Именно это называется воспроизводимостью.

Типичная ошибка новичка: открыть `raw/players.csv`, руками в редакторе удалить «странные» строки, сохранить под тем же именем. Теперь исходник испорчен, а пайплайн врёт. Правильно — фильтрация живёт в коде, результат уходит в `interim/` или `processed/`, raw остаётся нетронутым.

## Граница ноутбук / src — это граница зрелости

Главное архитектурное правило: код, который повторяется или пойдёт в прод, переезжает из ноутбука в `src/` как функции, и ноутбук их импортирует. Ноутбук — для исследования и демонстрации, `src/` — для логики. Это и есть граница между «поиграл с данными» и «построил систему».

Почему это важно технически: ноутбук исполняется ячейками в произвольном порядке, хранит скрытое состояние и почти не тестируется. Функция в `src/` — детерминирована, импортируется, покрывается тестом в `tests/`, переиспользуется и в обучении, и в проде. Когда логика расчёта оттока лежит в одной функции `make_features()`, а не размазана по 40 ячейкам, её можно вызвать одинаково и в ноутбуке-черновике, и в `train.py`, и в ночном джобе.

Было (вся логика в ноутбуке, скопирована в двух местах):

```python
# notebook.ipynb, ячейка 17
df["churned"] = (df["days_since_last_login"] > 30).astype(int)
df["deposits_30d"] = df.groupby("player_id")["deposit"].transform("sum")
# ... та же логика потом скопирована в train-скрипт — два источника правды
```

Стало (логика в `src/`, ноутбук только импортирует):

```python
# src/features.py
def make_features(df):
    df["churned"] = (df["days_since_last_login"] > 30).astype(int)
    df["deposits_30d"] = df.groupby("player_id")["deposit"].transform("sum")
    return df
```

```python
# notebook.ipynb
from src.features import make_features
df = make_features(load_raw())
```

Теперь определение оттока живёт в одном месте. Поменяли порог с 30 на 14 дней — правка в одной функции автоматически уезжает и в ноутбук, и в прод. Это и есть «один источник правды».

## Что осознанно не коммитят в git

В `.gitignore` отправляют то, что либо опасно, либо тяжело, либо легко пересоздаётся:

- `.env` — секреты (пароли к БД, API-ключи). Утёкший в публичный репозиторий ключ к платёжному API — это прямые деньги и инцидент безопасности.
- `.venv/` — виртуальное окружение, восстанавливается из `uv.lock` одной командой.
- `data/` — данные. Часто гигабайты, плюс могут содержать персональные данные игроков (GDPR/152-ФЗ). Версионируют отдельно (DVC, S3), а не в git.
- `models/` — обученные артефакты, тоже тяжёлые и пересоздаваемые из кода + данных.

При этом `pyproject.toml` и `uv.lock` коммитят обязательно — по ним любой воспроизведёт окружение. Тонкая грань: иногда в git кладут пустую папку с `.gitkeep` (например `data/raw/.gitkeep`), чтобы структура каталогов существовала у того, кто клонирует репозиторий, хотя сами данные игнорируются.

??? success "Ответы"

    1. Чтобы всегда можно было воспроизвести весь пайплайн с исходной точки. Любая обработка порождает новый файл в `interim/` или `processed/`, сырьё не правится на месте.
    2. Код, который повторяется или пойдёт в прод, переезжает из ноутбука в `src/` как функции, а ноутбук их импортирует. Ноутбук — для исследования и демонстрации, `src/` — для логики; наличие этой границы и есть признак зрелости.
    3. `.env` (секреты), `.venv/` (окружение), `data/` и `models/` (обученные модели) — они в `.gitignore`.
