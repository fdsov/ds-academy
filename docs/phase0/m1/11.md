# M1.11 · Секреты и .env

Это вопрос безопасности, на котором горят регулярно и больно.

Простыми словами: пароли от базы данных, API-ключи, токены — это секреты. Их нельзя писать прямо в коде, потому что код уходит в git, а оттуда — на GitHub, где их могут увидеть. Секреты кладут в отдельный файл `.env`, который git'у запрещено трогать.

Формально: `.env` — текстовый файл вида `KEY=value`, хранящий конфигурацию и секреты вне исходного кода. Он обязательно вносится в `.gitignore`. В репозиторий кладут только `.env.example` — шаблон без значений, показывающий, какие переменные нужны. Принцип разделения кода и конфигурации восходит к методологии Twelve-Factor App: конфигурация живёт в окружении, не в коде.

## На пальцах: почему именно окружение, а не код

Один и тот же код должен запускаться в трёх местах: на твоём ноутбуке (локальная база), на staging-сервере (тестовая база) и в production (боевая база с реальными деньгами игроков). Если адрес базы и пароль зашиты в код — придётся держать три версии кода или править файл перед каждым деплоем. Это путь к катастрофе: рано или поздно кто-то выкатит в production конфиг от локалки.

Решение: код один и тот же везде, а различается только окружение. Локально читается локальный `.env`, на сервере те же переменные приходят из настроек хостинга (Railway, Docker secrets, CI/CD variables). Код вообще не знает, откуда взялось значение — он просто берёт `os.environ["DB_PASSWORD"]`.

```python
# .env (в .gitignore, не коммитится)
# DB_PASSWORD=super_secret_123
# YOHOHO_API_KEY=sk-...

# Чтение в коде через python-dotenv
import os
from dotenv import load_dotenv

load_dotenv()
db_password = os.environ["DB_PASSWORD"]  # упадёт, если переменной нет — это хорошо
```

### Почему `os.environ[...]`, а не `os.getenv(...)`

Две формы чтения переменной ведут себя по-разному при отсутствии ключа, и это важно:

```python
import os

# Вариант 1: жёсткий — падает с KeyError, если переменной нет
db_password = os.environ["DB_PASSWORD"]

# Вариант 2: мягкий — возвращает None (или дефолт), молча
db_password = os.getenv("DB_PASSWORD")          # None, если нет
log_level = os.getenv("LOG_LEVEL", "INFO")      # дефолт "INFO" — это ок для НЕсекретов
```

Для секретов выбирают жёсткий вариант. Логика «fail fast»: лучше, чтобы сервис не стартовал вообще и громко упал на деплое, чем чтобы он поднялся с `db_password = None`, попытался подключиться к базе и выдал невнятную ошибку где-то в середине обработки запроса. `os.getenv` с дефолтом уместен только для безопасной конфигурации (уровень логов, размер пула), но не для паролей.

### Разбор: как `.env` превращается в переменные

`load_dotenv()` читает файл `.env` из текущей директории построчно, парсит `KEY=value` и подкладывает значения в `os.environ` — ровно туда, куда хостинг кладёт настоящие production-переменные. Поэтому код, читающий `os.environ`, работает одинаково и локально, и в проде. Несколько практических деталей:

- Значения всегда строки. `os.environ["PORT"]` вернёт `"5432"`, а не число — приводи тип явно: `int(os.environ["PORT"])`.
- `load_dotenv()` по умолчанию не перезаписывает уже существующие переменные окружения. То есть реальная переменная среды на сервере имеет приоритет над `.env` — это правильно: прод не должен случайно подхватить локальный файл.
- Пробелы и кавычки: `KEY = value` и `KEY="value"` парсятся, но лучше писать строго `KEY=value` без пробелов вокруг `=`.

## .env.example — контракт проекта

`.env.example` коммитится в репозиторий и служит документацией: какие переменные вообще нужны проекту, чтобы он запустился. Значения там — пустые или заведомо фейковые.

```bash
# .env.example (КОММИТИТСЯ в репозиторий)
DB_HOST=
DB_PORT=5432
DB_PASSWORD=
YOHOHO_API_KEY=
```

Новый разработчик (или ты сам через полгода) делает `cp .env.example .env`, заполняет значения — и проект готов. Без `.env.example` пришлось бы выяснять список переменных, ловя `KeyError` по очереди. Заодно это место для значений по умолчанию, которые не секретны (`DB_PORT=5432`).

!!! warning "Если секрет утёк в git — он скомпрометирован навсегда"

    Просто удалить ключ из файла и закоммитить недостаточно: он остаётся в истории git и доступен любому, кто склонирует репозиторий. Единственный правильный шаг — немедленно отозвать (revoke) и перевыпустить ключ на стороне сервиса. Чистка истории git (filter-repo) — вторична. Поэтому проще не допускать утечки: `.gitignore` настраивается до первого коммита.

### Почему «удалить и закоммитить» не спасает — на примере

Представь цепочку коммитов:

```
C1 — добавил config.py с YOHOHO_API_KEY="sk-real-123"
C2 — "удалил" ключ из config.py, закоммитил
```

В рабочей копии (HEAD) ключа нет. Но `git show C1:config.py` или `git checkout C1` мгновенно достаёт его обратно — история неизменна. Любой, кто склонировал репозиторий хоть раз, уже имеет ключ локально. Боты сканируют новые коммиты на GitHub за секунды и находят паттерны вроде `sk-...`. Поэтому единственная надёжная реакция на утечку — считать ключ скомпрометированным и отозвать его на стороне сервиса. Это особенно дорого в гемблинге: утёкший ключ к платёжному провайдеру или к боевой базе — это прямой денежный риск, а не абстрактная «уязвимость».

### Типичные ошибки

- Закоммитить `.env` до того, как добавил его в `.gitignore`. `.gitignore` не действует на уже отслеживаемые файлы — нужно `git rm --cached .env`, и всё равно ключ останется в истории. Настраивай `.gitignore` первым.
- Положить реальные значения в `.env.example`. Это тот же самый коммит секрета, только под другим именем файла.
- Печатать секреты в логи: `print(f"connecting with {db_password}")`. Логи тоже утекают — в Sentry, в облачный лог-агрегатор, в чужие глаза.
- Хранить один `.env` на всю команду в общем чате/диске. Секреты раздают через секрет-менеджер (Vault, Doppler, настройки хостинга), а не пересылают в Telegram.
