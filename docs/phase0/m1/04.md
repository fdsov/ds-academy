# M1.4 · Python 3.13 / 3.14

Python — основной язык DS, и версия имеет значение. На июнь 2026 актуальный рабочий выбор — Python 3.13 или 3.14 (3.14 вышел 7 октября 2025 и к лету 2026 уже зрелый). Эти релизы принесли две вещи, важные именно для нас.

Простыми словами: начиная с 3.13 Python научился (тогда в экспериментальном режиме) работать без «глобального замка», который раньше мешал по-настоящему параллельно использовать несколько ядер процессора в одном процессе, а в 3.14 эта сборка стала официально поддерживаемой. Плюс 3.13 принёс быстрее работающий интерактивный интерпретатор с подсветкой.

Формально: речь о GIL (Global Interpreter Lock) — мьютексе, который исторически разрешал исполнять байткод Python только одному потоку одновременно. Python 3.13 ввёл экспериментальную сборку free-threaded (`--disable-gil`, PEP 703), где GIL можно отключить, и режим экспериментального JIT-компилятора (PEP 744). В Python 3.14 эта сборка (`python3.14t`, PEP 779) перешла из экспериментальной в официально поддерживаемую — но по-прежнему не является дефолтной. Для DS это пока не «включил и стало в 4 раза быстрее» — большинство тяжёлых вычислений и так живут в NumPy/Polars, где работа идёт в нативном коде вне GIL. Но направление задано, и через пару лет free-threaded станет дефолтом.

!!! warning "Не бери самую свежую версию вслепую"

    Только что вышедший Python (свежий мажорный релиз в первые месяцы) часто опережает экосистему: бинарные колёса (wheels) для NumPy, PyTorch, scientific-стека под него ещё не собраны, и установка ломается или собирается из исходников часами. Практическое правило: бери предпоследний стабильный релиз. На июнь 2026 безопасный дефолт — 3.13, осознанный выбор для нового проекта — 3.14 (вышел 7 октября 2025 и к лету уже зрелый).

Важно: системный Python (тот, что идёт с macOS или Linux) трогать нельзя — от него зависит сама ОС. Версиями Python для проектов управляет `uv` (ниже), который умеет скачивать и ставить любую нужную версию изолированно.

!!! question "Проверь себя"

    1. Что такое GIL и почему отключение его во free-threaded-сборке пока слабо влияет на типичные DS-вычисления?
    2. Почему не стоит сразу ставить только что вышедшую версию Python в первые месяцы после релиза?
    3. Почему системный Python нельзя трогать, и кто управляет версиями Python для проектов?

??? success "Ответы"

    1. GIL (Global Interpreter Lock) — мьютекс, разрешавший исполнять байткод Python только одному потоку. Эффект отключения мал, потому что тяжёлые вычисления и так живут в NumPy/Polars, работающих в нативном коде вне GIL.
    2. Под свежий релиз ещё не собраны бинарные wheels для NumPy, PyTorch и scientific-стека — установка ломается или собирается из исходников часами. Правило: брать предпоследний стабильный релиз.
    3. От системного Python зависит сама ОС, его поломка ломает систему. Версиями Python для проектов управляет uv, скачивая и ставя нужную версию изолированно.
