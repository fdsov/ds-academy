# M1.4 · Python 3.13 / 3.14

Python — основной язык DS, и версия имеет значение. На июнь 2026 актуальный рабочий выбор — Python 3.13 или 3.14 (3.14 вышел 7 октября 2025 и к лету 2026 уже зрелый). Эти релизы принесли две вещи, важные именно для нас.

Простыми словами: начиная с 3.13 Python научился (тогда в экспериментальном режиме) работать без «глобального замка», который раньше мешал по-настоящему параллельно использовать несколько ядер процессора в одном процессе, а в 3.14 эта сборка стала официально поддерживаемой. Плюс 3.13 принёс быстрее работающий интерактивный интерпретатор с подсветкой.

Формально: речь о GIL (Global Interpreter Lock) — мьютексе, который исторически разрешал исполнять байткод Python только одному потоку одновременно. Python 3.13 ввёл экспериментальную сборку free-threaded (`--disable-gil`, PEP 703), где GIL можно отключить, и режим экспериментального JIT-компилятора (PEP 744). В Python 3.14 эта сборка (`python3.14t`, PEP 779) перешла из экспериментальной в официально поддерживаемую — но по-прежнему не является дефолтной. Для DS это пока не «включил и стало в 4 раза быстрее» — большинство тяжёлых вычислений и так живут в NumPy/Polars, где работа идёт в нативном коде вне GIL. Но направление задано, и через пару лет free-threaded станет дефолтом.

## Что такое GIL «на пальцах»

Представь кассу в супермаркете с одним кассиром (интерпретатор) и несколькими очередями (потоки). GIL — это правило «в любой момент кассир обслуживает только одну очередь». Даже если у тебя 8 ядер процессора (8 свободных касс), классический CPython держит ровно один открытый замок: потоки переключаются по очереди, но реально считают по одному. Поэтому CPU-bound код на чистом Python (циклы, арифметика в Python-объектах) от добавления потоков не ускоряется — он лишь делит то же одно ядро.

Зачем тогда GIL вообще нужен? Он сильно упрощает управление памятью CPython: счётчики ссылок (reference counting) меняются без гонок, не нужны блокировки на каждый объект. Цена — невозможность настоящего CPU-параллелизма внутри процесса. Поэтому до 3.13 единственный способ занять все ядра — запускать несколько процессов (`multiprocessing`), что дорого: каждый процесс — своя копия памяти, данные между ними надо сериализовать (pickle).

## Почему DS это пока почти не чувствует

Ключевая интуиция: GIL держит исполнение **байткода Python**, но отпускается, когда управление уходит в нативный код на C/Rust. NumPy, Polars, PyTorch, scikit-learn делают тяжёлую арифметику именно там — на время `arr @ arr2` или `df.group_by(...).agg(...)` GIL освобождён, и под капотом эти библиотеки уже используют все ядра (BLAS, OpenMP, Rayon в Polars). То есть DS-нагрузка уже параллельна — но не средствами потоков Python, а средствами библиотек.

Где free-threaded реально поможет: «склейка» из множества мелких Python-операций, которые нельзя векторизовать, — параллельный парсинг тысяч JSON, обход дерева, кастомные feature-функции на чистом Python поверх множества строк. Там, где сейчас спасает только `multiprocessing` с дорогой сериализацией, free-threaded даст потоки с общей памятью.

Маленький пример, где GIL виден. CPU-bound функция на чистом Python в потоках на классической сборке почти не ускоряется:

```python
import time
from concurrent.futures import ThreadPoolExecutor

def burn(n: int) -> int:
    total = 0
    for i in range(n):
        total += i * i
    return total

N = 20_000_000
t0 = time.perf_counter()
with ThreadPoolExecutor(max_workers=4) as ex:
    list(ex.map(burn, [N, N, N, N]))
print(f"{time.perf_counter() - t0:.2f} c")
# Классический GIL: ~ как при последовательном запуске (потоки делят одно ядро).
# Сборка python3.14t (free-threaded): время падает примерно в число ядер раз.
```

Проверить, в какой сборке ты находишься, можно так:

```python
import sys
# True только во free-threaded сборке (python3.13t / python3.14t)
print(getattr(sys, "_is_gil_enabled", lambda: True)())
print(sys.version)  # увидишь 'free-threading build', если это t-сборка
```

!!! note "Free-threaded — это отдельная сборка, не флаг по умолчанию"

    Обычный `python3.14` остаётся с GIL. Сборка без GIL ставится отдельно (`uv python install 3.14t`) и имеет суффикс `t`. Не всякий C-расширение умеет в неё загружаться: пакет должен явно объявить совместимость, иначе при импорте GIL автоматически включится обратно. Поэтому на середину 2026 free-threaded — режим для экспериментов и узких CPU-bound задач, а не дефолт для рабочих проектов.

## JIT — что это и почему «пока тихо»

PEP 744 (3.13) добавил экспериментальный JIT-компилятор: интерпретатор может на горячих участках кода сгенерировать машинный код вместо пошагового исполнения байткода. Это путь к ускорению чистого Python, но на 2026 выигрыш скромный (единицы процентов на типичных задачах) и JIT по умолчанию выключен. Для DS он, как и free-threaded, второстепенен ровно по той же причине — горячая математика уже в нативных библиотеках.

!!! warning "Не бери самую свежую версию вслепую"

    Только что вышедший Python (свежий мажорный релиз в первые месяцы) часто опережает экосистему: бинарные колёса (wheels) для NumPy, PyTorch, scientific-стека под него ещё не собраны, и установка ломается или собирается из исходников часами. Практическое правило: бери предпоследний стабильный релиз. На июнь 2026 безопасный дефолт — 3.13, осознанный выбор для нового проекта — 3.14 (вышел 7 октября 2025 и к лету уже зрелый).

Почему это бьёт именно по DS-стеку. Когда `pip`/`uv` не находит готового бинарного wheel под твою версию и платформу, он скачивает исходники и собирает их локально — для NumPy и особенно PyTorch это требует компилятора, системных библиотек (BLAS, LAPACK) и десятков минут, а часто просто падает с невнятной ошибкой компиляции. Wheel — это уже скомпилированный пакет под конкретную связку «версия Python + ОС + архитектура», поэтому экосистеме нужны недели-месяцы, чтобы выпустить колёса под новый мажорный релиз.

Аналогия из продуктовой/гемблинг-аналитики: это как катить новую версию платёжного провайдера в день её релиза прямо в прод на пятничном пике трафика. Технически «новее = лучше», практически — ловишь несобранные зависимости и чужие баги первым. Предпоследний стабильный релиз — это «обкатанная версия, на которой уже набили шишки другие».

Важно: системный Python (тот, что идёт с macOS или Linux) трогать нельзя — от него зависит сама ОС. Версиями Python для проектов управляет `uv` (ниже), который умеет скачивать и ставить любую нужную версию изолированно.

!!! danger "Типичные ошибки новичка"

    - `sudo pip install ...` в системный Python — рано или поздно ломает утилиты ОС, завязанные на него. Никогда не ставь пакеты в системный интерпретатор.
    - Поставить `3.14t` (free-threaded) «потому что новее» и удивляться, что половина пакетов не импортируется или GIL всё равно включается обратно. Для рабочих проектов бери обычную сборку.
    - Ждать ускорения от free-threaded на коде, который и так упирается в NumPy/Polars — там выигрыша не будет, узкое место не в GIL.
    - Ставить самый свежий мажор в первый месяц и часами собирать PyTorch из исходников вместо готового wheel.

!!! question "Проверь себя"

    1. Что такое GIL и почему отключение его во free-threaded-сборке пока слабо влияет на типичные DS-вычисления?
    2. Почему не стоит сразу ставить только что вышедшую версию Python в первые месяцы после релиза?
    3. Почему системный Python нельзя трогать, и кто управляет версиями Python для проектов?

??? success "Ответы"

    1. GIL (Global Interpreter Lock) — мьютекс, разрешавший исполнять байткод Python только одному потоку. Эффект отключения мал, потому что тяжёлые вычисления и так живут в NumPy/Polars, работающих в нативном коде вне GIL.
    2. Под свежий релиз ещё не собраны бинарные wheels для NumPy, PyTorch и scientific-стека — установка ломается или собирается из исходников часами. Правило: брать предпоследний стабильный релиз.
    3. От системного Python зависит сама ОС, его поломка ломает систему. Версиями Python для проектов управляет uv, скачивая и ставя нужную версию изолированно.
