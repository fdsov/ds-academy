# M1.3 · Воспроизводимость как принцип

Прежде чем трогать инструменты, нужно понять цель, ради которой они существуют. Все инструменты этого модуля решают одну задачу — сделать твою работу воспроизводимой. Если держать эту цель в голове, выбор инструмента становится очевидным.

## Зачем это вообще нужно

Воспроизводимость — это не академический ритуал и не «правильно по учебнику». Это страховка от вполне денежных провалов. Представь типичную сцену в продуктовой аналитике гемблинга: ты три недели назад показал руководству, что у когорты игроков из определённого гео LTV вырос на 18% после изменения бонусной механики. На основе твоего отчёта команда выкатила механику на все гео и заложила в прогноз дополнительные $120k GGR в месяц. Через месяц цифры не сошлись, тебя просят «пересчитать как тогда» — а ты не можешь. Тот ноутбук уже переписан, датасет в проде обновился, версия библиотеки сменилась, seed нигде не зафиксирован. Цифра 18% превратилась в недоказуемую легенду, а решение на $120k было принято на песке.

Невоспроизводимый анализ — это анализ, которому нельзя доверять, потому что его нельзя проверить. А в среде, где на основе твоих цифр двигают реальные деньги, недоказуемая цифра хуже, чем отсутствие цифры: отсутствие заставляет собрать данные, а ложная уверенность ведёт к дорогой ошибке.

## Слои воспроизводимости

Воспроизводимость распадается на слои, и каждый слой закрывается своим инструментом:

- Код — закрывается git (версионирование).
- Зависимости (какие библиотеки и каких версий) — закрывается `uv` и lockfile.
- Среда исполнения (какой Python, какая ОС) — закрывается виртуальным окружением, в пределе Docker.
- Данные — закрывается фиксацией версии датасета (хеши, DVC, snapshot в хранилище).
- Случайность — закрывается фиксацией random seed.

Важная интуиция: слои образуют цепочку, и результат воспроизводится только тогда, когда закрыты ВСЕ пять. Это логика слабого звена, а не суммы баллов. Можно идеально вести git, иметь чистый lockfile и зафиксированную среду — но если данные в источнике незаметно поменялись, результат всё равно «поплывёт». Один незакрытый слой обнуляет старания на остальных четырёх.

!!! info "Уровни строгости"

    Различают reproducibility (тот же код + те же данные дают тот же результат — минимальный обязательный уровень) и replicability (другой код или другие данные подтверждают тот же вывод — это уже про научную надёжность гипотезы). В прикладной аналитике сначала добиваются первого. Второе — признак зрелого исследования, когда вывод устойчив к деталям реализации.

Чтобы разница не была абстрактной: ты посчитал retention D7 своим SQL-запросом и получил 22%. Reproducibility — это когда коллега запускает твой же запрос на том же снапшоте данных и тоже получает ровно 22%. Replicability — это когда коллега пишет СВОЙ запрос, по-другому, на тех же сырых событиях, и получает 21–23%, подтверждая, что вывод «retention около 22%» — свойство данных, а не артефакт твоего конкретного кода. Replicability ловит ошибки, которые reproducibility пропускает: например, если в твоём запросе скрытый баг (неверный JOIN, который случайно даёт похожую цифру), то воспроизвести его дословно легко, а вот независимо подтвердить — уже нет.

## Почему случайность — отдельный слой

Слой «случайность» новичков удивляет чаще остальных: «при чём тут случайность, я же просто считаю метрики?». Дело в том, что половина инструментов DS внутри использует генератор случайных чисел: train/test split, инициализация весов модели, бутстрэп доверительных интервалов, сэмплирование данных, k-fold кросс-валидация. Без фиксации seed каждый запуск даёт чуть разные числа.

Минимальный пример, показывающий проблему и решение:

```python
import numpy as np

# Без фиксации seed — два запуска дадут разные выборки
sample_a = np.random.choice(1000, size=5)
sample_b = np.random.choice(1000, size=5)
print(sample_a, sample_b)  # каждый прогон скрипта печатает новое

# С фиксацией seed — результат детерминирован и воспроизводим
rng = np.random.default_rng(42)
print(rng.choice(1000, size=5))  # всегда одно и то же при seed=42
```

Практический эффект: если ты оцениваешь uplift A/B-теста через бутстрэп и не зафиксировал seed, доверительный интервал будет чуть-чуть «дышать» от запуска к запуску. На границе значимости (когда нижняя граница интервала болтается около нуля) это означает, что один прогон скажет «эффект значим, катим», а другой — «не значим, откатываем». Зафиксированный seed убирает этот шум: теперь любое расхождение в результате говорит о реальном изменении в коде или данных, а не о броске монетки внутри библиотеки.

## Практический критерий

Практический критерий: проект воспроизводим, если новый человек на чистой машине может выполнить три команды (склонировать, установить зависимости, запустить) и получить тот же результат, не задавая тебе вопросов. Всё, что мы дальше настраиваем, служит этому критерию.

```bash
git clone <repo-url> && cd project   # код — слой git
uv sync                              # зависимости + среда — слой uv/lockfile/venv
uv run python pipeline.py            # запуск; данные и seed зафиксированы внутри
```

Ключевые слова в критерии — «чистая машина» и «не задавая вопросов». «Чистая машина» означает: никаких глобально установленных пакетов, которые случайно подхватились; никаких файлов, которые лежат только у тебя на ноутбуке; никаких переменных окружения, про которые знаешь только ты. «Не задавая вопросов» означает: ни одного шага вида «а, там ещё надо вручную скачать файл из Slack» или «спроси у меня пароль к базе». Каждый такой вопрос — это незакрытый слой, замаскированный под устную традицию.

!!! warning "Типичные ошибки и пограничные случаи"

    - «У меня же работает» — самый опасный признак. Работает, потому что на твоей машине накопились невидимые зависимости (глобальный пакет, локальный файл, переменная окружения). На чистой машине это рассыпается.
    - `pip freeze > requirements.txt` фиксирует версии, но не lock-граф транзитивных зависимостей и не версию Python — это полумера, а не полноценный слой.
    - Данные «прибиты» к проду: запрос `SELECT ... FROM events` без фиксации снапшота даёт разный результат каждый день, потому что таблица пополняется. Нужен срез на дату или хеш версии.
    - Seed зафиксирован в одном месте, но библиотека внутри использует другой генератор (например, numpy зафиксирован, а PyTorch — нет). Фиксировать нужно все источники случайности, которые реально работают в пайплайне.
    - Воспроизводимость путают с правильностью. Воспроизводимый результат может быть стабильно неправильным — он лишь гарантирует, что ошибку можно найти и обсудить, а не что её нет.
