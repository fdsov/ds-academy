## Что отрабатываем

В модуле M0 ты разобрал пять фундаментальных ловушек данных и установку «по умолчанию себе не верь». Здесь ты применяешь это руками. Дан сырой гемблинг-датасет с зашитыми ловушками. На поверхности он подсказывает четыре красивых вывода — и все четыре ложные. Твоя задача — не построить ещё один график, а вскрыть, почему каждый вывод обманывает, на конкретных цифрах.

Отрабатываем напрямую:

- корреляция не равно причинность (общий конфаундер);
- survivorship bias (оценка только по выжившим);
- selection bias (анализ только по самоотобравшейся подвыборке);
- парадокс Симпсона (разворот тренда при агрегации);
- научный метод как процедура «атакуй свой вывод» и привычка считать неопределённость, а не голую точку.

Артефакт на выходе: разбор 3-4 ловушек, где для каждой выписано — наивный вывод с цифрой, контрцифра после корректного среза, и одно предложение «почему данные соврали». Это твой первый рабочий документ профессионального мышления.

## Бизнес-кейс

Ты — аналитик в команде iGaming-продукта. Перед квартальным планированием продакт-оунер приносит дашборд с четырьмя «победами» и хочет под них защитить бюджет: загнать всех игроков в live-чат, заложить в модель закупки трафика срок жизни игрока 14 месяцев, масштабировать retention-рассылку и откатить вариант B платёжной страницы. Все четыре вывода выглядят убедительно и подкреплены цифрами с дашборда. Твоя подпись под ними — это деньги, которые уйдут на инициативы, и решения, которые не отменить за квартал. До защиты бюджета у тебя один прогон по сырым данным.

!!! example "Ситуация"

    - **Чат и LTV.** «У пишущих в live-чат LTV ~250+ против ~40 — встроим чат-онбординг всем, поднимем средний LTV.» Под это просят бюджет на доработку CRM и онбординга.
    - **Срок жизни.** «VIP живут ~14 месяцев — закладываем эту цифру в окупаемость закупки трафика.» При таком lifetime команда готова поднять допустимый CAC и нарастить закуп, ориентировочно на сотни тысяч ₽ в месяц.
    - **Retention-кампания.** «Среди открывших письмо D30-retention 60% при базовых ~30% — масштабируем рассылку.» На кону регулярный бюджет на CRM-кампанию.
    - **A/B платёжной страницы.** «В агрегате A конвертит лучше — откатываем B.» Откат живой версии — потеря конверсии на всём платёжном трафике.

    Цена ошибки на каждом пункте — реальный бюджет или упущенная выручка. Цифры здесь иллюстративные (сценарий, не выгрузка из прода), но логика ловушек ровно та, что встречается в работе.

## Данные

Латентная переменная `engaged` (вовлечённость игрока) намеренно скрыта внутри генерации: именно она порождает большинство ловушек, и именно её студент не видит, пока не начнёт стратифицировать.

```python
import numpy as np
import polars as pl

rng = np.random.default_rng(42)
N = 10_000

engaged = rng.random(N) < 0.30                     # скрытый драйвер почти всего

used_chat = np.where(engaged, rng.random(N) < 0.60, rng.random(N) < 0.10)
ltv = np.where(engaged, rng.normal(300, 80, N), rng.normal(40, 20, N)).clip(0)

churned_w1 = np.where(engaged, rng.random(N) < 0.20, rng.random(N) < 0.70)
lifetime = np.where(
    churned_w1, rng.normal(0.25, 0.05, N),
    np.where(engaged, rng.normal(14, 3, N), rng.normal(3, 1, N)),
).clip(0.1)
is_vip = engaged & (~churned_w1) & (ltv > 250) & (lifetime > 10)

campaign = rng.choice(["test", "control"], N)
opened = np.where(engaged, rng.random(N) < 0.70, rng.random(N) < 0.15)
retained_d30 = rng.random(N) < np.where(engaged, 0.60, 0.20)   # эффект кампании ~ 0

players = pl.DataFrame({
    "used_chat": used_chat, "ltv": ltv, "churned_w1": churned_w1,
    "lifetime_m": lifetime, "is_vip": is_vip, "campaign": campaign,
    "opened_email": opened, "retained_d30": retained_d30,
})

# Отдельная A/B-таблица по платёжной странице для парадокса Симпсона
def make_ab(variant, n_desktop, n_mobile, p_desk, p_mob):
    dev = np.array(["desktop"] * n_desktop + ["mobile"] * n_mobile)
    conv = np.concatenate([
        rng.random(n_desktop) < p_desk,
        rng.random(n_mobile) < p_mob,
    ])
    return pl.DataFrame({"variant": variant, "device": dev, "converted": conv})

ab = pl.concat([
    make_ab("A", 3200, 800, 0.30, 0.15),   # A: трафик в основном desktop
    make_ab("B", 800, 3200, 0.35, 0.18),   # B: лучше в КАЖДОМ устройстве, но трафик в mobile
])

print(players.head())
print(ab.head())
```

Колонка `engaged` сознательно НЕ попала в `players` — ты должен прийти к ней сам, как в реальной таблице, где конфаундер не подписан.

## Ход работы

### Шаг 1: разведка вслепую

Зачем. Прежде чем верить любому выводу, профессионал смотрит, кто вообще в данных и как они распределены. Это первый барьер против selection и survivorship bias — вопрос «кого здесь нет?».

```python
print(players.describe())
print(players.select(
    pl.len().alias("rows"),
    pl.col("churned_w1").mean().alias("share_churned_w1"),
    pl.col("is_vip").mean().alias("share_vip"),
    pl.col("opened_email").mean().alias("share_opened"),
))
```

Что получилось. Около 49% игроков отвалились на первой неделе, VIP — единицы процентов, письмо открыли далеко не все. Запомни эти доли: дальше ровно эти отвалившиеся и не открывшие будут «исчезать» из наивных выводов и тащить их в ложь.

### Шаг 2: ложная корреляция (чат и LTV)

Зачем. Отрабатываем «корреляция не равно причинность». Наивный вывод: «у игроков, пишущих в live-чат, LTV выше — загоним всех в чат, LTV вырастет». Проверяем причинность стратификацией по подозреваемому конфаундеру.

```python
naive = players.group_by("used_chat").agg(
    pl.col("ltv").mean().round(1).alias("avg_ltv"), pl.len().alias("n")
).sort("used_chat")
print(naive)
```

Чата касается ловушка: разница огромна (примерно 40 против 250+). Но `used_chat` мы сами сгенерировали из вовлечённости. Восстановим прокси конфаундера и разрежем по нему:

```python
players = players.with_columns(
    (pl.col("ltv") > 150).alias("engaged_proxy")   # грубый прокси вовлечённости
)
stratified = players.group_by(["engaged_proxy", "used_chat"]).agg(
    pl.col("ltv").mean().round(1).alias("avg_ltv"), pl.len().alias("n")
).sort(["engaged_proxy", "used_chat"])
print(stratified)
```

Что получилось. Внутри каждой группы по вовлечённости разрыв LTV между «пишет в чат» и «не пишет» резко схлопывается. Значит чат — симптом вовлечённости, а не её причина. Решение «загнать всех в чат» не даст роста LTV.

!!! question "Проверь себя"

    2. Почему «загнать всех в чат» не сработает, хотя корреляция реальна и сильна?
    3. Какой эксперимент доказал бы причинность, если бы она была?

??? success "Ответы"

    1. Корреляция не равно причинность; конфаундер — вовлечённость (`engaged`), которая одновременно гонит игрока и в чат, и к высоким тратам.
    2. Чат не делает игрока ценным; ценные игроки и так пишут в чат. Принудив невовлечённых писать, ты не меняешь их вовлечённость, а значит и LTV.
    3. Рандомизированный A/B: случайно подталкивать часть игроков к чату и сравнивать LTV тест/контроль по всем назначенным, а не по среагировавшим.

### Шаг 3: survivorship bias (срок жизни VIP)

Зачем. Отрабатываем ошибку выжившего. Наивный вывод с дашборда: «наши VIP в среднем живут 14 месяцев, значит типичный игрок живёт долго». Проверяем, кто отобран в VIP-когорту и кого в ней нет по определению.

```python
vip_life = players.filter(pl.col("is_vip")).select(
    pl.col("lifetime_m").mean().round(1).alias("vip_avg_lifetime")
)
full_life = players.select(
    pl.col("lifetime_m").mean().round(1).alias("full_avg_lifetime")
)
print(vip_life, full_life)
```

Что получилось. По VIP — около 14 месяцев, по полной входной когорте — около 4 месяцев. Разрыв больше чем втрое. VIP по построению исключает всех, кто отвалился на первой неделе (а это ~49% входа). Считать lifetime и LTV по выжившим — гарантированно завысить прогноз ценности игрока.

### Шаг 4: selection bias (retention-кампания)

Зачем. Отрабатываем смещение отбора. Наивный вывод: «retention-кампания удерживает — среди получивших письмо D30-retention 60%, при базовых 30%». Проверяем, по кому считали и сравнили ли тест с контролем честно.

```python
naive_camp = players.filter(pl.col("opened_email")).select(
    pl.col("retained_d30").mean().round(3).alias("retention_among_openers")
)
honest_camp = players.group_by("campaign").agg(
    pl.col("retained_d30").mean().round(3).alias("retention"),
    pl.len().alias("n_sent"),
).sort("campaign")
print(naive_camp)
print(honest_camp)
```

Что получилось. Среди открывших — высокий retention, но test и control по ВСЕМ отправленным практически совпадают: истинный эффект кампании около нуля. «Эффект» был полностью смещением — письмо открывают более вовлечённые, у них retention высок и без всякой кампании. Сравнивать надо по всем, кому отправили, а не по среагировавшим.

!!! question "Проверь себя"

    1. Чем survivorship bias из шага 3 отличается от selection bias из шага 4?
    2. Почему retention 60% среди открывших — не доказательство, что кампания работает?
    3. По какой подвыборке надо сравнивать test и control, чтобы вывод был валиден?

??? success "Ответы"

    1. Survivorship — частный случай selection: критерий отбора — «дожитие» (VIP = не отвалившиеся), погибшие исключены. В шаге 4 критерий отбора — самоотбор по реакции (открыл письмо), тоже коррелированной с вовлечённостью.
    2. Открывшие письмо — уже более вовлечённые, у них retention высок сам по себе; «эффект» перепутан с составом подвыборки. Контроль по всем отправленным показывает ~0.
    3. По всем, кому отправили (intention-to-treat): test против control независимо от того, открыли письмо или нет.

### Шаг 5: парадокс Симпсона (A/B платёжной страницы) с доверительным интервалом

Зачем. Отрабатываем разворот тренда при агрегации и привычку M0 «точка без интервала — мнение». Наивный вывод: «в агрегате A конвертит лучше B — откатываем B». Проверяем разбивкой по конфаундеру `device` и считаем неопределённость.

```python
import numpy as np
from scipy import stats

def conv_ci(s: int, n: int, alpha: float = 0.05):
    p = s / n
    half = stats.norm.ppf(1 - alpha / 2) * np.sqrt(p * (1 - p) / n)
    return round(p, 3), (round(p - half, 3), round(p + half, 3))

by_cell = ab.group_by(["device", "variant"]).agg(
    pl.col("converted").sum().alias("s"), pl.len().alias("n")
).sort(["device", "variant"])
agg = ab.group_by("variant").agg(
    pl.col("converted").sum().alias("s"), pl.len().alias("n")
).sort("variant")

for row in by_cell.iter_rows(named=True):
    print(row["device"], row["variant"], conv_ci(row["s"], row["n"]))
print("---")
for row in agg.iter_rows(named=True):
    print(row["variant"], conv_ci(row["s"], row["n"]))
```

Что получилось. На desktop и на mobile по отдельности B конвертит выше A, а в агрегате A выигрывает. Конфаундер — состав трафика: в B загнали в основном mobile с низкой базовой конверсией, и агрегат смешал «эффект версии» с «эффектом устройства». Доверительные интервалы внутри ячеек не пересекаются — разворот не шум, а арифметика весов. Правильное решение — оставить B и/или стратифицировать рандомизацию, а не откатывать.

    2. Зачем здесь доверительный интервал, а не одна точка конверсии?
    3. Как изменить дизайн A/B, чтобы парадокс не возник в принципе?

??? success "Ответы"

    1. Из-за неравных весов устройств: B перекошен в mobile с низкой базой, и средневзвешенное по A обгоняет B, хотя в каждой ячейке B лучше.
    2. Чтобы отличить реальный разрыв от шума: непересекающиеся интервалы внутри ячеек подтверждают, что разворот — следствие весов, а не случайности.
    3. Стратифицированная рандомизация: балансировать долю desktop/mobile между вариантами, тогда агрегат не смешивает версию и устройство.

- [ ] **Риски и допущения.** Назови, на чём держится вывод: прокси вовлечённости грубый, истинный `engaged` не наблюдаем; эффект кампании оценён как ~0 в пределах доверительного интервала, а не строго ноль; разворот в A/B подтверждён непересекающимися ДИ по ячейкам.
- [ ] **Следующий шаг.** Чтобы доказать причинность там, где она возможна — рандомизированный A/B (чат, рассылка) с оценкой по всем назначенным (intention-to-treat); для закупки — считать lifetime по полной входной когорте, а не по выжившим.
- [ ] **Как подать стейкхолдеру.** Один слайд: «эти четыре вывода сэкономят/сожгут вот столько ₽; три из них — артефакт смещения, один разворачивает решение об откате. Прежде чем тратить — вот что проверить экспериментом.» Без терминов «конфаундер» и «стратификация» — на языке бюджета и риска.

## Развитие

- Замени прокси конфаундера: вместо `ltv > 150` восстанови `engaged` точнее (через несколько признаков) и посмотри, как сильнее схлопывается ложная корреляция чата.
- Поменяй распределение трафика в `make_ab` так, чтобы парадокс Симпсона исчез, и объясни в двух предложениях, какое именно свойство данных его порождало.
- Добавь пятую ловушку — p-hacking: прогони A/B-эффект кампании по 15 случайным сегментам (гео, день недели, устройство) и покажи, что хотя бы один «значим» на $\alpha=0.05$ при истинном эффекте ноль; посчитай $1-(1-\alpha)^m$.
- Заверни шаг 6 в функцию `audit(conclusion)`, которая по тексту вывода печатает чек-лист вопросов из M0 («кого нет в данных?», «есть ли конфаундер?», «сравнил ли по всем?») — заготовка командного adversarial-reviewer.
