# Воркшоп M0 · Разоблачение ловушек в сыром датасете

<span class="lecture-meta">Воркшоп к модулю M0 · ориентир 3-5 ч</span>

## Что ты построишь

В модуле M0 ты разобрал пять фундаментальных ловушек данных и установку «по умолчанию себе не верь». Здесь ты применяешь это руками. Дан сырой гемблинг-датасет с зашитыми ловушками. На поверхности он подсказывает четыре красивых вывода — и все четыре ложные. Твоя задача — не построить ещё один график, а вскрыть, почему каждый вывод обманывает, на конкретных цифрах.

Ты отработаешь напрямую:

- корреляция не равно причинность (общий конфаундер);
- survivorship bias (оценка только по выжившим);
- selection bias (анализ только по самоотобравшейся подвыборке);
- парадокс Симпсона (разворот тренда при агрегации);
- научный метод как процедура «атакуй свой вывод» и привычка считать неопределённость, а не голую точку.

Артефакт на выходе — разбор 3-4 ловушек, где для каждой выписано: наивный вывод с цифрой, контрцифра после корректного среза и одно предложение «почему данные соврали». Это твой первый рабочий документ профессионального мышления.

!!! info "Как устроен этот воркшоп"

    Это не лекция с готовым кодом, а задачник. Каждый содержательный шаг построен так:

    - **Задача** — что именно нужно сделать руками, с явными входами и именами выходных переменных.
    - **Критерий шага** — кусок кода с `assert`, который ты запускаешь после своего решения. Зелёный прогон (без ошибки) = шаг сдан. Это твой локальный авто-грейдер, аналог кнопки «Проверить» на Stepik.
    - **Решение** спрятано под спойлер `Решение`. Открывай его только после своей попытки — чтобы сверить подход, а не списать.
    - **Числовые подзадачи** проверяются прямо на странице: посчитай число, впиши в поле, нажми «Проверить».

    Все `assert` и числа рассчитаны на сгенерированные данные с `seed=42`. Не меняй seed, иначе цифры поплывут.

## Бизнес-кейс

!!! example "Ситуация"

    Ты — аналитик в команде iGaming-продукта. Перед квартальным планированием продакт-оунер приносит дашборд с четырьмя «победами» и хочет под них защитить бюджет: загнать всех игроков в live-чат, заложить в модель закупки трафика срок жизни игрока 14 месяцев, масштабировать retention-рассылку и откатить вариант B платёжной страницы. Все четыре вывода выглядят убедительно и подкреплены цифрами с дашборда. Твоя подпись под ними — это деньги, которые уйдут на инициативы, и решения, которые не отменить за квартал. До защиты бюджета у тебя один прогон по сырым данным.

    - **Чат и LTV.** «У пишущих в live-чат LTV кратно выше — встроим чат-онбординг всем, поднимем средний LTV.» Под это просят бюджет на доработку CRM и онбординга.
    - **Срок жизни.** «VIP живут ~14 месяцев — закладываем эту цифру в окупаемость закупки трафика.» При таком lifetime команда готова поднять допустимый CAC и нарастить закуп, ориентировочно на сотни тысяч ₽ в месяц.
    - **Retention-кампания.** «Среди открывших письмо D30-retention заметно выше базы — масштабируем рассылку.» На кону регулярный бюджет на CRM-кампанию.
    - **A/B платёжной страницы.** «В агрегате A конвертит лучше — откатываем B.» Откат живой версии — потеря конверсии на всём платёжном трафике.

    Цена ошибки на каждом пункте — реальный бюджет или упущенная выручка. Цифры здесь иллюстративные (сценарий, не выгрузка из прода), но логика ловушек ровно та, что встречается в работе.

## Данные

Никаких внешних файлов — генератор синтетики ниже запускается у любого, seed зафиксирован, цифры воспроизводимы. Латентная переменная `engaged` (вовлечённость игрока) намеренно скрыта внутри генерации: именно она порождает большинство ловушек, и именно её студент не видит, пока не начнёт стратифицировать.

Это setup-код, а не задача — скопируй и запусти как есть.

!!! warning "Зачем фиксировать seed"

    Без `seed` каждый запуск даст другие числа, и ты не сможешь сверить результат с воркшопом или воспроизвести баг. Фиксированный seed — основа воспроизводимого анализа. Все критерии-`assert` и поля ниже рассчитаны на `seed=42`.

```python
import numpy as np
import polars as pl

rng = np.random.default_rng(42)
N = 10_000

engaged = rng.random(N) < 0.30                     # скрытый драйвер почти всего

used_chat = np.where(engaged, rng.random(N) < 0.60, rng.random(N) < 0.10)
ltv = np.where(engaged, rng.normal(300, 80, N), rng.normal(40, 20, N)).clip(0)

churned_w1 = np.where(engaged, rng.random(N) < 0.20, rng.random(N) < 0.70)
lifetime = np.where(
    churned_w1, rng.normal(0.25, 0.05, N),
    np.where(engaged, rng.normal(14, 3, N), rng.normal(3, 1, N)),
).clip(0.1)
is_vip = engaged & (~churned_w1) & (ltv > 250) & (lifetime > 10)

campaign = rng.choice(["test", "control"], N)
opened = np.where(engaged, rng.random(N) < 0.70, rng.random(N) < 0.15)
retained_d30 = rng.random(N) < np.where(engaged, 0.60, 0.20)   # эффект кампании ~ 0

players = pl.DataFrame({
    "used_chat": used_chat, "ltv": ltv, "churned_w1": churned_w1,
    "lifetime_m": lifetime, "is_vip": is_vip, "campaign": campaign,
    "opened_email": opened, "retained_d30": retained_d30,
})

# Отдельная A/B-таблица по платёжной странице для парадокса Симпсона
def make_ab(variant, n_desktop, n_mobile, p_desk, p_mob):
    dev = np.array(["desktop"] * n_desktop + ["mobile"] * n_mobile)
    conv = np.concatenate([
        rng.random(n_desktop) < p_desk,
        rng.random(n_mobile) < p_mob,
    ])
    return pl.DataFrame({"variant": variant, "device": dev, "converted": conv})

ab = pl.concat([
    make_ab("A", 3200, 800, 0.30, 0.15),   # A: трафик в основном desktop
    make_ab("B", 800, 3200, 0.35, 0.18),   # B: лучше в КАЖДОМ устройстве, но трафик в mobile
])

print(players.head())
print(ab.head())
```

Колонка `engaged` сознательно НЕ попала в `players` — ты должен прийти к ней сам, как в реальной таблице, где конфаундер не подписан.

## Ход работы

### Шаг 1: Разведка вслепую

**Зачем.** Прежде чем верить любому выводу, профессионал смотрит, кто вообще в данных и как они распределены. Это первый барьер против selection и survivorship bias — вопрос «кого здесь нет?».

**Задача.** Посчитай ключевые доли входной когорты и сложи их в один фрейм `shares` с колонками `share_churned_w1`, `share_vip`, `share_opened`. Это твоя точка отсчёта: запомни, сколько игроков отвалилось и сколько вообще открыло письмо — дальше именно они будут «исчезать» из наивных выводов.

**Критерий шага:**

```python
assert 0.50 <= shares["share_churned_w1"][0] <= 0.60, "около половины отваливается на первой неделе"
assert 0.12 <= shares["share_vip"][0] <= 0.20, "VIP — заметная, но не доминирующая доля"
assert 0.28 <= shares["share_opened"][0] <= 0.37, "письмо открыли далеко не все"
print("OK: точка отсчёта зафиксирована")
```

??? success "Решение"

    ```python
    print(players.describe())
    shares = players.select(
        pl.col("churned_w1").mean().alias("share_churned_w1"),
        pl.col("is_vip").mean().alias("share_vip"),
        pl.col("opened_email").mean().alias("share_opened"),
    )
    print(shares)
    ```

    **Почему так.** Около 55% игроков отвалились на первой неделе, VIP ~16%, письмо открыли ~32%. Эти доли — не статистика ради статистики. Дальше ровно отвалившиеся и не открывшие будут вылетать из наивных срезов и тащить выводы в ложь. Привычка начинать с «кого здесь нет?» — это и есть защита от смещения отбора.

### Шаг 2: Ложная корреляция (чат и LTV)

**Зачем.** Отрабатываем «корреляция не равно причинность». Наивный вывод: «у игроков, пишущих в live-чат, LTV выше — загоним всех в чат, LTV вырастет». Проверяем причинность стратификацией по подозреваемому конфаундеру.

**Задача.** Сначала воспроизведи наивный срез: средний LTV по группам `used_chat`, результат в `naive` (колонка `avg_ltv`). Затем восстанови грубый прокси конфаундера `engaged_proxy = ltv > 150`, добавь его в `players` и пересчитай средний LTV по парам (`engaged_proxy`, `used_chat`) — результат в `strat`. Смотри, что происходит с разрывом «чат / не чат» внутри каждой группы вовлечённости.

??? tip "Подсказка"

    Прокси не обязан быть идеальным — он лишь грубо делит игроков на «вовлечённых» и «нет». Если внутри одной группы вовлечённости разрыв LTV между чатом и не-чатом почти исчезает, значит исходная корреляция держалась на третьей переменной, а не на самом чате.

**Критерий шага:**

```python
ltv_chat = naive.filter(pl.col("used_chat"))["avg_ltv"][0]
ltv_nochat = naive.filter(~pl.col("used_chat"))["avg_ltv"][0]
assert ltv_chat > 180 and ltv_nochat < 110, "наивный разрыв должен быть большим"
gap_naive = ltv_chat - ltv_nochat

hi = strat.filter(pl.col("engaged_proxy"))
gap_hi = abs(hi.filter(pl.col("used_chat"))["avg_ltv"][0]
             - hi.filter(~pl.col("used_chat"))["avg_ltv"][0])
assert gap_hi < 30, "внутри вовлечённой группы разрыв должен схлопнуться"
assert gap_hi < gap_naive / 3, "стратификация убивает большую часть разрыва"
print(f"OK: наивный разрыв {gap_naive:.0f}, внутри группы {gap_hi:.0f} — корреляция схлопнулась")
```

??? success "Решение"

    ```python
    naive = players.group_by("used_chat").agg(
        pl.col("ltv").mean().round(1).alias("avg_ltv"), pl.len().alias("n")
    ).sort("used_chat")
    print(naive)

    players = players.with_columns(
        (pl.col("ltv") > 150).alias("engaged_proxy")   # грубый прокси вовлечённости
    )
    strat = players.group_by(["engaged_proxy", "used_chat"]).agg(
        pl.col("ltv").mean().round(1).alias("avg_ltv"), pl.len().alias("n")
    ).sort(["engaged_proxy", "used_chat"])
    print(strat)
    ```

    **Почему так.** Наивно: чат ~226 против ~83 — разрыв ~142. Но внутри вовлечённой группы LTV ~303 против ~309 (разрыв ~6), внутри невовлечённой ~40 против ~44. То есть `used_chat` мы сами сгенерировали из вовлечённости: чат — симптом, а не причина. «Загнать всех в чат» не меняет вовлечённость, а значит и LTV. Корреляция реальна и сильна, но к причинности отношения не имеет.

Проверь понимание:

```text
Q: Какая ловушка тут основная и какой конфаундер её порождает?
[ ] Survivorship bias; конфаундер — размер первого депозита
[x] Корреляция не равно причинность; конфаундер — вовлечённость (engaged), гонящая игрока и в чат, и к высоким тратам
[ ] Парадокс Симпсона; конфаундер — устройство
> engaged одновременно повышает шанс писать в чат и повышает траты. Поэтому чат и LTV коррелируют, не будучи связаны причинно.
---
Q: Почему «загнать всех в чат» не поднимет LTV, хотя корреляция реальна и сильна?
[ ] Потому что чат технически нельзя включить всем
[x] Чат не делает игрока ценным — ценные и так пишут; принудив невовлечённых, ты не меняешь их вовлечённость
[ ] Потому что выборка слишком мала для вывода
> Ты воздействуешь на симптом, а не на драйвер. Вовлечённость невовлечённых не вырастет от того, что они напишут в чат, поэтому и LTV не сдвинется.
---
Q: Какой эксперимент доказал бы причинность, будь она реальной?
[ ] Сравнить LTV у писавших и не писавших в чат на большей выборке
[x] Рандомизированный A/B: случайно подталкивать часть к чату и сравнивать LTV по всем назначенным (intention-to-treat)
[ ] Построить регрессию LTV на факт использования чата
> Только рандомизация разрывает связь чата с вовлечённостью. Сравнивать надо по всем назначенным, а не по среагировавшим, иначе вернётся тот же самоотбор.
```

### Шаг 3: Survivorship bias (срок жизни VIP)

**Зачем.** Отрабатываем ошибку выжившего. Наивный вывод с дашборда: «наши VIP в среднем живут ~14 месяцев, значит типичный игрок живёт долго». Проверяем, кто отобран в VIP-когорту и кого в ней нет по определению.

**Задача.** Посчитай средний `lifetime_m` по VIP (`vip_life`, скаляр) и по всей входной когорте (`full_life`, скаляр). Сравни их и подумай, кто по построению исключён из VIP.

**Критерий шага:**

```python
assert vip_life > 12, "по VIP lifetime должен быть около 14 месяцев"
assert full_life < 6, "по полной когорте lifetime должен быть в разы меньше"
assert vip_life / full_life > 3, "разрыв должен быть больше чем втрое"
print(f"OK: VIP {vip_life:.1f} мес против полной когорты {full_life:.1f} мес")
```

Теперь оцени масштаб завышения числом:

```text
TASK: VIP средний lifetime ≈ 14.5 мес, полная входная когорта ≈ 4.2 мес. Во сколько раз прогноз по VIP завышает реальный срок жизни? Округли до 0.1.
ANSWER: 3.5
TOL: 0.3
PLACEHOLDER: 0.0
EXPLAIN: 14.5 / 4.2 = 3.45 ≈ 3.5 раза. VIP по построению исключает всех, кто отвалился на первой неделе (а это больше половины входа). Заложишь 14 месяцев в модель закупки — переплатишь CAC примерно втрое-вчетверо: окупаемость трафика рассчитана на «бессмертных» игроков, которых в реальной когорте меньшинство.
```

??? success "Решение"

    ```python
    vip_life = players.filter(pl.col("is_vip"))["lifetime_m"].mean()
    full_life = players["lifetime_m"].mean()
    print(f"VIP: {vip_life:.1f} мес | полная когорта: {full_life:.1f} мес")
    ```

    **Почему так.** По VIP ~14.5 месяцев, по полной входной когорте ~4.2. VIP-фильтр (`engaged & ~churned_w1 & ltv>250 & lifetime>10`) по построению выкидывает всех, кто отвалился на первой неделе — а это больше половины входа (помнишь долю с шага 1?). Считать lifetime и LTV по выжившим — гарантированно завысить прогноз ценности игрока. Это классический survivorship: оцениваешь самолёты, вернувшиеся с задания, и не видишь тех, что не вернулись.

### Шаг 4: Selection bias (retention-кампания)

**Зачем.** Отрабатываем смещение отбора. Наивный вывод: «retention-кампания удерживает — среди получивших письмо D30-retention заметно выше базы». Проверяем, по кому считали и сравнили ли тест с контролем честно.

**Задача.** Посчитай retention среди открывших письмо (`ret_openers`, скаляр). Затем честно сравни группы рандомизации: retention по `campaign` (test против control) по ВСЕМ отправленным — результат в `camp` (колонка `r`). Сопоставь наивную цифру с честным сравнением.

**Критерий шага:**

```python
assert ret_openers > 0.40, "среди открывших retention высокий"
r_ctrl = camp.filter(pl.col("campaign") == "control")["r"][0]
r_test = camp.filter(pl.col("campaign") == "test")["r"][0]
assert abs(r_test - r_ctrl) < 0.03, "истинный эффект кампании близок к нулю"
print(f"OK: openers={ret_openers:.3f}, но test={r_test:.3f} ≈ control={r_ctrl:.3f}")
```


### Шаг 5: Парадокс Симпсона (A/B платёжной страницы) с доверительным интервалом

    ```python
    import numpy as np
    from scipy import stats

    def conv_ci(s: int, n: int, alpha: float = 0.05):
        p = s / n
        half = stats.norm.ppf(1 - alpha / 2) * np.sqrt(p * (1 - p) / n)
        return round(p, 3), (round(p - half, 3), round(p + half, 3))

    by_cell = ab.group_by(["device", "variant"]).agg(
        pl.col("converted").sum().alias("s"), pl.len().alias("n")
    ).sort(["device", "variant"])
    agg = ab.group_by("variant").agg(
        pl.col("converted").sum().alias("s"), pl.len().alias("n")
    ).sort("variant")

    for row in by_cell.iter_rows(named=True):
        print(row["device"], row["variant"], conv_ci(row["s"], row["n"]))
    print("---")
    for row in agg.iter_rows(named=True):
        print(row["variant"], conv_ci(row["s"], row["n"]))
    ```

    **Почему так.** На desktop B ~0.356 против A ~0.305, на mobile B ~0.183 против A ~0.150 — B выигрывает в КАЖДОМ устройстве. А в агрегате A ~0.274 обгоняет B ~0.217. Конфаундер — состав трафика: в B загнали в основном mobile с низкой базовой конверсией, и агрегат смешал «эффект версии» с «эффектом устройства». Доверительные интервалы внутри ячеек не пересекаются — разворот не шум, а арифметика весов. Правильное решение — оставить B и/или стратифицировать рандомизацию, а не откатывать.

```text
[ ] ДИ требует регламент, на вывод не влияет
[x] Чтобы отличить реальный разрыв от шума: непересекающиеся интервалы внутри ячеек подтверждают, что разворот — следствие весов, а не случайности
[ ] ДИ нужен только для агрегата, в ячейках он лишний
```
