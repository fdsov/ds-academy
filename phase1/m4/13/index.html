    M4.13 · EXPLAIN — читаем план запроса¶

    Интуиция. Прежде чем база выполнит запрос, она строит план — пошаговую стратегию "как именно я это посчитаю". EXPLAIN показывает этот план, не выполняя запрос. EXPLAIN ANALYZE — выполняет и показывает реальное время.

    На пальцах. У одного и того же SQL-запроса бывает десяток способов выполнения: прочитать таблицу целиком и отфильтровать; прыгнуть через индекс прямо к нужным строкам; соединить две таблицы циклом или через хэш-таблицу. Планировщик (planner/optimizer) оценивает стоимость каждого варианта в условных единицах (cost) и выбирает самый дешёвый. EXPLAIN — это его "объяснительная записка": какой план он выбрал и почему он считает его дешёвым. Аналитик читает этот план, чтобы понять, где запрос буксует, и где база делает лишнюю работу.

    Формально:

    EXPLAIN ANALYZE
SELECT * FROM deposits WHERE player_id = 12345;

    Что искать в выводе:

    • Seq Scan (sequential scan) — база читает таблицу целиком. На большой таблице с фильтром это красный флаг: вероятно, не хватает индекса.
    • Index Scan — база использует индекс. То, что нужно.
    • Nested Loop / Hash Join / Merge Join — алгоритмы соединения. Hash Join хорош для больших таблиц без сортировки, Nested Loop — для маленьких.
    • Числа rows и actual time — оценка и реальность. Если планировщик ждал 100 строк, а пришёл миллион, статистика устарела — стоит выполнить ANALYZE table.

    Как читать дерево плана. План — это дерево, которое выполняется снизу вверх и изнутри наружу: самые вложенные (с наибольшим отступом) узлы выполняются первыми, их результат поднимается к родительскому узлу. Каждый узел описывает одну операцию. Разберём строку плана по частям:

    Index Scan using deposits_player_id_idx on deposits
  (cost=0.43..8.45 rows=1 width=64)
  (actual time=0.021..0.023 rows=1 loops=1)
    • cost=0.43..8.45 — два числа: оценочная стоимость старта (когда выдастся первая строка) и полная стоимость (когда выдастся последняя). Единицы условные, не миллисекунды: их нельзя сравнивать с реальным временем, только узлы между собой.
    • rows=1 — сколько строк планировщик ожидает от этого узла.
    • width=64 — средняя ширина строки в байтах.
    • В блоке actual — то, что произошло на самом деле при ANALYZE: actual time=0.021..0.023 (мс), rows=1 (фактически вернулось), loops=1 (сколько раз узел запускался; реальное суммарное время = actual time × loops).

    Разобранный пример из гемблинга. Дашборд считает депозиты игрока. Таблица deposits — 80 млн строк. Первый вариант запроса:

    EXPLAIN ANALYZE
SELECT player_id, SUM(amount_usd) AS total
FROM deposits
WHERE created_at::date = '2026-06-01'
GROUP BY player_id;

    В плане видим Seq Scan on deposits ... rows=80000000 ... actual time=... rows=42000: база прочитала все 80 млн строк, чтобы оставить 42 тысячи за нужный день. Причина — created_at::date оборачивает столбец функцией приведения типа, и индекс по сырому created_at не применяется (планировщик не знает, что ::date монотонна по исходному значению). Переписываем на диапазон:

    EXPLAIN ANALYZE
SELECT player_id, SUM(amount_usd) AS total
FROM deposits
WHERE created_at >= '2026-06-01' AND created_at < '2026-06-02'
GROUP BY player_id;

    Теперь в плане Index Scan (или Bitmap Index Scan + Bitmap Heap Scan) по created_at, прочитано ~42 тыс. строк вместо 80 млн, и actual time падает с секунд до миллисекунд. Это типичный переход от 30-секундного таймаута к мгновенному дашборду — без добавления железа, одной переписанной строкой.

    Пограничные случаи и типичные ошибки:

    • Seq Scan не всегда зло. Если запрос всё равно читает большую долю таблицы (например, агрегат по всем строкам без фильтра), последовательное чтение быстрее индекса — индекс выгоден, когда выбирается малая часть строк (низкая селективность фильтра).
    • Sargable-условия. Чтобы индекс работал, столбец в WHERE должен стоять "голым", без обёрток: created_at >= ... хорошо, date_trunc('day', created_at) = ... или EXTRACT(...) или amount * 1.0 > 100 — плохо. Альтернатива переписыванию — функциональный индекс: CREATE INDEX ON deposits ((created_at::date)).
    • EXPLAIN ANALYZE реально выполняет запрос. Для SELECT это безопасно, но EXPLAIN ANALYZE DELETE/UPDATE/INSERT физически изменит данные. Оборачивайте в транзакцию с откатом: BEGIN; EXPLAIN ANALYZE DELETE ...; ROLLBACK;.
    • LIMIT меняет план. Запрос с LIMIT 10 может выбрать другой план (дешёвый старт), чем тот же запрос без лимита (дешёвый итог) — потому планировщик и показывает стоимость двумя числами.
    • Расхождение rows. Сильный разрыв между оценочным rows и фактическим — почти всегда устаревшая статистика. Лечится ANALYZE deposits; (или настройкой autovacuum). Плохая оценка ведёт к плохому выбору join-алгоритма: планировщик думает, что строк мало, берёт Nested Loop, а строк миллион — и запрос встаёт.

    Умение читать EXPLAIN — это переход от "запрос работает" к "запрос работает за 50 мс вместо 30 секунд". На больших данных это разница между рабочим дашбордом и таймаутом.

    Как смотреть план в жизни

    Для сложных запросов используйте EXPLAIN (ANALYZE, BUFFERS) — добавится информация о том, сколько данных читалось из кэша (shared hit) против диска (read). А визуализаторы вроде explain.dalibo.com или PEV подсвечивают самый дорогой узел дерева — удобно, когда план в сотню строк.

    Проверь себя

    1. На что в выводе EXPLAIN указывает Seq Scan на большой таблице с фильтром и почему это красный флаг?
    2. Чем EXPLAIN отличается от EXPLAIN ANALYZE?
    3. Почему WHERE date_trunc('day', ts) = ... мешает использовать индекс по ts и как переписать условие?
    4. Что значит сильное расхождение оценки rows и реального числа строк, и что с этим делать?
    Ответы
    1. Seq Scan — чтение всей таблицы подряд; при точечном фильтре на больших данных это признак отсутствия подходящего индекса.
    2. EXPLAIN показывает план без выполнения; EXPLAIN ANALYZE реально выполняет запрос и показывает фактическое время и число строк.
    3. Функция оборачивает столбец, и индекс по сырому ts не применяется. Переписать на диапазон: ts >= ... AND ts < ... (или сделать функциональный индекс).
    4. Статистика планировщика устарела — стоит выполнить ANALYZE table, чтобы план строился по актуальному распределению.

    Выбери вариант — ответ подсветится сразу.
